# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 项目定位

这是一个**学习仓库**。用户拉取 yudao-cloud（ruoyi-vue-pro）是为了研究其架构和设计模式。主要工作是逐步生成项目的学习文档，让读者能跟着文档**从零搭建一个一模一样的项目**，最终具备搭建类似微服务平台的能力。

编写要求：
- 文档用**中文**编写
- 示例统一用**电商系统**举例，禁止使用国际税务等业务举例
- 优先**理解优于参考**——讲清设计背后的"为什么"，而不只是"是什么"
- 用项目中的真实代码做示例，从源码复制，不要凭记忆重写
- 将设计模式与通用软件工程概念关联（DDD、SOLID、微服务模式等）

提交规范：
- **每次完成文档内容的新增或修改后，立即提交并推送到远程仓库**
- 直接在 `master` 分支上提交，不要创建新分支
- 提交信息格式：`docs: {简要描述}`，如 `docs: 新增第零阶段课程式讲解`

## 学习路径

文档按**项目搭建顺序**组织，分为六个阶段。读者跟着走，每完成一个阶段就有一个可运行的成果。

### 第零阶段：全局视角（建立地图）

目标：读者在动手之前，先知道要建什么、为什么这样建。

包含两部分（分开编写，先讲解后文档）：
1. **课程式讲解**——从一个真实场景出发（如"我要做一个电商后台管理系统"），带读者走一遍设计思考过程
2. **架构设计文档**——完整的架构全景，供后续阶段随时回查

按需生成，不强制固定章节结构。典型内容：
- 项目定位与要解决的问题
- 整体架构全景（单体 vs 微服务、三层模块设计、部署模式）
- 技术选型与设计原则（为什么选这些技术栈、放弃了什么）
- 一个请求的完整生命周期（浏览器 → 网关 → 认证 → Controller → Service → Mapper → 响应）
- 核心设计模式速览（URL 前缀注入、RPC 机制、多租户隔离、数据权限等）

### 文档分层

每个组件的文档分为三部分，形成 **为什么这样设计 → 怎么一步步实现 → 怎么迁移改进** 的阅读顺序：

| 维度 | 功能设计 | 功能实现 | 深度学习 |
|------|---------|---------|---------|
| 核心问题 | 为什么这样设计 | 怎么一步步实现 | 怎么迁移、怎么改进、怎么选型 |
| 代码的角色 | 辅助说明设计意图（片段） | 主体内容（完整代码） | 引用源码做分析 |
| 读者收获 | 理解设计背后的"为什么" | 能动手实现一个类似组件 | 能带着经验去解决其他项目中的类似问题 |
| 生成时机 | 优先生成 | 设计完成后生成 | 按需生成 |

功能设计不逐行解读源码，功能实现不重复讲设计思想。两者通过交叉引用衔接。

### 功能设计（讲解为主）

定位：一门完整的组件设计课程。把读者当成初学者，从第三方技术的前置知识补课开始，到场景驱动的设计讲解，再到边界与扩展，完完整整地讲透一个组件的设计思想。

#### 前置知识

前置知识作为功能设计的章节，**分散放置在相关设计章节之前**，形成"先补课、再深入"的阅读节奏。

- 命名规范：`{编号}-前置知识：{主题}.md`
- 只讲解组件涉及到的部分（够用即可）
- 从底向上：先讲第三方技术本身，再讲 Spring 的封装抽象，再讲框架的设计
- 用类比和示意图帮助理解

#### 章节组织

章节数量不限制。核心宗旨是完完全全把内容讲透——简单组件可能 3 章就够，复杂组件 10 章甚至 20 章也没有问题。

常见的章节类型（根据组件实际情况选用和组合）：

- **总览章**（复杂组件需要）— 全景图：核心能力、整体架构、各部分协作
- **问题域与设计目标** — 从真实场景切入，分析痛点，对比替代方案
- **前置知识**（需要时）— 第三方技术讲解，放在相关设计章节之前
- **核心设计逐一深入** — 每章聚焦一个设计主题
- **配置设计与使用场景**（需要时）— 配置项的设计权衡
- **边界与扩展** — 不做什么、与其他组件的分工、已知限制
- **精简汇总章**（必选）— 两部分精简版，方便回顾复习：
  - 能力与使用速查（表格/清单形式）
  - 设计与概念精要（要点列表，不展开论证）
  - 篇幅控制在总章数的 1/3 以内

#### 风格参考

核心原则：**把内容讲透，让读者真正理解设计背后的"为什么"**。

- **"先破后立"开场**（按需）— 先列出读者可能有的不够完整的常见理解，然后揭示更深层的本质
- **场景驱动** — 每章从一个具体的、读者能共情的真实场景切入
- **设计取舍对比** — 不只讲"选了什么"，更要讲"放弃了什么、为什么"。用对比表呈现多方案差异
- **用类比建立直觉**（按需）— 用生活或工作中的类比帮助理解抽象概念
- **ASCII 流程图贯穿全文** — 用文字图表达架构、流程、状态转换
- **思考题**（按需）— 每章末尾 2-4 道，引导反思核心设计决策
- **不限制篇幅** — 讲清楚为止，不追求"精炼"而追求"讲透"
- **决策框架**（按需）— 用决策树或选型速查表帮读者做判断
- **章节内部推进节奏** — 场景切入 → 问题定义 → 方案对比 → 核心机制 → 配置/使用 → 扩展思考

### 功能实现（实操为主）

定位：一份完整的实现指南。从零开始，一步步实现整个组件。读者看完后应具备独立实现一个类似组件的能力。

#### 核心原则

1. **从零开始** — 第一章从空白项目起步
2. **完整过程** — 覆盖从第一行代码到最终可运行组件的全部步骤
3. **讲清为什么** — 对复杂的实现选择讲清原因
4. **代码即主体** — 每章核心内容是完整代码实现
5. **渐进式构建** — 每次只展示一个步骤的代码变化，不要一上来贴完整代码
6. **讲解节奏自然** — 有设计意图的地方讲清楚，顺理成章的地方一笔带过

#### 章节组织

按**实现顺序**组织，读者可以跟着从头到尾做一遍：

1. **起步：骨架搭建** — 项目结构、核心接口、Spring Boot 自动配置空壳
2. **核心机制逐步实现** — 从最底层开始，逐步向上构建
3. **配置与集成** — 配置属性类、条件装配
4. **完善与加固** — 错误处理、边界情况
5. **业务层对接** — 业务代码如何使用这个组件
6. **测试支持与基础设施** — 测试如何与组件交互

#### 代码呈现规范

- **完整可运行** — 不是伪代码或片段
- **从源码复制** — 不要凭记忆重写
- **标注来源** — 代码块前注明源文件路径
- **渐进式构建** — 同一类在多章中逐步完善时，后续只展示新增或变更部分

#### 与功能设计的衔接

- 首章开头简要回顾功能设计的核心结论（1-2 段）
- 实现选择处用简短的"为什么这么做"链接到功能设计对应章节
- 章末导航链接到功能设计中相关的设计章节

**按需生成**——根据组件内容自行判断是否值得生成，而非等用户要求。如果组件有足够的设计深度（设计模式提炼、多方案权衡、可迁移经验等），应主动生成；确实简单的组件可跳过。以下为推荐方向，根据组件实际情况选用：

### 生成与审查流程

1. **阅读源码** — 读取该组件涉及的所有核心类，理解实际运行时行为
2. **规划章节结构** — 功能设计：从底向上补课；功能实现：从骨架到完整
3. **先写功能设计，再写功能实现** — 实现中的"为什么"依赖设计中已建立的结论
4. **逐章编写** — 设计从场景切入，实现从上一章成果继续
5. **子代理审查** — 技术准确性、编号一致性、代码与源码一致性
6. **根据审查结果修复** — 重点修复编号不一致、交叉引用错误、源码注释与实际行为不符
7. **更新组件一览表** — 在 `项目模块全景一览表.md` 中更新生成记录

## 架构速查

以下为项目关键架构信息，供生成文档时快速参考。

### 技术栈

| 关注点 | 技术 |
|--------|------|
| 框架 | Spring Boot 2.7.18, Spring Cloud 2021.0.9, Spring Cloud Alibaba 2021.0.6.2 |
| Java 版本 | JDK 8 |
| ORM | MyBatis-Plus 3.5.16（无 JPA，无 XML mapper） |
| 数据库 | MySQL（默认），多数据库支持 |
| 注册/配置中心 | Nacos |
| API 网关 | Spring Cloud Gateway（响应式） |
| RPC | OpenFeign |
| 认证 | 自定义 OAuth2 + Spring Security |
| 授权 | `@PreAuthorize` + 自定义 `@ss` SpEL Bean |
| 多租户 | 共享 Schema，TenantLineHandler 自动追加 WHERE |
| 缓存 | Redis + Spring Cache + Guava LoadingCache |
| 消息队列 | 可插拔（Redis, RocketMQ, RabbitMQ, Kafka） |
| API 文档 | SpringDoc OpenAPI 3 + Knife4j |
| 对象映射 | MapStruct + 自定义 BeanUtils |
| 测试 | JUnit 5 + Mockito + H2 内存数据库 |

### 三层模块设计

1. **`yudao-dependencies/`** — BOM，集中管理所有依赖版本
2. **`yudao-framework/`** — 17 个自定义 Spring Boot Starter
3. **`yudao-module-*/`** — 业务模块，每个拆分为 `api`（Feign 客户端 + DTO）+ `server`（实现）

### 部署模式

- **单体**：`YudaoServerApplication`，默认只启用 system 和 infra 模块
- **微服务**：每个 `*-server` 有独立启动类，通过 Feign + Nacos 通信
- **网关**：`GatewayServerApplication`，处理路由和令牌校验

### 关键设计模式

- **URL 前缀注入**：`YudaoWebAutoConfiguration` 根据包路径自动拼接 `/admin-api` 或 `/app-api`
- **跨模块 RPC**：业务模块用 `-api` 子模块的 Feign 客户端；框架级用 `yudao-common` 中的 `*CommonApi` 接口。统一 `/rpc-api/{module}/...` 前缀
- **MyBatis-Plus 扩展**：`BaseMapperX` 封装分页/批量操作；`LambdaQueryWrapperX` 的 `*IfPresent` 方法消除空值判断
- **异常处理**：`ErrorCodeConstants` 定义错误码 → `ServiceExceptionUtil.exception()` 抛出 → `GlobalExceptionHandler` 统一转换为 `CommonResult`
- **DI 风格**：使用 `@Resource`（JSR-250），不使用 `@Autowired`
- **VO 命名**：`*SaveReqVO`（增改请求）、`*PageReqVO`（分页查询）、`*RespVO`（完整响应）、`*SimpleRespVO`（轻量响应）
