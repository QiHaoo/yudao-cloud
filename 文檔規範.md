# docs/ 文档编写指南

## 注意
生成的文档的示例统一用电商系统来举例，禁止使用国际税务等相关的业务来做举例。

## 目录结构

文档按组件编号组织，编号对应 `docs/项目框架基础设施组件全景一览表.md` 中的序号：

```
docs/
├── CLAUDE.md                              ← 本文档（文档编写规范）
├── 功能讲解参考/                           ← 课程式讲解的参考范例
├── 项目框架基础设施组件全景一览表.md       ← 主索引（跟踪组件的文档状态）
├── 框架基础设施组件全景.md                 ← 全景概览（能力域梳理）
├── {序号} {组件名}/                       ← 每个组件一个目录
│   ├── 功能设计/                          ← 第一部分：讲解为主
│   │   ├── 01-前置知识：xxx.md            ← 前置知识（放在相关设计章节之前）
│   │   ├── 02-问题域与设计目标.md          ← 为什么需要这个组件
│   │   ├── 03-前置知识：xxx.md            ← 更多前置知识
│   │   ├── 04-xxx设计.md                  ← 各设计主题深入讲解
│   │   └── ...
│   ├── 功能实现/                          ← 第二部分：实操为主
│   │   ├── 01-起步：骨架搭建.md            ← 从零开始的第一步
│   │   ├── 02-xxx实现.md                  ← 逐步实现
│   │   ├── 03-xxx实现.md
│   │   └── ...
│   ├── TDD功能实现/                       ← 测试驱动实现（按需生成，用户明确要求时才生成）
│   │   ├── 01-xxx.md                      ← 每章包含若干功能点，每个功能点走一遍 TDD 循环
│   │   └── ...
│   ├── 深度学习/                          ← 提炼为主（按需生成）
│   │   ├── 01-设计模式提炼.md
│   │   ├── 02-决策框架与思维模型.md
│   │   ├── 03-动手练习与扩展挑战.md
│   │   ├── 04-横向对比与反模式.md
│   │   ├── 05-反思与改进.md
│   │   ├── 06-业界方案全景.md
│   │   ├── 07-扩展阅读.md
│   │   ├── 08-DDD视角.md
│   │   └── 09-边界与集成.md
│   └── 源码索引.md                        ← 组件涉及的所有源码文件索引，供 AI 快速定位
└── ...
```

## 文档分层

每个组件的文档分为四部分，形成 **为什么这样设计 → 怎么一步步实现 → 怎么用 TDD 驱动实现 → 怎么迁移改进** 的阅读顺序：

| 维度 | 功能设计 | 功能实现 | TDD 功能实现 | 深度学习 |
|------|---------|---------|-------------|---------|
| 核心问题 | 为什么这样设计 | 怎么一步步实现 | 怎么用测试驱动的方式实现 | 怎么迁移、怎么改进、怎么选型 |
| 驱动力 | 设计思想和取舍 | 实操过程和细节 | Red → Green → Refactor | 提炼、反思、扩展认知 |
| 代码的角色 | 辅助说明设计意图（片段） | 主体内容（完整代码） | 测试代码先行，实现代码跟随 | 引用源码做分析 |
| 读者收获 | 理解设计背后的"为什么" | 能动手实现一个类似组件 | 掌握用 TDD 构建框架组件的能力 | 能带着经验去解决其他项目中的类似问题 |
| 生成时机 | 优先生成 | 设计完成后生成 | 用户明确要求时才生成 | 按需生成 |

功能设计不逐行解读源码，功能实现不重复讲设计思想。两者通过交叉引用衔接。TDD 功能实现以功能实现的章节结构为参考，不重复讲设计思想。深度学习在前三者的基础上做提炼和反思。

## 功能设计（讲解为主）

**定位**：一门完整的组件设计课程。把读者当成初学者，从第三方技术的前置知识补课开始，到场景驱动的设计讲解，再到边界与扩展，完完整整地讲透一个组件的设计思想。

### 前置知识融入课程

前置知识作为功能设计的章节，**分散放置在相关设计章节之前**，形成"先补课、再深入"的阅读节奏：

```
功能设计/
├── 01-前置知识：Redis Stream 核心概念.md    ← 先讲 Redis Stream 是什么
├── 02-问题域与设计目标.md                   ← 再讲为什么用 Redis Stream 做消息队列
├── 03-前置知识：Spring Data Redis 抽象.md   ← 先讲 Spring 怎么封装 Redis Stream
├── 04-消息流转设计.md                       ← 再讲框架怎么基于 Spring 抽象做消息流转
├── ...
```

**命名规范**：前置知识章节以 `{编号}-前置知识：{主题}.md` 命名，与设计章节统一编号。

**拆分原则**：
- 若组件只涉及一个第三方技术，一篇前置知识即可，放在设计章节之前
- 若涉及多个第三方技术，拆分为多篇前置知识，各自放在它支撑的设计章节之前
- 前置知识的篇幅由其复杂度决定——简单的一两千字即可，复杂的可能需要几千字

**内容要求**：
- **够用即可** — 只讲解组件涉及到的部分，不需要全面覆盖该技术的所有内容
- **从底向上** — 先讲第三方技术本身，再讲 Spring 的封装抽象，让读者理解每一层在解决什么问题
- **贴近上下文** — 将技术讲解与本组件代码的实际用法结合起来，避免脱离代码的纯理论介绍
- **用类比和示意图** — 让零基础的读者也能看懂

### 章节组织

功能设计的章节按学习路径组织，没有固定的章节数限制。以下为常见的章节类型，根据组件实际情况选用和组合：

- **总览章**（复杂组件需要）— 在深入各设计主题之前，先给一张全景图：组件有哪些核心能力、整体架构长什么样、各部分怎么协作。让读者先建立全局认知，再逐章深入细节
- **问题域与设计目标** — 从真实场景切入，分析痛点，对比替代方案，明确设计目标
- **前置知识**（需要时）— 讲解组件涉及的第三方技术，放在相关设计章节之前
- **核心设计逐一深入** — 每个章节聚焦一个设计主题，深入讲解设计考量
- **配置设计与使用场景**（需要时）— 各配置项的设计权衡，典型部署场景的推荐配置
- **边界与扩展** — 组件不做什么、与其他组件的分工、已知限制、扩展点
- **精简汇总章**（必选）— 最后一章，汇集前文所有内容的两份精简版，方便回顾复习：
  - **第一部分：能力与使用速查** — 组件提供了哪些能力、各能力对应的核心 API/入口是什么、典型使用方式。采用表格或清单形式，让读者无需翻阅全文即可快速定位要用的功能
  - **第二部分：设计与概念精要** — 将前文所有设计决策、核心机制、关键概念压缩为精简的要点列表。每条用一句话讲清一个设计要点，不展开论证（论证已在正文完成）。按主题分组，方便读者快速回顾整个组件的设计全貌
  - **篇幅控制** — 两部分各自控制在组件总章数的 1/3 以内（10 章的组件，汇总章最多 2-3 页当量的内容）。这不是缩写全文，而是提炼骨架——只保留关键词、核心结论和关键入口，去掉所有铺垫、论证和示例

**章节数量不限制。** 核心宗旨是完完全全把内容讲透——简单组件可能 3 章就够，复杂组件 10 章甚至 20 章也没有问题。

### 风格参考

`docs/功能讲解参考/` 目录下提供了课程式讲解的参考范例。以下要点仅为风格参考，不是强制模板。核心原则只有一个：**把内容讲透，让读者真正理解设计背后的"为什么"**。

- **"先破后立"开场**（按需使用）— 先列出读者可能有的不够完整的常见理解，然后揭示更深层的本质
- **场景驱动** — 每章从一个具体的、读者能共情的真实场景切入，而不是一上来就列概念
- **"药"式一句话总结**（按需使用）— 在展开讲解之前，先用一句话点明核心价值，让读者带着清晰的问题和期待往下读
- **核心问题前置** — 先讲清楚"为什么需要这个"，再讲"这是什么"，最后讲"怎么用"
- **用类比建立直觉**（按需使用）— 用生活或工作中的类比帮助理解抽象概念，找不到合适的类比不必强求
- **层次化讲解**（适合复杂系统）— 复杂系统按层级逐步展开，每一层配合具体示例
- **量化分析**（按需使用）— 有数据支撑时用数字说话，没有数据时不编造
- **经验法则**（按需使用）— 给出实用的判断口诀或阈值
- **设计取舍对比** — 不只讲"我们选了什么"，更要讲"我们放弃了什么、为什么"。用对比表呈现多方案差异，让读者看到全局而非单点
- **"从使用者到驾驭者"的学习旅程** — 帮读者建立"设计者思维"
- **ASCII 流程图贯穿全文** — 用文字图表达架构、流程、状态转换
- **思考题**（按需使用）— 每章末尾放 2-4 道思考题，引导反思核心设计决策。前置知识类章节可省略
- **不限制篇幅** — 讲清楚为止，不追求"精炼"而追求"讲透"
- **章节内部的推进节奏** — 一章内部的内容按"场景切入 → 问题定义 → 方案对比 → 核心机制 → 配置/使用 → 扩展思考"推进。不是每章都要走完全部环节，但大方向是从具体到抽象、从问题到方案。这样读者始终知道"现在在解决什么问题"
- **决策框架**（按需使用）— 当组件涉及多种模式/策略/配置选择时，用决策树或选型速查表帮读者做判断（如"什么时候用 X vs Y"）。不只解释过去的取舍，更要给读者一个自己决策的工具
- **认知锚点**（按需使用）— 在章节中间的关键转折处，用一句话提炼刚讲完的核心（如"子代理的核心工程价值归结起来是四件事：隔离、约束、复用"），帮读者在信息密集的段落中锁定要点

## 功能实现（实操为主）

**定位**：一份完整的实现指南。从零开始，一步步实现整个组件，包含所有细节。读者看完后应具备独立实现一个类似组件的能力。

### 核心原则

1. **从零开始** — 第一章从空白项目起步，不做任何"显然你已经知道"的假设
2. **完整过程** — 覆盖从第一行代码到最终可运行组件的全部步骤，不留缺口
3. **讲清为什么** — 对于复杂的实现选择，讲清楚为什么这么做而不是另一种做法
4. **代码即主体** — 每一章的核心内容是完整的代码实现，辅以必要的讲解
5. **渐进式构建，忌一上来贴完整代码** — 每一章内部应按实现步骤推进。每次只展示一个步骤的代码变化，下一步基于当前成果继续。读者应该能跟着步骤动笔，而不是面对一个 300 行的完整类无从下手。主体内容必须是分步的，且实现过程的代码都是完整的。
6. **讲解节奏自然** — 有设计意图、有取舍的地方尽量讲清楚，顺理成章的地方一笔带过。不需要强制每步都写"为什么要这样做"——一个依赖注入字段不需要解释，一个设计模式的选择需要。判断标准：读者看到这段代码时，会不会产生"为什么这样写"的疑问？会就讲，不会就过

### 章节组织

功能实现按**实现顺序**组织章节——读者可以跟着从头到尾做一遍。以下是典型的章节推进方式，根据组件实际情况调整：

1. **起步：骨架搭建** — 项目结构、核心接口定义、Spring Boot 自动配置类的空壳。让读者先有一个能编译运行的骨架
2. **核心机制逐步实现** — 按组件的实际构建顺序，每章实现一个核心能力。通常从最底层的机制开始，逐步向上构建
3. **配置与集成** — 配置属性类、条件装配、与 Spring Boot 的集成点
4. **完善与加固** — 错误处理、边界情况、守护机制等补充能力
5. **业务层对接** — 业务代码如何使用这个组件，扩展点如何对接
6. **测试支持与基础设施** — 测试如何与组件交互、如何跳过或mock对应的内容、如何设计并实现配套的测试工具

**章节数量不限制**。简单组件可能 3-4 章走完全部流程，复杂组件可能需要 10 章以上。

### 实现讲解的深浅节奏

不是所有代码都需要逐行讲解，但對於需要講解的地方，不限篇幅，力求講透。按以下节奏把握深浅：

- **顺理成章的代码** — 贴出完整代码，用 1-2 句话说明它在做什么即可（如简单的 getter/setter、直接的委托调用）
- **有设计意图的代码** — 贴出代码后，讲清楚为什么这样写而不是另一种写法（如选择继承而非组合、选择同步而非异步）
- **复杂或反直觉的代码** — 完整讲解，包括设计背景、实现思路、逐段解析、可能踩的坑

判断标准：**如果删掉这段讲解，读者自己看代码时会不会困惑？** 会困惑就讲，不会就简洁带过。

### 代码呈现规范

- **完整可运行** — 贴出的代码必须是完整的（不是伪代码或片段），读者应能直接复制使用
- **从源码复制** — 代码片段从实际源码复制，不要凭记忆重写
- **标注来源** — 在代码块前注明源文件路径，如 `// lumina-bf-essential/.../AsyncJobManager.java`
- **渐进式构建** — 如果同一个类在多章中逐步完善，后续章节只展示新增或变更的部分，并说明与之前版本的差异

### 与功能设计的衔接

功能实现不重复讲设计思想，但需要：
- **首章开头**：简要回顾功能设计的核心结论（1-2 段），让只看实现的读者也能理解大方向
- **实现选择处**：用简短的"为什么这么做"链接到功能设计的对应章节，如"关于为什么选择先持久化后入队，详见功能设计/03-核心设计决策"
- **章末导航**：每章末尾链接到功能设计中相关的设计章节，供想深入理解的读者参考

## TDD 功能实现（测试驱动）

**定位**：用 TDD（测试驱动开发）的方式重新走一遍组件的实现过程。与「功能实现」的区别：功能实现展示最终代码，TDD 功能实现展示 **"先写测试 → 看它失败 → 写最少代码让它通过 → 重构"** 的完整开发过程。

**生成时机：用户明确要求时才生成。** 生成组件文档时默认不包含 TDD 部分，只有用户明确说"包含 TDD"时才生成。

### TDD 循环

每个功能点的实现遵循 TDD 三步循环：

```
Red（写失败测试）
  → 确认测试确实失败（不是编译错误，是断言失败）
    → Green（写最少的代码让测试通过）
      → 确认测试通过
        → Refactor（重构，测试保持通过）
          → 小结：这个功能点的 TDD 过程教会了什么
```

**一章可能包含多个功能点**，每个功能点各自走一遍 TDD 循环。章节内部的结构是：

```
第 N 章：xxx 实现
  │
  ├── 功能点 A
  │   ├── 写测试（Red）
  │   ├── 最小实现（Green）
  │   ├── 重构（Refactor）
  │   └── 小结
  │
  ├── 功能点 B
  │   ├── 写测试（Red）
  │   ├── 最小实现（Green）
  │   ├── 重构（Refactor）
  │   └── 小结
  │
  └── 本章回顾
```

### 章节组织

TDD 功能实现的章节顺序不必与「功能实现」一致。按**可测试的功能点**自行组织——TDD 的推进顺序通常是由底向上（先测核心逻辑，再测集成），和功能实现的顺序可能不同。

### 内容要求

- **测试代码是主体** — 每步先展示完整的测试代码，再展示实现代码
- **最小实现** — Green 阶段只写刚好让测试通过的代码，不过度实现
- **重构的价值** — Refactor 阶段讲清楚"从哪里到哪里"，为什么这样重构
- **Red 阶段点到为止** — 读者跟着文档自己敲代码时自然能看到 Red，不需要在文档中展示运行失败的输出。只在有必要提醒或有注意事项时提一下（如"这个测试的失败信息可能会让你困惑，它实际含义是..."）
- **与功能实现的交叉引用** — TDD 版本不重复讲设计思想，引用功能实现中的设计解释

## 测试支持与基础设施

当前各组件几乎没有内建的测试支持，业务代码在测试时往往需要手动 Mock 或绕过。每个组件的功能实现部分应包含一个"测试支持"章节作为最后一章，承担两项任务：

1. **盘点现状** — 当前业务代码如何与组件交互来写测试（即使只是手动 Mock 或绕过，也要如实记录）
2. **设计与实现缺失的测试基础设施** — 识别痛点，设计并实现该组件配套的测试工具，在功能实现中落地

核心问题：**使用这个组件的业务代码，怎么写测试最方便？** 以下为需要覆盖的方向。每个方向都应当回答"现在有什么"和"还缺什么、怎么补"。各组件根据自身特点选择相关内容，不必全部覆盖。

### 需要覆盖的方向

**1. 测试集成方式**
- 单元测试中如何模拟/替代该组件（Mock？Stub？Fake？）
- 集成测试中组件如何参与（自动启用？需显式配置？）
- 是否需要切片测试注解（如 `@XxxTest`）只加载该组件相关的 Bean

**2. 组件提供的测试工具/注解**
- 是否有类似 `@WithMockUser`、`@SkipAuth` 的注解，用于跳过或简化非测试目标的逻辑
- 是否有测试专用的自动配置类
- 是否有 Mock Factory 或 Test Fixture 构建器
- **缺失时**：设计并实现这些工具，给出使用示例

**3. 常见测试场景的简化方案**
- "只想测试接口逻辑，不想关心权限校验" → 是否提供跳过权限的选项
- "只想测试业务逻辑，不想启动消息队列/定时任务" → 如何在测试中禁用
- "只想测单个 Service，不需要完整 Spring 上下文" → 是否有轻量级测试支持
- **缺失时**：设计简化方案，在功能实现中落地

**4. 测试配置参考**
- 测试 profile 下需要的 YAML 配置项
- 如何覆盖生产 Bean 为测试版本

**5. 测试数据准备**
- 是否有 Builder 模式或工厂方法简化测试数据构造
- 是否有预置的测试数据集合

**6. 可测试性设计**
- 组件在设计时预留了哪些扩展点/接口方便测试
- 如果组件本身设计不便测试，是否需要在组件代码中增加可测试性改进

### 不同组件类型的侧重点

- **基础设施组件**（如权限、消息队列、异步任务）→ 重点讲 Mock/替代方案和跳过机制
- **数据访问组件** → 重点讲测试数据准备和事务隔离
- **工具类组件** → 可能只需要简单的使用示例
- **流程引擎类组件** → 重点讲如何模拟流程节点

## 深度学习（提炼为主）

**定位**：组件文档的第三层。功能设计讲"为什么这样设计"，功能实现讲"怎么一步步实现"，深度学习讲**"怎么迁移、怎么改进、怎么选型"**。目标是让读者学完一个组件后，工程能力真正有所提升——不只是"看懂了代码"，而是能带着这些经验去解决其他项目中的类似问题。

### 目录结构

每个组件的深度学习文档放在 `深度学习/` 文件夹下：

```
{序号} {组件名}/
├── 功能设计/
├── 功能实现/
├── 深度学习/
│   ├── 00-组件设计方法论.md
│   ├── 01-设计模式提炼.md
│   ├── 02-决策框架与思维模型.md
│   ├── 03-动手练习与扩展挑战.md
│   ├── 04-横向对比与反模式.md
│   ├── 05-反思与改进.md
│   ├── 06-业界方案全景.md
│   └── 07-扩展阅读.md
└── 源码索引.md
```

**按需生成**：以上 10 个方向是推荐，不是强制。根据组件的实际情况选用——简单组件可能只有 2-3 个文件，复杂组件可能 10 个都有。原则是：有内容可写才写，不为凑数。

### 九个方向的内容定义

#### 00-组件设计方法论

聚焦"拿到一个需求后，怎么从零推导出一个好的组件设计"。这是设计过程中的思考方法，不是模式识别，也不是选型决策。

以下为常见的设计启发式，**根据组件实际情况选用**——不是每个组件都能提炼出所有条目，有内容可写才写：

- **先写理想 API 再倒推实现** — 站在调用方角度，先写出最自然、最好用的调用方式，再想怎么实现。如果 API 用起来别扭，说明内部设计有问题
- **用动词识别职责** — 一个组件该做什么？列出它该响应的动词，动词超过三个就该考虑拆分
- **通过用例划定边界** — 画出组件的所有调用场景，找出职责的自然聚类——哪些操作总是被一起调用，哪些从不一起出现
- **暴露能力而非实现** — 接口契约应该描述"能做什么"，而不是"怎么做的"。调用方不需要知道内部是用 Redis 还是数据库
- **设计不容易用错的接口** — 让错误的用法编译不过、或者明显反直觉。参数顺序、命名、默认值都是设计的一部分
- **识别变化点，预留而非实现** — 识别"这里将来大概率会变"的位置，用接口/策略模式留口子，但不过度实现未来需求
- **从最复杂的用例开始设计** — 先把最难的场景跑通，简单的 case 往往自然浮现。反过来做容易导致简单场景设计得很优雅、复杂场景被迫打补丁

#### 01-设计模式提炼

从组件中识别使用的设计模式（Builder、Decorator、策略模式、模板方法等），抽象成通用知识。

每个模式包含：
- 一句话定义
- 在本组件中的具体体现（引用源码）
- 通用化的应用场景——什么问题适合用这个模式
- 识别口诀——看到什么信号就想到这个模式

**目标**：让读者在其他项目中遇到类似问题时能识别并套用。

#### 02-决策框架与思维模型

聚焦"为什么选 A 而不选 B"的决策过程，训练架构决策能力。

每个决策包含：
- 背景约束——当时面临什么限制
- 候选方案——考虑过哪些选项
- 决策依据——为什么选了这个
- 适用边界——什么情况下该换一个方案

提炼可复用的决策框架，如"什么时候用输入清洗 vs 输出编码"、"什么时候用静态初始化 vs 懒加载"。

#### 03-动手练习与扩展挑战

渐进式练习题，从阅读到实践到独立构建：

- **L1 阅读理解** — 阅读源码回答问题，验证是否真正理解了设计
- **L2 扩展实现** — 给组件添加新功能或修改现有行为
- **L3 从零构建** — 不看源码，独立实现一个类似组件

每道题有明确的目标、验收标准、提示（不直接给答案）。

#### 04-横向对比与反模式

跨组件对比相同的设计手法，建立全局视角。例如：验证码和 XSS 防护都用了 `HttpServletRequestWrapper` 装饰器模式，可以对比它们的异同。

同时总结常见反模式和踩坑——什么是不该做的，以及为什么。

#### 05-反思与改进

对组件设计实现的批判性反思。**不限篇幅。**

内容包括：
- 设计上的不足之处——当初的决策在什么场景下会暴露问题
- 潜在的性能瓶颈——是否有 N+1 查询、不必要的序列化、同步阻塞等
- 可改进的具体方向——给出改进方案，而不只是指出问题
- 已知的技术债——框架作者留下的 TODO 或 workaround

**不回避问题**——即使是框架的设计缺陷也如实分析。培养批判性思维。

#### 06-业界方案全景

同类问题在业界有哪些解决方案——不限于当前框架选择的方案，尽可能讲透。**不限篇幅。如果有多種方案且篇幅較大，也可以分成多篇文檔**

每个方案包含：
- 原理——它是怎么工作的
- 优缺点——和当前方案相比的差异
- 适用场景——什么情况下该选这个方案
- 代表产品/库——哪些知名项目在用

**必须标注来源和依据**（论文、官方文档、知名博客、开源项目等）。这是扩展认知的核心文档——让读者面对类似问题时有更多选择。

#### 07-扩展阅读

汇总本组件相关的扩展阅读资源，供读者顺藤摸瓜。

分类组织：
- 官方文档 — 框架/库的官方文档链接
- 经典论文 — 相关的学术论文或技术规范
- 高质量博客 — 深度分析文章
- 开源项目 — 值得参考的实现
- 书籍章节 — 相关技术书籍的推荐章节

每条资源附一句话说明其价值——读者看到这句话就能判断是否值得花时间去读。

**必须标注来源。**

#### 08-DDD视角

用 DDD（领域驱动设计）的完整设计过程重新审视组件，展示"如果从零用 DDD 来设计这个组件，思考路径是什么样的"。

不是泛泛介绍 DDD 概念，而是结合本组件的实际设计，走一遍 DDD 的核心步骤：

- **领域发现** — 这个组件解决的核心领域问题是什么？关键概念有哪些？
- **限界上下文** — 组件的边界在哪？和哪些上下文有关系（上游/下游/共享内核）？
- **统一语言** — 组件中的核心术语如何精确定义？和框架/业务层的叫法是否一致？
- **聚合与聚合根** — 如果用 DDD 的方式建模，聚合边界怎么画？谁是聚合根？
- **值对象** — 哪些概念应该是值对象而非实体？不变性如何保证？
- **领域服务** — 哪些逻辑不属于任何实体/值对象，应该抽为领域服务？
- **领域事件** — 组件在什么时机产生事件？谁应该监听？
- **应用服务 vs 领域服务** — 哪些是编排逻辑（应用服务），哪些是核心业务规则（领域服务）？

最后对比 DDD 设计和框架实际设计的差异，分析框架选择（或不选择）DDD 的原因。

**根据组件实际情况选用**——不是每个组件都能走完所有步骤，有内容可写才写。

#### 09-边界与集成

聚焦"组件如何与外部世界打交道"——两个层面：

**解耦机制**：组件如何与外部依赖（其他组件、业务模块、基础设施）保持松耦合？通过哪些接口/抽象层实现隔离？数据流向是怎样的？

**接入指南**：业务项目如何接入这个组件？需要实现哪些接口？替换哪些 Bean？数据怎么流进流出？

内容结构（根据组件实际情况选用）：

- **依赖关系图** — 组件与外部的依赖关系、接口层次、数据流向
- **接口契约** — 组件对外暴露的接口定义、数据格式、调用约定
- **接入步骤** — 从业务项目视角，一步步接入的指南
- **数据一致性**（涉及缓存/异步时）— 写入后何时可见、如何主动失效
- **降级策略** — 外部依赖不可用时组件的行为
- **常见接入陷阱** — 接入时容易犯的错误及其规避方法

**根据组件实际情况选用**——不是每个组件都有外部依赖需要解耦，有内容可写才写。

## 生成与审查流程

编写文档时，遵循以下流程可以显著减少事实错误和编号混乱。

### 第一步：阅读源码

在动笔之前，必须完成：
1. **读源码** — 读取 `FramworkSource/` 下该组件涉及的所有核心类（接口、抽象类、配置类），理解实际运行时行为
2. **读业务层使用代码** — 读取 `dsf-tax-taxexchange-ext/` 中该组件的使用方式，理解实际业务场景，但不能将业务层的内容写到文档中去，如有需要，可以换成电商系统的例子。

### 第二步：规划章节结构

**功能设计**：根据组件涉及的技术栈，确定前置知识章节的插入位置。原则：从底向上——先底层技术本身，再 Spring 封装，再框架设计。

**功能实现**：根据组件的实际构建顺序，规划从骨架到完整的章节推进。原则：从底向上——先基础机制，再核心流程，再补充能力，最后业务对接。

### 第三步：先写功能设计，再写功能实现

功能设计完成后才开始写功能实现。原因：
- 功能实现中"为什么这么做"的讲解依赖于功能设计中已经建立的设计结论
- 功能设计的章节结构为功能实现的章节顺序提供了天然的参考

### 第四步：逐章编写

- 功能设计：每章从一个真实场景切入
- 功能实现：每章从上一章的成果继续，展示渐进式构建
- 关键设计/实现引用的源码行为必须与 `FramworkSource/` 中的实际代码一致
- 代码片段尽可能从源码复制，不要凭记忆重写
- 每章末尾用交叉引用导航（功能设计 ↔ 功能实现）

### 第五步：子代理审查

全部章节写完后，用子代理审查：

**代理一**：审查技术准确性及敏感信息
- 逐文件检查所有技术描述（API 语义、类关系、运行时行为）是否与源码一致
- 生成的文档的示例统一用电商系统来举例，禁止使用国际税务等相关的业务来做举例。
- 特别关注：代码注释中的说明与实际代码行为是否一致（注释可能过时）

审查代理的提示词示例：

```
你是技术文档审查员。请审查 {目录} 下的所有 .md 文件。
审查目标：
1. 技术准确性：对照源码 {源码目录} 验证所有技术描述
2. 逻辑一致性：标题编号、交叉引用、文内引用是否正确
3. 表述问题：语句是否通顺、概念是否清晰、前后是否矛盾
4. 代码示例：语法是否正确、是否与实际源码一致
请逐文件审查，每个文件报告发现的问题。
```

### 第六步：根据审查结果修复

审查代理通常会发现问题集中在：
- **标题编号与文件名不一致**
- **交叉引用指向旧编号**
- **源码注释与实际行为不符被文档照搬**
- **技术事实错误**

修复后重新跑一遍标题编号检查确认无遗漏。

### 第七步：更新组件一览表的生成记录

每次生成完一个组件的文档后，**必须**在 `项目框架基础设施组件全景一览表.md` 的"已完成组件详情"表格中更新该组件的记录，注明文档由哪个大模型生成。格式示例：

| # | 组件 | 文档路径 | 章节数 | 生成模型 |
|---|------|---------|--------|---------|
| 3 | **验证码** | [`3 验证码/功能设计/01-xxx.md`](...) | 功能设计 N 章 + 功能实现 N 章 | mimo-v2.5-pro |
