# 01-设计模式提炼

## 模式一：过滤器链模式（Filter Chain Pattern）

### 在本项目中的体现

`TokenAuthenticationFilter` 是 Spring Security 过滤器链中的一环。每个过滤器负责一个特定的关注点：

```
SecurityContextPersistenceFilter → TokenAuthenticationFilter → UsernamePasswordAuthenticationFilter → ...
```

### 通用化

过滤器链模式将请求处理分解为多个独立的步骤，每个步骤封装在一个过滤器中。新功能通过添加过滤器实现，不修改已有代码。

**适用场景**：HTTP 请求预处理、日志记录、权限校验、数据压缩等。

## 模式二：策略模式（Strategy Pattern）

### 在本项目中的体现

`TokenAuthenticationFilter` 内部使用了策略模式的变体——按优先级尝试三种认证策略：

1. Header 透传解析
2. Token 校验
3. Mock 模拟

每种策略独立实现，失败后降级到下一种。

### 通用化

当一个操作有多种实现方式，且需要根据上下文选择时，使用策略模式。本项目的特殊之处是策略有优先级（降级机制）。

## 模式三：模板方法模式（Template Method Pattern）

### 在本项目中的体现

`AuthorizeRequestsCustomizer` 是一个抽象类，定义了 URL 安全规则的扩展点。每个模块继承它并实现 `customize()` 方法。

```java
// 模板：父类定义结构
public abstract class AuthorizeRequestsCustomizer {
    protected String buildAdminApi(String url) { ... }  // 工具方法
    public abstract void customize(...);                 // 子类实现
}

// 实现：子类填充细节
@Component
public class SecurityConfiguration extends AuthorizeRequestsCustomizer {
    @Override
    public void customize(...) {
        registry.requestMatchers(buildAdminApi("/v3/api-docs/**")).permitAll();
    }
}
```

## 模式四：代理模式（Proxy Pattern）

### 在本项目中的体现

`SecurityFrameworkServiceImpl`（@ss Bean）是权限校验的代理。业务代码调用 `@ss.hasPermission()`，实际的权限查询通过 Feign RPC 代理到 system 服务。

```
Controller → @ss.hasPermission() → Guava Cache → Feign RPC → system 服务
```

缓存层也是一个代理——它拦截请求，命中则直接返回，未命中才调用远程服务。

## 模式五：ThreadLocal 上下文模式

### 在本项目中的体现

`SecurityContextHolder` 使用 `ThreadLocal`（TTL 版本）存储当前请求的认证信息。这是一种典型的"上下文传播"模式：

- **写入**：`TokenAuthenticationFilter` 在请求开始时设置
- **读取**：业务代码在任何时候通过 `SecurityFrameworkUtils.getLoginUser()` 读取
- **清理**：请求结束后自动清除

### 注意事项

ThreadLocal 上下文模式的陷阱：
- 线程池复用导致上下文泄漏
- 异步场景上下文丢失（用 TTL 解决）
- 忘记清理导致内存泄漏
