# 认证鉴权框架：问题域与设计目标

---

## 1. 从一个"裸奔"的电商后台说起

你刚搭好电商后台的骨架，商品管理、订单管理、用户管理等模块的 CRUD 都能跑了。你把服务部署到测试环境，发给同事试用。

第二天，你发现了一堆问题：

- **问题一**：同事直接用浏览器访问 `http://your-server/admin-api/product/list`，不用登录就能看到所有商品数据。
- **问题二**：有人"手滑"调了 `DELETE /admin-api/user/delete?id=1`，把超级管理员删了。
- **问题三**：前端同事把 Token 写死在代码里，任何人都能用这个 Token 访问所有接口。

你意识到：**你的系统完全没有"门"**。任何人只要知道 URL，就能随意进出、随意操作。

这就是没有认证鉴权的后果——系统处于"裸奔"状态。

---

## 2. 三个核心问题

要给系统装上"门"，你需要解决三个层层递进的问题：

```
请求进入系统
    │
    ▼
┌──────────────────┐
│  认证（AuthN）    │  ← 你是谁？
│  Authentication  │
└────────┬─────────┘
         │ 身份确认后
         ▼
┌──────────────────┐
│  鉴权（AuthZ）    │  ← 你能做什么？
│  Authorization   │
└────────┬─────────┘
         │ 权限确认后
         ▼
┌──────────────────┐
│  会话管理         │  ← 怎么保持登录状态？
│  Session/Token   │
└────────┬─────────┘
         │
         ▼
      业务逻辑
```

### 问题一：认证——"你是谁？"

认证的本质是**身份识别**。当一个请求到达服务器时，服务器需要回答一个问题：这个请求是谁发的？

最朴素的方式是用户名+密码。但 HTTP 是无状态协议——用户第一次请求输入了密码，第二次请求时服务器已经"忘了"他是谁。你不可能让用户每次点一个按钮都输一次密码。

所以你需要一种机制，让用户"登录一次，后续自动识别"。这就是会话管理要解决的问题，我们稍后讨论。

### 问题二：鉴权——"你能做什么？"

认证解决了"是谁"，但"是谁"和"能做什么"是两回事。

在电商后台中：
- 仓库管理员能看商品库存，但不能改价格
- 客服能看订单详情，但不能删除订单
- 超级管理员能做任何事

你需要一套权限模型，把**用户**、**角色**、**权限**关联起来。典型的 RBAC（基于角色的访问控制）模型：

```
用户 ──(N:N)──> 角色 ──(N:N)──> 菜单/权限标识

例如：
张三 ──> 仓库管理员 ──> product:stock:view, product:stock:update
李四 ──> 客服       ──> order:view, order:refund
王五 ──> 超级管理员 ──> *:*（通配符，所有权限）
```

鉴权发生在两个层面：
- **URL 层面**：这个 URL 是否需要登录才能访问？（比如 `/admin-api/captcha` 不需要登录）
- **方法层面**：当前用户有没有执行这个操作的权限？（比如 `system:user:delete`）

### 问题三：会话管理——"怎么保持登录状态？"

用户登录后，你需要在后续请求中记住他。主流方案有两种：

**Session 模式**：服务器保存会话信息，客户端只持有 Session ID（通过 Cookie 传递）。

```
客户端                          服务器
  │                              │
  │── POST /login ──────────────>│
  │   username=zhangsan          │ 验证密码，创建 Session
  │   password=***               │ Session ID = "abc123"
  │                              │ 存储：{abc123 → {userId:1, roles:[...]}}
  │<── Cookie: JSESSIONID=abc123 │
  │                              │
  │── GET /product/list ────────>│
  │   Cookie: JSESSIONID=abc123  │ 从 Cookie 取 Session ID
  │                              │ 查 Session 存储，找到用户 1
  │<── 200 OK (商品列表) ───────│
```

**Token 模式**：服务器不保存会话，客户端持有自包含的 Token（通常放在 Header 中）。

```
客户端                          服务器
  │                              │
  │── POST /login ──────────────>│
  │   username=zhangsan          │ 验证密码，生成 Token
  │   password=***               │ Token = "xyz789"（包含用户信息或映射关系）
  │                              │
  │<── {token: "xyz789"} ───────│
  │                              │
  │── GET /product/list ────────>│
  │   Authorization: xyz789      │ 解析/校验 Token
  │                              │ 从 Token 获取用户信息
  │<── 200 OK (商品列表) ───────│
```

这两种模式的选择，直接影响了后续的架构设计。我们在方案对比中详细讨论。

## 3. 替代方案对比

明确了三个核心问题后，我们来看看业界有哪些成熟的解决方案，以及各自的优劣。

### 方案一：Session + Cookie（传统方案）

这是 Java Web 最经典的方式，Spring Security 默认支持。

**工作原理**：

```
┌─────────┐         ┌─────────────────────────────┐
│ 浏览器  │         │         服务器                │
│         │         │                               │
│ Cookie  │ <─────> │  Session 存储（内存/Redis）    │
│ JSESSIONID│        │  {sessionId → UserContext}    │
└─────────┘         └─────────────────────────────┘
```

用户登录后，服务器创建 Session，将 Session ID 通过 Cookie 返回浏览器。后续请求浏览器自动带上 Cookie，服务器通过 Session ID 找到用户信息。

**优势**：
- Spring Security 原生支持，开箱即用
- 服务端控制会话，随时可以踢人下线
- Cookie 自动携带，前端无感知

**劣势**：
- **有状态**：服务器必须保存 Session，集群部署需要 Session 同步（如 Spring Session + Redis）
- **跨域困难**：Cookie 受同源策略限制，前后端分离、跨域 API 调用时很麻烦
- **移动端不友好**：原生 App 对 Cookie 的支持不如浏览器自然
- **CSRF 风险**：Cookie 会被自动携带，容易受到跨站请求伪造攻击

**适用场景**：传统单体应用、前后端不分离、浏览器端为主的系统。

### 方案二：Spring Security OAuth2（标准方案）

Spring Security OAuth2 是 Spring 官方的 OAuth2 实现，提供完整的授权服务器和资源服务器。

**工作原理**：

```
┌─────────┐     ┌─────────────────┐     ┌─────────────────┐
│  客户端  │────>│  授权服务器      │────>│  资源服务器      │
│ (浏览器) │     │ (Authorization  │     │ (Resource       │
│         │<────│  Server)        │     │  Server)        │
│         │     │                 │     │                 │
│  Token  │     │  - 用户认证     │     │  - Token 校验   │
│         │     │  - 令牌签发     │     │  - 资源保护     │
│         │     │  - 客户端管理   │     │                 │
└─────────┘     └─────────────────┘     └─────────────────┘
```

OAuth2 定义了四种授权模式（Authorization Code、Implicit、Client Credentials、Resource Owner Password），每种模式对应不同的场景。

**优势**：
- 行业标准，RFC 6749 规范定义
- 功能完整：支持多种授权模式、令牌刷新、范围控制（Scope）
- 生态成熟：各种语言和框架都有实现

**劣势**：
- **复杂度高**：授权服务器、资源服务器、客户端三个角色，配置项众多
- **学习成本大**：OAuth2 的概念（授权码、隐式模式、客户端凭证等）对初学者不友好
- **过度设计**：如果你只需要"用户名密码登录 + Token 认证"，OAuth2 的大部分能力用不上
- **Spring Security OAuth2 已废弃**：Spring 官方已将 OAuth2 Server 功能迁移到 Spring Authorization Server，原 spring-security-oauth2 处于维护模式

**适用场景**：需要支持第三方登录、多客户端授权、开放平台等标准 OAuth2 场景。

### 方案三：自研 Token + Spring Security（本项目方案）

yudao-cloud 选择了一条"中间路线"：用 Spring Security 的过滤器和注解体系，但不使用 OAuth2 授权服务器，自己实现 Token 的创建、存储和校验。

**工作原理**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     yudao-spring-boot-starter-security           │
│                     （框架层：通用能力）                           │
│                                                                 │
│  TokenAuthenticationFilter ──> OAuth2TokenCommonApi (Feign)     │
│  @PreAuthorize + @ss         PermissionCommonApi (Feign)        │
│  LoginUser 统一身份模型       SecurityFrameworkUtils 工具类      │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               │ Feign RPC
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                     yudao-module-system                          │
│                     （业务层：具体实现）                           │
│                                                                 │
│  OAuth2TokenService ──> Redis 缓存 + MySQL 持久化               │
│  PermissionService  ──> RBAC 权限查询                           │
│  AdminAuthService   ──> 登录/登出/刷新                          │
└─────────────────────────────────────────────────────────────────┘
```

核心思路是：
- **用 Spring Security 的"壳"**：过滤器链、`@PreAuthorize` 注解、`SecurityContextHolder` 上下文
- **不用 OAuth2 的"核"**：不搞授权服务器，Token 的创建和校验通过自研的 `OAuth2TokenService` 实现
- **框架层与业务层分离**：框架层提供认证鉴权的"骨架"（过滤器、注解、工具类），业务层实现"血肉"（Token 存储、权限查询）

**优势**：
- **轻量**：不引入 OAuth2 授权服务器，代码量和配置量大幅减少
- **灵活**：Token 存储策略（Redis + MySQL 双写）、过期策略、刷新策略完全自控
- **学习成本低**：核心概念只有"Token 校验"和"权限校验"两个
- **架构统一**：单体和微服务共用同一套认证鉴权框架，通过 Feign RPC 解耦

**劣势**：
- **非标准**：不遵循 OAuth2 规范，无法直接对接第三方 OAuth2 客户端
- **需要自研**：Token 管理、刷新机制等需要自己实现和维护
- **生态有限**：没有现成的 OAuth2 生态工具可用

**适用场景**：企业内部管理系统、前后端分离的后台系统、不需要 OAuth2 授权服务器的场景。

### 三种方案对比总览

| 维度 | Session + Cookie | Spring Security OAuth2 | 自研 Token + Spring Security |
|------|-----------------|----------------------|----------------------------|
| **状态管理** | 有状态（服务端存 Session） | 无状态（Token 自包含或查库） | 无状态（Token 存 Redis+MySQL） |
| **集群部署** | 需要 Session 同步 | 天然支持 | 天然支持（共享 Redis） |
| **前后端分离** | Cookie 跨域麻烦 | Header 传 Token，天然支持 | Header 传 Token，天然支持 |
| **移动端适配** | Cookie 不自然 | Token 天然支持 | Token 天然支持 |
| **CSRF 风险** | 高（Cookie 自动携带） | 低（Token 手动携带） | 低（Token 手动携带） |
| **学习成本** | 低 | 高（OAuth2 概念多） | 中（需要理解 Spring Security） |
| **功能完整性** | 基础 | 完整（多种授权模式） | 够用（登录+Token+权限） |
| **灵活性** | 低（受限于 Session 机制） | 中（受限于 OAuth2 规范） | 高（完全自控） |
| **代码复杂度** | 低 | 高 | 中 |
| **适用场景** | 传统单体 | 开放平台、第三方授权 | 企业内部管理系统 |

## 4. 设计目标

综合以上分析，`yudao-spring-boot-starter-security` 的设计目标如下：

### 核心目标

| 目标 | 具体要求 | 对应的痛点 |
|------|---------|-----------|
| **统一认证入口** | `TokenAuthenticationFilter` 拦截所有请求，自动识别用户身份 | 每个 Controller 手动检查 Token |
| **方法级鉴权** | `@PreAuthorize("@ss.hasPermission('...')")` 声明式权限校验 | URL 级权限粒度太粗 |
| **无状态 Token** | 基于 Token 的认证，不依赖 Session | 集群部署的 Session 同步问题 |
| **框架与业务分离** | 框架层提供通用能力，业务层实现具体逻辑 | 认证鉴权逻辑与业务代码耦合 |
| **单体微服务统一** | 同一套框架，单体和微服务都能用 | 切换架构时重写认证鉴权 |

### 扩展目标

| 目标 | 具体要求 |
|------|---------|
| **免认证 URL 管理** | `@PermitAll` 注解 + 配置化，灵活控制哪些 URL 不需要登录 |
| **多用户类型** | 支持管理员（`/admin-api`）和 C 端用户（`/app-api`）两套认证 |
| **微服务用户透传** | Feign 调用时自动传递当前用户信息，下游服务无需重复认证 |
| **异步上下文传递** | `@Async` 异步执行时不丢失用户上下文 |
| **开发调试友好** | Mock 模式：开发环境用密钥直接模拟任意用户，无需真实登录 |
| **Guava Cache 缓存** | 权限校验结果缓存 1 分钟，减少 RPC 调用开销 |

## 5. 整体设计思路：框架层与业务层分离

认证鉴权框架最核心的设计决策是**框架层与业务层分离**。我们用一个类比来说明：

> 想象你在建一栋写字楼。框架层是"建筑结构"——承重墙、楼梯、电梯井，这些是通用的，不管里面开餐厅还是开公司都一样。业务层是"室内装修"——隔间怎么分、灯光怎么布、家具怎么摆，这些取决于具体用途。

对应到认证鉴权框架：

```
┌─────────────────────────────────────────────────────────────────┐
│  框架层（yudao-spring-boot-starter-security）                    │
│  "建筑结构"——通用能力，不依赖具体业务                              │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ TokenAuthenticationFilter                                │  │
│  │ - 拦截请求，提取 Token                                    │  │
│  │ - 调用 OAuth2TokenCommonApi 校验 Token（通过 Feign）     │  │
│  │ - 构建 LoginUser，放入 SecurityContextHolder              │  │
│  └──────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ SecurityFrameworkServiceImpl (@ss Bean)                   │  │
│  │ - @PreAuthorize("@ss.hasPermission('...')")               │  │
│  │ - 调用 PermissionCommonApi 校验权限（通过 Feign）         │  │
│  │ - Guava Cache 缓存 1 分钟                                │  │
│  └──────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ 基础设施                                                  │  │
│  │ - LoginUser：统一身份模型                                  │  │
│  │ - SecurityFrameworkUtils：安全工具类                       │  │
│  │ - YudaoWebSecurityConfigurerAdapter：Spring Security 配置 │  │
│  │ - AuthorizeRequestsCustomizer：URL 安全规则扩展点          │  │
│  └──────────────────────────────────────────────────────────┘  │
└────────────────────────────────┬────────────────────────────────┘
                                 │
                                 │ Feign RPC 调用
                                 │ OAuth2TokenCommonApi
                                 │ PermissionCommonApi
                                 │
┌────────────────────────────────▼────────────────────────────────┐
│  业务层（yudao-module-system）                                   │
│  "室内装修"——具体实现，可替换可扩展                                │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ OAuth2TokenService                                        │  │
│  │ - Token 创建：生成 access_token + refresh_token            │  │
│  │ - Token 校验：Redis 快速查询 + MySQL 持久化               │  │
│  │ - Token 刷新：refresh_token 换新 access_token              │  │
│  │ - Token 移除：登出时删除 Token                             │  │
│  └──────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ PermissionService                                         │  │
│  │ - 查询用户的角色列表                                        │  │
│  │ - 查询用户的权限标识列表                                    │  │
│  │ - 判断用户是否有指定权限/角色                               │  │
│  └──────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ AdminAuthService / MemberAuthService                       │  │
│  │ - 账号密码登录 / 手机号登录 / 社交登录                      │  │
│  │ - 登出：移除 Token                                        │  │
│  │ - 刷新 Token                                              │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 为什么这样分离？

**原因一：框架层不依赖业务**。`TokenAuthenticationFilter` 不知道 Token 是存在 Redis 还是 MySQL，它只调用 `OAuth2TokenCommonApi` 这个接口。如果将来你要换 Token 存储方案（比如从 Redis 迁移到 Memcached），只需要改业务层，框架层一行代码都不用动。

**原因二：单体和微服务共用**。Feign 接口在单体模式下退化为本地调用（Spring 自动注入本地 Bean），微服务模式下走网络调用。框架层代码完全不变。

**原因三：关注点分离**。框架层关注"怎么拦截请求、怎么注入用户上下文"，业务层关注"怎么存 Token、怎么查权限"。两者各司其职，互不干扰。

### 一个请求的完整流转

以"张三创建商品"为例，看整个认证鉴权链路：

```
浏览器                    框架层                          业务层
  │                        │                               │
  │ POST /admin-api/       │                               │
  │ product/create         │                               │
  │ Authorization:         │                               │
  │ Bearer abc123          │                               │
  │───────────────────────>│                               │
  │                        │                               │
  │                        │ TokenAuthenticationFilter     │
  │                        │ 从 Header 提取 Token          │
  │                        │ "abc123"                      │
  │                        │                               │
  │                        │──Feign──> OAuth2TokenCommonApi │
  │                        │          checkAccessToken()   │
  │                        │                               │
  │                        │          Redis 查询 Token     │
  │                        │          返回 {userId:1,      │
  │                        │            userType:2,        │
  │                        │            tenantId:1, ...}   │
  │                        │<──Feign──                     │
  │                        │                               │
  │                        │ 构建 LoginUser                │
  │                        │ 放入 SecurityContextHolder     │
  │                        │                               │
  │                        │ Spring Security URL 检查      │
  │                        │ /product/create 需要认证 ✓    │
  │                        │                               │
  │                        │ Controller 方法被调用          │
  │                        │ @PreAuthorize("@ss            │
  │                        │   .hasPermission(             │
  │                        │   'system:product:create'))   │
  │                        │                               │
  │                        │ SecurityFrameworkServiceImpl  │
  │                        │ 检查 Guava Cache              │
  │                        │                               │
  │                        │──Feign──> PermissionCommonApi  │
  │                        │          hasAnyPermissions()  │
  │                        │                               │
  │                        │          查询 用户1→角色→权限  │
  │                        │          匹配 system:product: │
  │                        │          create ✓             │
  │                        │<──Feign──                     │
  │                        │                               │
  │                        │ 权限校验通过 ✓                │
  │                        │                               │
  │                        │ 执行业务逻辑                   │
  │                        │ productMapper.insert(...)     │
  │                        │                               │
  │<───────────────────────│                               │
  │ 200 OK {id: 1001}      │                               │
```

这个流程中，框架层做了三件事：
1. **认证**：从 Token 识别出"这是张三"（步骤 1-4）
2. **URL 鉴权**：确认这个 URL 需要登录（步骤 5）
3. **方法鉴权**：确认张三有创建商品的权限（步骤 6-8）

业务代码（Controller/Service）完全不需要关心认证鉴权，它只需要声明"我需要什么权限"，框架自动完成校验。

## 思考题

1. 如果你的电商系统需要支持第三方商家接入（商家有自己的系统，需要调用你的 API），你会选择哪种认证方案？自研 Token 方案能否满足需求？需要做什么改造？

2. 在"框架层与业务层分离"的设计中，`TokenAuthenticationFilter` 通过 Feign 调用 `OAuth2TokenCommonApi` 来校验 Token。为什么不直接在框架层实现 Token 校验逻辑（比如直接查 Redis）？这样不是更快吗？

3. `@PreAuthorize("@ss.hasPermission('system:product:create')")` 这种鉴权方式，权限标识是硬编码在代码里的。如果运营人员想在不改代码的情况下调整某个角色的权限，系统该如何支持？这属于框架层还是业务层的职责？

4. 在方案对比中，Session 模式的 CSRF 风险"高"而 Token 模式的 CSRF 风险"低"，这是为什么？Token 模式是否完全没有安全风险？如果不是，它面临的主要安全威胁是什么？
