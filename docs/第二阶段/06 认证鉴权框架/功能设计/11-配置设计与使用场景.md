# 11-配置设计与使用场景

## 场景切入

框架搭建好了，但不同项目的需求不同：
- 有的项目 Token 通过 Cookie 传递，有的通过 Header
- 开发环境需要 Mock 登录方便调试，线上环境必须关闭
- 有些特殊 URL 不需要认证（如内部健康检查）

这些差异通过 `SecurityProperties` 配置类统一管理。

## 配置类：SecurityProperties

```java
@ConfigurationProperties(prefix = "yudao.security")
@Data
public class SecurityProperties {
    private String tokenHeader = "Authorization";
    private String tokenParameter = "token";
    private Boolean mockEnable = false;
    private String mockSecret = "test";
    private List<String> permitAllUrls = Collections.emptyList();
    private Integer passwordEncoderLength = 4;
}
```

### 配置项详解

#### `tokenHeader`：Token 的 Header 名称

```yaml
yudao:
  security:
    token-header: Authorization
```

- **默认值**：`Authorization`
- **用途**：指定从哪个 HTTP Header 中提取 Token
- **使用场景**：
  - 标准场景：`Authorization: Bearer abc123`
  - 自定义场景：`X-Token: abc123`（某些前端框架的约定）
- **源码位置**：`SecurityFrameworkUtils.obtainAuthorization()` 中使用

#### `tokenParameter`：Token 的参数名称

```yaml
yudao:
  security:
    token-parameter: token
```

- **默认值**：`token`
- **用途**：当 Header 中没有 Token 时，从 URL 参数中提取
- **使用场景**：
  - WebSocket 连接：`ws://localhost:8080/ws?token=abc123`（WebSocket 无法设置 Header）
  - 文件下载：`GET /file/download?token=abc123`（某些下载组件不支持 Header）
- **优先级**：Header > Parameter

#### `mockEnable` + `mockSecret`：模拟登录

```yaml
yudao:
  security:
    mock-enable: true
    mock-secret: my-secret-key-2024
```

- **默认值**：`mockEnable=false`，`mockSecret="test"`
- **用途**：开发环境下，用"密钥 + 用户ID"直接模拟登录，无需走完整登录流程
- **使用方式**：请求时带 `Authorization: my-secret-key-20241`，即模拟用户 ID=1 登录
- **安全警告**：线上环境**必须关闭**（`mockEnable=false`）

**为什么需要 Mock 登录？**

开发阶段频繁需要测试不同用户的权限，每次都走完整登录流程太麻烦。Mock 登录让你直接指定用户 ID，快速切换身份。

#### `permitAllUrls`：免认证 URL 列表

```yaml
yudao:
  security:
    permit-all-urls:
      - /druid/**
      - /actuator/**
      - /error
```

- **默认值**：空列表
- **用途**：配置不需要认证的 URL 模式
- **适用场景**：框架组件的 URL（Druid 监控、Actuator 健康检查等）

#### `passwordEncoderLength`：BCrypt 强度

```yaml
yudao:
  security:
    password-encoder-length: 4
```

- **默认值**：4
- **用途**：BCrypt 密码加密的强度（log rounds）
- **性能影响**：
  - 4 → 约 16ms（开发环境推荐）
  - 10 → 约 100ms（生产环境推荐）
  - 每增加 1，耗时翻倍

## 典型配置示例

### 开发环境配置

```yaml
# application-dev.yml
yudao:
  security:
    mock-enable: true                    # 开启 Mock 登录
    mock-secret: "test"                  # 简单密钥
    password-encoder-length: 4           # 低强度，加快启动速度
```

### 生产环境配置

```yaml
# application-prod.yml
yudao:
  security:
    mock-enable: false                   # 关闭 Mock 登录
    password-encoder-length: 10          # 高强度，保障密码安全
    permit-all-urls:
      - /actuator/health                 # 只放行健康检查
```

### 前后端分离配置

```yaml
yudao:
  security:
    token-header: "Authorization"        # 标准 Bearer Token
    token-parameter: "token"             # WebSocket 场景备用
```

### 网关统一认证配置

```yaml
# Gateway 配置
yudao:
  security:
    permit-all-urls:
      - /system/auth/login               # 登录
      - /system/auth/register            # 注册
      - /system/captcha/**               # 验证码
      - /admin-api/system/oauth2/token   # OAuth2 Token 端点
```

## 配置的生效机制

`SecurityProperties` 通过 `@EnableConfigurationProperties` 注册：

```java
@AutoConfiguration
@EnableConfigurationProperties(SecurityProperties.class)
public class YudaoSecurityAutoConfiguration {
    // ...
}
```

各组件通过依赖注入获取配置：

```java
// TokenAuthenticationFilter 中
private final SecurityProperties securityProperties;

// 使用配置
String token = SecurityFrameworkUtils.obtainAuthorization(
    request,
    securityProperties.getTokenHeader(),      // 从配置读取
    securityProperties.getTokenParameter()     // 从配置读取
);
```

## 与 Spring Boot Profile 的结合

不同环境的配置差异通过 Spring Profile 管理：

```
application.yml           ← 公共配置
application-dev.yml       ← 开发环境
application-test.yml      ← 测试环境
application-prod.yml      ← 生产环境
```

启动时指定 Profile：
```bash
java -jar app.jar --spring.profiles.active=prod
```

## 配置项速查表

| 配置项 | 默认值 | 类型 | 说明 |
|--------|--------|------|------|
| `yudao.security.token-header` | `Authorization` | String | Token 的 Header 名称 |
| `yudao.security.token-parameter` | `token` | String | Token 的参数名称 |
| `yudao.security.mock-enable` | `false` | Boolean | Mock 登录开关 |
| `yudao.security.mock-secret` | `test` | String | Mock 登录密钥 |
| `yudao.security.permit-all-urls` | `[]` | List\<String\> | 免认证 URL 列表 |
| `yudao.security.password-encoder-length` | `4` | Integer | BCrypt 加密强度 |

## 思考题

1. **`tokenHeader` 和 `tokenParameter` 的优先级设计合理吗？** 如果反过来（Parameter 优先于 Header）会有什么问题？

2. **Mock 登录用 `mockSecret + userId` 的方式，有没有更安全的替代方案？** 比如：生成一个有签名的 Mock Token。

3. **`passwordEncoderLength` 设置为 4 在生产环境够安全吗？** OWASP 推荐的 BCrypt 强度是多少？提示：考虑暴力破解的成本。

4. **如果需要支持"同一接口在不同租户下有不同的免认证规则"，该怎么扩展配置？** 现有的 `permitAllUrls` 能支持吗？
