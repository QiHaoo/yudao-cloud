# 12-边界与扩展

## 框架不做什么

一个好的框架应该"做该做的事，不越界"。明确框架的边界，才能让使用者知道什么时候需要自己扩展。

### 1. 不管理用户的登录/登出

`TokenAuthenticationFilter` 只负责**校验 Token**，不负责**创建/删除 Token**。登录和登出的逻辑在 `yudao-module-system` 的 `AdminAuthService` 中实现。

这种分离的原因：
- 框架层不关心"用户怎么登录"（账号密码？短信？社交登录？）
- 框架层只关心"这个 Token 是否有效"
- 登录逻辑可以灵活替换，不影响认证框架

### 2. 不实现完整的 OAuth2 授权服务器

本项目只使用了 OAuth2 的**密码模式**和**客户端模式**，没有实现完整的授权服务器（如 Spring Authorization Server）。

不实现的功能：
- 授权码模式的完整流程（授权页面、回调处理）
- JWT Token（使用 UUID 替代）
- Token 自省端点（RFC 7662）

### 3. 不处理"记住我"（Remember-Me）

Spring Security 内置了 Remember-Me 功能，但本项目没有使用。原因是 Token 机制本身就可以通过较长的过期时间实现类似效果。

### 4. 不处理 Session 并发控制

传统 Web 应用需要限制同一用户的并发 Session 数量（如"同一账号只能在一个地方登录"）。本项目基于无状态 Token，不使用 Session，所以不需要这个功能。

### 5. 不处理 CSRF 防护

CSRF（跨站请求伪造）防护依赖于 Session 机制。本项目禁用了 Session（`SessionCreationPolicy.STATELESS`），所以也禁用了 CSRF：

```java
.csrf(AbstractHttpConfigurer::disable)
```

Token 认证天然免疫 CSRF：攻击者无法获取用户的 Token，所以无法伪造请求。

## 与其他组件的分工

### 与 yudao-spring-boot-starter-web 的分工

| 关注点 | security starter | web starter |
|--------|-----------------|-------------|
| 认证 | ✅ TokenAuthenticationFilter | ❌ |
| 鉴权 | ✅ @PreAuthorize + @ss | ❌ |
| 异常处理 | ✅ AuthenticationEntryPoint、AccessDeniedHandler | ✅ GlobalExceptionHandler |
| CORS | ✅ 在 SecurityFilterChain 中配置 | ✅ CorsFilter |
| XSS | ❌ | ✅ XSS 过滤器 |
| API 日志 | ❌ | ✅ ApiAccessLogFilter |

两者有协作关系：security starter 的异常处理器调用 web starter 的 `GlobalExceptionHandler` 来格式化错误响应。

### 与 yudao-module-system 的分工

| 关注点 | security starter（框架层） | system 模块（业务层） |
|--------|--------------------------|---------------------|
| Token 校验 | ✅ 调用 RPC 接口 | ✅ 实现校验逻辑 |
| Token 存储 | ❌ | ✅ Redis + MySQL |
| 用户信息 | ✅ LoginUser 载体 | ✅ 用户 CRUD |
| 权限校验 | ✅ @ss Bean 调用 RPC | ✅ 实现权限查询 |
| 角色管理 | ❌ | ✅ 角色 CRUD |
| 菜单管理 | ❌ | ✅ 菜单 CRUD |

### 与网关的分工

| 关注点 | Gateway | 服务内部 |
|--------|---------|---------|
| Token 校验 | ✅ 网关层校验（可选） | ✅ 服务层校验（必选） |
| URL 路由 | ✅ 路由转发 | ❌ |
| 免认证 URL | ✅ 网关层放行 | ✅ 服务层放行 |
| 用户透传 | ✅ 注入 login-user Header | ✅ 解析 login-user Header |

网关的 Token 校验是可选的——即使网关不校验，服务内部也会校验。网关层校验的好处是：拦截无效请求，减少后端服务的压力。

## 扩展点

### 扩展点一：AuthorizeRequestsCustomizer

每个 Maven 模块可以实现 `AuthorizeRequestsCustomizer` 来自定义 URL 安全规则：

```java
@Component
public class CustomSecurityConfiguration extends AuthorizeRequestsCustomizer {

    @Override
    public void customize(AuthorizeHttpRequestsConfigurer<HttpSecurity>
            .AuthorizationManagerRequestMatcherRegistry registry) {
        // 自定义规则
        registry.requestMatchers(buildAdminApi("/custom/**")).permitAll();
    }

    @Override
    public int getOrder() {
        return 0;  // 控制执行顺序
    }
}
```

**使用场景**：
- 放行模块特有的 URL（如 Druid 监控、代码生成预览）
- 限制某些 URL 只允许特定 HTTP 方法
- 为特定 URL 配置自定义的权限规则

### 扩展点二：SecurityFrameworkService

`SecurityFrameworkService` 是一个接口，可以替换默认实现：

```java
// 默认实现
@Bean("ss")
public SecurityFrameworkService securityFrameworkService(PermissionCommonApi permissionApi) {
    return new SecurityFrameworkServiceImpl(permissionApi);
}

// 自定义实现
@Bean("ss")
public SecurityFrameworkService customSecurityService() {
    return new CustomSecurityFrameworkServiceImpl();
}
```

**使用场景**：
- 权限校验逻辑与默认不同（如基于 ABAC 而非 RBAC）
- 需要集成外部权限系统（如 Keycloak、Casbin）
- 需要更复杂的缓存策略

### 扩展点三：LoginUser 的 info Map

`LoginUser.info` 使用 `Map<String, String>` 存储额外信息，可以自由扩展：

```java
// 在创建 Token 时，将自定义信息放入 info
Map<String, String> info = new HashMap<>();
info.put("nickname", user.getNickname());
info.put("deptId", String.valueOf(user.getDeptId()));
info.put("customField", "customValue");  // 扩展字段

// 在业务代码中获取
String customValue = SecurityFrameworkUtils.getLoginUser()
    .getInfo().get("customField");
```

### 扩展点四：OAuth2TokenCommonApi

Token 的存储和校验逻辑通过 `OAuth2TokenCommonApi` 接口定义。如果需要替换 Token 实现（如使用 JWT 替代 UUID），只需实现这个接口：

```java
// 默认：UUID Token + Redis/Mysql 存储
// 替换：JWT Token + 无状态校验
```

## 安全最佳实践

### 1. 生产环境必须关闭 Mock 登录

```yaml
yudao:
  security:
    mock-enable: false  # 生产环境必须为 false
```

### 2. Token 过期时间不宜过长

- Access Token：建议 2 小时
- Refresh Token：建议 7 天
- 过期时间越长，Token 泄露的风险越大

### 3. 密码加密强度适当

```yaml
yudao:
  security:
    password-encoder-length: 10  # 生产环境推荐 10-12
```

### 4. 免认证 URL 最小化

只放行真正需要免认证的 URL，避免过度使用 `@PermitAll`。

### 5. 敏感操作二次验证

对于删除用户、修改密码等敏感操作，除了权限校验外，建议增加二次验证（如确认密码、短信验证码）。

## 与业界方案的对比

| 维度 | 本项目方案 | Spring Security OAuth2 | Shiro |
|------|-----------|----------------------|-------|
| 认证方式 | 自研 Token | OAuth2 Token | Session/Token |
| 鉴权方式 | @PreAuthorize + @ss | @PreAuthorize + Authority | @RequiresPermissions |
| Token 存储 | UUID + Redis/MySQL | JWT（默认） | Session |
| 微服务支持 | ✅ Feign 透传 | 需要额外配置 | ❌ 单体为主 |
| 学习成本 | 中等 | 高 | 低 |
| 灵活性 | 高（自研） | 中（框架约束） | 中 |

## 思考题

1. **框架不实现完整的 OAuth2 授权服务器，这意味着什么？** 如果未来需要支持第三方应用接入，需要做什么改动？

2. **`AuthorizeRequestsCustomizer` 的 `getOrder()` 方法控制执行顺序。如果有多个模块都实现了这个接口，顺序对结果有什么影响？** 提示：考虑规则冲突的场景。

3. **如果要支持"基于属性的访问控制（ABAC）"（如"只能操作自己创建的订单"），需要怎么扩展框架？** 现有的 RBAC 模型能兼容吗？

4. **框架禁用了 CSRF 防护，这在所有场景下都安全吗？** 如果前端通过 Cookie 存储 Token 呢？
