# 01-总览：认证鉴权框架全景

## 从一个最简单的问题说起

你搭建了一个电商后台管理系统，有商品管理、订单管理、用户管理等模块。现在问题来了：

**怎么防止任何人都能直接访问你的接口？**

最朴素的想法：每个 Controller 方法开头写一段代码，检查请求里有没有带正确的"通行证"。但你很快发现——几十个 Controller、几百个接口，每个都写一遍？而且"正确"的标准还会变：有的接口不需要登录（比如验证码），有的接口需要特定角色（比如只有"超级管理员"才能删除用户）。

你需要一个**统一的、自动化的、可配置的**认证鉴权机制。这就是 `yudao-spring-boot-starter-security` 要解决的问题。

## 两个核心概念：认证与鉴权

在深入之前，先厘清两个经常被混淆的概念：

| 概念 | 回答的问题 | 典型场景 |
|------|-----------|---------|
| **认证（Authentication）** | "你是谁？" | 用户输入账号密码，系统验证通过，知道你是"张三" |
| **鉴权（Authorization）** | "你能做什么？" | 张三想删除商品，系统检查他有没有 `system:product:delete` 权限 |

**先认证，后鉴权**——你得先知道"是谁"，才能判断"能做什么"。

### 第一部分：认证链（你是谁？）

核心是 `TokenAuthenticationFilter`，它拦截每一个请求，按以下优先级识别用户身份：

1. **Header 透传**：从 `login-user` Header 解析（微服务间 RPC 场景）
2. **Token 校验**：从 `Authorization` Header 或 `token` 参数提取 Token，调用 `OAuth2TokenCommonApi.checkAccessToken()` 校验
3. **Mock 模拟**：开发环境下，以特定密钥开头的 Token 直接模拟用户

认证通过后，构建 `LoginUser` 对象，放入 Spring Security 上下文——后续所有代码都能通过 `SecurityFrameworkUtils.getLoginUser()` 获取当前用户。

### 第二部分：鉴权体系（你能做什么？）

认证解决了"你是谁"，但"你能做什么"需要额外的权限模型。框架采用 **RBAC（基于角色的访问控制）** 模型：

```
用户 ──(N:N)──> 角色 ──(N:N)──> 菜单（权限标识）
```

鉴权通过两种方式触发：

- **方法级**：`@PreAuthorize("@ss.hasPermission('system:user:create')")` —— 在 Controller/Service 方法上声明需要的权限
- **URL 级**：`@PermitAll` 标记免认证 URL，其余 URL 默认需要认证

`@ss` 是一个 Spring Bean（`SecurityFrameworkServiceImpl`），它通过 `PermissionCommonApi`（Feign RPC）调用 system 服务校验权限，内部用 Guava Cache 做 1 分钟缓存。

### 第三部分：基础设施支撑

- **`LoginUser`**：统一的登录用户信息载体，包含用户 ID、类型、租户、授权范围等
- **`SecurityFrameworkUtils`**：安全工具类，提供获取/设置当前用户、提取 Token 等静态方法
- **`TransmittableThreadLocal`**：替换 Spring Security 默认的 ThreadLocal，解决 `@Async` 异步场景下用户上下文丢失问题
- **`LoginUserRequestInterceptor`**：Feign 请求拦截器，实现微服务间用户信息透传
- **`AuthorizeRequestsCustomizer`**：抽象类，每个 Maven 模块可实现它来自定义 URL 安全规则

## 一个请求的完整认证鉴权流程

以"张三创建商品"为例：

```
1. 前端发送 POST /admin-api/product/create
   Header: Authorization: Bearer abc123xyz

2. TokenAuthenticationFilter 拦截请求
   ├── 从 Header 提取 Token: "abc123xyz"
   ├── 调用 OAuth2TokenCommonApi.checkAccessToken("abc123xyz")
   │   └── system 服务检查 Redis/MySQL，返回用户信息
   │       {userId: 1, userType: 2, tenantId: 1, scopes: [...]}
   ├── 构建 LoginUser，放入 SecurityContextHolder
   └── 继续过滤链

3. Spring Security 检查 URL 权限
   ├── /admin-api/product/create 不在 @PermitAll 列表中
   ├── 不在 permitAllUrls 配置中
   └── 需要认证 → 已认证 ✓ → 放行

4. Controller 方法被调用
   @PreAuthorize("@ss.hasPermission('system:product:create')")
   public CommonResult<Long> createProduct(@Valid @RequestBody ProductCreateReqVO reqVO) { ... }

5. Spring Security 方法级鉴权
   ├── 调用 @ss.hasPermission('system:product:create')
   │   └── SecurityFrameworkServiceImpl.hasAnyPermissions()
   │       ├── 检查缓存：用户 1 是否有 system:product:create 权限？
   │       ├── 缓存未命中 → 调用 PermissionCommonApi.hasAnyPermissions(1, "system:product:create")
   │       │   └── system 服务查询：用户 1 → 角色列表 → 菜单权限列表 → 匹配
   │       └── 返回 true/false
   ├── true → 执行方法体
   └── false → 返回 403 FORBIDDEN

6. 业务代码执行，返回结果
```

## 为什么这样设计？

### 设计决策 1：自研而非使用 Spring Security OAuth2

**选择**：自研 Token 管理（`OAuth2TokenService`），不使用 Spring Security OAuth2 框架。

**原因**：
- Spring Security OAuth2 过于复杂，学习成本高
- 项目只需要 Token 认证，不需要完整的 OAuth2 授权服务器
- 自研更灵活，能精确控制 Token 存储策略（Redis + MySQL 双写）

### 设计决策 2：框架层与业务层分离

**选择**：认证鉴权的"骨架"在 `yudao-spring-boot-starter-security`（框架层），"血肉"在 `yudao-module-system`（业务层）。

**原因**：
- 框架层提供通用能力（过滤器、鉴权注解、工具类），不依赖具体业务
- 业务层实现具体逻辑（Token 存储、权限查询），可替换可扩展
- 通过 Feign RPC 接口解耦，单体和微服务都能用

### 设计决策 3：@PreAuthorize + SpEL Bean 而非 URL 级权限

**选择**：主要使用方法级鉴权（`@PreAuthorize("@ss.hasPermission('...')")`），而非在 SecurityFilterChain 中配置 URL 权限。

**原因**：
- URL 级权限粒度太粗，无法区分同一 URL 的不同操作
- 方法级权限与代码紧耦合，更直观、更安全
- SpEL Bean 方式灵活，可以扩展任意鉴权逻辑
