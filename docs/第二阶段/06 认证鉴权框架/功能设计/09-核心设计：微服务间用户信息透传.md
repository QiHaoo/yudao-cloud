# 09-核心设计：微服务间用户信息透传

## 场景切入

假设你有一个电商系统，拆分为两个微服务：
- **system 服务**：用户管理、权限管理
- **order 服务**：订单管理

现在，管理员张三（userId=1）在 order 服务中创建了一个订单。order 服务需要记录"这个订单是谁创建的"。但问题是：张三的请求先到达网关，网关校验 Token 后转发给 order 服务。order 服务怎么知道当前用户是张三？

更复杂的情况：order 服务在处理过程中需要调用 system 服务查询用户信息。这个 Feign 调用怎么带上"当前用户是张三"的上下文？

这就是**用户信息透传**要解决的问题。

## 两种透传场景

```
场景一：客户端 → 网关 → 服务
  浏览器 → Gateway → order 服务
  透传方式：Gateway 解析 Token，将用户信息放入 login-user Header

场景二：服务 → 服务
  order 服务 → (Feign) → system 服务
  透传方式：LoginUserRequestInterceptor 自动将当前用户放入 login-user Header
```

### 场景一：网关到服务的透传

```
浏览器                    Gateway                    order 服务
  │                         │                          │
  │  Authorization:         │                          │
  │  Bearer abc123          │                          │
  │ ──────────────────────> │                          │
  │                         │                          │
  │                    ① 校验 Token                    │
  │                    ② 获取用户信息                   │
  │                    ③ 构建 login-user Header        │
  │                         │                          │
  │                         │  login-user:             │
  │                         │  %7B%22id%22%3A1...%7D   │
  │                         │ ───────────────────────> │
  │                         │                          │
  │                         │                    ④ 从 Header 解析 LoginUser
  │                         │                    ⑤ 放入 SecurityContextHolder
  │                         │                          │
  │ <──────────────────────────────────────────────── │
  │                    返回响应                         │
```

Gateway 负责：
1. 校验 Token 的有效性
2. 从 Token 中获取用户信息
3. 将用户信息序列化为 JSON，URL 编码后放入 `login-user` Header

order 服务的 `TokenAuthenticationFilter` 通过**策略一（Header 透传解析）**识别用户：

```java
private LoginUser buildLoginUserByHeader(HttpServletRequest request) {
    String loginUserStr = request.getHeader("login-user");
    if (StrUtil.isEmpty(loginUserStr)) {
        return null;
    }
    loginUserStr = URLDecoder.decode(loginUserStr, StandardCharsets.UTF_8.name());
    return JsonUtils.parseObject(loginUserStr, LoginUser.class);
}
```

### 场景二：服务间 Feign 调用的透传

```
order 服务                      system 服务
  │                                │
  │  当前用户：LoginUser{id=1}      │
  │                                │
  │  orderService.create()         │
  │    → 需要查询用户信息            │
  │    → Feign 调用 system 服务     │
  │                                │
  │  LoginUserRequestInterceptor   │
  │    ① 获取当前 LoginUser         │
  │    ② 序列化为 JSON              │
  │    ③ URL 编码                   │
  │    ④ 放入 login-user Header    │
  │                                │
  │  GET /rpc-api/system/user/get  │
  │  login-user: %7B%22id%22...%7D │
  │ ─────────────────────────────> │
  │                                │
  │                          ⑤ TokenAuthenticationFilter
  │                            从 Header 解析 LoginUser
  │                          ⑥ 放入 SecurityContextHolder
  │                          ⑦ 执行业务逻辑
  │                                │
  │ <──────────────────────────── │
  │        返回结果                 │
```

核心组件是 `LoginUserRequestInterceptor`，它实现了 Feign 的 `RequestInterceptor` 接口：

```java
public class LoginUserRequestInterceptor implements RequestInterceptor {

    @Override
    public void apply(RequestTemplate requestTemplate) {
        LoginUser user = SecurityFrameworkUtils.getLoginUser();
        if (user == null) {
            return;  // 没有登录用户，不透传
        }
        String userStr = JsonUtils.toJsonString(user);
        userStr = URLEncoder.encode(userStr, StandardCharsets.UTF_8.name());
        requestTemplate.header("login-user", userStr);
    }
}
```

**每次 Feign 调用时，这个拦截器会自动执行**，不需要业务代码手动传递用户信息。

## 为什么用 Header 而不是其他方式？

| 方式 | 优点 | 缺点 |
|------|------|------|
| **Header（采用）** | 标准 HTTP 机制，所有框架都支持 | 需要 URL 编码处理中文 |
| 参数 | 简单直观 | 暴露在 URL 中，会被日志记录；不适合复杂对象 |
| Cookie | 浏览器自动携带 | 微服务间调用不适用；有跨域限制 |
| 独立 RPC 调用 | 可以传递任意数据 | 多一次网络开销；增加延迟 |

## URL 编码的必要性

LoginUser 的 JSON 可能包含中文（如昵称"张三"），直接放在 Header 中可能有编码问题。URL 编码可以确保安全传输：

```
原始 JSON: {"id":1,"info":{"nickname":"张三"}}
URL 编码:  %7B%22id%22%3A1%2C%22info%22%3A%7B%22nickname%22%3A%22%E5%BC%A0%E4%B8%89%22%7D%7D
```

发送方（`LoginUserRequestInterceptor`）负责编码，接收方（`TokenAuthenticationFilter`）负责解码。

## 上下文传播的完整性

用户信息透传不是一次性的，它需要在整个调用链中保持：

```
浏览器 → Gateway → A 服务 → B 服务 → C 服务
         ↓          ↓         ↓         ↓
      LoginUser  LoginUser  LoginUser  LoginUser
      (网关注入)  (Header解析) (Feign透传) (Feign透传)
```

每一层都通过相同机制透传：
1. **网关层**：Token → LoginUser → login-user Header
2. **服务层**：login-user Header → LoginUser → SecurityContextHolder
3. **Feign 层**：SecurityContextHolder → LoginUser → login-user Header

## 与 Spring Cloud Sleuth/Zipkin 的对比

Spring Cloud Sleuth 通过 `TracingFeignClient` 透传 Trace ID，用于分布式链路追踪。本项目的 `LoginUserRequestInterceptor` 做的是类似的事情，但透传的是用户身份信息。

| 维度 | Sleuth Trace ID | LoginUser 透传 |
|------|----------------|---------------|
| 透传内容 | Trace ID + Span ID | 完整的用户信息 |
| 存储位置 | MDC (ThreadLocal) | SecurityContextHolder |
| 生命周期 | 单次请求链路 | 单次请求链路 |
| 序列化 | 简单字符串 | JSON + URL 编码 |

## 安全考量

### 1. login-user Header 的可信度

在微服务架构中，login-user Header 来自内部网络，应该被信任。但要确保：
- **网关是唯一的入口**：外部请求不能直接访问内部服务
- **RPC 路径受保护**：`/rpc-api/**` 路径应该只允许内网访问

### 2. Header 大小限制

LoginUser 的 JSON 可能比较大（特别是 `info` Map 包含很多信息时）。HTTP Header 通常有大小限制（默认 8KB），需要注意不要超过限制。

### 3. 序列化安全

`LoginUser` 使用 Jackson 序列化/反序列化。要确保：
- `@JsonIgnore` 标注了 `context` 字段，避免序列化临时数据
- 没有敏感信息（如密码）被放入 `info` Map

## 思考题

1. **如果 order 服务需要调用 system 服务，system 服务又需要调用 infra 服务，用户信息能正确透传吗？** 提示：考虑 Feign 拦截器的执行时机。

2. **如果 Gateway 和 order 服务之间有 Nginx 反向代理，login-user Header 会被 Nginx 丢弃吗？** 需要做什么配置？

3. **为什么不把 login-user Header 的透传逻辑放在 Spring Cloud Gateway 的全局过滤器中，而是放在每个服务的 TokenAuthenticationFilter 中？** 提示：考虑单体部署模式。

4. **如果要支持"服务间调用不带用户信息"的场景（如定时任务触发的调用），该怎么处理？** 现有的设计是否支持？
