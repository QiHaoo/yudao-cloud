# 01-总览：角色权限与菜单全景

## 从一个电商后台说起

你的电商平台上线了，团队也扩大了。现在有运营人员、仓库管理员、客服、财务……每个人需要访问后台系统，但你显然不希望仓库管理员能修改商品价格，也不希望客服能看到财务报表。

问题很明确：**不同的人，能做的事情不一样。**

在第六章（认证鉴权框架）中，我们已经解决了"你是谁"的问题——通过 Token 认证，系统能识别每个请求来自哪个用户。但认证只是第一步，更关键的是：**这个用户能做什么？**

这就是本章要解决的问题——**权限管理**。

## 权限管理的三个核心问题

一个完整的权限管理体系需要回答三个问题：

| 问题 | 对应能力 | 举例 |
|------|---------|------|
| **谁能登录？** | 认证（第六章已解决） | 张三输入账号密码，系统验证通过 |
| **谁能做什么操作？** | 功能权限（本章核心） | 张三是运营，能编辑商品；李四是仓库，只能查看订单 |
| **谁能看哪些数据？** | 数据权限（本章扩展） | 北京店店长只能看到北京店的订单，看不到上海店的 |

## RBAC：角色与权限的桥梁

项目采用 **RBAC（Role-Based Access Control，基于角色的访问控制）** 模型。其核心思想是：**用户不直接拥有权限，而是通过角色间接获得权限。**

```
┌──────────┐       ┌──────────┐       ┌──────────────────────┐
│  用户     │──N:N──>│  角色     │──N:N──>│  菜单（权限标识）      │
│  User    │       │  Role    │       │  Menu                │
└──────────┘       └──────────┘       └──────────────────────┘
                          │
                          │  数据范围
                          ▼
                   ┌──────────┐
                   │  部门     │
                   │  Dept    │
                   └──────────┘
```

为什么不让用户直接关联权限？想象一个电商平台有 50 个运营人员，每个人的权限都一样。如果直接关联，你需要维护 50 份相同的权限配置。而通过角色，你只需要创建一个"运营"角色，把 50 个人都关联到这个角色上——改一次角色权限，50 个人同时生效。

## 三个核心实体

### 1. 菜单（MenuDO）——权限的载体

在本项目中，"菜单"不仅仅是前端侧边栏的导航项，更是**权限的基本单位**。一个菜单记录代表一个可访问的资源，通过 `permission` 字段标识：

```
system_menu 表核心字段：
┌────────────────────────────────────────────────────────────────┐
│ id          │ 菜单编号                                          │
│ name        │ 菜单名称，如"用户管理"                              │
│ permission  │ 权限标识，如"system:user:create"                   │
│ type        │ 类型：DIR(1)=目录, MENU(2)=菜单, BUTTON(3)=按钮    │
│ parent_id   │ 父菜单ID，形成树形结构                              │
│ sort        │ 排序                                              │
│ path        │ 路由地址                                          │
│ component   │ 前端组件路径                                       │
│ status      │ 状态：启用/禁用                                    │
└────────────────────────────────────────────────────────────────┘
```

三种类型各司其职：

- **目录（DIR）**：纯分组，如"系统管理"，对应侧边栏的一级菜单
- **菜单（MENU）**：对应一个页面，如"用户管理"，点击后跳转到 `/system/user`
- **按钮（BUTTON）**：对应一个操作，如"新增用户"，不展示在导航栏，但控制按钮的显示/隐藏和后端接口的访问权限

### 2. 角色（RoleDO）——权限的集合

角色是一组权限的集合，同时附带数据范围信息：

```
system_role 表核心字段：
┌────────────────────────────────────────────────────────────────┐
│ id              │ 角色编号                                      │
│ name            │ 角色名称，如"运营主管"                          │
│ code            │ 角色标识，如"tenant_admin"                     │
│ type            │ 类型：SYSTEM(1)=内置, CUSTOM(2)=自定义          │
│ data_scope      │ 数据范围：全部/指定部门/本部门/仅本人 等          │
│ data_scope_dept_ids │ 自定义部门ID集合（data_scope=DEPT_CUSTOM时）│
│ sort            │ 排序                                          │
│ status          │ 状态                                          │
└────────────────────────────────────────────────────────────────┘
```

项目预置了几个内置角色（`RoleCodeEnum`）：

| 角色标识 | 名称 | 说明 |
|---------|------|------|
| `super_admin` | 超级管理员 | 拥有所有权限，不受权限校验限制 |
| `tenant_admin` | 租户管理员 | 租户内的管理员 |
| `crm_admin` | CRM 管理员 | CRM 模块专用 |

### 3. 用户（AdminUserDO）——操作的主体

用户通过两张中间表与角色和菜单关联：

```
system_user_role 表（用户-角色关联）：
┌──────────────────────────────┐
│ id       │ 自增主键           │
│ user_id  │ 用户ID             │
│ role_id  │ 角色ID             │
└──────────────────────────────┘

system_role_menu 表（角色-菜单关联）：
┌──────────────────────────────┐
│ id       │ 自增主键           │
│ role_id  │ 角色ID             │
│ menu_id  │ 菜单ID             │
└──────────────────────────────┘
```

## 两种权限控制方式

项目同时支持 URL 级和方法级两种权限控制，它们解决不同层面的问题：

### URL 级权限（第六章 SecurityFilterChain）

在 Spring Security 的过滤器链中，通过 `@PermitAll` 标记哪些 URL 不需要认证。这是**粗粒度**的控制——要么能访问，要么不能。

```
@PermitAll                              → 任何人都能访问（如验证码接口）
无注解                                   → 必须登录才能访问
```

### 方法级权限（@PreAuthorize + @ss Bean）

在 Controller/Service 方法上，通过 `@PreAuthorize` 注解声明需要的权限。这是**细粒度**的控制——精确到某个操作。

```java
// 来源：MenuController.java
@PostMapping("/create")
@PreAuthorize("@ss.hasPermission('system:menu:create')")
public CommonResult<Long> createMenu(@Valid @RequestBody MenuSaveVO createReqVO) {
    // ...
}
```

`@ss` 是一个 Spring Bean（`SecurityFrameworkServiceImpl`），它暴露了以下方法供 SpEL 表达式调用：

| 方法 | 用途 | 示例 |
|------|------|------|
| `hasPermission(permission)` | 判断是否有某个权限 | `@ss.hasPermission('system:user:create')` |
| `hasAnyPermissions(p1, p2)` | 判断是否有任一权限 | `@ss.hasAnyPermissions('system:user:query', 'system:user:export')` |
| `hasRole(role)` | 判断是否有某个角色 | `@ss.hasRole('tenant_admin')` |
| `hasAnyRoles(r1, r2)` | 判断是否有任一角色 | `@ss.hasAnyRoles('tenant_admin', 'crm_admin')` |

## 一次权限校验的完整流程

以"运营主管张三编辑商品"为例：

```
1. 前端发送 PUT /admin-api/product/update
   Header: Authorization: Bearer abc123xyz
   Body: {id: 1001, name: "iPhone 16", price: 5999}

2. TokenAuthenticationFilter（第六章）
   ├── 提取 Token，校验通过
   └── 构建 LoginUser {userId: 1, tenantId: 1}

3. Controller 方法被调用
   @PreAuthorize("@ss.hasPermission('system:product:update')")
   public CommonResult<Boolean> updateProduct(...) { ... }

4. Spring Security 拦截，执行 SpEL 表达式
   └── @ss.hasPermission('system:product:update')
       └── SecurityFrameworkServiceImpl.hasAnyPermissions()
           ├── Guava Cache 查询：用户1 + "system:product:update"
           ├── 缓存未命中
           └── 调用 PermissionCommonApi.hasAnyPermissions(1, "system:product:update")

5. PermissionServiceImpl.hasAnyPermissions()（本章核心逻辑）
   ├── 查询用户1的角色：["运营主管"]
   ├── 遍历权限列表 "system:product:update"
   │   ├── 通过 permission 查到对应的 menuId
   │   ├── 通过 menuId 查到拥有该菜单的角色列表
   │   └── 判断用户角色与菜单角色是否有交集 → 有！
   └── 返回 true

6. 权限校验通过，执行业务逻辑，返回结果
```

整个过程对业务代码**完全透明**——开发者只需要在方法上加一行 `@PreAuthorize` 注解，权限校验由框架自动完成。

## 数据范围：行级数据过滤

功能权限控制"能做什么操作"，数据权限控制"能看哪些数据"。

一个电商平台的典型场景：北京店店长只能看到北京店的订单，上海店店长只能看到上海店的订单，而总部运营能看到所有订单。

项目通过 `RoleDO.dataScope` 字段定义数据范围（`DataScopeEnum`）：

| 枚举值 | 含义 | 效果 |
|-------|------|------|
| `ALL(1)` | 全部数据 | 不做任何过滤 |
| `DEPT_CUSTOM(2)` | 指定部门 | 只能看到指定部门的数据 |
| `DEPT_ONLY(3)` | 本部门 | 只能看到自己所在部门的数据 |
| `DEPT_AND_CHILD(4)` | 本部门及下级 | 能看到自己部门及所有子部门的数据 |
| `SELF(5)` | 仅本人 | 只能看到自己创建的数据 |

数据权限的实现原理（第十一章会详细展开）：在 SQL 执行前，自动追加 `WHERE dept_id IN (...)` 或 `WHERE create_user_id = ?` 条件，对业务代码无侵入。
