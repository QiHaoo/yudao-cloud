# 06-配置设计与边界

## 配置项设计

角色权限与菜单模块本身没有独立的配置项，但依赖以下配置：

```yaml
# 组件 6 的配置（认证鉴权框架）
yudao:
  security:
    permit-all-urls: []  # 额外放行的 URL，不受权限控制

# Spring Security 配置
spring:
  security:
    filter:
      order: -100  # SecurityFilterChain 的过滤器顺序
```

### 为什么没有独立配置？

角色权限与菜单的配置全部通过数据库管理：
- 角色定义 → `system_role` 表
- 菜单定义 → `system_menu` 表
- 角色-菜单关联 → `system_role_menu` 表
- 用户-角色关联 → `system_user_role` 表

这种"数据库驱动配置"的方式比"配置文件驱动"更灵活，管理员可以在界面上直接操作。

## 组件边界

### 本组件做了什么

1. **角色 CRUD**：创建、修改、删除、查询角色
2. **菜单 CRUD**：创建、修改、删除、查询菜单
3. **权限分配**：角色-菜单关联、用户-角色关联
4. **数据权限配置**：为角色设置数据范围

### 本组件不做什么

| 不做的事 | 交给谁 | 原因 |
|---------|--------|------|
| 权限校验的执行 | 组件 6（SecurityFrameworkServiceImpl） | 本组件只管理数据，组件 6 负责校验 |
| 用户管理 | 组件 7 | 用户 CRUD 在用户模块 |
| 数据权限的 SQL 改写 | 数据权限框架 | 由 MyBatis 拦截器自动完成 |
| 菜单的前端渲染 | 前端 | 前端根据菜单数据渲染导航栏 |

### 与组件 6 的数据流

```
本组件（管理权限数据）
    │
    ▼
system_menu 表 → permission 字段（如 "system:user:create"）
    │
    ▼
system_role_menu 表 → 角色与菜单的关联
    │
    ▼
组件 6 的 SecurityFrameworkServiceImpl
    │
    ▼
@PreAuthorize("@ss.hasPermission('system:user:create')") → 校验
```

### 与数据权限框架的协作

```
本组件：配置角色的数据范围（dataScope, dataScopeDeptIds）
    │
    ▼
PermissionServiceImpl.getDeptDataPermission() → 计算用户的可见部门
    │
    ▼
数据权限框架：@DataPermission 注解 + MyBatis 拦截器
    │
    ▼
SQL 改写：WHERE dept_id IN (可见部门列表)
```

## 设计约束

### 1. 菜单是全局共享的

`MenuDO` 使用 `@TenantIgnore`，意味着所有租户共享同一套菜单。这是一个设计决策：

- ✅ 简化了菜单管理，不需要每个租户重复配置
- ❌ 不同租户无法自定义菜单

如果需要租户级自定义菜单，需要：
1. 移除 `@TenantIgnore`
2. 菜单增加 `tenantId` 字段
3. 权限校验时按租户过滤菜单

### 2. 角色是租户隔离的

`RoleDO` 继承 `TenantBaseDO`，每个租户有独立的角色体系。同一个角色名在不同租户下可以重复。

### 3. 超级管理员有特殊处理

```java
// RoleCodeEnum
TENANT_ADMIN("super_admin")  // 租户超管
```

超管跳过所有权限检查。删除超管角色会被系统拒绝。
