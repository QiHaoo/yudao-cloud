# 03-核心设计：菜单与权限模型

## 一个容易被误解的概念

很多初学者看到"菜单管理"，第一反应是："这不就是管侧边栏导航吗？加个目录、加个页面，前端渲染一下就完了。"

如果你也这么想，请先看这个场景：

运营主管张三登录后台，侧边栏显示"商品管理"菜单，点击进入商品列表页。页面上有"新增商品"、"编辑商品"、"删除商品"三个按钮。张三能点"新增"和"编辑"，但"删除"按钮是灰色的——因为他的角色没有"删除商品"的权限。

这里涉及两个层面的权限控制：
1. **页面级**：张三能不能看到"商品管理"这个菜单？（侧边栏导航）
2. **操作级**：张三能不能点"删除商品"这个按钮？（按钮显隐 + 接口校验）

在本项目中，这两个层面的权限**都由菜单（MenuDO）承载**。菜单不只是导航项，更是**权限的基本单位**。

## MenuDO：一个实体，三种类型

让我们从源码出发，看看菜单实体的完整设计：

```java
// 来源：MenuDO.java
@TableName("system_menu")
@TenantIgnore  // 菜单跨租户共享，不受租户过滤
public class MenuDO extends BaseDO {

    public static final Long ID_ROOT = 0L;  // 根节点的 parentId

    private Long id;           // 菜单编号
    private String name;       // 菜单名称，如"用户管理"
    private String permission; // 权限标识，如"system:user:create"
    private Integer type;      // 菜单类型：DIR(1) / MENU(2) / BUTTON(3)
    private Integer sort;      // 显示顺序
    private Long parentId;     // 父菜单ID
    private String path;       // 路由地址
    private String icon;       // 菜单图标
    private String component;  // 前端组件路径
    private String componentName; // 组件名（用于 keep-alive 缓存）
    private Integer status;    // 状态：启用/禁用
    private Boolean visible;   // 是否在侧边栏可见
    private Boolean keepAlive; // 是否启用 Vue 路由缓存
    private Boolean alwaysShow; // 是否总是显示（即使只有一个子菜单）
}
```

菜单有三种类型（`MenuTypeEnum`），各司其职：

```java
// 来源：MenuTypeEnum.java
public enum MenuTypeEnum {
    DIR(1),    // 目录：纯分组容器
    MENU(2),   // 菜单：对应一个页面
    BUTTON(3)  // 按钮：对应一个操作
}
```

用一个类比来理解：如果把后台管理系统比作一本书，目录（DIR）就是章，菜单（MENU）就是节，按钮（BUTTON）就是节里的某个具体操作步骤。

```
目录（DIR）     → "第3章 商品管理"          → 侧边栏一级菜单
菜单（MENU）    → "3.1 商品列表"            → 侧边栏二级菜单，点击跳转页面
按钮（BUTTON）  → "3.1.1 新增商品"          → 页面上的操作按钮，不在侧边栏显示
```

## 菜单树：parentId 构建的层级结构

菜单通过 `parentId` 字段形成树形结构。根节点的 `parentId` 固定为 `0`（`ID_ROOT`）。

以一个电商后台为例，完整的菜单树如下：

```
parentId=0 (根)
│
├── [DIR] 系统管理 (id=1, sort=10)
│   │
│   ├── [MENU] 用户管理 (id=11, parentId=1, path=/system/user)
│   │   ├── [BUTTON] 用户查询 (id=111, permission="system:user:query")
│   │   ├── [BUTTON] 用户新增 (id=112, permission="system:user:create")
│   │   ├── [BUTTON] 用户编辑 (id=113, permission="system:user:update")
│   │   └── [BUTTON] 用户删除 (id=114, permission="system:user:delete")
│   │
│   ├── [MENU] 角色管理 (id=12, parentId=1, path=/system/role)
│   │   ├── [BUTTON] 角色查询 (id=121, permission="system:role:query")
│   │   ├── [BUTTON] 角色新增 (id=122, permission="system:role:create")
│   │   ├── [BUTTON] 角色编辑 (id=123, permission="system:role:update")
│   │   └── [BUTTON] 角色删除 (id=124, permission="system:role:delete")
│   │
│   └── [MENU] 菜单管理 (id=13, parentId=1, path=/system/menu)
│       └── [BUTTON] 菜单查询 (id=131, permission="system:menu:query")
│       └── ...
│
├── [DIR] 商品管理 (id=2, sort=20)
│   │
│   ├── [MENU] 商品列表 (id=21, parentId=2, path=/product/list)
│   │   ├── [BUTTON] 商品查询 (id=211, permission="product:spu:query")
│   │   ├── [BUTTON] 商品新增 (id=212, permission="product:spu:create")
│   │   ├── [BUTTON] 商品编辑 (id=213, permission="product:spu:update")
│   │   └── [BUTTON] 商品删除 (id=214, permission="product:spu:delete")
│   │
│   └── [MENU] 分类管理 (id=22, parentId=2, path=/product/category)
│       └── ...
│
├── [DIR] 订单管理 (id=3, sort=30)
│   │
│   ├── [MENU] 订单列表 (id=31, parentId=3, path=/order/list)
│   │   ├── [BUTTON] 订单查询 (id=311, permission="order:order:query")
│   │   ├── [BUTTON] 订单发货 (id=312, permission="order:order:deliver")
│   │   └── [BUTTON] 订单退款 (id=313, permission="order:order:refund")
│   │
│   └── ...
│
└── [DIR] 数据统计 (id=4, sort=40)
    │
    └── [MENU] 销售报表 (id=41, parentId=4, path=/statistics/sales)
        ├── [BUTTON] 报表查询 (id=411, permission="statistics:report:query")
        └── [BUTTON] 报表导出 (id=412, permission="statistics:report:export")
```

这棵树同时服务于前端和后端：

```
前端视角：
├── 目录（DIR）  → 侧边栏的一级菜单（可展开的分组）
├── 菜单（MENU） → 侧边栏的二级菜单（可点击跳转的页面）
└── 按钮（BUTTON）→ 页面上的操作按钮（通过 v-if / v-auth 指令控制显隐）

后端视角：
├── 目录（DIR）  → 无权限含义，纯前端展示用
├── 菜单（MENU） → 对应一个页面路由，可选地也有 permission 标识
└── 按钮（BUTTON）→ 核心权限单位，permission 字段与 @PreAuthorize 配合
```

## 权限标识符：`{模块}:{实体}:{操作}`

权限标识符（`permission` 字段）是连接前后端权限控制的纽带。项目采用三段式命名规范：

```
{模块}:{实体}:{操作}

示例：
system:user:create    → 系统模块的用户实体，创建操作
product:spu:update    → 商品模块的SPU实体，更新操作
order:order:deliver   → 订单模块的订单实体，发货操作
```

各段的含义：

| 段 | 含义 | 示例 |
|----|------|------|
| 第一段：模块 | 业务模块或服务名 | `system`、`product`、`order` |
| 第二段：实体 | 业务实体 | `user`、`role`、`spu`、`order` |
| 第三段：操作 | CRUD 或业务操作 | `create`、`update`、`delete`、`query`、`export`、`deliver` |

从源码中可以看到，`MenuDO` 的注释明确说明了这个规范：

```java
// 来源：MenuDO.java
/**
 * 权限标识
 *
 * 一般格式为：${系统}:${模块}:${操作}
 * 例如说：system:admin:add，即 system 服务的添加管理员。
 *
 * 当我们把该 MenuDO 赋予给角色后，意味着该角色有该资源：
 * - 对于后端，配合 @PreAuthorize 注解，配置 API 接口需要该权限，从而对 API 接口进行权限控制。
 * - 对于前端，配合前端标签，配置按钮是否展示，避免用户没有该权限时，结果可以看到该操作。
 */
private String permission;
```

### 权限标识符如何连接前后端

这是一个关键设计——**同一个 permission 字段，同时被前端和后端使用**：

**后端使用**（接口级权限控制）：

```java
// 来源：RoleController.java
@PostMapping("/create")
@PreAuthorize("@ss.hasPermission('system:role:create')")
public CommonResult<Long> createRole(@Valid @RequestBody RoleSaveReqVO createReqVO) {
    return success(roleService.createRole(createReqVO, null));
}
```

**前端使用**（按钮级权限控制）：

```vue
<!-- 前端伪代码：基于 permission 控制按钮显隐 -->
<el-button v-auth="'system:role:create'" @click="handleCreate">新增</el-button>
<el-button v-auth="'system:role:update'" @click="handleUpdate">编辑</el-button>
<el-button v-auth="'system:role:delete'" @click="handleDelete">删除</el-button>
```

前端通过 `v-auth` 指令（或类似实现），在渲染时检查当前用户是否拥有指定的 permission。如果没有，按钮不会渲染到 DOM 中——用户连看都看不到。

这种设计的优势：**权限定义一次，前后端同时生效**。不需要分别在前端和后端维护两套权限配置。

## 按钮类型（BUTTON）与 @PreAuthorize 的映射

三种菜单类型中，按钮（BUTTON）是权限控制的核心。它的 `permission` 字段直接对应 `@PreAuthorize` 注解中的权限标识。

映射关系如下：

```
MenuDO (type=BUTTON, permission="product:spu:create")
    │
    ├── 前端：v-auth="'product:spu:create'" 控制按钮显隐
    │
    └── 后端：@PreAuthorize("@ss.hasPermission('product:spu:create')") 控制接口访问
```

一个完整的映射示例（以商品管理为例）：

| 菜单记录 | type | permission | 前端效果 | 后端效果 |
|---------|------|-----------|---------|---------|
| 商品管理 | DIR(1) | - | 侧边栏一级菜单 | 无 |
| 商品列表 | MENU(2) | - | 侧边栏二级菜单，点击跳转 | 对应页面路由 |
| 商品查询 | BUTTON(3) | `product:spu:query` | 查询按钮可见 | 列表接口可访问 |
| 商品新增 | BUTTON(3) | `product:spu:create` | 新增按钮可见 | 新增接口可访问 |
| 商品编辑 | BUTTON(3) | `product:spu:update` | 编辑按钮可见 | 编辑接口可访问 |
| 商品删除 | BUTTON(3) | `product:spu:delete` | 删除按钮可见 | 删除接口可访问 |

注意：目录和菜单也可以有 `permission` 字段，但通常不设置。它们的主要作用是前端导航，而非权限控制。

## RoleMenuDO：角色与权限的桥梁

角色和菜单是多对多关系，通过 `system_role_menu` 中间表关联：

```java
// 来源：RoleMenuDO.java
@TableName("system_role_menu")
public class RoleMenuDO extends TenantBaseDO {
    private Long id;      // 自增主键
    private Long roleId;  // 角色ID
    private Long menuId;  // 菜单ID
}
```

分配权限的过程就是往这张表里增删记录。例如，给"运营主管"角色分配"商品新增"和"商品编辑"权限：

```sql
INSERT INTO system_role_menu (role_id, menu_id) VALUES (100, 212);  -- 商品新增
INSERT INTO system_role_menu (role_id, menu_id) VALUES (100, 213);  -- 商品编辑
```

权限校验时，反向查询：

```
用户张三 → system_user_role → 角色ID列表 → system_role_menu → 菜单ID列表 → 匹配 permission
```

这个查询链路在 `PermissionServiceImpl.hasAnyPermissions()` 中实现：

```java
// 来源：PermissionServiceImpl.java（简化）
public boolean hasAnyPermissions(Long userId, String... permissions) {
    // 1. 获得用户的角色列表
    List<RoleDO> roles = getEnableUserRoleListByUserIdFromCache(userId);
    if (CollUtil.isEmpty(roles)) {
        return false;
    }

    // 2. 遍历权限标识，逐一检查
    for (String permission : permissions) {
        if (hasAnyPermission(roles, permission)) {
            return true;
        }
    }

    // 3. 超级管理员拥有所有权限
    return roleService.hasAnySuperAdmin(convertSet(roles, RoleDO::getId));
}

private boolean hasAnyPermission(List<RoleDO> roles, String permission) {
    // 通过 permission 查到对应的 menuId 列表
    List<Long> menuIds = menuService.getMenuIdListByPermissionFromCache(permission);
    if (CollUtil.isEmpty(menuIds)) {
        return false;
    }

    // 判断用户的任一角色是否拥有这些菜单
    Set<Long> roleIds = convertSet(roles, RoleDO::getId);
    for (Long menuId : menuIds) {
        Set<Long> menuRoleIds = getSelf().getMenuRoleIdListByMenuIdFromCache(menuId);
        if (CollUtil.containsAny(menuRoleIds, roleIds)) {
            return true;
        }
    }
    return false;
}
```

## 严格模式：权限标识必须存在

注意源码中的这行注释：

```java
// 采用严格模式，如果权限找不到对应的 Menu 的话，也认为没有权限
if (CollUtil.isEmpty(menuIds)) {
    return false;
}
```

这是一个重要的设计决策。项目要求：**如果一个 `@PreAuthorize` 注解引用的 permission 在菜单表中不存在，校验直接失败。**

为什么采用严格模式？因为它能及早发现配置错误。如果你在代码里写了 `@PreAuthorize("@ss.hasPermission('product:spu:delete')")`，但忘了在菜单表里创建对应的 BUTTON 记录，校验会直接失败——你能立刻发现问题。

相反，如果采用宽松模式（找不到就放行），权限配置的遗漏会被隐藏，可能导致未授权访问。

## @TenantIgnore：菜单的跨租户特性

注意 `MenuDO` 上的 `@TenantIgnore` 注解：

```java
@TenantIgnore
public class MenuDO extends BaseDO { ... }
```

这意味着菜单数据**不受租户过滤**——所有租户共享同一套菜单定义。

为什么？因为菜单定义的是"系统有哪些功能"，这是全局统一的。一个电商平台的商品管理功能，不管哪个租户都在用。租户之间的差异体现在**角色**上——不同租户的角色可以分配不同的菜单权限。

```
菜单（全局共享）              角色（租户隔离）
├── 商品查询                  租户A的运营主管 → 商品查询 ✓
├── 商品新增                  租户A的运营主管 → 商品新增 ✓
├── 商品删除                  租户A的运营主管 → 商品删除 ✗
└── 系统配置                  租户B的运营主管 → 商品查询 ✓
                            租户B的运营主管 → 商品新增 ✓
                            租户B的运营主管 → 商品删除 ✓
                            租户B的运营主管 → 系统配置 ✗
```

## 前端如何使用菜单数据

前端拿到菜单数据后，用于三个目的：

### 1. 渲染侧边栏导航

前端调用 `GET /system/menu/list` 获取完整的菜单树，过滤出 `type=DIR` 和 `type=MENU` 的记录，递归渲染侧边栏：

```
┌─────────────────────────────┐
│  系统管理          [DIR]    │
│  ├── 用户管理      [MENU]   │
│  ├── 角色管理      [MENU]   │
│  └── 菜单管理      [MENU]   │
│                             │
│  商品管理          [DIR]    │
│  ├── 商品列表      [MENU]   │
│  └── 分类管理      [MENU]   │
│                             │
│  订单管理          [DIR]    │
│  └── 订单列表      [MENU]   │
└─────────────────────────────┘
```

### 2. 配置路由表

每个 `type=MENU` 的记录包含 `path`（路由地址）和 `component`（前端组件路径），前端据此动态生成 Vue Router 路由配置：

```javascript
// 伪代码：根据菜单数据生成路由
{
  path: '/product/list',
  component: () => import('@/views/product/list.vue'),
  meta: { title: '商品列表', icon: 'shopping' }
}
```

### 3. 控制按钮显隐

前端获取用户拥有的权限列表（permission 字段集合），通过自定义指令控制按钮的显示/隐藏：

```javascript
// 伪代码：v-auth 指令实现
app.directive('auth', {
  mounted(el, binding) {
    const permissions = getUserPermissions() // ['product:spu:query', 'product:spu:create']
    if (!permissions.includes(binding.value)) {
      el.parentNode.removeChild(el) // 移除按钮
    }
  }
})
```

## 设计决策回顾

| 决策 | 选择 | 原因 |
|------|------|------|
| 权限载体 | 菜单（MenuDO） | 一套数据服务前后端，避免重复定义 |
| 菜单类型 | DIR / MENU / BUTTON 三级 | 区分导航分组、页面、操作三个层次 |
| 权限标识格式 | `{模块}:{实体}:{操作}` | 结构化、可读、易管理 |
| 严格模式 | permission 不存在则拒绝 | 及早发现配置错误 |
| 菜单租户隔离 | @TenantIgnore 跨租户共享 | 菜单定义全局功能，租户差异在角色层体现 |
| 树形结构 | parentId 自引用 | 简单直观，支持无限层级 |

## 小结

本章深入了菜单与权限模型的细节设计。核心要点：

1. **菜单不只是导航**：MenuDO 同时承载前端导航和后端权限两种职责
2. **三种类型各司其职**：DIR 是分组容器，MENU 是页面，BUTTON 是操作权限
3. **权限标识是纽带**：`{模块}:{实体}:{操作}` 格式的 permission 字段连接前后端
4. **严格模式保安全**：权限标识必须在菜单表中存在，否则校验失败
5. **跨租户共享菜单**：菜单定义全局功能，通过角色实现租户隔离
