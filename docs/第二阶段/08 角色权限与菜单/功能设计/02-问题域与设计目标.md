# 02-问题域与设计目标

## 一个电商后台的权限困境

你的电商平台发展迅速，团队从最初的 3 个人扩展到了 30 人。每个人都在用后台管理系统，但职责各不相同：

- **超级管理员**：掌控一切，能操作系统的所有功能
- **运营主管**：管理商品上下架、查看销售数据，但不能修改系统配置
- **仓库管理员**：查看订单、处理发货，但不能修改商品价格
- **客服专员**：查看订单详情、处理退款，但不能查看财务数据
- **店长**：只能看到自己门店的订单和库存

你面临一个核心问题：**怎么控制每个人能做什么、能看到什么？**

## 最朴素的方案：硬编码

最直接的想法——在代码里写死：

```java
public void updateProductPrice(Long productId, BigDecimal newPrice) {
    Long userId = SecurityFrameworkUtils.getLoginUserId();

    // 硬编码：只有 user_id = 1（超级管理员）和 user_id = 5（运营主管）能改价格
    if (!Arrays.asList(1L, 5L).contains(userId)) {
        throw new ServiceException(ErrorCodeConstants.NO_PERMISSION);
    }

    productService.updatePrice(productId, newPrice);
}
```

这种方案的问题显而易见：

**问题一：维护噩梦。** 每加一个人、每加一个接口，都要改代码。30 个员工、500 个接口，你要写 15000 个 if-else？

**问题二：权限变更需要发版。** 运营主管离职了，新人接替，你得改代码、重新部署。

**问题三：无法动态配置。** 想临时给客服加一个"查看商品"的权限？改代码、测试、发布，至少半天。

**问题四：数据权限更难处理。** "店长只能看自己门店的数据"——你需要在每个查询接口的 SQL 里手动加 `WHERE store_id = ?`。漏一个就是安全漏洞。

结论很明确：**硬编码的权限控制不可维护、不可扩展、不安全。**

## 三种主流权限模型

业界在长期实践中发展出了三种主要的访问控制模型。我们逐一分析，看看为什么项目最终选择了 RBAC。

### 方案一：ACL（Access Control List，访问控制列表）

ACL 是最简单的权限模型——直接为每个用户分配权限。

```
┌──────────┐       ┌──────────────────────┐
│  用户 A   │──N:N──>│  权限列表             │
│  用户 B   │       │  product:create      │
│  用户 C   │       │  product:update      │
│          │       │  order:query         │
│          │       │  ...                 │
└──────────┘       └──────────────────────┘
```

**优点**：简单直观，容易理解。

**缺点**：
- **用户多时维护困难**。100 个运营人员，每人 20 个权限，你需要维护 2000 条权限记录。新人入职时，要一个个勾选权限，容易遗漏。
- **权限变更成本高**。运营团队的权限要调整？100 个人逐个修改。
- **缺乏抽象层**。权限直接挂在用户上，没有"角色"这个中间概念，无法批量管理。

**适用场景**：用户极少、权限简单的系统，如个人博客的后台。

### 方案二：RBAC（Role-Based Access Control，基于角色的访问控制）

RBAC 在用户和权限之间引入了**角色**这个中间层：

```
┌──────────┐       ┌──────────┐       ┌──────────────────────┐
│  用户     │──N:N──>│  角色     │──N:N──>│  权限（菜单）         │
└──────────┘       └──────────┘       └──────────────────────┘

示例：
用户: 张三 ──> 角色: 运营主管 ──> 权限: product:create, product:update, order:query, ...
用户: 李四 ──> 角色: 仓库管理员 ──> 权限: order:query, stock:update, ...
用户: 王五 ──> 角色: 运营主管 ──> 权限: product:create, product:update, order:query, ...
```

**优点**：
- **批量管理**。100 个运营人员共享"运营主管"角色，改一次角色权限，100 人同时生效。
- **符合组织架构**。角色天然对应岗位职责，业务人员容易理解。
- **行业标准**。RBAC 是最广泛采用的权限模型，有成熟的理论和实践基础。
- **灵活性适中**。比 ACL 强大，比 ABAC 简单，适合绝大多数业务系统。

**缺点**：
- **角色爆炸风险**。如果每个细微的权限差异都创建新角色，角色数量会膨胀。
- **无法处理上下文相关的权限**。比如"工作日才能审批"这种基于时间/环境的规则，RBAC 无法直接表达。

**适用场景**：绝大多数企业应用、后台管理系统、电商平台。

### 方案三：ABAC（Attribute-Based Access Control，基于属性的访问控制）

ABAC 不使用角色，而是基于用户属性、资源属性、环境属性的组合策略来决定权限：

```
策略示例：
IF 用户.部门 == "财务部"
   AND 用户.职级 >= 3
   AND 资源.类型 == "报表"
   AND 时间.在工作时间内
   AND 环境.IP 在公司网段内
THEN 允许访问
```

**优点**：
- **极度灵活**。可以表达任意复杂的权限规则。
- **细粒度控制**。可以精确到"某个人在某个时间段从某个 IP 访问某个资源"。
- **无需预定义角色**。权限规则动态组合，不需要提前规划角色体系。

**缺点**：
- **复杂度高**。策略引擎的学习和维护成本远高于 RBAC。
- **性能开销**。每次权限校验都要评估多条策略，性能不如 RBAC 的简单查表。
- **调试困难**。多条策略叠加时，很难判断最终结果是允许还是拒绝。
- **过度设计**。对于大多数业务系统，ABAC 的灵活性是用不上的。

**适用场景**：安全要求极高的系统（如军事、金融核心系统），或权限规则极度复杂的场景。

### 三种方案对比

| 维度 | ACL | RBAC | ABAC |
|------|-----|------|------|
| 复杂度 | 低 | 中 | 高 |
| 灵活性 | 低 | 中 | 高 |
| 维护成本 | 用户多时极高 | 适中 | 策略多时极高 |
| 学习曲线 | 平缓 | 平缓 | 陡峭 |
| 批量管理 | 不支持 | 角色天然支持 | 需要策略分组 |
| 数据权限 | 需自行实现 | 可集成数据范围 | 天然支持 |
| 适用规模 | 小型系统 | 中大型系统 | 特定领域 |
| 业界采用度 | 低 | 最高 | 低 |

**项目选择 RBAC**，原因是它在简单性和灵活性之间取得了最佳平衡，是后台管理系统的行业标准方案。

## 设计目标

基于以上分析，项目确定了以下设计目标：

### 目标一：基于角色的权限管理

用户不直接拥有权限，通过角色间接获得权限。支持一个用户拥有多个角色，一个角色包含多个权限。

```
设计约束：
- 用户与角色是 N:N 关系（一个用户可以有多个角色）
- 角色与菜单是 N:N 关系（一个角色可以有多个权限）
- 权限校验时，取用户所有角色的权限并集
```

### 目标二：菜单驱动的权限标识

权限不是独立存在的，而是挂在菜单（Menu）上。菜单同时服务于两个目的：

- **前端**：渲染侧边栏导航、控制按钮显隐
- **后端**：通过 `permission` 字段与 `@PreAuthorize` 注解配合，实现接口级权限控制

这种设计避免了"权限列表"和"菜单列表"分开维护的问题——一套数据，两种用途。

### 目标三：数据范围的行级过滤

功能权限解决"能做什么操作"，数据权限解决"能看到哪些数据"。两者结合才是完整的权限体系。

通过角色的 `dataScope` 字段，支持五种数据范围：全部、指定部门、本部门、本部门及下级、仅本人。数据过滤对业务代码透明，通过 SQL 改写自动实现。

### 目标四：租户隔离

在多租户场景下，每个租户有自己的角色和权限体系。租户 A 的"运营主管"和租户 B 的"运营主管"可以拥有不同的权限。

系统预置的内置角色（如 `super_admin`）跨租户共享，自定义角色则严格隔离。

### 目标五：缓存优化

权限校验是高频操作——每个请求都可能触发。因此必须有缓存机制，避免每次都查数据库。

项目使用 Guava Cache（1 分钟过期）做本地缓存，同时支持 Redis 做分布式缓存。权限变更时自动清除相关缓存。

## 不做什么

明确边界同样重要。以下能力**不属于**本组件的职责：

| 不做的事 | 为什么不做 | 谁来做 |
|---------|-----------|--------|
| Token 的生成与校验 | 属于认证范畴 | 第六章：认证鉴权框架 |
| 用户的增删改查 | 用户管理是独立模块 | 第七章：用户管理与登录 |
| 部门的增删改查 | 部门管理是独立模块 | 第七章：用户管理与登录 |
| SQL 改写实现数据过滤 | 属于数据权限框架 | 第十一章：数据权限 |
| 前端菜单渲染逻辑 | 属于前端实现 | 前端项目 |

本组件专注于：**角色、菜单、权限分配、权限校验**这四个核心能力。

## 与第六章的关系：定义与执行的分离

理解本组件与第六章（认证鉴权框架）的关系至关重要：

```
┌─────────────────────────────────────────────────────────┐
│  第六章：认证鉴权框架                                      │
│  （执行者：负责"检查"权限）                                │
│                                                         │
│  @PreAuthorize → @ss.hasPermission() → 调用 RPC 查权限   │
│                                                         │
│  提供：                                                  │
│  - TokenAuthenticationFilter（认证）                     │
│  - SecurityFrameworkServiceImpl（鉴权入口）              │
│  - @PreAuthorize + @ss Bean（方法级鉴权注解）             │
└───────────────────────┬─────────────────────────────────┘
                        │
                        │ "权限是什么？" —— 需要查数据库
                        ▼
┌─────────────────────────────────────────────────────────┐
│  第八章：角色权限与菜单                                    │
│  （定义者：负责"管理"权限）                                │
│                                                         │
│  提供：                                                  │
│  - 菜单 CRUD（定义权限）                                  │
│  - 角色 CRUD（聚合权限）                                  │
│  - 权限分配（用户↔角色↔菜单）                             │
│  - 权限校验逻辑（查用户角色→查角色菜单→匹配权限）           │
│  - 数据范围管理（行级数据过滤）                            │
└─────────────────────────────────────────────────────────┘
```

简单来说：第六章是"执法者"，本章是"立法者"。第六章定义了检查权限的机制（怎么查），本章定义了权限的内容（查什么）。两者通过 `PermissionCommonApi`（Feign RPC 接口）连接。

## 小结

本章从一个电商平台的真实场景出发，分析了权限管理的核心问题，对比了 ACL、RBAC、ABAC 三种方案，最终确定了 RBAC 作为项目的权限模型。

核心设计决策：

| 决策 | 选择 | 原因 |
|------|------|------|
| 权限模型 | RBAC | 简单性与灵活性的最佳平衡 |
| 权限载体 | 菜单（Menu） | 一套数据服务前后端两个目的 |
| 数据权限 | 角色绑定数据范围 | 与功能权限统一管理 |
| 租户隔离 | 角色表带租户字段 | 每个租户独立的权限体系 |
| 缓存策略 | Guava Cache + Redis | 高频校验必须有缓存 |
