# 05-核心设计：数据权限（行级过滤）

## 场景引入

电商平台有多个门店：
- **总部运营**：可以看到所有门店的订单数据
- **华东区经理**：只能看到华东区门店的数据
- **门店店长**：只能看到自己门店的数据
- **普通客服**：只能看到自己处理的订单

URL 权限和方法权限控制的是"能不能访问这个接口"，数据权限控制的是"能看到哪些数据"。

## 数据权限 vs 功能权限

```
功能权限（@PreAuthorize）：
  "你能不能访问订单查询接口？"
  → 能 / 不能

数据权限（DataPermission）：
  "你能看到哪些订单？"
  → 全部 / 本部门 / 本部门及下级 / 仅自己
```

两者是正交的：
- 有功能权限 + 有数据权限 → 能访问，能看到数据
- 有功能权限 + 无数据权限 → 能访问，看不到数据（返回空）
- 无功能权限 → 直接 403

## DataScopeEnum 五级数据范围

```java
public enum DataScopeEnum {
    ALL(1),              // 全部数据权限
    DEPT_CUSTOM(2),      // 指定部门数据权限
    DEPT_ONLY(3),        // 部门数据权限
    DEPT_AND_CHILD(4),   // 部门及以下数据权限
    SELF(5);             // 仅本人数据权限
}
```

### 五级数据范围的含义

```
全部数据（ALL）
├── 可以看到所有部门、所有人的数据
└── 适用于：超级管理员、总部运营

指定部门（DEPT_CUSTOM）
├── 可以看到指定部门的数据
├── 例如：华东区经理看到 [上海店, 杭州店, 南京店]
└── dataScopeDeptIds 字段存储指定的部门 ID 列表

本部门（DEPT_ONLY）
├── 只能看到自己所在部门的数据
└── 适用于：门店店长

本部门及下级（DEPT_AND_CHILD）
├── 可以看到自己部门 + 所有子部门的数据
└── 适用于：区域经理（能看到区域下所有门店）

仅本人（SELF）
├── 只能看到自己创建/负责的数据
└── 适用于：普通客服、销售人员
```

## 数据权限的计算

```java
// PermissionServiceImpl.getDeptDataPermission()
public DeptDataPermissionRespDTO getDeptDataPermission(Long userId) {
    // 获取用户的所有启用角色
    List<RoleDO> roles = getEnableUserRoleListByUserIdFromCache(userId);

    DeptDataPermissionRespDTO result = new DeptDataPermissionRespDTO();
    for (RoleDO role : roles) {
        if (role.getDataScope() == DataScopeEnum.ALL) {
            result.setAll(true);
        } else if (role.getDataScope() == DataScopeEnum.DEPT_CUSTOM) {
            result.getDeptIds().addAll(role.getDataScopeDeptIds());
        } else if (role.getDataScope() == DataScopeEnum.DEPT_ONLY) {
            result.getDeptIds().add(userDeptId);
        } else if (role.getDataScope() == DataScopeEnum.DEPT_AND_CHILD) {
            result.getDeptIds().addAll(deptService.getChildDeptIdList(deptId));
            result.getDeptIds().add(deptId);
        } else if (role.getDataScope() == DataScopeEnum.SELF) {
            result.setSelf(true);
        }
    }
    return result;
}
```

### 多角色的数据范围合并

一个用户可能有多个角色，每个角色有不同的数据范围。合并规则：

1. **取并集**：任何一个角色能看的数据，用户都能看
2. **ALL 优先**：如果任一角色是 ALL，结果就是全部
3. **SELF 兜底**：如果所有角色都是 SELF，只能看自己的数据

```
示例：
用户 A 有两个角色：
  - 角色 1：DEPT_ONLY（本部门 100）
  - 角色 2：DEPT_AND_CHILD（部门 200，子部门 201, 202）

合并结果：deptIds = {100, 200, 201, 202}
```

## 数据权限的实现原理

数据权限通过 SQL 改写实现。在查询语句中自动追加 `WHERE dept_id IN (...)` 条件：

```sql
-- 原始 SQL
SELECT * FROM order WHERE status = 'paid'

-- 数据权限改写后（本部门）
SELECT * FROM order WHERE status = 'paid' AND dept_id IN (100)

-- 数据权限改写后（本部门及下级）
SELECT * FROM order WHERE status = 'paid' AND dept_id IN (100, 101, 102)
```

这个改写过程由 `@DataPermission` 注解和 MyBatis 拦截器自动完成。

### @DataPermission 注解

```java
// 在需要数据权限的方法上标注
@DataPermission(enable = false)  // 关闭数据权限
public CommonResult<AuthPermissionInfoRespVO> getPermissionInfo() { ... }

@DataPermission  // 开启数据权限（默认）
public CommonResult<PageResult<OrderDO>> getOrderPage(...) { ... }
```

## 数据权限的角色配置

管理员在"角色管理"页面为每个角色配置数据范围：

```
角色：华东区经理
├── 数据范围：指定部门（DEPT_CUSTOM）
└── 指定部门：[华东区, 上海店, 杭州店, 南京店]

角色：门店店长
├── 数据范围：本部门（DEPT_ONLY）
└── 指定部门：N/A（自动取用户所在部门）
```

## 数据权限与租户的关系

```
租户隔离（横向）：tenant_id
├── 不同租户的数据完全隔离
└── 由 TenantLineHandler 自动追加

数据权限（纵向）：dept_id
├── 同一租户内，按部门隔离
└── 由 DataPermission 拦截器追加
```

两者是正交的：先过滤租户，再过滤部门。
