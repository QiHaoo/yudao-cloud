# 起步：角色与菜单实体

> 本文档对应源码：`yudao-module-system-server` 模块下 `dal/dataobject/permission/` 目录

## 一、整体 ER 关系图

角色权限体系的核心是四张表，通过两张中间表将用户、角色、菜单串联起来：

```
┌─────────────────────┐         ┌──────────────────────┐
│   system_user_role   │         │   system_role_menu    │
│─────────────────────│         │──────────────────────│
│ id (PK)             │         │ id (PK)              │
│ userId (FK → user)  │         │ roleId (FK → role)   │
│ roleId (FK → role)  │         │ menuId (FK → menu)   │
│ tenant_id           │         │ tenant_id            │
└────────┬────────────┘         └──────────┬───────────┘
         │  N:1                            │  N:1
         ▼                                 ▼
┌─────────────────────┐         ┌──────────────────────┐
│   system_role        │         │   system_menu         │
│─────────────────────│         │──────────────────────│
│ id (PK)             │         │ id (PK)              │
│ name                │         │ name                 │
│ code                │         │ permission           │
│ sort                │         │ type (DIR/MENU/BTN)  │
│ status              │         │ sort                 │
│ type (SYSTEM/CUSTOM)│         │ parentId             │
│ remark              │         │ path / icon / component│
│ dataScope           │         │ status / visible      │
│ dataScopeDeptIds    │         │ keepAlive / alwaysShow│
│ tenant_id           │         │ (无 tenant_id，全局共享)│
└─────────────────────┘         └──────────────────────┘
```

**关键设计**：`system_role` 和 `system_role_menu` 都带 `tenant_id`（继承自 `TenantBaseDO`），表示角色是租户隔离的——每个租户有自己的角色集合。而 `system_menu` 标注了 `@TenantIgnore`，菜单是全局共享的，所有租户看到同一套菜单（通过租户套餐控制可见性）。

## 二、RoleDO —— 角色实体

源码路径：`cn.iocoder.yudao.module.system.dal.dataobject.permission.RoleDO`

```java
@TableName(value = "system_role", autoResultMap = true)
public class RoleDO extends TenantBaseDO {

    @TableId
    private Long id;
    private String name;            // 角色名称，如"商品管理员"
    private String code;            // 角色标识，如"product_admin"
    private Integer sort;           // 显示顺序
    private Integer status;         // 状态：0=开启，1=禁用（CommonStatusEnum）
    private Integer type;           // 类型：1=内置，2=自定义（RoleTypeEnum）
    private String remark;          // 备注

    // ---- 数据权限相关 ----
    private Integer dataScope;      // 数据范围（DataScopeEnum）
    @TableField(typeHandler = JacksonTypeHandler.class)
    private Set<Long> dataScopeDeptIds; // 自定义部门 ID 集合（仅 DEPT_CUSTOM 时使用）
}
```

**字段解读**：

| 字段 | 说明 | 电商场景举例 |
|------|------|-------------|
| `name` | 显示名称 | "商品管理员"、"订单客服" |
| `code` | 程序标识，全局唯一 | `product_admin`、`order_service` |
| `type` | SYSTEM(1) 为系统内置不可删，CUSTOM(2) 为用户自建 | 超级管理员是 SYSTEM 类型 |
| `dataScope` | 控制该角色能看哪些部门的数据 | "订单客服"只能看自己部门的订单 |
| `dataScopeDeptIds` | 当 dataScope=DEPT_CUSTOM 时，指定具体部门 | 指定"华东区"和"华南区"两个部门 |

**继承关系**：`RoleDO` 继承 `TenantBaseDO`，自动拥有 `tenantId`、`creator`、`createTime`、`updater`、`updateTime`、`deleted` 字段。MyBatis-Plus 的多租户拦截器会自动在 SQL 中追加 `WHERE tenant_id = ?` 条件。

## 三、MenuDO —— 菜单实体

源码路径：`cn.iocoder.yudao.module.system.dal.dataobject.permission.MenuDO`

```java
@TableName("system_menu")
@TenantIgnore  // 菜单全局共享，不受租户过滤
public class MenuDO extends BaseDO {

    public static final Long ID_ROOT = 0L; // 根节点 ID

    @TableId
    private Long id;
    private String name;           // 菜单名称
    private String permission;     // 权限标识，格式：${系统}:${模块}:${操作}
    private Integer type;          // 类型：1=目录，2=菜单，3=按钮（MenuTypeEnum）
    private Integer sort;          // 显示顺序
    private Long parentId;         // 父菜单 ID，0 表示根节点
    private String path;           // 路由地址（http 开头则为外链）
    private String icon;           // 菜单图标
    private String component;      // 组件路径，如 "system/user/index"
    private String componentName;  // 组件名，如 "SystemUser"
    private Integer status;        // 状态：0=开启，1=禁用
    private Boolean visible;       // 是否在侧边栏可见
    private Boolean keepAlive;     // 是否启用 Vue keep-alive 缓存
    private Boolean alwaysShow;    // 是否总是显示（即使只有一个子菜单）
}
```

**permission 字段的命名规范**：`${系统}:${模块}:${操作}`

```
system:user:list       → 查看用户列表
system:user:create     → 创建用户
system:user:update     → 编辑用户
system:user:delete     → 删除用户
system:order:export    → 导出订单
system:product:audit   → 审核商品
```

前端通过 `v-permission` 指令控制按钮是否渲染，后端通过 `@PreAuthorize("@ss.hasPermission('system:user:create')")` 做接口级鉴权。

**三种菜单类型**（`MenuTypeEnum`）：

| 类型 | 值 | 说明 | 电商举例 |
|------|-----|------|---------|
| DIR | 1 | 目录，用于侧边栏分组 | "商品管理"、"订单管理" |
| MENU | 2 | 页面，对应一个 Vue 路由 | "商品列表"、"订单详情" |
| BUTTON | 3 | 按钮，对应一个操作权限 | "创建商品"、"审核订单" |

**注意**：BUTTON 类型的菜单不需要 `path`、`icon`、`component`，在创建时 `MenuServiceImpl.initMenuProperty()` 会自动将这些字段置空。

## 四、中间表：RoleMenuDO 与 UserRoleDO

### 4.1 RoleMenuDO —— 角色与菜单的多对多关联

源码路径：`cn.iocoder.yudao.module.system.dal.dataobject.permission.RoleMenuDO`

```java
@TableName("system_role_menu")
public class RoleMenuDO extends TenantBaseDO {

    @TableId
    private Long id;
    private Long roleId;   // 角色 ID
    private Long menuId;   // 菜单 ID
}
```

一张角色-菜单关联记录表示"某个角色拥有某个菜单的访问权限"。例如：

```
roleId=1, menuId=101  →  "商品管理员" 拥有 "商品列表" 权限
roleId=1, menuId=102  →  "商品管理员" 拥有 "创建商品" 权限
roleId=2, menuId=101  →  "订单客服"   拥有 "商品列表" 权限（只读）
```

### 4.2 UserRoleDO —— 用户与角色的多对多关联

源码路径：`cn.iocoder.yudao.module.system.dal.dataobject.permission.UserRoleDO`

```java
@TableName("system_user_role")
public class UserRoleDO extends BaseDO {

    @TableId
    private Long id;
    private Long userId;   // 用户 ID
    private Long roleId;   // 角色 ID
}
```

注意 `UserRoleDO` 继承的是 `BaseDO` 而非 `TenantBaseDO`，因为用户本身已经通过 `system_user` 表的 `tenant_id` 实现了租户隔离，这里不需要重复。

## 五、核心枚举

### 5.1 MenuTypeEnum —— 菜单类型

源码路径：`cn.iocoder.yudao.module.system.enums.permission.MenuTypeEnum`

```java
public enum MenuTypeEnum {
    DIR(1),    // 目录
    MENU(2),   // 菜单（页面）
    BUTTON(3)  // 按钮（操作权限）
}
```

### 5.2 DataScopeEnum —— 数据范围

源码路径：`cn.iocoder.yudao.module.system.enums.permission.DataScopeEnum`

```java
public enum DataScopeEnum {
    ALL(1),           // 全部数据权限
    DEPT_CUSTOM(2),   // 指定部门数据权限
    DEPT_ONLY(3),     // 仅本部门数据权限
    DEPT_AND_CHILD(4),// 本部门及以下数据权限
    SELF(5)           // 仅本人数据权限
}
```

**电商场景举例**：

| 枚举值 | 含义 | 场景 |
|--------|------|------|
| ALL | 能看所有部门的数据 | 平台运营总监 |
| DEPT_CUSTOM | 只看指定几个部门 | 华东大区经理（只能看华东、华南） |
| DEPT_ONLY | 只看自己所在部门 | 仓储部主管 |
| DEPT_AND_CHILD | 看自己部门及下属子部门 | 物流中心负责人（含下属分拣站） |
| SELF | 只看自己创建的数据 | 普通销售员 |

### 5.3 RoleTypeEnum —— 角色类型

源码路径：`cn.iocoder.yudao.module.system.enums.permission.RoleTypeEnum`

```java
public enum RoleTypeEnum {
    SYSTEM(1),  // 内置角色，不可删除
    CUSTOM(2)   // 自定义角色
}
```

内置角色（如 `super_admin`）在代码中硬编码保护，`RoleServiceImpl.validateRoleForUpdate()` 会拒绝修改或删除 SYSTEM 类型的角色。

## 六、实体关系总结

```
User (用户)  ──N:N──  Role (角色)  ──N:N──  Menu (菜单)
              UserRoleDO              RoleMenuDO

一个用户可以有多个角色（如同时是"商品管理员"和"订单审核员"）
一个角色可以分配给多个用户
一个角色可以拥有多个菜单权限
一个菜单权限可以被分配给多个角色
```

**权限校验的计算路径**：`用户 → UserRoleDO → 角色 → RoleMenuDO → 菜单(permission) → 是否匹配接口要求的权限标识`
