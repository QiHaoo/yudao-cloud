# 设计模式提炼：角色权限与菜单

> 从角色权限与菜单组件中识别出的设计模式，抽象为可迁移的通用知识。

---

## 1. 访问控制列表模式（ACL Pattern）

### 模式识别

RBAC（基于角色的访问控制）的核心是两张关联表，它们本质上是访问控制列表：

| 关联表 | DO 类 | 语义 |
|--------|-------|------|
| `system_user_role` | `UserRoleDO` | 用户 X 拥有角色 Y |
| `system_role_menu` | `RoleMenuDO` | 角色 Y 拥有菜单（权限）Z |

一条 ACL 记录就是一个 `(subject, object)` 二元组。完整的权限判断需要两次 ACL 查询：

```
用户 → [UserRoleDO] → 角色 → [RoleMenuDO] → 菜单/权限
```

### 数据结构

```java
// UserRoleDO — 用户与角色的关联
public class UserRoleDO extends BaseDO {
    private Long id;
    private Long userId;    // 主体：谁
    private Long roleId;    // 被授权的角色
}

// RoleMenuDO — 角色与菜单的关联
public class RoleMenuDO extends TenantBaseDO {
    private Long id;
    private Long roleId;    // 主体：哪个角色
    private Long menuId;    // 资源：哪个菜单/权限
}
```

### 设计意图

- **解耦主体与资源**：用户不直接关联权限，通过角色间接关联。修改角色的权限，自动影响所有拥有该角色的用户
- **多租户隔离**：`RoleMenuDO` 继承 `TenantBaseDO`，同一个角色在不同租户下可以有不同的菜单权限
- **批量授权**：给一个角色分配权限，等同于给该角色下的所有用户分配权限

### 可迁移经验

电商系统中的商品分类权限可以采用同样的模式：

- `SellerCategoryRole`：卖家 X 的角色 Y 管理分类 Z
- 通过角色间接关联，实现"运营主管管理所有分类，运营专员只管部分分类"

## 2. 组合模式（Composite Pattern）

### 模式识别

菜单通过 `parentId` 形成树形结构，包含三种节点类型：

```
目录（DIR）── 顶级导航分组
  ├── 菜单（MENU）── 可访问的页面
  │     └── 按钮（BUTTON）── 页面内的操作权限
  └── 菜单（MENU）
        └── 按钮（BUTTON）
```

`MenuDO` 中通过 `parentId` 字段建立父子关系：

```java
public class MenuDO extends BaseDO {
    public static final Long ID_ROOT = 0L;  // 根节点
    private Long parentId;                    // 父菜单 ID
    private Integer type;                     // DIR=1, MENU=2, BUTTON=3
    private String permission;                // 权限标识，如 system:user:add
}
```

### 树遍历与权限收集

`MenuServiceImpl.filterDisableMenus()` 实现了从叶子到根的递归遍历：

```java
private boolean isMenuDisabled(MenuDO node, Map<Long, MenuDO> menuMap, Set<Long> disabledMenuCache) {
    if (disabledMenuCache.contains(node.getId())) {
        return true;
    }
    if (CommonStatusEnum.isDisable(node.getStatus())) {
        disabledMenuCache.add(node.getId());
        return true;
    }
    Long parentId = node.getParentId();
    if (ObjUtil.equal(parentId, ID_ROOT)) {
        return false;
    }
    // 递归检查父节点
    MenuDO parent = menuMap.get(parentId);
    if (parent == null || isMenuDisabled(parent, menuMap, disabledMenuCache)) {
        disabledMenuCache.add(node.getId());
        return true;
    }
    return false;
}
```

### 设计意图

- **统一数据结构**：目录、菜单、按钮用同一张表、同一个 DO，通过 `type` 字段区分
- **灵活组装**：前端可以递归构建菜单树，后端可以递归收集权限标识
- **级联控制**：父节点禁用时，子节点自动失效

### 与经典组合模式的差异

经典组合模式定义统一的 `Component` 接口，`Leaf` 和 `Composite` 分别实现。本项目更轻量——所有节点都是同一个 `MenuDO`，通过 `parentId` 自引用实现树形结构。这是关系数据库中的常见做法，比面向对象的组合模式更实用。

### 可迁移经验

电商系统中的商品分类同样适用：

- 一级分类（家电）→ 二级分类（冰箱）→ 三级分类（双门冰箱）
- 通过 `parentId` 建树，递归计算分类路径
- 禁用上级分类时，下级分类自动不可见

## 3. 代理模式（Proxy Pattern）

### 模式识别

`SecurityFrameworkServiceImpl` 是权限校验的代理层，它在调用方和实际权限服务之间插入了缓存逻辑：

```
Controller（@PreAuthorize）
    │
    ▼
SecurityFrameworkServiceImpl（代理：缓存 + 路由）
    │
    ▼
PermissionCommonApi（RPC 接口）
    │
    ▼
PermissionServiceImpl（实际权限判断）
```

源码位置：`SecurityFrameworkServiceImpl.java`

```java
public class SecurityFrameworkServiceImpl implements SecurityFrameworkService {
    private final PermissionCommonApi permissionApi;

    // Guava LoadingCache — 缓存代理
    private final LoadingCache<KeyValue<Long, List<String>>, Boolean> hasAnyPermissionsCache = buildCache(
            Duration.ofMinutes(1L),
            new CacheLoader<>() {
                @Override
                public Boolean load(KeyValue<Long, List<String>> key) {
                    return permissionApi.hasAnyPermissions(key.getKey(),
                            key.getValue().toArray(new String[0])).getCheckedData();
                }
            });

    @Override
    public boolean hasAnyPermissions(String... permissions) {
        if (skipPermissionCheck()) return true;  // 跨租户跳过
        Long userId = getLoginUserId();
        if (userId == null) return false;
        return hasAnyPermissionsCache.get(new KeyValue<>(userId, Arrays.asList(permissions)));
    }
}
```

### 代理的三层职责

| 层次 | 职责 | 实现方式 |
|------|------|---------|
| 路由代理 | 跨租户场景跳过校验 | `skipPermissionCheck()` 判断 |
| 空值代理 | 未登录用户直接拒绝 | `userId == null` 判断 |
| 缓存代理 | 1 分钟内相同请求不重复调用 RPC | Guava `LoadingCache` |

### 设计意图

- **性能优化**：权限校验是每个请求必经之路，缓存可以显著减少 RPC 调用
- **透明调用**：`@PreAuthorize("@ss.hasPermission('system:user:list')")` 中的 `@ss` 是 SpEL Bean，调用方完全不知道背后有缓存
- **防御性编程**：未登录、跨租户等边界情况在代理层统一处理

### 可迁移经验

任何高频调用的远程服务都可以用缓存代理包装：

- 电商系统中的会员等级查询：每次请求都要判断会员等级以计算折扣
- 商品价格查询：秒杀场景下价格信息需要高频读取
- 关键原则：**缓存时间要短**（本项目用 1 分钟），避免数据变更后长时间不生效

## 4. 观察者模式（Observer Pattern）

### 模式识别

当角色或菜单被删除时，需要级联清理关联数据。项目通过直接方法调用实现了一个轻量的"观察者"：

```java
// RoleServiceImpl.deleteRole() — 角色删除时通知权限服务
public void deleteRole(Long id) {
    RoleDO role = validateRoleForUpdate(id);
    roleMapper.deleteById(id);
    permissionService.processRoleDeleted(id);  // 观察者：清理关联数据
}

// MenuServiceImpl.deleteMenu() — 菜单删除时通知权限服务
public void deleteMenu(Long id) {
    // ... 校验 ...
    menuMapper.deleteById(id);
    permissionService.processMenuDeleted(id);  // 观察者：清理关联数据
}
```

被调用的清理方法：

```java
// PermissionServiceImpl — 级联清理
public void processRoleDeleted(Long roleId) {
    userRoleMapper.deleteListByRoleId(roleId);      // 删除用户-角色关联
    roleMenuMapper.deleteListByRoleId(roleId);      // 删除角色-菜单关联
}

public void processMenuDeleted(Long menuId) {
    roleMenuMapper.deleteListByMenuId(menuId);      // 删除角色-菜单关联
}
```

### 缓存驱逐也是观察者

数据变更时清除缓存，本质上也是一种观察者反应：

```java
// 角色更新时驱逐缓存
@CacheEvict(value = RedisKeyConstants.ROLE, key = "#updateReqVO.id")
public void updateRole(RoleSaveReqVO updateReqVO) { ... }

// 角色删除时驱逐缓存
@CacheEvict(value = RedisKeyConstants.ROLE, key = "#id")
public void deleteRole(Long id) { ... }

// 分配角色菜单时清空所有相关缓存
@Caching(evict = {
    @CacheEvict(value = RedisKeyConstants.MENU_ROLE_ID_LIST, allEntries = true),
    @CacheEvict(value = RedisKeyConstants.PERMISSION_MENU_ID_LIST, allEntries = true)
})
public void assignRoleMenu(Long roleId, Set<Long> menuIds) { ... }
```

### 设计意图

- **数据完整性**：删除角色时必须清理 UserRole 和 RoleMenu，否则会产生孤儿数据
- **缓存一致性**：数据变更时驱逐缓存，避免读到过期数据
- **轻量实现**：没有引入事件总线（如 Spring Event），直接方法调用，简单可控

### 何时该用事件总线

当前的直接调用方式适合同模块内的级联操作。如果跨模块需要响应（如"角色删除时通知审计模块记录日志"），则应使用 Spring Event 或消息队列，避免模块间直接依赖。

## 5. 缓存旁路模式（Cache-Aside Pattern）

### 模式识别

权限组件大量使用 Spring Cache 的 `@Cacheable` 和 `@CacheEvict`，实现经典的缓存旁路模式：

| 操作 | 注解 | 缓存 Key | 说明 |
|------|------|---------|------|
| 读用户角色 | `@Cacheable` | `USER_ROLE_ID_LIST::{userId}` | 优先读缓存，未命中则查 DB |
| 读菜单权限 | `@Cacheable` | `MENU_ROLE_ID_LIST::{menuId}` | 优先读缓存，未命中则查 DB |
| 读角色信息 | `@Cacheable` | `ROLE::{id}` | 优先读缓存，未命中则查 DB |
| 写用户角色 | `@CacheEvict` | `USER_ROLE_ID_LIST::{userId}` | 写 DB 后清缓存 |
| 写角色菜单 | `@CacheEvict` | `MENU_ROLE_ID_LIST` (allEntries) | 写 DB 后清所有缓存 |
| 写角色信息 | `@CacheEvict` | `ROLE::{id}` | 写 DB 后清缓存 |

### 读写流程

```
读操作：
  1. 先查 Redis 缓存
  2. 缓存命中 → 直接返回
  3. 缓存未命中 → 查 MySQL → 写入 Redis → 返回

写操作：
  1. 更新 MySQL
  2. 删除 Redis 缓存（而非更新缓存）
  3. 下次读取时重新加载
```

### 设计意图

- **为什么删除缓存而不是更新缓存**：更新缓存可能出现并发写入导致缓存与 DB 不一致。删除缓存更安全，下次读取时自然重建
- **为什么用 `allEntries = true`**：分配角色菜单时，一个操作可能影响多个缓存 Key，逐个删除不如批量清空简单可靠
- **Guava + Redis 双层缓存**：`SecurityFrameworkServiceImpl` 中的 Guava LoadingCache 是进程内缓存，Redis 是分布式缓存，形成两级防线

### 可迁移经验

电商系统中适合使用缓存旁路的场景：

- **商品分类树**：很少变更，读取频繁，适合 `@Cacheable`
- **用户购物车**：频繁变更，每次变更后 `@CacheEvict`
- **热门商品库存**：需要更复杂的缓存策略（如预扣减），不适合简单的 Cache-Aside

---

## 总结

| 设计模式 | 在本组件中的体现 | 核心收益 |
|---------|----------------|---------|
| ACL 模式 | UserRoleDO + RoleMenuDO | 主体与资源解耦，批量授权 |
| 组合模式 | MenuDO 的 parentId 树形结构 | 统一数据结构，递归处理 |
| 代理模式 | SecurityFrameworkServiceImpl | 缓存加速 + 边界防御 |
| 观察者模式 | processRoleDeleted / processMenuDeleted | 级联清理，数据完整性 |
| 缓存旁路 | @Cacheable + @CacheEvict | 高频读取性能优化 |

这五种模式在权限组件中形成了完整的协作链：**ACL 模式**定义权限关系 → **组合模式**组织菜单树 → **缓存旁路**加速读取 → **代理模式**统一入口 → **观察者模式**保证数据一致性。理解这个协作链，就能理解为什么权限系统要这样设计。
