# 04-核心设计：登录流程与安全保障

## 场景引入

小王在做一个电商后台管理系统。产品经理说："用户要能登录，登录后能看到自己的权限菜单。"听起来很简单，但小王很快发现一连串问题：

- 登录时要不要验证码？要，防暴力破解
- 登录失败要不要记录？要，安全审计需要
- 登录成功返回什么？Token，不是 Session
- 用户被禁用了还能登录吗？不能，但已经登录的怎么办？

本章从这个场景出发，讲清登录流程的完整设计。

## 登录流程全景

```
用户提交账号密码
    │
    ▼

│  2. 身份认证      │

│  └─ 检查状态     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  3. 创建令牌      │
│  OAuth2Token     │
│  ├─ 创建刷新令牌  │

│  └─ 双写存储     │

         ▼

└────────┬────────┘
         │
         ▼
    返回 Token 响应
```

## 设计决策一：验证码策略

### 问题

登录接口是公开的（`@PermitAll`），任何人都可以调用。如果没有保护，攻击者可以用脚本暴力破解密码。

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| 图形验证码 | 简单、成熟 | 用户体验差 |
| 滑块验证码 | 体验好、防机器 | 实现复杂 |
| 无验证码 + 限流 | 体验最好 | 防护弱 |
| 可配置验证码 | 灵活 | 多一层配置 |

### 本项目的选择

```java
@Value("${yudao.captcha.enable:true}")
private Boolean captchaEnable;
```

验证码默认开启，但可以通过配置关闭。这是一个务实的设计：

- **生产环境**：开启验证码，防止暴力破解
- **开发/测试环境**：关闭验证码，方便调试
- **内网环境**：如果已有 WAF/限流，可以关闭验证码减少摩擦

验证码本身不直接校验，而是委托给 `CaptchaService`（可插拔实现），通过 `captchaVerification` 字符串传递验证结果。这样可以灵活对接不同的验证码提供商（极验、AJ-Captcha 等）。

## 设计决策二：身份认证的三层校验

```java
public AdminUserDO authenticate(String username, String password) {
    // 第一层：用户是否存在
    AdminUserDO user = userService.getUserByUsername(username);
    if (user == null) {
        throw exception(AUTH_LOGIN_BAD_CREDENTIALS);
    }
    // 第二层：密码是否正确
    if (!userService.isPasswordMatch(password, user.getPassword())) {
        throw exception(AUTH_LOGIN_BAD_CREDENTIALS);
    }
    // 第三层：账号是否启用
    if (CommonStatusEnum.isDisable(user.getStatus())) {
        throw exception(AUTH_LOGIN_USER_DISABLED);
    }
    return user;
}
```

### 为什么"用户不存在"和"密码错误"返回相同错误码？

这是一个安全设计。如果分别返回"用户不存在"和"密码错误"，攻击者可以：

1. 先用"用户不存在"枚举出有效用户名
2. 再针对有效用户暴力破解密码

统一返回 `AUTH_LOGIN_BAD_CREDENTIALS`，攻击者无法区分是哪种失败，大大增加了攻击成本。

### 为什么先查用户再验密码？

虽然 `getUserByUsername` + `isPasswordMatch` 看起来是两步，但实际上：

```java
// AdminUserServiceImpl
public boolean isPasswordMatch(String rawPassword, String encodedPassword) {
    return passwordEncoder.matches(rawPassword, encodedPassword);
}
```

BCrypt 校验本身就需要读取数据库中的 encoded password，所以先查用户是必要的。如果用 JWT 的 sub claim 直接存密码哈希（不推荐），才可能避免这次查询。

## 设计决策三：登录成功后的令牌创建

```java
private AuthLoginRespVO createTokenAfterLoginSuccess(
        Long userId, String username, LoginLogTypeEnum logType) {
    // 1. 记录登录日志
    createLoginLog(userId, username, logType, LoginResultEnum.SUCCESS);
    // 2. 创建访问令牌
    OAuth2AccessTokenDO accessTokenDO = oauth2TokenService.createAccessToken(
            userId, getUserType().getValue(),
            OAuth2ClientConstants.CLIENT_ID_DEFAULT, null);
    // 3. 构建返回结果
    return BeanUtils.toBean(accessTokenDO, AuthLoginRespVO.class);
}
```

### 为什么登录日志和令牌创建放在一起？

因为它们是"登录成功"这个事件的两个侧面：

- **令牌**：证明用户已认证，后续请求用它来标识身份
- **日志**：记录谁在什么时候从哪里登录了，用于安全审计

如果分开处理，可能出现"令牌创建了但日志没记录"的不一致状态。

### 为什么使用 CLIENT_ID_DEFAULT？

```java
public interface OAuth2ClientConstants {
    String CLIENT_ID_DEFAULT = "default";
}
```

这是一个设计简化。在标准 OAuth2 中，不同客户端（Web、App、小程序）有不同的 clientId。本项目为管理后台统一使用 `default` 客户端，因为：

1. 管理后台只有一个客户端类型
2. 简化了客户端管理的复杂度
3. 如果未来需要区分客户端，可以扩展

## 设计决策四：多登录方式的统一

本项目支持四种登录方式：

```
┌─────────────────────────────────────────────────┐
│              登录方式                              │
├─────────────┬─────────────┬──────────┬──────────┤
│  账号密码    │  短信验证码   │  社交登录  │  注册     │
│  login()    │  smsLogin() │ social() │ register │
└──────┬──────┴──────┬──────┴────┬─────┴────┬─────┘
       │             │           │          │
       └─────────────┴───────────┴──────────┘
                        │
                        ▼
            createTokenAfterLoginSuccess()
                        │
                        ▼
               返回 AuthLoginRespVO
```

所有登录方式最终都汇聚到 `createTokenAfterLoginSuccess()`，这个方法做了两件事：

1. 记录登录日志（不同登录方式用不同的 `LoginLogTypeEnum`）
2. 创建 OAuth2 令牌

### 为什么这样设计？

**开闭原则**：添加新的登录方式（如 LDAP、SAML），只需要：

1. 在 `AuthController` 添加新端点
2. 在 `AdminAuthService` 添加新方法
3. 最终调用 `createTokenAfterLoginSuccess()`

不需要修改令牌创建逻辑、日志记录逻辑。

## 登出设计

```java
public void logout(String token, Integer logType) {
    OAuth2AccessTokenDO accessTokenDO = oauth2TokenService.removeAccessToken(token);
    if (accessTokenDO == null) {
        return; // token 不存在，可能已过期
    }
    createLogoutLog(accessTokenDO.getUserId(), accessTokenDO.getUserType(), logType);
}
```

### 为什么登出要删除 Token 而不是标记过期？

两种方案对比：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 删除 Token | 立即生效，释放存储 | 需要 Redis + MySQL 双删 |
| 标记过期 | 不需要删除操作 | 需要额外字段，查询时判断 |

本项目选择删除，因为：

1. 用户期望登出后立即失效
2. 不需要在每次 Token 校验时增加额外判断
3. Redis 删除操作是 O(1)，性能开销极小

## 刷新令牌设计

```java
public AuthLoginRespVO refreshToken(String refreshToken) {
    OAuth2AccessTokenDO accessTokenDO = oauth2TokenService
            .refreshAccessToken(refreshToken, OAuth2ClientConstants.CLIENT_ID_DEFAULT);
    return BeanUtils.toBean(accessTokenDO, AuthLoginRespVO.class);
}
```

### 刷新流程

```
客户端携带 refreshToken 请求
    │
    ▼
查找 refreshToken 对应的记录
    │
    ▼
校验 clientId 匹配
    │
    ▼
删除旧的 accessToken（Redis + MySQL）
    │
    ▼
检查 refreshToken 是否过期
    │
    ▼
创建新的 accessToken
    │
    ▼
返回新的 Token 信息
```

### 为什么刷新时要删除旧的 AccessToken？

这是"令牌轮换"（Token Rotation）策略：

1. 每次刷新，旧的 AccessToken 立即失效
2. 如果攻击者窃取了旧的 AccessToken，它在刷新后就无法使用
3. 同时创建新的 AccessToken，用户无感知

### 特殊处理：refreshToken 也能当 accessToken 用

```java
// OAuth2TokenServiceImpl.getAccessToken()
// 特殊：从 MySQL 中获取刷新令牌。原因：解决部分场景不方便刷新访问令牌场景
OAuth2RefreshTokenDO refreshTokenDO = oauth2RefreshTokenMapper.selectByRefreshToken(accessToken);
```

这是一个务实的设计。有些第三方组件（如积木报表、WebSocket）只能传递一个 token，无法区分 access token 和 refresh token。允许 refresh token 作为 access token 使用，解决了这些集成场景的问题。

## 重置密码设计

```java
public void resetPassword(AuthResetPasswordReqVO reqVO) {
    // 1. 校验手机验证码
    smsCodeApi.useSmsCode(...).checkError();
    // 2. 更新密码
    userService.updateUserPassword(userByMobile.getId(), reqVO.getPassword());
}
```

### 为什么重置密码需要短信验证码？

这是"证明你是你"的二次验证。只有能接收手机验证码的人，才有权限重置密码。防止：

- 攻击者通过 CSRF 攻击修改他人密码
- 内部人员恶意修改他人密码

## 思考题

1. 如果要支持"记住我"功能，应该如何设计？是延长 AccessToken 的过期时间，还是使用单独的 RememberMe Token？
2. 本项目登录失败时会记录日志，但没有"连续失败 N 次锁定账号"的机制。如果要实现，应该在哪里加？加在 `authenticate()` 方法中还是单独的限流组件？
3. 为什么 `logout()` 方法中，如果 `removeAccessToken` 返回 null（token 不存在），直接返回而不抛异常？从用户体验角度考虑这样设计合理吗？
