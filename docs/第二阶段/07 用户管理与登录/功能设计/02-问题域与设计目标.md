# 问题域与设计目标

## 场景引入

假设你正在为一个电商平台开发后台管理系统。系统有几十个运营人员，他们需要登录系统来管理商品、处理订单、查看报表。

最开始，你可能觉得"登录"这件事很简单——用户输入账号密码，服务器验证通过后记住"这个人已经登录了"。但当你真正开始设计时，会发现一连串问题：

- "记住登录状态"是怎么实现的？
- 服务器怎么在第二次请求时知道"这个人已经登录了"？
- 如果有 10 台服务器，用户的登录状态存在哪台机器上？
- 用户点了"退出登录"，怎么让令牌立刻失效？
- 如果用户的令牌被泄露了，怎么紧急吊销？

这些问题的本质是：**在无状态的 HTTP 协议上，如何可靠地维护用户的身份状态？**

## 认证的本质问题

HTTP 是无状态协议——每个请求都是独立的，服务器不会自动记住"你是谁"。但业务场景要求有状态：用户登录一次后，后续请求应该自动识别身份，不需要每次都输密码。

所以认证系统的核心问题是：**在无状态协议上建立有状态的会话。**

解决方案只有一个：**令牌（Token）**。用户登录成功后，服务器发放一个令牌；后续请求携带这个令牌，服务器通过令牌识别身份。

但"怎么发令牌"和"怎么管令牌"，不同的方案差异巨大。

## 三种方案对比

### 方案一：Session + Cookie

这是最传统的方案，几乎所有 Java Web 教程的第一课。

```
浏览器                         服务器
  |                               |
  |  POST /login                  |
  |  { username, password }       |
  |------------------------------>|
  |                               | 1. 验证用户名密码
  |                               | 2. 创建 Session
  |                               | 3. Session 中存储 {userId: 1}
  |                               | 4. 生成 SessionId: "abc123"
  |                               |
  |  200 OK                       |
  |  Set-Cookie: JSESSIONID=abc123|
  |<------------------------------|
  |                               |
  |  GET /api/orders              |
  |  Cookie: JSESSIONID=abc123    |
  |------------------------------>|
  |                               | 1. 根据 SessionId 查找 Session
  |                               | 2. 从 Session 中取出 userId=1
  |                               | 3. 处理业务
  |  200 OK [{orderId: 1}, ...]   |
  |<------------------------------|
```

**优点：**
- 实现简单，Spring Boot 开箱即用
- 服务端完全掌控会话生命周期
- Session 中可以存储任意数据

**致命问题：**

```
  用户请求 --> 负载均衡器 --> 服务器A (Session 在这)
  用户请求 --> 负载均衡器 --> 服务器B (找不到 Session!)
```

Session 存储在单台服务器的内存中。当应用扩展到多台服务器时，会话粘滞（Session Sticky）或集中式 Session 存储就成了必须解决的问题。虽然可以用 Spring Session + Redis 解决，但这本质上已经把 Session 方案改造成了"集中式令牌存储"方案——和后面要讲的方案三殊途同归。

### 方案二：JWT（JSON Web Token）

JWT 是近年来最流行的方案。它的核心思想是：**把用户信息直接编码在令牌中，服务器不需要存储任何东西。**

```
JWT 的结构：
  Header.Payload.Signature

  Header:   { "alg": "HS256", "typ": "JWT" }
  Payload:  { "userId": 1, "role": "admin", "exp": 1700000000 }
  Signature: HMAC-SHA256(base64(header) + "." + base64(payload), secret)

  最终令牌：eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjF9.abc123...
```

验证过程：

```
浏览器                         服务器
  |                               |
  |  GET /api/orders              |
  |  Authorization: Bearer eyJ... |
  |------------------------------>|
  |                               | 1. 分离 Header、Payload、Signature
  |                               | 2. 用密钥验证 Signature
  |                               | 3. 检查 exp 是否过期
  |                               | 4. 从 Payload 中取出 userId=1
  |                               |    (不需要查数据库/Redis!)
  |  200 OK                       |
  |<------------------------------|
```

**优点：**
- 无状态，服务器不需要存储令牌，天然支持水平扩展
- 性能好，验证只需要一次 HMAC 计算
- 跨服务传递方便，令牌中自带用户信息

**致命问题：**

JWT 一旦签发，就无法在服务端主动失效。假设一个场景：

> 运营人员小王的账号被盗了。管理员发现后立刻禁用了小王的账号。但小王之前登录获取的 JWT 还没过期（比如有效期 2 小时），盗号者仍然可以用这个 JWT 访问系统 2 小时。

这就是 JWT 的"签发即失控"问题。虽然可以通过黑名单（Blacklist）机制缓解，但一旦引入黑名单，JWT 的"无状态"优势就消失了——你还是要查一次存储。

### 方案三：自定义 UUID Token + Redis（本项目的选择）

本项目选择了第三条路：**用随机 UUID 作为令牌，存储在 Redis 中，同时持久化到 MySQL。**

```
浏览器                         服务器
  |                               |
  |  POST /login                  |
  |  { username, password }       |
  |------------------------------>|
  |                               | 1. 验证用户名密码
  |                               | 2. 生成 UUID 令牌
  |                               | 3. 令牌信息写入 Redis (快速查询)
  |                               | 4. 令牌信息写入 MySQL (持久化)
  |                               |
  |  200 OK                       |
  |  { accessToken: "a1b2c3...",  |
  |    refreshToken: "d4e5f6..." }|
  |<------------------------------|
  |                               |
  |  GET /api/orders              |
  |  Authorization: Bearer a1b2c3 |
  |------------------------------>|
  |                               | 1. 用 token 去 Redis 查
  |                               | 2. 找到 -> 读取 userId
  |                               | 3. 处理业务
  |  200 OK                       |
  |<------------------------------|
```

**优点：**
- 服务端掌控令牌生命周期，随时可以撤回
- Redis 查询延迟极低（亚毫秒级），性能接近 JWT
- 令牌本身不携带信息，安全性更好（不会泄露用户数据）
- MySQL 双写提供审计能力和故障兜底

**缺点：**
- 需要维护 Redis 存储，增加运维复杂度
- 令牌查询引入了网络开销（虽然很小）

## 三种方案对比表

| 维度 | Session + Cookie | JWT | UUID + Redis（本项目） |
|------|-----------------|-----|----------------------|
| **状态存储** | 服务端内存/集中存储 | 无（客户端持有） | Redis + MySQL |
| **水平扩展** | 需要 Session 共享 | 天然支持 | 需要 Redis 集群 |
| **令牌撤回** | 删除 Session 即可 | 无法直接撤回 | 删除 Redis 记录即可 |
| **验证性能** | 内存查询，极快 | 本地计算，极快 | Redis 查询，很快 |
| **安全性** | SessionId 无意义 | Payload 可解码 | 令牌无意义 |
| **审计能力** | 弱 | 无 | MySQL 提供完整审计 |
| **跨域支持** | Cookie 受域限制 | Header 传递，无限制 | Header 传递，无限制 |
| **运维复杂度** | 低 | 最低 | 中（需维护 Redis） |

## 为什么选择 UUID + Redis

本项目选择 UUID + Redis 方案，核心原因有三个：

### 1. 可撤回性

对于后台管理系统，令牌撤回是刚需：

- 员工离职，需要立刻禁用其所有会话
- 发现账号被盗，需要紧急吊销令牌
- 管理员强制踢人下线
- 修改权限后，需要旧令牌失效

UUID + Redis 方案天然支持这些场景——只需删除 Redis 中的记录即可。JWT 做不到这一点（除非引入黑名单，但那就失去了 JWT 的核心优势）。

### 2. 可观测性

通过 Redis 和 MySQL，管理员可以清楚地看到：

- 当前有多少活跃会话
- 每个用户的登录时间和过期时间
- 每个令牌对应的客户端信息

这对于一个后台管理系统来说非常重要——你需要知道"谁在线"。

### 3. 实现简单

相比 JWT 需要处理密钥管理、算法选择、Payload 设计等复杂问题，UUID + Redis 方案的实现更直接：

```java
// 生成令牌
String accessToken = IdUtil.fastSimpleUUID();

// 存储令牌
redisTemplate.set("oauth2_access_token:" + accessToken, tokenInfo, expireTime, TimeUnit.SECONDS);

// 验证令牌
OAuth2AccessTokenDO token = redisTemplate.get("oauth2_access_token:" + accessToken);
if (token != null && !isExpired(token)) {
    return token; // 有效
}
```

没有签名算法的选择困难，没有 Payload 结构的设计纠结，也没有密钥轮换的运维负担。

## 设计目标

基于以上分析，本组件的设计目标如下：

### 目标一：无状态验证（通过 Redis）

虽然系统是有状态的（需要 Redis），但从应用服务器的角度看，验证过程是"无状态"的——任何一台应用服务器都能独立完成令牌验证，不需要会话粘滞。

```
应用服务器A ──┐
              ├──> Redis ──> 查到令牌信息 ──> 验证通过
应用服务器B ──┘
```

### 目标二：双写保障（Redis + MySQL）

```
  创建令牌时：
  +----------+     +-------+
  | 创建令牌  | --> | Redis |  <-- 快速查询，自动过期
  +----------+     +-------+
        |
        v
  +----------+     +-------+
  | 创建令牌  | --> | MySQL |  <-- 持久化存储，审计查询
  +----------+     +-------+

  查询令牌时：
  +----------+     +-------+
  | 查询令牌  | --> | Redis |  <-- 命中？直接返回
  +----------+     +-------+
        |
        | 未命中
        v
  +----------+     +-------+
  | 查询令牌  | --> | MySQL |  <-- 兜底查询，命中后回写 Redis
  +----------+     +-------+
```

Redis 承担日常的高频查询，MySQL 提供持久化保障和审计能力。当 Redis 故障或数据丢失时，MySQL 可以兜底——`OAuth2TokenServiceImpl.getAccessToken()` 方法中体现了这个设计：

```java
// 优先从 Redis 中获取
OAuth2AccessTokenDO accessTokenDO = oauth2AccessTokenRedisDAO.get(accessToken);
if (accessTokenDO != null) {
    return accessTokenDO;
}
// 获取不到，从 MySQL 中获取访问令牌
accessTokenDO = oauth2AccessTokenMapper.selectByAccessToken(accessToken);
// 如果在 MySQL 存在，则往 Redis 中写入（回补缓存）
if (accessTokenDO != null && !DateUtils.isExpired(accessTokenDO.getExpiresTime())) {
    oauth2AccessTokenRedisDAO.set(accessTokenDO);
}
```

### 目标三：多客户端支持

不同客户端（管理后台、移动端 App、小程序）可能需要不同的令牌策略：

- 管理后台：较长的访问令牌有效期（如 2 小时），较长的刷新令牌有效期（如 30 天）
- 移动端：更长的访问令牌有效期（如 7 天），减少用户重新登录的频率
- 第三方应用：支持授权码模式

这通过 `OAuth2ClientDO` 实现——每个客户端有独立的配置：

```java
public class OAuth2ClientDO {
    private String clientId;                    // 客户端标识
    private String secret;                      // 客户端密钥
    private Integer accessTokenValiditySeconds;  // 访问令牌有效期
    private Integer refreshTokenValiditySeconds; // 刷新令牌有效期
    private List<String> authorizedGrantTypes;   // 支持的授权模式
    // ...
}
```

### 目标四：完整的审计追踪

每次登录、登出、令牌刷新都会记录日志，包含：

- 用户 ID 和用户名
- 登录方式（账号密码、短信、社交）
- 登录结果（成功、密码错误、用户禁用、验证码错误）
- 客户端 IP 和 User-Agent
- 关联的 TraceId（用于链路追踪）

这些信息对于安全审计至关重要——当发生安全事件时，你需要知道"谁在什么时候从哪里以什么方式登录了系统"。

## 思考题

1. 如果你的电商平台日活用户只有 100 人，你会选择哪种令牌方案？如果日活是 1000 万呢？
2. 本项目的双写策略中，Redis 和 MySQL 的数据可能不一致（比如 Redis 写成功但 MySQL 写失败）。这种不一致会带来什么问题？项目是如何处理的？
3. JWT 的"签发即失控"问题，在哪些业务场景下是不可接受的？在哪些场景下可以接受？
4. 如果要支持"同一账号只允许在一个设备登录"（互踢），UUID + Redis 方案和 JWT 方案分别怎么实现？
