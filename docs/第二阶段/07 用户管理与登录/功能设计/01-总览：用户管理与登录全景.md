# 总览：用户管理与登录全景

## 本章定位

在第六个组件"认证鉴权框架"中，我们解决了一个核心问题：**请求进来后，怎么验证它携带的令牌是否合法？** 那里有一个关键的前提假设——令牌已经存在了。

但令牌从哪来？

这就是本组件要解决的问题。如果把认证鉴权框架比作酒店的"门禁系统"，那用户管理与登录就是酒店的"前台"——它负责办理入住（登录）、发放房卡（令牌）、办理退房（登出）、以及补办房卡（刷新令牌）。

## 这个组件做什么

一句话概括：**管理用户信息的全生命周期，并通过 OAuth2 协议完成身份认证与令牌发放。**

具体拆解为三个子系统：

```
+------------------------------------------------------------------+
|                    用户管理与登录                                  |
|                                                                    |
|  +------------------+  +-------------------+  +------------------+ |
|  | 用户管理子系统    |  | 令牌服务子系统     |  | 认证端点子系统    | |
|  |                  |  |                   |  |                  | |
|  | - 用户 CRUD      |  | - 创建令牌        |  | - 账号密码登录   | |
|  | - 密码加密存储    |  | - 刷新令牌        |  | - 短信验证码登录 | |
|  | - 用户状态管理    |  | - 校验令牌        |  | - 社交账号登录   | |
|  | - 部门/岗位关联   |  | - 撤回令牌        |  | - 用户注册       | |
|  |                  |  | - 清理过期令牌     |  | - 密码重置       | |
|  +------------------+  +-------------------+  +------------------+ |
|         |                       |                      |          |
|         v                       v                      v          |
|    AdminUserService      OAuth2TokenService       AdminAuthService |
|    AdminUserMapper       OAuth2AccessTokenMapper   AuthController  |
|                         OAuth2RefreshTokenMapper                   |
|                         OAuth2AccessTokenRedisDAO                  |
+------------------------------------------------------------------+
```

### 子系统一：用户管理

负责用户实体的增删改查。核心实体是 `AdminUserDO`，对应数据库表 `system_users`。

关键字段：

| 字段 | 说明 | 设计要点 |
|------|------|---------|
| `username` | 用户账号 | 登录标识，全局唯一 |
| `password` | 加密密码 | BCrypt 加密，不存储明文 |
| `nickname` | 用户昵称 | 展示用，不要求唯一 |
| `deptId` | 部门 ID | 关联组织架构，后续数据权限依赖此字段 |
| `postIds` | 岗位数组 | JSON 存储，支持一人多岗 |
| `status` | 账号状态 | 启用/禁用，禁用后无法登录 |
| `loginIp` / `loginDate` | 最后登录信息 | 审计追踪 |

这个子系统相对"朴素"——它本质是一个标准的 CRUD 模块。复杂度不在这里，而在下面两个子系统。

### 子系统二：OAuth2 令牌服务

这是整个组件的**核心引擎**。它不关心用户怎么进来的（密码、短信还是社交），只关心一件事：**给合法用户发放令牌，并管理令牌的完整生命周期。**

令牌的生命周期可以画成这样：

```
  用户认证成功
       |
       v
  +----------+     +----------+     +----------+     +----------+
  | 创建令牌  | --> | 使用令牌  | --> | 刷新令牌  | --> | 过期/撤回 |
  +----------+     +----------+     +----------+     +----------+
       |                 |                |                |
       v                 v                v                v
  写入 Redis        Redis 快速查询    旧令牌作废       Redis 自动过期
  写入 MySQL        MySQL 兜底查询    新令牌发放       MySQL 标记清理
```

核心类是 `OAuth2TokenServiceImpl`，它同时操作两个存储：

- **Redis**：快速查询，每次请求校验令牌时直接命中 Redis，避免数据库压力
- **MySQL**：持久化存储，用于审计和兜底（Redis 故障时仍可工作）

这种"双写"策略是本组件的一个重要设计决策，后面的章节会详细展开。

### 子系统三：认证端点

面向前端的 HTTP 接口层。`AuthController` 暴露了以下端点：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/system/auth/login` | POST | 账号密码登录 |
| `/system/auth/sms-login` | POST | 短信验证码登录 |
| `/system/auth/social-login` | POST | 社交账号登录 |
| `/system/auth/register` | POST | 用户注册 |
| `/system/auth/logout` | POST | 登出 |
| `/system/auth/refresh-token` | POST | 刷新令牌 |
| `/system/auth/send-sms-code` | POST | 发送短信验证码 |
| `/system/auth/reset-password` | POST | 重置密码 |
| `/system/auth/social-auth-redirect` | GET | 社交授权跳转 |
| `/system/auth/get-permission-info` | GET | 获取当前用户权限信息 |

注意所有认证端点都标注了 `@PermitAll`——它们不需要令牌就能访问（否则就成了"没有钥匙就进不了门，但钥匙在门里面"的死循环）。

## 与第六个组件的关系

这两个组件的关系可以用一个简单的比喻来理解：

```
  第六组件（认证鉴权框架）          第七组件（用户管理与登录）
  ========================          ========================

  "保安"                           "前台"

  职责：验证令牌                     职责：发放令牌
  输入：HTTP 请求中的 token          输入：用户名 + 密码
  输出：LoginUser（当前用户信息）     输出：token + refreshToken

  核心类：                          核心类：
  TokenAuthenticationFilter         AuthController
  SecurityFrameworkService          AdminAuthServiceImpl
                                    OAuth2TokenServiceImpl

  依赖关系：
  保安需要去前台查询令牌信息
  -------> 调用 OAuth2TokenService.checkAccessToken()
```

更技术化地说：

- **第六组件**的 `TokenAuthenticationFilter` 拦截每个请求，从 Header 中提取 token，然后调用 `OAuth2TokenService.checkAccessToken()` 验证
- **第七组件**的 `OAuth2TokenServiceImpl` 提供 `checkAccessToken()` 实现，从 Redis/MySQL 中查询令牌信息
- 第六组件拿到令牌信息后，构建 `LoginUser` 对象，注入到 Spring Security 上下文中

两者是**消费者和提供者**的关系：第六组件消费令牌校验能力，第七组件提供令牌管理能力。

## 一个登录请求的完整旅程

为了让读者对整个组件的协作有直观感受，我们跟踪一个完整的登录请求：

```
浏览器                          服务器
  |                               |
  |  POST /system/auth/login      |
  |  { username, password,        |
  |    captchaVerification }      |
  |------------------------------>|
  |                               |
  |                     +---------+---------+
  |                     | AuthController    |
  |                     |   login()         |
  |                     +--------+----------+
  |                              |
  |                     +--------v----------+
  |                     | AdminAuthServiceImpl|
  |                     |   login()          |
  |                     +--------+----------+
  |                              |
  |                     1. validateCaptcha()
  |                        校验图形验证码
  |                              |
  |                     2. authenticate()
  |                        2.1 getUserByUsername()
  |                        2.2 isPasswordMatch()  -- BCrypt 校验
  |                        2.3 检查用户状态
  |                              |
  |                     3. createTokenAfterLoginSuccess()
  |                        3.1 createLoginLog()  -- 记录登录日志
  |                        3.2 oauth2TokenService.createAccessToken()
  |                              |
  |                     +--------v----------+
  |                     | OAuth2TokenServiceImpl|
  |                     |   createAccessToken() |
  |                     +--------+----------+
  |                              |
  |                     3.2.1 validOAuthClientFromCache()
  |                           校验客户端配置
  |                              |
  |                     3.2.2 createOAuth2RefreshToken()
  |                           生成 refresh_token (UUID)
  |                           写入 MySQL
  |                              |
  |                     3.2.3 createOAuth2AccessToken()
  |                           生成 access_token (UUID)
  |                           写入 MySQL
  |                           写入 Redis (带 TTL)
  |                              |
  |  { accessToken,               |
  |    refreshToken,              |
  |    expiresTime }              |
  |<------------------------------|
```

整个过程涉及的核心调用链：

```
AuthController.login()
  -> AdminAuthServiceImpl.login()
    -> validateCaptcha()           // 验证码校验
    -> authenticate()              // 用户认证
      -> userService.getUserByUsername()
      -> userService.isPasswordMatch()
    -> createTokenAfterLoginSuccess()
      -> loginLogService.createLoginLog()  // 审计日志
      -> oauth2TokenService.createAccessToken()
        -> oauth2ClientService.validOAuthClientFromCache()
        -> createOAuth2RefreshToken()      // 写 MySQL
        -> createOAuth2AccessToken()       // 写 MySQL + Redis
```

```
                    | /system/auth/*    |

    +---------v---------+           +---------v---------+

```

## 小结

本章建立了用户管理与登录的全景认知：

1. **三个子系统**各司其职：用户管理负责"数据"，令牌服务负责"引擎"，认证端点负责"接口"
2. **与第六组件的关系**：第六组件是"保安"（验证令牌），第七组件是"前台"（发放令牌）
3. **双写策略**：Redis 提供快速校验，MySQL 提供持久化和审计
4. **一个登录请求**经过 验证码校验 → 用户认证 → 令牌创建 → 双写存储 的完整链路

接下来的章节，我们会逐一深入每个子系统的设计决策。
