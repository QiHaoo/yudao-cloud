# 06-配置设计与边界

## 配置项设计

本组件涉及的配置项：

```yaml
yudao:
  security:
    token-header: "Authorization"      # Token 的 Header 名称
    token-parameter: "access_token"    # Token 的参数名
    mock-enable: false                 # 是否开启模拟用户
    mock-secret: ""                    # 模拟用户的密钥
    mock-user-id: 1L                   # 模拟用户的编号
    mock-nickname: "芋道"              # 模拟用户的昵称
    mock-dept-id: 100L                 # 模拟用户的部门编号
    permit-all-urls: []                # 额外放行的 URL 列表
    password-encoder-length: 4         # BCrypt 加密强度

yudao:
  captcha:
    enable: true                       # 验证码开关
```

### 配置项说明

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| token-header | Authorization | HTTP Header 中传递 Token 的字段名 |
| token-parameter | access_token | URL 参数中传递 Token 的字段名 |
| mock-enable | false | 开启后，可通过 mock-secret 模拟任意用户 |
| permit-all-urls | [] | 额外放行的 URL，不经过认证 |
| password-encoder-length | 4 | BCrypt 的 strength，值越大越安全但越慢 |
| captcha.enable | true | 是否开启验证码 |

### mock-enable 的使用场景

```java
// TokenAuthenticationFilter 中
if (securityProperties.getMockEnable()) {
    String mockSecret = obtainAuthorization(request, "mock-secret");
    if (StrUtil.equals(mockSecret, securityProperties.getMockSecret())) {
        // 模拟用户登录
        LoginUser loginUser = new LoginUser()
            .setId(securityProperties.getMockUserId())
            .setNickname(securityProperties.getMockNickname());
        SecurityFrameworkUtils.setLoginUser(loginUser);
    }
}
```

这是一个危险但实用的功能：
- **开发环境**：方便调试，不需要真实登录
- **测试环境**：自动化测试可以模拟任意用户
- **生产环境**：必须关闭（`mock-enable: false`）

## OAuth2 客户端配置

OAuth2 客户端通过数据库配置，支持多客户端：

```sql
INSERT INTO system_oauth2_client (client_id, secret, access_token_validity, refresh_token_validity, authorized_grant_types)
VALUES ('default', 'default', 1800, 86400, 'password,social');
```

| 字段 | 说明 | 示例 |
|------|------|------|
| clientId | 客户端标识 | default, app, miniapp |
| secret | 客户端密钥 | default |
| accessTokenValiditySeconds | AccessToken 有效期（秒） | 1800（30分钟） |
| refreshTokenValiditySeconds | RefreshToken 有效期（秒） | 86400（7天） |
| authorizedGrantTypes | 允许的授权类型 | password, sms, social |

## 组件边界

### 本组件做了什么

1. **用户 CRUD**：创建、修改、删除、查询用户
2. **登录认证**：账号密码、短信、社交登录
3. **Token 管理**：创建、刷新、删除 Token
4. **注册与重置密码**：用户自助注册、密码重置

### 本组件不做什么

| 不做的事 | 交给谁 | 原因 |
|---------|--------|------|
| 权限校验 | 组件 6（认证鉴权框架） | 认证与授权分离 |
| 角色/菜单管理 | 组件 8（角色权限与菜单） | 职责分离 |
| 短信发送 | 短信模块 | 通过 SmsCodeApi RPC 调用 |
| 社交登录 | 社交模块 | 通过 SocialUserService 封装 |
| 部门管理 | 部门模块 | 用户通过 deptId 关联 |

### 与组件 6 的分工

```
组件 6（认证鉴权框架）：验证 Token 是否有效，提取用户信息
组件 7（用户管理与登录）：创建 Token，管理用户实体

请求流程：
客户端 → TokenAuthenticationFilter（组件6）→ Controller（组件7）
                                                    ↑
                                              这里需要登录用户的 ID
                                              由组件 6 从 Token 中提取
```

## 思考题

1. 如果要支持"单点登录"（SSO），即用户在一个系统登录后，其他系统自动登录，需要怎么改造？
2. 本项目 RefreshToken 的有效期默认是 7 天。如果用户 7 天内没有操作，RefreshToken 过期，需要重新登录。这个策略是否合理？有没有更好的方案？
3. mock-enable 配置如果误开到生产环境，会有什么安全风险？如何通过技术手段防止这种情况？
