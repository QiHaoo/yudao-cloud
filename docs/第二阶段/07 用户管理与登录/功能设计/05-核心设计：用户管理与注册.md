# 05-核心设计：用户管理与注册

## 场景引入

电商平台的管理员需要管理后台用户——创建运营账号、修改密码、分配部门、启用/禁用账号。同时，C 端用户需要自行注册成为会员。

本章讲清用户管理的设计，以及注册流程如何与登录体系打通。

## 用户实体设计

```
AdminUserDO
├── id              用户编号（主键）
├── username        用户账号（唯一）
├── password        加密后的密码（BCrypt）
├── nickname        用户昵称
├── deptId          部门 ID（关联部门模块）
├── postIds         岗位编号数组（JSON 存储）
├── email           邮箱
├── mobile          手机号
├── sex             性别
├── avatar          头像 URL
├── status          状态（启用/禁用）
├── loginIp         最后登录 IP
├── loginDate       最后登录时间
└── 继承 TenantBaseDO（tenantId, creator, createTime, updater, updateTime, deleted）
```

### 设计要点

**1. 密码存储：BCrypt 单向加密**

```java
// AdminUserDO 中的注释
/**
 * 加密后的密码
 * 因为目前使用 BCryptPasswordEncoder 加密器，所以无需自己处理 salt 盐
 */
private String password;
```

BCrypt 的特点：
- 每次加密结果不同（内置随机 salt）
- 不可逆（无法从密文推导明文）
- 可调节计算强度（cost factor）
- 验证时自动提取 salt 进行比对

**2. postIds 用 JSON 存储**

```java
@TableField(typeHandler = JacksonTypeHandler.class)
private Set<Long> postIds;
```

一个用户可以有多个岗位。用 JSON 数组存储而非关联表，是因为：
- 岗位关联查询频率低，不需要 JOIN
- 简化了表结构
- MyBatis-Plus 的 `JacksonTypeHandler` 自动处理序列化

**3. 继承 TenantBaseDO**

所有用户数据都带 `tenantId`，实现多租户数据隔离。同一个用户名在不同租户下可以重复。

### 创建用户

```java
// AdminUserServiceImpl
public Long createUser(UserCreateReqVO reqVO) {
    // 校验用户名唯一
    checkUsernameUnique(reqVO.getUsername(), null);
    // 校验手机号唯一
    checkMobileUnique(reqVO.getMobile(), null);
    // 校验邮箱唯一
    checkEmailUnique(reqVO.getEmail(), null);

    // 密码加密
    String encodePassword = passwordEncoder.encode(reqVO.getPassword());
    // 插入用户
    AdminUserDO user = BeanUtils.toBean(reqVO, AdminUserDO.class);
    user.setPassword(encodePassword);
    userMapper.insert(user);
    return user.getId();
}
```

关键点：
- 创建时就校验 username/mobile/email 的唯一性
- 密码使用 BCrypt 加密后存储
- 如果指定了角色，同时调用 `permissionService.assignUserRole()` 关联

### 修改密码

```java
public void updateUserPassword(Long userId, String password) {
    AdminUserDO user = getUser(userId);
    if (user == null) {
        throw exception(USER_NOT_EXISTS);
    }
    // 更新密码
    userMapper.updateById(new AdminUserDO().setId(userId)
            .setPassword(passwordEncoder.encode(password)));
}
```

密码修改后，不需要立即让已有的 Token 失效。如果需要强制下线，可以调用 `oauth2TokenService.removeAccessToken(userId, userType)`。

## 注册流程

```java
// AdminAuthServiceImpl
public AuthLoginRespVO register(AuthRegisterReqVO registerReqVO) {
    // 1. 校验验证码
    validateCaptcha(registerReqVO);
    // 2. 校验用户名是否已存在，创建用户
    Long userId = userService.registerUser(registerReqVO);
    // 3. 创建 Token 令牌，记录登录日志
    return createTokenAfterLoginSuccess(userId, registerReqVO.getUsername(), LoginLogTypeEnum.LOGIN_USERNAME);
}
```

### 注册即登录

注册完成后直接创建 Token，用户无需再次登录。这是一个常见的 UX 优化：

```
传统流程：注册 → 跳转登录页 → 登录 → 进入系统
优化流程：注册 → 直接进入系统
```

### 注册的安全考量

1. **验证码**：防止批量注册
2. **用户名唯一性**：防止重复注册
3. **密码强度**：由前端校验，后端可以配合 `@Size` 等注解

## 用户类型

本项目支持两种用户类型：

| 类型 | 枚举值 | 说明 | 对应实体 |
|------|--------|------|---------|
| 管理员 | ADMIN(1) | 管理后台用户 | AdminUserDO |
| 会员 | MEMBER(2) | C 端用户 | MemberUserDO |

`UserTypeEnum` 定义在 `yudao-common` 中，被 OAuth2 令牌、权限校验等多个模块引用。

### 为什么需要区分用户类型？

1. **不同的登录入口**：管理员走 `/system/auth/login`，会员走 `/app-api/member/auth/login`
2. **不同的权限体系**：管理员用 RBAC，会员可能用更简单的权限模型
3. **不同的数据隔离**：管理员按部门隔离，会员按个人隔离

## 思考题

1. 本项目用 `postIds`（JSON 数组）存储用户的岗位关联。如果需要"查询某个岗位下有哪些用户"，这种设计有什么问题？你会怎么改进？
2. 注册流程中，`validateCaptcha` 和 `registerUser` 之间没有事务。如果 `registerUser` 失败了，验证码已经被消费了。这是否是一个问题？如何改进？
3. 如果要支持"用户头像上传"功能，头像文件应该存在哪里？本地文件系统、OSS、还是数据库？每种方案的权衡是什么？
