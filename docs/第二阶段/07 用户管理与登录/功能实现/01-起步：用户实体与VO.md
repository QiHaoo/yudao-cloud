# 01 - 起步：用户实体与 VO

> **功能设计参考**：[02-问题域与设计目标](../功能设计/02-问题域与设计目标.md) 中"用户模型设计"一节。

本章从最底层的数据模型出发，梳理用户管理与登录功能涉及的所有实体和 VO。读者完成后，能清楚地知道：用户长什么样、登录请求长什么样、登录响应长什么样。

## 1. AdminUserDO：用户实体

### 1.1 继承体系

```
TenantBaseDO          ← 多租户基础字段（tenantId, creator, updater, ...）
  └── AdminUserDO     ← 用户特有字段
```

`AdminUserDO` 继承自 `TenantBaseDO`，自动获得租户隔离能力和审计字段（`creator`、`createTime`、`updater`、`updateTime`、`deleted`、`tenantId`）。这意味着：
- 每个用户天然属于某个租户，MyBatis-Plus 的租户拦截器会自动在 SQL 中追加 `WHERE tenant_id = ?` 条件
- 所有写操作自动记录操作人和时间

### 1.2 完整字段定义

> 源码路径：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/dal/dataobject/user/AdminUserDO.java`

```java
@TableName(value = "system_users", autoResultMap = true)
@KeySequence("system_users_seq") // Oracle/PostgreSQL/Kingbase/DB2/H2 主键自增
@Data
@EqualsAndHashCode(callSuper = true)
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class AdminUserDO extends TenantBaseDO {

    /**
     * 用户ID
     */
    @TableId
    private Long id;
    /**
     * 用户账号
     */
    private String username;
    /**
     * 加密后的密码
     *
     * 因为目前使用 BCryptPasswordEncoder 加密器，所以无需自己处理 salt 盐
     */
    private String password;
    /**
     * 用户昵称
     */
    private String nickname;
    /**
     * 备注
     */
    private String remark;
    /**
     * 部门 ID
     */
    private Long deptId;
    /**
     * 岗位编号数组
     */
    @TableField(typeHandler = JacksonTypeHandler.class)
    private Set<Long> postIds;
    /**
     * 用户邮箱
     */
    private String email;
    /**
     * 手机号码
     */
    private String mobile;
    /**
     * 用户性别
     *
     * 枚举类 SexEnum
     */
    private Integer sex;
    /**
     * 用户头像
     */
    private String avatar;
    /**
     * 帐号状态
     *
     * 枚举 CommonStatusEnum
     */
    private Integer status;
    /**
     * 最后登录IP
     */
    private String loginIp;
    /**
     * 最后登录时间
     */
    private LocalDateTime loginDate;

}
```

### 1.3 字段设计要点

| 关注点 | 说明 |
|--------|------|
| **密码存储** | 使用 BCrypt 单向哈希，数据库中存的是哈希值，不可逆。源码注释明确写了"无需自己处理 salt 盐"，因为 BCrypt 的 salt 已编码在哈希值中 |
| **postIds** | 用 `JacksonTypeHandler` 将 `Set<Long>` 序列化为 JSON 存入单个字段。这避免了额外的关联表，适合"一个用户关联少量岗位"的场景 |
| **deptId** | 直接存储部门 ID，不做外键约束。后续数据权限功能通过此字段实现部门级数据过滤 |
| **status** | 复用 `CommonStatusEnum`（0=开启，1=禁用），而非单独的用户状态枚举。设计原则：能复用的通用枚举就复用 |
| **loginIp / loginDate** | 登录成功后由 `AdminUserService.updateUserLogin()` 更新，用于安全审计（例如发现异地登录） |

### 1.4 对应数据库表

表名 `system_users`，核心 DDL 结构（简化版）：

```sql
CREATE TABLE system_users (
    id          BIGINT       NOT NULL AUTO_INCREMENT COMMENT '用户ID',
    username    VARCHAR(30)  NOT NULL                COMMENT '用户账号',
    password    VARCHAR(100) NOT NULL DEFAULT ''     COMMENT '密码',
    nickname    VARCHAR(30)  NOT NULL                COMMENT '用户昵称',
    remark      VARCHAR(500)                         COMMENT '备注',
    dept_id     BIGINT                                COMMENT '部门ID',
    post_ids    VARCHAR(512)                         COMMENT '岗位编号数组(JSON)',
    email       VARCHAR(50)                          COMMENT '邮箱',
    mobile      VARCHAR(11)                          COMMENT '手机号',
    sex         TINYINT                               COMMENT '性别',
    avatar      VARCHAR(512)                         COMMENT '头像',
    status      TINYINT      NOT NULL DEFAULT 0     COMMENT '帐号状态',
    login_ip    VARCHAR(50)                          COMMENT '最后登录IP',
    login_date  DATETIME                             COMMENT '最后登录时间',
    -- TenantBaseDO 继承字段
    tenant_id   BIGINT       NOT NULL DEFAULT 0     COMMENT '租户编号',
    creator     VARCHAR(64)  DEFAULT ''              COMMENT '创建者',
    create_time DATETIME     DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updater     VARCHAR(64)  DEFAULT ''              COMMENT '更新者',
    update_time DATETIME     DEFAULT CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted     BIT(1)       NOT NULL DEFAULT b'0'  COMMENT '是否删除',
    PRIMARY KEY (id),
    UNIQUE KEY uk_username (username, tenant_id)
) COMMENT '管理后台用户表';
```

注意唯一键 `uk_username (username, tenant_id)`——同一个租户下用户名不能重复，不同租户可以有同名用户。

## 2. VO 体系概览

用户管理与登录涉及的 VO 分布在 `controller/admin/auth/vo/` 包下，按职责分为三类：

```
vo/
├── CaptchaVerificationReqVO    ← 验证码基类（被多个 ReqVO 继承）
├── AuthLoginReqVO              ← 账号密码登录请求
├── AuthLoginRespVO             ← 登录成功响应（所有登录方式共用）
├── AuthSmsLoginReqVO           ← 短信验证码登录请求
├── AuthSmsSendReqVO            ← 发送短信验证码请求
├── AuthSocialLoginReqVO        ← 社交登录请求
├── AuthRegisterReqVO           ← 注册请求
├── AuthResetPasswordReqVO      ← 重置密码请求
├── AuthPermissionInfoRespVO    ← 权限信息响应（含嵌套 UserVO、MenuVO）
└── AuthMenuRespVO              ← 菜单响应
```

下面逐一讲解核心 VO。

---

## 3. CaptchaVerificationReqVO：验证码基类

> 源码路径：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/auth/vo/CaptchaVerificationReqVO.java`

```java
@Schema(description = "管理后台 - 验证码 Request VO")
@Data
public class CaptchaVerificationReqVO {

    @Schema(description = "验证码，验证码开启时，需要传递",
            requiredMode = Schema.RequiredMode.REQUIRED,
            example = "PfcH6mgr8tpXuMWFjvW6YVaqrswIuwmWI5dsVZSg7sGpWtDCUbHuDEXl3cFB1+VvCC/rAkSwK8Fad52FSuncVg==")
    @NotEmpty(message = "验证码不能为空", groups = CodeEnableGroup.class)
    private String captchaVerification;

    /**
     * 开启验证码的 Group
     */
    public interface CodeEnableGroup {
    }
}
```

**设计要点**：

- 这是一个**基类**，`AuthLoginReqVO`、`AuthRegisterReqVO`、`AuthSmsSendReqVO` 都继承它
- 使用 **JSR-303 分组校验**：`@NotEmpty` 只在 `CodeEnableGroup` 分组下生效。`AdminAuthServiceImpl` 中通过 `ValidationUtils.validate(validator, reqVO, CaptchaVerificationReqVO.CodeEnableGroup.class)` 手动触发这个分组的校验
- 这样做的好处：验证码是否开启由配置项 `yudao.captcha.enable` 控制。关闭时，根本不会触发校验；开启时，才走分组校验逻辑

**电商场景类比**：用户登录电商后台管理系统时，如果管理员开启了验证码功能，前端需要先完成滑块验证，然后把验证结果（`captchaVerification` 字段）随登录请求一起提交。

## 4. AuthLoginReqVO：账号密码登录请求

> 源码路径：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/auth/vo/AuthLoginReqVO.java`

```java
@Schema(description = "管理后台 - 账号密码登录 Request VO，如果登录并绑定社交用户，需要传递 social 开头的参数")
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class AuthLoginReqVO extends CaptchaVerificationReqVO {

    @Schema(description = "账号", requiredMode = Schema.RequiredMode.REQUIRED, example = "yudaoyuanma")
    @NotEmpty(message = "登录账号不能为空")
    @Length(min = 4, max = 30, message = "账号长度为 4-30 位")
    @Pattern(regexp = "^[a-zA-Z0-9]{4,30}$", message = "账号格式为数字以及字母")
    private String username;

    @Schema(description = "密码", requiredMode = Schema.RequiredMode.REQUIRED, example = "buzhidao")
    @NotEmpty(message = "密码不能为空")
    @Length(min = 4, max = 16, message = "密码长度为 4-16 位")
    private String password;

    // ========== 绑定社交登录时，需要传递如下参数 ==========

    @Schema(description = "社交平台的类型，参见 SocialTypeEnum 枚举值", requiredMode = Schema.RequiredMode.REQUIRED, example = "10")
    @InEnum(SocialTypeEnum.class)
    private Integer socialType;

    @Schema(description = "授权码", requiredMode = Schema.RequiredMode.REQUIRED, example = "1024")
    private String socialCode;

    @Schema(description = "state", requiredMode = Schema.RequiredMode.REQUIRED, example = "9b2ffbc1-7425-4155-9894-9d5c08541d62")
    private String socialState;

    @AssertTrue(message = "授权码不能为空")
    public boolean isSocialCodeValid() {
        return socialType == null || StrUtil.isNotEmpty(socialCode);
    }

    @AssertTrue(message = "授权 state 不能为空")
    public boolean isSocialState() {
        return socialType == null || StrUtil.isNotEmpty(socialState);
    }

}
```

**设计要点**：

| 关注点 | 说明 |
|--------|------|
| **继承 CaptchaVerificationReqVO** | 自动获得 `captchaVerification` 字段 |
| **username 校验** | 4-30 位，只允许字母和数字。用 `@Pattern` 做格式校验，防止注入攻击 |
| **password 校验** | 4-16 位，只做长度校验，不做格式限制（允许特殊字符） |
| **socialType / socialCode / socialState** | 用于"登录并绑定社交账号"的场景。如果 `socialType` 非空，说明用户想在登录的同时绑定微信/GitHub 等社交账号 |
| **isSocialCodeValid() / isSocialState()** | 自定义的 `@AssertTrue` 校验：当 `socialType` 不为空时，`socialCode` 和 `socialState` 也必须不为空。这是一种**条件校验**的经典写法 |

**电商场景**：电商后台管理员用账号密码登录，同时可以选择绑定自己的 GitHub 账号，方便下次直接用 GitHub 授权登录。

## 5. AuthLoginRespVO：登录成功响应

> 源码路径：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/auth/vo/AuthLoginRespVO.java`

```java
@Schema(description = "管理后台 - 登录 Response VO")
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class AuthLoginRespVO {

    @Schema(description = "用户编号", requiredMode = Schema.RequiredMode.REQUIRED, example = "1024")
    private Long userId;

    @Schema(description = "访问令牌", requiredMode = Schema.RequiredMode.REQUIRED, example = "happy")
    private String accessToken;

    @Schema(description = "刷新令牌", requiredMode = Schema.RequiredMode.REQUIRED, example = "nice")
    private String refreshToken;

    @Schema(description = "过期时间", requiredMode = Schema.RequiredMode.REQUIRED)
    private LocalDateTime expiresTime;

}
```

**设计要点**：

- **所有登录方式共用同一个响应 VO**：账号密码登录、短信登录、社交登录、注册、刷新令牌——全部返回 `AuthLoginRespVO`
- **前端拿到 `accessToken` 后**，存入 localStorage 或 Cookie，后续请求通过 Header 携带
- **`refreshToken`** 用于在 `accessToken` 过期后获取新的 `accessToken`，而不需要用户重新登录
- **`expiresTime`** 告诉前端令牌何时过期，前端可以提前做无感刷新

这个 VO 的数据来自 `OAuth2AccessTokenDO`，通过 `BeanUtils.toBean()` 自动映射。

## 6. AuthSmsLoginReqVO：短信登录请求

> 源码路径：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/auth/vo/AuthSmsLoginReqVO.java`

```java
@Schema(description = "管理后台 - 短信验证码的登录 Request VO")
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class AuthSmsLoginReqVO {

    @Schema(description = "手机号", requiredMode = Schema.RequiredMode.REQUIRED, example = "yudaoyuanma")
    @NotEmpty(message = "手机号不能为空")
    @Mobile
    private String mobile;

    @Schema(description = "短信验证码", requiredMode = Schema.RequiredMode.REQUIRED, example = "1024")
    @NotEmpty(message = "验证码不能为空")
    private String code;

}
```

**设计要点**：

- 短信登录不继承 `CaptchaVerificationReqVO`，因为短信本身就是一种验证码形式，不需要额外的图形验证码
- `@Mobile` 是项目自定义的手机号校验注解，校验手机号格式是否合法

## 7. AuthSocialLoginReqVO：社交登录请求

> 源码路径：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/auth/vo/AuthSocialLoginReqVO.java`

```java
@Schema(description = "管理后台 - 社交绑定登录 Request VO，使用 code 授权码 + 账号密码")
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class AuthSocialLoginReqVO {

    @Schema(description = "社交平台的类型，参见 UserSocialTypeEnum 枚举值",
            requiredMode = Schema.RequiredMode.REQUIRED, example = "10")
    @InEnum(SocialTypeEnum.class)
    @NotNull(message = "社交平台的类型不能为空")
    private Integer type;

    @Schema(description = "授权码", requiredMode = Schema.RequiredMode.REQUIRED, example = "1024")
    @NotEmpty(message = "授权码不能为空")
    private String code;

    @Schema(description = "state", requiredMode = Schema.RequiredMode.REQUIRED,
            example = "9b2ffbc1-7425-4155-9894-9d5c08541d62")
    @NotEmpty(message = "state 不能为空")
    private String state;

}
```

**设计要点**：

- `type` 用 `@InEnum(SocialTypeEnum.class)` 校验，确保传入的社交平台类型是合法的（如微信、GitHub、Gitee 等）
- `code` 是 OAuth2 授权码，前端从社交平台回调 URL 的 query 参数中获取
- `state` 是 OAuth2 防 CSRF 攻击的随机字符串，前端在发起授权前生成，回调时带回，需要与之前存储的值匹配

## 8. AuthPermissionInfoRespVO：权限信息响应

> 源码路径：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/auth/vo/AuthPermissionInfoRespVO.java`

```java
@Schema(description = "管理后台 - 登录用户的权限信息 Response VO，额外包括用户信息和角色列表")
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class AuthPermissionInfoRespVO {

    @Schema(description = "用户信息", requiredMode = Schema.RequiredMode.REQUIRED)
    private UserVO user;

    @Schema(description = "角色标识数组", requiredMode = Schema.RequiredMode.REQUIRED)
    private Set<String> roles;

    @Schema(description = "操作权限数组", requiredMode = Schema.RequiredMode.REQUIRED)
    private Set<String> permissions;

    @Schema(description = "菜单树", requiredMode = Schema.RequiredMode.REQUIRED)
    private List<MenuVO> menus;

    @Schema(description = "用户信息 VO")
    @Data
    @NoArgsConstructor
    @AllArgsConstructor
    @Builder
    public static class UserVO {

        @Schema(description = "用户编号", requiredMode = Schema.RequiredMode.REQUIRED, example = "1024")
        private Long id;

        @Schema(description = "用户昵称", requiredMode = Schema.RequiredMode.REQUIRED, example = "芋道源码")
        private String nickname;

        @Schema(description = "用户头像", requiredMode = Schema.RequiredMode.REQUIRED,
                example = "https://www.iocoder.cn/xx.jpg")
        private String avatar;

        @Schema(description = "部门编号", requiredMode = Schema.RequiredMode.REQUIRED, example = "2048")
        private Long deptId;

        @Schema(description = "用户账号", requiredMode = Schema.RequiredMode.REQUIRED, example = "yudao")
        private String username;

        @Schema(description = "用户邮箱", example = "yudao@iocoder.cn")
        private String email;

    }

    @Schema(description = "管理后台 - 登录用户的菜单信息 Response VO")
    @Data
    @NoArgsConstructor
    @AllArgsConstructor
    @Builder
    public static class MenuVO {

        @Schema(description = "菜单名称", requiredMode = Schema.RequiredMode.REQUIRED, example = "芋道")
        private Long id;

        @Schema(description = "父菜单 ID", requiredMode = Schema.RequiredMode.REQUIRED, example = "1024")
        private Long parentId;

        @Schema(description = "菜单名称", requiredMode = Schema.RequiredMode.REQUIRED, example = "芋道")
        private String name;

        @Schema(description = "路由地址,仅菜单类型为菜单或者目录时，才需要传", example = "post")
        private String path;

        @Schema(description = "组件路径,仅菜单类型为菜单时，才需要传", example = "system/post/index")
        private String component;

        @Schema(description = "组件名", example = "SystemUser")
        private String componentName;

        @Schema(description = "菜单图标,仅菜单类型为菜单或者目录时，才需要传", example = "/menu/list")
        private String icon;

        @Schema(description = "是否可见", requiredMode = Schema.RequiredMode.REQUIRED, example = "false")
        private Boolean visible;

        @Schema(description = "是否缓存", requiredMode = Schema.RequiredMode.REQUIRED, example = "false")
        private Boolean keepAlive;

        @Schema(description = "是否总是显示", example = "false")
        private Boolean alwaysShow;

        /**
         * 子路由
         */
        private List<MenuVO> children;

    }

}
```

**设计要点**：

- 这个 VO 是前端渲染**左侧菜单**和**按钮权限**的数据来源
- `roles` 是角色标识集合（如 `["admin", "shop_manager"]`），前端用它判断是否显示某些角色专属的 UI 元素
- `permissions` 是权限标识集合（如 `["system:user:create", "system:order:export"]`），前端用它控制按钮级别的显示/隐藏
- `menus` 是**已构建好的菜单树**，后端完成树形组装，前端直接渲染，不需要前端再做一次树构建
- `UserVO` 是内部静态类，只暴露安全字段（不暴露密码、手机号等敏感信息）

**电商场景**：管理员登录电商后台后，前端调用 `get-permission-info` 接口，拿到当前用户能看到的菜单（如"商品管理"、"订单管理"、"营销活动"）和操作权限（如"导出订单"、"修改商品价格"），然后动态渲染侧边栏和按钮。

## 9. 其他 VO 简述

### AuthSmsSendReqVO

继承 `CaptchaVerificationReqVO`，包含 `mobile`（手机号）和 `scene`（短信场景，如登录、重置密码）。发送短信前需要先校验图形验证码。

### AuthRegisterReqVO

继承 `CaptchaVerificationReqVO`，包含 `username`、`nickname`、`password`。注册前需要校验图形验证码。

### AuthResetPasswordReqVO

包含 `password`、`mobile`、`code`（短信验证码）。通过手机号 + 短信验证码确认身份后重置密码。

## 10. VO 与实体的映射关系

```
前端请求                          后端处理                          后端响应
─────────                        ─────────                        ─────────
AuthLoginReqVO ──────────→ AdminAuthServiceImpl.login()
                              ↓
                           AdminUserDO (从数据库查询)
                              ↓
                           OAuth2AccessTokenDO (创建令牌)
                              ↓
                           AuthLoginRespVO ←── BeanUtils.toBean()

AuthSmsLoginReqVO ───────→ AdminAuthServiceImpl.smsLogin()
                              ↓
                           AdminUserDO (按手机号查询)
                              ↓
                           OAuth2AccessTokenDO
                              ↓
                           AuthLoginRespVO

AuthSocialLoginReqVO ────→ AdminAuthServiceImpl.socialLogin()
                              ↓
                           SocialUserRespDTO (社交用户绑定关系)
                              ↓
                           AdminUserDO (按绑定的userId查询)
                              ↓
                           OAuth2AccessTokenDO
                              ↓
                           AuthLoginRespVO
```

所有登录路径最终都汇聚到 `createTokenAfterLoginSuccess()` 这个模板方法，统一创建令牌并返回 `AuthLoginRespVO`。

## 小结

| 组件 | 职责 | 核心字段 |
|------|------|---------|
| `AdminUserDO` | 用户实体，映射 `system_users` 表 | username, password(BCrypt), deptId, status |
| `CaptchaVerificationReqVO` | 验证码基类，被多个请求 VO 继承 | captchaVerification |
| `AuthLoginReqVO` | 账号密码登录请求 | username, password, socialType/Code/State |
| `AuthLoginRespVO` | 所有登录方式的统一响应 | userId, accessToken, refreshToken, expiresTime |
| `AuthSmsLoginReqVO` | 短信登录请求 | mobile, code |
| `AuthSocialLoginReqVO` | 社交登录请求 | type, code, state |
| `AuthPermissionInfoRespVO` | 权限信息响应（用户+角色+菜单树） | user, roles, permissions, menus |
