# 设计模式提炼：用户管理与登录

> 从用户管理与登录组件中识别出的设计模式，抽象为可迁移的通用知识。

---

## 1. 模板方法模式（Template Method Pattern）

### 模式识别

`AdminAuthServiceImpl` 中的 `createTokenAfterLoginSuccess()` 是一个典型的模板方法。它定义了"登录成功后"的固定流程：

```
登录成功 → 记录登录日志 → 创建访问令牌 → 构建返回结果
```

所有登录方式最终都收敛到这个模板：

| 登录入口 | 方法 | 调用模板的参数 |
|---------|------|--------------|
| 账号密码登录 | `login()` | `userId, username, LOGIN_USERNAME` |
| 手机验证码登录 | `smsLogin()` | `userId, mobile, LOGIN_MOBILE` |
| 社交登录 | `socialLogin()` | `userId, username, LOGIN_SOCIAL` |
| 用户注册 | `register()` | `userId, username, LOGIN_USERNAME` |

源码位置：`AdminAuthServiceImpl.java` 第 213-221 行

```java
private AuthLoginRespVO createTokenAfterLoginSuccess(Long userId, String username, LoginLogTypeEnum logType) {
    // 1. 插入登陆日志
    createLoginLog(userId, username, logType, LoginResultEnum.SUCCESS);
    // 2. 创建访问令牌
    OAuth2AccessTokenDO accessTokenDO = oauth2TokenService.createAccessToken(userId, getUserType().getValue(),
            OAuth2ClientConstants.CLIENT_ID_DEFAULT, null);
    // 3. 构建返回结果
    return BeanUtils.toBean(accessTokenDO, AuthLoginRespVO.class);
}
```

### 设计意图

- **消除重复**：四种登录方式共享"日志 + 令牌 + 响应"的后处理逻辑，避免代码复制
- **开放扩展**：新增登录方式（如扫码登录）只需一个新的入口方法，最后调用同一个模板
- **固定骨架**：日志记录和令牌创建的顺序不可更改，保证审计完整性

### 可迁移经验

当你发现多个操作的"后处理"步骤完全一致时，就适合提取模板方法。典型场景：

- 电商下单：`下单成功 → 扣减库存 → 发送通知 → 记录日志`
- 文件上传：`上传成功 → 记录审计 → 更新索引 → 触发回调`

## 2. 策略模式（Strategy Pattern）

### 模式识别

`OAuth2GrantServiceImpl` 实现了多种授权策略，每种策略有不同的认证逻辑，但最终都创建访问令牌：

| 授权类型 | 方法 | 认证逻辑 |
|---------|------|---------|
| 授权码模式 | `grantAuthorizationCodeForAccessToken()` | 验证授权码 + clientId + redirectUri |
| 密码模式 | `grantPassword()` | 验证用户名 + 密码 |
| 隐式授权 | `grantImplicit()` | 直接创建令牌 |
| 客户端凭证 | `grantClientCredentials()` | 仅验证 clientId |
| 刷新令牌 | `grantRefreshToken()` | 验证刷新令牌有效性 |

源码位置：`OAuth2GrantServiceImpl.java`

每种策略的认证过程完全不同，但最终都调用 `oauth2TokenService.createAccessToken()`。

### 设计意图

- **单一职责**：每种授权方式的认证逻辑独立封装，互不干扰
- **开闭原则**：新增授权方式（如 PKCE）只需添加新方法，不修改已有代码
- **统一出口**：所有策略最终收敛到令牌创建，保证安全性一致

### 与经典策略模式的差异

经典策略模式通常通过接口 + 多实现类 + 运行时选择来实现。本项目将多种策略放在同一个 Service 类中，通过不同的 Controller 方法路由到不同的策略方法。这是一种更轻量的做法——当策略数量固定且不频繁扩展时，无需引入接口抽象层。

### 可迁移经验

电商系统中的支付方式选择就是典型的策略场景：

- 微信支付：调用微信 SDK → 回调处理 → 更新订单状态
- 支付宝支付：调用支付宝 SDK → 回调处理 → 更新订单状态
- 余额支付：扣减余额 → 更新订单状态

共同的后半段（更新订单状态）可以提取为模板方法，前半段（调用不同支付 SDK）保持策略独立。

## 3. 仓储模式（Repository Pattern）

### 模式识别

Token 的存储被拆分为两个仓储，屏蔽了底层存储细节：

| 仓储 | 位置 | 职责 |
|------|------|------|
| `OAuth2AccessTokenRedisDAO` | `dal/redis/oauth2/` | Redis 读写，快速查询 |
| `OAuth2AccessTokenMapper` | `dal/mysql/oauth2/` | MySQL 持久化，审计追溯 |

`OAuth2TokenServiceImpl` 通过这两个仓储操作 Token，不直接操作 Redis 或 MySQL。

### 读取流程中的仓储协作

```
getAccessToken(accessToken)
    │
    ├─ 1. 先查 Redis（OAuth2AccessTokenRedisDAO.get）
    │     └─ 命中 → 直接返回
    │
    ├─ 2. Redis 未命中，查 MySQL（OAuth2AccessTokenMapper.selectByAccessToken）
    │     └─ 也未命中 → 尝试当刷新令牌查
    │
    └─ 3. MySQL 命中 → 回写 Redis（OAuth2AccessTokenRedisDAO.set）→ 返回
```

源码位置：`OAuth2TokenServiceImpl.java` 第 104-128 行

### 设计意图

- **存储透明**：Service 层只关心"存 Token"和"取 Token"，不关心用 Redis 还是 MySQL
- **职责分离**：Redis DAO 只管缓存操作，MySQL Mapper 只管持久化操作
- **可替换性**：如果未来要换缓存方案（如 Memcached），只需替换 RedisDAO 实现

### 可迁移经验

电商系统中的商品库存可以采用同样的模式：

- `StockRedisDAO`：Redis 中维护热点商品库存，支撑高并发扣减
- `StockMapper`：MySQL 中维护最终库存，作为数据源

Service 层通过两个仓储协作，实现"先扣 Redis，异步同步 MySQL"的高性能方案。

## 4. 双重写入模式（Dual-Write Pattern）

### 模式识别

Token 创建时同时写入 Redis 和 MySQL：

```java
// OAuth2TokenServiceImpl.createOAuth2AccessToken() 第 193-195 行
oauth2AccessTokenMapper.insert(accessTokenDO);      // 写 MySQL
oauth2AccessTokenRedisDAO.set(accessTokenDO);        // 写 Redis
```

Token 删除时也同时清理两个存储：

```java
// OAuth2TokenServiceImpl.removeAccessToken() 第 150-154 行
oauth2AccessTokenMapper.deleteById(accessTokenDO.getId());       // 删 MySQL
oauth2AccessTokenRedisDAO.delete(accessToken);                    // 删 Redis
```

### 一致性分析

| 场景 | MySQL | Redis | 结果 |
|------|-------|-------|------|
| 正常创建 | 写入成功 | 写入成功 | 一致 |
| Redis 写入失败 | 写入成功 | 未写入 | 不一致，但读取时会从 MySQL 回填 |
| MySQL 写入失败（事务回滚） | 未写入 | 已写入 | 不一致，Redis 中有脏数据（Token 会自然过期） |

### 设计意图

- **性能优先**：Token 校验是高频操作（每次 API 请求都要走），Redis 毫秒级响应
- **可用性优先**：Redis 不可用时降级到 MySQL 查询，保证系统不中断
- **最终一致**：Token 自带过期时间，脏数据会自然清理，无需人工干预

### 权衡取舍

这不是强一致性方案。项目选择"可用性优先"的原因是：

1. Token 校验是读多写少的场景，读性能比写一致性更重要
2. Token 有 TTL，即使出现不一致也会在过期后自动恢复
3. 最坏情况是多查一次 MySQL，不会造成业务错误

### 可迁移经验

电商秒杀场景中的库存扣减可以借鉴此模式：

- Redis 扣减库存（快速响应）
- MySQL 异步同步（最终持久化）
- 不一致时以 MySQL 为准，Redis 定期校准

关键原则：**只在能容忍短暂不一致的场景使用双重写入**。涉及资金的操作（如支付）必须用分布式事务保证强一致。

---

## 总结

| 设计模式 | 在本组件中的体现 | 核心收益 |
|---------|----------------|---------|
| 模板方法 | `createTokenAfterLoginSuccess()` | 消除登录后处理的重复代码 |
| 策略模式 | `OAuth2GrantServiceImpl` 的多种授权方式 | 各策略独立演化，互不干扰 |
| 仓储模式 | Redis DAO + MySQL Mapper | 存储层对 Service 透明 |
| 双重写入 | Token 同时写 Redis 和 MySQL | 高性能读取 + 数据持久化 |

这四种模式在本组件中协同工作：**策略模式**分流不同登录方式 → **模板方法**统一后处理 → **仓储模式**封装存储细节 → **双重写入**平衡性能与可靠性。理解它们的协作关系，比单独理解每个模式更有价值。
