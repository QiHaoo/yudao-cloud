# 设计模式提炼：字典与配置管理

> 从字典与配置管理组件中识别出的设计模式，抽象为可迁移的通用知识。

---

## 1. 元数据模式（Metadata Pattern）

### 模式识别

字典系统由两张表构成，形成"描述数据的数据"结构：

| 表 | DO 类 | 角色 | 示例 |
|----|-------|------|------|
| `system_dict_type` | `DictTypeDO` | 元数据：描述一类字典 | "性别"、"订单状态"、"支付方式" |
| `system_dict_data` | `DictDataDO` | 数据：字典的具体选项 | "男/女"、"待付款/已付款/已发货" |

```java
// 元数据层 —— 描述"这是什么类型的字典"
public class DictTypeDO extends BaseDO {
    private Long id;
    private String name;    // 字典名称，如"订单状态"
    private String type;    // 字典类型，如"order_status"
    private Integer status; // 是否启用
}

// 数据层 —— 描述"这个字典有哪些选项"
public class DictDataDO extends BaseDO {
    private Long id;
    private String dictType;    // 关联的字典类型，如"order_status"
    private String label;       // 显示文本，如"待付款"
    private String value;       // 存储值，如"0"
    private Integer sort;       // 排序
    private Integer status;     // 是否启用
}
```

### 设计意图

- **数据与描述分离**：字典的结构信息（类型）和内容信息（数据）分开管理，各自独立演进
- **类型作为聚合键**：`dictType` 字段将零散的字典数据聚合为一组，前端通过一个类型标识就能获取所有选项
- **自描述性**：字典类型本身包含名称和状态，不需要额外的文档来解释每个字典的含义

### 配置系统中的元数据

`ConfigDO` 将元数据和数据合并到一条记录中，每个配置项都是自描述的：

```java
public class ConfigDO extends BaseDO {
    private String category;   // 分类（元数据）
    private String name;       // 名称（元数据）
    private String configKey;  // 键名（标识）
    private String value;      // 键值（数据）
    private Integer type;      // 类型：系统内置 or 自定义（元数据）
    private Boolean visible;   // 是否可见（元数据）
}
```

配置项的 `name` + `configKey` + `type` + `visible` 共同构成元数据，描述了 `value` 的含义和使用规则。

### 可迁移经验

电商系统中的商品属性可以采用同样的分层设计：

- **属性类型（元数据）**：颜色、尺码、材质
- **属性值（数据）**：红色/蓝色、S/M/L、棉/涤纶

将属性类型独立管理后，运营人员可以在后台自由添加新的属性类型，无需开发介入。

## 2. 软删除模式（Soft Delete Pattern）

### 模式识别

`DictTypeDO` 使用软删除而非物理删除：

```java
public class DictTypeDO extends BaseDO {
    // ... 其他字段 ...
    private LocalDateTime deletedTime;  // 删除时间，null 表示未删除
}
```

删除操作不是 `DELETE FROM`，而是更新 `deletedTime`：

```java
// DictTypeServiceImpl.deleteDictType()
dictTypeMapper.updateToDelete(id, LocalDateTime.now());
```

删除前还会检查是否有关联的字典数据：

```java
if (dictDataService.getDictDataCountByDictType(dictType.getType()) > 0) {
    throw exception(DICT_TYPE_HAS_CHILDREN);
}
```

### 为什么用软删除

| 考量 | 物理删除 | 软删除 |
|------|---------|--------|
| 数据恢复 | 不可能 | 可以恢复 |
| 外键引用 | 可能破坏引用完整性 | 引用仍然有效 |
| 存储空间 | 释放空间 | 占用更多空间 |
| 查询复杂度 | 简单 | 需要过滤已删除记录 |
| 唯一索引 | 简单 | 需要特殊处理（见下文） |

### 软删除与唯一索引的冲突

软删除带来一个问题：如果记录被"删除"了，唯一索引仍然生效，导致无法重新创建同名记录。

项目的解决方案是设置 `deletedTime` 的默认值：

```java
// DictTypeServiceImpl.createDictType()
dictType.setDeletedTime(LocalDateTimeUtils.EMPTY); // 唯一索引，避免 null 值
```

这样唯一索引基于 `(type, deletedTime)` 组合，未删除的记录 `deletedTime` 为固定空值，已删除的记录 `deletedTime` 为实际时间，两者不冲突。

### 设计意图

- **数据安全**：字典类型被删除后，已有的业务数据（如订单的状态字段值 "0"）仍然有据可查
- **审计追溯**：保留删除时间和操作人，满足合规要求
- **防御误操作**：管理员误删字典后可以恢复

### 可迁移经验

电商系统中适合软删除的实体：

- **商品分类**：删除后历史订单中的分类引用不应失效
- **优惠券模板**：删除后已发放的优惠券仍需显示来源信息
- **用户账号**：注销后订单、评论等数据仍需保留关联

不适合软删除的场景：

- **临时数据**：验证码、会话日志等，用完即弃
- **高频写入表**：软删除会显著增加表体积，影响查询性能

## 3. 配置即数据模式（Configuration-as-Data）

### 模式识别

`ConfigDO` 将系统配置存储在数据库中，而非传统的 YAML/properties 文件：

```java
public class ConfigDO extends BaseDO {
    private String category;   // 分类，如"登录"
    private String name;       // 名称，如"账号初始密码"
    private String configKey;  // 键名，如"sys.user.init-password"
    private String value;      // 键值，如"123456"
    private Integer type;      // 系统内置(1) or 自定义(2)
    private Boolean visible;   // 前端是否可见
}
```

### 与传统配置的对比

| 维度 | YAML/properties | 数据库存储 |
|------|----------------|-----------|
| 修改方式 | 改文件 + 重启应用 | 管理后台实时修改 |
| 版本控制 | Git 天然支持 | 需要额外实现 |
| 多环境 | profile 机制 | 不同环境不同数据库 |
| 权限控制 | 文件系统权限 | 应用层权限控制 |
| 审计追踪 | Git log | 操作日志 |
| 敏感信息 | 需要加密工具 | `visible=false` 控制可见性 |

### 系统内置 vs 自定义配置

```java
// ConfigServiceImpl.createConfig()
config.setType(ConfigTypeEnum.CUSTOM.getType());  // 用户创建的都是自定义类型

// ConfigServiceImpl.deleteConfig()
if (ConfigTypeEnum.SYSTEM.getType().equals(config.getType())) {
    throw exception(CONFIG_CAN_NOT_DELETE_SYSTEM_TYPE);  // 系统内置不可删除
}
```

| 类型 | 说明 | 可编辑 | 可删除 |
|------|------|--------|--------|
| SYSTEM（系统内置） | 框架预设的配置，如初始密码、验证码开关 | 是 | 否 |
| CUSTOM（自定义） | 管理员通过 UI 新增的配置 | 是 | 是 |

### 设计意图

- **运维友好**：不需要登录服务器改配置文件，管理员在后台即可调整
- **实时生效**：修改后立即生效（配合缓存刷新），无需重启
- **敏感信息保护**：`visible=false` 的配置项前端无法获取，用于存储 API 密钥等敏感信息

### 权衡与风险

| 风险 | 应对措施 |
|------|---------|
| 配置被误改 | 操作日志记录每次变更 |
| 数据库挂了导致配置丢失 | 系统内置配置有默认值兜底 |
| 无法回滚到历史版本 | 可以考虑增加配置版本管理 |
| 多实例配置不一致 | 数据库是共享的，天然一致 |

### 可迁移经验

电商系统中适合存数据库的配置：

- **运费模板参数**：满减金额、包邮门槛，运营频繁调整
- **促销规则参数**：折扣上限、限购数量，需要实时生效
- **第三方接口密钥**：支付宝 AppId、物流 API Key，需要 `visible=false`

不适合存数据库的配置：

- **数据库连接信息**：应用启动时就需要，此时可能还没连上数据库
- **JVM 参数**：堆大小、GC 策略，必须在启动前确定
- **日志级别**：通常需要在不重启的情况下调整，但更推荐用 Spring Boot Actuator

## 4. 值对象模式（Value Object Pattern）

### 模式识别

在 DDD 的语境下，`DictDataDO` 是 `DictTypeDO` 的值对象：

| 特征 | 实体（Entity） | 值对象（Value Object） |
|------|---------------|---------------------|
| 标识 | 有唯一 ID | 由属性值标识 |
| 可变性 | 可以独立修改 | 通常整体替换 |
| 生命周期 | 独立存在 | 依附于实体 |

`DictDataDO` 虽然有自己的 `id`（数据库层面），但在业务语义上，它通过 `(dictType, value)` 复合键标识：

```java
// DictDataServiceImpl.validateDictDataValueUnique()
DictDataDO dictData = dictDataMapper.selectByDictTypeAndValue(dictType, value);
```

一个字典数据条目的业务含义完全由它的属性（label、value、sort、status）决定，ID 只是数据库的技术需求。

### 值对象的依附性

字典数据不能脱离字典类型独立存在：

```java
// DictDataServiceImpl.createDictData()
validateDictTypeExists(createReqVO.getDictType());  // 先验证类型存在

// DictTypeServiceImpl.deleteDictType()
if (dictDataService.getDictDataCountByDictType(dictType.getType()) > 0) {
    throw exception(DICT_TYPE_HAS_CHILDREN);  // 有数据时不允许删除类型
}
```

### 设计意图

- **聚合边界**：`DictTypeDO` 是聚合根，`DictDataDO` 是聚合内的值对象
- **完整性保护**：删除聚合根前检查值对象，防止产生孤儿数据
- **业务一致性**：字典数据的有效性依赖于字典类型的存在和状态

### 为什么 DictDataDO 有 ID

纯粹是数据库层面的技术需求。在严格的 DDD 实践中，值对象不应该有独立的 ID。但实际项目中，有 ID 方便单独更新某一条字典数据（如修改排序），这是实用性的妥协。

### 可迁移经验

电商系统中的值对象：

- **订单项（OrderItem）**：是订单（Order）的值对象，由 (orderId, skuId) 标识
- **商品规格（ProductSpec）**：是商品（Product）的值对象，由 (productId, specName) 标识
- **收货地址中的省市区**：是地址的值对象，由名称标识

判断是否应该建模为值对象的标准：**如果删除了实体，这个对象还有独立存在的意义吗？** 如果没有，它就是值对象。

---

## 总结

| 设计模式 | 在本组件中的体现 | 核心收益 |
|---------|----------------|---------|
| 元数据模式 | DictType 描述 DictData，Config 自描述 | 数据与描述分离，灵活扩展 |
| 软删除模式 | DictType 的 deletedTime | 数据安全，审计追溯 |
| 配置即数据 | ConfigDO 存储在数据库中 | 实时修改，无需重启 |
| 值对象模式 | DictDataDO 依附于 DictTypeDO | 聚合边界清晰，完整性保护 |

这四种模式在字典与配置组件中各有侧重：**元数据模式**解决了"如何描述数据"的问题，**值对象模式**解决了"数据之间如何组织"的问题，**软删除模式**解决了"数据如何安全删除"的问题，**配置即数据模式**解决了"配置如何灵活管理"的问题。它们共同构成了一个既灵活又可靠的字典配置体系。
