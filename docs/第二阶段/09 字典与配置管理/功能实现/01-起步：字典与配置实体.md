# 01 - 起步：字典与配置实体

## 本章目标

在动手写业务代码之前，我们需要先搞清楚"字典"和"配置"这两个概念在数据库里长什么样。本章将：

1. 理解 `DictTypeDO`（字典类型）、`DictDataDO`（字典数据）、`ConfigDO`（参数配置）三个实体的字段含义
2. 理解它们为什么都使用 `@TenantIgnore` 注解
3. 理解字典模块放在 `yudao-module-system`，而配置模块放在 `yudao-module-infra` 的原因

---

## 一、字典类型的实体：DictTypeDO

> 源码位置：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/dal/dataobject/dict/DictTypeDO.java`

### 1.1 字段说明

```java
@TableName("system_dict_type")
@TenantIgnore
public class DictTypeDO extends BaseDO {

    @TableId
    private Long id;          // 字典主键
    private String name;      // 字典名称，如"性别"、"订单状态"
    private String type;      // 字典类型（唯一编码），如"sys_common_sex"、"trade_order_status"
    private Integer status;   // 状态，参见 CommonStatusEnum（0=开启，1=禁用）
    private String remark;    // 备注
    private LocalDateTime deletedTime; // 删除时间（软删除标记）
}
```

### 1.2 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键，自增 |
| name | varchar(100) | 字典名称，如"性别" |
| type | varchar(100) | 字典类型编码，唯一索引，如 `sys_common_sex` |
| status | tinyint | 状态：0=开启，1=禁用 |
| remark | varchar(500) | 备注 |
| deleted | tinyint | 是否删除（0=未删除，1=已删除） |
| deleted_time | datetime | 删除时间，与 `deleted` 配合实现软删除 |
| create_time | datetime | 创建时间（来自 BaseDO） |
| creator | varchar(64) | 创建人（来自 BaseDO） |
| update_time | datetime | 更新时间（来自 BaseDO） |
| updater | varchar(64) | 更新人（来自 BaseDO） |

### 1.3 关键设计决策

**为什么用 `type` 字符串而不是数字 ID 来标识字典类型？**

想象一个电商系统，你需要在前端页面展示"订单状态"的下拉框。如果用数字 ID（比如 `dict_type_id = 5`）来关联，你需要记住"5 代表订单状态"——这在代码里毫无语义。而用字符串编码 `trade_order_status`，任何人看到都能立即理解含义。

**为什么使用 `deletedTime` 而不是物理删除？**

字典类型被删除后，可能有历史数据引用了它。物理删除会导致外键关联断裂。软删除通过 `deleted = 1` 和 `deleted_time` 记录删除状态，数据仍保留在数据库中，不会影响历史记录。

注意 `DictTypeMapper.updateToDelete()` 方法的实现：

```java
@Update("UPDATE system_dict_type SET deleted = 1, deleted_time = #{deletedTime} WHERE id = #{id}")
void updateToDelete(@Param("id") Long id, @Param("deletedTime") LocalDateTime deletedTime);
```

这是一种手动软删除，不依赖 MyBatis-Plus 的 `@TableLogic` 注解，而是通过自定义 SQL 精确控制删除行为。

### 1.4 为什么继承 BaseDO 而不是 TenantBaseDO？

`DictTypeDO` 继承的是 `BaseDO`（通用基础字段），而不是 `TenantBaseDO`（带租户字段的基础类）。配合 `@TenantIgnore` 注解，表示字典类型是**全局共享**的——所有租户看到的是同一套字典类型。

为什么？因为字典类型属于"元数据"，比如"性别"这个字典，不管哪个租户，男/女的定义是一样的。如果每个租户各自维护一套字典类型，不仅浪费存储，还会导致数据不一致。

## 二、字典数据的实体：DictDataDO

> 源码位置：`yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/dal/dataobject/dict/DictDataDO.java`

### 2.1 字段说明

```java
@TableName("system_dict_data")
@TenantIgnore
public class DictDataDO extends BaseDO {

    @TableId
    private Long id;          // 字典数据编号
    private Integer sort;     // 排序值，数字越小越靠前
    private String label;     // 字典标签，如"男"、"女"
    private String value;     // 字典值，如"1"、"2"
    private String dictType;  // 所属字典类型（字符串外键，不是 ID）
    private Integer status;   // 状态：0=开启，1=禁用
    private String colorType; // 颜色类型，对应 element-ui 颜色
    private String cssClass;  // 自定义 CSS 样式
    private String remark;    // 备注
}
```

### 2.2 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键，自增 |
| sort | int | 排序值，升序排列 |
| label | varchar(100) | 字典标签，前端展示用，如"待付款" |
| value | varchar(100) | 字典值，后端存储用，如"0" |
| dict_type | varchar(100) | 所属字典类型编码，关联 `system_dict_type.type` |
| status | tinyint | 状态：0=开启，1=禁用 |
| color_type | varchar | 颜色类型：default/primary/success/info/warning/danger |
| css_class | varchar | 自定义 CSS 类名 |
| remark | varchar(500) | 备注 |
| create_time | datetime | 创建时间（来自 BaseDO） |
| creator | varchar(64) | 创建人（来自 BaseDO） |
| update_time | datetime | 更新时间（来自 BaseDO） |
| updater | varchar(64) | 更新人（来自 BaseDO） |

### 2.3 关键设计决策

**为什么 `dictType` 是字符串而不是数字 ID 外键？**

如果用数字 ID（如 `dict_type_id = 5`），每次查询字典数据都需要先 JOIN 字典类型表，再去匹配编码。直接存储类型编码字符串（如 `trade_order_status`），查询时只需一条 SQL：`SELECT * FROM system_dict_data WHERE dict_type = 'trade_order_status'`，效率更高且语义更清晰。

**`colorType` 和 `cssClass` 是干什么的？**

这两个字段服务于前端 UI 渲染。以电商订单状态为例：

| label | value | colorType | 效果 |
|-------|-------|-----------|------|
| 待付款 | 0 | warning | 橙色标签 |
| 已付款 | 1 | primary | 蓝色标签 |
| 已发货 | 2 | success | 绿色标签 |
| 已完成 | 3 | default | 灰色标签 |

`colorType` 映射到 element-ui 的内置颜色（default、primary、success、info、warning、danger），`cssClass` 则允许前端开发者自定义更复杂的样式。

**为什么 `@TableField(updateStrategy = FieldStrategy.ALWAYS)` 用在 `cssClass` 上？**

默认情况下，MyBatis-Plus 更新字段时会跳过 `null` 值。但 `cssClass` 允许被清空（设为空字符串或 null），所以需要 `ALWAYS` 策略，确保即使传入 null 也会更新到数据库。

## 三、参数配置的实体：ConfigDO

> 源码位置：`yudao-module-infra/yudao-module-infra-server/src/main/java/cn/iocoder/yudao/module/infra/dal/dataobject/config/ConfigDO.java`

### 3.1 字段说明

```java
@TableName("infra_config")
@TenantIgnore
public class ConfigDO extends BaseDO {

    @TableId
    private Long id;          // 参数主键
    private String category;  // 参数分类，如"biz"、"system"
    private String name;      // 参数名称，如"数据库用户名"
    private String configKey; // 参数键名，如"yunai.db.username"
    private String value;     // 参数键值，如"root"
    private Integer type;     // 参数类型：1=系统内置，2=自定义
    private Boolean visible;  // 是否可见（false=敏感配置，前端不可获取）
    private String remark;    // 备注
}
```

### 3.2 表结构

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键，自增 |
| category | varchar(50) | 参数分类，如"biz"、"mail"、"sms" |
| name | varchar(100) | 参数名称，如"电商-商品库存预警阈值" |
| config_key | varchar(100) | 参数键名，唯一索引，如 `yunai.biz.stock-warning` |
| value | varchar(500) | 参数键值，如"10" |
| type | tinyint | 参数类型：1=系统内置（SYSTEM），2=自定义（CUSTOM） |
| visible | tinyint | 是否可见：true=前端可获取，false=敏感配置不可获取 |
| remark | varchar(500) | 备注 |
| create_time | datetime | 创建时间（来自 BaseDO） |
| creator | varchar(64) | 创建人（来自 BaseDO） |
| update_time | datetime | 更新时间（来自 BaseDO） |
| updater | varchar(64) | 更新人（来自 BaseDO） |

### 3.3 关键设计决策

**为什么字段名用 `configKey` 而不是 `key`？**

源码注释已经说明了原因：

```java
/**
 * 支持多 DB 类型时，无法直接使用 key + @TableField("config_key") 来实现转换，
 * 原因是 "config_key" AS key 而存在报错
 */
private String configKey;
```

`key` 是 SQL 保留字（如 `PRIMARY KEY`），在某些数据库中会导致语法错误。为了避免跨数据库兼容性问题，直接使用 `configKey` 作为 Java 字段名，对应数据库列名 `config_key`。

**`visible = false` 的使用场景**

想象一个电商系统中有以下配置：

| configKey | value | visible |
|-----------|-------|---------|
| yunai.biz.shop-name | "芋道商城" | true |
| yunai.pay.alipay-app-id | "2021001100001" | false |
| yunai.sms.aliyun-access-key-secret | "aBcDeFgHiJk" | false |

`visible = true` 的配置可以被前端通过 `/infra/config/get-value-by-key` 接口直接获取，用于页面展示。`visible = false` 的配置（如支付密钥、短信密钥）则会被拒绝返回，防止敏感信息泄露。

**SYSTEM 和 CUSTOM 类型的区别**

```java
public enum ConfigTypeEnum {
    SYSTEM(1),  // 系统内置配置
    CUSTOM(2);  // 用户自定义配置
}
```

- `SYSTEM(1)`：系统预置的配置项，如数据库连接参数、缓存过期时间等。这些配置**不允许用户删除**，只能修改值。
- `CUSTOM(2)`：用户自行创建的配置项，如业务参数、开关等。用户可以自由增删改。

`ConfigServiceImpl.deleteConfig()` 中有明确的保护逻辑：

```java
if (ConfigTypeEnum.SYSTEM.getType().equals(config.getType())) {
    throw exception(CONFIG_CAN_NOT_DELETE_SYSTEM_TYPE);
}
```

## 四、模块归属：为什么字典在 system，配置在 infra？

这是很多初学者会困惑的问题——字典和配置看起来功能类似，为什么不放在同一个模块？

| 维度 | 字典（Dict） | 配置（Config） |
|------|-------------|---------------|
| 所属模块 | `yudao-module-system` | `yudao-module-infra` |
| 关注点 | 业务枚举值管理 | 系统运行参数管理 |
| 使用场景 | 前端下拉框、标签渲染 | 后端功能开关、参数读取 |
| 数据特征 | 类型+数据的两层结构 | 键值对的扁平结构 |
| 典型例子 | 订单状态、商品分类 | 短信配置、文件存储路径 |

**字典属于"系统管理"**，因为它管理的是业务领域的枚举数据，是用户可见、可操作的元数据。在 RBAC 权限模型中，字典管理属于系统管理员的职责。

**配置属于"基础设施"**，因为它管理的是系统运行时的技术参数，更多是给后端代码读取的。比如数据库连接配置、第三方 API 密钥等，属于基础设施层面的关注点。

这种模块归属的设计遵循了**关注点分离**原则：系统模块聚焦于用户和权限管理，基础设施模块聚焦于技术支撑能力。

## 六、小结

| 要点 | 说明 |
|------|------|
| DictTypeDO | 字典类型，用字符串 `type` 作为唯一编码，支持软删除 |
| DictDataDO | 字典数据，用字符串 `dictType` 关联类型（非 ID 外键），支持颜色和 CSS 自定义 |
| ConfigDO | 参数配置，用 `configKey` 避免 SQL 保留字问题，`visible` 控制敏感配置可见性 |
| @TenantIgnore | 三个实体都是全局共享的，不按租户隔离 |
| 模块归属 | 字典在 system（业务元数据），配置在 infra（技术参数） |
