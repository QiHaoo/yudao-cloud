## 一、测试支持

### 1.1 测试字典类型服务

字典类型的核心逻辑是唯一性校验和删除保护，测试时需要 mock Mapper 层。

**测试创建字典类型——名称重复：**

```java
@ExtendWith(MockitoExtension.class)
class DictTypeServiceImplTest {

    @InjectMocks
    private DictTypeServiceImpl dictTypeService;

    @Mock
    private DictTypeMapper dictTypeMapper;

    @Mock
    private DictDataService dictDataService;

    @Test
    void testCreateDictType_nameDuplicate() {
        // 准备参数
        DictTypeSaveReqVO reqVO = new DictTypeSaveReqVO()
                .setName("性别").setType("sys_common_sex").setStatus(0);

        // mock 已存在同名记录
        DictTypeDO existing = new DictTypeDO().setId(1L).setName("性别");
        when(dictTypeMapper.selectByName("性别")).thenReturn(existing);

        // 调用，断言异常
        assertServiceException(() -> dictTypeService.createDictType(reqVO),
                DICT_TYPE_NAME_DUPLICATE);
    }
}
```

**测试删除字典类型——存在子数据：**

```java
@Test
void testDeleteDictType_hasChildren() {
    // 准备参数
    Long dictTypeId = 1L;

    // mock 字典类型存在
    DictTypeDO dictType = new DictTypeDO().setId(dictTypeId).setType("sys_common_sex");
    when(dictTypeMapper.selectById(dictTypeId)).thenReturn(dictType);

    // mock 有子数据
    when(dictDataService.getDictDataCountByDictType("sys_common_sex")).thenReturn(5L);

    // 调用，断言异常
    assertServiceException(() -> dictTypeService.deleteDictType(dictTypeId),
            DICT_TYPE_HAS_CHILDREN);
}
```

### 1.2 测试字典数据服务

字典数据的核心逻辑是字典类型有效性校验和值唯一性校验。

**测试创建字典数据——字典类型不存在：**

```java
@ExtendWith(MockitoExtension.class)
class DictDataServiceImplTest {

    @InjectMocks
    private DictDataServiceImpl dictDataService;

    @Mock
    private DictDataMapper dictDataMapper;

    @Mock
    private DictTypeService dictTypeService;

    @Test
    void testCreateDictData_dictTypeNotExists() {
        // 准备参数
        DictDataSaveReqVO reqVO = new DictDataSaveReqVO()
                .setLabel("男").setValue("1").setDictType("sys_common_sex").setSort(1).setStatus(0);

        // mock 字典类型不存在
        when(dictTypeService.getDictType("sys_common_sex")).thenReturn(null);

        // 调用，断言异常
        assertServiceException(() -> dictDataService.createDictData(reqVO),
                DICT_TYPE_NOT_EXISTS);
    }
}
```

**测试批量校验字典数据：**

```java
@Test
void testValidateDictDataList_valueNotEnabled() {
    // 准备参数
    String dictType = "trade_order_status";
    List<String> values = Arrays.asList("0", "1");

    // mock 查询结果：value=1 的数据被禁用
    DictDataDO data0 = new DictDataDO().setValue("0").setStatus(CommonStatusEnum.ENABLE.getStatus());
    DictDataDO data1 = new DictDataDO().setValue("1").setStatus(CommonStatusEnum.DISABLE.getStatus()).setLabel("已付款");
    when(dictDataMapper.selectByDictTypeAndValues(dictType, values))
            .thenReturn(Arrays.asList(data0, data1));

    // 调用，断言异常
    assertServiceException(() -> dictDataService.validateDictDataList(dictType, values),
                DICT_DATA_NOT_ENABLE);
}
```

### 1.3 测试配置服务

配置的核心逻辑是唯一性校验、系统配置删除保护和可见性控制。

**测试删除系统内置配置：**

```java
@ExtendWith(MockitoExtension.class)
class ConfigServiceImplTest {

    @InjectMocks
    private ConfigServiceImpl configService;

    @Mock
    private ConfigMapper configMapper;

    @Test
    void testDeleteConfig_systemType() {
        // 准备参数
        Long configId = 1L;

        // mock 系统内置配置
        ConfigDO config = new ConfigDO().setId(configId).setType(ConfigTypeEnum.SYSTEM.getType());
        when(configMapper.selectById(configId)).thenReturn(config);

        // 调用，断言异常
        assertServiceException(() -> configService.deleteConfig(configId),
                CONFIG_CAN_NOT_DELETE_SYSTEM_TYPE);
    }
}
```

**测试按 Key 获取配置——不可见：**

```java
@Test
void testGetConfigByKey_invisible() {
    // 这个测试实际上在 Controller 层，因为 visible 检查在 Controller 中
    // 准备参数
    String key = "yunai.pay.alipay-private-key";

    // mock 配置存在但不可见
    ConfigDO config = new ConfigDO().setConfigKey(key).setValue("secret").setVisible(false);
    when(configMapper.selectByKey(key)).thenReturn(config);

    // 在 Controller 层调用 get-value-by-key 时会抛出异常
    // 这里验证 Service 层能正确返回配置
    ConfigDO result = configService.getConfigByKey(key);
    assertNotNull(result);
    assertFalse(result.getVisible());
}
```

### 1.4 测试要点总结

| 测试场景 | 关键断言 |
|---------|---------|
| 创建字典类型-名称重复 | 抛出 `DICT_TYPE_NAME_DUPLICATE` |
| 创建字典类型-编码重复 | 抛出 `DICT_TYPE_TYPE_DUPLICATE` |
| 删除字典类型-有子数据 | 抛出 `DICT_TYPE_HAS_CHILDREN` |
| 创建字典数据-类型不存在 | 抛出 `DICT_TYPE_NOT_EXISTS` |
| 创建字典数据-类型已禁用 | 抛出 `DICT_TYPE_NOT_ENABLE` |
| 创建字典数据-值重复 | 抛出 `DICT_DATA_VALUE_DUPLICATE` |
| 创建配置-key 重复 | 抛出 `CONFIG_KEY_DUPLICATE` |
| 删除配置-系统内置 | 抛出 `CONFIG_CAN_NOT_DELETE_SYSTEM_TYPE` |
| 获取配置值-不可见 | 抛出 `CONFIG_GET_VALUE_ERROR_IF_VISIBLE` |

### 2.4 关键设计模式速查

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| @TenantIgnore | 所有三个 DO | 全局共享数据，不按租户隔离 |
| 软删除 | DictTypeMapper.updateToDelete() | 字典类型支持软删除，保留历史数据 |
| 唯一性校验 | 所有 Service 实现 | 新增时 id=null 直接判重，修改时排除自身 |
| 删除保护 | DictTypeServiceImpl.deleteDictType() | 有子数据时拒绝删除 |
| 类型保护 | ConfigServiceImpl.deleteConfig() | 系统内置配置不允许删除 |
| 可见性控制 | ConfigController.getConfigKey() | 敏感配置不返回给前端 |
| MapStruct 转换 | ConfigConvert | VO/DO 字段名不同时使用 MapStruct |
| BeanUtils.toBean() | 字典模块 | VO/DO 字段名相同时使用通用转换 |
| LambdaQueryWrapperX | 所有 Mapper | *IfPresent 方法消除空值判断 |

## 三、跨模块协作说明

字典和配置虽然分属不同模块，但在业务使用中经常配合：

```
┌─────────────────────────────────────────────────────────────────┐
│                     业务模块（如 yudao-module-trade）              │
│                                                                 │
│   创建订单时：                                                    │
│   1. 调用 DictDataService.validateDictDataList()                │
│      校验订单状态值是否合法                                        │
│   2. 调用 ConfigService.getConfigByKey()                        │
│      读取库存预警阈值等业务配置                                     │
└────────────┬──────────────────────────────────┬─────────────────┘
             │                                  │
             ▼                                  ▼
┌────────────────────────┐        ┌────────────────────────┐
│  yudao-module-system   │        │  yudao-module-infra    │
│                        │        │                        │
│  字典类型 + 字典数据     │        │  参数配置               │
│  (枚举值管理)           │        │  (运行时参数)           │
└────────────────────────┘        └────────────────────────┘
```

在微服务架构下，跨模块调用通过 Feign 客户端实现。在单体架构下，直接注入 Service 即可。

## 四、小结

| 要点 | 说明 |
|------|------|
| 测试策略 | Mock Mapper 层，测试 Service 层的校验逻辑 |
| 重点测试 | 唯一性校验、删除保护、可见性控制 |
| 源码组织 | 字典在 system 模块，配置在 infra 模块，各自独立完整 |
| 跨模块协作 | 业务模块通过 Service 接口调用，不直接依赖实现类 |
