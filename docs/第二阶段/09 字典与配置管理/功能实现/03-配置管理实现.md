# 03 - 配置管理实现

## 本章目标

实现参数配置的完整 CRUD 功能，包括：

1. 配置的增删改查与导出
2. 通过 configKey 获取配置值的能力
3. 敏感配置的可见性控制
4. 系统内置配置的删除保护

---

## 一、配置控制器：ConfigController

> 源码位置：`yudao-module-infra/yudao-module-infra-server/src/main/java/cn/iocoder/yudao/module/infra/controller/admin/config/ConfigController.java`

### 1.1 API 总览

| 接口 | 方法 | URL | 权限 | 说明 |
|------|------|-----|------|------|
| 创建配置 | POST | `/infra/config/create` | `infra:config:create` | 新增参数配置 |
| 修改配置 | PUT | `/infra/config/update` | `infra:config:update` | 修改参数配置 |
| 删除配置 | DELETE | `/infra/config/delete` | `infra:config:delete` | 删除单个配置 |
| 批量删除 | DELETE | `/infra/config/delete-list` | `infra:config:delete` | 批量删除配置 |
| 查询详情 | GET | `/infra/config/get` | `infra:config:query` | 查询单个配置 |
| 按键取值 | GET | `/infra/config/get-value-by-key` | 无需权限 | 根据 configKey 获取值 |
| 分页查询 | GET | `/infra/config/page` | `infra:config:query` | 分页查询配置 |
| 导出 Excel | GET | `/infra/config/export-excel` | `infra:config:export` | 导出配置数据 |

### 1.2 URL 前缀说明

注意配置管理的 URL 前缀是 `/infra/config`，而不是 `/system/config`。这是因为配置管理属于 `yudao-module-infra` 模块，URL 前缀由模块所在包路径决定（参见 YudaoWebAutoConfiguration 的 URL 前缀注入机制）。

### 1.3 核心接口实现

**按 Key 获取配置值：**

```java
@GetMapping(value = "/get-value-by-key")
@Operation(summary = "根据参数键名查询参数值", description = "不可见的配置，不允许返回给前端")
public CommonResult<String> getConfigKey(@RequestParam("key") String key) {
    ConfigDO config = configService.getConfigByKey(key);
    if (config == null) {
        return success(null);
    }
    if (!config.getVisible()) {
        throw exception(ErrorCodeConstants.CONFIG_GET_VALUE_ERROR_IF_VISIBLE);
    }
    return success(config.getValue());
}
```

这个接口的实现逻辑：

1. 根据 `key` 查询配置
2. 如果配置不存在，返回 null（不报错，因为可能是可选配置）
3. 如果配置存在但 `visible = false`，抛出 `CONFIG_GET_VALUE_ERROR_IF_VISIBLE` 错误
4. 只有 `visible = true` 的配置才返回值

**创建配置：**

```java
@PostMapping("/create")
@PreAuthorize("@ss.hasPermission('infra:config:create')")
public CommonResult<Long> createConfig(@Valid @RequestBody ConfigSaveReqVO createReqVO) {
    return success(configService.createConfig(createReqVO));
}
```

### 1.4 请求/响应 VO

**ConfigSaveReqVO（创建/修改请求）：**

```java
@Data
public class ConfigSaveReqVO {
    private Long id;           // 编号（修改时必填）
    @NotEmpty private String category; // 参数分组，如"biz"、"mail"
    @NotBlank private String name;     // 参数名称，如"电商-库存预警阈值"
    @NotBlank private String key;      // 参数键名，如"yunai.biz.stock-warning"
    @NotBlank private String value;    // 参数键值，如"10"
    @NotNull  private Boolean visible; // 是否可见
    private String remark;             // 备注
}
```

注意 `key` 字段在 VO 中叫 `key`，但在 DO 中叫 `configKey`。这是因为 VO 面向前端，用简洁的 `key` 更友好；DO 面向数据库，用 `configKey` 避免 SQL 保留字问题。两者通过 `ConfigConvert` 进行转换。

## 二、配置服务层：ConfigServiceImpl

> 源码位置：`yudao-module-infra/yudao-module-infra-server/src/main/java/cn/iocoder/yudao/module/infra/service/config/ConfigServiceImpl.java`

### 2.1 创建配置

```java
@Override
public Long createConfig(ConfigSaveReqVO createReqVO) {
    // 1. 校验参数配置 key 的唯一性
    validateConfigKeyUnique(null, createReqVO.getKey());

    // 2. 插入参数配置
    ConfigDO config = ConfigConvert.INSTANCE.convert(createReqVO);
    config.setType(ConfigTypeEnum.CUSTOM.getType()); // 用户创建的都是自定义类型
    configMapper.insert(config);
    return config.getId();
}
```

关键设计决策：**用户创建的配置自动设置为 `CUSTOM` 类型**。这意味着用户无法创建"系统内置"类型的配置——只有开发人员通过 SQL 脚本或初始化代码才能创建 `SYSTEM` 类型的配置。

### 2.2 删除配置

```java
@Override
public void deleteConfig(Long id) {
    // 1. 校验配置存在
    ConfigDO config = validateConfigExists(id);
    // 2. 内置配置，不允许删除
    if (ConfigTypeEnum.SYSTEM.getType().equals(config.getType())) {
        throw exception(CONFIG_CAN_NOT_DELETE_SYSTEM_TYPE);
    }
    // 3. 删除
    configMapper.deleteById(id);
}
```

删除流程的关键保护：**系统内置配置不允许删除**。比如数据库连接配置、缓存过期时间等，这些是系统运行必需的参数，如果被误删会导致系统异常。

批量删除也有同样的保护：

```java
@Override
public void deleteConfigList(List<Long> ids) {
    // 校验是否有内置配置
    List<ConfigDO> configs = configMapper.selectByIds(ids);
    configs.forEach(config -> {
        if (ConfigTypeEnum.SYSTEM.getType().equals(config.getType())) {
            throw exception(CONFIG_CAN_NOT_DELETE_SYSTEM_TYPE);
        }
    });
    // 批量删除
    configMapper.deleteByIds(ids);
}
```

### 2.3 唯一性校验

```java
@VisibleForTesting
public void validateConfigKeyUnique(Long id, String key) {
    ConfigDO config = configMapper.selectByKey(key);
    if (config == null) {
        return;
    }
    if (id == null) {
        throw exception(CONFIG_KEY_DUPLICATE);
    }
    if (!config.getId().equals(id)) {
        throw exception(CONFIG_KEY_DUPLICATE);
    }
}
```

这与字典管理的唯一性校验模式完全一致——项目中所有唯一性校验都遵循这个模式。

### 2.4 Mapper 层实现

> 源码位置：`yudao-module-infra/yudao-module-infra-server/src/main/java/cn/iocoder/yudao/module/infra/dal/mysql/config/ConfigMapper.java`

```java
@Mapper
public interface ConfigMapper extends BaseMapperX<ConfigDO> {

    default ConfigDO selectByKey(String key) {
        return selectOne(ConfigDO::getConfigKey, key);
    }

    default PageResult<ConfigDO> selectPage(ConfigPageReqVO reqVO) {
        return selectPage(reqVO, new LambdaQueryWrapperX<ConfigDO>()
                .likeIfPresent(ConfigDO::getName, reqVO.getName())
                .likeIfPresent(ConfigDO::getConfigKey, reqVO.getKey())
                .eqIfPresent(ConfigDO::getType, reqVO.getType())
                .betweenIfPresent(ConfigDO::getCreateTime, reqVO.getCreateTime())
                .orderByDesc(ConfigDO::getId));
    }
}
```

注意 `selectByKey()` 方法使用 `ConfigDO::getConfigKey` 而不是 `ConfigDO::getKey`——因为 DO 中的字段名是 `configKey`，对应数据库列 `config_key`。

## 三、ConfigConvert 转换器

配置模块使用 MapStruct 进行 VO/DO 转换，而不是通用的 `BeanUtils.toBean()`。这是因为 VO 中的字段名 `key` 需要映射到 DO 中的 `configKey`。

```java
@Mapper
public interface ConfigConvert {

    ConfigConvert INSTANCE = Mappers.getMapper(ConfigConvert.class);

    ConfigDO convert(ConfigSaveReqVO bean);

    ConfigRespVO convert(ConfigDO bean);

    PageResult<ConfigRespVO> convertPage(PageResult<ConfigDO> page);

    List<ConfigRespVO> convertList(List<ConfigDO> list);
}
```

MapStruct 会自动处理 `key` -> `configKey` 的映射（基于命名约定或 `@Mapping` 注解）。

## 四、ConfigTypeEnum 枚举

> 源码位置：`yudao-module-infra/yudao-module-infra-api/src/main/java/cn/iocoder/yudao/module/infra/enums/config/ConfigTypeEnum.java`

```java
@Getter
@AllArgsConstructor
public enum ConfigTypeEnum {
    SYSTEM(1),  // 系统内置配置
    CUSTOM(2);  // 自定义配置

    private final Integer type;
}
```

两种类型的区别：

| 维度 | SYSTEM(1) | CUSTOM(2) |
|------|-----------|-----------|
| 创建方式 | 开发人员通过 SQL 脚本预置 | 用户通过管理后台创建 |
| 可否修改 | 可以修改值 | 可以修改所有字段 |
| 可否删除 | 不允许删除 | 允许删除 |
| 典型场景 | 数据库配置、缓存配置 | 业务开关、阈值参数 |

## 五、电商场景实战示例

### 5.1 配置项规划

以电商系统为例，常见的配置项如下：

| category | name | configKey | value | type | visible |
|----------|------|-----------|-------|------|---------|
| biz | 商品库存预警阈值 | yunai.biz.stock-warning | 10 | CUSTOM | true |
| biz | 订单自动取消时间（分钟） | yunai.biz.order-auto-cancel-minutes | 30 | CUSTOM | true |
| pay | 支付宝 App ID | yunai.pay.alipay-app-id | 2021001100001 | SYSTEM | false |
| pay | 支付宝私钥 | yunai.pay.alipay-private-key | MIIEvg... | SYSTEM | false |
| sms | 短信验证码有效期（秒） | yunai.sms.code-expire-seconds | 300 | CUSTOM | true |
| mail | 邮件发送人 | yunai.mail.from-address | noreply@shop.com | SYSTEM | true |

### 5.2 创建配置

```
POST /infra/config/create
{
    "category": "biz",
    "name": "商品库存预警阈值",
    "key": "yunai.biz.stock-warning",
    "value": "10",
    "visible": true,
    "remark": "当商品库存低于此值时，系统发送预警通知"
}
```

### 5.3 前端获取配置值

```
GET /infra/config/get-value-by-key?key=yunai.biz.stock-warning
```

返回：

```json
{
    "code": 0,
    "msg": "",
    "data": "10"
}
```

如果尝试获取不可见配置：

```
GET /infra/config/get-value-by-key?key=yunai.pay.alipay-private-key
```

返回错误：

```json
{
    "code": 1001000004,
    "msg": "获取参数配置失败，原因：不允许获取不可见配置",
    "data": null
}
```

### 5.4 后端代码中使用配置

在业务代码中，可以通过 `ConfigService.getConfigByKey()` 读取配置值：

```java
@Resource
private ConfigService configService;

public void checkStockWarning(TradeOrderDO order) {
    // 读取库存预警阈值
    ConfigDO config = configService.getConfigByKey("yunai.biz.stock-warning");
    int warningThreshold = Integer.parseInt(config.getValue());

    // 检查商品库存
    for (TradeOrderItemDO item : order.getItems()) {
        if (item.getSkuStock() < warningThreshold) {
            // 发送库存预警通知
            notifyService.sendStockWarning(item.getSkuId(), item.getSkuStock());
        }
    }
}
```

## 七、字典 vs 配置：实现差异对比

| 维度 | 字典管理 | 配置管理 |
|------|---------|---------|
| 所属模块 | yudao-module-system | yudao-module-infra |
| URL 前缀 | `/system/dict-type`、`/system/dict-data` | `/infra/config` |
| 权限前缀 | `system:dict:*` | `infra:config:*` |
| 数据结构 | 两层：类型 + 数据 | 扁平：键值对 |
| 软删除 | 字典类型支持软删除 | 不支持软删除 |
| 类型保护 | 删除类型前检查子数据 | 系统内置配置不允许删除 |
| 可见性控制 | 无 | `visible` 字段控制敏感配置 |
| 转换方式 | BeanUtils.toBean() | MapStruct (ConfigConvert) |
| 前端缓存 | list-all-simple 全量缓存 | 按需通过 get-value-by-key 获取 |

## 八、小结

| 要点 | 说明 |
|------|------|
| URL 前缀 | `/infra/config`，因为配置属于基础设施模块 |
| 可见性控制 | `visible = false` 的配置不会返回给前端，保护敏感信息 |
| 类型保护 | 系统内置配置（SYSTEM）不允许用户删除，只能修改值 |
| 用户创建 | 用户通过管理后台创建的配置自动标记为 CUSTOM 类型 |
| key 命名 | 推荐使用 `yunai.{模块}.{功能}` 的命名规范，如 `yunai.biz.stock-warning` |
