# 04-核心设计：系统配置管理

## 场景引入

电商平台需要各种系统级配置：
- 站点名称、Logo
- 文件上传大小限制
- 短信服务的 API Key
- 是否开放注册
- 数据库连接信息

这些配置有的是公开的（站点名称），有的是敏感的（API Key、数据库密码）。本章讲清配置管理的设计。

## 配置实体设计

```
ConfigDO
├── id              配置编号（主键）
├── category        配置分类（如 "biz", "sms", "db"）
├── name            配置名称（如 "用户初始密码"）
├── configKey       配置键名（如 "system.user.init-password"）
├── value           配置键值（如 "123456"）
├── type            配置类型（SYSTEM=1 / CUSTOM=2）
├── visible         是否可见（true=公开, false=敏感）
├── remark          备注
└── 继承 BaseDO（creator, createTime, updater, updateTime, deleted）
```

### 设计要点

**1. configKey 而不是 key**

```java
private String configKey;  // 而不是 "key"
```

原因是 `key` 在某些数据库中是保留字（如 SQL Server），使用 `configKey` 避免兼容性问题。通过 MapStruct 的 `@Mapping` 注解在 VO 和 DO 之间转换：

```java
@Mapping(source = "configKey", target = "key")   // DO → VO
@Mapping(source = "key", target = "configKey")    // VO → DO
```

**2. @TenantIgnore：配置全局共享**

与字典类型一样，配置也是全局共享的。所有租户使用同一套系统配置，不需要每个租户重复设置。

**3. visible 字段：敏感数据保护**

```java
// ConfigController.getConfigKey()
if (!config.getVisible()) {
    throw exception(CONFIG_GET_VALUE_ERROR_IF_VISIBLE);
}
```

不可见的配置（如数据库密码、API Key）不允许通过 `get-value-by-key` 接口返回给前端。只有内部代码可以通过 `configService.getConfigByKey()` 直接读取。

## 两种配置类型

```java
public enum ConfigTypeEnum {
    SYSTEM(1),  // 系统内置配置
    CUSTOM(2)   // 自定义配置
}
```

### 系统配置 vs 自定义配置

| 维度 | 系统配置（SYSTEM） | 自定义配置（CUSTOM） |
|------|-------------------|---------------------|
| 创建方式 | 数据库初始化脚本 | 管理员在界面创建 |
| 能否修改 | 可以修改值 | 可以修改所有字段 |
| 能否删除 | 不允许删除 | 可以删除 |
| 典型用途 | 系统运行依赖的配置 | 业务自定义配置 |

```java
// ConfigServiceImpl.deleteConfig()
if (ConfigTypeEnum.SYSTEM.getType().equals(config.getType())) {
    throw exception(CONFIG_CAN_NOT_DELETE_SYSTEM_TYPE);
}
```

### 为什么系统配置不能删除？

系统配置是框架运行依赖的。例如：
- `system.user.init-password`：用户导入时的初始密码
- `system.user.register-enabled`：是否开放注册
- `sys.captcha.enable`：验证码开关

如果误删这些配置，系统功能会异常。通过 `type=SYSTEM` 标记，防止误操作。

## 配置分类（category）

配置通过 `category` 字段分组：

```
biz        业务配置
├── system.user.init-password    用户初始密码
├── system.user.register-enabled 是否开放注册
└── ...

sms        短信配置
├── sms.aliyun.access-key        阿里云 AccessKey
├── sms.aliyun.secret-key        阿里云 SecretKey
└── ...

db         数据库配置
├── db.name                      数据库名
└── ...
```

分类不影响功能逻辑，纯粹是为了管理界面的组织。管理员可以按分类筛选配置。

## 配置的缓存策略

配置值被频繁读取（如每次用户注册都要检查 `register-enabled`），需要缓存：

```
读取配置：
  优先从 Redis 缓存读取
  ↓ 缓存未命中
  从 MySQL 读取
  ↓ 写入缓存
  返回配置值

修改配置：
  更新 MySQL
  ↓
  清除 Redis 缓存
  ↓
  下次读取时重新加载
```

## 配置与字典的区别

| 维度 | 字典（Dict） | 配置（Config） |
|------|-------------|---------------|
| 用途 | 代码→标签映射 | 系统参数键值对 |
| 数据结构 | 一对多（类型→数据项） | 一对一（键→值） |
| 典型数据 | 性别、状态、订单状态 | 站点名称、API Key |
| 前端使用 | 下拉框、标签、徽章 | 功能开关、参数读取 |
| 所在模块 | yudao-module-system | yudao-module-infra |
| 租户共享 | 是（@TenantIgnore） | 是（@TenantIgnore） |

### 为什么配置在 infra 模块而不是 system 模块？

这是一个模块职责划分的决策：
- **system 模块**：管理业务实体（用户、角色、菜单、字典）
- **infra 模块**：管理基础设施（配置、文件、代码生成、日志）

配置属于"基础设施"范畴，因为它影响的是系统运行行为，而非业务数据。

## 思考题

1. 如果需要支持"配置版本管理"（每次修改都保留历史版本，支持回滚），需要怎么设计？
2. 本项目配置值都是字符串类型。如果需要支持 JSON 格式的复杂配置（如邮件服务器配置包含 host/port/username/password），应该怎么处理？
3. 如果要支持"配置热更新"（修改配置后不需要重启服务立即生效），有哪些方案？本项目的 Redis 缓存机制是否已经支持？
