# 03-核心设计：字典类型与字典数据

## 从"订单状态"说起

上一章我们分析了字典要解决的问题。现在让我们看看，这个项目是如何设计字典的数据模型的。

假设我们要为电商系统添加"订单状态"字典。最终效果是：后台管理员在界面上配置好映射关系，前端的所有页面都能自动获取到这些数据来渲染下拉框和状态标签。

那么问题来了：这个映射关系应该存在哪？怎么存？怎么读？怎么缓存？

## 数据模型：两张表的经典设计

字典采用**主表-从表**的两层结构：

```
┌─────────────────────────────────────────────────────────────────┐
│                    字典数据模型                                    │
│                                                                 │
│  ┌───────────────────────┐         ┌──────────────────────────┐ │
│  │  system_dict_type      │  1 : N  │  system_dict_data         │ │
│  │  （字典类型表）          │ ──────> │  （字典数据表）            │ │
│  │                       │         │                          │ │
│  │  id           BIGINT  │         │  id           BIGINT     │ │
│  │  name         VARCHAR │         │  dictType     VARCHAR    │ │ ← 用 type 字符串关联，不用 ID
│  │  type         VARCHAR │         │  label        VARCHAR    │ │
│  │  status       INT     │         │  value        VARCHAR    │ │
│  │  remark       VARCHAR │         │  sort         INT        │ │
│  │  createTime   DATETIME│         │  status       INT        │ │
│  │  updateTime   DATETIME│         │  colorType    VARCHAR    │ │
│  │  deletedTime  DATETIME│         │  cssClass     VARCHAR    │ │
│  └───────────────────────┘         │  remark       VARCHAR    │ │
│                                    │  createTime   DATETIME   │ │
│                                    │  updateTime   DATETIME   │ │
│                                    └──────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

### DictTypeDO：字典类型（"分类"）

```java
// 源码位置：yudao-module-system-server/.../dal/dataobject/dict/DictTypeDO.java
@TableName("system_dict_type")
@TenantIgnore  // 全局共享，不按租户隔离
public class DictTypeDO extends BaseDO {
    @TableId
    private Long id;          // 字典主键
    private String name;      // 字典名称（如"订单状态"）
    private String type;      // 字典类型编码（如"order_status"）—— 唯一索引
    private Integer status;   // 状态（启用/禁用）
    private String remark;    // 备注
    private LocalDateTime deletedTime;  // 逻辑删除时间
}
```

DictTypeDO 是字典的"分类目录"。每个 DictType 代表一组同类的映射关系。关键字段：

- **`name`**：人类可读的名称，用于管理界面显示（如"订单状态"、"支付方式"）
- **`type`**：程序使用的唯一编码，是整个字典体系的核心标识。前端和后端都通过这个编码来获取对应的字典数据

### DictDataDO：字典数据（"条目"）

```java
// 源码位置：yudao-module-system-server/.../dal/dataobject/dict/DictDataDO.java
@TableName("system_dict_data")
@TenantIgnore  // 全局共享，不按租户隔离
public class DictDataDO extends BaseDO {
    @TableId
    private Long id;           // 字典数据编号
    private String dictType;   // 字典类型（冗余自 DictTypeDO.type）
    private String label;      // 字典标签（如"待付款"）
    private String value;      // 字典值（如"0"）
    private Integer sort;      // 排序
    private Integer status;    // 状态（启用/禁用）
    private String colorType;  // 颜色类型（Element UI: default/primary/success/info/warning/danger）
    private String cssClass;   // CSS 样式
    private String remark;     // 备注
}
```

DictDataDO 是字典的"具体条目"。每个 DictData 代表一个编码到标签的映射。关键字段：

- **`label`**：显示文本，前端用它来渲染（如"待付款"、"支付宝"）
- **`value`**：编码值，数据库中存储的就是这个值（如"0"、"alipay"）
- **`sort`**：排序号，决定在下拉框中的显示顺序
- **`colorType`**：颜色类型，前端可以直接用它来渲染状态标签的颜色
- **`cssClass`**：自定义 CSS 样式，用于更复杂的渲染需求

## 关键设计决策一：用 type 字符串而非 ID 作外键

这是一个非常有意思的设计选择。传统的关系数据库设计中，外键通常引用主表的 ID：

```
传统设计（用 ID 做外键）：
  dict_type.id = 1, type = "order_status"
  dict_data.dict_type_id = 1   ← 通过 ID 关联

本项目设计（用 type 字符串做外键）：
  dict_type.id = 1, type = "order_status"
  dict_data.dict_type = "order_status"   ← 通过 type 字符串关联
```

为什么选择后者？

```
┌─────────────────────────────────────────────────────────────────┐
│                    两种外键方案对比                                │
├──────────────────────┬──────────────────────────────────────────┤
│                      │  用 ID 做外键     │  用 type 字符串做外键  │
├──────────────────────┼──────────────────┼──────────────────────┤
│ 可读性               │  看到 "1" 不知道  │  看到 "order_status"  │
│                      │  是什么字典       │  立刻知道含义          │
├──────────────────────┼──────────────────┼──────────────────────┤
│ 调试友好             │  需要 JOIN 才能   │  直接在数据表中        │
│                      │  看到字典名称     │  就能看到类型编码      │
├──────────────────────┼──────────────────┼──────────────────────┤
│ 前端使用             │  需要先查 ID 再   │  直接用 type 字符串    │
│                      │  查字典数据       │  获取字典数据          │
├──────────────────────┼──────────────────┼──────────────────────┤
│ 跨环境迁移           │  ID 可能不一致    │  type 字符串天然一致   │
├──────────────────────┼──────────────────┼──────────────────────┤
│ 存储空间             │  8 字节 (BIGINT)  │  可变长度（通常更长）   │
├──────────────────────┼──────────────────┼──────────────────────┤
│ 唯一性保证           │  主键自动保证     │  需要额外唯一索引       │
└──────────────────────┴──────────────────┴──────────────────────┘
```

**核心原因**：字典的使用场景决定了"可读性"和"前端直接使用"比"存储效率"更重要。

前端获取字典数据的典型调用是：`GET /system/dict-data/list-all-simple?dictType=order_status`。如果用 ID 做外键，前端需要先查字典类型列表拿到 ID，再用 ID 去查字典数据——多了一次请求，而且 ID 在不同环境中可能不同（开发环境 ID=1 是订单状态，生产环境 ID=1 可能是支付方式）。

用 type 字符串做外键，type 本身就是业务含义的编码，在所有环境中都是一致的。

## 关键设计决策二：@TenantIgnore 全局共享

字典类型和字典数据都标注了 `@TenantIgnore` 注解：

```java
@TenantIgnore  // DictTypeDO 上
public class DictTypeDO extends BaseDO { ... }

@TenantIgnore  // DictDataDO 上
public class DictDataDO extends BaseDO { ... }
```

这意味着在多租户环境下，所有租户共享同一份字典数据。系统会自动跳过对这些表的租户过滤（不再自动追加 `WHERE tenant_id = ?` 条件）。

为什么这样设计？

```
如果按租户隔离字典会怎样？

  租户 A 的字典：
  ├── order_status: 待付款(0), 已付款(1), 已完成(2)
  └── gender: 男(0), 女(1)

  租户 B 的字典：
  ├── order_status: 待付款(0), 已付款(1), 已发货(2), 已完成(3), 已取消(4)
  └── gender: 男(0), 女(1), 未知(2)

  问题：
  1. 基础字典（如性别）在每个租户都要重复创建，浪费且容易不一致
  2. 框架层代码（如 Excel 导出的字典翻译）需要知道当前租户才能查字典，增加复杂度
  3. 前端缓存需要按租户隔离，内存开销翻倍
```

字典数据（如性别、订单状态）本质上是**系统级元数据**，不属于任何特定租户。把它们设为全局共享，简化了数据管理、缓存策略和跨模块调用。

如果某个租户确实需要自定义的字典条目（比如租户 A 要增加一个"部分发货"的订单状态），可以通过扩展字典数据的租户字段来实现——但在当前设计中，项目选择了全局共享的简洁方案。

## 关键设计决策三：视觉渲染属性

字典数据的 `colorType` 和 `cssClass` 字段是专门为前端渲染设计的。

```
字典数据的视觉属性如何工作：

  数据库中的字典数据：
  ┌────────┬─────────┬───────┬────────────┬──────────┐
  │ label  │ value   │ sort  │ colorType  │ cssClass │
  ├────────┼─────────┼───────┼────────────┼──────────┤
  │ 待付款  │ 0       │ 1     │ warning    │          │
  │ 已付款  │ 1       │ 2     │ primary    │          │
  │ 已发货  │ 2       │ 3     │ primary    │          │
  │ 已完成  │ 3       │ 4     │ success    │          │
  │ 已取消  │ 4       │ 5     │ danger     │          │
  └────────┴─────────┴───────┴────────────┴──────────┘

  前端渲染效果（Element UI 的 el-tag）：
  ┌────────────────────────────────────────┐
  │ 订单号      金额       状态             │
  ├────────────────────────────────────────┤
  │ 10001      ¥99.00    [待付款]  ← 黄色  │
  │ 10002      ¥199.00   [已完成]  ← 绿色  │
  │ 10003      ¥299.00   [已取消]  ← 红色  │
  └────────────────────────────────────────┘

  前端代码：
  <el-tag :type="dictData.colorType">{{ dictData.label }}</el-tag>
```

`colorType` 对应 Element UI 的 `el-tag` 组件的 `type` 属性，支持以下值：

| colorType | 含义 | 视觉效果 |
|-----------|------|---------|
| `default` | 默认 | 灰色 |
| `primary` | 主要 | 蓝色 |
| `success` | 成功 | 绿色 |
| `info` | 信息 | 灰色 |
| `warning` | 警告 | 黄色 |
| `danger` | 危险 | 红色 |

`cssClass` 则提供了更灵活的自定义样式能力，当内置的颜色类型不够用时，可以定义自定义 CSS 类。

这个设计体现了**字典面向显示层**的定位——它不仅仅存储编码到文本的映射，还携带了渲染信息，让前端可以直接使用字典数据来驱动 UI 渲染。

## 关键设计决策四：排序与状态控制

### sort 字段：控制显示顺序

字典数据的 `sort` 字段决定了在下拉框和列表中的显示顺序。在 `DictDataServiceImpl` 中，获取字典数据列表时会按 `dictType` 和 `sort` 双重排序：

```java
// 源码位置：DictDataServiceImpl.java
private static final Comparator<DictDataDO> COMPARATOR_TYPE_AND_SORT = Comparator
        .comparing(DictDataDO::getDictType)
        .thenComparingInt(DictDataDO::getSort);
```

这保证了同一个字典类型下的数据项按 sort 值从小到大排列。管理员可以在后台调整 sort 值来改变显示顺序，比如把"待付款"排在最前面。

### status 字段：启用/禁用控制

字典类型和字典数据都有 `status` 字段（使用 `CommonStatusEnum`，0=开启，1=停用）。这个设计解决了一个实际问题：

```
场景：某电商系统要下线"银行转账"支付方式

  方案一：直接删除
  ├── 问题：历史订单中 pay_type="bank" 的数据将无法显示对应文本
  └── 结果：订单详情页显示空白或原始编码 "bank"

  方案二：禁用（status=1）
  ├── 效果：新订单不再显示"银行转账"选项
  ├── 历史订单仍能正确显示"银行转账"文本
  └── 管理员可以随时重新启用
```

禁用而非删除，是一种"软下线"策略。在 `DictDataServiceImpl.getDictDataList()` 中，调用时传入 `status` 参数，只获取启用状态的字典数据：

```java
// 源码位置：DictDataServiceImpl.java
public List<DictDataDO> getDictDataList(Integer status, String dictType) {
    List<DictDataDO> list = dictDataMapper.selectListByStatusAndDictType(status, dictType);
    list.sort(COMPARATOR_TYPE_AND_SORT);
    return list;
}
```

前端获取下拉框选项时，传入 `status=0`（启用），只看到可用选项；后台管理界面查看全部数据时，传入 `status=null`，可以看到禁用的数据。

## 校验机制

字典服务实现了一套完整的校验机制，确保数据的一致性和完整性。

### 字典类型校验

```
创建字典类型时的校验流程：

  createDictType(createReqVO)
      │
      ├── 1. 校验 name 唯一性
      │   └── validateDictTypeNameUnique(null, name)
      │       ├── 查询：selectByName(name)
      │       ├── 已存在 → 抛出 DICT_TYPE_NAME_DUPLICATE
      │       └── 不存在 → 通过
      │
      ├── 2. 校验 type 唯一性
      │   └── validateDictTypeUnique(null, type)
      │       ├── 查询：selectByType(type)
      │       ├── 已存在 → 抛出 DICT_TYPE_TYPE_DUPLICATE
      │       └── 不存在 → 通过
      │
      └── 3. 插入数据
          └── 设置 deletedTime 为 EMPTY（避免唯一索引的 null 值问题）
```

### 字典数据校验

```
创建字典数据时的校验流程：

  createDictData(createReqVO)
      │
      ├── 1. 校验字典类型存在且启用
      │   └── validateDictTypeExists(dictType)
      │       ├── 查询：dictTypeService.getDictType(type)
      │       ├── 不存在 → 抛出 DICT_TYPE_NOT_EXISTS
      │       ├── 禁用 → 抛出 DICT_TYPE_NOT_ENABLE
      │       └── 存在且启用 → 通过
      │
      ├── 2. 校验 value 在同一类型下唯一
      │   └── validateDictDataValueUnique(null, dictType, value)
      │       ├── 查询：selectByDictTypeAndValue(dictType, value)
      │       ├── 已存在 → 抛出 DICT_DATA_VALUE_DUPLICATE
      │       └── 不存在 → 通过
      │
      └── 3. 插入数据
```

### 删除保护

删除字典类型时，会检查是否有关联的字典数据：

```
删除字典类型时的校验流程：

  deleteDictType(id)
      │
      ├── 1. 校验字典类型存在
      │   └── validateDictTypeExists(id)
      │
      ├── 2. 检查是否有子数据
      │   └── dictDataService.getDictDataCountByDictType(type)
      │       ├── count > 0 → 抛出 DICT_TYPE_HAS_CHILDREN
      │       └── count = 0 → 通过
      │
      └── 3. 逻辑删除
          └── dictTypeMapper.updateToDelete(id, now)  // 不是物理删除
```

注意：字典类型使用的是**逻辑删除**（`updateToDelete` 而非 `deleteById`），通过 `deletedTime` 字段标记删除时间。这与 `@TenantIgnore` 和唯一索引的组合有关——逻辑删除可以避免物理删除后唯一索引的冲突问题。

## 缓存策略

字典数据的读取频率远高于变更频率，是天然的缓存友好型数据。项目在不同层面采用了不同的缓存策略。

### 前端缓存：一次性加载

```
前端缓存策略：

  应用启动时：
  ┌────────┐                    ┌────────────────────┐
  │  前端   │ ── GET ──────────> │ /system/dict-data   │
  │        │     /list-all-simple│ /list-all-simple    │
  │        │ <── 返回全部 ────── │                    │
  │        │     启用的字典数据   │                    │
  └────┬───┘                    └────────────────────┘
       │
       │ 存入本地状态（Vuex/Pinia）
       ▼
  ┌────────────────────────────────────────┐
  │  本地字典缓存                             │
  │                                        │
  │  dictDataMap: {                        │
  │    "order_status": [                   │
  │      { label: "待付款", value: "0", ... },│
  │      { label: "已付款", value: "1", ... },│
  │      ...                               │
  │    ],                                  │
  │    "pay_type": [ ... ],               │
  │    "gender": [ ... ],                 │
  │    ...                                 │
  │  }                                     │
  └────────────────────────────────────────┘

  后续使用：
  1. 渲染下拉框 → 从 dictDataMap["order_status"] 取数据
  2. 渲染状态标签 → 从 dictDataMap["order_status"] 中匹配 value
  3. 导出 Excel → 用 DictFrameworkUtils.parseDictDataLabel() 翻译
```

前端缓存的优势是**零网络开销**——页面切换、表格渲染、下拉框展开都不需要请求后端。代价是首次加载会一次性获取所有字典数据，但字典数据量通常很小（几十到几百条），这个开销可以忽略。

### 后端缓存：Guava LoadingCache

框架层的 `DictFrameworkUtils` 使用 Guava 的 `LoadingCache` 做后端缓存：

```java
// 源码位置：yudao-spring-boot-starter-excel/.../dict/core/DictFrameworkUtils.java
private static final LoadingCache<String, List<DictDataRespDTO>> GET_DICT_DATA_CACHE =
    CacheUtils.buildAsyncReloadingCache(
        Duration.ofMinutes(1L),  // 过期时间 1 分钟
        new CacheLoader<String, List<DictDataRespDTO>>() {
            @Override
            public List<DictDataRespDTO> load(String dictType) {
                return dictDataApi.getDictDataList(dictType).getCheckedData();
            }
        });
```

这个缓存的 key 是字典类型编码（如 `"order_status"`），value 是该类型下的所有字典数据列表。缓存 1 分钟后自动过期重新加载。

```
后端缓存的使用场景：

  Excel 导出时翻译字典值：

  DictFrameworkUtils.parseDictDataLabel("order_status", "0")
      │
      ├── 1. 查缓存：GET_DICT_DATA_CACHE.get("order_status")
      │   ├── 缓存命中 → 直接返回缓存数据
      │   └── 缓存未命中 → 调用 DictDataCommonApi（Feign RPC）
      │                    → 获取数据 → 存入缓存 → 返回
      │
      ├── 2. 在列表中查找 value="0" 的条目
      │   └── 找到 → 返回 label="待付款"
      │
      └── 3. 返回 "待付款"
```

### 缓存失效

当管理员通过后台修改了字典数据，缓存需要失效。项目提供了 `DictFrameworkUtils.clearCache()` 方法：

```java
// 源码位置：DictFrameworkUtils.java
public static void clearCache() {
    GET_DICT_DATA_CACHE.invalidateAll();
}
```

这个方法会清除所有字典缓存。在字典数据变更时（创建、更新、删除），应该调用此方法来确保缓存一致性。由于字典数据变更频率极低（管理员手动操作），全量清除缓存的代价可以接受。

## 前端使用方式全景

让我们完整地走一遍前端使用字典数据的流程。

### 场景一：渲染下拉框

```
需求：订单搜索页面的"订单状态下拉框"

  1. 前端在应用启动时已加载全部字典数据到本地缓存

  2. 订单搜索组件渲染时：
     ┌─────────────────────────────────────────┐
     │ // 前端伪代码                            │
     │ const dictStore = useDictStore()        │
     │ const orderStatusList = dictStore        │
     │     .getDataByType('order_status')      │
     │                                         │
     │ // 渲染下拉框                             │
     │ <el-select v-model="queryForm.status">  │
     │   <el-option                            │
     │     v-for="item in orderStatusList"     │
     │     :key="item.value"                   │
     │     :label="item.label"                 │  ← "待付款"
     │     :value="item.value" />              │  ← "0"
     │ </el-select>                            │
     └─────────────────────────────────────────┘

  3. 用户选择"待付款" → 表单值为 "0" → 提交到后端查询
```

### 场景二：渲染状态标签


## 设计模式提炼

回顾字典类型和字典数据的设计，我们可以识别出几个通用的设计模式：

### 1. 主表-从表模式（Master-Detail）

字典类型是主表，字典数据是从表。这种模式适用于所有"一个分类下有多条明细"的场景，如商品分类-商品、文章分类-文章、部门-员工等。

### 2. 逻辑删除模式

字典类型使用 `deletedTime` 做逻辑删除，而非物理删除。这在需要保留历史关联数据的场景中非常重要——禁用一个字典类型比删除它更安全，因为历史数据中的编码仍然需要翻译。

### 3. 编码即外键模式

用业务编码（`type` 字符串）而非数据库 ID 做外键，牺牲了一点存储效率，换来了更好的可读性和跨环境一致性。这在元数据管理类的表中很常见。

### 4. 多层缓存模式

前端本地缓存（全量、零网络开销）+ 后端 Guava Cache（1 分钟过期、自动刷新）+ 数据库（最终数据源）。三层缓存逐级回退，保证了极高的读取性能。

## 小结

本章深入分析了字典类型和字典数据的核心设计：

- **两层数据模型**：DictTypeDO（分类）和 DictDataDO（条目），一对多关系
- **用 type 字符串做外键**：可读性好、前端直接使用、跨环境一致
- **@TenantIgnore 全局共享**：字典是系统级元数据，不需要按租户隔离
- **视觉渲染属性**：colorType 和 cssClass 让前端可以直接用字典数据驱动 UI
- **排序与状态控制**：sort 控制显示顺序，status 支持启用/禁用的软下线
- **完整的校验机制**：唯一性校验、存在性校验、删除保护
- **多层缓存**：前端本地缓存 + 后端 Guava Cache + 数据库
