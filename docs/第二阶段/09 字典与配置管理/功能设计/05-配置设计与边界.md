# 05-配置设计与边界

## 配置项设计

字典与配置管理模块本身没有独立的配置项（它们本身就是配置的管理者）。

但它们被其他模块通过配置键引用：

```yaml
# 系统配置（通过 infra_config 表管理，非 YAML）
# system.user.init-password = 123456
# system.user.register-enabled = true
# sys.captcha.enable = true
```

## 组件边界

### 字典管理（yudao-module-system）

**做了什么：**
- 字典类型的 CRUD
- 字典数据的 CRUD
- 字典数据的缓存

**不做什么：**
- 字典数据的前端渲染（交给前端）
- 字典数据的业务校验（交给业务模块，如 `validateDictDataList()`）

### 配置管理（yudao-module-infra）

**做了什么：**
- 配置的 CRUD
- 配置值的缓存
- 敏感配置的可见性控制

**不做什么：**
- 配置的生效逻辑（交给使用配置的模块）
- 配置的初始化（通过 SQL 脚本）

### 跨模块引用

```
yudao-module-system
├── AdminUserServiceImpl
│   └── configApi.getConfigValueByKey("system.user.init-password")
│   └── configApi.getConfigValueByKey("system.user.register-enabled")
│
yudao-module-infra
├── ConfigController
├── ConfigServiceImpl
└── ConfigDO
```

system 模块通过 `ConfigApi`（RPC 接口）读取 infra 模块的配置值，实现了模块间的解耦。

## 设计约束

### 1. 字典和配置都是全局共享的

`DictTypeDO`、`DictDataDO`、`ConfigDO` 都使用 `@TenantIgnore`，所有租户共享。这意味着：
- 租户 A 创建的字典类型，租户 B 也能看到
- 系统配置对所有租户生效

如果需要租户级隔离的字典/配置，需要：
1. 移除 `@TenantIgnore`
2. 继承 `TenantBaseDO`
3. 数据按 `tenantId` 隔离

### 2. 字典类型使用软删除

```java
// DictTypeMapper
@Update("UPDATE system_dict_type SET deleted = 1, deleted_time = #{deletedTime} WHERE id = #{id}")
void updateToDelete(@Param("id") Long id, @Param("deletedTime") LocalDateTime deletedTime);
```

字典类型使用软删除（`deleted + deletedTime`），而非物理删除。原因是：
- 字典类型被业务代码引用（通过 `dictType` 字符串）
- 物理删除后，已引用的字典数据会成为孤儿数据
- 软删除保留了数据完整性

### 3. 配置的 configKey 全局唯一

```java
// ConfigServiceImpl.validateConfigKeyUnique()
ConfigDO config = configMapper.selectByKey(key);
if (config != null && !config.getId().equals(id)) {
    throw exception(CONFIG_KEY_DUPLICATE);
}
```

`configKey` 是配置的唯一标识，业务代码通过它读取配置值。重复的 configKey 会导致配置读取混乱。

## 思考题

1. 如果要支持"配置加密"（如数据库密码在数据库中加密存储，读取时自动解密），应该在哪里实现？在 `ConfigDO` 的 getter 中，还是在 `ConfigServiceImpl` 中？
2. 本项目字典数据的 `colorType` 和 `cssClass` 字段是给前端用的。如果前端框架从 Element UI 迁移到 Ant Design Vue，这些字段还适用吗？应该怎么设计更通用？
3. 如果系统有 1000+ 条字典数据，前端每次加载页面都要请求字典接口，如何优化？
