# 01-总览：字典与配置管理全景

## 从两个常见的开发烦恼说起

你正在开发一个电商后台管理系统。产品经理走过来说：

> "订单状态的下拉框，前端那边写死的 `'待付款', '已付款', '已发货'`，现在要加一个 `'已退款'`，你改一下代码重新部署。"

你叹了口气，改了前端代码，打包、部署。第二天，产品经理又来了：

> "对了，我们系统的名字要从'芋道商城'改成'芋道优选'，这个写在哪的？配置文件里？那改完也得重启吧？"

这两个场景，分别代表了开发中两类非常普遍的需求：

1. **"码值映射"**——系统里有大量的"编码 → 显示文本"映射关系（订单状态、支付方式、性别……），它们散落在前端代码、后端枚举、数据库注释里，改一次就要改代码、重新部署。
2. **"系统参数"**——系统运行时需要的各种可调参数（系统名称、注册开关、短信配置、文件上传限制……），它们要么写死在 YAML 配置文件里（改了要重启），要么硬编码在业务代码里（改了要重新打包）。

**字典与配置管理**就是为了解决这两个问题：让"码值映射"和"系统参数"都可以通过后台管理界面动态维护，不需要改代码、不需要重启。

### 字典管理：解决"码值映射"问题

字典管理维护的是**一组一组的"编码-标签"映射**。每组映射有一个唯一的类型标识（`type`），组内有多条数据项，每条数据项包含编码（`value`）和显示文本（`label`）。

在电商系统中，典型的字典类型包括：

| 字典类型 type | 字典名称 | 数据项示例 |
|--------------|---------|-----------|
| `gender` | 性别 | `0→男`, `1→女`, `2→未知` |
| `order_status` | 订单状态 | `0→待付款`, `1→已付款`, `2→已发货`, `3→已完成`, `4→已取消` |
| `pay_type` | 支付方式 | `alipay→支付宝`, `wechat→微信支付`, `bank→银行转账` |
| `product_status` | 商品状态 | `0→下架`, `1→上架` |
| `refund_status` | 退款状态 | `0→无退款`, `1→退款中`, `2→退款成功`, `3→退款拒绝` |

字典数据除了基本的 `label` 和 `value`，还包含一些前端渲染相关的字段：`colorType`（颜色类型，对应 Element UI 的 default/primary/success/info/warning/danger）和 `cssClass`（自定义 CSS 样式）。这让前端可以直接用字典数据来渲染状态标签的颜色，比如"已完成"显示绿色、"已取消"显示红色。

### 参数配置：解决"系统参数"问题

参数配置维护的是**一组键值对（key-value）**，每个配置项有一个唯一的键名（`configKey`）和对应的值（`value`）。

在电商系统中，典型的配置项包括：

| 配置键 configKey | 配置名称 | 配置值 | 类型 |
|-----------------|---------|--------|------|
| `sys.name` | 系统名称 | 芋道优选 | 系统内置 |
| `sys.user.register-able` | 是否开启注册功能 | true | 系统内置 |
| `sms.api.key` | 短信 API Key | sk-xxxxx | 自定义 |
| `file.max-size` | 文件上传大小限制（MB） | 10 | 系统内置 |
| `pay.alipay.app-id` | 支付宝应用 ID | 2021xxxx | 自定义 |

配置项区分**系统内置**（`type = 1`）和**用户自定义**（`type = 2`）。系统内置配置不允许删除，因为系统代码依赖它们；用户自定义配置可以随意增删。

部分配置项涉及敏感信息（如 API Key、密码），通过 `visible` 字段控制是否对前端可见。不可见的配置项，前端无法通过接口获取其值。

## 两个子系统的关系与分工

你可能会问：字典和配置看起来都是 key-value，为什么要分成两个子系统？

```
┌──────────────────────────────────────────────────────────────────────┐
│                        核心区别对比                                    │
├──────────────┬─────────────────────┬─────────────────────────────────┤
│              │  字典（Dict）         │  配置（Config）                  │
├──────────────┼─────────────────────┼─────────────────────────────────┤
│ 数据结构     │  一对多（类型→数据项） │  一对一（key→value）             │
│ 典型用途     │  下拉框、状态标签     │  功能开关、系统参数               │
│ 值的数量     │  一个类型有多条数据   │  一个 key 只有一个 value          │
│ 渲染需求     │  需要颜色、样式信息   │  不需要                          │
│ 排序需求     │  需要自定义排序       │  不需要                          │
│ 跨模块使用   │  所有业务模块都用     │  主要在框架层和 infra 模块使用     │
│ 敏感性       │  一般不敏感          │  部分敏感（API Key、密码）        │
│ 存储位置     │  system 模块         │  infra 模块                      │
└──────────────┴─────────────────────┴─────────────────────────────────┘
```

字典和配置解决的是**两类不同的问题域**：

- **字典**关注的是"显示层"——把编码翻译成人能看懂的文本，带有视觉呈现属性（颜色、样式）
- **配置**关注的是"控制层"——决定系统的行为和能力，带有安全属性（可见性控制）

把它们放在同一个组件里讲解，是因为它们在管理界面上经常放在一起，且都属于"系统管理"的范畴。但在代码层面，它们分别位于不同的模块中：字典在 `yudao-module-system`，配置在 `yudao-module-infra`。

### 字典的跨模块使用

字典的一个关键特点是**跨模块使用**。商品模块需要 `product_status` 字典，订单模块需要 `order_status` 字典，会员模块需要 `member_level` 字典——几乎所有业务模块都会用到字典。

为了解决跨模块调用，项目提供了两种机制：

1. **前端缓存**：`DictDataController` 提供 `/system/dict-data/list-all-simple` 接口，前端启动时一次性加载所有字典数据到本地缓存，后续直接从缓存中查找，不需要每次都请求后端。
2. **RPC 接口**：`DictDataCommonApi`（位于 `yudao-common` 模块）提供 Feign 客户端，其他微服务可以通过 RPC 调用获取字典数据。框架层的 `DictFrameworkUtils` 工具类封装了基于 Guava LoadingCache 的 1 分钟本地缓存。

### 配置的使用方式

配置的使用相对简单。业务代码通过 `ConfigService.getConfigByKey(key)` 获取配置值。框架层代码如果需要读取配置，通常通过 `DictDataCommonApi` 的类似机制（Feign RPC）来跨模块获取。

几个重要的关联关系：

- **与业务模块**：字典被所有业务模块使用。前端在加载页面时，需要根据字典类型获取选项列表来渲染下拉框、状态标签等。
- **与导出导入**：Excel 导出时，需要把数据库中的编码（如 `0`）翻译成可读文本（如"男"）。`DictFrameworkUtils.parseDictDataLabel()` 提供了这个能力。
- **与多租户**：字典类型和配置都标记了 `@TenantIgnore`，意味着它们是**全局共享**的，不按租户隔离。这很合理——"男/女"的映射关系不会因为租户不同而不同。
- **与权限管理**：字典操作需要 `system:dict:*` 权限，配置操作需要 `infra:config:*` 权限，通过 `@PreAuthorize` 注解实现方法级鉴权。

## 小结

本章建立了字典与配置管理的全景认知：

- **字典管理**解决"编码→显示文本"的映射问题，让前端的下拉框、状态标签可以动态配置，不需要改代码
- **参数配置**解决"系统运行参数"的管理问题，让功能开关、集成配置可以在后台动态调整
- 两者虽然经常一起出现，但解决的是不同层面的问题：字典面向"显示层"，配置面向"控制层"
- 字典位于 `yudao-module-system`，配置位于 `yudao-module-infra`，它们都标记了 `@TenantIgnore` 实现全局共享
