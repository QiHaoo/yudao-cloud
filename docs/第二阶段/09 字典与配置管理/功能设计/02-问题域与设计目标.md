# 02-问题域与设计目标

## 一个电商系统的"码值地狱"

想象你接手了一个电商后台管理系统。上线第一个月，一切正常。第二个月，需求开始变了。

### 场景一：订单状态满天飞

最初，订单只有三个状态：待付款、已付款、已完成。前端代码里写死了：

```javascript
// 订单状态
const ORDER_STATUS = {
  0: '待付款',
  1: '已付款',
  2: '已完成'
}
```

后来产品经理说要加"已发货"和"已取消"。你改了前端代码，重新部署。再后来，退款功能上线，又加了"退款中"和"已退款"。你发现订单状态相关的代码散落在至少五个地方：

```
前端散落的订单状态定义：

1. 订单列表页的表格列渲染      → ORDER_STATUS_MAP
2. 订单详情页的状态显示        → getStatusText()
3. 订单搜索的下拉框           → ORDER_STATUS_OPTIONS
4. 订单导出 Excel 的文本转换   → exportStatusText()
5. 移动端 App 的订单状态显示   → orderStatusText（另一套代码）

每个地方都要改，漏一个就是 bug。
```

### 场景二：支付方式的"方言"

支付方式更混乱。数据库里存的是 `alipay`、`wechat`、`bank_transfer`，但不同页面显示的文本不一致：

- 订单列表显示"支付宝"、"微信"、"银行转账"
- 财务报表显示"支付宝支付"、"微信支付"、"银行汇款"
- 导出的 Excel 显示"Alipay"、"WeChat Pay"、"Bank Transfer"

**同一个编码，三种不同的显示文本**，因为是不同开发者在不同时间写的。

### 场景三：系统名称的"重启噩梦"

系统名称写在 `application.yml` 里：

```yaml
sys:
  name: 芋道商城
```

运营说要改成"芋道优选"。你改了 YAML 文件，打包，部署，重启服务。改一个名字，花了 20 分钟。如果线上有用户在操作，重启期间服务不可用。

### 场景四：敏感配置的"泄露风险"

短信服务的 API Key 也写在 YAML 里。但 YAML 文件提交到了 Git 仓库，所有有仓库权限的人都能看到这个 Key。你想把它从 YAML 里移出来，但移出来放在哪？数据库里？那数据库被拖库了怎么办？而且这个值在后台管理界面能被看到吗？如果能，普通管理员也能看到？

## 没有字典和配置管理时的常见做法

让我们系统地梳理一下，在没有统一的字典和配置管理之前，开发者通常怎么处理这些问题。

### 做法一：前端硬编码

```javascript
// 前端代码中直接写死所有状态映射
const ORDER_STATUS_OPTIONS = [
  { value: 0, label: '待付款' },
  { value: 1, label: '已付款' },
  { value: 2, label: '已发货' },
  { value: 3, label: '已完成' },
  { value: 4, label: '已取消' }
]
```

**优点**：简单直接，零依赖，开发速度快。

**缺点**：
- 改一个状态就要改前端代码、重新打包部署
- 多个页面各自维护一份，容易不一致
- 后端无法控制这些映射，前端和后端可能对同一个编码有不同的理解
- 国际化困难——要支持英文怎么办？每个映射都加一套？

### 做法二：后端枚举类

```java
public enum OrderStatusEnum {
    UNPAID(0, "待付款"),
    PAID(1, "已付款"),
    SHIPPED(2, "已发货"),
    COMPLETED(3, "已完成"),
    CANCELLED(4, "已取消");

    private final int code;
    private final String label;
    // ...
}
```

**优点**：
- 类型安全，编译期检查
- 前后端可以用同一套枚举（如果用 TypeScript）
- 重构友好，IDE 能追踪引用

**缺点**：
- 新增状态需要改代码、重新编译部署
- 非技术人员（运营、产品）无法自行维护
- 枚举值和显示文本绑定在一起，不灵活（同一个状态，不同场景可能需要不同的显示文本）
- 无法附加额外属性（如颜色、样式）

### 做法三：配置文件（YAML/Properties）

```yaml
# application.yml
sys:
  name: 芋道商城
  register-able: true
  sms:
    api-key: sk-xxxxx
    api-secret: yyyy
```

**优点**：
- 版本控制友好，变更可追溯
- 不同环境可以用不同的配置文件（dev/test/prod）

**缺点**：
- 修改需要重启服务
- 敏感信息明文存储在文件中，可能被提交到 Git
- 非技术人员无法维护
- 无法在运行时动态调整

**对比枚举方案的优势**：

| 维度 | 枚举方案 | 字典表方案 |
|------|---------|-----------|
| 新增状态 | 改代码、编译、部署 | 后台管理界面添加，即时生效 |
| 维护者 | 只有开发者 | 运营、产品也能维护 |
| 显示文本 | 绑定在枚举中，一处定义 | 可以附加颜色、样式等渲染信息 |
| 跨模块一致性 | 依赖代码引用 | 统一数据源，天然一致 |
| 前端集成 | 需要接口返回枚举 | 提供标准接口，前端缓存 |

**对比硬编码方案的优势**：

- **单一数据源**：所有页面从同一个字典获取数据，不会出现不一致
- **后端可控**：后端决定有哪些状态、每个状态的显示文本，前端只是消费方
- **缓存友好**：字典数据变化频率低，非常适合缓存，性能开销极小

## 为什么分两个子系统而不是一个？

字典和配置表面上都是 key-value，为什么项目把它们分成两个独立的子系统？让我们从数据模型和使用场景两个角度来分析。

### 数据模型的差异

```
字典的数据模型：一对多

  DictType (order_status)
      │
      ├── DictData { label: "待付款", value: "0", sort: 1, colorType: "warning" }
      ├── DictData { label: "已付款", value: "1", sort: 2, colorType: "primary" }
      ├── DictData { label: "已发货", value: "2", sort: 3, colorType: "primary" }
      └── DictData { label: "已完成", value: "3", sort: 4, colorType: "success" }

  特点：
  - 一个类型下有多条数据
  - 数据有排序需求
  - 数据有视觉属性（颜色、样式）
  - 数据有启用/禁用状态


配置的数据模型：一对一

  Config { configKey: "sys.name", value: "芋道优选" }

  特点：
  - 一个 key 只有一个 value
  - 不需要排序
  - 不需要视觉属性
  - 需要敏感数据保护（visible）
  - 需要区分系统内置和用户自定义
```

如果强行合并成一张表，要么字典的"一对多"需求无法满足（一张表无法表达类型和数据项的层级关系），要么配置的"一对一"场景要冗余大量不需要的字段（sort、colorType、cssClass 对配置来说毫无意义）。

### 使用场景的差异

```
字典的典型使用场景：

  1. 前端下拉框
     ┌──────────────────┐
     │ 订单状态 ▼        │
     │ ┌──────────────┐ │
     │ │ 全部          │ │
     │ │ 待付款        │ │  ← 从字典加载
     │ │ 已付款        │ │
     │ │ 已发货        │ │
     │ └──────────────┘ │
     └──────────────────┘

  2. 表格状态标签
     ┌────────┬────────┬──────────┐
     │ 订单号  │ 金额   │ 状态      │
     ├────────┼────────┼──────────┤
     │ 10001  │ ¥99.00 │ [待付款]  │  ← 黄色标签
     │ 10002  │ ¥199.00│ [已完成]  │  ← 绿色标签
     └────────┴────────┴──────────┘

  3. Excel 导出文本转换
     数据库值: 0 → 字典翻译 → "待付款"


配置的典型使用场景：

  1. 功能开关
     if (configService.getConfigByKey("sys.user.register-able").equals("true")) {
         // 允许注册
     }

  2. 集成配置
     String apiKey = configService.getConfigByKey("sms.api.key");
     smsClient.send(apiKey, phone, message);

  3. 系统行为控制
     String systemName = configService.getConfigByKey("sys.name");
     // 用于页面标题、邮件模板等
```

字典的消费者是前端渲染层，配置的消费者是后端业务层。把它们分开，各司其职，代码结构更清晰。

## 设计目标

基于上面的分析，我们为字典与配置管理设定以下设计目标：

### 目标一：运行时可配置

**核心诉求**：修改码值映射和系统参数不需要改代码、不需要重新部署。

这意味着数据必须存储在数据库中（而不是代码或配置文件中），并且提供后台管理界面让运维和产品人员也能操作。

### 目标二：高性能缓存

**核心诉求**：字典和配置的读取频率很高（每次页面加载都可能读取），但变更频率很低。

```
读取频率 vs 变更频率

  字典数据：
  ├── 读取：每次页面加载、每次表格渲染、每次下拉框展开
  ├── 变更：可能几个月才改一次
  └── 缓存策略：全量缓存到前端本地 + 后端 Guava Cache 1 分钟

  配置数据：
  ├── 读取：系统启动时、业务逻辑判断时
  ├── 变更：偶尔调整
  └── 缓存策略：按 key 缓存，变更时清除
```

### 目标三：多租户全局共享

**核心诉求**：字典类型、字典数据、配置项都是全局共享的，不需要按租户隔离。

理由很直观——"男/女"的映射不会因为租户不同而不同，"系统名称"也不会。这也是为什么字典和配置的 DO 类都标注了 `@TenantIgnore`。

```
┌─────────────────────────────────────────────┐
│  租户 A  │  租户 B  │  租户 C               │
│          │          │                       │
│  ┌───────┴──────────┴────────────────────┐  │
│  │       共享的字典和配置                  │  │
│  │       （@TenantIgnore）                │  │
│  │                                       │  │
│  │  order_status: 待付款/已付款/已发货     │  │
│  │  sys.name: 芋道优选                    │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 目标四：敏感数据保护

**核心诉求**：部分配置项（如 API Key、数据库密码）是敏感的，不能暴露给前端。

配置表的 `visible` 字段就是为此设计的。当 `visible = false` 时，前端通过 `/get-value-by-key` 接口获取配置值会直接拒绝。

```
敏感数据保护流程：

  前端请求: GET /infra/config/get-value-by-key?key=sms.api.key

  后端处理:
  1. 查询配置: configKey = "sms.api.key", visible = false
  2. 判断: visible == false?
     ├── 是 → 抛出异常 CONFIG_GET_VALUE_ERROR_IF_VISIBLE
     └── 否 → 返回 value
```

注意：`visible` 控制的是**前端接口**的可见性，后端代码读取配置不受此限制。这是合理的——后端代码是可信的，前端（尤其是面向用户的界面）是不可信的。

### 目标五：类型安全与校验

**核心诉求**：防止脏数据导致系统异常。

字典方面：
- 字典类型的 `type` 字段必须唯一（唯一索引）
- 同一字典类型下的 `value` 必须唯一
- 删除字典类型前，必须确保没有关联的字典数据
- 新增字典数据时，必须校验关联的字典类型存在且启用

配置方面：
- 配置的 `configKey` 必须唯一
- 系统内置配置（`type = 1`）不允许删除
- 用户创建的配置自动标记为 `type = 2`（CUSTOM）

## 小结

本章从四个真实的开发场景出发，分析了没有字典和配置管理时的痛点：

1. **码值映射散落各处**：前端硬编码、后端枚举、数据库注释，改一次要改多个地方
2. **显示文本不一致**：不同页面对同一编码的显示文本不同
3. **配置修改需要重启**：YAML 配置改了要重新打包部署
4. **敏感信息泄露风险**：API Key 等敏感信息明文存储

然后对比了三种常见方案（硬编码、枚举类、配置文件），解释了为什么这个项目选择字典表+配置表的方案：

- **字典表**：灵活、可维护、支持视觉属性、单一数据源
- **配置表**：运行时可调、支持敏感数据保护、区分系统内置和用户自定义

最后设定了五个设计目标：运行时可配置、高性能缓存、多租户全局共享、敏感数据保护、类型安全与校验。

下一章我们将深入字典的核心设计：字典类型和字典数据的数据模型、它们之间的关系，以及缓存策略。
