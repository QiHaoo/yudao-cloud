# 04 - 业务层集成与使用

本章介绍如何在业务层（Controller/Service）中使用数据权限组件，包括 `@DataPermission` 注解的使用、`DataPermissionUtils` 工具类的使用、以及 `deptColumns/userColumns` 的配置方法。

## Controller 层使用 @DataPermission

### 场景一：查询用户详情时禁用数据权限

在查询用户个人中心时，需要禁用数据权限，否则当用户的数据范围是 SELF 时，查询部门信息会被过滤：

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/user/UserProfileController.java
@Tag(name = "管理后台 - 用户个人中心")
@RestController
@RequestMapping("/system/user/profile")
@Validated
@Slf4j
public class UserProfileController {

    @Resource
    private AdminUserService userService;
    @Resource
    private DeptService deptService;
    @Resource
    private PostService postService;
    @Resource
    private PermissionService permissionService;
    @Resource
    private RoleService roleService;

    @GetMapping("/get")
    @Operation(summary = "获得登录用户信息")
    @DataPermission(enable = false) // 关闭数据权限，避免只查看自己时，查询不到部门。
    public CommonResult<UserProfileRespVO> getUserProfile() {
        // 获得用户基本信息
        AdminUserDO user = userService.getUser(getLoginUserId());
        // 获得用户角色
        List<RoleDO> userRoles = roleService.getRoleListFromCache(permissionService.getUserRoleIdListByUserId(user.getId()));
        // 获得部门信息
        DeptDO dept = user.getDeptId() != null ? deptService.getDept(user.getDeptId()) : null;
        // 获得岗位信息
        List<PostDO> posts = CollUtil.isNotEmpty(user.getPostIds()) ? postService.getPostList(user.getPostIds()) : null;
        return success(UserConvert.INSTANCE.convert(user, userRoles, dept, posts));
    }

    // ... 其他方法
}
```

为什么这里需要 `@DataPermission(enable = false)`？

```
问题：如果当前用户的数据范围是 SELF
  → 查询用户表时，会自动追加 WHERE user_id = 当前用户ID
  → 查询部门表时，会自动追加 WHERE dept_id = 当前用户部门ID

  但 getUserProfile() 的目的是获取当前登录用户的完整信息，
  包括其所在部门的详细信息。如果部门表被过滤，可能查不到部门信息。

解决方案：禁用数据权限
  → 允许查询所有部门和用户数据
  → 因为这个接口本身就只返回当前登录用户的信息，不存在越权问题
```

### 场景二：类级别注解 + 方法级别覆盖

```java
// 类级别：作为默认值，对该类所有方法生效
@DataPermission
@RestController
@RequestMapping("/trade/order")
public class OrderController {

    // 方法级别：覆盖类级别的配置，禁用数据权限
    @DataPermission(enable = false)
    @GetMapping("/get")
    public CommonResult<OrderRespVO> getOrder(@RequestParam("id") Long id) {
        // 这个方法不进行数据权限过滤
    }

    // 未标注方法：使用类级别的 @DataPermission（enable = true）
    @GetMapping("/page")
    public CommonResult<PageResult<OrderRespVO>> getOrderPage(OrderPageReqVO reqVO) {
        // 这个方法进行数据权限过滤
    }
}
```

### 场景三：指定规则

```java
// 只使用部门规则，不使用其他自定义规则
@DataPermission(includeRules = DeptDataPermissionRule.class)
public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
    return orderMapper.selectPage(reqVO);
}

// 排除某个自定义规则
@DataPermission(excludeRules = RegionDataPermissionRule.class)
public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
    return orderMapper.selectPage(reqVO);
}
```

## Service 层使用 @DataPermission

### 场景一：获取数据权限时避免递归

在 `PermissionServiceImpl.getDeptDataPermission()` 方法上，需要禁用数据权限，否则会出现无限递归：

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/service/permission/PermissionServiceImpl.java
@Override
@DataPermission(enable = false) // 关闭数据权限，不然就会出现递归获取数据权限的问题
public DeptDataPermissionRespDTO getDeptDataPermission(Long userId) {
    // 获得用户的角色
    List<RoleDO> roles = getEnableUserRoleListByUserIdFromCache(userId);

    // 如果角色为空，则只能查看自己
    DeptDataPermissionRespDTO result = new DeptDataPermissionRespDTO();
    if (CollUtil.isEmpty(roles)) {
        result.setSelf(true);
        return result;
    }

    // 获得用户的部门编号的缓存，通过 Guava 的 Suppliers 惰性求值，即有且仅有第一次发起 DB 的查询
    Supplier<Long> userDeptId = Suppliers.memoize(() -> userService.getUser(userId).getDeptId());
    // 遍历每个角色，计算
    for (RoleDO role : roles) {
        // 为空时，跳过
        if (role.getDataScope() == null) {
            continue;
        }
        // 情况一，ALL
        if (Objects.equals(role.getDataScope(), DataScopeEnum.ALL.getScope())) {
            result.setAll(true);
            continue;
        }
        // 情况二，DEPT_CUSTOM
        if (Objects.equals(role.getDataScope(), DataScopeEnum.DEPT_CUSTOM.getScope())) {
            CollUtil.addAll(result.getDeptIds(), role.getDataScopeDeptIds());
            // 自定义可见部门时，保证可以看到自己所在的部门。否则，一些场景下可能会有问题。
            // 例如说，登录时，基于 t_user 的 username 查询会可能被 dept_id 过滤掉
            CollectionUtils.addIfNotNull(result.getDeptIds(), userDeptId.get());
            continue;
        }
        // 情况三，DEPT_ONLY
        if (Objects.equals(role.getDataScope(), DataScopeEnum.DEPT_ONLY.getScope())) {
            CollectionUtils.addIfNotNull(result.getDeptIds(), userDeptId.get());
            continue;
        }
        // 情况四，DEPT_DEPT_AND_CHILD
        if (Objects.equals(role.getDataScope(), DataScopeEnum.DEPT_AND_CHILD.getScope())) {
            Long deptId = userDeptId.get();
            // 用户未设置部门，直接跳过；否则 getChildDeptIdListFromCache 走缓存注解会因 null key 报错
            if (deptId == null) {
                continue;
            }
            CollUtil.addAll(result.getDeptIds(), deptService.getChildDeptIdListFromCache(deptId));
            // 添加本身部门编号
            result.getDeptIds().add(deptId);
            continue;
        }
        // 情况五，SELF
        if (Objects.equals(role.getDataScope(), DataScopeEnum.SELF.getScope())) {
            result.setSelf(true);
            continue;
        }
        // 未知情况，error log 即可
        log.error("[getDeptDataPermission][LoginUser({}) role({}) 无法处理]", userId, toJsonString(result));
    }
    return result;
}
```

递归问题示意：

```
getDeptDataPermission(userId)
    │
    ▼
查询用户角色 → userRoleMapper.selectListByUserId(userId)
    │
    ▼
DataPermissionRuleHandler 拦截 SQL
    │
    ▼
需要获取当前用户的数据权限 → getDeptDataPermission(userId)  ← 递归！
    │
    ▼
无限循环...

解决方案：@DataPermission(enable = false)
    → 在 getDeptDataPermission() 内部的 SQL 不会被数据权限拦截
    → 避免递归
```

## DataPermissionUtils 工具类使用

除了注解方式，yudao-cloud 还提供了编程式禁用数据权限的能力：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/util/DataPermissionUtils.java
package cn.iocoder.yudao.framework.datapermission.core.util;

import cn.iocoder.yudao.framework.datapermission.core.annotation.DataPermission;
import cn.iocoder.yudao.framework.datapermission.core.aop.DataPermissionContextHolder;
import lombok.SneakyThrows;

import java.util.concurrent.Callable;

/**
 * 数据权限 Util
 *
 * @author 芋道源码
 */
public class DataPermissionUtils {

    private static DataPermission DATA_PERMISSION_DISABLE;

    @DataPermission(enable = false)
    @SneakyThrows
    private static DataPermission getDisableDataPermissionDisable() {
        if (DATA_PERMISSION_DISABLE == null) {
            DATA_PERMISSION_DISABLE = DataPermissionUtils.class
                    .getDeclaredMethod("getDisableDataPermissionDisable")
                    .getAnnotation(DataPermission.class);
        }
        return DATA_PERMISSION_DISABLE;
    }

    /**
     * 忽略数据权限，执行对应的逻辑
     *
     * @param runnable 逻辑
     */
    public static void executeIgnore(Runnable runnable) {
        addDisableDataPermission();
        try {
            // 执行 runnable
            runnable.run();
        } finally {
            removeDataPermission();
        }
    }

    /**
     * 忽略数据权限，执行对应的逻辑
     *
     * @param callable 逻辑
     * @return 执行结果
     */
    @SneakyThrows
    public static <T> T executeIgnore(Callable<T> callable) {
        addDisableDataPermission();
        try {
            // 执行 callable
            return callable.call();
        } finally {
            removeDataPermission();
        }
    }

    /**
     * 添加忽略数据权限
     */
    public static void addDisableDataPermission(){
        DataPermission dataPermission = getDisableDataPermissionDisable();
        DataPermissionContextHolder.add(dataPermission);
    }

    public static void removeDataPermission(){
        DataPermissionContextHolder.remove();
    }

}
```

### 使用场景

#### 场景一：在 Lambda 或匿名类中禁用数据权限

```java
public void someMethod() {
    DataPermissionUtils.executeIgnore(() -> {
        // 这段代码执行时，数据权限被禁用
        List<OrderDO> orders = orderMapper.selectList(...);
        processOrders(orders);
    });
}
```

#### 场景二：需要返回值的场景

```java
public OrderDO getOrder(Long orderId) {
    return DataPermissionUtils.executeIgnore(() -> {
        return orderMapper.selectById(orderId);
    });
}
```

#### 场景三：手动控制上下文（不推荐，容易遗漏 remove）

```java
public void complexMethod() {
    DataPermissionUtils.addDisableDataPermission();
    try {
        // 复杂的业务逻辑
    } finally {
        DataPermissionUtils.removeDataPermission();
    }
}
```

### @DataPermission vs DataPermissionUtils 选择指南

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| Controller 方法 | `@DataPermission` | 声明式，清晰直观 |
| Service 方法 | `@DataPermission` | 声明式，AOP 自动管理 |
| Lambda/匿名类 | `DataPermissionUtils.executeIgnore()` | 无法使用注解 |
| 条件性禁用 | `DataPermissionUtils.executeIgnore()` | 需要运行时判断 |
| 复杂嵌套逻辑 | `DataPermissionUtils` 手动管理 | 需要精确控制入栈出栈时机 |

## 配置 deptColumns/userColumns 映射

### 通过 DeptDataPermissionRuleCustomizer 配置

`DeptDataPermissionRuleCustomizer` 是一个函数式接口，用于配置 `DeptDataPermissionRule` 的表字段映射：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rule/dept/DeptDataPermissionRuleCustomizer.java
@FunctionalInterface
public interface DeptDataPermissionRuleCustomizer {

    /**
     * 自定义 {@link DeptDataPermissionRule} 的配置
     *
     * @param rule 数据权限规则
     */
    void customize(DeptDataPermissionRule rule);

}
```

### system 模块配置示例

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/framework/datapermission/config/DataPermissionConfiguration.java
@Configuration(proxyBeanMethods = false)
public class DataPermissionConfiguration {

    @Bean
    public DeptDataPermissionRuleCustomizer sysDeptDataPermissionRuleCustomizer() {
        return rule -> {
            // dept
            rule.addDeptColumn(AdminUserDO.class);  // 用户表，部门字段默认为 dept_id
            rule.addDeptColumn(DeptDO.class, "id");  // 部门表，部门字段为 id（特殊：部门表的主键就是部门编号）
            // user
            rule.addUserColumn(AdminUserDO.class, "id");  // 用户表，用户字段为 id（特殊：用户表的主键就是用户编号）
        };
    }

}
```

### 业务模块配置示例（电商）

假设电商模块有订单表 `trade_order`，需要按部门和用户过滤：

```java
@Configuration(proxyBeanMethods = false)
public class TradeDataPermissionConfiguration {

    @Bean
    public DeptDataPermissionRuleCustomizer tradeDeptDataPermissionRuleCustomizer() {
        return rule -> {
            // 订单表：部门字段为 dept_id，用户字段为 user_id
            rule.addDeptColumn(TradeOrderDO.class);
            rule.addUserColumn(TradeOrderDO.class);
            
            // 客户表：部门字段为 dept_id，用户字段为 creator
            rule.addDeptColumn(TradeCustomerDO.class);
            rule.addUserColumn(TradeCustomerDO.class, "creator");
        };
    }

}
```

### 配置方式说明

| 方法 | 作用 | 示例 |
|------|------|------|
| `addDeptColumn(Class<?> entityClass)` | 注册部门字段（默认 `dept_id`） | `rule.addDeptColumn(OrderDO.class)` |
| `addDeptColumn(Class<?> entityClass, String column)` | 注册部门字段（自定义列名） | `rule.addDeptColumn(DeptDO.class, "id")` |
| `addDeptColumn(String tableName, String column)` | 直接指定表名和字段名 | `rule.addDeptColumn("trade_order", "dept_id")` |
| `addUserColumn(Class<?> entityClass)` | 注册用户字段（默认 `user_id`） | `rule.addUserColumn(OrderDO.class)` |
| `addUserColumn(Class<?> entityClass, String column)` | 注册用户字段（自定义列名） | `rule.addUserColumn(OrderDO.class, "creator")` |
| `addUserColumn(String tableName, String column)` | 直接指定表名和字段名 | `rule.addUserColumn("trade_order", "user_id")` |

## 完整使用流程

### 第一步：配置表字段映射

在业务模块中创建 `DataPermissionConfiguration`，配置需要数据权限过滤的表和字段：

```java
@Configuration(proxyBeanMethods = false)
public class TradeDataPermissionConfiguration {

    @Bean
    public DeptDataPermissionRuleCustomizer tradeDeptDataPermissionRuleCustomizer() {
        return rule -> {
            rule.addDeptColumn(TradeOrderDO.class);
            rule.addUserColumn(TradeOrderDO.class);
        };
    }

}
```

### 第二步：在 Controller/Service 中使用注解

```java
@RestController
@RequestMapping("/trade/order")
public class OrderController {

    @GetMapping("/page")
    @DataPermission  // 启用数据权限（默认）
    public CommonResult<PageResult<OrderRespVO>> getOrderPage(OrderPageReqVO reqVO) {
        // 查询订单列表，会自动过滤当前用户可见的数据
        return success(orderService.getOrderPage(reqVO));
    }

    @GetMapping("/get")
    @DataPermission(enable = false)  // 禁用数据权限
    public CommonResult<OrderRespVO> getOrder(@RequestParam("id") Long id) {
        // 查询订单详情，不过滤
        return success(orderService.getOrder(id));
    }

}
```

### 第三步：SQL 自动改写

当执行 SQL 时，MyBatis-Plus 的 `DataPermissionInterceptor` 会自动拦截并改写 SQL：

```sql
-- 原始 SQL
SELECT * FROM trade_order WHERE status = 1

-- 改写后的 SQL（假设当前用户可见部门：[100, 101]，用户 ID：42）
SELECT * FROM trade_order WHERE status = 1 AND (dept_id IN (100, 101) OR user_id = 42)
```

## 思考题

1. 在 `PermissionServiceImpl.getDeptDataPermission()` 方法上使用 `@DataPermission(enable = false)` 是为了避免递归。如果将来需要在这个方法内部启用数据权限（比如查询其他表），你会如何设计？

2. `DataPermissionUtils.executeIgnore()` 方法使用 `try-finally` 结构确保数据权限上下文被正确恢复。如果在 `runnable.run()` 或 `callable.call()` 抛出异常，会发生什么？这种设计合理吗？

3. 如果一个业务模块需要同时使用"基于部门"和"基于客户等级"两种数据权限规则，你会如何配置？需要创建几个 `DeptDataPermissionRuleCustomizer`？
