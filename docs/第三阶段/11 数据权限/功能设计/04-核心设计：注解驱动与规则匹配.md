# 04 - 核心设计：注解驱动与规则匹配

## 从一个实际问题开始

假设你在开发电商后台的"用户个人中心"页面。用户点击"个人信息"时，系统需要查询用户的基本信息、所属部门、角色列表。

问题来了：如果当前用户是"销售"角色（数据范围 SELF），当他查询自己的部门信息时，`DeptDataPermissionRule` 会自动追加 `WHERE dept_id = ?` 条件。但部门表查询的是 `system_dept` 表，如果用当前用户的 dept_id 去过滤，可能查不到其他部门的信息。

更严重的是：**如果查询逻辑中需要获取数据权限本身（`getDeptDataPermission()`），而获取数据权限又需要查询用户表，用户表又被数据权限过滤——这就形成了无限递归**。

这就是 `@DataPermission` 注解存在的意义：**在需要的地方，可以精确控制数据权限的行为**。

## @DataPermission 注解详解

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/annotation/DataPermission.java
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface DataPermission {

    /**
     * 当前类或方法是否开启数据权限
     * 即使不添加 @DataPermission 注解，默认是开启状态
     * 可通过设置 enable 为 false 禁用
     */
    boolean enable() default true;

    /**
     * 生效的数据权限规则数组，优先级高于 excludeRules
     */
    Class<? extends DataPermissionRule>[] includeRules() default {};

    /**
     * 排除的数据权限规则数组，优先级最低
     */
    Class<? extends DataPermissionRule>[] excludeRules() default {};
}
```

三个属性的作用：

| 属性 | 默认值 | 作用 | 优先级 |
|------|-------|------|--------|
| `enable` | `true` | 控制数据权限是否生效 | 最高 |
| `includeRules` | `{}` (空) | 指定生效的规则（白名单模式） | 中 |
| `excludeRules` | `{}` (空) | 指定排除的规则（黑名单模式） | 低 |

### 注解的生效范围

`@DataPermission` 可以标注在**类**或**方法**上：

```java
// 类级别：作为默认值，对该类所有方法生效
@DataPermission
@RestController
@RequestMapping("/trade/order")
public class OrderController {

    // 方法级别：覆盖类级别的配置
    @DataPermission(enable = false)
    @GetMapping("/get")
    public CommonResult<OrderRespVO> getOrder(@RequestParam("id") Long id) {
        // 这个方法不进行数据权限过滤
    }

    // 未标注方法：使用类级别的 @DataPermission（enable = true）
    @GetMapping("/page")
    public CommonResult<PageResult<OrderRespVO>> getOrderPage(OrderPageReqVO reqVO) {
        // 这个方法进行数据权限过滤
    }
}
```

**方法级别的注解优先于类级别**。这是一个经典的"就近原则"设计。

## AOP 拦截机制

`@DataPermission` 注解本身只是"声明"，真正让它生效的是 AOP 拦截机制。yudao-cloud 实现了两个核心类：

- `DataPermissionAnnotationAdvisor`：定义切点（哪些方法需要拦截）
- `DataPermissionAnnotationInterceptor`：定义通知（拦截后做什么）

### DataPermissionAnnotationAdvisor：切点定义

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/aop/DataPermissionAnnotationAdvisor.java
@Getter
@EqualsAndHashCode(callSuper = true)
public class DataPermissionAnnotationAdvisor extends AbstractPointcutAdvisor {

    private final Advice advice;
    private final Pointcut pointcut;

    public DataPermissionAnnotationAdvisor() {
        this.advice = new DataPermissionAnnotationInterceptor();
        this.pointcut = this.buildPointcut();
    }

    protected Pointcut buildPointcut() {
        // 切点一：类上有 @DataPermission 注解
        Pointcut classPointcut = new AnnotationMatchingPointcut(DataPermission.class, true);
        // 切点二：方法上有 @DataPermission 注解
        Pointcut methodPointcut = new AnnotationMatchingPointcut(null, DataPermission.class, true);
        // 两者的并集
        return new ComposablePointcut(classPointcut).union(methodPointcut);
    }
}
```

切点的含义是：**只要类或方法上有 `@DataPermission` 注解，就会被拦截**。

### DataPermissionAnnotationInterceptor：拦截逻辑

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/aop/DataPermissionAnnotationInterceptor.java
@DataPermission // 该注解用于 DATA_PERMISSION_NULL 空对象
public class DataPermissionAnnotationInterceptor implements MethodInterceptor {

    /**
     * 空对象，用于方法无 @DataPermission 注解时占位
     */
    static final DataPermission DATA_PERMISSION_NULL =
            DataPermissionAnnotationInterceptor.class.getAnnotation(DataPermission.class);

    private final Map<MethodClassKey, DataPermission> dataPermissionCache = new ConcurrentHashMap<>();

    @Override
    public Object invoke(MethodInvocation methodInvocation) throws Throwable {
        // 入栈：将 @DataPermission 注解压入上下文栈
        DataPermission dataPermission = this.findAnnotation(methodInvocation);
        if (dataPermission != null) {
            DataPermissionContextHolder.add(dataPermission);
        }
        try {
            // 执行业务方法
            return methodInvocation.proceed();
        } finally {
            // 出栈：将 @DataPermission 注解从上下文栈弹出
            if (dataPermission != null) {
                DataPermissionContextHolder.remove();
            }
        }
    }

    private DataPermission findAnnotation(MethodInvocation methodInvocation) {
        // 1. 从缓存中获取
        Method method = methodInvocation.getMethod();
        Object targetObject = methodInvocation.getThis();
        Class<?> clazz = targetObject != null ? targetObject.getClass() : method.getDeclaringClass();
        MethodClassKey methodClassKey = new MethodClassKey(method, clazz);
        DataPermission dataPermission = dataPermissionCache.get(methodClassKey);
        if (dataPermission != null) {
            return dataPermission != DATA_PERMISSION_NULL ? dataPermission : null;
        }

        // 2.1 先从方法上查找
        dataPermission = AnnotationUtils.findAnnotation(method, DataPermission.class);
        // 2.2 方法上没有，再从类上查找
        if (dataPermission == null) {
            dataPermission = AnnotationUtils.findAnnotation(clazz, DataPermission.class);
        }
        // 2.3 添加到缓存（使用 DATA_PERMISSION_NULL 占位，避免重复查找）
        dataPermissionCache.put(methodClassKey,
                dataPermission != null ? dataPermission : DATA_PERMISSION_NULL);
        return dataPermission;
    }
}
```

### 拦截流程图

```
业务方法被调用
    │
    ▼
DataPermissionAnnotationInterceptor.invoke()
    │
    ▼
findAnnotation(methodInvocation)
    │
    ▼
缓存中有？ ──是──▶ 返回缓存的注解
    │否
    ▼
方法上有 @DataPermission？ ──是──▶ 缓存并返回
    │否
    ▼
类上有 @DataPermission？ ──是──▶ 缓存并返回
    │否
    ▼
缓存 DATA_PERMISSION_NULL 占位，返回 null
    │
    ▼

    │
    ▼

    │
    ▼
finally: 注解不为 null？ ──是──▶ DataPermissionContextHolder.remove()
```

## DataPermissionContextHolder：上下文管理

`DataPermissionContextHolder` 是数据权限的"记忆"，它使用 `ThreadLocal` 存储当前方法的数据权限注解信息。

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/aop/DataPermissionContextHolder.java
public class DataPermissionContextHolder {

    /**
     * 使用 List 的原因：可能存在方法的嵌套调用
     */
    private static final ThreadLocal<LinkedList<DataPermission>> DATA_PERMISSIONS =
            TransmittableThreadLocal.withInitial(LinkedList::new);

    /**
     * 获得当前的 DataPermission 注解（栈顶）
     */
    public static DataPermission get() {
        return DATA_PERMISSIONS.get().peekLast();
    }

    /**
     * 入栈 DataPermission 注解
     */
    public static void add(DataPermission dataPermission) {
        DATA_PERMISSIONS.get().addLast(dataPermission);
    }

    /**
     * 出栈 DataPermission 注解
     */
    public static DataPermission remove() {
        DataPermission dataPermission = DATA_PERMISSIONS.get().removeLast();
        // 无元素时，清空 ThreadLocal
        if (DATA_PERMISSIONS.get().isEmpty()) {
            DATA_PERMISSIONS.remove();
        }
        return dataPermission;
    }
}
```

### 为什么用栈而不是单个值

关键在于**方法嵌套调用**的场景：

```java
@Service
public class OrderService {

    @DataPermission(enable = true)  // 注解 A
    public PageResult<OrderDO> getOrderPage() {
        // 内部调用了另一个有数据权限注解的方法
        return someHelper.process();
    }
}

@Service
public class SomeHelper {

    @DataPermission(enable = false)  // 注解 B
    public PageResult<OrderDO> process() {
        // 这里应该使用注解 B（enable = false）
        return orderMapper.selectPage(...);
    }
}
```

嵌套调用时的栈变化：

```
1. getOrderPage() 被调用
   栈：[A(enable=true)]

2. AOP 拦截，压入注解 A
   栈：[A(enable=true)]

3. process() 被调用
   栈：[A(enable=true)]

4. AOP 拦截，压入注解 B
   栈：[A(enable=true), B(enable=false)]

5. DataPermissionContextHolder.get() 返回 B（栈顶）
   → 数据权限被禁用 ✓

6. process() 执行完毕，弹出注解 B
   栈：[A(enable=true)]

7. getOrderPage() 执行完毕，弹出注解 A
   栈：[]
```

### 为什么使用 TransmittableThreadLocal

普通的 `ThreadLocal` 在线程池场景下会有问题：

```
问题场景：
  主线程设置了 @DataPermission(enable = false)
  然后提交了一个异步任务到线程池
  异步任务的线程无法获取主线程的数据权限上下文

解决方案：
  使用 TransmittableThreadLocal（阿里开源）
  它可以自动将 ThreadLocal 的值传递到子线程
```

## DataPermissionUtils：编程式禁用数据权限

除了注解方式，yudao-cloud 还提供了编程式禁用数据权限的能力：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/util/DataPermissionUtils.java
public class DataPermissionUtils {

    private static DataPermission DATA_PERMISSION_DISABLE;

    /**
     * 获取一个 enable = false 的注解实例（单例）
     */
    @DataPermission(enable = false)
    @SneakyThrows
    private static DataPermission getDisableDataPermissionDisable() {
        if (DATA_PERMISSION_DISABLE == null) {
            DATA_PERMISSION_DISABLE = DataPermissionUtils.class
                    .getDeclaredMethod("getDisableDataPermissionDisable")
                    .getAnnotation(DataPermission.class);
        }
        return DATA_PERMISSION_DISABLE;
    }

    /**
     * 忽略数据权限，执行对应的逻辑
     */
    public static void executeIgnore(Runnable runnable) {
        addDisableDataPermission();
        try {
            runnable.run();
        } finally {
            removeDataPermission();
        }
    }

    /**
     * 忽略数据权限，执行对应的逻辑（带返回值）
     */
    @SneakyThrows
    public static <T> T executeIgnore(Callable<T> callable) {
        addDisableDataPermission();
        try {
            return callable.call();
        } finally {
            removeDataPermission();
        }
    }

    /**
     * 添加忽略数据权限
     */
    public static void addDisableDataPermission() {
        DataPermission dataPermission = getDisableDataPermissionDisable();
        DataPermissionContextHolder.add(dataPermission);
    }

    /**
     * 移除数据权限上下文
     */
    public static void removeDataPermission() {
        DataPermissionContextHolder.remove();
    }
}
```

### 使用场景

`DataPermissionUtils.executeIgnore()` 适用于无法使用注解的场景：

```java
// 场景一：在 Lambda 或匿名类中禁用数据权限
public void someMethod() {
    DataPermissionUtils.executeIgnore(() -> {
        // 这段代码执行时，数据权限被禁用
        List<OrderDO> orders = orderMapper.selectList(...);
        processOrders(orders);
    });
}

// 场景二：需要返回值的场景
public OrderDO getOrder(Long orderId) {
    return DataPermissionUtils.executeIgnore(() -> {
        return orderMapper.selectById(orderId);
    });
}

// 场景三：手动控制上下文（不推荐，容易遗漏 remove）
public void complexMethod() {
    DataPermissionUtils.addDisableDataPermission();
    try {
        // 复杂的业务逻辑
    } finally {
        DataPermissionUtils.removeDataPermission();
    }
}
```

### @DataPermission vs DataPermissionUtils 选择指南

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| Controller 方法 | `@DataPermission` | 声明式，清晰直观 |
| Service 方法 | `@DataPermission` | 声明式，AOP 自动管理 |
| Lambda/匿名类 | `DataPermissionUtils.executeIgnore()` | 无法使用注解 |
| 条件性禁用 | `DataPermissionUtils.executeIgnore()` | 需要运行时判断 |
| 复杂嵌套逻辑 | `DataPermissionUtils` 手动管理 | 需要精确控制入栈出栈时机 |

## skipPermissionCheck()：跨服务调用时跳过检查

在微服务架构中，跨服务调用时需要跳过数据权限检查。这个判断在 `DataPermissionRuleHandler.getSqlSegment()` 的最开始：

```java
@Override
public Expression getSqlSegment(Table table, Expression where, String mappedStatementId) {
    // 特殊：跨租户访问时跳过
    if (skipPermissionCheck()) {
        return null;
    }
    // ...
}
```

`skipPermissionCheck()` 的实现：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-security/src/main/java/cn/iocoder/yudao/framework/security/core/util/SecurityFrameworkUtils.java
public static boolean skipPermissionCheck() {
    LoginUser loginUser = getLoginUser();
    if (loginUser == null) {
        return false;
    }
    // visitTenantId 为空，说明是跨租户访问（如系统级别的操作）
    if (loginUser.getVisitTenantId() == null) {
        return true;
    }
    return false;
}
```

## 典型使用场景

### 场景一：查询用户详情时跳过数据权限

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/user/UserProfileController.java
@GetMapping("/get")
@Operation(summary = "获得登录用户信息")
@DataPermission(enable = false) // 关闭数据权限，避免只查看自己时，查询不到部门
public CommonResult<UserProfileRespVO> getUserProfile() {
    // 获取用户基本信息
    AdminUserDO user = userService.getUser(getLoginUserId());
    // 获取用户角色
    List<RoleDO> userRoles = roleService.getRoleListFromCache(
            permissionService.getUserRoleIdListByUserId(user.getId()));
    // 获取部门信息
    DeptDO dept = user.getDeptId() != null ? deptService.getDept(user.getDeptId()) : null;
    // ...
}
```

为什么这里需要 `@DataPermission(enable = false)`？

```
问题：如果当前用户的数据范围是 SELF
  → 查询用户表时，会自动追加 WHERE user_id = 当前用户ID
  → 查询部门表时，会自动追加 WHERE dept_id = 当前用户部门ID

  但 getUserProfile() 的目的是获取当前登录用户的完整信息，
  包括其所在部门的详细信息。如果部门表被过滤，可能查不到部门信息。

解决方案：禁用数据权限
  → 允许查询所有部门和用户数据
  → 因为这个接口本身就只返回当前登录用户的信息，不存在越权问题
```

### 场景二：获取数据权限时避免递归

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/service/permission/PermissionServiceImpl.java
@Override
@DataPermission(enable = false) // 关闭数据权限，不然就会出现递归获取数据权限的问题
public DeptDataPermissionRespDTO getDeptDataPermission(Long userId) {
    // 获取用户角色
    List<RoleDO> roles = getEnableUserRoleListByUserIdFromCache(userId);
    // 遍历角色，计算数据权限...
}
```

递归问题示意：

```
getDeptDataPermission(userId)
    │
    ▼
查询用户角色 → userRoleMapper.selectListByUserId(userId)
    │
    ▼
DataPermissionRuleHandler 拦截 SQL
    │
    ▼
需要获取当前用户的数据权限 → getDeptDataPermission(userId)  ← 递归！
    │
    ▼
无限循环...

解决方案：@DataPermission(enable = false)
    → 在 getDeptDataPermission() 内部的 SQL 不会被数据权限拦截
    → 避免递归
```

## 注解缓存机制

`DataPermissionAnnotationInterceptor` 使用 `ConcurrentHashMap` 缓存注解查找结果，避免每次方法调用都进行反射查找：

```java
private final Map<MethodClassKey, DataPermission> dataPermissionCache = new ConcurrentHashMap<>();
```

缓存策略：
- 首次查找：通过 `AnnotationUtils.findAnnotation()` 进行反射查找
- 缓存命中：直接返回缓存结果
- 未找到注解：使用 `DATA_PERMISSION_NULL` 占位，避免重复查找

这个设计的精妙之处在于：**用空间换时间，一次反射查找，终身缓存**。

## 思考题

1. `DataPermissionContextHolder` 使用 `LinkedList` 作为栈结构。如果在同一个方法中多次调用 `add()` 但没有对应的 `remove()`（比如异常路径），会导致什么问题？当前代码是如何处理的？

2. `DataPermissionUtils.getDisableDataPermissionDisable()` 方法使用反射获取注解实例，而不是直接 `new` 一个匿名注解。为什么？直接创建一个实现 `DataPermission` 接口的匿名类不行吗？

3. 在微服务场景中，`@DataPermission(enable = false)` 的传播只传递了 `enable` 属性。如果将来需要传播 `includeRules` 和 `excludeRules`，你会如何设计？
