# 05 - 核心设计：跨服务数据权限传播

## 问题：微服务中的数据权限困境

在单体应用中，数据权限的上下文（`DataPermissionContextHolder`）存储在 `ThreadLocal` 中，整个请求链路在同一个线程内完成，上下文自然可用。

但微服务架构下，情况完全不同：

```
浏览器请求
    │
    ▼
订单服务 (Service A)
    │
    │  @DataPermission(enable = false)
    │  需要调用商品服务查询商品信息
    │
    ▼ Feign 调用
商品服务 (Service B)
    │
    │  收到 HTTP 请求
    │  此时 DataPermissionContextHolder 是空的！
    │  数据权限会按默认规则生效
    │
    ▼
商品 Mapper 查询
    │
    │  SQL 被数据权限拦截，追加了 dept_id 过滤条件
    │  但订单服务已经禁用了数据权限，这里不应该过滤！
    │
    ▼
返回被错误过滤的数据
```

问题的根源：**`ThreadLocal` 只在当前线程内有效，跨服务的 HTTP 调用会创建新的线程**。

## 解决方案：HTTP Header 传播

yudao-cloud 的解决方案是**通过 HTTP Header 传播数据权限上下文**。整个流程分为三个环节：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    跨服务数据权限传播流程                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Service A (调用方)                                                │
│   ┌─────────────────────────────────────────┐                       │
│   │ 1. Feign 发起请求前                       │                       │
│   │    DataPermissionRequestInterceptor      │                       │
│   │    读取 DataPermissionContextHolder       │                       │
│   │    如果 enable=false，设置 header          │                       │
│   │    data-permission-enable: false          │                       │
│   └─────────────────────┬───────────────────┘                       │
│                         │                                           │
│                         ▼ HTTP 请求                                 │
│   Service B (被调用方)                                              │
│   ┌─────────────────────────────────────────┐                       │
│   │ 2. 收到请求后，Filter 执行前              │                       │
│   │    DataPermissionRpcWebFilter            │                       │
│   │    读取 header: data-permission-enable   │                       │
│   │    如果是 false，调用 DataPermissionUtils │                       │
│   │    .executeIgnore() 禁用数据权限          │                       │
│   └─────────────────────┬───────────────────┘                       │
│                         │                                           │
│                         ▼                                           │
│   ┌─────────────────────────────────────────┐                       │
│   │ 3. Service B 的业务代码执行                │                       │
│   │    DataPermissionContextHolder 中已有     │                       │
│   │    enable=false 的注解                    │                       │
│   │    SQL 改写时会跳过数据权限                │                       │
│   └─────────────────────────────────────────┘                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## DataPermissionRequestInterceptor：发送端

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rpc/DataPermissionRequestInterceptor.java
public class DataPermissionRequestInterceptor implements RequestInterceptor {

    public static final String ENABLE_HEADER_NAME = "data-permission-enable";

    @Override
    public void apply(RequestTemplate requestTemplate) {
        // 获取当前上下文中的 DataPermission 注解
        DataPermission dataPermission = DataPermissionContextHolder.get();
        // 如果注解存在且 enable=false，通过 header 传递给被调用方
        if (dataPermission != null && Boolean.FALSE.equals(dataPermission.enable())) {
            requestTemplate.header(ENABLE_HEADER_NAME, "false");
        }
    }
}
```

关键设计点：

1. **只传播 `enable = false` 的情况**：默认情况下数据权限是开启的，只有需要禁用时才需要通知对方
2. **只传播 `enable` 属性**：`includeRules` 和 `excludeRules` 不传播，因为这些通常是调用方的本地需求
3. **使用 Feign `RequestInterceptor`**：这是 Feign 框架提供的标准扩展点，在请求发送前自动执行

### 为什么只传播 enable 而不传播完整注解

```
方案一：只传播 enable（yudao-cloud 的选择）
  Header: data-permission-enable: false

  优点：
  - 简单，一个 header 就够
  - 被调用方的行为清晰：要么禁用，要么使用默认规则

  缺点：
  - 无法传递 includeRules/excludeRules

方案二：传播完整注解
  Header: data-permission: {"enable":false,"includeRules":[...],"excludeRules":[...]}

  优点：
  - 可以精确控制被调用方的行为

  缺点：
  - 注解中的 Class 类型无法序列化为 JSON
  - 被调用方可能没有调用方的规则类，反序列化会失败
  - 增加了系统的耦合度
```

yudao-cloud 选择方案一，因为**跨服务的数据权限控制应该尽量简单**。复杂的数据权限逻辑应该在各自服务内部处理。

## DataPermissionRpcWebFilter：接收端

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rpc/DataPermissionRpcWebFilter.java
public class DataPermissionRpcWebFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        // 读取 header 中的数据权限标记
        String enable = request.getHeader(DataPermissionRequestInterceptor.ENABLE_HEADER_NAME);

        if (Objects.equals(enable, Boolean.FALSE.toString())) {
            // 调用方禁用了数据权限，本服务也禁用
            DataPermissionUtils.executeIgnore(() -> {
                try {
                    chain.doFilter(request, response);
                } catch (IOException | ServletException e) {
                    throw new RuntimeException(e);
                }
            });
        } else {
            // 正常执行，数据权限按默认规则生效
            chain.doFilter(request, response);
        }
    }
}
```

关键设计点：

1. **使用 `OncePerRequestFilter`**：保证每个请求只执行一次，避免重复处理
2. **使用 `DataPermissionUtils.executeIgnore()`**：将禁用操作包装在 try-finally 中，确保请求处理完毕后恢复上下文
3. **Filter 顺序**：在自动配置中，`DataPermissionRpcWebFilter` 的顺序设置为 `TENANT_CONTEXT_FILTER - 1`，确保在租户上下文 Filter 之前执行

### Filter 执行顺序

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/config/YudaoDataPermissionRpcAutoConfiguration.java
@Bean
public FilterRegistrationBean<DataPermissionRpcWebFilter> dataPermissionRpcFilter() {
    FilterRegistrationBean<DataPermissionRpcWebFilter> registrationBean = new FilterRegistrationBean<>();
    registrationBean.setFilter(new DataPermissionRpcWebFilter());
    // 顺序在租户 Filter 前面，稳妥点
    registrationBean.setOrder(TENANT_CONTEXT_FILTER - 1);
    return registrationBean;
}
```

为什么要在租户 Filter 之前执行？

```
请求进入
    │
    ▼
DataPermissionRpcWebFilter（顺序：TENANT_CONTEXT_FILTER - 1）
    │  如果 header 中有 data-permission-enable=false
    │  禁用数据权限
    │
    ▼
TenantContextFilter（顺序：TENANT_CONTEXT_FILTER）
    │  设置租户上下文
    │
    ▼
业务代码执行
    │  数据权限和租户上下文都已就绪
    │
    ▼
SQL 执行
    │  先应用租户过滤（DataPermissionInterceptor 顺序靠前）
    │  再应用数据权限过滤
```

## 自动配置

跨服务传播功能通过 `YudaoDataPermissionRpcAutoConfiguration` 自动配置：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/config/YudaoDataPermissionRpcAutoConfiguration.java
@AutoConfiguration
@ConditionalOnClass(name = "feign.RequestInterceptor") // 只有引入了 Feign 才生效
public class YudaoDataPermissionRpcAutoConfiguration {

    @Bean
    public DataPermissionRequestInterceptor dataPermissionRequestInterceptor() {
        return new DataPermissionRequestInterceptor();
    }

    @Bean
    public FilterRegistrationBean<DataPermissionRpcWebFilter> dataPermissionRpcFilter() {
        FilterRegistrationBean<DataPermissionRpcWebFilter> registrationBean = new FilterRegistrationBean<>();
        registrationBean.setFilter(new DataPermissionRpcWebFilter());
        registrationBean.setOrder(TENANT_CONTEXT_FILTER - 1);
        return registrationBean;
    }
}
```

`@ConditionalOnClass(name = "feign.RequestInterceptor")` 保证了：**只有在微服务模式下（引入了 Feign），才会注册 RPC 传播组件**。单体应用不会有这些 Bean。

## 与租户上下文传播的对比

在组件 10（多租户）中，我们也学习了类似的上下文传播机制。两者的设计思路相同，但实现细节有差异：

| 维度 | 数据权限传播 | 租户上下文传播 |
|------|------------|--------------|
| 传播内容 | `enable` 属性（布尔值） | `tenantId`（长整型） |
| Header 名 | `data-permission-enable` | 租户相关的 header |
| 发送端 | `DataPermissionRequestInterceptor` | 租户的 Feign 拦截器 |
| 接收端 | `DataPermissionRpcWebFilter` | 租户的 Web Filter |
| 默认行为 | 默认开启，需要显式禁用 | 必须传递租户 ID |
| Filter 顺序 | `TENANT_CONTEXT_FILTER - 1`（更早） | `TENANT_CONTEXT_FILTER` |

### 为什么数据权限 Filter 要在租户 Filter 之前

```
场景：订单服务调用商品服务，且需要禁用数据权限

如果顺序反过来（租户 Filter 先执行）：
  1. 租户 Filter 设置 tenant_id = 1
  2. 数据权限 Filter 禁用数据权限
  3. SQL 执行：WHERE tenant_id = 1（租户过滤生效）
  4. 数据权限虽然被禁用，但租户隔离仍然生效 ✓

如果数据权限 Filter 先执行（当前方案）：
  1. 数据权限 Filter 禁用数据权限
  2. 租户 Filter 设置 tenant_id = 1
  3. SQL 执行：WHERE tenant_id = 1（租户过滤生效）
  4. 数据权限被禁用，租户隔离仍然生效 ✓

两种顺序都能工作，但数据权限 Filter 先执行更稳妥：
  - 数据权限的禁用是"局部"的，只影响当前请求
  - 租户上下文是"全局"的，影响后续所有操作
  - 先设置数据权限，再设置租户，逻辑更清晰
```

## 数据权限 + 多租户的协作

在实际运行中，数据权限和多租户是叠加生效的。让我们跟踪一个完整的 SQL 改写过程：

```
原始 SQL：
  SELECT * FROM trade_order WHERE status = 1

第一步：多租户拦截器（TenantLineHandler）
  SELECT * FROM trade_order WHERE status = 1 AND tenant_id = 1

第二步：数据权限拦截器（DataPermissionRuleHandler）
  SELECT * FROM trade_order WHERE status = 1 AND tenant_id = 1 AND dept_id IN (100, 101, 102)
```

两者的协作关系：

```
┌─────────────────────────────────────────────────────────────┐
│                SQL 改写的两层过滤                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   原始 SQL                                                   │
│   ┌─────────────────────────────────────────┐               │
│   │ SELECT * FROM trade_order               │               │
│   │ WHERE status = 1                        │               │
│   └─────────────────────┬───────────────────┘               │
│                         │                                   │
│                         ▼                                   │
│   第一层：多租户过滤（全局生效）                               │
│   ┌─────────────────────────────────────────┐               │
│   │ WHERE status = 1                        │               │
│   │   AND tenant_id = 1                     │               │
│   └─────────────────────┬───────────────────┘               │
│                         │                                   │
│                         ▼                                   │
│   第二层：数据权限过滤（按规则生效）                           │
│   ┌─────────────────────────────────────────┐               │
│   │ WHERE status = 1                        │               │
│   │   AND tenant_id = 1                     │               │
│   │   AND dept_id IN (100, 101, 102)        │               │
│   └─────────────────────────────────────────┘               │
│                                                             │
│   最终效果：先隔离租户，再过滤部门数据                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 两层过滤的独立性

重要的是理解：**两层过滤是完全独立的**。

- 多租户过滤：由 `TenantLineHandler` 实现，对所有 SQL 全局生效，无法通过注解禁用
- 数据权限过滤：由 `DataPermissionRuleHandler` 实现，可以通过 `@DataPermission` 注解控制

这意味着：
- 即使数据权限被禁用（`@DataPermission(enable = false)`），租户隔离仍然生效
- 即使租户隔离被禁用（跨租户访问），数据权限仍然可能生效（除非同时禁用）

## 完整的跨服务调用链路

让我们跟踪一个完整的跨服务调用链路，理解数据权限是如何在服务间传播的：

```
1. 浏览器发起请求
   GET /admin-api/trade/order/page
   Header: Authorization: Bearer xxx

2. 订单服务 Controller
   @DataPermission(enable = false)
   public CommonResult<PageResult<OrderRespVO>> getOrderPage() {
       // 需要调用商品服务获取商品信息
       List<ProductRespVO> products = productApi.getProducts(...);
       // ...
   }

3. AOP 拦截
   DataPermissionAnnotationInterceptor.invoke()
   → DataPermissionContextHolder.add(@DataPermission(enable=false))
   栈：[@DataPermission(enable=false)]

4. Feign 调用商品服务
   DataPermissionRequestInterceptor.apply()
   → 读取 DataPermissionContextHolder.get()
   → enable = false
   → 设置 header: data-permission-enable: false

5. HTTP 请求发送到商品服务
   GET /rpc-api/product/get-products
   Header: data-permission-enable: false

6. 商品服务收到请求
   DataPermissionRpcWebFilter.doFilterInternal()
   → 读取 header: data-permission-enable = false
   → DataPermissionUtils.executeIgnore(() -> chain.doFilter(...))
   → DataPermissionContextHolder.add(@DataPermission(enable=false))
   栈：[@DataPermission(enable=false)]

7. 商品服务 Mapper 查询
   DataPermissionRuleHandler.getSqlSegment()
   → DataPermissionRuleFactory.getDataPermissionRule()
   → DataPermissionContextHolder.get() 返回 enable=false
   → 返回空列表，不改写 SQL

8. 商品服务返回数据
   DataPermissionRpcWebFilter 的 finally 块
   → DataPermissionContextHolder.remove()
   栈：[]

9. 订单服务收到返回数据
   Controller 方法继续执行

10. AOP 退出
    DataPermissionAnnotationInterceptor 的 finally 块
    → DataPermissionContextHolder.remove()
    栈：[]
```

## 边界情况处理

### 情况一：调用方没有禁用数据权限

```
订单服务：
  @DataPermission(enable = true)  // 或没有注解
  public void someMethod() {
      productApi.getProducts(...);
  }

Feign 拦截器：
  DataPermissionContextHolder.get() 返回 enable=true 或 null
  → 不设置 header

商品服务：
  DataPermissionRpcWebFilter 收到请求
  → header 中没有 data-permission-enable
  → 正常执行，数据权限按默认规则生效
```

### 情况二：调用链中有多个服务

```
订单服务 → 商品服务 → 库存服务

订单服务禁用数据权限
  → header: data-permission-enable: false

商品服务收到请求，禁用数据权限
  → 商品服务内部调用库存服务
  → Feign 拦截器读取 DataPermissionContextHolder
  → 当前上下文是 enable=false（由 DataPermissionRpcWebFilter 设置）
  → 继续传递 header: data-permission-enable: false

库存服务收到请求，禁用数据权限
  → 正常执行
```

这个设计保证了：**数据权限的禁用状态可以在整个调用链中传播**。

### 情况三：单体应用

```
单体应用中没有 Feign 调用
  → DataPermissionRequestInterceptor 不会注册（@ConditionalOnClass 条件不满足）
  → DataPermissionRpcWebFilter 不会注册
  → 不会有额外的开销
```

## 思考题

1. 当前方案只传播了 `enable=false` 的情况。如果业务需要在跨服务调用时传递"只使用某个特定规则"的需求，你会如何扩展当前设计？

2. `DataPermissionRpcWebFilter` 使用 `DataPermissionUtils.executeIgnore()` 来禁用数据权限。如果在 Filter 执行过程中抛出异常，`finally` 块是否能正常执行？这会导致什么问题？

3. 在微服务架构中，如果 Service A 调用 Service B，Service B 的数据权限规则可能和 Service A 不同。当前方案中，Service B 使用自己的规则还是 Service A 的规则？这种设计合理吗？
