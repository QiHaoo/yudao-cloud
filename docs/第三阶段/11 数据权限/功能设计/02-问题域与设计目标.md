# 02 - 问题域与设计目标

## 传统方案的痛点

假设你在开发一个电商后台管理系统，需要实现"区域经理只能看到本区域订单"的功能。最直觉的做法是什么？

### 方案一：业务代码中手动加 WHERE

在每个查询方法里，手动拼接部门过滤条件：

```java
// 订单 Service
public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
    // 获取当前用户的部门
    Long deptId = securityFrameworkUtils.getLoginUser().getDeptId();
    return orderMapper.selectPage(reqVO, new LambdaQueryWrapper<OrderDO>()
            .eq(OrderDO::getDeptId, deptId)  // 手动加部门过滤
            ...);
}

// 客户 Service
public PageResult<CustomerDO> getCustomerPage(CustomerPageReqVO reqVO) {
    Long deptId = securityFrameworkUtils.getLoginUser().getDeptId();
    return customerMapper.selectPage(reqVO, new LambdaQueryWrapper<CustomerDO>()
            .eq(CustomerDO::getDeptId, deptId)  // 又写一遍
            ...);
}

// 商品 Service（如果商品也需要按部门过滤）
public PageResult<ProductDO> getProductPage(ProductPageReqVO reqVO) {
    Long deptId = securityFrameworkUtils.getLoginUser().getDeptId();
    return productMapper.selectPage(reqVO, new LambdaQueryWrapper<ProductDO>()
            .eq(ProductDO::getDeptId, deptId)  // 再写一遍
            ...);
}
```

这种方案的痛点非常明显：

| 痛点 | 说明 |
|------|------|
| **代码重复** | 每个 Service 方法都要写一遍部门过滤逻辑，几十个方法就是几十遍 |
| **容易遗漏** | 新增一个查询方法时，开发者可能忘记加部门过滤，导致数据泄露 |
| **难以维护** | 如果数据范围规则变化（比如从"本部门"改为"本部门及子部门"），需要修改所有 Service 方法 |
| **规则不一致** | 不同开发者写的过滤逻辑可能不一样，有的用 `=`，有的用 `IN`，导致行为不一致 |
| **与业务耦合** | 数据权限逻辑和业务逻辑混在一起，Service 方法变得又长又难读 |

### 方案二：应用层过滤

另一个思路是：先查出所有数据，然后在应用层过滤掉没有权限的数据。

```java
public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
    // 先查出所有数据
    PageResult<OrderDO> pageResult = orderMapper.selectPage(reqVO);
    // 再在应用层过滤
    Long deptId = securityFrameworkUtils.getLoginUser().getDeptId();
    List<OrderDO> filtered = pageResult.getList().stream()
            .filter(order -> order.getDeptId().equals(deptId))
            .collect(Collectors.toList());
    pageResult.setList(filtered);
    return pageResult;
}
```

这种方案看似简单，但有致命缺陷：

```
问题场景：订单表有 100 万条数据，区域经理只能看 5000 条

应用层过滤：
  1. 从数据库查出 100 万条数据（大量 I/O，浪费内存）
  2. 在 Java 中过滤掉 99.5 万条（浪费 CPU）
  3. 最终返回 5000 条

SQL 层过滤：
  1. 在 SQL 中加 WHERE dept_id = 100
  2. 数据库只返回 5000 条（高效利用索引）
  3. 直接返回
```

| 维度 | 应用层过滤 | SQL 层过滤 |
|------|-----------|-----------|
| 数据库 I/O | 查全量数据，I/O 大 | 只查权限内数据，I/O 小 |
| 内存消耗 | 全量数据加载到 JVM | 只加载需要的数据 |
| 分页问题 | 分页不准确（每页 10 条可能过滤后只剩 3 条） | 分页准确 |
| 索引利用 | 无法利用 dept_id 索引 | 可以利用索引 |

结论：**数据量稍大时，应用层过滤完全不可行**。

## 为什么选择 SQL 改写

经过上面的分析，SQL 改写几乎是唯一合理的选择。它的核心思路是：

```
不改变业务代码，由框架在 SQL 执行前自动追加 WHERE 条件
```

SQL 改写方案的优势：

```
┌─────────────────────────────────────────────────────────────┐
│                    SQL 改写方案的优势                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 透明性                                                  │
│     业务代码不需要任何修改，数据权限对 Service 层完全透明     │
│                                                             │
│  2. 统一性                                                  │
│     所有 SQL 的过滤逻辑由同一个组件生成，保证一致            │
│                                                             │
│  3. 高效性                                                  │
│     利用数据库索引，只查询需要的数据                         │
│                                                             │
│  4. 安全性                                                  │
│     不可能遗漏——只要 SQL 涉及配置的表，就一定会被改写        │
│                                                             │
│  5. 可维护性                                                │
│     规则变化时只需修改配置，不需要改业务代码                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### SQL 改写的技术基础

yudao-cloud 选择 MyBatis-Plus 作为 ORM 框架，而 MyBatis-Plus 内置了 `DataPermissionInterceptor` 插件，提供了 SQL 改写的扩展点。

```java
// MyBatis-Plus 提供的接口
public interface MultiDataPermissionHandler {
    /**
     * 获取数据权限 SQL 片段
     *
     * @param table              表信息
     * @param where              原始 WHERE 条件
     * @param mappedStatementId  Mapper 方法的 ID
     * @return 追加的 WHERE 条件表达式
     */
    Expression getSqlSegment(Table table, Expression where, String mappedStatementId);
}
```

yudao-cloud 的 `DataPermissionRuleHandler` 实现了这个接口，在 `getSqlSegment()` 中根据规则生成 WHERE 条件，MyBatis-Plus 会自动将其拼接到原始 SQL 中。

这个设计的精妙之处在于：**框架负责拦截 SQL 并调用接口，业务只需要实现"为哪些表生成什么条件"**。

## 设计目标

基于以上分析，数据权限组件的设计目标可以归纳为四个：

### 目标一：声明式（注解驱动）

```
设计原则：数据权限应该是"声明"出来的，而不是"编码"出来的
```

通过 `@DataPermission` 注解，开发者可以声明式地控制数据权限：

```java
// 默认情况：开启数据权限，所有规则生效
public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
    return orderMapper.selectPage(reqVO);
}

// 特殊情况：禁用数据权限
@DataPermission(enable = false)
public AdminUserDO getUserByUsername(String username) {
    return userMapper.selectByUsername(username);
}

// 特殊情况：只使用特定规则
@DataPermission(includeRules = DeptDataPermissionRule.class)
public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
    return orderMapper.selectPage(reqVO);
}
```

注解可以放在方法上（方法级优先），也可以放在类上（作为默认值）。

### 目标二：可扩展（自定义规则）

```
设计原则：框架提供规则接口，业务决定如何实现
```

通过 `DataPermissionRule` 接口，业务可以自定义任意的数据权限规则：

```java
public interface DataPermissionRule {
    // 声明该规则要处理哪些表
    Set<String> getTableNames();
    // 根据表名和别名，生成过滤条件
    Expression getExpression(String tableName, Alias tableAlias);
}
```

内置的 `DeptDataPermissionRule` 处理"基于部门"的场景。如果业务需要"基于区域"、"基于客户等级"等自定义规则，只需实现 `DataPermissionRule` 接口并注册为 Spring Bean 即可。

```
┌─────────────────────────────────────────────────────────────┐
│                    规则扩展示意                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  DataPermissionRule (接口)                                  │
│      │                                                      │
│      ├── DeptDataPermissionRule     ← 内置：基于部门        │
│      │                                                      │
│      ├── RegionDataPermissionRule   ← 自定义：基于区域      │
│      │                                                      │
│      ├── CustomerLevelRule          ← 自定义：基于客户等级  │
│      │                                                      │
│      └── ... (任意扩展)                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 目标三：与 RBAC 集成

```
设计原则：数据权限的范围定义在角色上，运行时根据用户角色动态计算
```

数据权限不是独立存在的，它和 RBAC（角色权限）体系紧密集成：

```
角色表 (system_role)
┌──────┬────────────┬───────────┬──────────────────┐
│ id   │ name       │ data_scope│ data_scope_dept_ids│
├──────┼────────────┼───────────┼──────────────────┤
│ 1    │ 超级管理员  │ 1 (ALL)   │ null              │
│ 2    │ 区域经理    │ 4 (DEPT_AND_CHILD) │ null     │
│ 3    │ 销售       │ 5 (SELF)  │ null              │
│ 4    │ 运营总监    │ 2 (DEPT_CUSTOM) │ [100,200]   │
└──────┴────────────┴───────────┴──────────────────┘
```

运行时，`PermissionServiceImpl.getDeptDataPermission()` 根据用户的角色计算最终的数据权限：

```
用户"小王"拥有角色：销售(SELF) + 区域经理(DEPT_AND_CHILD)

计算过程：
  1. 销售角色(SELF) → self = true
  2. 区域经理角色(DEPT_AND_CHILD) → deptIds = [100, 101, 102]

最终结果：WHERE (dept_id IN (100, 101, 102) OR user_id = 42)
```

### 目标四：跨服务透明

```
设计原则：数据权限在微服务调用链中自动传播，不需要业务代码干预
```

在微服务架构中，Service A 调用 Service B 是很常见的。如果 Service A 已经做了数据权限检查，Service B 不应该重复检查（否则可能导致数据丢失或性能浪费）。

```
Service A (订单服务)                    Service B (商品服务)
    │                                       │
    │  @DataPermission(enable=false)        │
    │  调用商品服务查询商品 ────────────────▶│
    │                                       │  收到 header: data-permission-enable=false
    │                                       │  自动禁用数据权限
    │  ◀───────────────────────────────────│  返回数据
    │                                       │
```

通过 Feign 拦截器 + Web Filter 的组合，数据权限的 `enable` 属性会自动在服务间传播。

## 与组件 8（角色权限与菜单）的关系

数据权限和角色权限是 RBAC 体系的两个"翅膀"，它们共享角色这个核心概念：

```
┌─────────────────────────────────────────────────────────────┐
│                    RBAC 体系全景                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                      角色 (Role)                             │
│                    ┌─────────────┐                          │
│                    │             │                          │
│         ┌──────────┤   role_id   ├──────────┐               │
│         │          │             │          │               │
│         ▼          └─────────────┘          ▼               │
│  ┌─────────────┐                    ┌──────────────┐        │
│  │ 菜单权限     │                    │ 数据权限      │        │
│  │ (功能权限)   │                    │ (数据范围)    │        │
│  │             │                    │              │        │
│  │ @PreAuthorize│                   │ @DataPermission│      │
│  │             │                    │              │        │
│  │ 控制"能不能  │                    │ 控制"能看到   │        │
│  │  操作"      │                    │  哪些数据"    │        │
│  └─────────────┘                    └──────────────┘        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

具体关系：

| 维度 | 角色权限（组件 8） | 数据权限（组件 11） |
|------|-------------------|-------------------|
| 存储位置 | `system_role_menu` 表 | `system_role.data_scope` 字段 |
| 配置方式 | 角色绑定菜单 | 角色设置数据范围 |
| 执行位置 | Controller 层（Spring Security） | Mapper 层（MyBatis-Plus 插件） |
| 注解 | `@PreAuthorize("@ss.hasPermission('order:query')")` | `@DataPermission` |
| 检查时机 | 进入方法前 | SQL 执行前 |

两个组件共享的核心数据结构是 `RoleDO`：

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/dal/dataobject/permission/RoleDO.java
public class RoleDO extends TenantBaseDO {
    private Long id;
    private String name;
    private String code;
    // ... 其他字段

    /**
     * 数据范围
     * 枚举 DataScopeEnum
     */
    private Integer dataScope;
    /**
     * 数据范围(指定部门数组)
     * 适用于 dataScope 的值为 DEPT_CUSTOM 时
     */
    private Set<Long> dataScopeDeptIds;
}
```

## 设计取舍：为什么不用数据库行级安全（RLS）

有些数据库（如 PostgreSQL）支持行级安全（Row Level Security），可以在数据库层面实现数据过滤。为什么 yudao-cloud 没有选择这个方案？

| 维度 | 数据库 RLS | 应用层 SQL 改写 |
|------|-----------|----------------|
| 数据库兼容 | 仅 PostgreSQL 等少数数据库支持 | 支持所有数据库 |
| 灵活性 | 受限于数据库能力 | 可以用 Java 实现任意逻辑 |
| 调试困难 | 调试困难，需要数据库管理员配合 | 可以在应用层日志中看到改写后的 SQL |
| 性能 | 数据库层执行，理论上更高效 | JSQLParser 解析有一定开销，但可忽略 |
| 维护成本 | 需要 DBA 参与 | 开发者自行维护 |

yudao-cloud 选择应用层 SQL 改写，核心原因是**跨数据库兼容**和**开发自主权**。

## 设计取舍：为什么用注解而不是配置文件

另一个可选方案是通过配置文件定义数据权限规则，例如：

```yaml
data-permission:
  rules:
    - table: trade_order
      column: dept_id
    - table: trade_customer
      column: dept_id
```

为什么 yudao-cloud 选择注解而不是配置文件？

| 维度 | 注解驱动 | 配置文件驱动 |
|------|---------|-------------|
| 细粒度控制 | 可以精确到方法级别 | 通常只能到表级别 |
| 可读性 | 代码即文档，一目了然 | 需要对照配置文件和代码 |
| 动态性 | 可以根据业务逻辑动态启用/禁用 | 静态配置，难以动态调整 |
| IDE 支持 | IDE 可以做语法检查和重构 | 配置文件中的类名容易拼错 |

yudao-cloud 的设计是**两者结合**：`@DataPermission` 注解控制方法级别的行为，`DeptDataPermissionRuleCustomizer` 配置表级别的字段映射。

## 思考题

1. 如果一个系统同时需要"基于部门的数据权限"和"基于区域的数据权限"，你会如何设计？是写两个 `DataPermissionRule` 实现，还是扩展 `DeptDataPermissionRule`？
2. 数据权限的 SQL 改写在 SQL 执行前拦截，这意味着它无法覆盖原生 SQL（如 `@Select` 注解中的 SQL）。如果业务中有原生 SQL，你有什么解决方案？
3. 在微服务架构中，如果 Service A 调用 Service B，Service B 再调用 Service C，数据权限应该如何传播？当前方案只传播了 `enable=false`，是否足够？
