# 03 - 核心设计：数据权限规则体系

## 从一个问题开始

在上一章中，我们确定了数据权限的核心机制是 SQL 改写。但 SQL 改写只是"手段"，真正的问题是：

**如何定义"哪些表需要过滤"以及"过滤条件是什么"？**

这就是规则体系要解决的问题。yudao-cloud 设计了一个灵活的规则接口，让业务可以自定义任意的数据权限逻辑。

## DataPermissionRule 接口

整个规则体系的基石是 `DataPermissionRule` 接口，它只有两个方法：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rule/DataPermissionRule.java
public interface DataPermissionRule {

    /**
     * 返回需要生效的表名数组
     * DataPermission 基于 SQL 重写，通过 WHERE 返回只有权限的数据
     */
    Set<String> getTableNames();

    /**
     * 根据表名和别名，生成对应的 WHERE / OR 过滤条件
     *
     * @param tableName 表名
     * @param tableAlias 别名，可能为空
     * @return 过滤条件 Expression 表达式
     */
    Expression getExpression(String tableName, Alias tableAlias);
}
```

这两个方法的设计意图非常清晰：

```
getTableNames()  →  "我负责哪些表？"
getExpression()  →  "这些表的过滤条件是什么？"
```

### 为什么返回 JSQLParser 的 Expression 而不是 SQL 字符串

`getExpression()` 返回的是 JSQLParser 库的 `Expression` 对象，而不是普通的 SQL 字符串。这个设计选择有深意：

```
方案一：返回 SQL 字符串
  getExpression() → "dept_id IN (100, 101, 102)"

  问题：
  - 需要自己处理 SQL 注入风险
  - 需要自己处理表别名（如 t.dept_id vs dept_id）
  - 需要自己拼接 AND/OR，容易出错

方案二：返回 Expression 对象（yudao-cloud 的选择）
  getExpression() → InExpression(column, expressionList)

  优势：
  - JSQLParser 自动处理 SQL 语法
  - MyBatis-Plus 自动处理 AND 拼接
  - 类型安全，编译期检查
```

用一个类比来理解：返回 SQL 字符串就像用字符串拼 HTML（容易 XSS），返回 Expression 对象就像用 DOM API 构建 HTML（安全且结构化）。

## DataPermissionRuleFactory：规则的管理中心

`DataPermissionRuleFactory` 是规则的容器和管理中心，负责根据当前上下文决定哪些规则应该生效。

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rule/DataPermissionRuleFactory.java
public interface DataPermissionRuleFactory {

    /**
     * 获得所有数据权限规则数组
     */
    List<DataPermissionRule> getDataPermissionRules();

    /**
     * 获得指定 Mapper 的数据权限规则数组
     * 根据 @DataPermission 注解上下文过滤
     */
    List<DataPermissionRule> getDataPermissionRule(String mappedStatementId);
}
```

核心方法是 `getDataPermissionRule(String mappedStatementId)`，它根据当前的 `@DataPermission` 注解上下文来过滤规则。

### DataPermissionRuleFactoryImpl 的过滤逻辑

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rule/DataPermissionRuleFactoryImpl.java
@Override
public List<DataPermissionRule> getDataPermissionRule(String mappedStatementId) {
    // 1.1 无数据权限规则
    if (CollUtil.isEmpty(rules)) {
        return Collections.emptyList();
    }

    // 1.2 未配置 @DataPermission 注解，则默认开启所有规则
    DataPermission dataPermission = DataPermissionContextHolder.get();
    if (dataPermission == null) {
        return rules;
    }

    // 1.3 已配置，但禁用数据权限
    if (!dataPermission.enable()) {
        return Collections.emptyList();
    }

    // 1.4 特殊：数据翻译时，强制忽略数据权限
    if (isTranslateCall()) {
        return Collections.emptyList();
    }

    // 2.1 情况一：已配置，只选择部分规则（includeRules 优先级高于 excludeRules）
    if (ArrayUtil.isNotEmpty(dataPermission.includeRules())) {
        return rules.stream()
                .filter(rule -> ArrayUtil.contains(dataPermission.includeRules(), rule.getClass()))
                .collect(Collectors.toList());
    }

    // 2.2 已配置，只排除部分规则
    if (ArrayUtil.isNotEmpty(dataPermission.excludeRules())) {
        return rules.stream()
                .filter(rule -> !ArrayUtil.contains(dataPermission.excludeRules(), rule.getClass()))
                .collect(Collectors.toList());
    }

    // 2.3 已配置，全部规则
    return rules;
}
```

这个过滤逻辑可以用一个决策树来表示：

```
getDataPermissionRule() 被调用
    │
    ▼
规则列表为空？ ──是──▶ 返回空列表
    │否
    ▼
有 @DataPermission 注解？ ──否──▶ 返回所有规则（默认开启）
    │是
    ▼
enable = false？ ──是──▶ 返回空列表（禁用数据权限）
    │否
    ▼
是数据翻译调用？ ──是──▶ 返回空列表（避免干扰）
    │否
    ▼
配置了 includeRules？ ──是──▶ 返回 includeRules 中指定的规则
    │否
    ▼
配置了 excludeRules？ ──是──▶ 返回排除 excludeRules 后的规则
    │否
    ▼
返回所有规则
```

### 优先级：includeRules > excludeRules

`includeRules` 的优先级高于 `excludeRules`，这是一个重要的设计决策：

```java
// 场景：系统有 3 个规则：DeptRule、RegionRule、CustomerLevelRule

// 只使用 DeptRule
@DataPermission(includeRules = DeptDataPermissionRule.class)
public PageResult<OrderDO> getOrderPage() { ... }

// 排除 RegionRule（使用 DeptRule + CustomerLevelRule）
@DataPermission(excludeRules = RegionDataPermissionRule.class)
public PageResult<OrderDO> getOrderPage() { ... }

// 注意：如果同时配置了 includeRules 和 excludeRules
// 只有 includeRules 生效，excludeRules 被忽略
@DataPermission(includeRules = DeptDataPermissionRule.class,
               excludeRules = RegionDataPermissionRule.class)
// 实际效果：只使用 DeptRule
```

## DataPermissionRuleHandler：SQL 改写的核心

`DataPermissionRuleHandler` 是整个数据权限组件的"发动机"，它实现了 MyBatis-Plus 的 `MultiDataPermissionHandler` 接口，负责在 SQL 执行前拦截并改写 SQL。

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/db/DataPermissionRuleHandler.java
@RequiredArgsConstructor
public class DataPermissionRuleHandler implements MultiDataPermissionHandler {

    private final DataPermissionRuleFactory ruleFactory;

    @Override
    public Expression getSqlSegment(Table table, Expression where, String mappedStatementId) {
        // 步骤 1：特殊处理——跨租户访问时跳过
        if (skipPermissionCheck()) {
            return null;
        }

        // 步骤 2：获取 Mapper 对应的数据权限规则
        List<DataPermissionRule> rules = ruleFactory.getDataPermissionRule(mappedStatementId);
        if (CollUtil.isEmpty(rules)) {
            return null;
        }

        // 步骤 3：遍历规则，生成条件
        Expression allExpression = null;
        for (DataPermissionRule rule : rules) {
            // 判断表名是否匹配
            String tableName = MyBatisUtils.getTableName(table);
            if (!rule.getTableNames().contains(tableName)) {
                continue;
            }

            // 获取单条规则的条件
            Expression oneExpress = rule.getExpression(tableName, table.getAlias());
            if (oneExpress == null) {
                continue;
            }

            // 用 AND 拼接到总条件中
            allExpression = allExpression == null ? oneExpress
                    : new AndExpression(allExpression, oneExpress);
        }
        return allExpression;
    }
}
```

```
    │
    ▼

    │否

    │
    ▼

    │否
    ▼

    │
    ▼
┌─── 遍历每条规则 ───────────────────────────────────┐
│   │                                                 │
│   ▼                                                 │
│   rule.getTableNames() 包含当前表名？                │
│   │否                                              │
│   │  └──▶ continue（跳过这条规则）                  │

│   │                                                 │

```

### SQL 改写示例

让我们通过一个完整的例子来理解 SQL 改写过程：

```
原始 SQL：
  SELECT * FROM trade_order WHERE status = 1

规则配置：
  DeptDataPermissionRule:
    - trade_order 表的 dept_id 字段
    - 当前用户可见部门：[100, 101, 102]

改写过程：
  1. DataPermissionRuleHandler.getSqlSegment() 被调用
  2. 获取规则列表：[DeptDataPermissionRule]
  3. 遍历规则：
     - DeptDataPermissionRule.getTableNames() 包含 "trade_order"？
       → 是
     - DeptDataPermissionRule.getExpression("trade_order", null)
       → 返回：dept_id IN (100, 101, 102)
  4. allExpression = dept_id IN (100, 101, 102)

改写后的 SQL：
  SELECT * FROM trade_order
  WHERE status = 1
    AND dept_id IN (100, 101, 102)
```

## DeptDataPermissionRule：基于部门的数据权限实现

`DeptDataPermissionRule` 是内置的、也是最常用的数据权限规则实现。它基于部门（dept_id）和用户（user_id）来过滤数据。

### 核心字段

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rule/dept/DeptDataPermissionRule.java
@AllArgsConstructor
@Slf4j
public class DeptDataPermissionRule implements DataPermissionRule {

    private final PermissionCommonApi permissionApi;

    /**
     * 基于部门的表字段配置
     * key：表名，value：字段名
     * 一般情况下，每个表的部门编号字段是 dept_id
     */
    private final Map<String, String> deptColumns = new HashMap<>();

    /**
     * 基于用户的表字段配置
     * key：表名，value：字段名
     */
    private final Map<String, String> userColumns = new HashMap<>();

    /**
     * 所有表名，是 deptColumns 和 userColumns 的合集
     */
    private final Set<String> TABLE_NAMES = new HashSet<>();
}
```

### deptColumns 和 userColumns：表名到字段名的映射

这两个 Map 是 `DeptDataPermissionRule` 的核心配置。它们建立了**表名**到**字段名**的映射关系：

```
deptColumns:
  ┌─────────────────┬───────────┐
  │ 表名             │ 字段名    │
  ├─────────────────┼───────────┤
  │ trade_order     │ dept_id   │  ← 订单表的部门字段是 dept_id
  │ trade_customer  │ dept_id   │  ← 客户表的部门字段是 dept_id
  │ system_dept     │ id        │  ← 部门表的主键是 id（特殊处理）
  └─────────────────┴───────────┘

userColumns:
  ┌─────────────────┬───────────┐
  │ 表名             │ 字段名    │
  ├─────────────────┼───────────┤
  │ trade_order     │ user_id   │  ← 订单表的用户字段是 user_id
  │ trade_customer  │ creator   │  ← 客户表的用户字段是 creator
  └─────────────────┴───────────┘
```

通过 `addDeptColumn()` 和 `addUserColumn()` 方法配置：

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/framework/datapermission/config/DataPermissionConfiguration.java
@Configuration(proxyBeanMethods = false)
public class DataPermissionConfiguration {

    @Bean
    public DeptDataPermissionRuleCustomizer sysDeptDataPermissionRuleCustomizer() {
        return rule -> {
            // 部门过滤：AdminUserDO 表的 dept_id 字段
            rule.addDeptColumn(AdminUserDO.class);
            // 部门过滤：DeptDO 表的 id 字段（特殊：部门表的主键就是部门编号）
            rule.addDeptColumn(DeptDO.class, "id");
            // 用户过滤：AdminUserDO 表的 id 字段（特殊：用户表的主键就是用户编号）
            rule.addUserColumn(AdminUserDO.class, "id");
        };
    }
}
```

`addDeptColumn()` 和 `addUserColumn()` 方法支持两种配置方式：

```java
// 方式一：通过实体类自动获取表名，字段名默认为 dept_id / user_id
public void addDeptColumn(Class<? extends BaseDO> entityClass) {
    addDeptColumn(entityClass, "dept_id");  // 默认字段名
}

// 方式二：通过实体类 + 自定义字段名
public void addDeptColumn(Class<? extends BaseDO> entityClass, String columnName) {
    String tableName = TableInfoHelper.getTableInfo(entityClass).getTableName();
    addDeptColumn(tableName, columnName);
}

// 方式三：直接指定表名和字段名
public void addDeptColumn(String tableName, String columnName) {
    deptColumns.put(tableName, columnName);
    TABLE_NAMES.add(tableName);  // 同时注册到 TABLE_NAMES
}
```

### getExpression()：生成过滤条件的核心逻辑

`getExpression()` 是 `DeptDataPermissionRule` 最核心的方法，它根据当前用户的数据权限生成 SQL 过滤条件：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/core/rule/dept/DeptDataPermissionRule.java
@Override
public Expression getExpression(String tableName, Alias tableAlias) {
    // 步骤 1：只有有登录用户的情况下，才进行数据权限处理
    LoginUser loginUser = SecurityFrameworkUtils.getLoginUser();
    if (loginUser == null) {
        return null;
    }

    // 步骤 2：只有管理员类型的用户，才进行数据权限处理
    if (ObjectUtil.notEqual(loginUser.getUserType(), UserTypeEnum.ADMIN.getValue())) {
        return null;
    }

    // 步骤 3：获取数据权限（优先从缓存获取，避免重复 RPC 调用）
    DeptDataPermissionRespDTO deptDataPermission =
            loginUser.getContext(CONTEXT_KEY, DeptDataPermissionRespDTO.class);
    if (deptDataPermission == null) {
        deptDataPermission = permissionApi.getDeptDataPermission(loginUser.getId()).getCheckedData();
        // 缓存到 LoginUser 上下文中
        loginUser.setContext(CONTEXT_KEY, deptDataPermission);
    }

    // 步骤 4：情况一——ALL，可查看全部，无需拼接条件
    if (deptDataPermission.getAll()) {
        return null;  // 返回 null 表示不加条件
    }

    // 步骤 5：情况二——既不能查看部门，又不能查看自己，100% 无权限
    if (CollUtil.isEmpty(deptDataPermission.getDeptIds())
        && Boolean.FALSE.equals(deptDataPermission.getSelf())) {
        return new EqualsTo(null, null);  // WHERE null = null，返回空结果
    }

    // 步骤 6：情况三——拼接 Dept 和 User 的条件
    Expression deptExpression = buildDeptExpression(tableName, tableAlias, deptDataPermission.getDeptIds());
    Expression userExpression = buildUserExpression(tableName, tableAlias, deptDataPermission.getSelf(), loginUser.getId());

    // 组合条件
    if (deptExpression == null && userExpression == null) {
        return new EqualsTo(null, null);  // 无条件时返回空结果
    }
    if (deptExpression == null) {
        return userExpression;
    }
    if (userExpression == null) {
        return deptExpression;
    }
    // 目前，如果有指定部门 + 可查看自己，采用 OR 条件
    return new ParenthesedExpressionList(new OrExpression(deptExpression, userExpression));
}
```

### buildDeptExpression()：生成部门过滤条件

```java
private Expression buildDeptExpression(String tableName, Alias tableAlias, Set<Long> deptIds) {
    // 如果表没有配置 deptColumns，无需作为条件
    String columnName = deptColumns.get(tableName);
    if (StrUtil.isEmpty(columnName)) {
        return null;
    }
    // 如果部门列表为空，无需作为条件
    if (CollUtil.isEmpty(deptIds)) {
        return null;
    }
    // 拼接条件：dept_id IN (100, 101, 102)
    return new InExpression(
        MyBatisUtils.buildColumn(tableName, tableAlias, columnName),
        new ParenthesedExpressionList(
            new ExpressionList<LongValue>(
                CollectionUtils.convertList(deptIds, LongValue::new)
            )
        )
    );
}
```

生成的 SQL 条件：

```sql
-- 无别名时
dept_id IN (100, 101, 102)

-- 有别名时（如 SELECT * FROM trade_order o）
o.dept_id IN (100, 101, 102)
```

### buildUserExpression()：生成用户过滤条件

```java
private Expression buildUserExpression(String tableName, Alias tableAlias, Boolean self, Long userId) {
    // 如果不查看自己，无需作为条件
    if (Boolean.FALSE.equals(self)) {
        return null;
    }
    // 如果表没有配置 userColumns，无需作为条件
    String columnName = userColumns.get(tableName);
    if (StrUtil.isEmpty(columnName)) {
        return null;
    }
    // 拼接条件：user_id = 42
    return new EqualsTo(
        MyBatisUtils.buildColumn(tableName, tableAlias, columnName),
        new LongValue(userId)
    );
}
```

### 组合条件的最终效果

当用户同时拥有"部门"和"个人"的数据权限时，最终的 SQL 条件是：

```sql
-- 用户可见部门：[100, 101, 102]，用户 ID：42
-- trade_order 表同时配置了 deptColumns 和 userColumns

-- 最终条件：WHERE (dept_id IN (100, 101, 102) OR user_id = 42)
```

这个 OR 条件的含义是：**用户可以看到自己所在部门的数据，也可以看到自己创建的数据**。

## DeptDataPermissionRespDTO：数据权限的传输对象

`DeptDataPermissionRespDTO` 是数据权限计算结果的载体，它在 `PermissionCommonApi` 的 RPC 调用中传输：

```java
// 文件路径：yudao-framework/yudao-common/src/main/java/cn/iocoder/yudao/framework/common/biz/system/permission/dto/DeptDataPermissionRespDTO.java
@Data
public class DeptDataPermissionRespDTO {

    /**
     * 是否可查看全部数据
     */
    private Boolean all;

    /**
     * 是否可查看自己的数据
     */
    private Boolean self;

    /**
     * 可查看的部门编号数组
     */
    private Set<Long> deptIds;

    public DeptDataPermissionRespDTO() {
        this.all = false;
        this.self = false;
        this.deptIds = new HashSet<>();
    }
}
```

三个字段与 `DataScopeEnum` 的对应关系：

```
DataScopeEnum.ALL           → all = true
DataScopeEnum.DEPT_CUSTOM   → deptIds = [指定的部门编号]
DataScopeEnum.DEPT_ONLY     → deptIds = [用户所在部门]
DataScopeEnum.DEPT_AND_CHILD→ deptIds = [用户所在部门 + 所有子部门]
DataScopeEnum.SELF          → self = true
```

注意：一个用户可能拥有多个角色，每个角色有不同的数据范围。最终结果是所有角色的**并集**。

## getDeptDataPermission()：数据权限的计算逻辑

`PermissionServiceImpl.getDeptDataPermission()` 负责根据用户的角色计算最终的数据权限：

```java
// 文件路径：yudao-module-system/yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/service/permission/PermissionServiceImpl.java
@Override
@DataPermission(enable = false) // 关闭数据权限，避免递归获取数据权限
public DeptDataPermissionRespDTO getDeptDataPermission(Long userId) {
    // 获取用户的角色
    List<RoleDO> roles = getEnableUserRoleListByUserIdFromCache(userId);

    // 如果角色为空，则只能查看自己
    DeptDataPermissionRespDTO result = new DeptDataPermissionRespDTO();
    if (CollUtil.isEmpty(roles)) {
        result.setSelf(true);
        return result;
    }

    // 获取用户部门编号（惰性加载，只查一次）
    Supplier<Long> userDeptId = Suppliers.memoize(() -> userService.getUser(userId).getDeptId());

    // 遍历每个角色，计算并集
    for (RoleDO role : roles) {
        if (role.getDataScope() == null) {
            continue;
        }

        // ALL：查看全部
        if (Objects.equals(role.getDataScope(), DataScopeEnum.ALL.getScope())) {
            result.setAll(true);
            continue;
        }

        // DEPT_CUSTOM：指定部门
        if (Objects.equals(role.getDataScope(), DataScopeEnum.DEPT_CUSTOM.getScope())) {
            CollUtil.addAll(result.getDeptIds(), role.getDataScopeDeptIds());
            // 保证可以看到自己所在的部门
            CollectionUtils.addIfNotNull(result.getDeptIds(), userDeptId.get());
            continue;
        }

        // DEPT_ONLY：本部门
        if (Objects.equals(role.getDataScope(), DataScopeEnum.DEPT_ONLY.getScope())) {
            CollectionUtils.addIfNotNull(result.getDeptIds(), userDeptId.get());
            continue;
        }

        // DEPT_AND_CHILD：本部门及子部门
        if (Objects.equals(role.getDataScope(), DataScopeEnum.DEPT_AND_CHILD.getScope())) {
            Long deptId = userDeptId.get();
            if (deptId == null) {
                continue;
            }
            CollUtil.addAll(result.getDeptIds(), deptService.getChildDeptIdListFromCache(deptId));
            result.getDeptIds().add(deptId);  // 添加本身部门编号
            continue;
        }

        // SELF：仅本人
        if (Objects.equals(role.getDataScope(), DataScopeEnum.SELF.getScope())) {
            result.setSelf(true);
            continue;
        }
    }
    return result;
}
```

注意方法上的 `@DataPermission(enable = false)` 注解——这是为了避免递归。如果不关闭数据权限，`getDeptDataPermission()` 内部查询用户和部门时，又会触发数据权限检查，导致无限递归。

## 规则的自动配置

`DeptDataPermissionRule` 通过 Spring Boot 自动配置注册到容器中：

```java
// 文件路径：yudao-framework/yudao-spring-boot-starter-biz-data-permission/src/main/java/cn/iocoder/yudao/framework/datapermission/config/YudaoDeptDataPermissionAutoConfiguration.java
@AutoConfiguration
@ConditionalOnClass(LoginUser.class)
@ConditionalOnBean(value = {DeptDataPermissionRuleCustomizer.class})
public class YudaoDeptDataPermissionAutoConfiguration {

    @Bean
    public DeptDataPermissionRule deptDataPermissionRule(
            PermissionCommonApi permissionApi,
            List<DeptDataPermissionRuleCustomizer> customizers) {
        // Cloud 专属逻辑：优先使用本地的 PermissionApi 实现类
        try {
            PermissionCommonApi permissionApiImpl = SpringUtil.getBean("permissionApiImpl", PermissionCommonApi.class);
            if (permissionApiImpl != null) {
                permissionApi = permissionApiImpl;
            }
        } catch (Exception ignored) {}

        // 创建 DeptDataPermissionRule 对象
        DeptDataPermissionRule rule = new DeptDataPermissionRule(permissionApi);
        // 通过 Customizer 补全表配置
        customizers.forEach(customizer -> customizer.customize(rule));
        return rule;
    }
}
```

自动配置的条件：
- `@ConditionalOnClass(LoginUser.class)`：必须有安全框架
- `@ConditionalOnBean(DeptDataPermissionRuleCustomizer.class)`：必须有业务层的 Customizer 配置

这意味着：**如果业务层没有配置 `DeptDataPermissionRuleCustomizer`，数据权限规则不会被激活**。
