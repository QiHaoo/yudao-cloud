# 06 - 精简汇总

本章是前五章的精简版，方便回顾复习。

## 能力速查表

### @DataPermission 注解配置

| 属性 | 类型 | 默认值 | 作用 | 示例 |
|------|------|-------|------|------|
| `enable` | `boolean` | `true` | 控制数据权限是否生效 | `@DataPermission(enable = false)` |
| `includeRules` | `Class<?>[]` | `{}` | 指定生效的规则（白名单） | `@DataPermission(includeRules = DeptDataPermissionRule.class)` |
| `excludeRules` | `Class<?>[]` | `{}` | 指定排除的规则（黑名单） | `@DataPermission(excludeRules = RegionRule.class)` |

优先级：`enable = false` > `includeRules` > `excludeRules` > 默认全部规则

### DataPermissionRule 接口

| 方法 | 返回值 | 作用 |
|------|-------|------|
| `getTableNames()` | `Set<String>` | 声明该规则负责哪些表 |
| `getExpression(tableName, tableAlias)` | `Expression` | 根据表名生成 WHERE 过滤条件 |

### DeptDataPermissionRule 配置

| 方法 | 作用 | 示例 |
|------|------|------|
| `addDeptColumn(Class<?> entityClass)` | 注册部门字段（默认 `dept_id`） | `rule.addDeptColumn(OrderDO.class)` |
| `addDeptColumn(Class<?> entityClass, String column)` | 注册部门字段（自定义列名） | `rule.addDeptColumn(DeptDO.class, "id")` |
| `addUserColumn(Class<?> entityClass)` | 注册用户字段（默认 `user_id`） | `rule.addUserColumn(OrderDO.class)` |
| `addUserColumn(Class<?> entityClass, String column)` | 注册用户字段（自定义列名） | `rule.addUserColumn(OrderDO.class, "creator")` |

### DataScopeEnum 五级数据范围

| 枚举值 | 编号 | 含义 | 典型角色 |
|-------|------|------|---------|
| `ALL` | 1 | 查看全部数据 | 超级管理员 |
| `DEPT_CUSTOM` | 2 | 查看指定部门数据 | 运营总监 |
| `DEPT_ONLY` | 3 | 查看本部门数据 | 部门经理 |
| `DEPT_AND_CHILD` | 4 | 查看本部门及子部门数据 | 区域经理 |
| `SELF` | 5 | 查看本人数据 | 普通销售 |

### DataPermissionUtils 工具方法

| 方法 | 作用 |
|------|------|
| `executeIgnore(Runnable)` | 禁用数据权限执行逻辑（无返回值） |
| `executeIgnore(Callable<T>)` | 禁用数据权限执行逻辑（有返回值） |
| `addDisableDataPermission()` | 手动压入禁用注解 |
| `removeDataPermission()` | 手动弹出注解 |

## 核心设计决策列表

| 决策 | 选择 | 理由 |
|------|------|------|
| 数据过滤位置 | SQL 层改写 | 性能高、不可遗漏、与业务解耦 |
| SQL 改写技术 | MyBatis-Plus MultiDataPermissionHandler | 框架内置支持，扩展点清晰 |
| 条件表达式 | JSQLParser Expression 对象 | 类型安全、自动处理 AND 拼接 |
| 注解驱动 vs 配置文件 | 注解为主 + Customizer 配置为辅 | 方法级精细控制 + 表级统一配置 |
| 上下文存储 | TransmittableThreadLocal + LinkedList 栈 | 支持嵌套调用 + 线程池传播 |
| 跨服务传播 | HTTP Header（仅 enable） | 简单可靠，避免类序列化问题 |
| 规则匹配策略 | 注解级优先于全局级 | 灵活性：默认全部 + 按需调整 |
| 无权限处理 | `WHERE null = null` | 返回空结果集，而非抛异常 |
| 递归防护 | `@DataPermission(enable = false)` | 在获取数据权限的方法上禁用 |

## SQL 改写示例

### 示例一：部门过滤

```
配置：
  DeptDataPermissionRule
  deptColumns: { "trade_order": "dept_id" }
  用户可见部门: [100, 101, 102]

原始 SQL：
  SELECT * FROM trade_order WHERE status = 1

改写后 SQL：
  SELECT * FROM trade_order
  WHERE status = 1
    AND dept_id IN (100, 101, 102)
```

### 示例二：用户过滤

```
配置：
  DeptDataPermissionRule
  userColumns: { "trade_order": "user_id" }
  用户 ID: 42，数据范围: SELF

原始 SQL：
  SELECT * FROM trade_order WHERE status = 1

改写后 SQL：
  SELECT * FROM trade_order
  WHERE status = 1
    AND user_id = 42
```

### 示例三：部门 + 用户组合过滤

```
配置：
  DeptDataPermissionRule
  deptColumns: { "trade_order": "dept_id" }
  userColumns: { "trade_order": "user_id" }
  用户可见部门: [100, 101]，用户 ID: 42

原始 SQL：
  SELECT * FROM trade_order WHERE status = 1

改写后 SQL：
  SELECT * FROM trade_order
  WHERE status = 1
    AND (dept_id IN (100, 101) OR user_id = 42)
```

### 示例四：带表别名的 SQL

```
配置：
  同示例三

原始 SQL：
  SELECT o.* FROM trade_order o LEFT JOIN trade_customer c ON o.customer_id = c.id WHERE o.status = 1

改写后 SQL：
  SELECT o.* FROM trade_order o LEFT JOIN trade_customer c ON o.customer_id = c.id
  WHERE o.status = 1
    AND (o.dept_id IN (100, 101) OR o.user_id = 42)

注意：customer 表如果没有配置 deptColumns/userColumns，不会被过滤
```

### 示例五：禁用数据权限

```
配置：
  @DataPermission(enable = false)

原始 SQL：
  SELECT * FROM trade_order WHERE status = 1

改写后 SQL：
  （不改写，保持原样）
  SELECT * FROM trade_order WHERE status = 1
```

## 数据权限请求生命周期

```
1. 请求进入 Controller
   │
   ▼
2. AOP 拦截 @DataPermission
   │  压入 DataPermissionContextHolder 栈
   │
   ▼
3. Service 层执行业务逻辑
   │
   ▼
4. Mapper 层执行 SQL
   │
   ▼
5. MyBatis-Plus DataPermissionInterceptor 拦截
   │  调用 DataPermissionRuleHandler.getSqlSegment()
   │
   ▼
6. DataPermissionRuleHandler 处理
   │  ├─ skipPermissionCheck()？→ 跳过
   │  ├─ 获取规则列表（根据注解上下文过滤）
   │  ├─ 遍历规则，匹配表名
   │  ├─ 调用 rule.getExpression() 获取条件
   │  └─ AND 组合所有条件
   │
   ▼
7. SQL 被改写并执行
   │
   ▼
8. 返回结果
   │
   ▼
9. AOP 退出
   │  弹出 DataPermissionContextHolder 栈
   │
   ▼
10. 返回响应给客户端
```

## 常见问题

**Q: 数据权限和多租户会冲突吗？**
A: 不会。两者是叠加关系，先多租户过滤，再数据权限过滤。数据权限禁用不影响租户隔离。

**Q: 如何临时禁用数据权限？**
A: 两种方式：注解 `@DataPermission(enable = false)` 或工具类 `DataPermissionUtils.executeIgnore()`。

**Q: 数据权限只对管理员生效吗？**
A: 是的。`DeptDataPermissionRule.getExpression()` 中判断了 `UserTypeEnum.ADMIN`，非管理员类型用户不处理。

**Q: 用户没有角色时会怎样？**
A: `PermissionServiceImpl.getDeptDataPermission()` 中，如果用户没有角色，设置 `self = true`，即只能查看自己创建的数据。

**Q: 多个角色的数据范围如何合并？**
A: 取并集。遍历所有角色，`deptIds` 合并，`self` 取 OR，`all` 取 OR。
