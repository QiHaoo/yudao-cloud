# 01 - 总览：数据权限全景

## 场景引入

想象你正在运营一个电商后台管理系统。系统里有各种数据：订单、客户、商品、财务报表……现在来了三个角色：

- **超级管理员**：能看到所有数据，没有限制。
- **华东区域经理**：只能看到华东区域的订单和客户。
- **销售小王**：只能看到自己跟进的客户和自己签下的订单。

如果系统不做任何处理，这三个人登录系统后，看到的将是**完全相同的数据列表**。区域经理能看到其他区域的订单，销售小王能看到同事的客户信息——这显然不行。

这就是**数据权限**要解决的问题：**同一个系统、同一个页面、同一个接口，不同用户看到的数据范围不同**。

## 数据权限 vs 功能权限

在学习数据权限之前，有必要把它和功能权限做一个彻底的区分。很多人容易把两者混为一谈，但它们解决的是完全不同的问题。

### 功能权限：能不能操作

功能权限（也叫菜单权限、操作权限）控制的是**用户能否访问某个功能**。

```
用户点击"订单管理"菜单
    → 系统检查：该用户的角色是否有"订单管理"菜单权限？
        → 有：显示菜单，进入页面
        → 无：菜单不显示，或提示"无权限"
```

功能权限的典型实现是 RBAC（基于角色的访问控制）：角色绑定菜单，用户绑定角色，通过 `@PreAuthorize` 注解在接口层做校验。

### 数据权限：能看到哪些数据

数据权限控制的是**用户能看到哪些数据行**。

```
用户进入"订单管理"页面，查询订单列表
    → 系统返回所有订单？还是只返回该用户所在区域的订单？
        → 数据权限决定
```

关键区别在于：

| 维度 | 功能权限 | 数据权限 |
|------|---------|---------|
| 控制粒度 | 接口/菜单级别 | 数据行级别 |
| 典型问题 | "你能不能看订单列表？" | "你能看哪些订单？" |
| 实现位置 | Controller 层（Spring Security） | Mapper 层（SQL 改写） |
| 注解 | `@PreAuthorize` | `@DataPermission` |

用一句话总结：**功能权限管"门"，数据权限管"窗"**。功能权限决定你能不能进这个房间，数据权限决定你进了房间之后能从窗户看到多远的风景。

## 数据权限 vs 多租户

在本系列的第 10 个组件中，我们学习了多租户。多租户和数据权限都涉及"数据过滤"，但它们的定位完全不同。

### 多租户：租户间的隔离

多租户解决的是**不同租户之间的数据隔离**。电商 SaaS 平台中，商家 A 和商家 B 的数据必须严格隔离——商家 A 绝对不能看到商家 B 的订单。

```
商家 A 登录 → 所有 SQL 自动追加 WHERE tenant_id = A
商家 B 登录 → 所有 SQL 自动追加 WHERE tenant_id = B
```

多租户是**全局生效**的，不需要业务代码做任何特殊处理。

### 数据权限：租户内的过滤

数据权限解决的是**同一个租户内部的数据可见范围**。商家 A 内部，区域经理只能看本区域数据，销售只能看自己的数据。

```
商家 A 的区域经理 → WHERE tenant_id = A AND dept_id = 100
商家 A 的销售     → WHERE tenant_id = A AND user_id = 42
```

### 两者的协作关系

在实际运行中，多租户和数据权限是**叠加生效**的。一个 SQL 最终可能同时被两个组件改写：

```
原始 SQL：SELECT * FROM trade_order

多租户改写后：SELECT * FROM trade_order WHERE tenant_id = 1

数据权限改写后：SELECT * FROM trade_order WHERE tenant_id = 1 AND dept_id = 100
```

两者的执行顺序是：**先多租户，后数据权限**。这在自动配置中通过 `MyBatisUtils.addInterceptor()` 的顺序来保证。

| 维度 | 多租户 | 数据权限 |
|------|-------|---------|
| 隔离范围 | 租户之间 | 租户内部 |
| 生效范围 | 全局，所有表 | 按规则配置，指定表 |
| 配置方式 | `TenantDatabaseInterceptor` 自动追加 | `@DataPermission` 注解 + `DataPermissionRule` |
| 典型场景 | 商家 A 看不到商家 B 的数据 | 区域经理看不到其他区域的数据 |

### 请求的完整生命周期

让我们跟踪一个完整的请求，看看数据权限是如何生效的：

```
1. 浏览器发起请求
   GET /admin-api/trade/order/page

2. Controller 方法被调用
   ┌─────────────────────────────────────────────────────┐
   │ @DataPermission                                      │
   │ public CommonResult<PageResult<OrderRespVO>> page()  │
   └─────────────────────────────────────────────────────┘
                          │
                          ▼
3. AOP 拦截：DataPermissionAnnotationInterceptor
   - 读取 @DataPermission 注解
   - 将注解信息压入 DataPermissionContextHolder 栈
                          │
                          ▼
4. Service 层执行业务逻辑
                          │
                          ▼
5. Mapper 层执行 SQL
   ┌─────────────────────────────────────────────────────┐
   │ SELECT * FROM trade_order WHERE status = 1           │
   └─────────────────────────────────────────────────────┘
                          │
                          ▼
6. MyBatis-Plus DataPermissionInterceptor 拦截
   - 调用 DataPermissionRuleHandler.getSqlSegment()
                          │
                          ▼
7. DataPermissionRuleHandler 处理流程
   - 检查 skipPermissionCheck()？→ 跳过
   - 通过 DataPermissionRuleFactory 获取规则列表
   - 遍历规则，匹配表名
   - 调用 rule.getExpression() 获取 WHERE 条件
   - 用 AND 组合所有条件
                          │
                          ▼
8. SQL 被改写
   ┌─────────────────────────────────────────────────────┐
   │ SELECT * FROM trade_order                           │
   │ WHERE status = 1                                    │
   │   AND dept_id IN (100, 101, 102)                    │
   └─────────────────────────────────────────────────────┘
                          │
                          ▼
9. 返回过滤后的结果

10. AOP 退出：DataPermissionAnnotationInterceptor
    - 将注解信息从 DataPermissionContextHolder 栈弹出
```

| `DataPermissionUtils` | core/util | 工具类，提供编程式禁用数据权限的能力 |
| `DataPermissionRequestInterceptor` | core/rpc | Feign 拦截器，跨服务传播数据权限上下文 |
| `DataPermissionRpcWebFilter` | core/rpc | RPC 请求过滤器，恢复数据权限上下文 |

## 五级数据范围

数据权限的"范围"由 `DataScopeEnum` 定义，共五级：

```
┌─────────────────────────────────────────────────────────────┐
│                     五级数据范围                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ALL (全部)                                                │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  能看到所有数据，不加任何过滤条件                      │   │
│   │  典型角色：超级管理员                                 │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   DEPT_CUSTOM (指定部门)                                     │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  能看到指定的若干部门的数据                           │   │
│   │  典型角色：运营总监，需要看华东+华南两个区域           │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   DEPT_ONLY (本部门)                                         │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  只能看到自己所在部门的数据                           │   │
│   │  典型角色：部门经理                                   │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   DEPT_AND_CHILD (本部门及子部门)                            │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  能看到自己所在部门及其所有子部门的数据                │   │
│   │  典型角色：区域经理，能看到区域下所有门店的数据         │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   SELF (仅本人)                                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  只能看到自己创建的数据                               │   │
│   │  典型角色：普通销售                                   │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

源码定义如下：

```java
// 文件路径：yudao-module-system/yudao-module-system-api/src/main/java/cn/iocoder/yudao/module/system/enums/permission/DataScopeEnum.java
@Getter
@AllArgsConstructor
public enum DataScopeEnum implements ArrayValuable<Integer> {

    ALL(1),              // 全部数据权限
    DEPT_CUSTOM(2),      // 指定部门数据权限
    DEPT_ONLY(3),        // 部门数据权限
    DEPT_AND_CHILD(4),   // 部门及以下数据权限
    SELF(5);             // 仅本人数据权限

    private final Integer scope;
}
```

这五级范围不是互斥的——一个用户可能拥有多个角色，每个角色有不同的数据范围。最终的可见数据是**所有角色数据范围的并集**。这个计算逻辑在 `PermissionServiceImpl.getDeptDataPermission()` 中实现。

## 思考题

1. 数据权限和多租户都通过 SQL 改写实现数据过滤，为什么它们需要分成两个独立的组件，而不是合并成一个？
2. 五级数据范围中，`DEPT_AND_CHILD` 需要递归查询子部门。如果部门层级很深（比如 10 层），这对性能会有什么影响？项目是如何缓解这个问题的？
3. `@DataPermission` 默认是开启的（`enable = true`），为什么选择"默认开启"而不是"默认关闭"？这两种策略各有什么优缺点？
