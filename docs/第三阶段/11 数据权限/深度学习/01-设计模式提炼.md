# 01 - 设计模式提炼：数据权限组件

本文从数据权限组件中提炼可复用的设计模式，帮助读者在遇到类似"动态 SQL 增强"需求时，能够举一反三地设计出灵活、可扩展的方案。

## 策略模式：DataPermissionRule 接口

### 模式识别

数据权限组件最核心的设计是 `DataPermissionRule` 接口。它只有两个方法，却定义了一个完整的"过滤策略"：

```java
public interface DataPermissionRule {
    Set<String> getTableNames();                              // 我负责哪些表？
    Expression getExpression(String tableName, Alias alias);  // 过滤条件是什么？
}
```

这是典型的**策略模式（Strategy Pattern）**：将"如何过滤"的算法封装成独立的策略接口，让调用方（`DataPermissionRuleHandler`）不关心具体实现，只按接口约定调用。

### 策略模式的结构对比

```
经典策略模式：
  Context ──uses──> Strategy (接口)
                      │
                      ├── ConcreteStrategyA
                      ├── ConcreteStrategyB
                      └── ConcreteStrategyC

数据权限组件：
  DataPermissionRuleHandler ──uses──> DataPermissionRule (接口)
                                        │
                                        ├── DeptDataPermissionRule（基于部门）
                                        ├── RegionDataPermissionRule（自定义：基于区域）
                                        └── CustomerLevelRule（自定义：基于客户等级）
```

### 为什么策略模式适合这个场景

在电商系统中，数据过滤的规则是高度可变的：

- 系统管理模块需要按**部门**过滤
- 商品模块可能需要按**仓库**过滤
- 促销模块可能需要按**活动范围**过滤
- CRM 模块需要按**客户等级**过滤

如果把这些规则都硬编码到一个处理器里，代码会变成一团意大利面。策略模式的精髓在于：**把变化的部分抽离成接口，让不变的部分（SQL 拦截 + 条件拼接）保持稳定**。

### 通用化提炼

当你遇到以下特征的需求时，可以考虑策略模式：

| 特征 | 数据权限组件中的体现 |
|------|---------------------|
| 核心算法有多种变体 | 不同表有不同的过滤逻辑 |
| 变体可能在运行时切换 | 通过 `@DataPermission` 注解选择规则 |
| 变体可能由业务方扩展 | 业务模块实现 `DataPermissionRule` 接口 |
| 算法的调用方式统一 | 都是 `getTableNames()` + `getExpression()` |

### 电商场景举例：促销规则引擎

假设电商系统需要支持多种促销策略（满减、折扣、秒杀、拼团），每种策略的计算逻辑不同，但都需要在订单结算时被调用。可以参考数据权限的设计：

```java
// 策略接口
public interface PromotionRule {
    Set<String> getApplicableProductCategories();  // 适用于哪些商品分类
    BigDecimal calculateDiscount(OrderDO order);   // 计算折扣金额
}

// 策略工厂
public interface PromotionRuleFactory {
    List<PromotionRule> getApplicableRules(Long orderId);
}

// 处理器
public class PromotionCalculationHandler {
    private final PromotionRuleFactory ruleFactory;

    public BigDecimal calculate(Long orderId) {
        List<PromotionRule> rules = ruleFactory.getApplicableRules(orderId);
        BigDecimal totalDiscount = BigDecimal.ZERO;
        for (PromotionRule rule : rules) {
            totalDiscount = totalDiscount.add(rule.calculateDiscount(order));
        }
        return totalDiscount;
    }
}
```

这个结构和 `DataPermissionRuleHandler` + `DataPermissionRuleFactory` + `DataPermissionRule` 完全同构。

## 责任链模式：多规则 AND 组合

### 模式识别

`DataPermissionRuleHandler.getSqlSegment()` 中有一个关键的循环：

```java
Expression allExpression = null;
for (DataPermissionRule rule : rules) {
    String tableName = MyBatisUtils.getTableName(table);
    if (!rule.getTableNames().contains(tableName)) {
        continue;
    }
    Expression oneExpress = rule.getExpression(tableName, table.getAlias());
    if (oneExpress == null) {
        continue;
    }
    allExpression = allExpression == null ? oneExpress
            : new AndExpression(allExpression, oneExpress);
}
```

这个循环将多个规则的条件用 `AND` 串接起来，形成一条**责任链**。每个规则只负责自己关心的表，不关心的表直接 `continue` 跳过。

### 与经典责任链的区别

经典的责任链模式中，每个处理器可以选择"处理"或"传递给下一个"。数据权限的规则链略有不同：

| 维度 | 经典责任链 | 数据权限规则链 |
|------|-----------|--------------|
| 处理方式 | 通常只有一个处理器实际处理 | 所有匹配的规则都会参与处理 |
| 组合方式 | 选择性传递 | AND 组合（所有条件叠加） |
| 终止条件 | 某个处理器处理后终止 | 遍历完所有规则 |

更准确地说，这是一个**管道模式（Pipeline Pattern）**或**组合过滤器模式**：每个规则独立生成条件，最终所有条件被组合成一个复合条件。

### 通用化提炼

当你遇到以下特征的需求时，可以考虑类似的规则链设计：

| 特征 | 数据权限组件中的体现 |
|------|---------------------|
| 多个过滤条件需要叠加 | 部门条件 AND 区域条件 AND 客户等级条件 |
| 每个过滤器只关心自己的领域 | `getTableNames()` 做表名匹配 |
| 过滤器数量可扩展 | 新增 `DataPermissionRule` Bean 即可 |
| 过滤器之间互不依赖 | 每个规则独立生成 `Expression` |

### 电商场景举例：订单校验链

电商下单时需要经过多个校验（库存校验、优惠校验、地址校验、支付校验），每个校验独立但需要全部通过：

```java
public interface OrderValidationRule {
    String getName();
    ValidationResult validate(OrderDO order);
}

public class OrderValidationChain {
    private final List<OrderValidationRule> rules;

    public ValidationResult validate(OrderDO order) {
        for (OrderValidationRule rule : rules) {
            ValidationResult result = rule.validate(order);
            if (!result.isSuccess()) {
                return result;  // 校验失败，立即返回
            }
        }
        return ValidationResult.success();
    }
}
```

## 注解驱动：@DataPermission 声明式配置

### 模式识别

`@DataPermission` 注解是数据权限组件的声明式入口。它将数据权限的控制从"编码"变为"声明"：

```java
// 声明式：一目了然
@DataPermission(enable = false)
public AdminUserDO getUserByUsername(String username) {
    return userMapper.selectByUsername(username);
}

// 命令式：需要理解内部实现
public AdminUserDO getUserByUsername(String username) {
    DataPermissionUtils.executeIgnore(() -> {
        return userMapper.selectByUsername(username);
    });
}
```

### 注解驱动的三层架构

数据权限组件的注解驱动采用了三层架构：

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：注解定义（@DataPermission）                          │
│  职责：声明"要做什么"                                         │
│  特点：纯数据，无逻辑                                         │
├─────────────────────────────────────────────────────────────┤
│  第二层：AOP 拦截（Advisor + Interceptor）                    │
│  职责：拦截注解，管理上下文                                    │
│  特点：通用框架代码，与业务无关                                │
├─────────────────────────────────────────────────────────────┤
│  第三层：业务处理器（DataPermissionRuleHandler）               │
│  职责：根据上下文执行具体的 SQL 改写                           │
│  特点：业务逻辑，依赖上下文                                    │
└─────────────────────────────────────────────────────────────┘
```

这种分层的好处是：**注解可以自由演化（增加属性），AOP 层保持稳定，业务处理器按需实现**。

### 上下文栈设计：支持嵌套调用

`DataPermissionContextHolder` 使用 `LinkedList` 作为栈结构，这是一个精妙的设计：

```java
private static final ThreadLocal<LinkedList<DataPermission>> DATA_PERMISSIONS =
        TransmittableThreadLocal.withInitial(LinkedList::new);
```

栈结构解决了注解嵌套的问题：

```
方法 A: @DataPermission(enable = true)
    └── 方法 B: @DataPermission(enable = false)
            └── 方法 C: @DataPermission(enable = true)

栈变化：
  方法 A 进入 → 栈: [A(true)]
  方法 B 进入 → 栈: [A(true), B(false)]  ← get() 返回 B(false)
  方法 C 进入 → 栈: [A(true), B(false), C(true)]  ← get() 返回 C(true)
  方法 C 退出 → 栈: [A(true), B(false)]
  方法 B 退出 → 栈: [A(true)]
  方法 A 退出 → 栈: []
```

### 通用化提炼

注解驱动适合以下场景：

| 特征 | 数据权限组件中的体现 |
|------|---------------------|
| 行为需要在方法级别精细控制 | 每个 Controller 方法的数据权限不同 |
| 默认行为明确 | 默认开启数据权限 |
| 需要支持嵌套 | 方法 A 调用方法 B，两者注解不同 |
| 与 AOP 天然契合 | Spring AOP 自动拦截注解 |

### 电商场景举例：操作日志注解

```java
@OperationLog(module = "订单", type = "查询")
@GetMapping("/page")
public CommonResult<PageResult<OrderRespVO>> getOrderPage() { ... }

@OperationLog(module = "订单", type = "修改")
@PostMapping("/update")
public CommonResult<Boolean> updateOrder(@Valid @RequestBody OrderUpdateReqVO reqVO) { ... }
```

通过 `@OperationLog` 注解声明操作日志的元数据，AOP 拦截器自动记录操作日志，业务代码无需关心日志逻辑。

## SQL 改写模式：JSqlParser 动态注入 WHERE 条件

### 模式识别

数据权限组件的核心技术手段是 SQL 改写——在 SQL 执行前拦截，动态注入 WHERE 条件。这本质上是一个**装饰器模式**的变体：不改变原始 SQL 的结构，只在 WHERE 子句上"装饰"额外的条件。

```
原始 SQL：
  SELECT * FROM trade_order WHERE status = 1

装饰后：
  SELECT * FROM trade_order WHERE status = 1 AND dept_id IN (100, 101, 102)
```

### SQL 改写的分层设计

```
┌─────────────────────────────────────────────────────────────┐
│  MyBatis-Plus DataPermissionInterceptor（框架层）             │
│  职责：拦截 SQL，调用 Handler，拼接条件                        │
│  扩展点：MultiDataPermissionHandler 接口                      │
├─────────────────────────────────────────────────────────────┤
│  DataPermissionRuleHandler（组件层）                          │
│  职责：遍历规则，收集条件                                      │
│  扩展点：DataPermissionRule 接口                              │
├─────────────────────────────────────────────────────────────┤
│  DeptDataPermissionRule（业务层）                             │
│  职责：为特定表生成 WHERE 条件                                 │
│  输出：JSQLParser Expression 对象                             │
└─────────────────────────────────────────────────────────────┘
```

### 为什么返回 Expression 而不是 SQL 字符串

这是一个关键的设计决策。用类比来理解：

| 方案 | 类比 | 安全性 | 可组合性 |
|------|------|--------|----------|
| 返回 SQL 字符串 | 字符串拼 HTML | 有 SQL 注入风险 | 手动拼接 AND/OR，容易出错 |
| 返回 Expression 对象 | DOM API 构建 HTML | 类型安全，无注入风险 | 框架自动处理 AND 拼接 |

这个决策体现了**面向对象的类型安全**原则：用对象结构代替字符串拼接，让编译器帮你检查错误。

### 通用化提炼

SQL 改写模式适合以下场景：

| 特征 | 数据权限组件中的体现 |
|------|---------------------|
| 需要在不改业务代码的情况下增强 SQL | 自动追加 WHERE 条件 |
| 增强逻辑可配置 | 不同表、不同规则 |
| 增强逻辑可叠加 | 多个规则的条件 AND 组合 |
| 需要保证安全性 | Expression 对象避免 SQL 注入 |

### 电商场景举例：数据脱敏

在查询用户手机号、银行卡号等敏感信息时，可以在 SQL 层自动脱敏：

```java
public class DataMaskingRule implements DataMaskingHandler {
    @Override
    public Expression getMaskingExpression(Table table, String columnName) {
        // 将 phone 字段替换为 CONCAT(LEFT(phone, 3), '****', RIGHT(phone, 4))
        Function left = new Function().withName("LEFT")
            .withParameters(new ExpressionList(new Column(columnName), new LongValue(3)));
        // ...
    }
}
```

## 工厂模式：DataPermissionRuleFactory 的规则过滤

### 模式识别

`DataPermissionRuleFactory` 不仅是规则的容器，更是规则的"调度中心"。它的 `getDataPermissionRule()` 方法根据当前上下文（注解信息）决定哪些规则应该生效：

```java
public List<DataPermissionRule> getDataPermissionRule(String mappedStatementId) {
    DataPermission dataPermission = DataPermissionContextHolder.get();
    if (dataPermission == null) return rules;           // 无注解，全部生效
    if (!dataPermission.enable()) return emptyList();   // 禁用，不生效
    if (isTranslateCall()) return emptyList();           // 数据翻译，不生效
    if (ArrayUtil.isNotEmpty(dataPermission.includeRules()))
        return filterByInclude(rules, dataPermission);  // 白名单过滤
    if (ArrayUtil.isNotEmpty(dataPermission.excludeRules()))
        return filterByExclude(rules, dataPermission);  // 黑名单过滤
    return rules;                                        // 默认全部生效
}
```

这是一个**工厂模式**的变体——不是创建对象，而是根据上下文选择对象。更准确地说，它是**策略选择器（Strategy Selector）**。

### 决策树设计

工厂的过滤逻辑形成了一个清晰的决策树：

```
getDataPermissionRule()
    │
    ├── 规则列表为空 → 返回空
    │
    ├── 无注解 → 返回全部规则（默认开启）
    │
    ├── enable = false → 返回空（禁用）
    │
    ├── 是数据翻译调用 → 返回空（特殊处理）
    │
    ├── 有 includeRules → 返回白名单规则
    │
    ├── 有 excludeRules → 返回排除后的规则
    │
    └── 其他 → 返回全部规则
```

决策树的每个分支都有明确的业务含义，这种设计让代码的可读性和可维护性极高。

### 优先级设计

`includeRules > excludeRules > 默认全部` 的优先级设计是一个重要的工程决策：

```
优先级：enable = false > includeRules > excludeRules > 默认全部

设计原则：
  - 白名单（includeRules）比黑名单（excludeRules）更精确
  - 精确的声明优先于模糊的声明
  - 禁用是最强的声明，优先级最高
```

### 通用化提炼

工厂模式 + 决策树适合以下场景：

| 特征 | 数据权限组件中的体现 |
|------|---------------------|
| 对象集合需要根据上下文动态选择 | 不同注解选择不同规则 |
| 选择逻辑有明确的优先级 | includeRules > excludeRules |
| 选择逻辑可能扩展 | 未来可能增加新的过滤条件 |
| 选择逻辑需要可测试 | 决策树的每个分支都可以独立测试 |

## AOP 与关注点分离

### 模式识别

数据权限组件是 AOP（面向切面编程）的经典应用。它将"数据过滤"这个横切关注点从业务代码中抽离出来：

```
没有 AOP 的代码（耦合）：
  public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
      // 业务逻辑
      LambdaQueryWrapper<OrderDO> wrapper = ...;
      // 数据权限逻辑（与业务耦合）
      Long deptId = securityUtils.getLoginUser().getDeptId();
      wrapper.in(OrderDO::getDeptId, getVisibleDeptIds(deptId));
      return orderMapper.selectPage(reqVO, wrapper);
  }

使用 AOP 的代码（分离）：
  @DataPermission  // 声明式，一行搞定
  public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
      // 纯业务逻辑
      return orderMapper.selectPage(reqVO);
  }
```

### 关注点分离的体现

数据权限组件将以下关注点分离到不同的层次：

| 关注点 | 负责的类 | 层次 |
|--------|---------|------|
| "数据权限是否开启" | `@DataPermission` 注解 | 声明层 |
| "注解在哪里生效" | `DataPermissionAnnotationAdvisor` | AOP 切面层 |
| "上下文如何传递" | `DataPermissionContextHolder` | 上下文层 |
| "规则如何选择" | `DataPermissionRuleFactory` | 工厂层 |
| "SQL 如何改写" | `DataPermissionRuleHandler` | SQL 处理层 |
| "条件如何生成" | `DataPermissionRule` 实现类 | 规则层 |
| "跨服务如何传播" | `DataPermissionRequestInterceptor` | RPC 层 |

每个层次只关注自己的职责，层次之间通过接口解耦。这是**单一职责原则（SRP）**和**依赖倒置原则（DIP）**的完美体现。

### 与 SOLID 原则的关联

| SOLID 原则 | 数据权限组件中的体现 |
|-----------|---------------------|
| **S**（单一职责） | 每个类只做一件事：注解只声明，AOP 只拦截，Handler 只改写 |
| **O**（开闭原则） | 新增规则只需实现 `DataPermissionRule` 接口，不修改现有代码 |
| **L**（里氏替换） | 任何 `DataPermissionRule` 实现都可以替换 `DeptDataPermissionRule` |
| **I**（接口隔离） | `DataPermissionRule` 接口只有两个方法，职责清晰 |
| **D**（依赖倒置） | `DataPermissionRuleHandler` 依赖 `DataPermissionRuleFactory` 接口，不依赖具体实现 |

## 设计模式的组合使用

数据权限组件的精妙之处在于，它不是单一模式的应用，而是多种模式的有机组合：

```
┌─────────────────────────────────────────────────────────────┐
│                    设计模式组合全景                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  注解驱动（@DataPermission）                                 │
│      │                                                      │
│      ▼                                                      │
│  AOP 拦截（Advisor + Interceptor）                           │
│      │                                                      │
│      ▼                                                      │
│  工厂模式（DataPermissionRuleFactory）                       │
│      │  ┌─ 策略模式（DataPermissionRule 接口）               │
│      ├─→│  ├── DeptDataPermissionRule                       │
│      │  └── ...                                             │
│      │                                                      │
│      ▼                                                      │
│  责任链/管道模式（多规则 AND 组合）                            │
│      │                                                      │
│      ▼                                                      │
│  SQL 改写模式（JSQLParser Expression）                       │
│      │                                                      │
│      ▼                                                      │
│  装饰器模式（在原始 SQL 上追加 WHERE 条件）                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

每种模式解决一个特定的问题，组合在一起形成了一个完整的、可扩展的数据权限框架。

## 反模式警示

### 反模式一：把所有规则塞进一个类

```java
// 反模式：一个类处理所有表的数据权限
public class AllInOneDataPermissionRule implements DataPermissionRule {
    @Override
    public Expression getExpression(String tableName, Alias alias) {
        if ("trade_order".equals(tableName)) {
            // 订单表的过滤逻辑...
        } else if ("trade_customer".equals(tableName)) {
            // 客户表的过滤逻辑...
        } else if ("system_dept".equals(tableName)) {
            // 部门表的过滤逻辑...
        }
        // 20 个 else if...
    }
}
```

**问题**：违反开闭原则，每次新增表都要修改这个类，最终会变成一个几千行的"上帝类"。

**正确做法**：每个表（或每类表）一个 `DataPermissionRule` 实现。

### 反模式二：在业务代码中手动拼接 SQL 条件

```java
// 反模式：业务代码中直接拼接数据权限条件
public PageResult<OrderDO> getOrderPage(OrderPageReqVO reqVO) {
    String sql = "SELECT * FROM trade_order WHERE status = 1";
    if (!isAdmin()) {
        sql += " AND dept_id IN (" + getVisibleDeptIds() + ")";
    }
    return orderMapper.rawQuery(sql);
}
```

**问题**：SQL 注入风险、代码重复、难以维护、容易遗漏。

**正确做法**：通过 `@DataPermission` 注解 + SQL 改写，在框架层统一处理。

### 反模式三：过度使用 @DataPermission(enable = false)

```java
// 反模式：到处禁用数据权限
@DataPermission(enable = false)
public PageResult<OrderDO> getOrderPage() { ... }

@DataPermission(enable = false)
public OrderDO getOrder(Long id) { ... }

@DataPermission(enable = false)
public List<OrderDO> getOrderList() { ... }
```

**问题**：如果到处禁用，数据权限形同虚设，系统回到"谁都能看所有数据"的状态。

**正确做法**：只在确实需要的地方禁用（如查询自己的个人信息、避免递归），其他地方保持默认开启。

---

## 总结

数据权限组件是设计模式的教科书级应用。它的核心价值不在于"用了哪些模式"，而在于**每种模式都用在了最合适的地方**：

| 模式 | 解决的问题 | 选择的理由 |
|------|-----------|-----------|
| 策略模式 | 过滤逻辑可变 | 不同表需要不同的过滤规则 |
| 责任链/管道 | 多条件叠加 | 多个规则的条件需要 AND 组合 |
| 注解驱动 | 声明式配置 | 方法级别的精细控制 |
| SQL 改写 | 透明增强 | 不改业务代码，自动过滤 |
| 工厂模式 | 规则选择 | 根据上下文动态选择规则 |
| AOP | 关注点分离 | 数据过滤逻辑与业务逻辑解耦 |

当你在自己的项目中遇到类似的需求时，不妨回到这个组件，看看它是如何用这些模式解决问题的。设计模式不是银弹，但用对了地方，就是最好的武器。
