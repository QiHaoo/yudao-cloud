# 设计模式提炼

本文从基础设施模块中提炼出五个核心设计模式，通过项目中的真实代码示例，帮助读者理解这些模式在实际业务中的应用，并掌握可迁移的通用经验。

---

## 一、策略模式：FileClient 的多后端实现

### 1.1 模式定义

**策略模式（Strategy Pattern）**定义了一系列算法，将每个算法封装起来，并使它们可以互相替换。策略模式让算法独立于使用它的客户端而变化。

在电商系统中，文件上传是一个典型场景：商品图片、订单附件、用户头像等都需要存储。不同的存储后端（本地磁盘、阿里云 OSS、MinIO、FTP）各有优劣，策略模式让我们可以在运行时灵活切换。

### 1.2 项目中的应用

项目通过 `FileClient` 接口统一文件操作，每种存储后端实现为一个独立策略：

```
┌─────────────────────────────────────────────────────────────┐
│                      FileClient 接口                         │
├─────────────────────────────────────────────────────────────┤
│  + upload(content, path, type): String                      │
│  + delete(path): void                                       │
│  + getContent(path): byte[]                                 │
│  + presignPutUrl(path): String                              │
│  + presignGetUrl(url, expiration): String                   │
└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
        ▼                     ▼                     ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│ LocalFileClient│   │ S3FileClient  │   │  FtpFileClient│
│  (本地磁盘)    │   │ (S3/阿里云/   │   │   (FTP服务器) │
│               │   │  MinIO/腾讯云) │   │               │
└───────────────┘   └───────────────┘   └───────────────┘
        │
        ▼
┌───────────────┐   ┌───────────────┐
│ DBFileClient  │   │SftpFileClient │
│  (数据库存储)  │   │  (SFTP服务器) │
└───────────────┘   └───────────────┘
```

**源码位置**：`yudao-module-infra/yudao-module-infra-server/src/main/java/cn/iocoder/yudao/module/infra/framework/file/core/client/`

### 1.3 代码示例

**策略接口定义**（`FileClient.java`）：

```java
public interface FileClient {
    Long getId();
    String upload(byte[] content, String path, String type) throws Exception;
    void delete(String path) throws Exception;
    byte[] getContent(String path) throws Exception;
    default String presignPutUrl(String path) {
        throw new UnsupportedOperationException("不支持的操作");
    }
    default String presignGetUrl(String url, Integer expirationSeconds) {
        throw new UnsupportedOperationException("不支持的操作");
    }
}
```

**具体策略实现**（`S3FileClient.java`）：

```java
public class S3FileClient extends AbstractFileClient<S3FileClientConfig> {
    private S3Client client;
    private S3Presigner presigner;

    @Override
    protected void doInit() {
        // 初始化 S3 客户端
        AwsCredentialsProvider credentialsProvider = StaticCredentialsProvider.create(
                AwsBasicCredentials.create(config.getAccessKey(), config.getAccessSecret()));
        client = S3Client.builder()
                .credentialsProvider(credentialsProvider)
                .region(region)
                .endpointOverride(endpoint)
                .build();
    }

    @Override
    public String upload(byte[] content, String path, String type) {
        PutObjectRequest putRequest = PutObjectRequest.builder()
                .bucket(config.getBucket())
                .key(path)
                .contentType(type)
                .build();
        client.putObject(putRequest, RequestBody.fromBytes(content));
        return presignGetUrl(path, null);
    }
    // ...
}
```

**策略枚举注册**（`FileStorageEnum.java`）：

```java

    DB(1, DBFileClientConfig.class, DBFileClient.class),
```

### 1.4 可迁移经验

1. **枚举注册策略类**：用枚举将策略编号与实现类映射，避免 `if-else` 或 `switch` 硬编码
2. **泛型约束配置类型**：`AbstractFileClient<Config extends FileClientConfig>` 确保每个实现有强类型配置
3. **默认方法提供可选能力**：`presignPutUrl` 等方法用 `default` 实现，子类按需覆盖
4. **配置与客户端分离**：`FileClientConfig` 用 `@JsonTypeInfo` 支持多态序列化，配置存数据库，运行时反序列化

## 二、模板方法模式：代码生成器的 Velocity 模板

### 2.1 模式定义

**模板方法模式（Template Method Pattern）**在父类中定义算法骨架，将某些步骤延迟到子类实现。模板方法使得子类可以在不改变算法结构的情况下，重新定义算法的某些步骤。

在代码生成场景中，生成流程是固定的：读取表结构 → 构建上下文 → 渲染模板 → 输出文件。但不同类型的代码（Java 后端、Vue 前端、SQL）有不同的模板和输出路径。

### 2.2 项目中的应用

项目使用 Velocity 模板引擎实现代码生成，`CodegenEngine` 类定义了生成流程骨架：

```
┌─────────────────────────────────────────────────────────────┐
│                    CodegenEngine.execute()                   │
├─────────────────────────────────────────────────────────────┤
│  1. initBindingMap()        ← 初始化上下文变量               │
│  2. getTemplates()          ← 获取模板列表                   │
│  3. 遍历模板：                                               │
│     ├─ isSubTemplate()      ← 判断是否子表模板               │
│     ├─ isPageReqVOTemplate()← 判断是否分页请求模板           │
│     └─ generateCode()       ← 渲染模板生成代码               │
│        ├─ formatFilePath()  ← 格式化输出路径                 │
│        ├─ templateEngine.getTemplate().render() ← 渲染       │
│        └─ prettyCode()      ← 格式化代码                     │
└─────────────────────────────────────────────────────────────┘
```

**源码位置**：`yudao-module-infra/yudao-module-infra-server/src/main/java/cn/iocoder/yudao/module/infra/service/codegen/inner/CodegenEngine.java`

### 2.3 代码示例

**生成流程骨架**（`CodegenEngine.java` 第 374-409 行）：

```java
public Map<String, String> execute(DbType dbType, CodegenTableDO table, List<CodegenColumnDO> columns,
                                   List<CodegenTableDO> subTables, List<List<CodegenColumnDO>> subColumnsList) {
    // 1.1 初始化 bindMap 上下文
    Map<String, Object> bindingMap = initBindingMap(dbType, table, columns, subTables, subColumnsList);
    // 1.2 获得模版
    Map<String, String> templates = getTemplates(table.getFrontType());

    // 2. 执行生成
    Map<String, String> result = Maps.newLinkedHashMapWithExpectedSize(templates.size());
    templates.forEach((vmPath, filePath) -> {
        // 2.1 特殊：主子表专属逻辑
        if (isSubTemplate(vmPath)) {
            generateSubCode(table, subTables, result, vmPath, filePath, bindingMap);
            return;
        }
        // 2.2 默认生成
        generateCode(result, vmPath, filePath, bindingMap);
    });
    return result;
}
```

### 2.4 可迁移经验

1. **模板与逻辑分离**：模板文件（`.vm`）定义代码结构，Java 类定义生成逻辑，互不干扰
2. **上下文变量注入**：通过 `bindingMap` 将表结构、字段信息注入模板，模板只负责渲染
3. **路径模板化**：输出路径也用 `${table.moduleName}` 等变量，实现动态路径生成
4. **后处理格式化**：`prettyCode()` 方法统一处理生成后的代码格式，避免模板过于复杂

## 三、观察者模式：@LogRecord 的 AOP 拦截

### 3.1 模式定义

**观察者模式（Observer Pattern）**定义了对象之间的一对多依赖关系，当一个对象状态改变时，所有依赖于它的对象都会得到通知并自动更新。

在业务系统中，操作日志是一个典型的横切关注点。例如：创建订单、修改商品价格、删除用户等操作都需要记录日志，但日志记录不应侵入业务代码。

### 3.2 项目中的应用

项目基于 `mzt-logapi` 实现 `@LogRecord` 注解，通过 AOP 拦截方法执行，自动记录操作日志：

```
┌─────────────────────────────────────────────────────────────┐
│                    业务方法调用                               │
│  @LogRecord(type = "订单", subType = "创建订单", bizNo = "#id")│
│  public Long createOrder(CreateOrderReqVO reqVO) { ... }    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    AOP 拦截器                                │
│  1. 解析 @LogRecord 注解                                    │
│  2. 执行业务方法                                             │
│  3. 构建 LogRecord 对象                                      │
│  4. 调用 ILogRecordService.record()                         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│              LogRecordServiceImpl.record()                   │
│  1. fillUserFields()      ← 填充用户信息                     │
│  2. fillModuleFields()    ← 填充模块信息                     │
│  3. fillRequestFields()   ← 填充请求信息                     │
│  4. operateLogApi.createOperateLogAsync() ← 异步记录         │
└─────────────────────────────────────────────────────────────┘
```

**源码位置**：`yudao-framework/yudao-spring-boot-starter-security/src/main/java/cn/iocoder/yudao/framework/operatelog/core/service/LogRecordServiceImpl.java`

### 3.3 代码示例

**日志记录服务实现**（`LogRecordServiceImpl.java`）：

```java
public class LogRecordServiceImpl implements ILogRecordService {
    @Resource
    private OperateLogCommonApi operateLogApi;

    @Override
    public void record(LogRecord logRecord) {
        OperateLogCreateReqDTO reqDTO = new OperateLogCreateReqDTO();
        try {
            reqDTO.setTraceId(TracerUtils.getTraceId());
            // 补充用户信息
            fillUserFields(reqDTO);
            // 补全模块信息
            fillModuleFields(reqDTO, logRecord);
            // 补全请求信息
            fillRequestFields(reqDTO);
            // 异步记录日志
            operateLogApi.createOperateLogAsync(reqDTO);
        } catch (Throwable ex) {
            log.error("[record][url({}) log({}) 发生异常]", reqDTO.getRequestUrl(), reqDTO, ex);
        }
    }

    private static void fillUserFields(OperateLogCreateReqDTO reqDTO) {
        LoginUser loginUser = SecurityFrameworkUtils.getLoginUser();
        if (loginUser == null) {
            return;
        }
        reqDTO.setUserId(loginUser.getId());
        reqDTO.setUserType(loginUser.getUserType());
    }

    public static void fillModuleFields(OperateLogCreateReqDTO reqDTO, LogRecord logRecord) {
        reqDTO.setType(logRecord.getType());       // 大模块类型，例如：订单
        reqDTO.setSubType(logRecord.getSubType());  // 操作名称，例如：创建订单
        reqDTO.setBizId(Long.parseLong(logRecord.getBizNo())); // 业务编号
        reqDTO.setAction(logRecord.getAction());    // 操作内容
        reqDTO.setExtra(logRecord.getExtra());      // 扩展字段
    }
}
```

### 3.4 可迁移经验

1. **注解驱动**：用 `@LogRecord` 注解标记需要记录日志的方法，业务代码零侵入
2. **SpEL 表达式**：`bizNo = "#id"` 支持从方法参数中动态提取业务标识
3. **异步记录**：日志记录通过 `createOperateLogAsync` 异步执行，不阻塞业务流程
4. **容错处理**：日志记录异常不影响业务执行，只打印错误日志

## 四、过滤器模式：ApiAccessLogFilter 的请求链

### 4.1 模式定义

**过滤器模式（Filter Pattern）**或**责任链模式（Chain of Responsibility Pattern）**通过给多个对象处理请求的机会，将请求的发送者和接收者解耦。请求沿着链传递，直到有一个对象处理它。

在 Web 应用中，HTTP 请求需要经过多个处理环节：认证、授权、日志记录、参数校验等。过滤器模式让我们可以灵活组合这些处理环节。

### 4.2 项目中的应用

项目通过 `ApiAccessLogFilter` 记录 API 访问日志，它继承自 `ApiRequestFilter`，在请求处理前后插入日志记录逻辑：

```
┌─────────────────────────────────────────────────────────────┐
│                      HTTP 请求                               │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                  ApiAccessLogFilter                          │
│  1. 记录开始时间 beginTime                                   │
│  2. 提前获取请求参数（避免 XSS 过滤）                         │
│  3. filterChain.doFilter() ← 继续过滤器链                    │
│  4. 记录结束时间，计算耗时                                    │
│  5. 构建日志对象，异步写入数据库                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                  业务 Controller 处理                        │
└─────────────────────────────────────────────────────────────┘
```

**源码位置**：`yudao-framework/yudao-spring-boot-starter-web/src/main/java/cn/iocoder/yudao/framework/apilog/core/filter/ApiAccessLogFilter.java`

### 4.3 代码示例

**过滤器核心逻辑**（`ApiAccessLogFilter.java` 第 66-84 行）：

```java
@Override
protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain filterChain)
        throws ServletException, IOException {
    // 获得开始时间
    LocalDateTime beginTime = LocalDateTime.now();
    // 提前获得参数，避免 XssFilter 过滤处理
    Map<String, String> queryString = ServletUtils.getParamMap(request);
    String requestBody = ServletUtils.getBody(request);

    try {
        // 继续过滤器
        filterChain.doFilter(request, response);
        // 正常执行，记录日志
        createApiAccessLog(request, beginTime, queryString, requestBody, null);
    } catch (Exception ex) {
        // 异常执行，记录日志
        createApiAccessLog(request, beginTime, queryString, requestBody, ex);
        throw ex;
    }
}
```

**日志构建逻辑**（`ApiAccessLogFilter.java` 第 100-160 行）：

```java
private boolean buildApiAccessLog(ApiAccessLogCreateReqDTO accessLog, HttpServletRequest request,
                                  LocalDateTime beginTime, Map<String, String> queryString,
                                  String requestBody, Exception ex) {
    // 判断：是否要记录操作日志
    HandlerMethod handlerMethod = (HandlerMethod) request.getAttribute(ATTRIBUTE_HANDLER_METHOD);
    ApiAccessLog accessLogAnnotation = null;
    if (handlerMethod != null) {
        accessLogAnnotation = handlerMethod.getMethodAnnotation(ApiAccessLog.class);
        if (accessLogAnnotation != null && BooleanUtil.isFalse(accessLogAnnotation.enable())) {
            return false; // 通过注解控制是否记录
        }
    }

    // 处理用户信息
    accessLog.setUserId(WebFrameworkUtils.getLoginUserId(request))
            .setUserType(WebFrameworkUtils.getLoginUserType(request));

    // 设置请求字段
    accessLog.setTraceId(TracerUtils.getTraceId())
            .setApplicationName(applicationName)
            .setRequestUrl(request.getRequestURI())
            .setRequestMethod(request.getMethod())
            .setUserAgent(ServletUtils.getUserAgent(request))
            .setUserIp(ServletUtils.getClientIP(request));

    // 持续时间
    accessLog.setBeginTime(beginTime)
            .setEndTime(LocalDateTime.now())
            .setDuration((int) LocalDateTimeUtil.between(beginTime, LocalDateTime.now(), ChronoUnit.MILLIS));
    return true;
}
```

**注解控制**（`ApiAccessLog.java`）：

```java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface ApiAccessLog {
    boolean enable() default true;           // 是否记录
    boolean requestEnable() default true;    // 是否记录请求参数
    boolean responseEnable() default false;  // 是否记录响应结果
    String[] sanitizeKeys() default {};      // 敏感参数脱敏
    String operateModule() default "";       // 操作模块
    String operateName() default "";         // 操作名称
    OperateTypeEnum[] operateType() default {}; // 操作类型
}
```

### 4.4 可迁移经验

1. **提前获取参数**：在 `filterChain.doFilter()` 之前获取请求参数，避免被其他过滤器修改
2. **注解控制粒度**：通过 `@ApiAccessLog` 注解灵活控制哪些接口需要记录、记录哪些内容
3. **敏感数据脱敏**：`sanitizeKeys` 支持自定义脱敏字段，内置 `password`、`token` 等默认脱敏
4. **异常也要记录**：`catch` 块中记录日志后重新抛出异常，确保异常请求也有日志

## 五、工厂模式：FileClientFactory 的客户端创建

### 5.1 模式定义

**工厂模式（Factory Pattern）**提供一个创建对象的接口，让子类决定实例化哪个类。工厂方法使一个类的实例化延迟到其子类。

在文件管理场景中，需要根据配置动态创建不同的文件客户端。工厂模式封装了创建逻辑，客户端代码无需关心具体的创建细节。

### 5.2 项目中的应用

项目通过 `FileClientFactory` 和 `FileClientFactoryImpl` 实现文件客户端的创建和管理：


### 5.3 代码示例

**工厂实现**（`FileClientFactoryImpl.java`）：

```java
public class FileClientFactoryImpl implements FileClientFactory {
    /**
     * 文件客户端 Map
     * key：配置编号
     */
    private final ConcurrentMap<Long, AbstractFileClient<?>> clients = new ConcurrentHashMap<>();

    @Override
    public FileClient getFileClient(Long configId) {
        AbstractFileClient<?> client = clients.get(configId);
        if (client == null) {
            log.error("[getFileClient][配置编号({}) 找不到客户端]", configId);
        }
        return client;
    }

    @Override
    @SuppressWarnings("unchecked")
    public <Config extends FileClientConfig> void createOrUpdateFileClient(Long configId, Integer storage, Config config) {
        AbstractFileClient<Config> client = (AbstractFileClient<Config>) clients.get(configId);
        if (client == null) {
            client = this.createFileClient(configId, storage, config);
            client.init();
            clients.put(client.getId(), client);
        } else {
            client.refresh(config);
        }
    }

    @SuppressWarnings("unchecked")
    private <Config extends FileClientConfig> AbstractFileClient<Config> createFileClient(
            Long configId, Integer storage, Config config) {
        FileStorageEnum storageEnum = FileStorageEnum.getByStorage(storage);
        Assert.notNull(storageEnum, String.format("文件配置(%s) 为空", storageEnum));
        // 创建客户端
        return (AbstractFileClient<Config>) ReflectUtil.newInstance(storageEnum.getClientClass(), configId, config);
    }
}
```

**抽象客户端的初始化模板**（`AbstractFileClient.java`）：

```java
public abstract class AbstractFileClient<Config extends FileClientConfig> implements FileClient {
    private final Long id;
    protected Config config;
    private Config originalConfig;

    public final void init() {
        doInit();  // 模板方法，子类实现
        log.debug("[init][配置({}) 初始化完成]", config);
    }

    protected abstract void doInit();

    public final void refresh(Config config) {
        // 判断是否更新
        if (config.equals(this.originalConfig)) {
            return;
        }
        log.info("[refresh][配置({})发生变化，重新初始化]", config);
        this.config = config;
        this.originalConfig = config;
        this.init();
    }
}
```

### 5.4 可迁移经验

1. **缓存客户端实例**：用 `ConcurrentMap` 缓存已创建的客户端，避免重复创建
2. **反射创建实例**：通过 `ReflectUtil.newInstance(storageEnum.getClientClass())` 动态创建，避免 `switch-case`
3. **配置变更检测**：`refresh()` 方法对比原始配置，只在配置变更时重新初始化
4. **模板方法组合**：工厂创建对象，抽象类定义初始化骨架，具体子类实现初始化细节

## 六、设计模式对比与选型

| 模式 | 解决的问题 | 项目中的应用 | 适用场景 |
|------|-----------|-------------|---------|
| 策略模式 | 算法/实现的灵活切换 | FileClient 多后端 | 需要在运行时切换实现 |
| 模板方法 | 流程固定，步骤可变 | CodegenEngine 代码生成 | 算法骨架固定，子步骤可定制 |
| 观察者模式 | 事件通知，解耦 | @LogRecord 日志记录 | 一个事件触发多个动作 |
| 过滤器模式 | 请求链处理 | ApiAccessLogFilter | 请求需要多个处理环节 |
| 工厂模式 | 对象创建封装 | FileClientFactory | 根据配置创建不同实现 |

## 七、通用化提炼

### 7.1 如何识别设计模式的应用场景

1. **看到多个 `if-else` 或 `switch` 处理不同类型** → 考虑策略模式
2. **看到固定流程中的可变步骤** → 考虑模板方法模式
3. **看到"当...发生时，需要做..."** → 考虑观察者模式
4. **看到请求需要经过多个处理环节** → 考虑过滤器/责任链模式
5. **看到根据参数创建不同类型的对象** → 考虑工厂模式

### 7.2 设计模式的组合使用

项目中的设计模式往往组合使用：

- **策略 + 工厂**：`FileClientFactory` 创建不同的 `FileClient` 策略
- **模板方法 + 策略**：`AbstractFileClient` 定义初始化模板，子类实现具体策略
- **观察者 + 过滤器**：`ApiAccessLogFilter` 作为过滤器，内部通过注解驱动日志记录

### 7.3 避免过度设计

1. **简单场景不需要模式**：只有 2-3 种实现时，简单的 `if-else` 可能更清晰
2. **模式是手段不是目的**：先理解问题，再选择合适的模式
3. **YAGNI 原则**：不要为了"可能的扩展"而提前引入复杂模式
