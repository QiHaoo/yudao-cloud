# 02-问题域与设计目标

## 场景引入

想象你是一个电商系统的开发团队负责人。项目启动三个月后，你发现了以下问题：

**问题一：重复的 CRUD 代码**

商品表、订单表、用户表、地址表……每个表都需要手写 Controller、Service、Mapper、VO、DO。代码结构高度相似，却不得不一遍遍重复。新来的程序员小王抱怨："我这周写了 8 个 Controller，代码几乎一模一样，感觉在做体力活。"

**问题二：文件存储的混乱**

商品图片存在本地磁盘，用户头像用的阿里云 OSS，发票 PDF 用的腾讯云 COS。当服务器迁移时，本地文件全部丢失；当需要统一存储方案时，发现每个模块的文件上传代码都不一样。

**问题三：操作日志的缺失**

管理员修改了商品价格，但没有记录是谁改的、什么时候改的、从多少钱改到多少钱。出了问题，只能查数据库 binlog，效率极低。

## 问题域分析

### 问题一：CRUD 的重复劳动

一个典型的业务表需要以下代码：

```
数据库表: product
    │
    ├── Controller 层
    │   ├── ProductController.java          ← 创建、更新、删除、查询、分页
    │   └── vo/
    │       ├── ProductSaveReqVO.java       ← 新增/修改请求 VO
    │       ├── ProductPageReqVO.java       ← 分页查询 VO
    │       └── ProductRespVO.java          ← 响应 VO
    │
    ├── Service 层
    │   ├── ProductService.java             ← 接口
    │   └── ProductServiceImpl.java         ← 实现
    │
    ├── Mapper 层
    │   └── ProductMapper.java              ← MyBatis-Plus Mapper
    │
    └── DO 实体
        └── ProductDO.java                  ← 数据库实体
```

**统计**：一个表需要 7-8 个文件，每个文件 100-300 行代码。如果系统有 50 个业务表，就是 350-400 个文件，其中 80% 的代码是重复的。

**更深层的问题**：
- 代码风格不统一：每个开发者写的代码结构略有差异
- 容易出错：手动编写容易遗漏字段或写错类型
- 维护困难：表结构变更时，需要手动同步修改多个文件

### 问题二：文件存储的多样性

不同的业务场景需要不同的存储方案：

| 场景 | 推荐存储 | 原因 |
|------|---------|------|
| 开发环境 | 本地存储 | 简单快速，无需外部依赖 |
| 小型项目 | 本地存储 / 数据库存储 | 成本低，维护简单 |
| 中型项目 | 阿里云 OSS / 腾讯云 COS | 稳定可靠，按量付费 |
| 大型项目 | MinIO 自建 | 数据自主可控，成本可控 |
| 跨云部署 | S3 兼容存储 | 标准协议，兼容性好 |

**痛点**：
- 存储方案一旦选定，更换成本极高
- 不同模块各自实现文件上传，代码重复
- 缺少统一的文件元数据管理（文件大小、类型、上传时间等）

### 问题三：操作日志的侵入性

传统的手动日志记录方式：

```java
// 方式一：在业务代码中直接记录
public void updateProductPrice(Long productId, BigDecimal newPrice) {
    // 业务逻辑
    ProductDO product = productMapper.selectById(productId);
    BigDecimal oldPrice = product.getPrice();
    product.setPrice(newPrice);
    productMapper.updateById(product);

    // 手动记录日志
    OperateLogDO log = new OperateLogDO();
    log.setUserId(SecurityFrameworkUtils.getLoginUserId());
    log.setOperateType("UPDATE");
    log.setOperateName("修改商品价格");
    log.setRequestParam("{\"productId\":" + productId + ",\"newPrice\":" + newPrice + "}");
    log.setTimestamp(LocalDateTime.now());
    operateLogMapper.insert(log);
}
```

**痛点**：
- **侵入性强**：日志代码与业务逻辑混在一起，违反单一职责原则
- **容易遗漏**：开发者可能忘记记录日志，或者记录的信息不完整
- **维护困难**：日志格式变更时，需要修改所有业务代码
- **性能影响**：同步写入日志可能影响主业务性能

### 问题四：API 访问监控的缺失

系统上线后，面临以下监控需求：
- 哪些接口访问量最大？
- 哪些接口响应时间最长？
- 哪些接口经常报错？
- 异常请求的详细堆栈是什么？

如果没有统一的访问日志，这些问题都无法回答。

## 设计目标

基于以上问题分析，我们定义以下设计目标：

### 目标一：代码生成自动化

```
设计目标：
┌─────────────────────────────────────────────────────────────┐
│  输入：数据库表结构                                           │
│    ↓                                                         │
│  处理：读取表元数据 → 应用模板 → 生成代码                      │
│    ↓                                                         │
│  输出：完整的 Controller/Service/Mapper/VO/DO + 前端页面      │
└─────────────────────────────────────────────────────────────┘

验收标准：
✓ 一键生成，无需手动修改
✓ 生成的代码符合项目规范
✓ 支持单表和主子表
✓ 支持自定义模板
```

### 目标二：文件存储可插拔

```
设计目标：
┌─────────────────────────────────────────────────────────────┐
│  接口层：统一的 FileClient 接口                               │
│    ↓                                                         │
│  实现层：Local / S3 / FTP / SFTP / DB                        │
│    ↓                                                         │
│  配置层：通过配置切换存储后端，业务代码无需修改                  │
└─────────────────────────────────────────────────────────────┘

验收标准：
✓ 业务代码只依赖接口，不依赖具体实现
✓ 切换存储后端只需修改配置
✓ 支持多种存储后端并存（如主配置用 OSS，备份用本地）
✓ 支持文件元数据管理
```

### 目标三：日志记录声明式

```
设计目标：
┌─────────────────────────────────────────────────────────────┐
│  方式：注解 + AOP / Filter                                   │
│    ↓                                                         │
│  效果：业务代码零侵入，自动记录日志                            │
│    ↓                                                         │
│  内容：操作人、操作时间、请求参数、响应结果、异常信息、耗时      │
└─────────────────────────────────────────────────────────────┘

验收标准：
✓ 通过注解标记需要记录日志的方法
✓ 自动捕获方法执行信息
✓ 异步写入，不影响主业务性能
✓ 支持敏感参数脱敏
```

### 目标四：API 访问全面监控

```
设计目标：
┌─────────────────────────────────────────────────────────────┐
│  范围：所有 API 请求                                          │
│    ↓                                                         │
│  方式：Servlet Filter 自动拦截                                │
│    ↓                                                         │
│  记录：请求URL、请求参数、响应结果、耗时、用户信息、异常信息    │
└─────────────────────────────────────────────────────────────┘

验收标准：
✓ 自动记录所有 API 请求，无需手动配置
✓ 支持通过注解排除特定接口
✓ 支持敏感参数脱敏
✓ 日志异步写入，不影响请求性能
```

## 为什么独立成模块？

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **分散在各业务模块** | 就近原则，代码内聚 | 代码重复，维护困难 |
| **放在 yudao-module-system** | 已有系统管理模块 | 职责不清，system 膨胀 |
| **独立为 yudao-module-infra** | 职责清晰，独立演进 | 增加模块数量 |

### 选择独立模块的理由

1. **职责清晰**
   - system 管理业务实体（用户、角色、菜单）
   - infra 提供技术基础设施（代码生成、文件、日志）
   - 两者关注点不同，不应混在一起

2. **独立演进**
   - 文件存储从本地切换到 OSS，只需要修改 infra 模块
   - 代码生成器升级模板，不影响业务模块
   - 日志记录策略变更，对业务模块透明

3. **复用性高**
   - 所有业务模块都可以使用 infra 的能力
   - 避免每个模块重复实现文件上传、日志记录

4. **部署灵活**
   - 微服务模式下，infra 可以独立部署
   - 可以根据负载单独扩展 infra 服务

## 设计原则

### 1. 接口抽象原则

文件存储采用接口抽象，业务代码只依赖 `FileClient` 接口，不依赖具体实现：

```java
// 源码位置：yudao-module-infra-server/.../framework/file/core/client/FileClient.java
public interface FileClient {
    String upload(byte[] content, String path, String type) throws Exception;
    void delete(String path) throws Exception;
    byte[] getContent(String path) throws Exception;
}
```

### 2. 声明式编程原则

日志记录采用注解声明，避免侵入业务代码：

```java
// 源码位置：yudao-framework/.../apilog/core/annotation/ApiAccessLog.java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface ApiAccessLog {
    boolean enable() default true;
    boolean requestEnable() default true;
    boolean responseEnable() default false;
    String[] sanitizeKeys() default {};
}
```

### 3. 模板方法原则

代码生成采用模板方法模式，通过 Velocity 模板引擎生成代码：

```
数据库表结构 ──► 读取元数据 ──► 应用模板 ──► 生成代码
                      │
                      ▼
              Velocity 模板引擎
                      │
                      ▼
              可自定义的模板文件
```

### 4. 异步处理原则

日志写入采用异步方式，避免影响主业务性能：

```
请求处理 ──► 记录日志到内存 ──► 立即返回响应
                    │
                    ▼ (异步)
              批量写入数据库
```

## 本章小结

本章分析了基础设施模块要解决的四个核心问题：

| 问题 | 根因 | 设计目标 |
|------|------|---------|
| CRUD 重复劳动 | 代码结构高度相似 | 代码生成自动化 |
| 文件存储混乱 | 缺少统一抽象 | 文件存储可插拔 |
| 日志记录侵入 | 手动记录代码分散 | 日志记录声明式 |
| 访问监控缺失 | 没有统一拦截机制 | API 访问全面监控 |
