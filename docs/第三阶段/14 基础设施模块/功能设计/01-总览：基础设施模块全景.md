# 01-总览：基础设施模块全景

## 场景引入

假设你正在开发一个电商后台管理系统。项目启动后，你发现团队面临着一系列重复性的技术工作：

- **代码生成**：商品表、订单表、用户表……每个业务表都需要手写 Controller、Service、Mapper、VO、DO，代码结构高度相似，却不得不一遍遍重复
- **文件管理**：商品图片、用户头像、发票PDF……文件上传功能散落在各个业务模块中，当需要从本地存储切换到阿里云 OSS 时，改动量巨大
- **操作日志**：管理员修改商品价格、删除订单、调整库存……这些关键操作需要记录日志用于审计，但手动在每个方法里添加日志代码既繁琐又容易遗漏
- **API 访问日志**：系统上线后，需要监控所有 API 的访问情况，用于性能分析和问题排查

这些功能有一个共同特点：**它们不是业务逻辑，而是技术基础设施**。每个业务系统都需要，但它们本身与具体的业务无关。

## yudao-module-infra 的定位

`yudao-module-infra` 是芋道项目的技术基础设施模块，它的定位非常清晰：

```
┌─────────────────────────────────────────────────────────────────┐
│                        业务模块                                  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │   system     │  │    bpm       │  │   pay        │  ...     │
│  │  系统管理     │  │   工作流     │  │   支付       │          │
│  └──────────────┘  └──────────────┘  └──────────────┘          │
│         │                │                  │                   │
│         └────────────────┼──────────────────┘                   │
│                          ▼                                      │
│              ┌───────────────────────┐                          │
│              │   yudao-module-infra  │                          │
│              │     技术基础设施       │                          │
│              └───────────────────────┘                          │
└─────────────────────────────────────────────────────────────────┘
```

**核心原则**：infra 提供技术能力，业务模块使用技术能力。

- `yudao-module-system` 负责管理业务实体：用户、角色、菜单、字典、部门等
- `yudao-module-infra` 负责提供技术基础设施：代码生成、文件管理、日志记录等

这种分离使得基础设施可以独立演进，不会与业务逻辑耦合。

## 四大子功能

yudao-module-infra 提供四大核心子功能：

| 子功能 | 定位 | 典型场景 |
|--------|------|---------|
| **代码生成** | 从数据库表自动生成 CRUD 代码 | 电商系统新建商品表，一键生成前后端代码 |
| **文件管理** | 统一的文件上传/下载/删除 API | 商品图片上传到阿里云 OSS，用户头像存储在本地 |
| **API 访问日志** | 自动记录所有 API 请求 | 监控系统性能，排查接口异常 |
| **API 错误日志** | 自动记录异常请求 | 定位系统错误，追踪异常堆栈 |

## 与 yudao-module-system 的分工

| 维度 | yudao-module-system | yudao-module-infra |
|------|--------------------|--------------------|
| **定位** | 业务核心 | 技术基础设施 |
| **管理内容** | 用户、角色、菜单、字典、部门、岗位 | 代码生成、文件、日志、配置、数据源 |
| **数据特点** | 业务数据，多租户隔离 | 技术数据，通常全局共享 |
| **依赖方向** | 被业务模块依赖 | 被业务模块依赖 |
| **典型使用** | 权限校验、用户信息查询 | 文件上传、日志记录、代码生成 |

**为什么这样分工？**

1. **职责单一**：system 专注业务实体管理，infra 专注技术能力提供
2. **独立演进**：文件存储从本地切换到 OSS，只需要修改 infra 模块
3. **复用性高**：所有业务模块都可以使用 infra 的能力，无需重复实现
4. **部署灵活**：微服务模式下，infra 可以独立部署和扩展

## 关键设计决策

1. **日志与业务分离**：访问日志通过 Filter 实现，对业务代码零侵入
2. **文件存储可插拔**：通过 FileClient 接口抽象，支持多种存储后端
3. **代码生成模板化**：使用 Velocity 模板引擎，支持自定义模板
4. **配置动态化**：文件配置存储在数据库中，支持运行时切换

## 本章小结

yudao-module-infra 作为技术基础设施模块，通过提供代码生成、文件管理、日志记录等通用能力，避免了业务模块的重复造轮子。它的设计遵循"关注点分离"原则，将技术基础设施与业务逻辑解耦，使得系统更易于维护和扩展。
