# 01-起步：基础设施模块结构

本章从零搭建 `yudao-module-infra` 模块的整体结构，让读者理解每个子模块的职责和模块间的依赖关系。

> **前置阅读**：本章的实现对应功能设计中的「01-总览：基础设施模块全景」和「02-问题域与设计目标」。如果对 infra 模块的定位和设计目标还不清楚，建议先阅读功能设计部分。

## 模块划分

`yudao-module-infra` 采用标准的 `api` + `server` 双子模块结构，与项目中其他业务模块（system、bpm、pay 等）保持一致。

```
yudao-module-infra/
├── pom.xml                          ← 父 POM，packaging=pom
├── yudao-module-infra-api/          ← RPC 接口定义 + DTO + 枚举
└── yudao-module-infra-server/       ← 服务实现（Controller/Service/Mapper/DO）
```

### 父 POM

源码位置：`yudao-module-infra/pom.xml`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <parent>
        <groupId>cn.iocoder.cloud</groupId>
        <artifactId>yudao</artifactId>
        <version>${revision}</version>
    </parent>
    <modelVersion>4.0.0</modelVersion>
    <modules>
        <module>yudao-module-infra-api</module>
        <module>yudao-module-infra-server</module>
    </modules>
    <artifactId>yudao-module-infra</artifactId>
    <packaging>pom</packaging>

    <name>${project.artifactId}</name>
    <description>
        infra 模块，主要提供两块能力：
        1. 我们放基础设施的运维与管理，支撑上层的通用与核心业务。例如说：定时任务的管理、服务器的信息等等
        2. 研发工具，提升研发效率与质量。例如说：代码生成器、接口文档等等
    </description>
</project>
```

**设计要点**：
- `packaging=pom` 表明这是一个聚合模块，本身不产出 JAR
- `<parent>` 指向根 POM，继承统一的依赖版本管理
- `<modules>` 声明两个子模块

## yudao-module-infra-api

`api` 模块存放**跨模块共享的契约**：RPC 接口定义、DTO（数据传输对象）、枚举常量。其他业务模块只需依赖此模块即可调用 infra 的能力，无需依赖 `server` 模块。

### 核心 RPC 接口

**FileApi.java** -- 文件服务 RPC 接口

源码位置：`yudao-module-infra-api/src/main/java/cn/iocoder/yudao/module/infra/api/file/FileApi.java`

```java
@FeignClient(name = ApiConstants.NAME)
@Tag(name = "RPC 服务 - 文件")
public interface FileApi {

    String PREFIX = ApiConstants.PREFIX + "/file";

    /**
     * 保存文件，并返回文件的访问路径
     */
    @PostMapping(PREFIX + "/create")
    CommonResult<String> createFile(@Valid @RequestBody FileCreateReqDTO createReqDTO);

    /**
     * 生成文件预签名地址，用于读取
     */
    @GetMapping(PREFIX + "/presigned-url")
    CommonResult<String> presignGetUrl(@RequestParam("url") String url,
                                       @RequestParam(value = "expirationSeconds", required = false) Integer expirationSeconds);
}
```

**ConfigApi.java** -- 配置服务 RPC 接口

源码位置：`yudao-module-infra-api/src/main/java/cn/iocoder/yudao/module/infra/api/config/ConfigApi.java`

```java
@FeignClient(name = ApiConstants.NAME)
@Tag(name = "RPC 服务 - 配置")
public interface ConfigApi {

    String PREFIX = ApiConstants.PREFIX + "/config";

    /**
     * 获得配置值
     */
    @GetMapping(PREFIX + "/get-value-by-key")
    CommonResult<String> getConfigValueByKey(@RequestParam("key") String key);
}
```

### ApiConstants

源码位置：`yudao-module-infra-api/src/main/java/cn/iocoder/yudao/module/infra/enums/ApiConstants.java`

```java
public interface ApiConstants {
    String NAME = "infra-server";                    // 服务名，用于 Feign 路由
    String PREFIX = "/rpc-api/infra";                // RPC 前缀，网关路由用
}
```

**设计要点**：
- `NAME = "infra-server"` 对应 Nacos 注册的服务名，Feign 客户端通过此名称找到 infra 服务
- `PREFIX = "/rpc-api/infra"` 是所有 RPC 接口的统一前缀，与管理后台的 `/admin-api/infra` 前缀区分开

## 框架层与业务层的分工

在 infra 模块中，有一个容易混淆的点：为什么 `framework/` 目录放在 `server` 模块内部，而不是放在 `yudao-framework` 中？

```
┌─────────────────────────────────────────────────────────────────┐
│                    框架层 vs 业务层的分工                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  yudao-framework/ (全局框架层)                                   │
│  ├── yudao-spring-boot-starter-web                              │
│  │   └── apilog/                                                │
│  │       ├── @ApiAccessLog 注解                                 │
│  │       ├── ApiAccessLogFilter                                 │
│  │       └── ApiAccessLogInterceptor                            │
│  │   所有模块都需要的日志能力 ↓                                  │
│                                                                 │
│  yudao-module-infra-server/framework/ (模块内框架层)             │
│  ├── codegen/config/                                            │
│  │   └── CodegenProperties                                      │
│  └── file/core/                                                 │
│      └── client/ (FileClient 等)                                │
│   只有 infra 模块需要的框架能力 ↓                                │
│                                                                 │
│  yudao-module-infra-server/service/ + controller/ (业务层)      │
│  └── 具体的业务逻辑实现                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**原则**：
- **全局框架层**（`yudao-framework`）：所有模块都需要的能力，如 `@ApiAccessLog` 注解、`ApiAccessLogFilter`
- **模块内框架层**（`server/framework/`）：只有 infra 模块使用的框架能力，如 `FileClient` 接口、`CodegenProperties`
- **业务层**（`server/service/` + `controller/`）：具体的业务逻辑实现

```
┌─────────────────────────────────────────────────────────────────┐
│                    模块依赖关系图                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  yudao-module-product-server                                     │

│       ├──► yudao-module-system-api (用户、权限)                  │
│       │                                                         │

│                                                                 │
│  yudao-module-system-server                                      │
│       │                                                         │
│       └──► yudao-module-infra-api (文件、配置)                   │
│                                                                 │
│  yudao-module-infra-server                                       │
│       │                                                         │
│       └──► yudao-module-system-api (用户信息，用于日志记录)      │
│                                                                 │
│  yudao-framework (框架层)                                        │
│       │                                                         │
│       ├──► yudao-common (通用工具、CommonApi 接口)               │
│       │                                                         │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键依赖方向**：
- 业务模块依赖 `infra-api`，不依赖 `infra-server`
- `infra-server` 依赖 `system-api`（获取用户信息用于日志）
- 框架层依赖 `infra-api` 中的 DTO（用于日志异步写入的参数定义）

## 搭建步骤

如果你要从零搭建 infra 模块，按以下步骤进行：

### 步骤一：创建父 POM

在项目根目录下创建 `yudao-module-infra/pom.xml`，设置 `packaging=pom`，声明两个子模块。

### 步骤二：创建 api 子模块

创建 `yudao-module-infra-api/`，在其中定义：
1. `ApiConstants` -- 服务名和 RPC 前缀
2. `ErrorCodeConstants` -- 错误码常量
3. `FileApi`、`ConfigApi` -- RPC 接口
4. 相关的 DTO 和枚举

### 步骤三：创建 server 子模块

创建 `yudao-module-infra-server/`，按功能划分子包：
1. `controller/` -- 按 admin/app 分层，每个功能一个子包
2. `service/` -- 每个功能一个子包
3. `dal/` -- dataobject（DO）和 mysql（Mapper）
4. `framework/` -- 模块特有的框架扩展
5. `api/` -- RPC 接口实现
6. `job/` -- 定时任务

### 步骤四：在根 POM 中注册

在项目根 POM 的 `<modules>` 中添加 `yudao-module-infra`：

```xml
<module>yudao-module-infra</module>
```

## 本章小结

本章完成了 infra 模块的整体结构搭建。核心要点：

1. **双子模块结构**：`api`（契约）+ `server`（实现），与项目其他模块保持一致
2. **四层代码组织**：controller -> service -> dal -> framework，职责清晰
3. **框架层的位置**：全局能力放 `yudao-framework`，模块内能力放 `server/framework/`
4. **依赖方向**：业务模块只依赖 `api`，`server` 可以依赖其他模块的 `api`
