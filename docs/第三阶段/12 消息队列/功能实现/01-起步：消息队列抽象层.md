# 01-起步：消息队列抽象层

> 本章对应功能设计：[03-核心设计：Redis MQ 实现](../功能设计/03-核心设计：Redis%20MQ实现.md) 中的"消息基类体系"部分。

## 本章目标

从零搭建消息队列框架的核心抽象层。完成本章后，你将拥有：

1. 消息基类 `AbstractRedisMessage` —— 所有消息的根基
2. 两种消息子类 —— `AbstractRedisChannelMessage`（广播）和 `AbstractRedisStreamMessage`（可靠）
3. 拦截器接口 `RedisMessageInterceptor` —— 消息处理的扩展点

这些是整个 MQ 框架的地基，后续的发送、消费、自动配置都建立在此之上。

## 第二步：消息基类 AbstractRedisMessage

所有消息的根基，提供一个 `headers` Map 用于携带元数据。

```java
// 源码位置：yudao-framework/yudao-spring-boot-starter-mq/src/main/java/cn/iocoder/yudao/framework/mq/redis/core/message/AbstractRedisMessage.java
@Data
public abstract class AbstractRedisMessage {

    /**
     * 头
     */
    private Map<String, String> headers = new HashMap<>();

    public String getHeader(String key) {
        return headers.get(key);
    }

    public void addHeader(String key, String value) {
        headers.put(key, value);
    }

}
```

**为什么用 `Map<String, String>` 而不是强类型字段？**

headers 是扩展点——拦截器可以往里面塞任何东西（租户 ID、链路追踪 ID、时间戳等），用 Map 提供了最大的灵活性。如果用强类型字段，每新增一种元数据都要改基类，违反开闭原则。

**为什么 headers 默认初始化为 `new HashMap<>()`？**

避免空指针。消息创建后就可以直接调用 `addHeader()`，不需要先判断 headers 是否为 null。

## 第三步：广播消息基类 AbstractRedisChannelMessage

基于 Redis Pub/Sub 的广播消息。所有订阅同一个 channel 的消费者都会收到消息。

```java
// 源码位置：yudao-framework/yudao-spring-boot-starter-mq/src/main/java/cn/iocoder/yudao/framework/mq/redis/core/pubsub/AbstractRedisChannelMessage.java
public abstract class AbstractRedisChannelMessage extends AbstractRedisMessage {

    /**
     * 获得 Redis Channel，默认使用类名
     *
     * @return Channel
     */
    @JsonIgnore // 避免序列化。原因是，Redis 发布 Channel 消息的时候，已经会指定。
    public String getChannel() {
        return getClass().getSimpleName();
    }

}
```

**关键设计点：**

1. **`getChannel()` 默认返回类的简单名**。比如 `RedisWebSocketMessage` 类的 channel 就是 `"RedisWebSocketMessage"`。每个消息类自动拥有唯一的频道名，不需要额外配置。

2. **`@JsonIgnore` 注解**。channel 信息在发送时已经通过 `convertAndSend(channel, json)` 指定了，不需要再序列化到消息体中。如果不忽略，JSON 里会多出一个冗余的 `channel` 字段。

3. **子类可以覆盖 `getChannel()`**。如果需要自定义频道名（比如加前缀），覆盖这个方法即可。

## 第四步：可靠消息基类 AbstractRedisStreamMessage

基于 Redis Stream 的可靠消息。支持持久化、消费者组、ACK 机制。

```java
// 源码位置：yudao-framework/yudao-spring-boot-starter-mq/src/main/java/cn/iocoder/yudao/framework/mq/redis/core/stream/AbstractRedisStreamMessage.java
public abstract class AbstractRedisStreamMessage extends AbstractRedisMessage {

    /**
     * 获得 Redis Stream Key，默认使用类名
     *
     * @return Channel
     */
    @JsonIgnore // 避免序列化
    public String getStreamKey() {
        return getClass().getSimpleName();
    }

}
```

与 `AbstractRedisChannelMessage` 几乎一样，区别在于语义：Channel 对应 Pub/Sub 的频道，StreamKey 对应 Stream 的键。之所以分成两个类，是为了在编译期区分两种消息模式——`RedisMQTemplate` 的两个 `send` 方法通过参数类型来决定使用哪种发送方式。

## 第五步：拦截器接口 RedisMessageInterceptor

拦截器是消息处理的扩展点，可以在消息发送/消费前后插入横切逻辑。

```java
// 源码位置：yudao-framework/yudao-spring-boot-starter-mq/src/main/java/cn/iocoder/yudao/framework/mq/redis/core/interceptor/RedisMessageInterceptor.java
public interface RedisMessageInterceptor {

    default void sendMessageBefore(AbstractRedisMessage message) {
    }

    default void sendMessageAfter(AbstractRedisMessage message) {
    }

    default void consumeMessageBefore(AbstractRedisMessage message) {
    }

    default void consumeMessageAfter(AbstractRedisMessage message) {
    }

}
```

**为什么四个方法都是 `default` 空实现？**

这是**接口适配模式**——避免实现类被迫实现所有方法。大多数拦截器只需要关心一两个钩子点（比如租户拦截器只关心 `sendMessageBefore` 和 `consumeMessageBefore`），强制实现所有方法会产生大量空方法体。

**四个钩子点的典型用途：**

| 钩子点 | 执行时机 | 典型用途 |
|--------|---------|---------|
| `sendMessageBefore` | 发送前 | 写入租户 ID、记录发送日志 |
| `sendMessageAfter` | 发送后 | 清理上下文、统计发送耗时 |
| `consumeMessageBefore` | 消费前 | 读取租户 ID、设置上下文 |
| `consumeMessageAfter` | 消费后 | 清理上下文、记录消费日志 |

**拦截器的执行顺序遵循"洋葱模型"**——Before 正序执行，After 倒序执行。这保证了"谁先设置的，谁最后清理"。详细分析见功能设计文档 [05-核心设计：消息拦截与扩展](../功能设计/05-核心设计：消息拦截与扩展.md)。

## 第六步：Maven 依赖

`yudao-spring-boot-starter-mq` 模块的核心依赖：

```xml
<!-- pom.xml 关键依赖 -->
<dependencies>
    <!-- Spring Data Redis，提供 RedisTemplate、Stream 操作等 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-data-redis</artifactId>
    </dependency>
    <!-- Redisson，提供分布式锁（用于后台任务） -->
    <dependency>
        <groupId>org.redisson</groupId>
        <artifactId>redisson-spring-boot-starter</artifactId>
    </dependency>
    <!-- Jackson JSON 序列化 -->
    <dependency>
        <groupId>com.fasterxml.jackson.core</groupId>
        <artifactId>jackson-databind</artifactId>
    </dependency>
</dependencies>
```

注意：这里没有引入 RocketMQ、RabbitMQ、Kafka 的依赖。Redis MQ 是默认实现，其他 MQ 的依赖由业务模块按需引入。

## 验证：类继承体系

完成本章后，类继承体系如下：

```
AbstractRedisMessage                  ← 消息基类，持有 headers Map
    ├── AbstractRedisChannelMessage   ← Pub/Sub 消息，getChannel() 指定频道
    └── AbstractRedisStreamMessage    ← Stream 消息，getStreamKey() 指定流

RedisMessageInterceptor              ← 拦截器接口，四个 default 方法
```

下一章我们将实现 `RedisMQTemplate`——统一的发送 API，它根据消息类型自动选择 Pub/Sub 或 Stream 发送方式。

## 本章小结

| 类 | 职责 | 关键设计 |
|----|------|---------|
| `AbstractRedisMessage` | 消息基类 | `headers` Map 提供灵活的元数据扩展 |
| `AbstractRedisChannelMessage` | 广播消息 | `getChannel()` 默认用类名，`@JsonIgnore` 避免冗余 |
| `AbstractRedisStreamMessage` | 可靠消息 | `getStreamKey()` 默认用类名，与 Channel 消息编译期区分 |
| `RedisMessageInterceptor` | 拦截器接口 | 四个 `default` 方法，只需覆盖关心的钩子 |
