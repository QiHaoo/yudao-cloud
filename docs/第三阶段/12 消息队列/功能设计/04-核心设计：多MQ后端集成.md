# 04-核心设计：多 MQ 后端集成

## 集成策略总览

yudao-cloud 的 MQ 集成采用**分层策略**：

- **Redis MQ**：框架提供统一的 `RedisMQTemplate` 和消息基类，业务代码依赖框架抽象
- **Spring Event**：直接使用 Spring 原生 API，零封装
- **RocketMQ / RabbitMQ / Kafka**：直接使用 Spring Boot Starter 的原生 API，框架只做租户等横切关注点的集成

```
┌─────────────────────────────────────────────────────────────────┐
│                        业务代码                                  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│  │ Redis MQ │ │Spring Ev.│ │ RocketMQ │ │ RabbitMQ │  Kafka   │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘  ┌───┐  │
└───────┼────────────┼────────────┼────────────┼────────┼───┼────┘
        ↓            ↓            ↓            ↓        ↓   ↓
┌──────────────┐ ┌────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐
│RedisMQTemplate│ │Spring  │ │RocketMQ  │ │RabbitMQ  │ │Kafka   │
│(框架封装)    │ │原生API │ │原生API   │ │原生API   │ │原生API │
└──────────────┘ └────────┘ └──────────┘ └──────────┘ └────────┘
```

## Spring Event 集成

Spring Event 是最轻量的消息机制，完全基于 Spring Framework 内置能力，不需要任何额外封装。

### 使用模式

```java
// Producer：注入 ApplicationEventPublisher
@AllArgsConstructor
public class BpmProcessInstanceEventPublisher {
    private final ApplicationEventPublisher publisher;

    public void sendProcessInstanceResultEvent(BpmProcessInstanceStatusEvent event) {
        publisher.publishEvent(event);
    }
}

// Consumer：@EventListener 监听
@Component
public class CouponTakeByRegisterConsumer {
    @Resource
    private CouponService couponService;

    @EventListener
    @Async // 关键：异步执行，不阻塞 Producer
    public void onMessage(MemberUserCreateMessage message) {
        couponService.takeCouponByRegister(message.getUserId());
    }
}
```

源码位置：
- Producer: `yudao-module-bpm/.../event/BpmProcessInstanceEventPublisher.java`
- Consumer: `yudao-module-mall/.../consumer/coupon/CouponTakeByRegisterConsumer.java`

### @Async 的重要性

Spring Event 默认是**同步执行**的——Producer 调用 `publishEvent()` 后会等待所有 Listener 执行完毕才返回。这意味着：

```
不加 @Async：
  publishEvent() ──→ Listener1() ──→ Listener2() ──→ 返回
  Producer 线程阻塞，等待所有消费者完成

加 @Async：
  publishEvent() ──→ 立即返回
  另一个线程：Listener1()
  另一个线程：Listener2()
```

在 yudao-cloud 中，几乎所有 Spring Event 的 Consumer 都加了 `@Async`，实现真正的异步解耦。

### Spring Event 的局限

1. **进程内 only**：消息不出 JVM，跨服务通信需要其他方案
2. **无持久化**：进程崩溃，未处理的消息丢失
3. **无重试**：消费失败不会自动重试
4. **无消费组**：所有 Listener 都会收到消息，无法实现集群消费

这些局限决定了 Spring Event 只适合"进程内解耦"场景，不适合需要可靠投递的场景。

## RocketMQ 集成

RocketMQ 是阿里巴巴开源的分布式消息队列，适合大规模生产环境。

### 自动配置

RocketMQ 的集成通过 Spring Boot Starter 实现，yudao-cloud 没有额外的封装层。只需要：

1. 引入 `rocketmq-spring-boot-starter` 依赖
2. 配置 NameServer 地址
3. 使用 `RocketMQTemplate` 发送，`@RocketMQMessageListener` 接收

### 发送消息

```java
@Resource
private RocketMQTemplate rocketMQTemplate;

// 同步发送
rocketMQTemplate.syncSend("order-topic:tag", orderMessage);

// 异步发送
rocketMQTemplate.asyncSend("order-topic:tag", orderMessage, new SendCallback() {
    @Override
    public void onSuccess(SendResult sendResult) { /* 发送成功 */ }
    @Override
    public void onException(Throwable e) { /* 发送失败 */ }
});

// 单向发送（不关心结果）
rocketMQTemplate.sendOneWay("order-topic:tag", orderMessage);
```

### 消费消息

```java
@Component
@RocketMQMessageListener(topic = "order-topic", selectorExpression = "tag", consumerGroup = "order-consumer-group")
public class OrderConsumer implements RocketMQListener<OrderMessage> {
    @Override
    public void onMessage(OrderMessage message) {
        // 处理订单消息
    }
}
```

### RocketMQ 的特色能力

| 特性 | 说明 | 适用场景 |
|------|------|---------|
| Tag 过滤 | 一个 Topic 下用 Tag 区分消息类型 | 订单状态变更：tag=created/paid/shipped |
| 延迟消息 | 消息在指定时间后才投递 | 订单超时取消（30 分钟后检查） |
| 事务消息 | 本地事务与消息发送的原子性 | 下单：创建订单 + 发消息要么都成功要么都失败 |
| 顺序消息 | 同一个 Queue 内消息有序 | 订单状态变更必须按顺序处理 |
| 消息回溯 | 按时间点重新消费 | 数据修复、问题排查 |

## RabbitMQ 集成

RabbitMQ 是 AMQP 协议的经典实现，以灵活的路由能力著称。

### 自动配置

```java
// 源码位置：cn/iocoder/yudao/framework/mq/rabbitmq/config/YudaoRabbitMQAutoConfiguration.java
@AutoConfiguration
@ConditionalOnClass(name = "org.springframework.amqp.rabbit.core.RabbitTemplate")
public class YudaoRabbitMQAutoConfiguration {

    @Bean
    public MessageConverter createMessageConverter() {
        return new Jackson2JsonMessageConverter(); // 使用 JSON 序列化
    }
}
```

这个配置类做了一件关键的事——将消息序列化方式从 JDK 默认的 `SimpleMessageConverter`（序列化为 Java 对象字节流）改为 `Jackson2JsonMessageConverter`（序列化为 JSON）。为什么？因为 JDK 序列化有安全风险，而且跨语言不兼容。

### Exchange/Queue/Binding 声明

RabbitMQ 的消息路由依赖 Exchange 和 Binding 的配置：

```java
@Configuration
public class RabbitMQConfig {
    // 声明 Exchange
    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order.exchange");
    }

    // 声明 Queue
    @Bean
    public Queue orderQueue() {
        return QueueBuilder.durable("order.queue").build();
    }

    // 声明 Binding（绑定 Exchange 和 Queue）
    @Bean
    public Binding orderBinding() {
        return BindingBuilder.bind(orderQueue()).to(orderExchange()).with("order.created");
    }
}
```

### 发送消息

```java
@Resource
private RabbitTemplate rabbitTemplate;

// 发送到指定 Exchange 和 Routing Key
rabbitTemplate.convertAndSend("order.exchange", "order.created", orderMessage);
```

### 消费消息

```java
@Component
@RabbitListener(queues = "order.queue")
public class OrderConsumer {
    @RabbitHandler
    public void onMessage(OrderMessage message) {
        // 处理订单消息
    }
}
```

### RabbitMQ 的特色能力

| 特性 | 说明 | 适用场景 |
|------|------|---------|
| 灵活路由 | Direct/Topic/Fanout/Headers 四种 Exchange 类型 | 复杂的消息分发规则 |
| 消息确认 | 手动 ACK + 消息重回队列 | 需要精确控制消费确认 |
| 死信队列 | 消费失败/过期的消息进入 DLQ | 消息重试、问题排查 |
| 优先级队列 | 高优先级消息先消费 | VIP 订单优先处理 |
| 延迟队列 | 通过 TTL + 死信实现 | 订单超时取消 |

## Kafka 集成

Kafka 是分布式流处理平台，以超高的吞吐量著称。

### 集成方式

Kafka 的集成通过 Spring Boot 的 `spring-kafka` Starter 实现：

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
    consumer:
      group-id: my-consumer-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
```

### 发送消息

```java
@Resource
private KafkaTemplate<String, Object> kafkaTemplate;

// 同步发送
kafkaTemplate.send("order-topic", "order-123", orderMessage).get();

// 异步发送
kafkaTemplate.send("order-topic", "order-123", orderMessage)
    .addCallback(
        result -> { /* 成功 */ },
        ex -> { /* 失败 */ }
    );
```

### 消费消息

```java
@Component
public class OrderConsumer {
    @KafkaListener(topics = "order-topic", groupId = "order-consumer-group")
    public void onMessage(ConsumerRecord<String, OrderMessage> record) {
        OrderMessage message = record.value();
        // 处理订单消息
    }
}
```

### Kafka 的特色能力

| 特性 | 说明 | 适用场景 |
|------|------|---------|
| 超高吞吐 | 单机百万级 TPS | 日志收集、大数据管道 |
| 消息回溯 | 按 offset 或时间点重新消费 | 数据重放、问题排查 |
| 分区有序 | 同一个分区内消息有序 | 需要局部有序的场景 |
| 流处理 | Kafka Streams / ksqlDB | 实时数据分析 |
| 持久化 | 消息持久化到磁盘，支持长期保留 | 数据仓库、审计日志 |

## 各 MQ 的自动配置对比

| MQ | 配置类 | 关键 Bean | 条件装配 |
|----|--------|----------|---------|
| Redis (Producer) | `YudaoRedisMQProducerAutoConfiguration` | `RedisMQTemplate` | 无（默认启用） |
| Redis (Consumer) | `YudaoRedisMQConsumerAutoConfiguration` | `RedisMessageListenerContainer`、`StreamMessageListenerContainer` | `@ConditionalOnBean(AbstractRedisXxxMessageListener.class)` |
| RabbitMQ | `YudaoRabbitMQAutoConfiguration` | `Jackson2JsonMessageConverter` | `@ConditionalOnClass(RabbitTemplate.class)` |
| RocketMQ | 无额外配置 | 直接用 Starter | 依赖 `rocketmq-spring-boot-starter` |
| Kafka | 无额外配置 | 直接用 Starter | 依赖 `spring-kafka` |

## 与租户模块的集成

多租户场景下，消息需要携带租户信息，消费者需要恢复租户上下文。yudao-cloud 为每种 MQ 都实现了租户集成。

### Redis MQ：TenantRedisMessageInterceptor

```java
// 源码位置：cn/iocoder/yudao/framework/tenant/core/mq/redis/TenantRedisMessageInterceptor.java
public class TenantRedisMessageInterceptor implements RedisMessageInterceptor {

    @Override
    public void sendMessageBefore(AbstractRedisMessage message) {
        Long tenantId = TenantContextHolder.getTenantId();
        if (tenantId != null) {
            message.addHeader(HEADER_TENANT_ID, tenantId.toString());
        }
    }

    @Override
    public void consumeMessageBefore(AbstractRedisMessage message) {
        String tenantIdStr = message.getHeader(HEADER_TENANT_ID);
        if (StrUtil.isNotEmpty(tenantIdStr)) {
            TenantContextHolder.setTenantId(Long.valueOf(tenantIdStr));
        }
    }

    @Override
    public void consumeMessageAfter(AbstractRedisMessage message) {
        TenantContextHolder.clear(); // 清理，防止污染后续请求
    }
}
```

### RocketMQ：Hook 机制

RocketMQ 通过 `SendMessageHook` 和 `ConsumeMessageHook` 实现租户传递：

```java
// 发送时：SendMessageHook
public class TenantRocketMQSendMessageHook implements SendMessageHook {
    @Override
    public void sendMessageBefore(SendMessageContext context) {
        Long tenantId = TenantContextHolder.getTenantId();
        if (tenantId != null) {
            context.getMessage().putUserProperty(HEADER_TENANT_ID, tenantId.toString());
        }
    }
}

// 消费时：ConsumeMessageHook
public class TenantRocketMQConsumeMessageHook implements ConsumeMessageHook {
    @Override
    public void consumeMessageBefore(ConsumeMessageContext context) {
        List<MessageExt> messages = context.getMsgList();
        Assert.isTrue(messages.size() == 1, "消息条数({})不正确", messages.size());
        String tenantId = messages.get(0).getUserProperty(HEADER_TENANT_ID);
        if (StrUtil.isNotEmpty(tenantId)) {
            TenantContextHolder.setTenantId(Long.parseLong(tenantId));
        }
    }

    @Override
    public void consumeMessageAfter(ConsumeMessageContext context) {
        TenantContextHolder.clear();
    }
}
```

源码位置：`yudao-framework/yudao-spring-boot-starter-biz-tenant/.../mq/rocketmq/`

注意：`Assert.isTrue(messages.size() == 1)` 这个校验——租户信息是设置在线程上下文中的，如果一批消息包含多个不同租户的消息，设置就会出错。所以要求批量消费时只能有一条消息。

### RabbitMQ：MessagePostProcessor

```java
// 源码位置：cn/iocoder/yudao/framework/tenant/core/mq/rabbitmq/TenantRabbitMQMessagePostProcessor.java
public class TenantRabbitMQMessagePostProcessor implements MessagePostProcessor {
    @Override
    public Message postProcessMessage(Message message) throws AmqpException {
        Long tenantId = TenantContextHolder.getTenantId();
        if (tenantId != null) {
            message.getMessageProperties().getHeaders().put(HEADER_TENANT_ID, tenantId);
        }
        return message;
    }
}
```

通过 `BeanPostProcessor` 自动注册到所有 `RabbitTemplate`：

```java
// 源码位置：cn/iocoder/yudao/framework/tenant/core/mq/rabbitmq/TenantRabbitMQInitializer.java
public class TenantRabbitMQInitializer implements BeanPostProcessor {
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        if (bean instanceof RabbitTemplate) {
            ((RabbitTemplate) bean).addBeforePublishPostProcessors(new TenantRabbitMQMessagePostProcessor());
        }
        return bean;
    }
}
```

### Kafka：ProducerInterceptor

```java
// 源码位置：cn/iocoder/yudao/framework/tenant/core/mq/kafka/TenantKafkaProducerInterceptor.java
public class TenantKafkaProducerInterceptor implements ProducerInterceptor<Object, Object> {
    @Override
    public ProducerRecord<Object, Object> onSend(ProducerRecord<Object, Object> record) {
        Long tenantId = TenantContextHolder.getTenantId();
        if (tenantId != null) {
            Headers headers = (Headers) ReflectUtil.getFieldValue(record, "headers");
            headers.add(HEADER_TENANT_ID, tenantId.toString().getBytes());
        }
        return record;
    }
}
```

注意：Kafka 的 `ProducerRecord.headers` 字段是 `private` 的且没有 getter，只能通过反射获取。这是一个不太优雅但必要的 hack。

通过 `EnvironmentPostProcessor` 自动注册：

```java
// 源码位置：cn/iocoder/yudao/framework/tenant/core/mq/kafka/TenantKafkaEnvironmentPostProcessor.java
public class TenantKafkaEnvironmentPostProcessor implements EnvironmentPostProcessor {
    @Override
    public void postProcessEnvironment(ConfigurableEnvironment environment, SpringApplication application) {
        String value = environment.getProperty("spring.kafka.producer.properties.interceptor.classes");
        if (StrUtil.isEmpty(value)) {
            value = TenantKafkaProducerInterceptor.class.getName();
        } else {
            value += "," + TenantKafkaProducerInterceptor.class.getName();
        }
        environment.getSystemProperties().put("spring.kafka.producer.properties.interceptor.classes", value);
    }
}
```

## 租户集成方式对比

| MQ | 发送拦截 | 消费拦截 | 注册方式 |
|----|---------|---------|---------|
| Redis | `RedisMessageInterceptor` | `RedisMessageInterceptor` | Spring Bean 自动注入 |
| RocketMQ | `SendMessageHook` | `ConsumeMessageHook` | `BeanPostProcessor` 注册 |
| RabbitMQ | `MessagePostProcessor` | `@Header` 注解 | `BeanPostProcessor` 注册 |
| Kafka | `ProducerInterceptor` | `@Header` 注解 | `EnvironmentPostProcessor` 注入配置 |

所有 MQ 的租户集成都遵循同一个模式：

```
发送前：TenantContextHolder.getTenantId() → 写入消息 Header
消费前：读取消息 Header → TenantContextHolder.setTenantId()
消费后：TenantContextHolder.clear()
```

## 思考题

1. **为什么 yudao-cloud 对 Redis MQ 做了封装（RedisMQTemplate），但对 RocketMQ/RabbitMQ/Kafka 没有？** 提示：考虑项目的主要部署模式和依赖关系。

2. **RabbitMQ 的 `YudaoRabbitMQAutoConfiguration` 为什么要用 `Jackson2JsonMessageConverter` 替换默认的消息转换器？** 提示：考虑 JDK 序列化的安全风险和跨语言兼容性。

3. **在多租户场景下，如果一条消息需要发送给多个租户，当前的设计能支持吗？需要怎么改？**
