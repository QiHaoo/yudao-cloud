# 05-核心设计：消息拦截与扩展

## 拦截器模式：AOP 的消息版本

在 Web 开发中，我们用 `HandlerInterceptor` 在请求处理前后插入逻辑（鉴权、日志、CORS）。消息队列的拦截器做的是同样的事——在消息发送/消费前后插入横切逻辑。

```
Web 请求拦截：
  preHandle() → Controller.handle() → postHandle()

消息发送拦截：
  sendMessageBefore() → 实际发送 → sendMessageAfter()

消息消费拦截：
  consumeMessageBefore() → 实际消费 → consumeMessageAfter()
```

### 洋葱模型

yudao-cloud 的拦截器执行顺序遵循"洋葱模型"：

```
发送流程：
  拦截器A.sendMessageBefore()   ← 最先执行
    拦截器B.sendMessageBefore()
      拦截器C.sendMessageBefore()
        ↓
      实际发送消息
        ↓
      拦截器C.sendMessageAfter()  ← 最后执行（倒序）
    拦截器B.sendMessageAfter()
  拦截器A.sendMessageAfter()

消费流程：同理
```

Before 正序、After 倒序。这个设计保证了"谁先设置的，谁最后清理"——比如拦截器 A 设置了租户上下文，那么它在 After 中最后清理，不会影响其他拦截器的执行。

源码实现：

```java
// 源码位置：RedisMQTemplate.java
private void sendMessageBefore(AbstractRedisMessage message) {
    interceptors.forEach(interceptor -> interceptor.sendMessageBefore(message));
}

private void sendMessageAfter(AbstractRedisMessage message) {
    for (int i = interceptors.size() - 1; i >= 0; i--) {
        interceptors.get(i).sendMessageAfter(message);
    }
}
```

### try-finally 保障清理

发送和消费都用 `try-finally` 包裹，确保 After 方法一定执行：

```java
public <T extends AbstractRedisChannelMessage> void send(T message) {
    try {
        sendMessageBefore(message);
        redisTemplate.convertAndSend(message.getChannel(), JsonUtils.toJsonString(message));
    } finally {
        sendMessageAfter(message); // 即使发送失败也会执行
    }
}
```

这在多租户场景下尤为重要——如果 `sendMessageBefore` 设置了某些上下文，但发送失败了，`sendMessageAfter` 必须清理这些上下文，否则会"泄漏"到后续请求。

## RedisMessageInterceptor 接口

```java
// 源码位置：cn/iocoder/yudao/framework/mq/redis/core/interceptor/RedisMessageInterceptor.java
public interface RedisMessageInterceptor {

    default void sendMessageBefore(AbstractRedisMessage message) {}
    default void sendMessageAfter(AbstractRedisMessage message) {}
    default void consumeMessageBefore(AbstractRedisMessage message) {}
    default void consumeMessageAfter(AbstractRedisMessage message) {}
}
```

四个方法都是 `default` 空实现，拦截器只需覆盖关心的方法。这是**接口适配模式**——避免实现类被迫实现所有方法。

### 自动注册

Producer 配置类会自动收集所有 `RedisMessageInterceptor` Bean：

```java
// 源码位置：YudaoRedisMQProducerAutoConfiguration.java
@Bean
public RedisMQTemplate redisMQTemplate(StringRedisTemplate redisTemplate,
                                       List<RedisMessageInterceptor> interceptors) {
    RedisMQTemplate redisMQTemplate = new RedisMQTemplate(redisTemplate);
    interceptors.forEach(redisMQTemplate::addInterceptor);
    return redisMQTemplate;
}
```

要新增一个拦截器，只需要：
1. 实现 `RedisMessageInterceptor` 接口
2. 加上 `@Component` 注解
3. 框架自动注册，无需修改任何配置

## 租户消息拦截：完整实现

这是 yudao-cloud 中最重要的拦截器实现——确保消息在跨进程传递时不会丢失租户信息。

### 问题场景

```
用户请求（tenant_id=1）
    ↓
Controller → Service → 发送消息
    ↓
消息进入 Redis（没有租户信息！）
    ↓
消费者收到消息（不知道是哪个租户的）
    ↓
执行 SQL（没有 tenant_id 条件 → 查询到所有租户的数据 → 数据泄露！）
```

### 解决方案

```java
// 源码位置：cn/iocoder/yudao/framework/tenant/core/mq/redis/TenantRedisMessageInterceptor.java
public class TenantRedisMessageInterceptor implements RedisMessageInterceptor {

    @Override
    public void sendMessageBefore(AbstractRedisMessage message) {
        // 从当前线程上下文获取租户 ID，写入消息 header
        Long tenantId = TenantContextHolder.getTenantId();
        if (tenantId != null) {
            message.addHeader(HEADER_TENANT_ID, tenantId.toString());
        }
    }

    @Override
    public void consumeMessageBefore(AbstractRedisMessage message) {
        // 从消息 header 读取租户 ID，设置到当前线程上下文
        String tenantIdStr = message.getHeader(HEADER_TENANT_ID);
        if (StrUtil.isNotEmpty(tenantIdStr)) {
            TenantContextHolder.setTenantId(Long.valueOf(tenantIdStr));
        }
    }

    @Override
    public void consumeMessageAfter(AbstractRedisMessage message) {
        // 清理上下文，防止污染后续请求
        TenantContextHolder.clear();
    }
}
```

### 完整流程

```
Producer 线程                                    Consumer 线程
┌─────────────────────┐                          ┌─────────────────────┐
│ TenantContextHolder │                          │ TenantContextHolder │
│   tenantId = 1      │                          │   (空)              │
└──────────┬──────────┘                          └──────────┬──────────┘
           │                                                │
           ↓                                                │
sendMessageBefore()                                         │
  message.addHeader("tenant-id", "1")                       │
           │                                                │
           ↓                                                │
  实际发送消息 ──────→ Redis ──────→ Consumer 收到           │
                                         │                  │
                                         ↓                  │
                               consumeMessageBefore()       │
                                 tenantId = "1"             │
                                 TenantContextHolder        │
                                   .setTenantId(1) ────────→│
                                         │           tenantId = 1
                                         ↓                  │
                               实际消费消息                  │
                               （SQL 自动带 tenant_id=1）   │
                                         │                  │
                                         ↓                  │
                               consumeMessageAfter()        │
                                 TenantContextHolder        │
                                   .clear() ───────────────→│
                                                   tenantId = (空)
```

### 为什么 consumeMessageAfter 要 clear？

消费逻辑的入口可以看作一个"新的请求"——它不属于任何用户会话。如果不清理，租户 ID 会残留在 `ThreadLocal` 中，影响后续在同一线程上执行的其他逻辑（比如线程池复用线程时）。

注意源码注释："Consumer 是一个逻辑的入口，所以不考虑原本上下文就存在租户编号的情况"。这意味着消费者不应该假设自己的线程已经有租户上下文。

## 消息序列化：JSON 统一方案

所有消息都通过 `JsonUtils.toJsonString()` 序列化为 JSON 字符串，再存入 Redis。

### 为什么选 JSON？

| 序列化方式 | 优点 | 缺点 |
|-----------|------|------|
| JDK 序列化 | Java 原生 | 不跨语言、有安全风险、不可读 |
| JSON | 可读、跨语言、生态丰富 | 体积略大、性能略低 |
| Protobuf | 体积小、性能高 | 不可读、需要定义 schema |
| Kryo | 体积小、性能高 | 不跨语言 |

yudao-cloud 选择 JSON 的核心原因是**可读性**——在 Redis 中直接查看消息内容，方便调试。对于大多数业务场景，JSON 的性能完全够用。

### 消息结构

消息序列化后的 JSON 结构：

```json
{
    "headers": {
        "tenant-id": "1"
    },
    "sessionId": "abc123",
    "userType": 1,
    "userId": 100,
    "messageType": "text",
    "messageContent": "Hello"
}
```

`headers` 是基类 `AbstractRedisMessage` 的字段，业务字段在同级。

## 消息幂等性

### 问题

在分布式环境中，消息可能被重复投递：

1. **网络抖动**：Producer 发送消息后，ACK 丢失，Producer 认为发送失败并重试
2. **消费者崩溃**：消息处理了一半，消费者崩溃，Pending 消息被重发
3. **RedisPendingMessageResendJob**：超时 5 分钟的消息会被自动重发

### 解决方案

消息幂等性**不在框架层面实现**，而是留给业务层处理。原因：

1. 不同业务的幂等判断逻辑不同（订单号？优惠券 ID？）
2. 框架无法知道业务是否已经处理过某条消息
3. 强行实现会导致性能下降（每条消息都要查数据库判断是否重复）

### 业务层实现模式

```java
// 消费者代码示例
@EventListener
@Async
public void onMessage(OrderPaidMessage message) {
    // 幂等检查：查询订单状态，如果已经是"已支付"，跳过
    OrderDO order = orderMapper.selectById(message.getOrderId());
    if (order.getStatus() == OrderStatusEnum.PAID.getStatus()) {
        log.warn("[onMessage][订单({})已处理，跳过]", message.getOrderId());
        return;
    }
    // 处理业务逻辑
    orderService.handlePayment(message);
}
```

### 常见的幂等方案

| 方案 | 实现方式 | 适用场景 |
|------|---------|---------|
| 状态检查 | 查询业务状态，判断是否已处理 | 订单状态变更 |
| 唯一约束 | 数据库唯一索引防止重复插入 | 优惠券领取记录 |
| Redis 标记 | SETNX 标记消息已处理 | 高并发场景 |
| 乐观锁 | UPDATE ... WHERE version = ? | 库存扣减 |

## 事务消息：与 Spring 事务的集成

### 问题场景

```java
@Transactional
public void createOrder(OrderCreateReqVO reqVO) {
    // 1. 创建订单（数据库操作）
    orderMapper.insert(order);
    // 2. 发送消息
    redisMQTemplate.send(new OrderCreatedMessage(order.getId()));
    // 3. 如果后续代码抛异常，事务回滚，但消息已经发出去了！
}
```

消息发送成功但事务回滚，导致"幽灵消息"——消费者收到消息去查订单，发现订单不存在。

### 解决方案

yudao-cloud 当前没有内置事务消息支持（RocketMQ 原生支持事务消息，Redis 不支持）。业务层可以通过以下方式处理：

**方案一：事务后置发送**

```java
// 利用 Spring 的 TransactionSynchronization
@Transactional
public void createOrder(OrderCreateReqVO reqVO) {
    orderMapper.insert(order);

    TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
        @Override
        public void afterCommit() {
            // 事务提交后再发送消息
            redisMQTemplate.send(new OrderCreatedMessage(order.getId()));
        }
    });
}
```

**方案二：本地消息表**

```java
@Transactional
public void createOrder(OrderCreateReqVO reqVO) {
    orderMapper.insert(order);
    // 1. 将消息写入本地消息表（同一个事务）
    messageMapper.insert(new MessageDO("order.created", order.getId()));
}

// 后台任务定期扫描消息表，发送未投递的消息
```

**方案三：RocketMQ 事务消息**

如果使用 RocketMQ，可以利用其原生的事务消息机制：

```java
// RocketMQ 事务消息：本地事务和消息发送是原子的
rocketMQTemplate.sendMessageInTransaction("order-topic", message, new RocketMQLocalTransactionListener() {
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地事务
        try {
            createOrder(...);
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        // 回查：检查本地事务是否成功
        OrderDO order = orderMapper.selectById(orderId);
        return order != null ? COMMIT : ROLLBACK;
    }
});
```

## 自定义拦截器示例：消息日志

以下是一个实用的自定义拦截器示例——记录消息的发送和消费日志：

```java
@Component
@Slf4j
public class MessageLoggingInterceptor implements RedisMessageInterceptor {

    @Override
    public void sendMessageBefore(AbstractRedisMessage message) {
        log.info("[sendMessage][消息类型={}][Headers={}]",
                message.getClass().getSimpleName(), message.getHeaders());
    }

    @Override
    public void consumeMessageBefore(AbstractRedisMessage message) {
        log.info("[consumeMessage][消息类型={}][Headers={}]",
                message.getClass().getSimpleName(), message.getHeaders());
    }

    @Override
    public void consumeMessageAfter(AbstractRedisMessage message) {
        log.info("[consumeMessage][完成][消息类型={}]", message.getClass().getSimpleName());
    }
}
```

加上 `@Component` 后，框架自动注册，无需修改任何配置。

## 扩展点总结

yudao-cloud 的消息队列提供了以下扩展点：

| 扩展点 | 机制 | 典型用途 |
|--------|------|---------|
| 消息定义 | 继承 AbstractRedisMessage | 定义新的消息类型 |
| 消费者 | 继承 AbstractRedisXxxMessageListener | 处理特定消息 |
| 拦截器 | 实现 RedisMessageInterceptor | 租户传递、日志、监控 |
| 消息头 | headers Map | 携带元数据（租户 ID、traceId） |
| Channel/StreamKey | 覆盖 getChannel()/getStreamKey() | 自定义消息路由 |

## 思考题

1. **如果要实现消息的链路追踪（在消息中携带 traceId），应该如何设计拦截器？** 提示：考虑与 Sleuth/Zipkin 的集成。

2. **当前的拦截器是全局的（所有消息都经过），如果需要只对特定消息类型生效的拦截器，应该怎么改？** 提示：考虑在拦截器中加 `instanceof` 判断，或者设计消息类型匹配器。

3. **`consumeMessageAfter` 中清理上下文用的是 `TenantContextHolder.clear()`，这会清掉所有租户信息。如果未来有其他拦截器也在 consumeMessageBefore 中设置了其他上下文，这个设计会有问题吗？**
