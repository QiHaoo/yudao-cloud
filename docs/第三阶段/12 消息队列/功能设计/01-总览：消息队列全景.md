# 01-总览：消息队列全景

## 场景引入：电商下单后的困境

想象一个典型的电商下单场景。用户点击"提交订单"按钮后，系统需要完成一系列操作：

```
用户下单
  ├─ 1. 创建订单记录（核心业务）
  ├─ 2. 扣减库存（核心业务）
  ├─ 3. 发送短信通知（辅助业务）
  ├─ 4. 更新用户积分（辅助业务）
  └─ 5. 通知物流系统（辅助业务）
```

如果把这些操作全部写在一个方法里，会产生三个严重问题：

**问题一：强耦合**。订单服务直接依赖短信服务、积分服务、物流服务。任何一个下游服务出问题，都会导致下单失败。短信网关挂了，用户就下不了单——这显然不合理。

**问题二：同步阻塞**。发送短信需要调用第三方 API，更新积分需要跨服务 RPC，这些操作加起来可能耗时 500ms 以上。用户等得不耐烦，体验极差。

**问题三：流量尖峰**。大促期间，每秒可能有上万个订单。如果同步处理所有操作，系统负载会瞬间飙升，数据库连接池被打满，最终导致整个系统崩溃。

消息队列正是为了解决这三个问题而生。

## 消息队列的核心价值

### 解耦：各做各的事

引入消息队列后，订单服务只需要做一件事——把"下单成功"这个事件发布出去。至于谁关心这个事件、怎么处理，订单服务完全不关心。

```
                    ┌─────────────┐
                    │  短信服务    │ ← 订阅"下单成功"
                    └─────────────┘
                           ↑
┌──────────┐    ┌─────────────────────┐    ┌─────────────┐
│ 订单服务 │───→│     消息队列        │───→│  积分服务    │ ← 订阅"下单成功"
└──────────┘    └─────────────────────┘    └─────────────┘
                           ↓
                    ┌─────────────┐
                    │  物流服务    │ ← 订阅"下单成功"
                    └─────────────┘
```

订单服务不再依赖任何下游服务。新增一个"推荐系统"要监听下单事件？只需要加一个消费者，订单服务一行代码都不用改。

### 异步：不用傻等

消息发布是微秒级操作。订单服务把消息扔进队列就立即返回，用户瞬间看到"下单成功"。短信、积分、物流这些耗时操作，由消费者在后台慢慢处理。

```
同步模式：用户等待 = 创建订单 + 扣库存 + 发短信 + 更新积分 + 通知物流
异步模式：用户等待 = 创建订单 + 扣库存 + 发布消息（微秒级）
```

### 削峰：保护下游

大促期间，消息队列充当缓冲区。上游的请求先堆积在队列里，下游服务按照自己的处理能力匀速消费。就像水坝调节水流一样，避免下游被洪水冲垮。

```
大促瞬间：10000 条消息/秒
         ↓ 进入消息队列（缓冲）
匀速消费：500 条消息/秒（下游处理能力）
```

## yudao-cloud 的消息队列架构

yudao-cloud 采用**可插拔**的消息队列设计。项目没有绑定某一种 MQ，而是提供了统一的抽象层，支持多种 MQ 后端：

| MQ 后端 | 适用场景 | 特点 |
|---------|---------|------|
| Spring Event | 进程内解耦 | 零依赖，最轻量 |
| Redis Pub/Sub | 进程间广播 | 无持久化，实时性好 |
| Redis Stream | 进程间集群消费 | 持久化，消费者组，ACK 机制 |
| RocketMQ | 大规模生产环境 | 高可靠，事务消息，延迟消息 |
| RabbitMQ | 企业级应用 | 灵活路由，多种协议 |
| Kafka | 大数据场景 | 超高吞吐，日志收集 |

## 两种消息模式

yudao-cloud 中的消息传递分为两个层次：

### 模式一：Spring Event（进程内）

适用于**同一个应用内部**的事件解耦。比如用户注册后发送优惠券，这两个逻辑在同一个服务中，用 Spring Event 就够了。

```
┌─────────────────────────────────────────────┐
│              同一个 JVM 进程                 │
│                                             │
│  用户注册 ──publishEvent──→ 优惠券发放      │
│  (Producer)                  (Consumer)     │
│                                             │
│  特点：                                     │
│  - 零外部依赖                               │
│  - 默认同步执行（加 @Async 可异步）         │
│  - 无持久化，进程挂了消息就丢了             │
└─────────────────────────────────────────────┘
```

**典型使用**：

```java
// Producer：发布事件
publisher.publishEvent(new MailSendMessage().setLogId(1L).setToMails(...));

// Consumer：监听事件
@EventListener
@Async // 加上异步，不阻塞 Producer 线程
public void onMessage(MailSendMessage message) {
    mailSendService.doSendMail(message);
}
```

源码位置：`yudao-module-system/yudao-module-system-server/.../mq/consumer/mail/MailSendConsumer.java`

### 模式二：Redis MQ（进程间）

适用于**跨服务、跨进程**的消息传递。比如订单服务（一个微服务）发布消息，物流服务（另一个微服务）消费消息。

```
┌──────────────┐          ┌──────────────┐          ┌──────────────┐
│  订单服务    │          │    Redis     │          │  物流服务    │
│  (Producer)  │──send──→│   Server     │──recv──→│  (Consumer)  │
└──────────────┘          └──────────────┘          └──────────────┘

Redis 提供两种消息通道：
┌─────────────────────────────────────────────────────────────────┐
│ Channel (Pub/Sub)                                               │
│ - 广播模式：所有订阅者都收到消息                                 │
│ - 不持久化：消息发完就没了                                       │
│ - 适用场景：WebSocket 消息广播、实时通知                         │
├─────────────────────────────────────────────────────────────────┤
│ Stream                                                          │
│ - 集群模式：同一个消费组内只有一个消费者收到消息                 │
│ - 持久化：消息保存在 Redis 中，支持回溯                         │
│ - ACK 机制：消费者确认后才标记为已消费                           │
│ - 适用场景：订单处理、库存扣减等需要可靠投递的场景               │
└─────────────────────────────────────────────────────────────────┘
```

## 核心组件一览

| 组件 | 职责 | 关键类 |
|------|------|--------|
| 消息基类 | 定义消息结构 | `AbstractRedisMessage`、`AbstractRedisChannelMessage`、`AbstractRedisStreamMessage` |
| 发送模板 | 统一发送 API | `RedisMQTemplate` |
| 消费者基类 | 统一消费逻辑 | `AbstractRedisChannelMessageListener`、`AbstractRedisStreamMessageListener` |
| 拦截器 | 消息处理扩展点 | `RedisMessageInterceptor` |
| 后台任务 | 保障消息可靠性 | `RedisPendingMessageResendJob`、`RedisStreamMessageCleanupJob` |
| 自动配置 | Spring Boot 集成 | `YudaoRedisMQProducerAutoConfiguration`、`YudaoRedisMQConsumerAutoConfiguration` |
| 租户集成 | 多租户消息隔离 | `TenantRedisMessageInterceptor`、`TenantRocketMQSendMessageHook` 等 |

## 本章小结

消息队列是微服务架构中的"神经系统"，负责在各个服务之间传递事件。yudao-cloud 的设计哲学是**渐进式选择**：

1. 同进程内用 Spring Event——最简单，零依赖
2. 跨进程但规模不大用 Redis Stream——复用已有 Redis，无需额外部署
3. 大规模生产环境用 RocketMQ/RabbitMQ——高可靠、功能丰富
