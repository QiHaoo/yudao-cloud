# 02-问题域与设计目标

## 为什么需要统一的 MQ 抽象

假设你正在开发一个电商系统，一开始用 Redis 做消息队列，代码里到处写着 `redisTemplate.convertAndSend(...)`。后来业务增长，Redis Pub/Sub 的不持久化成了瓶颈，老板说"换 RocketMQ 吧"。

这时候你面临一个噩梦：**所有业务代码都要改**。每个 Producer 要从 `redisTemplate.convertAndSend` 改成 `rocketMQTemplate.syncSend`，每个 Consumer 要从 `AbstractRedisChannelMessageListener` 改成 `@RocketMQMessageListener`。几十个消息，几百行代码，改完还要全量回归测试。

这就是**厂商锁定**问题——业务代码与具体 MQ 实现强耦合。

### 解决方案：抽象层

yudao-cloud 的做法是在业务代码和 MQ 实现之间加一层抽象：

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────┐
│  业务代码    │────→│   统一抽象层     │────→│  具体 MQ     │
│              │     │ (RedisMQTemplate) │     │  实现        │
└──────────────┘     └──────────────────┘     └──────────────┘
                            ↑
                     业务代码只依赖这一层
                     换 MQ 只改配置，不改业务代码
```

但 yudao-cloud 没有做得太极端——它没有为所有 MQ 提供统一的 `MQTemplate` 接口。为什么？因为不同 MQ 的特性差异太大（Redis Stream 的消费者组 vs RocketMQ 的 Topic/Tag vs RabbitMQ 的 Exchange/Queue），强行统一反而会丢失各自的特色能力。

实际策略是：**Redis MQ 提供统一模板，其他 MQ 直接使用原生 API**。这体现了务实的设计哲学——不是所有东西都需要抽象，当抽象的成本大于收益时，直接用原生 API 更好。

## Spring Event vs 消息队列：进程内 vs 跨进程

这是最容易混淆的选择。很多开发者一提到"解耦"就想到消息队列，但其实在同一个应用内部，Spring Event 往往是更好的选择。

### 对比分析

| 维度 | Spring Event | Redis MQ | 专业 MQ（RocketMQ 等） |
|------|-------------|----------|----------------------|
| 适用范围 | 同一 JVM 进程 | 跨进程（Redis 网络） | 跨进程（独立集群） |
| 外部依赖 | 无 | Redis | 专用 MQ 集群 |
| 持久化 | 无 | Stream 支持 | 完整支持 |
| 消费模式 | 同步/异步 | 广播/集群 | 广播/集群/顺序/事务 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 可靠性 | 进程崩溃丢失 | Redis 崩溃丢失 | 高可靠（多副本） |
| 复杂度 | 最低 | 低 | 中等 |

### 决策流程

```
需要解耦的操作
      │
      ├─ 在同一个服务内？──→ 是 ──→ 用 Spring Event
      │                         （加 @Async 实现异步）
      │
      └─ 跨服务？──→ 是 ──→ 消息量大？对可靠性要求高？
                                    │
                                    ├─ 是 ──→ 用 RocketMQ/RabbitMQ
                                    │
                                    └─ 否 ──→ 用 Redis Stream
```

### yudao-cloud 中的实际选择

项目中大量使用 Spring Event 做进程内解耦：

```java
// 消息定义：就是一个普通的 POJO
@Data
public class MailSendMessage {
    private Long logId;
    private Collection<String> toMails;
    private String title;
    private String content;
    // ...
}

// Producer：注入 ApplicationEventPublisher，发布事件
publisher.publishEvent(new MailSendMessage().setLogId(1L)...);

// Consumer：@EventListener 监听，@Async 异步执行
@EventListener
@Async
public void onMessage(MailSendMessage message) {
    mailSendService.doSendMail(message);
}
```

源码位置：`yudao-module-system/yudao-module-system-server/.../mq/consumer/mail/MailSendConsumer.java`

注意 `@Async` 的作用——Spring Event 默认在 Producer 线程中同步执行，加上 `@Async` 后才会异步执行。如果不加异步，"解耦"只是代码层面的解耦，性能上并没有提升。

## Redis 作为 MQ 的优缺点

yudao-cloud 选择 Redis 作为默认的 MQ 实现，这是一个务实的选择——大多数项目已经部署了 Redis，不需要额外搭建 MQ 集群。

### 优点

1. **零额外部署**。Redis 已经是标配组件，不需要再装 RocketMQ/RabbitMQ
2. **低延迟**。Redis 基于内存，消息传递延迟在毫秒级
3. **API 简单**。Redis 的 Pub/Sub 和 Stream 命令直观易懂
4. **Stream 弥补了 Pub/Sub 的不足**。Redis 5.0 引入的 Stream 支持持久化、消费者组、ACK 机制，让 Redis 具备了"正经 MQ"的能力

### 缺点

1. **内存限制**。Redis 数据都在内存中，大量消息会占用宝贵内存
2. **可靠性有限**。Redis 的持久化（RDB/AOF）不如专业 MQ 的多副本机制
3. **功能简单**。没有延迟消息、事务消息、死信队列等高级特性
4. **运维风险**。Redis 重启时，Pub/Sub 的消息会丢失；Stream 的消息虽然持久化了，但故障恢复时间可能较长

### Stream 的关键改进

Redis Stream（5.0+）是 yudao-cloud 重点使用的特性，它解决了 Pub/Sub 的核心痛点：

```
Pub/Sub 的问题：
  Producer ──publish──→ Channel ──→ Consumer A ✓ 收到
                                ──→ Consumer B ✓ 收到
                                ──→ Consumer C ✗ 没订阅（或当时不在线）
  消息不会保留，后来者看不到

Stream 的改进：
  Producer ──XADD──→ Stream Key ──→ Consumer Group 1 ──→ Consumer A
                                  ──→ Consumer Group 2 ──→ Consumer B
  消息持久化，消费者组内只有一个消费者收到（集群消费）
  消费者组间各自独立消费（广播消费）
  支持 XACK 确认机制
```

yudao-cloud 的自动配置会检查 Redis 版本，低于 5.0 会直接报错：

```java
// 源码位置：YudaoRedisMQConsumerAutoConfiguration.java
int majorVersion = Integer.parseInt(StrUtil.subBefore(version, '.', false));
if (majorVersion < 5) {
    throw new IllegalStateException("您当前的 Redis 版本为 " + version + "，小于最低要求的 5.0.0 版本！");
}
```

## 设计目标

基于以上分析，yudao-cloud 的消息队列组件有四个核心设计目标：

### 目标一：可插拔

一行配置切换 MQ 后端。业务代码不感知底层用的是 Redis 还是 RocketMQ。

```
# application.yml
yudao:
  mq:
    type: redis    # 切换为 redis / rocketmq / rabbitmq / kafka
```

**实现方式**：Redis MQ 有自己的 `RedisMQTemplate`；其他 MQ 直接用 Spring Boot Starter 的原生模板（`RocketMQTemplate`、`RabbitTemplate`、`KafkaTemplate`）。切换 MQ 意味着切换依赖和配置，业务层的 Producer/Consumer 代码需要适配，但消息定义（POJO）可以复用。

### 目标二：统一 API（Redis MQ）

对于 Redis MQ，提供统一的发送和消费 API，屏蔽 Pub/Sub 和 Stream 的底层差异。

```
RedisMQTemplate.send(message)
      │
      ├─ message 是 ChannelMessage？──→ redisTemplate.convertAndSend(channel, json)
      │                                 （Pub/Sub 广播）
      │
      └─ message 是 StreamMessage？──→ redisTemplate.opsForStream().add(...)
                                        （Stream 集群消费）
```

### 目标三：支持消息拦截

通过拦截器机制，在消息发送/消费前后插入横切逻辑（如多租户、日志、监控）。

```
发送流程：
  sendMessageBefore() ──→ 实际发送 ──→ sendMessageAfter()
       ↑                                    ↑
  正序执行拦截器                      倒序执行拦截器

消费流程：
  consumeMessageBefore() ──→ 实际消费 ──→ consumeMessageAfter()
       ↑                                      ↑
  正序执行拦截器                        倒序执行拦截器
```

这个设计借鉴了 Spring MVC 的 `HandlerInterceptor` 模式——Before 正序、After 倒序，形成"洋葱模型"。

### 目标四：保障消息可靠性

Redis Stream 的消息虽然持久化了，但消费者可能崩溃导致消息卡在 Pending 队列。yudao-cloud 通过两个后台任务解决这个问题：

1. **RedisPendingMessageResendJob**：每分钟扫描一次，将超时 5 分钟的 Pending 消息重新投递
2. **RedisStreamMessageCleanupJob**：每小时清理一次，每个 Stream 只保留最近 10000 条消息，防止内存溢出

## MQ 选型决策框架

面对一个具体场景，如何选择合适的消息方案？以下是一个实用的决策框架：

```
┌─────────────────────────────────────────────────────────────────┐
│                    消息方案选型决策树                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Q1: 消息需要跨服务传递吗？                                     │
│  ├─ 否 → Spring Event（最简单，零依赖）                         │
│  └─ 是 → Q2                                                    │
│                                                                 │
│  Q2: 消息量有多大？对可靠性要求多高？                           │
│  ├─ 中小规模，可接受偶尔丢失 → Redis Stream                    │
│  └─ 大规模，不能丢失 → Q3                                      │
│                                                                 │
│  Q3: 需要事务消息、延迟消息等高级特性吗？                       │
│  ├─ 是 → RocketMQ                                              │
│  └─ 否 → Q4                                                    │
│                                                                 │
│  Q4: 需要灵活的路由规则（Topic/Tag/Exchange）吗？              │
│  ├─ 是 → RabbitMQ                                              │
│  └─ 否 → Q5                                                    │
│                                                                 │
│  Q5: 主要用于日志收集、大数据管道吗？                          │
│  ├─ 是 → Kafka                                                 │
│  └─ 否 → Redis Stream（默认选择）                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 各方案适用场景速查

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 用户注册后发优惠券 | Spring Event | 同一服务内，简单高效 |
| 订单创建后通知库存服务 | Redis Stream | 跨服务，需要可靠投递 |
| WebSocket 消息广播到所有节点 | Redis Pub/Sub | 广播模式，实时性优先 |
| 大促期间订单处理 | RocketMQ | 高吞吐，支持事务消息 |
| 电商下单后发短信 | Spring Event 或 Redis Stream | 取决于短信服务是否独立部署 |
| 日志收集到 ELK | Kafka | 超高吞吐，顺序写入 |

## 思考题

1. **为什么 yudao-cloud 没有为所有 MQ 提供统一的 Template 接口？** 提示：考虑不同 MQ 的特性差异（如 RocketMQ 的 Tag 过滤、RabbitMQ 的 Exchange 路由）。

2. **Redis Stream 的消费者组和 RocketMQ 的 Consumer Group 有什么异同？** 提示：从消费模式、负载均衡、消息回溯三个角度对比。

3. **如果业务需要"消息必达"（at-least-once），Redis Stream 够用吗？还需要做什么？** 提示：考虑 Redis 主从切换时的数据丢失风险。
