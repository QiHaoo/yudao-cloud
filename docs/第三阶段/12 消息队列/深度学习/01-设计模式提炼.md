# 01-设计模式提炼

本文从消息队列组件中识别出五种核心设计模式，将它们从具体实现中抽象为通用知识，帮助读者在其他项目中识别和应用相同的设计手法。

---

## 模式一：发布订阅模式（Publish-Subscribe Pattern）

### 在本项目中的体现

yudao-cloud 的消息队列组件提供了两种发布订阅的实现：

**1. Spring Event（进程内发布订阅）**

```java
// Producer：发布事件
publisher.publishEvent(new MailSendMessage().setLogId(1L).setToMails(...));

// Consumer：监听事件
@EventListener
@Async
public void onMessage(MailSendMessage message) {
    mailSendService.doSendMail(message);
}
```

**2. Redis Pub/Sub（跨进程发布订阅）**

```java
// Producer：发送 Channel 消息
redisMQTemplate.send(new RedisWebSocketMessage().setSessionId("abc"));

// Consumer：监听 Channel
public class RedisWebSocketMessageConsumer extends AbstractRedisChannelMessageListener<RedisWebSocketMessage> {
    @Override
    public void onMessage(RedisWebSocketMessage message) {
        redisWebSocketMessageSender.send(message.getSessionId(), ...);
    }
}
```

### 通用化

发布订阅模式的核心思想是：**发布者和订阅者解耦，发布者不需要知道有多少订阅者，订阅者也不需要知道消息从哪来**。

```
┌─────────────────────────────────────────────────────────────┐
│                    发布订阅模式                                │
│                                                             │
│  发布者 ──publish──→ 事件通道 ──notify──→ 订阅者 A           │
│                                        ──notify──→ 订阅者 B  │
│                                        ──notify──→ 订阅者 C  │
│                                                             │
│  发布者不知道订阅者的存在                                      │
│  订阅者之间互不知道彼此的存在                                  │
│  新增订阅者不需要修改发布者代码                                │
└─────────────────────────────────────────────────────────────┘
```

**两种变体的对比**：

| 维度 | 观察者模式（Spring Event） | 消息中间件模式（Redis MQ） |
|------|--------------------------|--------------------------|
| 耦合度 | 进程内，通过 Spring 容器耦合 | 跨进程，通过消息通道耦合 |
| 持久化 | 无 | Stream 支持 |
| 可靠性 | 进程崩溃丢失 | Redis 崩溃丢失 |
| 适用范围 | 同一 JVM 进程内 | 跨服务、跨进程 |

**在本项目中的选型决策**：

```
需要解耦的操作
  │
  ├── 在同一个服务内？→ Spring Event（最简单）
  │
  └── 跨服务？
        ├── 中小规模，可接受偶尔丢失 → Redis Stream
        └── 大规模，不能丢失 → RocketMQ/RabbitMQ
```

## 模式二：拦截器链模式（Interceptor Chain Pattern）

### 在本项目中的体现

`RedisMQTemplate` 在消息发送和消费前后执行拦截器链，支持多租户信息传递、日志记录等横切逻辑。

```java
// RedisMQTemplate.java
public <T extends AbstractRedisChannelMessage> void send(T message) {
    try {
        sendMessageBefore(message);  // 正序执行拦截器
        redisTemplate.convertAndSend(message.getChannel(), JsonUtils.toJsonString(message));
    } finally {
        sendMessageAfter(message);   // 倒序执行拦截器
    }
}

private void sendMessageBefore(AbstractRedisMessage message) {
    interceptors.forEach(interceptor -> interceptor.sendMessageBefore(message));
}

private void sendMessageAfter(AbstractRedisMessage message) {
    for (int i = interceptors.size() - 1; i >= 0; i--) {
        interceptors.get(i).sendMessageAfter(message);
    }
}
```

拦截器接口定义：

```java
public interface RedisMessageInterceptor {
    default void sendMessageBefore(AbstractRedisMessage message) {}
    default void sendMessageAfter(AbstractRedisMessage message) {}
    default void consumeMessageBefore(AbstractRedisMessage message) {}
    default void consumeMessageAfter(AbstractRedisMessage message) {}
}
```

### 通用化

拦截器链模式的核心思想是：**将请求处理分解为多个独立的步骤，每个步骤封装在一个拦截器中，按顺序执行**。yudao-cloud 的实现遵循"洋葱模型"——Before 正序、After 倒序。

```
┌─────────────────────────────────────────────────────────────┐
│                    拦截器链（洋葱模型）                        │
│                                                             │
│  拦截器A.sendMessageBefore()   ← 最先执行                    │
│    拦截器B.sendMessageBefore()                               │
│      拦截器C.sendMessageBefore()                             │
│        ↓                                                    │
│      实际发送消息                                             │
│        ↓                                                    │
│      拦截器C.sendMessageAfter()  ← 最后执行（倒序）           │
│    拦截器B.sendMessageAfter()                                │
│  拦截器A.sendMessageAfter()                                  │
└─────────────────────────────────────────────────────────────┘
```

**与其他类似模式的区别**：

| 模式 | 特点 | 本项目实例 |
|------|------|-----------|
| 过滤器链（Filter Chain） | 每个 Filter 决定是否继续 | Spring Security Filter Chain |
| 拦截器链（Interceptor Chain） | 所有拦截器都执行，Before/After 对称 | RedisMQTemplate 拦截器 |
| 责任链（Chain of Responsibility） | 每个 Handler 决定是否处理 | Netty ChannelPipeline |

**适用场景**：

| 场景 | 拦截器职责 | 本项目实例 |
|------|-----------|-----------|
| 多租户 | 消息 Header 写入/读取租户 ID | TenantRedisMessageInterceptor |
| 日志记录 | 记录消息发送/消费日志 | 自定义 MessageLoggingInterceptor |
| 链路追踪 | 注入 traceId | 可扩展 |
| 监控埋点 | 统计消息发送/消费耗时 | 可扩展 |

### 设计要点

1. **try-finally 保障清理**：即使发送失败，After 方法也必须执行，防止上下文泄漏
2. **default 空实现**：拦截器接口的方法都是 default，实现类只需覆盖关心的方法
3. **自动注册**：通过 Spring 的 `List<RedisMessageInterceptor>` 自动收集所有拦截器 Bean

## 模式三：模板方法模式（Template Method Pattern）

### 在本项目中的体现

`AbstractRedisChannelMessageListener` 和 `AbstractRedisStreamMessageListener` 是两个消费者基类，它们定义了消息处理的固定流程（反序列化 → 拦截器 Before → 业务消费 → ACK → 拦截器 After），子类只需实现 `onMessage(T)` 方法。

```java
// AbstractRedisChannelMessageListener.java
public abstract class AbstractRedisChannelMessageListener<T extends AbstractRedisChannelMessage>
        implements MessageListener {

    @Override
    public final void onMessage(Message message, byte[] bytes) {
        // 1. 反序列化（固定流程）
        T messageObj = JsonUtils.parseObject(message.getBody(), messageType);
        try {
            // 2. 拦截器 before（固定流程）
            consumeMessageBefore(messageObj);
            // 3. 业务消费（子类实现）
            this.onMessage(messageObj);
        } finally {
            // 4. 拦截器 after（固定流程）
            consumeMessageAfter(messageObj);
        }
    }

    // 子类只需实现这个方法
    public abstract void onMessage(T message);
}
```

Stream 消费者额外增加了 ACK 机制：

```java
// AbstractRedisStreamMessageListener.java
@Override
public void onMessage(ObjectRecord<String, String> message) {
    T messageObj = JsonUtils.parseObject(message.getValue(), messageType);
    try {
        consumeMessageBefore(messageObj);
        this.onMessage(messageObj);
        // 关键：消费成功后 ACK
        redisMQTemplate.getRedisTemplate().opsForStream().acknowledge(group, message);
    } finally {
        consumeMessageAfter(messageObj);
    }
}
```

### 通用化

模板方法模式的核心思想是：**在父类中定义算法的骨架，将某些步骤延迟到子类实现**。父类控制整体流程，子类填充细节。

```
┌─────────────────────────────────────────────────────────────┐
│                    模板方法模式                                │
│                                                             │
│  父类（定义骨架）：                                            │
│    step1()        ← 固定实现                                  │
│    step2()        ← 固定实现                                  │
│    doSomething()  ← 抽象方法，子类实现                         │
│    step3()        ← 固定实现                                  │
│                                                             │
│  子类（填充细节）：                                            │
│    doSomething()  ← 实现具体业务逻辑                          │
└─────────────────────────────────────────────────────────────┘
```

**在本项目中的体现层次**：

```
AbstractRedisMessage                    ← 消息基类
    ├── AbstractRedisChannelMessage     ← 定义 getChannel()
    └── AbstractRedisStreamMessage      ← 定义 getStreamKey()

AbstractRedisChannelMessageListener     ← 消费者模板
    └── RedisWebSocketMessageConsumer   ← 实现 onMessage()

AbstractRedisStreamMessageListener      ← 消费者模板
    └── 具体 Stream 消费者               ← 实现 onMessage()
```

### 设计要点

1. **`final` 方法保护骨架**：`onMessage(Message, byte[])` 是 final 的，子类不能改变处理流程
2. **泛型推断**：通过 `TypeUtil.getTypeArgument(getClass(), 0)` 自动获取消息类型，子类不需要手动传 Class
3. **channel/streamKey 自动推导**：通过反射创建消息实例并调用 `getChannel()`/`getStreamKey()`，避免硬编码

## 模式四：工厂模式（Factory Pattern）

### 在本项目中的体现

yudao-cloud 的消息队列组件通过 Spring Boot 的自动配置机制，根据条件自动创建不同的 MQ 组件实例。这本质上是一种工厂模式——根据配置和环境条件，决定创建哪些 Bean。

```java
// YudaoRedisMQConsumerAutoConfiguration.java
@AutoConfiguration(after = YudaoRedisAutoConfiguration.class)
@EnableScheduling
public class YudaoRedisMQConsumerAutoConfiguration {

    // 只有存在 Channel 监听器时才创建 Pub/Sub 容器
    @Bean
    @ConditionalOnBean(AbstractRedisChannelMessageListener.class)
    public RedisMessageListenerContainer redisMessageListenerContainer(...) { ... }

    // 只有存在 Stream 监听器时才创建 Stream 容器
    @Bean
    @ConditionalOnBean(AbstractRedisStreamMessageListener.class)
    public StreamMessageListenerContainer redisStreamMessageListenerContainer(...) { ... }

    // 只有存在 Stream 监听器时才启动重发任务
    @Bean
    @ConditionalOnBean(AbstractRedisStreamMessageListener.class)
    public RedisPendingMessageResendJob redisPendingMessageResendJob(...) { ... }
}
```

### 通用化

工厂模式的核心思想是：**将对象的创建逻辑封装起来，让调用方不需要知道创建的细节**。Spring Boot 的自动配置是工厂模式的一种高级形式——通过条件注解实现"条件化工厂"。

```
┌─────────────────────────────────────────────────────────────┐
│                    条件化工厂（Spring Boot 自动配置）           │
│                                                             │
│  条件检查：                                                   │
│    @ConditionalOnClass      → 依赖存在吗？                    │
│    @ConditionalOnBean       → Bean 存在吗？                   │
│    @ConditionalOnProperty   → 配置满足吗？                    │
│                                                             │
│  所有条件满足 → 创建 Bean                                    │
│  任一条件不满足 → 跳过，不创建                                │
└─────────────────────────────────────────────────────────────┘
```

**各 MQ 后端的工厂策略**：

| MQ | 工厂方式 | 条件 |
|----|---------|------|
| Redis (Producer) | 自动配置创建 RedisMQTemplate | 无条件（默认启用） |
| Redis (Consumer) | 自动配置创建 ListenerContainer | `@ConditionalOnBean` 监听器存在 |
| RabbitMQ | 自动配置创建 Jackson2JsonMessageConverter | `@ConditionalOnClass(RabbitTemplate.class)` |
| RocketMQ | Spring Boot Starter 原生创建 | 依赖 rocketmq-spring-boot-starter |
| Kafka | Spring Boot Starter 原生创建 | 依赖 spring-kafka |

### 设计要点

1. **`@ConditionalOnBean` 按需加载**：没有定义 Stream 监听器就不会创建 Stream 容器和后台任务，节省资源
2. **`@ConditionalOnMissingBean` 可覆盖**：允许用户自定义实现覆盖默认 Bean
3. **分层工厂**：Producer 和 Consumer 的自动配置分离，各自独立生效

## 模式五：适配器模式（Adapter Pattern）

### 在本项目中的体现

yudao-cloud 面临一个挑战：需要为 Redis、RocketMQ、RabbitMQ、Kafka 四种 MQ 实现统一的多租户消息传递。每种 MQ 的 API 完全不同，但租户传递的逻辑是一样的。

解决方案是为每种 MQ 编写一个适配器，将统一的租户传递逻辑"适配"到各 MQ 的原生扩展点上：

```
统一逻辑：发送前写入 tenant-id，消费前读取 tenant-id，消费后清理
    │
    ├── Redis 适配器：RedisMessageInterceptor
    │     sendMessageBefore → message.addHeader("tenant-id", ...)
    │
    ├── RocketMQ 适配器：SendMessageHook + ConsumeMessageHook
    │     sendMessageBefore → context.getMessage().putUserProperty(...)
    │
    ├── RabbitMQ 适配器：MessagePostProcessor
    │     postProcessMessage → message.getMessageProperties().getHeaders().put(...)
    │
    └── Kafka 适配器：ProducerInterceptor
          onSend → record.headers().add(...)
```

### 通用化

适配器模式的核心思想是：**将一个类的接口转换成客户端期望的另一个接口，使原本不兼容的类可以协同工作**。

```
┌─────────────────────────────────────────────────────────────┐
│                    适配器模式                                  │
│                                                             │
│  统一接口（租户传递逻辑）                                      │
│      │                                                      │
│      ├── 适配器 A → Redis API                                │
│      ├── 适配器 B → RocketMQ API                             │
│      ├── 适配器 C → RabbitMQ API                             │
│      └── 适配器 D → Kafka API                                │
│                                                             │
│  每个适配器将统一逻辑"翻译"成特定 MQ 的原生 API                │
└─────────────────────────────────────────────────────────────┘
```

**各 MQ 适配器的注册方式差异**：

| MQ | 适配器类型 | 注册方式 | 特殊处理 |
|----|-----------|---------|---------|
| Redis | `RedisMessageInterceptor` | Spring Bean 自动注入 | 无 |
| RocketMQ | `SendMessageHook` + `ConsumeMessageHook` | `BeanPostProcessor` 注册 | 要求批量消费只能有一条消息 |
| RabbitMQ | `MessagePostProcessor` | `BeanPostProcessor` 注册到 RabbitTemplate | 无 |
| Kafka | `ProducerInterceptor` | `EnvironmentPostProcessor` 注入配置 | 需要反射访问 private headers 字段 |

### 与其他模式的区别

| 模式 | 目的 | 关系 |
|------|------|------|
| 适配器模式 | 接口转换，使不兼容的类协同工作 | 改变接口 |
| 装饰器模式 | 增强对象功能，不改变接口 | 增强功能 |
| 代理模式 | 控制对象访问 | 控制访问 |

yudao-cloud 的多租户 MQ 集成更接近适配器模式——它将"写入租户 ID 到消息"这个统一逻辑，适配到四种完全不同的 MQ API 上。

### 设计要点

1. **统一逻辑，分散适配**：租户传递的核心逻辑（写入 Header、读取 Header、清理上下文）是统一的，只是"写入/读取"的方式因 MQ 而异
2. **利用原生扩展点**：每种适配器都使用 MQ 原生的扩展机制（Hook、Interceptor、PostProcessor），不修改 MQ 核心代码
3. **自动注册**：通过 `BeanPostProcessor`、`EnvironmentPostProcessor` 等机制自动注册适配器，业务代码无感知

## 模式间的协作

这五种模式在消息队列组件中协同工作：

```
一个消息从发送到消费的完整生命周期中涉及的模式：

1. 发布订阅模式
   Producer 发布消息 → 消息通道 → Consumer 订阅消息

2. 拦截器链模式
   消息发送前后执行拦截器链（租户传递、日志记录等）

3. 模板方法模式
   Consumer 基类定义固定的处理流程，子类只实现业务逻辑

4. 工厂模式
   Spring Boot 自动配置根据条件创建 MQ 组件

5. 适配器模式
   统一的租户传递逻辑适配到不同 MQ 的原生 API

协作关系：
  工厂(4) 创建 MQ 组件 → 发布订阅(1) 传递消息
    → 拦截器链(2) 增强消息 → 适配器(5) 适配租户传递
    → 模板方法(3) 消费消息
```

## 思考题

1. **为什么 yudao-cloud 对 Redis MQ 做了统一模板（RedisMQTemplate），但对 RocketMQ/RabbitMQ/Kafka 没有？** 提示：考虑不同 MQ 的特性差异（Tag 过滤、Exchange 路由、分区有序）。

2. **拦截器链模式中的"洋葱模型"与 Spring MVC 的 HandlerInterceptor 有什么异同？** 提示：考虑 After 方法的执行条件。

3. **如果要为消息队列组件添加链路追踪支持（自动注入 traceId），应该用哪种模式？** 提示：拦截器链模式是最自然的选择。
