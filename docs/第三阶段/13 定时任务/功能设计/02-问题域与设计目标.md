# 02-问题域与设计目标

## 从电商场景说起

上一章我们看到了定时任务的全景。现在让我们把镜头拉近，看看在真实业务中，定时任务到底会遇到哪些"坑"。

### 场景一：商品定时上下架

运营团队配置了一个促销活动：商品 A 在明天上午 10 点自动上架，后天晚上 12 点自动下架。

```
问题：这个"自动"由谁来执行？

方案 A：在商品服务里写一个 while 循环，每秒检查一次是否到了上架时间
        → CPU 空转，资源浪费严重

方案 B：用 Thread.sleep() 等到指定时间再执行
        → 服务重启后丢失，无法恢复

方案 C：用定时任务，每分钟扫描一次"到达上架时间"的商品
        → 正解。但谁来扫描？怎么保证只扫描一次？
```

在单体应用中，一个 `@Scheduled` 就搞定了。但当系统部署了 3 个实例：

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  实例 A       │  │  实例 B       │  │  实例 C       │
│              │  │              │  │              │
│ @Scheduled   │  │ @Scheduled   │  │ @Scheduled   │
│ 每分钟扫描   │  │ 每分钟扫描   │  │ 每分钟扫描   │
│      ↓       │  │      ↓       │  │      ↓       │
│ 发现100件商品│  │ 发现100件商品│  │ 发现100件商品│
│ 执行上架     │  │ 执行上架     │  │ 执行上架     │
└──────────────┘  └──────────────┘  └──────────────┘
        ↓                ↓                ↓
   商品被上架了 3 次，触发了 3 次缓存刷新，
   发送了 3 条"商品上架"消息通知
```

**核心矛盾**：@Scheduled 是"每人一份"的执行模式，而定时任务通常需要"全局只执行一次"。

### 场景二：每日对账

每天凌晨 2 点，系统需要和第三方支付平台（如支付宝、微信支付）进行对账：逐笔核对前一天的交易记录，找出差异。

```
对账流程：
1. 下载第三方的对账文件（可能几 MB ~ 几百 MB）
2. 逐笔比对本地订单和第三方记录
3. 标记差异记录，生成对账报告
4. 差异订单进入人工审核流程
```

这个任务有几个特点：

```
执行时间长 ──── 数据量大时可能需要 30 分钟以上
                不能中途失败就放弃，否则第二天的对账会更复杂

失败不可怕 ──── 网络抖动导致下载失败，应该自动重试
                而不是等到第二天才发现对账没做

需要可观测 ──── 财务总监问"昨天的对账做了吗？结果怎么样？"
                你需要一个地方能查到执行记录
```

**核心矛盾**：简单的 @Scheduled 只管"到点触发"，不管"触发之后发生了什么"。

### 场景三：多租户的数据统计

你的电商 SaaS 平台有 50 个租户（50 家独立的店铺），每天凌晨需要统计每家店铺的销售数据。

```
天真方案：为每个租户创建一个定时任务
         → 50 个租户 = 50 个任务配置
         → 新增一个租户？手动去 XXL-Job 控制台再加一个任务
         → 删除一个租户？手动去删。忘了？任务执行报错

期望方案：一个任务，自动遍历所有租户执行
         → 新增租户？自动覆盖，不用改任何配置
         → 删除租户？自动跳过
```

**核心矛盾**：定时任务的配置（几个任务、什么时候执行）是相对固定的，但租户数量是动态变化的。

## 问题域分析

把上面的场景抽象一下，分布式定时任务要解决的核心问题可以归纳为四类：

### 问题一：防重复执行

```
问题本质：多个实例共享同一个调度触发点

          触发信号
             │
    ┌────────┼────────┐
    ▼        ▼        ▼
 实例 A   实例 B   实例 C
 (执行)   (执行)   (执行)    ← 重复执行

解决方向：
  a) 分布式锁 ── 只有一个实例能抢到锁
  b) 集中调度 ── 由一个中心节点决定谁执行
  c) 分片 ──── 把任务拆分，每个实例执行一部分
```

### 问题二：失败处理

```
问题本质：任务执行失败后怎么办？

时间线：
  02:00  对账任务启动
  02:05  下载对账文件成功
  02:15  比对到第 5000 条时，数据库连接超时
  02:15  任务失败 ──→ ???

  选项 A：什么都不做，等明天凌晨再执行
           → 问题：今天的对账就丢了

  选项 B：自动重试
           → 问题：从头重试还是从失败点继续？

  选项 C：告警通知 + 人工介入
           → 至少不会悄悄失败
```

### 问题三：可观测性

```
问题本质：任务执行的"黑盒"状态

  运营："昨天的报表生成任务执行了吗？"
  开发："我查查日志……嗯，服务器 A 的日志里有，但不确定
        是不是最新的，等我再去服务器 B 和 C 看看……"

期望：
  ┌──────────────────────────────────────────┐
  │          任务执行记录（集中管理）           │
  ├──────┬──────┬───────┬────────┬───────────┤
  │ 任务 │ 时间 │ 状态  │ 耗时   │ 结果详情  │
  ├──────┼──────┼───────┼────────┼───────────┤
  │ 对账 │ 02:00│ 成功  │ 25min  │ 差异3条   │
  │ 报表 │ 03:00│ 失败  │ 5min   │ DB超时    │
  │ 清理 │ 04:00│ 成功  │ 2min   │ 删1000条  │
  └──────┴──────┴───────┴────────┴───────────┘
```

### 问题四：任务与租户的耦合

```
问题本质：SaaS 场景下，定时任务需要覆盖所有租户

静态配置（反模式）：
  任务 A：处理租户 1 的过期订单
  任务 B：处理租户 2 的过期订单
  任务 C：处理租户 3 的过期订单
  ... 每增加一个租户就要加一个任务

动态遍历（正确做法）：
  任务 X：遍历所有租户，逐个处理过期订单
  → 租户数量变化时，无需修改任务配置
```

## 设计目标

基于以上分析，yudao-cloud 定时任务组件的设计目标如下：

```
┌─────────────────────────────────────────────────────────────────┐
│                     设计目标                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 集中调度，防重复执行                                         │
│     ┌──────────┐                                                │
│     │ 调度中心  │ ─── 只触发一次 ──→ 执行器实例                  │
│     └──────────┘                                                │
│                                                                 │
│  2. 开箱即用的 Spring Boot Starter                               │
│     引入依赖 + 配置地址 = 自动装配完成                           │
│     不需要手动注册执行器、不需要手动初始化                        │
│                                                                 │
│  3. 多租户自动遍历                                               │
│     一个 @TenantJob 注解 = 自动遍历所有租户执行                  │
│     租户数量变化时无需修改任务配置                                │
│                                                                 │
│  4. 与 Spring @Scheduled 共存                                    │
│     简单任务用 @Scheduled，复杂任务用 XXL-Job                    │
│     两者互不干扰，各司其职                                       │
│                                                                 │
│  5. 最小侵入                                                     │
│     业务代码只需要写 @XxlJob 方法，不需要关心                    │
│     执行器注册、日志上报、路由策略等底层细节                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 目标与非目标

| 维度 | 目标（做什么） | 非目标（不做什么） |
|------|---------------|-------------------|
| 调度 | 依赖 XXL-Job Admin 集中调度 | 不自研调度引擎 |
| 执行 | 提供自动装配 + 多租户支持 | 不封装 XXL-Job 的 API（保持原生能力） |
| 监控 | 依赖 XXL-Job Admin 的控制台 | 不自建任务监控系统 |
| 部署 | 单体和微服务统一配置方式 | 不做任务的热部署/热更新 |
| 租户 | 自动遍历 + 幂等保障 | 不做租户级任务隔离（每个租户独立任务） |

## 设计取舍：为什么不在框架层自研调度

你可能会问：既然有这么多问题，为什么不自己实现一个调度引擎？

```
自研调度引擎的成本：
  ├── 调度算法（Cron 解析、下次触发时间计算）
  ├── 分布式锁（保证不重复触发）
  ├── 故障转移（调度节点挂了怎么办）
  ├── 日志收集（执行日志的存储和查询）
  ├── 管理界面（任务的增删改查、启停控制）
  └── 持久化（任务信息、执行记录存储）

vs.

使用 XXL-Job 的成本：
  ├── 部署 XXL-Job Admin（一个 Java 应用 + 一个 MySQL 库）
  ├── 引入 xxl-job-core 依赖
  └── 写一个自动配置类（~50 行代码）
```

结论很清晰：**调度引擎是基础设施，不是业务差异化所在**。用成熟的开源方案，把精力花在业务逻辑上，这才是正确的投入方向。

## 小结

本章从三个电商场景出发，分析了分布式定时任务的四个核心问题（防重复、失败处理、可观测、多租户），并明确了 yudao-cloud 的设计目标：以 XXL-Job 为底座，通过 Spring Boot Starter 提供开箱即用的自动装配，通过 @TenantJob 注解解决多租户场景的遍历问题。
