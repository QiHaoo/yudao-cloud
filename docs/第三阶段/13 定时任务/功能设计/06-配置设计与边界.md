# 06-配置设计与边界

## 配置项全景

yudao-cloud 的定时任务配置项并不多，但每一项都有明确的设计意图。让我们把它们梳理清楚。

### 完整配置树

```yaml
xxl:
  job:
    # ===== 开关 =====
    enabled: true                      # 是否启用 XXL-Job，默认 true

    # ===== 安全 =====
    accessToken: default_token         # Admin 与执行器之间的通信令牌

    # ===== 调度中心 =====
    admin:
      addresses: http://127.0.0.1:9090/xxl-job-admin  # Admin 地址（必填）

    # ===== 执行器 =====
    executor:
      appname: ${spring.application.name}              # 执行器名称（必填）
      logpath: ${user.home}/logs/xxl-job/${spring.application.name}  # 日志路径（必填）
      ip:                                # 执行器 IP（可选，默认自动获取）
      port: -1                           # 执行器端口（可选，默认随机）
      log-retention-days: 30             # 日志保留天数（可选，默认 30）
```

### 配置分层策略

yudao-cloud 将配置分为三层，各层的职责不同：

```
application.yaml（服务级公共配置）
├── executor.appname: ${spring.application.name}    ← 所有环境共享
├── executor.logpath: ${user.home}/logs/...         ← 所有环境共享
└── accessToken: default_token                      ← 所有环境共享

application-local.yaml（本地开发配置）
├── enabled: false                                  ← 本地开发禁用
└── admin.addresses: http://127.0.0.1:9090/...     ← 本地 Admin 地址

application-dev.yaml（开发环境配置）
└── admin.addresses: http://127.0.0.1:9090/...     ← 开发环境 Admin 地址

application-prod.yaml（生产环境配置）
├── enabled: true                                   ← 生产环境启用
└── admin.addresses: http://xxljob.prod:9090/...   ← 生产 Admin 地址
```

为什么这样分层？

```
问题：如果所有配置都写在一个 application.yaml 里

  本地开发：需要手动改 admin.addresses 为 localhost
  部署测试：需要手动改 admin.addresses 为测试环境地址
  部署生产：需要手动改 admin.addresses 为生产环境地址

  每次切换环境都要改配置文件，容易出错

解决：Spring Profile 机制

  application.yaml         → 公共配置，不随环境变化
  application-{profile}.yaml → 环境特定配置

  启动时通过 spring.profiles.active=local/dev/prod 选择
```

### 各环境的典型配置

**本地开发（application-local.yaml）**

```yaml
xxl:
  job:
    enabled: false  # 本地开发通常不需要 XXL-Job Admin
```

为什么本地开发要禁用？因为本地开发时通常没有运行 XXL-Job Admin 服务。如果不禁用，应用启动时会尝试连接 Admin 并注册执行器，连接失败会打印大量错误日志。

**开发/测试环境（application-dev.yaml）**

```yaml
xxl:
  job:
    admin:
      addresses: http://127.0.0.1:9090/xxl-job-admin
```

开发环境中，XXL-Job Admin 通常和应用部署在同一台机器上。

**生产环境（application-prod.yaml）**

```yaml
xxl:
  job:
    accessToken: your-secret-token-here  # 生产环境必须修改默认令牌
    admin:
      addresses: http://xxljob.internal.company.com:9090/xxl-job-admin
```

生产环境的关键点：
- `accessToken` 必须修改为强密码
- `addresses` 指向生产环境的 Admin 地址
- 不需要配置 `enabled`（默认就是 `true`）

## 配置的默认值设计

框架为大部分配置项设置了合理的默认值，只有少数必填项：

```
┌──────────────────────┬──────────┬──────────┬─────────────────────────┐
│ 配置项                │ 默认值    │ 是否必填 │ 设计意图                │
├──────────────────────┼──────────┼──────────┼─────────────────────────┤
│ enabled              │ true     │ 否       │ 默认启用，可选禁用      │
│ accessToken          │ null     │ 否       │ 默认无令牌              │
│ admin.addresses      │ -        │ 是       │ 必须知道 Admin 在哪     │
│ executor.appName     │ -        │ 是       │ 执行器必须有名称        │
│ executor.logPath     │ -        │ 是       │ 日志必须有存储路径      │
│ executor.ip          │ null     │ 否       │ 默认自动获取本机 IP     │
│ executor.port        │ -1       │ 否       │ 默认随机端口            │
│ executor.logRetentionDays │ 30  │ 否       │ 默认保留 30 天          │
└──────────────────────┴──────────┴──────────┴─────────────────────────┘
```

### port = -1（随机端口）的考量

为什么不默认使用固定端口？

```
场景：同一台机器部署了 3 个微服务
  yudao-system-server   → 需要 XXL-Job 端口
  yudao-trade-server    → 需要 XXL-Job 端口
  yudao-pay-server      → 需要 XXL-Job 端口

如果默认端口都是 9999：
  第 1 个服务启动成功，占用 9999
  第 2 个服务启动失败，端口冲突！
  第 3 个服务启动失败，端口冲突！

随机端口方案：
  每个服务自动选择可用端口，Admin 通过注册信息知道实际端口
  → 无端口冲突，无需手动规划端口分配
```

### logRetentionDays = 30 的考量

为什么默认 30 天？

```
太短（如 3 天）→ 无法回溯上个月的任务执行情况
太长（如 365 天）→ 日志表膨胀，查询变慢

30 天是一个平衡点：
  • 覆盖一个月内的执行记录
  • 满足大多数合规要求
  • 数据量可控

如果需要永久保留：设置为 -1
```

## 与 Spring @Scheduled 的关系

在 `YudaoXxlJobAutoConfiguration` 上有一个容易被忽略的注解：

```java
@AutoConfiguration
@EnableScheduling  // ← 这个
public class YudaoXxlJobAutoConfiguration { ... }
```

`@EnableScheduling` 开启了 Spring 自带的定时任务能力。这意味着在同一应用中，`@Scheduled` 和 `@XxlJob` 可以共存。

```
┌─────────────────────────────────────────────────────────────────┐
│                    两种定时任务共存                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Spring @Scheduled                    XXL-Job @XxlJob          │
│   ─────────────────                    ────────────────          │
│                                                                 │
│   触发方式：本地 TaskScheduler          触发方式：Admin 集中调度  │
│   执行位置：每个实例都执行              执行位置：指定实例执行    │
│   管理方式：代码即配置                  管理方式：Web 控制台      │
│   失败处理：无                          失败处理：重试 + 告警     │
│   日志追踪：无                          日志追踪：集中存储        │
│                                                                 │
│   适用：                              适用：                     │
│   • 缓存预热                          • 订单超时取消             │
│   • 本地文件清理                      • 佣金结算                 │
│   • 健康检查心跳                      • 数据统计                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

为什么把 `@EnableScheduling` 放在 XXL-Job 的配置类上？

```
设计考量：

  @EnableScheduling 放在 YudaoXxlJobAutoConfiguration 上
  → 引入 starter-job 就自动开启 @Scheduled
  → 不需要单独配置

  这样做的好处：
  1. 一个依赖解决所有定时任务需求
  2. @Scheduled 和 @XxlJob 可以在同一应用中共存
  3. 简单任务用 @Scheduled，复杂任务用 @XxlJob，开发者自行判断
```

选型决策表：

```
┌───────────────────────────────┬───────────┬──────────┐
│ 场景                           │ @Scheduled│ @XxlJob  │
├───────────────────────────────┼───────────┼──────────┤
│ 每个实例都要执行（如缓存预热） │ ✓         │          │
│ 全局只执行一次（如订单取消）   │           │ ✓        │
│ 需要失败重试                   │           │ ✓        │
│ 需要执行日志                   │           │ ✓        │
│ 需要分片并行                   │           │ ✓        │
│ 需要动态修改调度时间           │           │ ✓        │
│ 需要暂停/恢复任务              │           │ ✓        │
│ 简单的心跳/清理                │ ✓         │          │
└───────────────────────────────┴───────────┴──────────┘
```

## 边界：不做什么

好的框架设计不是做更多，而是**知道边界在哪里**。yudao-cloud 的定时任务组件明确不做以下事情：

### 1. 不自研调度引擎

```
XXL-Job 已经提供了：
  • Cron 解析和触发
  • 多种路由策略
  • 故障转移
  • 分片广播
  • 执行日志管理
  • Web 管理界面

框架不需要重复造轮子
```

### 2. 不封装 XXL-Job 的原生 API

```
XXL-Job 提供了丰富的 API：
  • XxlJobHelper.log()       → 输出执行日志
  • XxlJobHelper.handleSuccess() → 标记成功
  • XxlJobHelper.handleFail()    → 标记失败
  • XxlJobHelper.getJobParam()   → 获取任务参数
  • XxlJobContext                → 获取执行上下文

框架不封装这些 API
  → 业务代码直接使用原生 API
  → 不需要学习框架的封装层
  → 查 XXL-Job 官方文档就能解决问题
```

### 3. 不做任务的热部署

```
不支持的场景：
  • 在运行时动态创建新的 JobHandler
  • 在运行时修改 JobHandler 的代码逻辑
  • 不重启应用就生效新的任务代码

任务的变更流程：
  修改代码 → 构建 → 部署 → 在 Admin 控制台配置调度规则
```

### 4. 不做租户级任务隔离

```
不支持的场景：
  • 租户 A 每小时执行一次，租户 B 每天执行一次
  • 租户 A 用 Cron A，租户 B 用 Cron B

框架的设计：所有租户共享同一个调度规则
  → 一个任务，遍历所有租户执行
  → 不支持租户级别的差异化调度

如果确实需要：
  → 创建多个 JobHandler，代码中硬编码租户 ID 范围
  → 在 Admin 控制台配置不同的调度规则
```

### 5. 不做分布式锁保护

```
XXL-Job 本身通过调度中心保证"只触发一个实例"
框架不需要额外加分布式锁

但有一个边界情况：
  分片广播模式下，所有实例都会执行
  如果业务逻辑需要"每个租户只处理一次"：
  → 需要业务层自己处理（如数据库乐观锁）
  → 框架不提供这个能力
```

### 6. 不做跨模块的任务依赖

```
不支持的场景：
  任务 A 完成后 → 触发任务 B → 触发任务 C

框架的设计：每个任务独立执行
  → 任务之间没有依赖关系
  → 不支持 DAG（有向无环图）编排

如果确实需要：
  → 在任务 A 的代码末尾手动触发任务 B
  → 或者使用独立的工作流引擎
```

## 与其他组件的分工

### 与多租户组件的分工

```
定时任务组件（yudao-spring-boot-starter-job）：
  ├── 提供：XXL-Job 自动配置
  ├── 提供：XxlJobProperties 配置绑定
  └── 不涉及：租户相关逻辑

多租户组件（yudao-spring-boot-starter-biz-tenant）：
  ├── 提供：@TenantJob 注解
  ├── 提供：TenantJobAspect 切面
  ├── 提供：TenantUtils 工具类
  └── 条件依赖：xxl-job-core 存在时才激活

两者通过条件装配组合，互不耦合
```

### 与消息队列组件的分工

```
定时任务：按时间规则触发，处理"到点该做什么"
消息队列：按事件触发，处理"发生了什么之后该做什么"

示例：
  定时任务：每小时检查超时订单 → 发现超时订单
  消息队列：发送"订单超时"消息 → 通知库存服务释放库存

两者互补，不重叠
```

### 与 Spring Task 的分工

```
Spring @Scheduled：
  • 轻量级，无外部依赖
  • 每个实例独立执行
  • 适合"每个实例都需要做"的任务

XXL-Job @XxlJob：
  • 重量级，依赖 Admin + MySQL
  • 集中调度，防重复执行
  • 适合"全局只做一次"的任务

框架同时启用两者，由开发者根据场景选择
```

## 从单体到微服务的配置迁移

yudao-cloud 支持单体和微服务两种部署模式，定时任务的配置在两种模式下略有不同：

### 单体模式

```yaml
# yudao-server/src/main/resources/application.yaml
xxl:
  job:
    executor:
      appname: ${spring.application.name}  # yudao-server
    accessToken: default_token
```

单体模式下，所有模块的 JobHandler 都在一个进程中，只需一个执行器配置。

### 微服务模式

```yaml
# yudao-module-trade-server/src/main/resources/application.yaml
xxl:
  job:
    executor:
      appname: ${spring.application.name}  # yudao-module-trade-server
    accessToken: default_token

# yudao-module-pay-server/src/main/resources/application.yaml
xxl:
  job:
    executor:
      appname: ${spring.application.name}  # yudao-module-pay-server
    accessToken: default_token
```

微服务模式下，每个服务独立配置执行器，`appname` 自动使用 Spring 应用名，无需手动指定。

```
Admin 控制台看到的执行器列表：
  ┌──────────────────────────────┬─────────────┐
  │ 执行器名称                    │ 实例数      │
  ├──────────────────────────────┼─────────────┤
  │ yudao-server（单体）          │ 3           │
  │ yudao-module-trade-server    │ 2           │
  │ yudao-module-pay-server      │ 2           │
  │ yudao-module-member-server   │ 1           │
  └──────────────────────────────┴─────────────┘
```

迁移时唯一需要注意的是：`admin.addresses` 在单体和微服务模式下应该指向同一个 Admin 实例。

## 小结

本章讨论了定时任务组件的配置设计和边界：

- **配置分层**：公共配置 + 环境配置，通过 Spring Profile 切换
- **默认值设计**：大多数配置有合理默认值，只有 Admin 地址、执行器名称、日志路径是必填的
- **与 @Scheduled 共存**：两者互补，简单任务用 @Scheduled，复杂任务用 @XxlJob
- **明确的边界**：不自研调度引擎、不封装原生 API、不支持租户级隔离、不支持任务编排
- **组件分工**：定时任务组件和多租户组件正交组合，互不耦合
