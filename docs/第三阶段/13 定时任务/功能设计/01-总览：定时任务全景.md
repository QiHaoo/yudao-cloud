# 01-总览：定时任务全景

## 场景引入

想象你正在开发一个电商后台管理系统，随着业务的增长，你会发现有大量的"后台默默执行"的需求：

- **每天凌晨 2 点**：结算前一天的佣金，给分销员的账户打款
- **每小时整点**：检查超时未支付的订单，自动取消释放库存
- **每 30 分钟**：同步第三方仓库的库存数据，防止超卖
- **每天早上 8 点**：生成前一天的销售统计报表，推送给运营团队
- **每周一凌晨**：将超过 30 天未下单的会员自动降级

这些任务有一个共同特点：**不需要用户触发，按时间规则自动执行**。这就是定时任务的典型场景。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           电商后台管理系统                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐             │
│   │ 订单管理模块  │    │ 会员管理模块  │    │ 支付管理模块  │             │
│   └──────┬───────┘    └──────┬───────┘    └──────┬───────┘             │
│          │                   │                   │                      │
│          │    定时任务层      │                   │                      │
│          ▼                   ▼                   ▼                      │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐             │
│   │ 超时订单取消  │    │ 佣金结算     │    │ 支付单过期   │             │
│   │ 每小时执行   │    │ 每天凌晨执行  │    │ 每5分钟执行  │             │
│   └──────────────┘    └──────────────┘    └──────────────┘             │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 定时任务的核心需求

在单体应用时代，一个 `@Scheduled` 注解就能解决问题。但当系统演进到微服务架构，部署多个实例时，问题就来了：

### 1. 可靠调度

```
场景：电商系统的"超时订单取消"任务

问题：系统部署了 3 个实例，如果每个实例都执行一次取消逻辑：
      - 订单 A 被取消了 3 次（重复执行）
      - 库存被释放了 3 次（数据错乱）
      - 退款被触发了 3 次（资金风险）

期望：只有 1 个实例执行，其他实例"袖手旁观"
```

### 2. 失败重试

```
场景：凌晨的佣金结算任务

问题：任务执行到一半，数据库连接超时了
      - 已经结算的佣金记录：成功
      - 未结算的佣金记录：丢失了

期望：任务失败后能自动重试，或者能从失败点继续执行
```

### 3. 任务监控

```
场景：运营总监想知道"昨天的报表生成任务执行得怎么样"

问题：任务执行日志散落在各个服务器上，无法集中查看
      - 任务是否执行成功？
      - 执行了多长时间？
      - 处理了多少条数据？

期望：有一个统一的界面，能查看所有任务的执行历史和状态
```

### 4. 分布式执行

```
场景：需要同步 100 个仓库的库存数据

问题：单个实例执行太慢，需要多个实例协同工作
      - 把 100 个仓库分配给 3 个实例
      - 每个实例处理 33-34 个仓库

期望：任务能自动分片，多个实例并行执行
```

## yudao-cloud 的选择：XXL-Job

市面上有多种定时任务方案，yudao-cloud 选择了 XXL-Job。让我们先看看各方案的对比：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **Spring @Scheduled** | 简单、零配置 | 单机、无管理界面、无失败重试 | 开发测试、简单任务 |
| **Quartz** | 功能强大、集群支持 | 需要自己实现管理界面、配置复杂 | 需要精细控制的场景 |
| **Elastic-Job** | 分片能力强、依赖 ZooKeeper | 运维复杂、学习成本高 | 大规模分布式场景 |
| **XXL-Job** | 开箱即用、可视化管理、轻量 | 依赖数据库、Admin 单点 | 中小规模生产环境 |

### 为什么选 XXL-Job

```
XXL-Job 的核心优势：

1. 可视化管理 ──────────────────────────────────────────────
   ┌─────────────────────────────────────────┐
   │           XXL-Job Admin 控制台           │
   ├─────────────────────────────────────────┤
   │  任务管理  │  执行器管理  │  日志查询    │
   │  ────────  │  ──────────  │  ────────   │
   │  新增任务  │  在线机器    │  执行日志    │
   │  编辑任务  │  自动注册    │  调度日志    │
   │  启停任务  │  手动注册    │  失败告警    │
   └─────────────────────────────────────────┘

2. 丰富的路由策略 ──────────────────────────────────────────
   轮询  │  随机  │  一致性HASH  │  故障转移  │  分片广播
    ↓        ↓          ↓            ↓           ↓
   均匀    任意       固定节点     自动切换     并行执行

3. 故障转移 ──────────────────────────────────────────────
   执行器A（宕机）───×──→ 任务
                         ↓ 自动切换
   执行器B（正常）──────→ 任务 ✓

4. 日志追踪 ──────────────────────────────────────────────
   任务执行日志实时上报，集中存储，随时查看
```

## 整体架构

XXL-Job 采用经典的 **调度中心 + 执行器** 架构：

```
                    ┌─────────────────────────────────────┐
                    │         XXL-Job Admin (调度中心)      │
                    │         ┌───────────────┐           │
                    │         │  调度引擎      │           │
                    │         │  (CRON解析)    │           │
                    │         └───────┬───────┘           │
                    │                 │ 触发任务            │
                    │         ┌───────┴───────┐           │
                    │         │  任务管理      │           │
                    │         │  日志存储      │           │
                    │         └───────────────┘           │
                    └──────────┬──────────┬───────────────┘
                               │          │
              ┌────────────────┘          └────────────────┐
              │ HTTP/RPC 调度                               │ HTTP/RPC 调度
              ▼                                            ▼
   ┌──────────────────────┐                   ┌──────────────────────┐
   │   执行器实例 A        │                   │   执行器实例 B        │
   │  ┌────────────────┐  │                   │  ┌────────────────┐  │
   │  │ XxlJobExecutor │  │                   │  │ XxlJobExecutor │  │
   │  └───────┬────────┘  │                   │  └───────┬────────┘  │
   │          │            │                   │          │            │
   │    ┌─────┴─────┐     │                   │    ┌─────┴─────┐     │
   │    │ JobHandler│     │                   │    │ JobHandler│     │
   │    │ ┌───────┐ │     │                   │    │ ┌───────┐ │     │
   │    │ │@XxlJob│ │     │                   │    │ │@XxlJob│ │     │
   │    │ └───────┘ │     │                   │    │ └───────┘ │     │
   │    └───────────┘     │                   │    └───────────┘     │
   └──────────────────────┘                   └──────────────────────┘
           │                                           │
           │  注册                                     │  注册
           └───────────────────┬───────────────────────┘
                               ▼
                    ┌──────────────────────┐
                    │      MySQL 数据库     │
                    │  ┌────────────────┐  │
                    │  │  xxl_job_* 表  │  │
                    │  │  - 任务信息    │  │
                    │  │  - 调度日志    │  │
                    │  │  - 执行器注册  │  │
                    │  └────────────────┘  │
                    └──────────────────────┘
```

**核心流程**：

1. **Admin（调度中心）**：负责任务的管理和调度触发，基于 CRON 表达式计算下次执行时间
2. **Executor（执行器）**：负责任务的实际执行，启动时向 Admin 注册自己
3. **JobHandler（任务处理器）**：业务代码，用 `@XxlJob` 注解标记的方法

```
任务执行流程：

Admin                           Executor                      业务JobHandler
  │                                │                              │
  │  1. CRON触发，计算执行机器      │                              │
  │───────────────────────────────>│                              │
  │                                │  2. 调用对应的JobHandler      │
  │                                │─────────────────────────────>│
  │                                │                              │
  │                                │  3. 返回执行结果              │
  │                                │<─────────────────────────────│
  │  4. 记录执行日志               │                              │
  │<───────────────────────────────│                              │
  │                                │                              │
```

## 与 Spring @Scheduled 的关系

yudao-cloud 中，XXL-Job 和 Spring @Scheduled 是**共存**的，各司其职：

```
┌─────────────────────────────────────────────────────────────────┐
│                        定时任务分层                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │              XXL-Job（复杂任务）                         │  │
│   │                                                         │  │
│   │  • 需要分布式调度（只在一个实例执行）                     │  │
│   │  • 需要失败重试和告警                                    │  │
│   │  • 需要任务监控和日志追踪                                │  │
│   │  • 需要分片并行执行                                      │  │
│   │                                                         │  │
│   │  示例：                                                  │  │
│   │  - 超时订单取消（每小时）                                 │  │
│   │  - 佣金结算（每天凌晨）                                  │  │
│   │  - 库存同步（每30分钟）                                  │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │              Spring @Scheduled（简单任务）               │  │
│   │                                                         │  │
│   │  • 单机执行即可（每个实例都执行也没关系）                 │  │
│   │  • 不需要管理界面                                        │  │
│   │  • 不需要失败重试                                        │  │
│   │                                                         │  │
│   │  示例：                                                  │  │
│   │  - 缓存预热（应用启动时）                                │  │
│   │  - 本地临时文件清理（每天）                               │  │
│   │  - 健康检查心跳（每分钟）                                 │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**选型建议**：

| 特征 | Spring @Scheduled | XXL-Job |
|------|-------------------|---------|
| 是否需要只执行一次 | 否（每个实例都执行） | 是 |
| 是否需要管理界面 | 否 | 是 |
| 是否需要失败重试 | 否 | 是 |
| 是否需要执行日志 | 否 | 是 |
| 是否需要分片执行 | 否 | 是 |

简单来说：**能用 @Scheduled 解决的就用 @Scheduled，解决不了的用 XXL-Job**。

## 小结

本章从电商系统的实际场景出发，介绍了定时任务的核心需求，并解释了 yudao-cloud 为什么选择 XXL-Job 作为定时任务的解决方案。下一章我们将深入分析分布式环境下定时任务的具体挑战，以及 yudao-cloud 的设计目标。
