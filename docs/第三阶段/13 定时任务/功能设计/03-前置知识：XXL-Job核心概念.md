# 03-前置知识：XXL-Job 核心概念

## 为什么需要了解 XXL-Job 本身

yudao-cloud 的定时任务组件是基于 XXL-Job 的封装。如果不理解 XXL-Job 自身的架构和概念，后续看框架的集成代码时会很吃力——你不知道哪些是 XXL-Job 本身的能力，哪些是 yudao-cloud 额外添加的。

本章的目标：**让你对 XXL-Job 的核心概念建立清晰的心智模型**，为后续的集成设计做好铺垫。

## 整体架构：调度中心 + 执行器

XXL-Job 采用经典的中心化调度架构，由两个核心角色组成：

```
┌─────────────────────────────────────────────────────────────────┐
│                    XXL-Job Admin（调度中心）                      │
│                                                                 │
│   职责：                                                         │
│   • 管理任务的配置（Cron 表达式、路由策略、执行参数）             │
│   • 按时间规则触发任务执行                                       │
│   • 接收执行结果，记录执行日志                                   │
│   • 提供 Web 管理界面                                            │
│                                                                 │
│   本质：一个 Spring Boot 应用 + MySQL 数据库                     │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           │  HTTP 调度请求
                           │  （带路由策略选择目标执行器）
                           │
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│   执行器实例 A   │ │   执行器实例 B   │ │   执行器实例 C   │
│                 │ │                 │ │                 │
│  职责：          │ │  职责：          │ │  职责：          │
│  • 接收调度请求  │ │  • 接收调度请求  │ │  • 接收调度请求  │
│  • 执行业务代码  │ │  • 执行业务代码  │ │  • 执行业务代码  │
│  • 上报执行结果  │ │  • 上报执行结果  │ │  • 上报执行结果  │
│                 │ │                 │ │                 │
│  本质：你的应用  │ │  本质：你的应用  │ │  本质：你的应用  │
│  里嵌入的        │ │  里嵌入的        │ │  里嵌入的        │
│  xxl-job-core   │ │  xxl-job-core   │ │  xxl-job-core   │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

### 两个角色的关系

一个常见的误解是把"执行器"当成一个独立部署的服务。实际上：

```
你的电商应用 = 业务代码 + xxl-job-core（执行器）

┌──────────────────────────────────────────┐
│              电商应用实例                  │
│                                          │
│  ┌────────────────┐  ┌────────────────┐ │
│  │   业务代码      │  │  xxl-job-core  │ │
│  │  - 订单服务    │  │  (执行器)       │ │
│  │  - 商品服务    │  │                │ │
│  │  - 支付服务    │  │  负责：         │ │
│  │               │  │  - 向Admin注册  │ │
│  │               │  │  - 接收调度请求  │ │
│  │               │  │  - 调用JobHandler│ │
│  │               │  │  - 上报结果     │ │
│  └────────────────┘  └────────────────┘ │
└──────────────────────────────────────────┘
```

**执行器不是独立进程，它是嵌入在你的应用里的一个组件。** 启动时自动向 Admin 注册，之后 Admin 就知道"有哪些执行器实例可用"。

### 注册机制

执行器启动后，会通过心跳机制向 Admin 注册自己：

```
时间线：
  T0  执行器实例 A 启动
  T0  实例 A 向 Admin 注册（IP: 192.168.1.10, Port: 9999）
  T1  执行器实例 B 启动
  T1  实例 B 向 Admin 注册（IP: 192.168.1.11, Port: 9999）
  T2  Admin 知道当前有 2 个可用的执行器实例
  T3  任务触发时，Admin 根据路由策略选择其中一个实例

  ... 某天实例 A 宕机 ...
  T4  实例 A 的心跳超时
  T5  Admin 将实例 A 标记为不可用
  T6  后续任务只路由到实例 B
```

XXL-Job 支持两种注册方式：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 自动注册 | 执行器启动时自动向 Admin 注册，心跳维持 | 大多数场景，推荐 |
| 手动注册 | 在 Admin 控制台手动填写执行器的 IP 列表 | IP 固定的场景 |

yudao-cloud 默认使用自动注册，无需手动配置 IP。

## 核心概念一：执行器（Executor）

执行器是任务的"容器"，一个执行器下可以挂载多个任务。

```
执行器 "yudao-trade-server"
  ├── 任务 1: tradeOrderAutoCancelJob（超时订单取消）
  ├── 任务 2: tradeOrderAutoReceiveJob（自动确认收货）
  └── 任务 3: tradeOrderAutoCommentJob（自动好评）

执行器 "yudao-pay-server"
  ├── 任务 1: payOrderExpireJob（支付单过期）
  ├── 任务 2: payOrderSyncJob（支付单同步）
  └── 任务 3: payNotifyJob（支付回调通知）
```

在 Admin 控制台，你需要先创建执行器，然后在执行器下创建任务。执行器的 `appName` 就是它的标识，对应配置中的 `xxl.job.executor.appname`。

yudao-cloud 的做法是：**每个微服务的 appName 直接用 Spring 的应用名**：

```yaml
xxl:
  job:
    executor:
      appname: ${spring.application.name}  # 例如 yudao-trade-server
```

这样在 Admin 控制台看到的执行器名称就是服务名，一眼就知道是哪个服务。

## 核心概念二：任务（JobHandler）

任务是具体的业务逻辑。在 Java 代码中，用 `@XxlJob` 注解标记一个方法：

```java
@Component
public class TradeOrderAutoCancelJob {

    @Resource
    private TradeOrderUpdateService tradeOrderUpdateService;

    @XxlJob("tradeOrderAutoCancelJob")  // "tradeOrderAutoCancelJob" 是任务的 Handler 名称
    public String execute() {
        int count = tradeOrderUpdateService.cancelOrderBySystem();
        return String.format("过期订单 %s 个", count);
    }
}
```

这里有几个要点：

```
@XxlJob("tradeOrderAutoCancelJob")
     │              │
     │              └── Handler 名称，在 Admin 控制台创建任务时填写
     │                  必须全局唯一
     │
     └── 标记这个方法是一个 XXL-Job 的任务处理器

方法签名：
  - 可以无参数：public void execute()
  - 可以有参数：public String execute(String param)
    → param 由 Admin 控制台的任务参数传入
  - 返回值会作为执行结果记录到日志
```

### 任务的执行模型

一个常见的疑问：当 Admin 触发任务时，执行器是怎么找到对应的 JobHandler 的？

```
Admin 发送调度请求
  → 携带 Handler 名称："tradeOrderAutoCancelJob"
  → 到达执行器实例 A

执行器内部：
  ┌─────────────────────────────────────────┐
  │  Handler 注册表（Map<String, Method>）   │
  │                                         │
  │  "tradeOrderAutoCancelJob" → execute()  │
  │  "payOrderExpireJob"       → execute()  │
  │  "tokenCleanJob"           → execute()  │
  └─────────────────────────────────────────┘
         │
         │ 查找 "tradeOrderAutoCancelJob"
         ▼
  反射调用对应的 Java 方法
```

`@XxlJob` 注解在应用启动时会被扫描，注册到一个内部的 Map 中。调度请求到达时，通过 Handler 名称找到对应的方法并执行。

## 核心概念三：路由策略

当一个执行器有多个实例时，Admin 需要决定"这次调度发给谁"。这就是路由策略的作用。

```
执行器 "yudao-trade-server" 有 3 个实例：
  实例 A (192.168.1.10)
  实例 B (192.168.1.11)
  实例 C (192.168.1.12)

任务 "tradeOrderAutoCancelJob" 触发了
  → 发给谁？
```

XXL-Job 提供了多种路由策略：

```
┌─────────────┬─────────────────────────────────────────────────────┐
│ 策略         │ 行为                                                │
├─────────────┼─────────────────────────────────────────────────────┤
│ FIRST       │ 固定选择第一个实例（注册顺序）                        │
│ ROUND       │ 轮询：第1次→A，第2次→B，第3次→C，第4次→A...         │
│ RANDOM      │ 随机选择一个实例                                      │
│ CONSISTENT_HASH │ 一致性HASH，相同参数的任务总是路由到同一实例      │
│ LEAST_FREQUENTLY_USED │ 最不经常使用：选择历史上被调用次数最少的    │
│ LEAST_RECENTLY_USED   │ 最近最久未使用：选择最久没被调度的实例      │
│ FAILOVER    │ 故障转移：按顺序尝试，第一个失败就试下一个             │
│ BUSYOVER    │ 忙碌转移：跳过正在执行任务的实例                       │
│ SHARDING_BROADCAST │ 分片广播：所有实例都执行，每个实例拿到不同的分片│
└─────────────┴─────────────────────────────────────────────────────┘
```

最常用的三种：

```
1. 轮询（ROUND）── 最简单，均匀分配
   第1次 → A  第2次 → B  第3次 → C  第4次 → A ...

2. 故障转移（FAILOVER）── 最可靠，自动切换
   尝试 A → 失败 → 尝试 B → 失败 → 尝试 C → 成功

3. 分片广播（SHARDING_BROADCAST）── 适合大数据量并行
   所有实例同时执行，每个实例拿到自己的分片参数：
   实例 A: 分片号=0, 总分片数=3 → 处理第 0, 3, 6, 9... 条
   实例 B: 分片号=1, 总分片数=3 → 处理第 1, 4, 7, 10... 条
   实例 C: 分片号=2, 总分片数=3 → 处理第 2, 5, 8, 11... 条
```

yudao-cloud 中大多数任务使用默认策略（第一个注册的实例），因为定时任务通常不需要负载均衡——只需要一个实例执行就够了。

## 核心概念四：Cron 表达式

Cron 表达式是定时任务的"时间规则"。XXL-Job 使用 6 位 Cron 表达式（比 Linux Crontab 多一位，支持到秒）。

```
格式：秒 分 时 日 月 周

┌─────── 秒 (0-59)
│ ┌───── 分 (0-59)
│ │ ┌─── 时 (0-23)
│ │ │ ┌─ 日 (1-31)
│ │ │ │ ┌ 月 (1-12)
│ │ │ │ │ ┌ 周 (0-7, 0和7都是周日)
│ │ │ │ │ │
* * * * * *
```

电商系统中常见的 Cron 表达式：

```
"0 0 * * * ?"        → 每小时整点
                       用途：检查超时未支付的订单

"0 0 2 * * ?"        → 每天凌晨 2 点
                       用途：佣金结算、数据统计

"0 */30 * * * ?"     → 每 30 分钟
                       用途：库存同步

"0 0 0 1 * ?"        → 每月 1 号零点
                       用途：生成月度报表

"0 0 8 ? * 1"        → 每周一早上 8 点
                       用途：会员等级降级

"0 0/5 * * * ?"      → 每 5 分钟
                       用途：支付单过期检查
```

特殊字符说明：

```
*  任意值          用在"秒"位置：每秒都触发
?  不指定          用在"日"和"周"位置，表示"不关心"
/  递增            0/15 从0开始每隔15
-  范围            1-5 表示 1 到 5
,  列举            1,3,5 表示 1、3、5
L  最后            L 用在"日"位置：每月最后一天
W  最近工作日      15W：离15号最近的工作日
#  第几个星期几    6#3：第三个周五
```

### XXL-Job 的 Cron 与 Spring @Scheduled 的区别

```
Spring @Scheduled：
  @Scheduled(cron = "0 0 2 * * ?")  ← 6位，支持秒

XXL-Job Admin 控制台：
  在任务配置页面填写 Cron 表达式 → 由 Admin 解析和触发

两者格式相同，但触发机制完全不同：
  @Scheduled → 由 Spring 的 TaskScheduler 在本地触发
  XXL-Job    → 由 Admin 的调度引擎在中心节点触发，通过 HTTP 发给执行器
```

## 核心概念五：执行日志与回调

XXL-Job 有一套完整的日志上报机制：

```
执行器执行任务
    │
    ├── 1. 执行前：记录"开始执行"日志
    │
    ├── 2. 执行中：业务代码通过 XxlJobHelper.log() 输出日志
    │         XxlJobHelper.log("处理订单数量：{}", count);
    │         → 这些日志会实时上报到 Admin
    │
    ├── 3. 执行后：
    │         成功 → XxlJobHelper.handleSuccess("处理完成")
    │         失败 → XxlJobHelper.handleFail("处理失败：xxx")
    │
    └── 4. 回调 Admin：将执行结果（成功/失败 + 日志内容）发回 Admin
              → Admin 存入数据库
              → 在控制台可以查看
```

在 Admin 控制台的"调度日志"页面，你可以看到：

```
┌──────────────────────────────────────────────────────────────────┐
│ 调度日志                                                          │
├──────┬────────────┬──────┬──────┬─────────────────────────────────┤
│ ID   │ 触发时间    │ 状态 │ 耗时 │ 执行结果                        │
├──────┼────────────┼──────┼──────┼─────────────────────────────────┤
│ 1001 │ 02:00:00   │ 成功 │ 25s  │ 处理订单数量：156               │
│ 1000 │ 01:00:00   │ 成功 │ 18s  │ 处理订单数量：89                │
│ 999  │ 00:00:00   │ 失败 │ 5s   │ 数据库连接超时                  │
│ 998  │ 23:00:00   │ 成功 │ 22s  │ 处理订单数量：203               │
└──────┴────────────┴──────┴──────┴─────────────────────────────────┘
```

## 核心概念六：任务阻塞策略

当一个任务还没执行完，下一次调度又来了，怎么办？

```
场景：任务 A 每 5 分钟执行一次，但这次执行花了 8 分钟

  00:00  触发 → 开始执行
  00:05  触发 → 但上次还没执行完！怎么办？
  00:08  上次执行完成
  00:10  触发 → 正常执行
```

XXL-Job 提供了三种阻塞策略：

```
┌──────────────────┬──────────────────────────────────────────────┐
│ 策略              │ 行为                                         │
├──────────────────┼──────────────────────────────────────────────┤
│ SERIAL_EXECUTION │ 串行执行：等上一次执行完再执行下一次          │
│                  │ → 00:05 的调度排队等，00:08 后才开始执行       │
│                  │ → 不会丢任务，但可能延迟                       │
├──────────────────┼──────────────────────────────────────────────┤
│ DISCARD_LATER    │ 丢弃后来的：如果上次没执行完，直接丢弃本次    │
│                  │ → 00:05 的调度被丢弃                           │
│                  │ → 不会堆积，但可能丢任务                       │
├──────────────────┼──────────────────────────────────────────────┤
│ COVER_EARLY      │ 覆盖之前的：终止上一次执行，立即执行本次      │
│                  │ → 00:00 的执行被中断，00:05 的立即开始         │
│                  │ → 适用于"只需要最新结果"的场景                 │
└──────────────────┴──────────────────────────────────────────────┘
```

电商系统中，大多数定时任务选择 `SERIAL_EXECUTION`（串行），因为数据处理类任务不能丢。

## 核心概念七：失败重试与告警

XXL-Job 支持配置任务失败后的自动重试次数：

```
任务配置：重试次数 = 3

第 1 次执行：失败 → 重试
第 2 次执行：失败 → 重试
第 3 次执行：失败 → 重试
第 4 次执行：失败 → 标记为最终失败，触发告警
```

告警方式支持邮件通知（XXL-Job 内置），也可以扩展其他方式。

## 整体流程回顾

把所有概念串起来，一个任务从配置到执行的完整流程：

```
步骤 1：在 Admin 控制台创建执行器
        AppName: yudao-trade-server
        注册方式: 自动注册

步骤 2：在 Admin 控制台创建任务
        Handler: tradeOrderAutoCancelJob
        Cron: 0 0 * * * ?（每小时执行）
        路由策略: 轮询
        阻塞策略: 串行执行
        重试次数: 1

步骤 3：在业务代码中编写 JobHandler
        @XxlJob("tradeOrderAutoCancelJob")
        public String execute() { ... }

步骤 4：应用启动
        → xxl-job-core 初始化
        → 向 Admin 注册执行器实例
        → 注册所有 @XxlJob 方法到 Handler 映射表

步骤 5：到了触发时间
        → Admin 的调度引擎检测到该执行了
        → 根据路由策略选择一个执行器实例
        → 发送 HTTP 请求（携带 Handler 名称）
        → 执行器找到对应的 JobHandler 方法
        → 执行业务逻辑
        → 将执行结果回调给 Admin
        → Admin 记录日志
```

## 小结

本章介绍了 XXL-Job 的七个核心概念：执行器、任务（JobHandler）、路由策略、Cron 表达式、执行日志与回调、阻塞策略、失败重试与告警。这些是理解 yudao-cloud 集成设计的前置知识。
