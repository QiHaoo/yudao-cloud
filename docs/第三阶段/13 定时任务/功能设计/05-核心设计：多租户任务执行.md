# 05-核心设计：多租户任务执行

## 问题回顾

上一章我们实现了 XXL-Job 的基础集成，业务代码可以用 `@XxlJob` 注解定义定时任务。但在 SaaS 多租户场景下，还有一个关键问题没有解决。

假设你的电商 SaaS 平台有 50 个租户（50 家独立店铺），每家店铺都有"超时订单自动取消"的需求。如果按常规思路：

```
方案 A：每个租户一个任务
  任务 1: cancelOrder_tenant_1  → Cron: 0 0 * * * ?
  任务 2: cancelOrder_tenant_2  → Cron: 0 0 * * * ?
  任务 3: cancelOrder_tenant_3  → Cron: 0 0 * * * ?
  ...
  任务 50: cancelOrder_tenant_50 → Cron: 0 0 * * * ?

问题：
  • 50 个租户 = 50 个任务配置，维护噩梦
  • 新增租户？→ 去 Admin 控制台手动加一个任务
  • 删除租户？→ 去 Admin 控制台手动删一个任务，忘了就报错
  • 任务代码里要写 tenantId 硬编码
```

```
方案 B：一个任务，手动遍历租户

  @XxlJob("cancelOrder")
  public void execute() {
      List<Long> tenantIds = tenantService.getTenantIds();
      for (Long tenantId : tenantIds) {
          TenantContextHolder.setTenantId(tenantId);
          try {
              tradeOrderUpdateService.cancelOrderBySystem();
          } finally {
              TenantContextHolder.clear();
          }
      }
  }

问题：
  • 每个 JobHandler 都要写一遍遍历逻辑（重复代码）
  • 容易忘记清理 TenantContextHolder（内存泄漏）
  • 容易忘记 try-finally（异常时租户上下文错乱）
```

yudao-cloud 的方案是：**一个 `@TenantJob` 注解，自动完成租户遍历**。

## @TenantJob 注解设计

使用方式极其简单：

```java
@Component
public class TradeOrderAutoCancelJob {

    @Resource
    private TradeOrderUpdateService tradeOrderUpdateService;

    @XxlJob("tradeOrderAutoCancelJob")
    @TenantJob  // ← 加上这个注解，自动遍历所有租户
    public String execute() {
        int count = tradeOrderUpdateService.cancelOrderBySystem();
        return String.format("过期订单 %s 个", count);
    }
}
```

对比没有 `@TenantJob` 的情况：

```java
// 没有 @TenantJob：每个实例只执行一次，不区分租户
@XxlJob("accessLogCleanJob")
@TenantIgnore  // 明确标记：这个任务忽略租户
public void execute() {
    apiAccessLogService.cleanAccessLog(14, 100);
}
```

两种模式的对比：

```
┌─────────────────────────────────────────────────────────────────┐
│  模式 1：@TenantJob（多租户任务）                                │
│                                                                 │
│  任务触发                                                       │
│    ↓                                                           │
│  获取所有租户列表 [1, 2, 3, ..., 50]                            │
│    ↓                                                           │
│  遍历每个租户：                                                  │
│    租户 1 → 设置上下文 → 执行业务逻辑 → 清理上下文              │
│    租户 2 → 设置上下文 → 执行业务逻辑 → 清理上下文              │
│    ...                                                          │
│    租户 50 → 设置上下文 → 执行业务逻辑 → 清理上下文             │
│    ↓                                                           │
│  汇总结果，上报 Admin                                           │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│  模式 2：无注解 / @TenantIgnore（非租户任务）                    │
│                                                                 │
│  任务触发                                                       │
│    ↓                                                           │
│  直接执行业务逻辑（不设置租户上下文）                            │
│    ↓                                                           │
│  上报 Admin                                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 核心实现：TenantJobAspect

`@TenantJob` 的魔法来自 AOP 切面 `TenantJobAspect`。它是整个多租户任务执行的核心：

```java
@Aspect
@RequiredArgsConstructor
@Slf4j
public class TenantJobAspect {

    private final TenantFrameworkService tenantFrameworkService;

    @Around("@annotation(tenantJob)")
    public void around(ProceedingJoinPoint joinPoint, TenantJob tenantJob) {
        // 1. 获取所有租户
        List<Long> tenantIds = tenantFrameworkService.getTenantIds();
        if (CollUtil.isEmpty(tenantIds)) {
            return;
        }

        // 2. 逐个租户执行
        Map<Long, String> results = new ConcurrentHashMap<>();
        AtomicBoolean success = new AtomicBoolean(true);
        XxlJobContext xxlJobContext = XxlJobContext.getXxlJobContext();

        tenantIds.parallelStream().forEach(tenantId -> {
            TenantUtils.execute(tenantId, () -> {
                try {
                    XxlJobContext.setXxlJobContext(xxlJobContext);
                    Object result = joinPoint.proceed();
                    results.put(tenantId, StrUtil.toStringOrEmpty(result));
                } catch (Throwable e) {
                    results.put(tenantId, ExceptionUtil.getRootCauseMessage(e));
                    success.set(false);
                    XxlJobHelper.log(StrUtil.format(
                        "[多租户({}) 执行任务({})，发生异常：{}]",
                        tenantId, joinPoint.getSignature(),
                        ExceptionUtils.getStackTrace(e)));
                }
            });
        });

        // 3. 汇总结果
        if (success.get()) {
            XxlJobHelper.handleSuccess(JsonUtils.toJsonString(results));
        } else {
            XxlJobHelper.handleFail(JsonUtils.toJsonString(results));
        }
    }
}
```

让我们逐段拆解这个切面的设计。

### 第一步：获取租户列表

```java
List<Long> tenantIds = tenantFrameworkService.getTenantIds();
```

`TenantFrameworkService` 是一个接口，由业务层实现。它的职责很简单：

```java
public interface TenantFrameworkService {
    List<Long> getTenantIds();   // 获取所有启用的租户 ID
    void validTenant(Long id);   // 校验租户是否合法
}
```

实际实现中，`getTenantIds()` 会查询数据库中所有启用状态的租户。这意味着：

```
新增租户 → 数据库插入一条租户记录 → 自动被覆盖，无需改任务配置
删除租户 → 数据库标记为禁用 → 自动跳过，无需改任务配置
```

### 第二步：逐个租户执行

```java
tenantIds.parallelStream().forEach(tenantId -> {
    TenantUtils.execute(tenantId, () -> {
        // 执行业务逻辑
    });
});
```

`TenantUtils.execute()` 做了什么？

```java
public static void execute(Long tenantId, Runnable runnable) {
    Long oldTenantId = TenantContextHolder.getTenantId();
    Boolean oldIgnore = TenantContextHolder.isIgnore();
    try {
        TenantContextHolder.setTenantId(tenantId);  // 设置当前租户
        TenantContextHolder.setIgnore(false);         // 确保不忽略租户
        runnable.run();                                // 执行业务逻辑
    } finally {
        TenantContextHolder.setTenantId(oldTenantId); // 恢复原租户
        TenantContextHolder.setIgnore(oldIgnore);      // 恢复原忽略状态
    }
}
```

这段代码的设计要点：

```
1. 设置租户上下文
   TenantContextHolder.setTenantId(tenantId)
   → MyBatis 拦截器会自动追加 WHERE tenant_id = ?
   → 缓存操作会自动加租户前缀
   → 业务代码完全不需要感知租户

2. 确保不忽略租户
   TenantContextHolder.setIgnore(false)
   → 防止在 @TenantIgnore 的上下文中调用 @TenantJob 方法时出问题

3. finally 恢复上下文
   → 无论执行成功还是失败，都恢复原来的租户上下文
   → 防止租户上下文"泄漏"到下一个租户的执行中
```

`parallelStream()` 的含义：

```
parallelStream → 使用 ForkJoinPool 并行执行
  租户 1 ──→ ┐
  租户 2 ──→ ├──→ 并行执行（线程池）
  租户 3 ──→ ┘
  ...
  租户 50 ─→ ┘

vs.

普通 stream → 串行执行
  租户 1 → 租户 2 → 租户 3 → ... → 租户 50
```

并行执行的好处是：当租户数量多时，可以显著缩短总执行时间。但需要注意：

```
使用 parallelStream 的前提：
  1. 业务逻辑是幂等的（重试不会产生副作用）
  2. 租户之间没有执行顺序的依赖
  3. 数据库连接池足够大（并行执行需要更多连接）
```

### 第三步：XXL-Job 上下文传递

```java
XxlJobContext xxlJobContext = XxlJobContext.getXxlJobContext();

// 在并行流中：
XxlJobContext.setXxlJobContext(xxlJobContext);
```

为什么要手动传递 `XxlJobContext`？

```
问题：parallelStream 使用 ForkJoinPool 的线程，不是主线程
      XXL-Job 的上下文存储在 ThreadLocal 中
      子线程拿不到主线程的 ThreadLocal

解决：在并行流开始前，捕获主线程的 XxlJobContext
      在每个子线程中，手动设置 XxlJobContext
      这样 XxlJobHelper.log() 才能正常工作
```

```
主线程：XxlJobContext = {任务ID=1001, 执行参数=...}
  │
  │ parallelStream
  ▼
子线程（无上下文）：XxlJobContext = null
  → XxlJobHelper.log() 会报错！

解决后：
子线程（手动设置）：XxlJobContext = 主线程的上下文
  → XxlJobHelper.log() 正常工作
```

### 第四步：结果收集与上报

```java
Map<Long, String> results = new ConcurrentHashMap<>();
AtomicBoolean success = new AtomicBoolean(true);

// 每个租户执行后：
results.put(tenantId, StrUtil.toStringOrEmpty(result));  // 成功
results.put(tenantId, ExceptionUtil.getRootCauseMessage(e));  // 失败

// 最终上报：
if (success.get()) {
    XxlJobHelper.handleSuccess(JsonUtils.toJsonString(results));
} else {
    XxlJobHelper.handleFail(JsonUtils.toJsonString(results));
}
```

结果在 Admin 控制台的展示效果：

```
成功的情况：
  {"1":"过期订单 5 个","2":"过期订单 3 个","3":"过期订单 8 个",...}

失败的情况（租户 5 执行报错）：
  {"1":"过期订单 5 个","2":"过期订单 3 个","5":"数据库连接超时",...}
  → 整体标记为失败，Admin 可以触发重试
```

注意 `ConcurrentHashMap` 和 `AtomicBoolean` 的使用——因为是并行流，多个线程同时写入结果，必须用线程安全的数据结构。

## 幂等性要求

源码注释中有一段重要提醒：

```java
/**
 * 注意，需要保证 JobHandler 的幂等性。
 * 因为 Job 因为某个租户执行失败重试时，之前执行成功的租户也会再次执行。
 */
```

这是什么意思？

```
场景：50 个租户，第 37 个租户执行失败

  租户 1  → 成功
  租户 2  → 成功
  ...
  租户 36 → 成功
  租户 37 → 失败（数据库超时）
  租户 38 → 成功
  ...
  租户 50 → 成功

  整体结果：失败
  → Admin 触发重试
  → 所有 50 个租户重新执行一遍（包括之前成功的 1-36, 38-50）
```

因此，使用 `@TenantJob` 的任务必须是幂等的：

```
幂等的业务逻辑：
  • "取消超时订单" → 已取消的订单不会被重复取消（状态判断）
  • "统计昨日销售" → 重新统计会覆盖之前的结果（覆盖写入）
  • "清理过期数据" → 已删除的数据不会报错（DELETE 无副作用）

非幂等的业务逻辑（不适合 @TenantJob）：
  • "发送优惠券" → 重试会重复发送
  • "累加积分" → 重试会多加
  • "扣减库存" → 重试会多扣
```

## 注册机制：条件装配

`TenantJobAspect` 不在 `yudao-spring-boot-starter-job` 中定义，而是在 `yudao-spring-boot-starter-biz-tenant` 中：

```java
// 在 YudaoTenantAutoConfiguration 中
@Bean
@ConditionalOnClass(name = "com.xxl.job.core.handler.annotation.XxlJob")
public TenantJobAspect tenantJobAspect(TenantFrameworkService tenantFrameworkService) {
    return new TenantJobAspect(tenantFrameworkService);
}
```

条件注解的含义：

```
@ConditionalOnClass(name = "com.xxl.job.core.handler.annotation.XxlJob")
│
│  只有当 xxl-job-core 存在时才创建 TenantJobAspect
│
│  设计意图：
│  多租户组件（yudao-spring-boot-starter-biz-tenant）和定时任务组件
│  （yudao-spring-boot-starter-job）是独立的，可以单独使用
│  只有两者同时存在时，@TenantJob 才生效
```

模块依赖关系：

```
yudao-spring-boot-starter-job（定时任务 Starter）
  └── 提供：XxlJob 自动配置、XxlJobProperties

yudao-spring-boot-starter-biz-tenant（多租户 Starter）
  └── 提供：@TenantJob、TenantJobAspect、租户上下文管理
  └── 条件依赖：如果 xxl-job-core 存在，创建 TenantJobAspect

业务模块同时引入两个 Starter：
  └── 获得：XXL-Job 自动配置 + @TenantJob 支持
```

这种设计体现了**正交组合**的思想：定时任务和多租户是两个独立的关注点，通过条件装配按需组合。

## 三种任务模式的对比

在 yudao-cloud 中，一个 JobHandler 有三种使用模式：

```
模式 1：普通任务（无注解）

  @XxlJob("tokenCleanJob")
  public void execute() {
      tokenService.cleanExpiredTokens();
  }

  行为：执行一次，不区分租户
  适用：Token 清理、日志清理等全局操作


模式 2：多租户任务（@TenantJob）

  @XxlJob("tradeOrderAutoCancelJob")
  @TenantJob
  public String execute() {
      int count = tradeOrderUpdateService.cancelOrderBySystem();
      return String.format("过期订单 %s 个", count);
  }

  行为：遍历所有租户执行
  适用：订单取消、佣金结算、数据统计等租户级操作


模式 3：忽略租户任务（@TenantIgnore）

  @XxlJob("accessLogCleanJob")
  @TenantIgnore
  public void execute() {
      apiAccessLogService.cleanAccessLog(14, 100);
  }

  行为：明确标记忽略租户，即使在租户上下文中也强制忽略
  适用：系统级操作，与具体租户无关
```

选型决策树：

```
这个任务需要区分不同租户的数据吗？
  │
  ├── 不需要 → 用 @TenantIgnore 或不加注解
  │            例如：清理全局日志、刷新全局缓存
  │
  └── 需要 → 这个任务的逻辑是幂等的吗？
              │
              ├── 是 → 用 @TenantJob
              │        例如：取消过期订单、统计昨日数据
              │
              └── 否 → 不能用 @TenantJob
                       需要自己实现带幂等保护的遍历逻辑
                       例如：发送优惠券（需要记录已发送状态）
```

## 与异步任务的协作

回顾上一章提到的 TTL 支持。在 `@TenantJob` 的执行过程中，如果业务代码使用了 `@Async` 异步调用：

```java
@XxlJob("someJob")
@TenantJob
public void execute() {
    // 当前在租户 A 的上下文中
    someAsyncService.doSomething();  // @Async 异步调用
    // 异步线程能否继承租户 A 的上下文？
}
```

没有 TTL 支持时：

```
主线程：TenantContextHolder = 租户 A
  │
  │ @Async
  ▼
异步线程：TenantContextHolder = null  ← 上下文丢失！
  → 查询数据库时没有 WHERE tenant_id = ?
  → 可能查到所有租户的数据（数据泄露）
```

有 TTL 支持时（`YudaoAsyncAutoConfiguration` 自动配置）：

```
主线程：TenantContextHolder = 租户 A
  │
  │ @Async（TtlRunnable 包装）
  ▼
异步线程：TenantContextHolder = 租户 A  ← TTL 自动传递
  → 查询数据库时正确追加 WHERE tenant_id = A
```

## 小结

本章深入分析了 `@TenantJob` 的设计与实现：

- **注解设计**：一个注解解决多租户遍历，消除重复代码
- **AOP 实现**：`TenantJobAspect` 通过 `@Around` 切面，在任务执行前自动获取租户列表，逐个设置上下文执行
- **并行执行**：使用 `parallelStream` 加速多租户执行
- **上下文传递**：手动传递 `XxlJobContext` 到子线程，保证 `XxlJobHelper` 正常工作
- **幂等要求**：重试时所有租户会重新执行，业务逻辑必须幂等
- **正交组合**：定时任务组件和多租户组件独立，通过条件装配按需组合
