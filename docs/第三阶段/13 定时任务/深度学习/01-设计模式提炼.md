# 01-设计模式提炼

本文从定时任务组件中识别出三种核心设计模式，将它们从具体实现中抽象为通用知识，帮助读者在其他项目中识别和应用相同的设计手法。

---

## 模式一：自动配置模式（ConditionalOnClass / ConditionalOnProperty）

### 在本项目中的体现

`YudaoXxlJobAutoConfiguration` 通过一系列条件注解，实现了 XXL-Job 的零配置集成。业务模块只需要引入依赖和编写配置 YAML，框架自动完成执行器初始化、Handler 注册等繁琐工作。

```java
@AutoConfiguration
@ConditionalOnClass(XxlJobSpringExecutor.class)
@ConditionalOnProperty(prefix = "xxl.job", name = "enabled",
                       havingValue = "true", matchIfMissing = true)
@EnableConfigurationProperties({XxlJobProperties.class})
@EnableScheduling
public class YudaoXxlJobAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public XxlJobExecutor xxlJobExecutor(XxlJobProperties properties) {
        XxlJobSpringExecutor xxlJobExecutor = new XxlJobSpringExecutor();
        xxlJobExecutor.setAdminAddresses(properties.getAdmin().getAddresses());
        xxlJobExecutor.setAppname(properties.getExecutor().getAppName());
        xxlJobExecutor.setLogPath(properties.getExecutor().getLogPath());
        xxlJobExecutor.setAccessToken(properties.getAccessToken());
        return xxlJobExecutor;
    }
}
```

### 条件注解的设计意图

每个条件注解都解决一个具体的装配问题：

```
@ConditionalOnClass(XxlJobSpringExecutor.class)
│
│  问题：业务模块没有引入 xxl-job-core 依赖怎么办？
│  解决：classpath 中不存在 XxlJobSpringExecutor 时，整个配置类跳过
│  效果：不引入依赖 = 不启用 XXL-Job，零报错
│
@ConditionalOnProperty(prefix = "xxl.job", name = "enabled",
                       havingValue = "true", matchIfMissing = true)
│
│  问题：本地开发时不想启动 XXL-Job 怎么办？
│  解决：配置 xxl.job.enabled=false 时，配置类跳过
│  效果：不配置时默认启用（matchIfMissing=true）
│
@EnableConfigurationProperties({XxlJobProperties.class})
│
│  问题：YAML 配置怎么映射到 Java 对象？
│  解决：自动注册 XxlJobProperties Bean，绑定 xxl.job.* 配置
│  效果：配置有结构、有校验、有 IDE 提示
│
@EnableScheduling
│
│  问题：@Scheduled 和 @XxlJob 能共存吗？
│  解决：开启 Spring 的 @Scheduled 支持
│  效果：两种定时任务互不干扰
```

### 通用化

自动配置模式的核心思想是：**将框架的初始化逻辑封装在 Starter 中，通过条件注解按需装配，让业务模块"零配置"使用框架能力**。

```
┌─────────────────────────────────────────────────────────────┐
│                    自动配置模式                                │
│                                                             │
│  传统方式：                                                   │
│    手动编写 @Configuration 类 → 每个项目复制一遍              │
│                                                             │
│  自动配置方式：                                               │
│    编写一次 @AutoConfiguration 类 → 放在 Starter 里          │
│    → 所有引入依赖的项目自动生效                               │
│    → 通过条件注解控制生效条件                                 │
│    → 通过 @ConditionalOnMissingBean 允许覆盖                 │
└─────────────────────────────────────────────────────────────┘
```

**条件装配的三层保障**：

| 层次 | 条件注解 | 解决的问题 |
|------|---------|-----------|
| 依赖层 | `@ConditionalOnClass` | 依赖不存在时跳过 |
| 配置层 | `@ConditionalOnProperty` | 配置不满足时跳过 |
| 覆盖层 | `@ConditionalOnMissingBean` | 用户自定义时退让 |

**适用场景**：

| 场景 | 条件 | 示例 |
|------|------|------|
| 可选组件 | `@ConditionalOnClass` | XXL-Job、Redis、RocketMQ |
| 功能开关 | `@ConditionalOnProperty` | 多租户开关、调试模式 |
| 默认实现 | `@ConditionalOnMissingBean` | 默认的 XxlJobExecutor |
| 按需加载 | `@ConditionalOnBean` | 只有存在 Stream 监听器才创建容器 |

### 与 yudao-cloud 其他组件的对比

同样的自动配置模式在项目中广泛使用：

| 组件 | 配置类 | 关键条件注解 |
|------|--------|------------|
| 多租户 | YudaoTenantAutoConfiguration | `@ConditionalOnProperty(yudao.tenant.enable)` |
| 安全认证 | YudaoSecurityAutoConfiguration | 无条件（核心组件） |
| Redis MQ | YudaoRedisMQConsumerAutoConfiguration | `@ConditionalOnBean(AbstractRedisXxxMessageListener.class)` |
| RabbitMQ | YudaoRabbitMQAutoConfiguration | `@ConditionalOnClass(RabbitTemplate.class)` |
| 定时任务 | YudaoXxlJobAutoConfiguration | `@ConditionalOnClass` + `@ConditionalOnProperty` |

## 模式二：AOP 模式（@TenantJob）

### 在本项目中的体现

`@TenantJob` 注解配合 `TenantJobAspect` 切面，为定时任务自动添加多租户遍历能力。一个注解消除了每个 JobHandler 中重复的租户遍历代码。

```java
// 使用方：只需一个注解
@XxlJob("tradeOrderAutoCancelJob")
@TenantJob
public String execute() {
    int count = tradeOrderUpdateService.cancelOrderBySystem();
    return String.format("过期订单 %s 个", count);
}

// 切面实现：自动遍历所有租户
@Aspect
public class TenantJobAspect {

    @Around("@annotation(tenantJob)")
    public void around(ProceedingJoinPoint joinPoint, TenantJob tenantJob) {
        List<Long> tenantIds = tenantFrameworkService.getTenantIds();
        if (CollUtil.isEmpty(tenantIds)) {
            return;
        }

        Map<Long, String> results = new ConcurrentHashMap<>();
        AtomicBoolean success = new AtomicBoolean(true);
        XxlJobContext xxlJobContext = XxlJobContext.getXxlJobContext();

        tenantIds.parallelStream().forEach(tenantId -> {
            TenantUtils.execute(tenantId, () -> {
                try {
                    XxlJobContext.setXxlJobContext(xxlJobContext);
                    Object result = joinPoint.proceed();
                    results.put(tenantId, StrUtil.toStringOrEmpty(result));
                } catch (Throwable e) {
                    results.put(tenantId, ExceptionUtil.getRootCauseMessage(e));
                    success.set(false);
                }
            });
        });

        if (success.get()) {
            XxlJobHelper.handleSuccess(JsonUtils.toJsonString(results));
        } else {
            XxlJobHelper.handleFail(JsonUtils.toJsonString(results));
        }
    }
}
```

### 通用化

AOP 模式的核心思想是：**将横切关注点（如多租户、日志、事务）从业务代码中分离出来，通过切面在运行时织入**。

```
┌─────────────────────────────────────────────────────────────┐
│                    AOP 模式                                   │
│                                                             │
│  业务代码（只关注核心逻辑）：                                   │
│    @XxlJob("cancelOrder")                                   │
│    @TenantJob  ← 声明需要多租户支持                           │
│    public void execute() { cancelOrders(); }                │
│                                                             │
│  切面（处理横切关注点）：                                      │
│    @Around("@annotation(tenantJob)")                        │
│    public void around(...) {                                │
│        遍历租户 → 设置上下文 → 执行业务 → 清理上下文           │
│    }                                                        │
│                                                             │
│  效果：业务代码零感知，多租户逻辑自动生效                       │
└─────────────────────────────────────────────────────────────┘
```

**@TenantJob 切面的特殊之处**：

与普通的 AOP 切面（如 `@Transactional`、`@Cacheable`）不同，`@TenantJob` 切面会**多次调用** `joinPoint.proceed()`——每个租户调用一次。这是"一个切面对应多次执行"的特殊模式。

```
普通 AOP 切面：
  @Around → 前置处理 → joinPoint.proceed()（一次）→ 后置处理

@TenantJob 切面：
  @Around → 获取租户列表
         → 租户1: TenantUtils.execute(1, () -> joinPoint.proceed())
         → 租户2: TenantUtils.execute(2, () -> joinPoint.proceed())
         → ...
         → 租户N: TenantUtils.execute(N, () -> joinPoint.proceed())
         → 汇总结果
```

### 条件装配：正交组合

`TenantJobAspect` 的注册体现了**正交组合**的设计思想——定时任务和多租户是两个独立的关注点，通过条件装配按需组合：

```java
// 在 YudaoTenantAutoConfiguration 中
@Bean
@ConditionalOnClass(name = "com.xxl.job.core.handler.annotation.XxlJob")
public TenantJobAspect tenantJobAspect(TenantFrameworkService tenantFrameworkService) {
    return new TenantJobAspect(tenantFrameworkService);
}
```

```
组合矩阵：
  ┌─────────────────┬─────────────────┬─────────────────────┐
  │                 │ 有 xxl-job-core │ 无 xxl-job-core     │
  ├─────────────────┼─────────────────┼─────────────────────┤
  │ 有 tenant starter│ @TenantJob 生效 │ @TenantJob 不生效   │
  │                 │ TenantJobAspect │ TenantJobAspect 不创建│
  ├─────────────────┼─────────────────┼─────────────────────┤
  │ 无 tenant starter│ 无多租户支持    │ 两者都不存在         │
  └─────────────────┴─────────────────┴─────────────────────┘
```

### 设计要点

1. **注解驱动**：一个 `@TenantJob` 注解替代了几十行重复的遍历代码
2. **幂等性要求**：切面的重试机制要求业务逻辑必须幂等
3. **并行执行**：`parallelStream()` 加速多租户执行，但要求租户间无依赖
4. **上下文传递**：手动传递 `XxlJobContext` 到子线程，保证 `XxlJobHelper` 正常工作

## 模式三：模板方法模式（任务执行器）

### 在本项目中的体现

yudao-cloud 的 XXL-Job 集成本质上也是一种模板方法模式。`XxlJobSpringExecutor` 定义了任务执行的固定流程（接收调度请求 → 查找 Handler → 执行 → 上报结果），业务开发者只需实现 `@XxlJob` 方法。

```
XxlJobSpringExecutor（父类，定义骨架）：
  1. 启动时向 Admin 注册
  2. 启动内置 HTTP Server 监听调度请求
  3. 收到调度请求 → 查找 Handler → 反射调用
  4. 收集执行结果 → 回调 Admin

@XxlJob 方法（子类，填充细节）：
  具体的业务逻辑
```

`XxlJobSpringExecutor` 实现了 Spring 的 `SmartInitializingSingleton` 接口，在所有单例 Bean 初始化完成后自动扫描 `@XxlJob` 方法：

```
Spring 容器启动流程：

  1. 创建所有 Bean（包括业务组件、JobHandler 类等）
  2. 所有单例 Bean 创建完成
  3. XxlJobSpringExecutor.afterSingletonsInstantiated() 被调用
  4. 扫描所有 Bean 中的 @XxlJob 方法
  5. 注册到 Handler 映射表
  6. 启动内置 HTTP Server
  7. 向 Admin 注册执行器
```

### 通用化

模板方法模式的核心思想是：**在父类中定义算法的骨架，将某些步骤延迟到子类实现**。Spring Boot 的自动配置 + 接口约定，是模板方法模式在框架层面的体现。

```
┌─────────────────────────────────────────────────────────────┐
│                    模板方法模式（框架级）                       │
│                                                             │
│  框架定义骨架：                                               │
│    步骤1：初始化配置（XxlJobProperties）                      │
│    步骤2：创建执行器（XxlJobSpringExecutor）                  │
│    步骤3：扫描 Handler（@XxlJob 注解扫描）                    │
│    步骤4：注册到 Admin（心跳机制）                            │
│    步骤5：监听调度请求（内置 HTTP Server）                    │
│                                                             │
│  业务填充细节：                                               │
│    @XxlJob("myJob")                                         │
│    public void execute() { /* 具体业务逻辑 */ }              │
└─────────────────────────────────────────────────────────────┘
```

**与 Spring Boot Starter 的关系**：

Spring Boot Starter 本身就是模板方法模式的体现：

| 层次 | 角色 | 本项目实例 |
|------|------|-----------|
| Spring Boot | 定义 Starter 的加载骨架 | AutoConfiguration.imports 扫描 |
| Starter | 定义组件的初始化骨架 | YudaoXxlJobAutoConfiguration |
| 业务代码 | 填充具体的配置和实现 | @XxlJob 方法 + YAML 配置 |

### 设计要点

1. **SmartInitializingSingleton**：在所有 Bean 初始化后才扫描 `@XxlJob`，确保所有 Handler 都已注册
2. **XxlJobSpringExecutor vs XxlJobExecutor**：Spring 增强版自动扫描注解、支持依赖注入
3. **@ConditionalOnMissingBean**：允许用户自定义执行器，框架的自动配置退让

## 模式间的协作

这三种模式在定时任务组件中协同工作：

```
一个定时任务从配置到执行的完整生命周期中涉及的模式：

1. 自动配置模式
   Spring Boot 扫描 AutoConfiguration.imports
   → 条件检查 → 创建 XxlJobExecutor Bean
   → 绑定 XxlJobProperties 配置

2. 模板方法模式
   XxlJobSpringExecutor 定义任务执行骨架
   → 扫描 @XxlJob 方法 → 注册 Handler
   → 接收调度请求 → 调用 Handler → 上报结果

3. AOP 模式
   @TenantJob 切面增强 Handler
   → 获取租户列表 → 逐个设置上下文执行
   → 收集结果 → 上报

协作关系：
  自动配置(1) 创建执行器 → 模板方法(2) 管理任务生命周期
    → AOP(3) 增强任务执行（多租户遍历）
```

## 思考题

1. **`@ConditionalOnMissingBean` 在什么场景下会被触发？** 提示：考虑用户需要完全自定义执行器初始化逻辑的情况。

2. **`@TenantJob` 切面使用 `parallelStream()` 并行执行，如果改用 `CompletableFuture` 会有什么优势？** 提示：考虑异常处理、超时控制、线程池配置。

3. **如果要支持"按租户分片执行"（类似 XXL-Job 的分片广播），`@TenantJob` 切面需要怎么改？** 提示：考虑将租户列表分组，每组分配到不同的执行器实例。
