# 01-设计模式提炼

本文从多租户组件中识别出五种核心设计模式，将它们从具体实现中抽象为通用知识，帮助读者在其他项目中识别和应用相同的设计手法。

---

## 模式一：拦截器模式（SQL 自动改写）

### 在本项目中的体现

`TenantDatabaseInterceptor` 实现了 MyBatis-Plus 的 `TenantLineHandler` 接口，在 SQL 执行前自动改写语句，追加 `WHERE tenant_id = ?` 条件。业务代码完全不感知租户的存在。

```
业务代码：SELECT * FROM product WHERE status = 1
    │
    ▼ TenantLineInnerInterceptor 拦截
    │
    ▼ TenantDatabaseInterceptor.getTenantId() → 1
    │
    ▼ 改写后：SELECT * FROM product WHERE status = 1 AND tenant_id = 1
    │
    ▼ 执行改写后的 SQL
```

核心源码：

```java
// TenantDatabaseInterceptor.java
public class TenantDatabaseInterceptor implements TenantLineHandler {

    @Override
    public Expression getTenantId() {
        return new LongValue(TenantContextHolder.getRequiredTenantId());
    }

    @Override
    public boolean ignoreTable(String tableName) {
        // 三级判断：全局忽略 → 配置忽略 → 反射检查
        if (TenantContextHolder.isIgnore()) {
            return true;
        }
        tableName = SqlParserUtils.removeWrapperSymbol(tableName);
        Boolean ignore = ignoreTables.get(tableName.toLowerCase());
        if (ignore == null) {
            ignore = computeIgnoreTable(tableName);
            synchronized (ignoreTables) {
                addIgnoreTable(tableName, ignore);
            }
        }
        return ignore;
    }
}
```

### 通用化

拦截器模式的核心思想是：**在不修改业务代码的前提下，通过拦截执行流程并注入额外逻辑，实现横切关注点的处理**。

```
┌─────────────────────────────────────────────────────────────┐
│                    拦截器模式通用结构                          │
│                                                             │
│  请求 → [拦截器1] → [拦截器2] → [拦截器3] → 实际处理         │
│            │            │            │                       │
│         前置处理     前置处理     前置处理                     │
│                                                             │
│  实际处理 → [拦截器3] → [拦截器2] → [拦截器1] → 响应         │
│              后置处理     后置处理     后置处理                │
└─────────────────────────────────────────────────────────────┘
```

**适用场景**：

| 场景 | 拦截点 | 典型实现 |
|------|--------|---------|
| SQL 改写 | MyBatis 执行前 | TenantLineInnerInterceptor |
| 日志记录 | HTTP 请求前后 | HandlerInterceptor |
| 权限校验 | 方法调用前 | AOP @PreAuthorize |
| 缓存拦截 | 方法调用前后 | Spring Cache |
| 消息增强 | 消息发送前后 | RedisMessageInterceptor |

**与其他模式的关系**：拦截器模式常常与**责任链模式**结合——多个拦截器组成链，依次执行。yudao-cloud 中 MyBatis-Plus 的拦截器链（租户拦截器 → 数据权限拦截器 → 分页拦截器）就是典型例子。

### 设计要点

1. **透明性**：拦截器对被拦截的对象透明，业务代码不需要知道拦截器的存在
2. **可组合**：多个拦截器可以叠加，各自负责不同的关注点
3. **顺序敏感**：拦截器的执行顺序很重要（租户拦截器必须在分页拦截器之前）

## 模式二：ThreadLocal 上下文传播模式

### 在本项目中的体现

`TenantContextHolder` 使用 `TransmittableThreadLocal` 存储当前线程的租户 ID 和忽略标志。整条请求链路中，任何代码都可以通过 `TenantContextHolder.getTenantId()` 获取当前租户。

```
HTTP 请求进入
  │
  ▼ TenantContextWebFilter：写入 TenantContextHolder
  │
  ▼ Controller → Service → Mapper：读取 TenantContextHolder
  │
  ▼ TenantDatabaseInterceptor：从 TenantContextHolder 取值改写 SQL
  │
  ▼ TenantContextWebFilter (finally)：清理 TenantContextHolder
```

核心源码：

```java
// TenantContextHolder.java
public class TenantContextHolder {

    private static final ThreadLocal<Long> TENANT_ID = new TransmittableThreadLocal<>();
    private static final ThreadLocal<Boolean> IGNORE = new TransmittableThreadLocal<>();

    public static Long getTenantId() {
        return TENANT_ID.get();
    }

    public static void setTenantId(Long tenantId) {
        TENANT_ID.set(tenantId);
    }

    public static void clear() {
        TENANT_ID.remove();
        IGNORE.remove();
    }
}
```

### 通用化

ThreadLocal 上下文传播模式的核心思想是：**利用线程局部存储，在请求链路的不同层次之间隐式传递上下文信息，避免参数层层透传**。

```
┌─────────────────────────────────────────────────────────────┐
│              ThreadLocal 上下文传播模式                        │
│                                                             │
│  入口层：写入上下文                                           │
│    ↓                                                        │
│  中间层：无感知地传递                                         │
│    ↓                                                        │
│  使用层：读取上下文                                           │
│    ↓                                                        │
│  出口层：清理上下文                                           │
└─────────────────────────────────────────────────────────────┘
```

**常见应用**：

| 场景 | 上下文内容 | 写入时机 | 清理时机 |
|------|-----------|---------|---------|
| 多租户 | tenantId | Web Filter | Web Filter finally |
| 用户认证 | LoginUser | Token Filter | Token Filter finally |
| 链路追踪 | traceId | MDC Filter | MDC Filter finally |
| 数据权限 | deptId | AOP 切面 | AOP finally |

### 注意事项

ThreadLocal 上下文模式有三个常见陷阱：

**陷阱一：线程池复用导致上下文泄漏**

```
请求 A（tenantId=1）→ 线程池线程执行 → 设置 tenantId=1
请求 B（tenantId=2）→ 复用同一线程 → tenantId 还是 1！数据串租户！
```

**解决**：在 finally 块中清理上下文。yudao-cloud 的 `TenantContextWebFilter` 严格遵循这个原则。

**陷阱二：异步场景上下文丢失**

```
主线程：tenantId=1
  │ @Async 异步调用
  ▼
异步线程：tenantId=null → SQL 没有租户条件 → 数据泄露！
```

**解决**：使用 `TransmittableThreadLocal`（TTL），它能在线程池场景下自动捕获和回放上下文。

**陷阱三：忘记清理导致内存泄漏**

Web 应用中，线程是长期存活的。如果请求结束后不清理 ThreadLocal，其中的对象永远不会被 GC 回收。

**解决**：始终在 finally 块中调用 `clear()`。

## 模式三：策略模式（租户忽略策略三级检查）

### 在本项目中的体现

`TenantDatabaseInterceptor.ignoreTable()` 方法实现了一个三级判断策略，按优先级从高到低依次检查，决定是否忽略某张表的租户过滤：

```
ignoreTable(tableName)
  │
  ├── 策略 1：全局忽略（TenantContextHolder.isIgnore()）
  │     最快，直接返回 true
  │
  ├── 策略 2：配置忽略（ignoreTables 缓存命中）
  │     快，HashMap 查找
  │
  └── 策略 3：反射检查（computeIgnoreTable()）
        慢，涉及类元数据查询
        结果缓存到 ignoreTables 中
```

核心源码：

```java
@Override
public boolean ignoreTable(String tableName) {
    // 策略 1：全局忽略
    if (TenantContextHolder.isIgnore()) {
        return true;
    }
    // 策略 2：配置忽略（查缓存）
    tableName = SqlParserUtils.removeWrapperSymbol(tableName);
    Boolean ignore = ignoreTables.get(tableName.toLowerCase());
    if (ignore == null) {
        // 策略 3：反射检查
        ignore = computeIgnoreTable(tableName);
        synchronized (ignoreTables) {
            addIgnoreTable(tableName, ignore);
        }
    }
    return ignore;
}
```

### 通用化

策略模式的核心思想是：**将算法族封装成独立的策略类，让它们可以互相替换**。yudao-cloud 的特殊之处在于策略有**优先级**——先尝试快的策略，失败后降级到慢的策略。

```
┌─────────────────────────────────────────────────────────────┐
│                    策略模式（带优先级）                        │
│                                                             │
│  输入 → 策略1（最快）→ 命中？→ 是 → 返回                    │
│                  │                                          │
│                  否                                         │
│                  ↓                                          │
│         策略2（较快）→ 命中？→ 是 → 返回                    │
│                  │                                          │
│                  否                                         │
│                  ↓                                          │
│         策略3（最慢）→ 返回                                  │
└─────────────────────────────────────────────────────────────┘
```

**适用场景**：

| 场景 | 策略链 | 本项目实例 |
|------|--------|-----------|
| 多级缓存 | L1 本地缓存 → L2 Redis → L3 数据库 | Guava → Redis → MySQL |
| 认证方式 | Header 透传 → Token 校验 → Mock 模拟 | TokenAuthenticationFilter |
| 表忽略判断 | 全局标志 → 配置缓存 → 反射检查 | TenantDatabaseInterceptor |
| URL 匹配 | 精确匹配 → Ant 模糊匹配 | TenantSecurityWebFilter |

### 设计要点

1. **短路求值**：每个策略检查前先判断是否可以快速返回，避免不必要的计算
2. **结果缓存**：慢策略的结果缓存起来，后续相同输入直接从缓存获取
3. **保守默认**：无法确定时选择安全的默认值（"沉默的表"默认忽略）

## 模式四：代理模式（AOP @TenantIgnore）

### 在本项目中的体现

`@TenantIgnore` 注解配合 `TenantIgnoreAspect` 切面，为方法调用添加了"忽略租户"的代理逻辑。调用方不需要手动设置 `TenantContextHolder.setIgnore(true)`，切面自动完成。

```java
// 业务代码：只加注解，不操心上下文
@TenantIgnore
public void syncAllDictData() {
    // 这里的 SQL 不会追加 tenant_id 条件
    List<DictDataDO> allData = dictDataMapper.selectList();
}

// 切面自动处理上下文
@Around("@annotation(tenantIgnore)")
public Object around(ProceedingJoinPoint joinPoint, TenantIgnore tenantIgnore) throws Throwable {
    Boolean oldIgnore = TenantContextHolder.isIgnore();
    try {
        Object enable = SpringExpressionUtils.parseExpression(tenantIgnore.enable());
        if (Boolean.TRUE.equals(enable)) {
            TenantContextHolder.setIgnore(true);  // 设置忽略标志
        }
        return joinPoint.proceed();  // 执行原始方法
    } finally {
        TenantContextHolder.setIgnore(oldIgnore);  // 恢复原状态
    }
}
```

### 通用化

代理模式的核心思想是：**为一个对象提供一个替身，以控制对这个对象的访问**。AOP 切面是 Spring 中实现代理模式的主要手段。

```
┌─────────────────────────────────────────────────────────────┐
│                    AOP 代理模式                               │
│                                                             │
│  调用方 → 代理（切面）→ 目标方法                              │
│            │                                                │
│            ├── 前置处理（设置上下文、权限检查、日志记录）       │
│            ├── 执行目标方法                                   │
│            └── 后置处理（清理上下文、记录结果、异常处理）       │
└─────────────────────────────────────────────────────────────┘
```

**保存-恢复模式**：`TenantIgnoreAspect` 遵循一个经典模式——在修改共享状态前保存旧值，在 finally 块中恢复：

```java
Boolean oldIgnore = TenantContextHolder.isIgnore();  // 保存
try {
    TenantContextHolder.setIgnore(true);              // 修改
    return joinPoint.proceed();                        // 执行
} finally {
    TenantContextHolder.setIgnore(oldIgnore);          // 恢复
}
```

这个模式在任何需要临时修改共享状态的场景中都适用，比如：
- 临时切换数据源
- 临时修改日志级别
- 临时切换用户身份

## 模式五：缓存旁路模式（Guava LoadingCache）

### 在本项目中的体现

`TenantFrameworkServiceImpl` 使用 Guava `LoadingCache` 缓存租户列表和租户校验结果，避免每次请求都查询数据库。

```java
// TenantFrameworkServiceImpl.java
private final LoadingCache<Object, List<Long>> getTenantIdsCache = buildAsyncReloadingCache(
        Duration.ofMinutes(1L),
        new CacheLoader<Object, List<Long>>() {
            @Override
            public List<Long> load(Object key) {
                return tenantApi.getTenantIdList().getCheckedData();
            }
        });

private final LoadingCache<Long, CommonResult<Boolean>> validTenantCache = buildAsyncReloadingCache(
        Duration.ofMinutes(1L),
        new CacheLoader<Long, CommonResult<Boolean>>() {
            @Override
            public CommonResult<Boolean> load(Long id) {
                return tenantApi.validTenant(id);
            }
        });
```

### 通用化

缓存旁路模式（Cache-Aside Pattern）的核心思想是：**先查缓存，缓存未命中时才查数据库，查到后写入缓存**。

```
┌─────────────────────────────────────────────────────────────┐
│                    缓存旁路模式                               │
│                                                             │
│  请求 → 查缓存 → 命中？→ 是 → 返回缓存数据                   │
│                  │                                          │
│                  否                                         │
│                  ↓                                          │
│         查数据库 → 返回数据 → 写入缓存                        │
└─────────────────────────────────────────────────────────────┘
```

yudao-cloud 的特殊之处在于使用了 **`buildAsyncReloadingCache`**——异步刷新：

```
同步加载（build）：
  缓存过期 → 请求阻塞 → 等待 DB 返回 → 返回新值
  延迟：DB 查询时间（可能 10-100ms）

异步刷新（buildAsyncReloadingCache）：
  缓存过期 → 立即返回旧值（不阻塞）→ 后台异步加载新值
  延迟：≈ 0ms
```

**适用场景**：

| 场景 | 缓存对象 | 过期策略 | 刷新方式 |
|------|---------|---------|---------|
| 租户校验 | 租户合法性 | 1 分钟 | 异步刷新 |
| 租户列表 | 所有租户 ID | 1 分钟 | 异步刷新 |
| 权限校验 | 用户权限集合 | 5 分钟 | 异步刷新 |
| 字典数据 | 字典项列表 | 10 分钟 | 同步加载 |

### Guava LoadingCache vs Redis 缓存

| 维度 | Guava LoadingCache | Redis |
|------|-------------------|-------|
| 延迟 | 本地内存，纳秒级 | 网络调用，毫秒级 |
| 依赖 | 无外部依赖 | 需要 Redis 服务 |
| 一致性 | 每个 JVM 实例独立缓存 | 多实例共享 |
| 数据量 | 适合小数据集（百级） | 适合大数据集（万级+） |
| 适用场景 | 租户列表、权限配置 | 业务数据、会话信息 |

**选型原则**：数据量小、变化不频繁、对延迟敏感的配置类数据用 Guava；数据量大、需要多实例共享的业务数据用 Redis。

## 模式间的协作

这五种模式在多租户组件中不是孤立的，而是协同工作：

```
一个请求的完整生命周期中涉及的模式：

1. 拦截器模式
   TenantContextWebFilter 拦截 HTTP 请求

2. ThreadLocal 上下文传播模式
   TenantContextHolder 存储租户 ID

3. 策略模式
   TenantDatabaseInterceptor.ignoreTable() 三级检查

4. 代理模式
   TenantIgnoreAspect 处理 @TenantIgnore 注解

5. 缓存旁路模式
   TenantFrameworkServiceImpl 缓存租户校验结果

它们的协作关系：
  拦截器(1) → 写入上下文(2) → SQL 改写时读取上下文(2)
                              → 忽略判断时使用策略(3)
  代理(4)   → 临时修改上下文(2)的 ignore 标志
  缓存(5)   → 为拦截器(1)的校验提供快速响应
```

## 思考题

1. **拦截器模式与装饰器模式有什么区别？** 提示：考虑拦截器是否修改被拦截对象的行为，以及拦截器链的动态性。

2. **ThreadLocal 上下文模式与依赖注入（DI）有什么相似之处？** 提示：两者都是为了让代码不需要显式传递依赖。

3. **策略模式中的三级检查如果改为责任链模式，代码会有什么变化？哪种更合适？** 提示：考虑三级检查是否需要动态组合。
