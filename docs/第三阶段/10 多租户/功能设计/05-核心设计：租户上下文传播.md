# 05-核心设计：租户上下文传播

## 场景引入

上一章我们看到，`TenantDatabaseInterceptor` 在 SQL 执行时从 `TenantContextHolder` 获取租户编号。但问题来了：

- 用户的 HTTP 请求带着 `tenant-id: 100` 的头进来
- 经过 Controller、Service、Mapper 层层调用
- 最终 SQL 执行时，`TenantContextHolder.getTenantId()` 是怎么知道当前租户是 100 的？

更复杂的是，如果 Service A 通过 Feign 调用 Service B，租户信息怎么跨服务传递？

这就是**上下文传播**要解决的问题。

## 设计目标

| 目标 | 说明 |
|------|------|
| **请求入口提取** | 从 HTTP 请求头中解析租户编号，存入线程上下文 |
| **线程内传播** | 同一线程内的所有代码都能访问当前租户编号 |
| **跨线程传播** | 线程池中执行的任务也能继承父线程的租户编号 |
| **跨服务传播** | Feign 调用时自动将租户编号放入请求头 |
| **安全校验** | 防止用户越权访问其他租户的数据 |
| **上下文清理** | 请求结束后清理，避免内存泄漏和数据串租户 |

## 上下文存储：TenantContextHolder

### 为什么用 TransmittableThreadLocal？

`TenantContextHolder` 是整个上下文传播的核心枢纽。它使用 `TransmittableThreadLocal`（阿里巴巴开源的 TTL 库）存储租户信息：

```java
public class TenantContextHolder {

    /**
     * 当前租户编号
     */
    private static final ThreadLocal<Long> TENANT_ID = new TransmittableThreadLocal<>();

    /**
     * 是否忽略租户
     */
    private static final ThreadLocal<Boolean> IGNORE = new TransmittableThreadLocal<>();

    public static Long getTenantId() {
        return TENANT_ID.get();
    }

    public static Long getRequiredTenantId() {
        Long tenantId = getTenantId();
        if (tenantId == null) {
            throw new NullPointerException("TenantContextHolder 不存在租户编号！");
        }
        return tenantId;
    }

    public static void setTenantId(Long tenantId) {
        TENANT_ID.set(tenantId);
    }

    public static void setIgnore(Boolean ignore) {
        IGNORE.set(ignore);
    }

    public static boolean isIgnore() {
        return Boolean.TRUE.equals(IGNORE.get());
    }

    public static void clear() {
        TENANT_ID.remove();
        IGNORE.remove();
    }
}
```

**为什么不用普通的 `ThreadLocal`？**

在电商后台中，一个请求可能需要调用多个服务，或者提交到线程池异步执行。普通的 `ThreadLocal` 在线程切换时会丢失上下文：

```
主线程（tenant-id=100）
  │
  ├── 提交任务到线程池 ──→ 工作线程（tenant-id=null）❌ 租户信息丢失
  │
  └── 普通 ThreadLocal 的行为
```

`TransmittableThreadLocal` 通过包装 `ExecutorService`，在任务提交时自动捕获上下文，在任务执行时自动回放：

```
主线程（tenant-id=100）
  │
  ├── 提交任务到线程池 ──→ 工作线程（tenant-id=100）✅ 自动传播
  │
  └── TransmittableThreadLocal 的行为
```

### 两个标志位的分工

| 字段 | 用途 | 典型场景 |
|------|------|---------|
| `TENANT_ID` | 当前操作的租户编号 | 正常请求、切换租户查看 |
| `IGNORE` | 是否跳过租户过滤 | 全局操作（定时任务、数据迁移） |

两者配合使用：当 `IGNORE=true` 时，即使 `TENANT_ID` 有值，数据库层也会跳过租户过滤。这在"管理员查看所有租户的统计数据"等场景中非常有用。

## 请求入口：TenantContextWebFilter

### 工作流程

```
HTTP 请求进入
  │
  ▼
TenantContextWebFilter（Filter 优先级：TENANT_CONTEXT_FILTER）
  │
  ├── 1. 从请求头读取 tenant-id
  │     WebFrameworkUtils.getTenantId(request)
  │
  ├── 2. 如果有值，写入 TenantContextHolder
  │     TenantContextHolder.setTenantId(tenantId)
  │
  ├── 3. 继续执行后续 Filter → Controller → Service → Mapper
  │
  └── 4. finally 块中清理上下文
        TenantContextHolder.clear()
```

### 源码解析

```java
public class TenantContextWebFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        // 设置
        Long tenantId = WebFrameworkUtils.getTenantId(request);
        if (tenantId != null) {
            TenantContextHolder.setTenantId(tenantId);
        }
        try {
            chain.doFilter(request, response);
        } finally {
            // 清理
            TenantContextHolder.clear();
        }
    }
}
```

**设计要点**：

1. **继承 `OncePerRequestFilter`**：保证每个请求只执行一次，避免在 `forward`、`include` 等场景下重复执行。

2. **finally 清理**：无论请求成功还是异常，都必须清理上下文。否则线程池复用线程时，下一个请求会"继承"上一个请求的租户信息——这是典型的"数据串租户"Bug。

3. **从请求头读取**：`WebFrameworkUtils.getTenantId(request)` 读取的是 `tenant-id` 请求头。前端（或网关）在发起请求时需要携带这个头。

### 请求头约定

框架约定使用 `tenant-id` 作为请求头名称：

```java
// WebFrameworkUtils.java
public static final String HEADER_TENANT_ID = "tenant-id";
```

前端请求示例：

```http
GET /admin-api/system/user/list HTTP/1.1
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
tenant-id: 100
```

Swagger/Knife4j 中也自动配置了这个全局参数，方便开发调试。

## 安全防线：TenantSecurityWebFilter

`TenantContextWebFilter` 只负责"读取"租户编号，不负责"校验"。真正的安全校验由 `TenantSecurityWebFilter` 完成。

### 校验流程

```
TenantSecurityWebFilter 执行
  │
  ├── 1. 获取当前登录用户和请求中的 tenant-id
  │
  ├── 2. 已登录用户 + 请求带了 tenant-id
  │     │
  │     ├── tenant-id 与用户所属租户一致 ──→ 放行
  │     └── tenant-id 与用户所属租户不一致 ──→ 拒绝（防越权）
  │
  ├── 3. 已登录用户 + 请求没带 tenant-id
  │     └── 使用用户所属的 tenant-id（自动填充）
  │
  ├── 4. 是豁免 URL？
  │     │
  │     ├── 是 + 没有 tenant-id ──→ 设置 ignore=true，放行
  │     └── 是 + 有 tenant-id ──→ 继续校验
  │
  ├── 5. 非豁免 URL + 没有 tenant-id ──→ 拒绝（缺少租户标识）
  │
  └── 6. 校验租户是否合法（未禁用、未过期）
        │
        ├── 合法 ──→ 放行
        └── 不合法 ──→ 拒绝
```

### 源码解析

```java
public class TenantSecurityWebFilter extends ApiRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        Long tenantId = TenantContextHolder.getTenantId();

        // 1. 登陆的用户，校验是否有权限访问该租户，避免越权问题。
        LoginUser user = SecurityFrameworkUtils.getLoginUser();
        if (user != null) {
            // 如果获取不到租户编号，则尝试使用登陆用户的租户编号
            if (tenantId == null) {
                tenantId = user.getTenantId();
                TenantContextHolder.setTenantId(tenantId);
            // 如果传递了租户编号，则进行比对租户编号，避免越权问题
            } else if (!Objects.equals(user.getTenantId(), TenantContextHolder.getTenantId())) {
                log.error("[doFilterInternal][租户({}) User({}/{}) 越权访问租户({}) URL({}/{})]",
                        user.getTenantId(), user.getId(), user.getUserType(),
                        TenantContextHolder.getTenantId(), request.getRequestURI(), request.getMethod());
                ServletUtils.writeJSON(response, CommonResult.error(
                    GlobalErrorCodeConstants.FORBIDDEN.getCode(), "您无权访问该租户的数据"));
                return;
            }
        }

        // 如果非允许忽略租户的 URL，则校验租户是否合法
        if (!isIgnoreUrl(request)) {
            // 2. 如果请求未带租户的编号，不允许访问。
            if (tenantId == null) {
                ServletUtils.writeJSON(response, CommonResult.error(
                    GlobalErrorCodeConstants.BAD_REQUEST.getCode(), "请求的租户标识未传递"));
                return;
            }
            // 3. 校验租户是合法，例如说被禁用、到期
            try {
                tenantFrameworkService.validTenant(tenantId);
            } catch (Throwable ex) {
                CommonResult<?> result = globalExceptionHandler.allExceptionHandler(request, ex);
                ServletUtils.writeJSON(response, result);
                return;
            }
        } else {
            // 如果是允许忽略租户的 URL，若未传递租户编号，则默认忽略
            if (tenantId == null) {
                TenantContextHolder.setIgnore(true);
            }
        }

        chain.doFilter(request, response);
    }
}
```

### 越权防护详解

**场景**：商家 A 的员工（tenant-id=100）在请求头中伪造 `tenant-id: 200`，试图查看商家 B 的数据。

```
请求：GET /admin-api/product/list
Header: tenant-id: 200
用户：商家 A 的员工（user.tenantId = 100）

校验逻辑：
  user.getTenantId() = 100
  TenantContextHolder.getTenantId() = 200
  100 != 200 ──→ 拒绝，返回 403
```

这个校验在 `TenantSecurityWebFilter` 中完成，在业务代码执行之前就拦截了。

### 豁免 URL 的处理

某些 URL 不需要租户标识（如支付回调、短信回调），通过 `isIgnoreUrl()` 判断：

```java
private boolean isIgnoreUrl(HttpServletRequest request) {
    String apiUri = request.getRequestURI().substring(request.getContextPath().length());
    // 快速匹配（精确匹配）
    if (CollUtil.contains(tenantProperties.getIgnoreUrls(), apiUri)
            || CollUtil.contains(ignoreUrls, apiUri)) {
        return true;
    }
    // 逐个 Ant 路径匹配（模糊匹配）
    for (String url : tenantProperties.getIgnoreUrls()) {
        if (pathMatcher.match(url, apiUri)) {
            return true;
        }
    }
    // ...
    return false;
}
```

**注意**：`ignoreUrls` 有两个来源：
1. `TenantProperties.ignoreUrls`：配置文件中手动指定的
2. `getTenantIgnoreUrls()`：自动扫描 `@TenantIgnore` 注解标记的 Controller 方法

## 跨服务传播：TenantRequestInterceptor

当服务 A 通过 Feign 调用服务 B 时，租户信息需要通过 HTTP 请求头传递。

### 工作原理

```
服务 A（tenant-id=100）
  │
  ├── TenantContextHolder.getTenantId() = 100
  │
  ├── Feign 调用服务 B
  │     │
  │     └── TenantRequestInterceptor.apply()
  │           │
  │           └── requestTemplate.header("tenant-id", "100")
  │
  ▼
服务 B 收到请求
  │
  └── TenantContextWebFilter 读取 tenant-id = 100
      └── TenantContextHolder.setTenantId(100)
```

### 源码解析

```java
public class TenantRequestInterceptor implements RequestInterceptor {

    @Override
    public void apply(RequestTemplate requestTemplate) {
        Long tenantId = TenantContextHolder.getTenantId();
        if (tenantId != null) {
            requestTemplate.header(HEADER_TENANT_ID, String.valueOf(tenantId));
        }
    }
}
```

**设计要点**：

1. **条件传播**：只有当前线程有租户编号时才传播，避免传播 null 值到下游服务。

2. **Feign 自动装配**：在 `YudaoTenantRpcAutoConfiguration` 中注册为 Bean，所有 Feign 客户端自动使用。

3. **请求头约定**：使用与 HTTP 请求相同的 `tenant-id` 头，保持一致性。

### 自动配置

```java
@AutoConfiguration
@ConditionalOnProperty(prefix = "yudao.tenant", value = "enable", matchIfMissing = true)
@EnableFeignClients(clients = TenantCommonApi.class)
public class YudaoTenantRpcAutoConfiguration {

    @Bean
    public TenantRequestInterceptor tenantRequestInterceptor() {
        return new TenantRequestInterceptor();
    }
}
```

## 跨租户访问：TenantVisitContextInterceptor

除了正常的租户隔离，还有一个特殊场景：**平台管理员需要切换到某个商家的视角查看数据**。

### 工作流程

```
平台管理员（tenant-id=0，平台租户）
  │
  ├── 请求头携带 visit-tenant-id: 100（要查看的商家）
  │
  ▼
TenantVisitContextInterceptor
  │
  ├── 1. 读取 visit-tenant-id
  │
  ├── 2. 如果与当前租户相同，跳过
  │
  ├── 3. 校验用户是否有 system:tenant:visit 权限
  │     │
  │     ├── 无权限 ──→ 抛出 403
  │     └── 有权限 ──→ 继续
  │
  ├── 4. 切换 TenantContextHolder.setTenantId(visitTenantId)
  │
  └── 5. 请求结束后恢复原租户
```

### 源码解析

```java
@RequiredArgsConstructor
@Slf4j
public class TenantVisitContextInterceptor implements HandlerInterceptor {

    private static final String PERMISSION = "system:tenant:visit";

    private final TenantProperties tenantProperties;
    private final SecurityFrameworkService securityFrameworkService;

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        // 读取要访问的租户编号
        Long visitTenantId = WebFrameworkUtils.getVisitTenantId(request);
        if (visitTenantId == null) {
            return true;
        }
        // 如果和当前租户编号一致，则直接跳过
        if (ObjUtil.equal(visitTenantId, TenantContextHolder.getTenantId())) {
            return true;
        }
        // 必须是登录用户
        LoginUser loginUser = SecurityFrameworkUtils.getLoginUser();
        if (loginUser == null) {
            return true;
        }
        // 校验用户是否可切换租户
        if (!securityFrameworkService.hasAnyPermissions(PERMISSION)) {
            throw exception0(GlobalErrorCodeConstants.FORBIDDEN.getCode(), "您无权切换租户");
        }
        // 【重点】切换租户编号
        loginUser.setVisitTenantId(visitTenantId);
        TenantContextHolder.setTenantId(visitTenantId);
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) {
        // 【重点】清理切换，换回原租户编号
        LoginUser loginUser = SecurityFrameworkUtils.getLoginUser();
        if (loginUser != null && loginUser.getTenantId() != null) {
            TenantContextHolder.setTenantId(loginUser.getTenantId());
        }
    }
}
```

**设计要点**：

1. **权限校验**：只有拥有 `system:tenant:visit` 权限的用户才能切换租户，普通商家员工无法使用。

2. **请求结束后恢复**：`afterCompletion` 中恢复原始租户编号，确保后续代码（如日志记录）使用正确的租户上下文。

3. **排除路径**：通过 `ignoreVisitUrls` 配置排除某些不需要跨租户访问的 URL。

## 完整的上下文传播链路

把所有组件串起来，一个请求的完整生命周期：

```
前端请求
  │
  │  Header: tenant-id: 100
  │  Header: Authorization: Bearer xxx
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│  TenantContextWebFilter                                  │
│  读取 tenant-id: 100，写入 TenantContextHolder           │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│  TenantSecurityWebFilter                                 │
│  校验：用户属于租户 100，请求 tenant-id=100，一致 ✓      │
│  校验：租户 100 未禁用、未过期 ✓                         │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│  TenantVisitContextInterceptor（可选）                   │
│  如果有 visit-tenant-id，切换到目标租户                  │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│  Controller → Service → Mapper                           │
│  业务代码完全不感知租户                                  │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│  TenantDatabaseInterceptor                               │
│  SQL 改写：追加 WHERE tenant_id = 100                    │
└─────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│  TenantContextWebFilter (finally)                        │
│  清理：TenantContextHolder.clear()                       │
└─────────────────────────────────────────────────────────┘
```

## 设计取舍与思考

### 为什么用 Filter 而不是 Interceptor 做上下文设置？

`TenantContextWebFilter` 是 Filter，而 `TenantSecurityWebFilter` 也是 Filter。为什么不用 Spring MVC 的 HandlerInterceptor？

**原因**：Filter 在 DispatcherServlet 之前执行，能覆盖更早的阶段。如果用 Interceptor，某些在 Filter 阶段就抛异常的请求（如认证失败）将无法正确清理上下文。

### 为什么安全校验单独一个 Filter？

`TenantSecurityWebFilter` 可以合并到 `TenantContextWebFilter` 中，但拆分有好处：

1. **职责单一**：一个负责"读取"，一个负责"校验"
2. **顺序可控**：通过 `WebFilterOrderEnum` 精确控制执行顺序
3. **可测试性**：两个 Filter 可以独立单元测试

### 为什么用请求头而不是请求参数？

租户信息通过 `tenant-id` 请求头传递，而不是 URL 参数 `?tenantId=100`：

| 维度 | 请求头 | URL 参数 |
|------|--------|---------|
| 安全性 | 不会出现在 URL 日志中 | 可能被日志记录 |
| 一致性 | 前后端统一约定 | 参数名可能不一致 |
| 简洁性 | 全局配置一次 | 每个请求都要带 |
| 缓存友好 | 不影响 URL 唯一性 | 不同 tenantId 产生不同 URL |

## 小结

租户上下文传播的核心设计可以概括为：

1. **一个存储**：`TenantContextHolder` 使用 `TransmittableThreadLocal`，支持跨线程传播
2. **两个 Filter**：
   - `TenantContextWebFilter`：读取请求头，设置上下文，清理上下文
   - `TenantSecurityWebFilter`：防越权校验，租户合法性校验
3. **一个 Interceptor**：`TenantRequestInterceptor` 通过 Feign 请求头跨服务传播
4. **一个切换器**：`TenantVisitContextInterceptor` 支持管理员跨租户查看

整个传播链路的设计原则是：**入口统一设置，出口统一清理，中间各层自动透明**。业务开发人员完全不需要关心租户信息的传递。
