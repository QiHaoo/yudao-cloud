# 07-配置设计与边界

## 场景引入

多租户组件已经设计好了，但实际部署时总有一些"灰色地带"：

- 支付回调接口不需要 `tenant-id` 头，怎么配置豁免？
- 有些缓存是全局共享的（如字典缓存），怎么排除？
- `@TenantIgnore` 注解能不能根据条件动态生效？

本章解决这些配置与边界问题。

## TenantProperties 配置项总览

所有配置项都在 `yudao.tenant` 前缀下：

```java
@ConfigurationProperties(prefix = "yudao.tenant")
@Data
public class TenantProperties {

    /**
     * 是否开启多租户（默认 true）
     */
    private Boolean enable = true;

    /**
     * 需要忽略多租户的请求 URL
     */
    private Set<String> ignoreUrls = new HashSet<>();

    /**
     * 需要忽略跨租户访问的请求 URL
     */
    private Set<String> ignoreVisitUrls = Collections.emptySet();

    /**
     * 需要忽略多租户的表
     */
    private Set<String> ignoreTables = Collections.emptySet();

    /**
     * 需要忽略多租户的 Spring Cache 缓存
     */
    private Set<String> ignoreCaches = Collections.emptySet();
}
```

### 配置示例

```yaml
yudao:
  tenant:
    enable: true
    ignore-urls:
      - /admin-api/pay/callback/**       # 支付回调
      - /admin-api/sms/callback/**       # 短信回调
      - /admin-api/system/tenant/get-ids # 获取租户列表（管理接口）
    ignore-visit-urls:
      - /admin-api/user/profile/**       # 个人信息不允许跨租户访问
    ignore-tables:
      - system_tenant                    # 租户表本身
      - system_dict_type                 # 字典类型全局共享
      - system_dict_data                 # 字典数据全局共享
    ignore-caches:
      - dict_data                        # 字典缓存
      - system_config                    # 系统配置缓存
```

## 五个配置项详解

### 1. enable：全局开关

```yaml
yudao:
  tenant:
    enable: false  # 完全禁用多租户
```

通过 `@ConditionalOnProperty` 控制整个自动配置类是否生效：

```java
@AutoConfiguration
@ConditionalOnProperty(prefix = "yudao.tenant", value = "enable", matchIfMissing = true)
public class YudaoTenantAutoConfiguration {
    // ...
}
```

**使用场景**：
- 开发环境：本地调试时不想带 `tenant-id` 头
- 单租户部署：客户私有化部署，只有一个租户
- 灰度上线：先关闭多租户，逐步迁移

**注意**：`matchIfMissing = true` 表示不配置时默认开启。这是安全的选择——多租户是核心功能，不应该因为漏配置而失效。

### 2. ignoreUrls：豁免 URL

```yaml
yudao:
  tenant:
    ignore-urls:
      - /admin-api/pay/callback/alipay
      - /admin-api/pay/callback/wechat
      - /admin-api/sms/callback/**
```

**作用**：这些 URL 不要求携带 `tenant-id` 请求头，也不做租户有效性校验。

**匹配规则**：
1. 精确匹配：`/admin-api/pay/callback/alipay` 只匹配这个精确路径
2. Ant 风格匹配：`/admin-api/sms/callback/**` 匹配所有子路径

**底层实现**在 `TenantSecurityWebFilter.isIgnoreUrl()` 中：

```java
private boolean isIgnoreUrl(HttpServletRequest request) {
    String apiUri = request.getRequestURI().substring(request.getContextPath().length());
    // 快速匹配（精确匹配）
    if (CollUtil.contains(tenantProperties.getIgnoreUrls(), apiUri)
            || CollUtil.contains(ignoreUrls, apiUri)) {
        return true;
    }
    // 逐个 Ant 路径匹配
    for (String url : tenantProperties.getIgnoreUrls()) {
        if (pathMatcher.match(url, apiUri)) {
            return true;
        }
    }
    return false;
}
```

**设计要点**：先做精确匹配（O(1)），再做 Ant 匹配（O(n)），保证性能。

### 3. ignoreVisitUrls：跨租户访问排除

```yaml
yudao:
  tenant:
    ignore-visit-urls:
      - /admin-api/user/profile/**
      - /admin-api/user/password/**
```

**作用**：这些 URL 不允许通过 `visit-tenant-id` 请求头切换租户。

**场景**：用户修改自己的密码、查看自己的个人信息，这些操作应该始终作用于用户所属的租户，不允许管理员通过"切换租户"来操作。

**实现在自动配置中**：

```java
@Bean
public WebMvcConfigurer tenantWebMvcConfigurer(TenantProperties tenantProperties,
                                               TenantVisitContextInterceptor tenantVisitContextInterceptor) {
    return new WebMvcConfigurer() {
        @Override
        public void addInterceptors(InterceptorRegistry registry) {
            registry.addInterceptor(tenantVisitContextInterceptor)
                    .excludePathPatterns(tenantProperties.getIgnoreVisitUrls().toArray(new String[0]));
        }
    };
}
```

### 4. ignoreTables：忽略的表

```yaml
yudao:
  tenant:
    ignore-tables:
      - system_tenant
      - system_dict_type
      - system_dict_data
      - system_oauth2_client
```

**作用**：这些表的 SQL 不会自动追加 `tenant_id` 条件。

**典型需要忽略的表**：

| 表 | 忽略原因 |
|---|---------|
| `system_tenant` | 租户表本身不能有 `tenant_id`，否则无限递归 |
| `system_dict_type` | 字典类型全局共享，所有租户用同一套 |
| `system_dict_data` | 字典数据全局共享 |
| `system_oauth2_client` | OAuth2 客户端配置全局共享 |
| `system_error_code` | 错误码全局共享 |

**注意**：这个配置在 `TenantDatabaseInterceptor` 构造时一次性加载：

```java
public TenantDatabaseInterceptor(TenantProperties properties) {
    properties.getIgnoreTables().forEach(table -> {
        addIgnoreTable(table, true);
    });
    addIgnoreTable("DUAL", true);  // Oracle 的 DUAL 表
}
```

**运行时修改配置不会生效**，需要重启应用。

### 5. ignoreCaches：忽略的缓存

```yaml
yudao:
  tenant:
    ignore-caches:
      - dict_data
      - system_config
```

**作用**：这些缓存的 key 不会自动拼接租户后缀。

在 `TenantRedisCacheManager` 中实现：

```java
@Override
public Cache getCache(String name) {
    String[] names = StrUtil.splitToArray(name, SPLIT);
    // 如果开启多租户，则 name 拼接租户后缀
    if (!TenantContextHolder.isIgnore()
            && TenantContextHolder.getTenantId() != null
            && !CollUtil.contains(ignoreCaches, names[0])) {
        name = name + ":" + TenantContextHolder.getTenantId();
    }
    return super.getCache(name);
}
```

**不忽略时的缓存 key**：`dict_data:100`（租户 100 的字典缓存）
**忽略后的缓存 key**：`dict_data`（全局共享的字典缓存）

## @TenantIgnore 注解详解

### 三种使用位置

`@TenantIgnore` 可以标注在三种位置，作用各不相同：

```java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Inherited
public @interface TenantIgnore {
    String enable() default "true";
}
```

#### 1. 标注在 Controller 方法上

```java
@RestController
@RequestMapping("/admin-api/pay")
public class PayController {

    @PostMapping("/callback/alipay")
    @TenantIgnore  // 支付宝回调不需要租户标识
    public CommonResult<Boolean> alipayCallback(@RequestBody AlipayCallbackReqVO reqVO) {
        // ...
    }
}
```

**效果**：该 URL 自动加入 `ignoreUrls`，不需要在配置文件中手动添加。

实现在自动配置类的 `getTenantIgnoreUrls()` 方法中：

```java
private Set<String> getTenantIgnoreUrls() {
    Set<String> ignoreUrls = new HashSet<>();
    RequestMappingHandlerMapping handlerMapping = (RequestMappingHandlerMapping)
            applicationContext.getBean("requestMappingHandlerMapping");
    Map<RequestMappingInfo, HandlerMethod> handlerMethodMap = handlerMapping.getHandlerMethods();

    for (Map.Entry<RequestMappingInfo, HandlerMethod> entry : handlerMethodMap.entrySet()) {
        HandlerMethod handlerMethod = entry.getValue();
        // 检查方法级和类级注解
        if (!handlerMethod.hasMethodAnnotation(TenantIgnore.class)
            && !handlerMethod.getBeanType().isAnnotationPresent(TenantIgnore.class)) {
            continue;
        }
        // 添加到忽略 URL 集合
        if (entry.getKey().getPatternsCondition() != null) {
            ignoreUrls.addAll(entry.getKey().getPatternsCondition().getPatterns());
        }
    }
    return ignoreUrls;
}
```

#### 2. 标注在 Controller 类上

```java
@RestController
@RequestMapping("/admin-api/pay")
@TenantIgnore  // 整个支付回调 Controller 都不需要租户标识
public class PayCallbackController {
    // 所有接口都自动加入 ignoreUrls
}
```

**效果**：该 Controller 下所有接口的 URL 都加入 `ignoreUrls`。

#### 3. 标注在 DO 实体类上

```java
@Data
@TableName("system_tenant")
@TenantIgnore  // 租户表不需要租户隔离
public class TenantDO extends BaseDO {
    private Long id;
    private String name;
}
```

**效果**：等价于将 `system_tenant` 加入 `ignoreTables` 配置。判断逻辑在 `computeIgnoreTable()` 中：

```java
TenantIgnore tenantIgnore = tableInfo.getEntityType().getAnnotation(TenantIgnore.class);
return tenantIgnore != null;  // 有注解就忽略
```

### SpEL 条件表达式

`@TenantIgnore` 的 `enable` 属性支持 Spring 表达式语言（SpEL），可以实现动态条件：

```java
@TenantIgnore(enable = "@tenantIgnoreChecker.isIgnore()")  // 调用 Bean 方法
public void syncAllData() {
    // ...
}
```

**底层实现**在 `TenantIgnoreAspect` 中：

```java
@Around("@annotation(tenantIgnore)")
public Object around(ProceedingJoinPoint joinPoint, TenantIgnore tenantIgnore) throws Throwable {
    Boolean oldIgnore = TenantContextHolder.isIgnore();
    try {
        // 计算条件，满足的情况下，才进行忽略
        Object enable = SpringExpressionUtils.parseExpression(tenantIgnore.enable());
        if (Boolean.TRUE.equals(enable)) {
            TenantContextHolder.setIgnore(true);
        }
        return joinPoint.proceed();
    } finally {
        TenantContextHolder.setIgnore(oldIgnore);
    }
}
```

**SpEL 表达式示例**：

| 表达式 | 含义 |
|--------|------|
| `"true"` | 始终忽略（默认值） |
| `"false"` | 始终不忽略（等同于不加注解） |
| `"@tenantChecker.isGlobal()"` | 调用 Bean 方法动态判断 |
| `"T(cn.iocoder.yudao.TenantUtils).isGlobal()"` | 调用静态方法 |
| `"#userId == 1L"` | 基于方法参数判断（需要配合 `@Param`） |

### 注意事项

源码注释中明确指出：

> 注意，只有 DB 的场景会过滤，其它场景暂时不过滤：
> 1. Redis 场景：因为是基于 Key 实现多租户的能力，所以忽略没有意义
> 2. MQ 场景：有点难以抉择，目前可以通过 Consumer 手动在消费的方法上添加 @TenantIgnore 进行忽略

这意味着 `@TenantIgnore` 只影响数据库操作，不影响 Redis 缓存和消息队列。

## 组件边界与分工

### 多租户组件做什么

```
┌─────────────────────────────────────────────────────────────┐
│                    多租户组件的职责范围                        │
│                                                             │
│  ✅ 自动追加 SQL 的 tenant_id 条件                           │
│  ✅ 从请求头读取并传播租户编号                                │
│  ✅ 防越权校验（用户只能访问自己租户的数据）                   │
│  ✅ 租户有效性校验（禁用、过期）                              │
│  ✅ 缓存 key 自动加租户后缀                                  │
│  ✅ Feign 调用自动传播租户头                                  │
│  ✅ 定时任务自动遍历所有租户                                  │
└─────────────────────────────────────────────────────────────┘
```

### 多租户组件不做什么

```
┌─────────────────────────────────────────────────────────────┐
│                    多租户组件不负责的事                        │
│                                                             │
│  ❌ 租户的 CRUD 管理（由业务模块 system 负责）                │
│  ❌ 租户套餐、计费（由业务模块负责）                          │
│  ❌ 租户级别的数据备份与恢复                                  │
│  ❌ 租户级别的资源配额（CPU、内存、存储）                     │
│  ❌ 租户级别的功能开关（Feature Flag）                        │
│  ❌ 跨租户的数据迁移                                         │
└─────────────────────────────────────────────────────────────┘
```

### 与其他组件的分工

| 关注点 | 负责组件 | 说明 |
|--------|---------|------|
| 租户 CRUD | system 模块 | 租户的增删改查、套餐管理 |
| 用户认证 | security 组件 | 登录、Token 校验 |
| 租户隔离 | tenant 组件 | SQL 改写、上下文传播 |
| 数据权限 | data-permission 组件 | 部门级、用户级数据过滤 |
| 多租户缓存 | tenant 组件 | Redis Cache key 拼接 |
| 多租户消息 | tenant 组件 | MQ 消息自动带租户标识 |

**特别注意**：多租户和数据权限是两个独立的维度。一个请求可以同时有租户过滤和数据权限过滤：

```sql
-- 最终 SQL 可能同时有两个 WHERE 条件
SELECT * FROM product
WHERE tenant_id = 100          -- 多租户过滤
  AND dept_id IN (1, 2, 3)     -- 数据权限过滤（假设用户能看这些部门的数据）
```

## 与数据权限的交互

### 两个组件的执行顺序

```
SQL 执行
  │
  ▼
TenantLineInnerInterceptor（租户过滤，先执行）
  │  追加：WHERE tenant_id = 100
  │
  ▼
DataPermissionInterceptor（数据权限，后执行）
  │  追加：WHERE dept_id IN (1, 2, 3)
  │
  ▼
最终 SQL：WHERE tenant_id = 100 AND dept_id IN (1, 2, 3)
```

**为什么租户过滤先执行？** 因为在 `YudaoTenantAutoConfiguration` 中，`TenantLineInnerInterceptor` 被添加到拦截器链的第一个位置。这保证了租户过滤是"最底层"的过滤，其他过滤器在此基础上进一步缩小范围。

### 冲突场景

**场景**：管理员通过 `visit-tenant-id` 切换到租户 200，但数据权限仍然基于管理员自己的部门。

这通常不是问题，因为：
1. 租户过滤基于 `TenantContextHolder.getTenantId()`
2. 数据权限基于当前登录用户的部门

两者独立工作，互不干扰。但如果数据权限的部门数据分布在不同租户中，可能会出现"看不到数据"的情况。这需要在业务层做额外处理。

## 边界场景与局限性

### 局限一：跨租户查询需要显式处理

如果需要在管理后台查询所有租户的数据（如"全平台商品总数"），必须使用 `TenantUtils.executeIgnore()` 或逐个租户执行：

```java
// 方案一：忽略租户（适合全局统计）
Integer total = TenantUtils.executeIgnore(() -> productMapper.selectCount());

// 方案二：逐个租户（适合需要按租户分组的结果）
Map<Long, Integer> result = new HashMap<>();
for (Long tenantId : tenantFrameworkService.getTenantIds()) {
    Integer count = TenantUtils.execute(tenantId, () -> productMapper.selectCount());
    result.put(tenantId, count);
}
```

**局限**：方案一只能用于没有 `tenant_id` 条件需求的场景。如果表有 `tenant_id` 列但没有继承 `TenantBaseDO`，`executeIgnore()` 不会影响 SQL（因为 `ignoreTable()` 已经返回 true 了）。

### 局限二：动态数据源与多租户

如果使用动态数据源（如按租户分库），当前的行级隔离方案就不够了。需要扩展 `TenantDatabaseInterceptor`，在 `getTenantId()` 中不仅返回租户编号，还要切换数据源。

yudao-cloud 当前的设计是**共享数据源 + 行级隔离**，不支持按租户分库。

### 局限三：缓存一致性

```
时间线：
  T1: 管理员禁用租户 100（数据库更新）
  T2: 请求到达，validTenantCache 命中旧缓存，校验通过
  T3: 1 分钟后缓存过期，新请求才会被拦截
```

在 T2 到 T3 的时间窗口内，被禁用的租户仍然可以访问。这是"最终一致性"的代价，对于大多数业务场景是可接受的。

### 局限四：第三方 SQL 不拦截

手动编写的原生 SQL（如 `@Select` 注解中的 SQL 字符串）不会被 MyBatis-Plus 拦截器改写：

```java
@Select("SELECT * FROM product WHERE id = #{id}")
ProductDO selectByIdRaw(@Param("id") Long id);
```

这条 SQL 不会自动追加 `tenant_id` 条件。开发人员必须自己加上：

```java
@Select("SELECT * FROM product WHERE id = #{id} AND tenant_id = #{tenantId}")
ProductDO selectByIdRaw(@Param("id") Long id, @Param("tenantId") Long tenantId);
```

**建议**：尽量使用 MyBatis-Plus 的 Wrapper 查询，避免手写 SQL。

## 小结

配置设计与边界的核心要点：

1. **五个配置项**：`enable`（开关）、`ignoreUrls`（URL 豁免）、`ignoreVisitUrls`（跨租户排除）、`ignoreTables`（表豁免）、`ignoreCaches`（缓存豁免）
2. **@TenantIgnore 三种用法**：Controller 方法（URL 豁免）、Controller 类（批量 URL 豁免）、DO 实体类（表豁免）
3. **SpEL 表达式**：`enable` 属性支持动态条件判断
4. **组件边界**：多租户负责隔离，不负责 CRUD、计费、资源配额
5. **已知局限**：跨租户查询需显式处理、不支持分库、缓存有延迟、原生 SQL 不拦截
