# 04-核心设计：行级租户隔离

## 场景引入

回到我们的电商平台。商家 A 上架了一款手机壳，商家 B 上架了一条数据线。它们都存在同一张 `product` 表里：

```sql
SELECT * FROM product;
```

如果 SQL 不做任何处理，商家 A 登录后台，会看到所有商家的商品——这显然不行。

**核心问题**：如何让每一条 SQL 查询自动带上 `WHERE tenant_id = ?` 条件，让业务开发人员完全不用关心租户隔离？

## 设计目标

| 目标 | 说明 |
|------|------|
| **自动追加** | 所有 SQL 自动追加 `tenant_id` 条件，业务代码零侵入 |
| **三级豁免** | 全局配置 → 基类继承 → 注解标记，三种方式灵活排除不需要隔离的表 |
| **缓存优化** | 豁免判断结果缓存，避免每次 SQL 解析都做反射检查 |

## 核心机制：TenantDatabaseInterceptor

整个行级隔离的核心是 `TenantDatabaseInterceptor`，它实现了 MyBatis-Plus 的 `TenantLineHandler` 接口。在 03 章我们已经讲过这个接口的两个核心方法，这里来看框架是如何实现的。

### 整体工作流程

```
SQL 执行
  │
  ▼
TenantLineInnerInterceptor（MyBatis-Plus 内置）
  │
  ├── 调用 getTenantId() ──→ 从 TenantContextHolder 获取当前租户编号
  │
  ├── 调用 ignoreTable() ──→ 三级检查决定是否豁免
  │     │
  │     ├── 情况一：TenantContextHolder.isIgnore() == true ──→ 全局忽略
  │     ├── 情况二：ignoreTables 缓存命中 ──→ 直接返回
  │     └── 情况三：computeIgnoreTable() ──→ 反射检查后缓存
  │
  ▼
  不忽略 ──→ SQL 改写，追加 WHERE tenant_id = ?
  忽略   ──→ SQL 原样执行
```

### 源码解析

源码位置：`yudao-framework/yudao-spring-boot-starter-biz-tenant/.../core/db/TenantDatabaseInterceptor.java`

```java
public class TenantDatabaseInterceptor implements TenantLineHandler {

    /**
     * 忽略的表缓存
     * KEY：表名（大小写都存）
     * VALUE：是否忽略
     */
    private final Map<String, Boolean> ignoreTables = new HashMap<>();

    public TenantDatabaseInterceptor(TenantProperties properties) {
        // 从配置文件加载全局忽略的表
        properties.getIgnoreTables().forEach(table -> {
            addIgnoreTable(table, true);
        });
        // Oracle 的 DUAL 表也需要忽略
        addIgnoreTable("DUAL", true);
    }

    @Override
    public Expression getTenantId() {
        return new LongValue(TenantContextHolder.getRequiredTenantId());
    }

    @Override
    public boolean ignoreTable(String tableName) {
        // 情况一，全局忽略多租户
        if (TenantContextHolder.isIgnore()) {
            return true;
        }
        // 情况二，忽略多租户的表（查缓存）
        tableName = SqlParserUtils.removeWrapperSymbol(tableName);
        Boolean ignore = ignoreTables.get(tableName.toLowerCase());
        if (ignore == null) {
            ignore = computeIgnoreTable(tableName);
            synchronized (ignoreTables) {
                addIgnoreTable(tableName, ignore);
            }
        }
        return ignore;
    }
}
```

**关键设计点**：

1. **`getTenantId()` 从上下文取值**：不持有租户信息，而是每次从 `TenantContextHolder` 动态获取。这意味着同一个拦截器实例可以服务所有租户。

2. **`ignoreTable()` 的短路设计**：先检查全局忽略标志（最快），再查缓存（快），最后才做反射检查（慢）。大多数情况下前两步就能返回。

3. **缓存的双写策略**：`addIgnoreTable` 同时存大写和小写 key，解决不同数据库（MySQL 小写、Oracle 大写）的表名大小写差异。

## 三级豁免机制详解

### 第一级：全局忽略（TenantContextHolder.isIgnore()）

这是最粗粒度的豁免。当 `TenantContextHolder` 中的 `ignore` 标志为 `true` 时，**所有表**都跳过租户过滤。

触发场景：
- `@TenantIgnore` 注解标记的方法/类
- `TenantUtils.executeIgnore()` 编程式忽略
- 不带 `tenant-id` 头的豁免 URL 请求

```java
// ignoreTable() 方法的第一行
if (TenantContextHolder.isIgnore()) {
    return true;  // 全部忽略
}
```

### 第二级：配置文件忽略（ignoreTables）

在 `application.yml` 中配置的全局忽略表列表：

```yaml
yudao:
  tenant:
    ignore-tables:
      - system_tenant    # 租户表本身不能有 tenant_id
      - system_dict_type # 字典类型全局共享
```

这些表在 `TenantDatabaseInterceptor` 构造时就加载到 `ignoreTables` 缓存中。

### 第三级：反射检查（computeIgnoreTable）

当缓存未命中时，执行 `computeIgnoreTable()` 做最终判断：

```java
private boolean computeIgnoreTable(String tableName) {
    // 1. 找不到表信息 → 不是项目里的表 → 忽略
    TableInfo tableInfo = TableInfoHelper.getTableInfo(tableName);
    if (tableInfo == null) {
        return true;
    }
    // 2. 继承了 TenantBaseDO → 需要租户隔离 → 不忽略
    if (TenantBaseDO.class.isAssignableFrom(tableInfo.getEntityType())) {
        return false;
    }
    // 3. 有 @TenantIgnore 注解 → 忽略
    TenantIgnore tenantIgnore = tableInfo.getEntityType().getAnnotation(TenantIgnore.class);
    return tenantIgnore != null;
}
```

**判断逻辑**可以用一张决策树表示：

```
表名在 MyBatis-Plus 中注册了吗？
  │
  ├── 否 ──→ 外部表，不拦截 ──→ 忽略
  │
  └── 是 ──→ 实体类继承了 TenantBaseDO？
               │
               ├── 是 ──→ 需要租户隔离 ──→ 不忽略
               │
               └── 否 ──→ 实体类有 @TenantIgnore 注解？
                          │
                          ├── 是 ──→ 忽略
                          └── 否 ──→ 不忽略（默认行为）
```

这里有一个**隐含的设计决策**：如果一个表既没有继承 `TenantBaseDO`，也没有标注 `@TenantIgnore`，它会被**忽略**（不加 `tenant_id` 条件）。这是因为框架无法确定一个"沉默"的表是否需要租户隔离，选择保守策略——不拦截。

## TenantBaseDO：租户字段的承载体

```java
@Data
@EqualsAndHashCode(callSuper = true)
public abstract class TenantBaseDO extends BaseDO {
    /**
     * 多租户编号
     */
    private Long tenantId;
}
```

业务 DO 继承 `TenantBaseDO` 后，MyBatis-Plus 的 `insert` 操作会自动填充 `tenant_id` 字段（通过 `MetaObjectHandler`），查询时会被 `TenantLineInnerInterceptor` 自动追加 WHERE 条件。

以电商为例：

```java
/**
 * 商品 DO
 */
@Data
@TableName("product")
public class ProductDO extends TenantBaseDO {
    private Long id;
    private String name;
    private Long price;
    // tenantId 字段继承自 TenantBaseDO，无需声明
}
```

对应的 SQL 变化：

```sql
-- 开发人员写的
INSERT INTO product (name, price) VALUES ('手机壳', 29.9);

-- 实际执行的（自动填充 tenant_id）
INSERT INTO product (name, price, tenant_id) VALUES ('手机壳', 29.9, 100);

-- 开发人员写的
SELECT * FROM product WHERE id = 1;

-- 实际执行的（自动追加 tenant_id 条件）
SELECT * FROM product WHERE id = 1 AND tenant_id = 100;
```

## @TenantIgnore 注解：类级别标记

`@TenantIgnore` 可以标注在 DO 实体类上，表示该表不需要租户隔离：

```java
@Data
@TableName("system_tenant")
@TenantIgnore  // 租户表本身不能有 tenant_id
public class TenantDO extends BaseDO {
    private Long id;
    private String name;
    private Integer status;
}
```

这等价于在配置文件中添加 `system_tenant` 到 `ignoreTables`，但更直观——"这个表不需要租户隔离"的意图直接写在实体类上。

## 与 MyBatis-Plus 的集成点

在自动配置类中，`TenantLineInnerInterceptor` 被注册到 MyBatis-Plus 的拦截器链中：

```java
@Bean
public TenantLineInnerInterceptor tenantLineInnerInterceptor(TenantProperties properties,
                                                             MybatisPlusInterceptor interceptor) {
    TenantLineInnerInterceptor inner = new TenantLineInnerInterceptor(
        new TenantDatabaseInterceptor(properties));
    // 必须添加到第一个位置，在分页插件前面（MyBatis-Plus 的规定）
    MyBatisUtils.addInterceptor(interceptor, inner, 0);
    return inner;
}
```

**为什么必须是第一个？** 因为 MyBatis-Plus 要求 `TenantLineInnerInterceptor` 在 `PaginationInnerInterceptor` 之前。如果顺序反了，分页插件会先改写 SQL，然后租户插件再改写，可能导致 SQL 语法错误。

## 设计取舍与思考

### 为什么用缓存而不是每次都反射？

反射检查（`TableInfoHelper.getTableInfo()` + `isAssignableFrom()` + `getAnnotation()`）涉及类元数据查询，在高并发场景下有一定开销。框架用 `HashMap` 缓存结果，每个表只检查一次。

**代价**：如果运行时动态修改注解（极少场景），缓存不会自动失效。但这个场景在实际业务中几乎不存在。

### 为什么同时存大写和小写 key？

```java
private void addIgnoreTable(String tableName, boolean ignore) {
    ignoreTables.put(tableName.toLowerCase(), ignore);
    ignoreTables.put(tableName.toUpperCase(), ignore);
}
```

MySQL 默认表名不区分大小写，Oracle 默认大写。如果不做双写，同一个表在不同数据库下可能缓存未命中，导致重复反射检查。

### 为什么"沉默的表"默认忽略？

如果一个实体类既没有继承 `TenantBaseDO`，也没有标注 `@TenantIgnore`，框架选择忽略它。这是**保守策略**：

- 优点：不会误伤第三方库的表、中间表、外部数据源表
- 缺点：如果开发人员忘记继承 `TenantBaseDO`，该表不会有租户隔离

这个取舍在实际项目中是合理的——新表创建时，开发人员会明确选择继承 `TenantBaseDO` 还是 `BaseDO`，不太可能"忘记"。

## 与传统方案的对比

| 维度 | 手动 WHERE | MyBatis 拦截器 | yudao 方案 |
|------|-----------|---------------|-----------|
| 业务侵入 | 每个 SQL 都要加 | 无侵入 | 无侵入 |
| 灵活性 | 最灵活 | 一般 | 三级豁免，最灵活 |
| 性能 | 最优 | 有拦截开销 | 缓存优化，开销极小 |
| 维护成本 | 极高 | 中等 | 低 |

## 小结

行级租户隔离的核心设计可以概括为：

1. **一个拦截器**：`TenantDatabaseInterceptor` 实现 `TenantLineHandler`，在 SQL 执行前自动改写
2. **三级豁免**：全局忽略 → 配置忽略 → 反射检查，从快到慢逐级判断
3. **一个基类**：`TenantBaseDO` 标记需要隔离的表，同时承载 `tenant_id` 字段
4. **一个缓存**：`ignoreTables` HashMap 缓存反射检查结果，避免重复开销
