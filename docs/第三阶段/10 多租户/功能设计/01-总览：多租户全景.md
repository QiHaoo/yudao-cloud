# 01-总览：多租户全景

## 场景引入

假设你正在运营一个 SaaS 电商平台，服务着数百个商家（租户）。每个商家有自己的商品、订单、会员数据，但共享同一套应用代码和基础设施。

**核心矛盾**：如何让"商家 A"只能看到自己的商品和订单，而"商家 B"完全不知道"商家 A"的存在？更关键的是——如何让业务开发人员**几乎不需要写任何隔离代码**，系统就能自动完成数据隔离？

这就是多租户（Multi-Tenancy）要解决的问题。

## 三种多租户隔离方案

业界主流的多租户数据隔离方案有三种，各有取舍：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        方案一：独立数据库                              │
│                                                                     │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐                       │
│  │ 商家 A   │    │ 商家 B   │    │ 商家 C   │                       │
│  │ 专用 DB  │    │ 专用 DB  │    │ 专用 DB  │                       │
│  └──────────┘    └──────────┘    └──────────┘                       │
│  隔离性：★★★★★  成本：★★★★★  运维：★★★★★                           │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                      方案二：共享 Schema                              │
│                                                                     │
│  ┌─────────────────────────────────────────┐                        │
│  │              共享数据库                   │                        │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ │                        │
│  │  │Schema A  │ │Schema B  │ │Schema C  │ │                        │
│  │  └──────────┘ └──────────┘ └──────────┘ │                        │
│  └─────────────────────────────────────────┘                        │
│  隔离性：★★★★   成本：★★★    运维：★★★★                             │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                  方案三：共享 Schema 行级隔离                          │

│  ┌─────────────────────────────────────────┐                        │
│  │              共享数据库                   │                        │
│  │  ┌─────────────────────────────────┐    │                        │
│  │  │        共享表 + tenant_id 列     │    │                        │
│  │  │  ┌─────┐ ┌─────┐ ┌─────┐       │    │                        │
│  │  │  │Row A│ │Row B│ │Row C│ ...    │    │                        │
```

### 详细对比

| 维度 | 独立数据库 | 共享 Schema | 共享 Schema 行级隔离 |
|------|-----------|------------|-------------------|
| **数据隔离强度** | 最强（物理隔离） | 强（Schema 级隔离） | 中等（逻辑隔离） |
| **成本** | 每个租户一套 DB，成本高 | 一套 DB，多个 Schema | 一套 DB，一套表，成本最低 |
| **运维复杂度** | 高（备份、迁移、扩容） | 中等 | 低 |
| **跨租户查询** | 极难（需要联邦查询） | 难（需要跨 Schema） | 简单（去掉 WHERE 条件即可） |
| **租户定制** | 可独立定制表结构 | 可部分定制 | 不可定制 |
| **开发复杂度** | 低（连接不同 DB 即可） | 中等 | 高（需要 SQL 改写） |
| **适用场景** | 金融、政府（强隔离需求） | 中型 SaaS | 通用 SaaS |

## yudao-cloud 的选择：共享 Schema 行级隔离

yudao-cloud 选择了**方案三：共享 Schema 行级隔离**，原因如下：

1. **成本最低**：不需要为每个租户创建独立的数据库或 Schema，一套表结构服务所有租户
2. **运维最简**：备份、迁移、扩容都只针对一套数据库
3. **开发效率高**：通过 MyBatis-Plus 拦截器自动改写 SQL，业务代码**零感知**
4. **SaaS 电商场景适配**：电商平台的租户（商家）通常共享相同的数据模型，不需要定制表结构

**代价是什么？** 隔离性相对较弱——如果 SQL 改写出 bug，可能导致数据泄露。因此，yudao-cloud 在多个层面设置了安全防线。

## 核心组件一览

| 组件 | 职责 | 所在层 |

| `TenantContextHolder` | 存储当前线程的租户 ID 和忽略标记 | 上下文 |

## 一条请求的完整生命周期

以"商家 A 查询自己的商品列表"为例：

```
1. 浏览器发送请求
   GET /admin-api/product/page?name=手机
   Header: tenant-id: 1

2. TenantContextWebFilter
   → 从 Header 取出 tenant-id=1
   → TenantContextHolder.setTenantId(1)

3. TenantSecurityWebFilter
   → 检查登录用户是否属于租户 1
   → 检查租户 1 是否合法（未禁用、未过期）

4. TenantVisitContextInterceptor
   → 无切换操作，直接放行

5. ProductController.page()
   → 正常的业务逻辑

6. ProductMapper.selectPage()
   → MyBatis-Plus 拦截器触发
   → TenantDatabaseInterceptor.ignoreTable("product")
     → product 表继承了 TenantBaseDO → 不忽略
   → SQL 改写：
     SELECT * FROM product WHERE name LIKE '%手机%'
     → SELECT * FROM product WHERE name LIKE '%手机%' AND tenant_id = 1

7. 返回结果（只包含租户 1 的商品）

8. TenantContextWebFilter 清理
   → TenantContextHolder.clear()
```

## 小结

多租户的核心思路是：**通过一个 tenant_id 字段，配合 SQL 自动改写，让业务代码完全不需要感知租户的存在**。yudao-cloud 在这个基础上，构建了从 Web 层到数据层、从 HTTP 请求到 RPC 调用、从数据库到缓存到消息队列的**全链路租户隔离体系**。
