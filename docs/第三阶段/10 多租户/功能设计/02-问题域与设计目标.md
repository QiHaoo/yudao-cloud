# 02-问题域与设计目标

## 从一个"数据泄露"事故说起

某电商平台上线后，运营人员发现了一个令人不安的现象：在商品列表页面，偶尔能看到其他商家的商品。排查后发现，是某个新来的开发人员写了一条 SQL，忘了加 `WHERE tenant_id = ?` 条件。

这个事故揭示了多租户的核心挑战：**数据隔离不能依赖人的自觉，必须是系统自动完成的**。

## 多租户的核心挑战

### 挑战一：数据隔离（最核心）

这是多租户最基本也最重要的需求。在共享 Schema 行级隔离方案下，所有租户的数据都存在同一张表中，通过 `tenant_id` 列区分。问题在于：

**场景**：电商平台上，商家 A 的商品表查询：
```sql
-- 正确：只看到自己的商品
SELECT * FROM product WHERE tenant_id = 1 AND status = 1;

-- 危险：忘了加租户条件，看到所有商品
SELECT * FROM product WHERE status = 1;
```

**要求**：业务开发人员不需要手动加 `tenant_id` 条件，框架自动完成。

**难点**：
- SELECT 需要自动追加 `WHERE tenant_id = ?`
- INSERT 需要自动追加 `tenant_id` 字段和值
- UPDATE/DELETE 需要自动追加 `WHERE tenant_id = ?`
- 哪些表需要隔离、哪些不需要，要有灵活的配置机制

### 挑战二：上下文传播

租户信息需要在整条请求链路中传播，包括：

```
HTTP 请求 → Filter → Controller → Service → Mapper → DB
                ↓
           RPC 调用 → 远程 Filter → 远程 Controller → ...
                ↓
           定时任务 → Service → Mapper → DB
                ↓
           MQ 消息 → Consumer → Service → Mapper → DB
```

**要求**：租户信息在所有链路中自动传播，业务代码零感知。

**难点**：
- HTTP 请求：从 Header 中提取，存入 ThreadLocal
- RPC 调用（Feign）：透传 tenant-id Header
- 定时任务：没有 HTTP 请求上下文，需要按租户逐个执行
- MQ 消息：发送时携带租户信息，消费时恢复上下文
- 线程池：子线程需要继承父线程的租户信息

### 挑战三：缓存隔离

Redis 缓存是全局共享的，如果不做隔离，不同租户的缓存会互相覆盖。

**场景**：
```
商家 A 查询商品分类 → 缓存 key: "product:categories" → 数据是商家 A 的
商家 B 查询商品分类 → 命中同一个缓存 key → 拿到的是商家 A 的数据！
```

**要求**：缓存 key 自动追加租户标识，不同租户的缓存互不干扰。

### 挑战四：MQ 消息隔离

消息队列中，一条消息的消费者可能处理的是任意租户的数据。如果没有租户上下文，消费者不知道当前消息属于哪个租户。

**场景**：
```
商家 A 创建订单 → 发送 MQ 消息 → 消费者处理
                                   ↓
                              此时 TenantContextHolder 为空
                              下游的数据库查询没有租户条件
                              可能查到所有租户的数据！
```

**要求**：消息发送时自动携带租户信息，消费时自动恢复上下文。

## 设计目标

### 目标一：零感知

业务开发人员在写 CRUD 代码时，**完全不需要考虑租户**。不需要手动加 `WHERE tenant_id = ?`，不需要手动在 INSERT 语句中加上 `tenant_id` 字段。框架在 MyBatis 拦截器层面自动完成。

```java
// 业务代码完全不感知租户
public List<ProductDO> getProductList(String name) {
    return productMapper.selectList(
        new LambdaQueryWrapperX<ProductDO>()
            .likeIfPresent(ProductDO::getName, name)
    );
    // 框架自动改写为：
    // SELECT * FROM product WHERE name LIKE '%手机%' AND tenant_id = 1
}
```

### 目标二：自动隔离

从 HTTP 请求到数据库查询，从 RPC 调用到 MQ 消费，租户信息的传播和隔离都是**自动完成**的。整个链路中，租户信息像"血液"一样自然流动：

```
HTTP Header: tenant-id=1
    ↓ TenantContextWebFilter 自动提取
ThreadLocal: tenantId=1
    ↓ TenantDatabaseInterceptor 自动改写 SQL
SQL: WHERE tenant_id=1
    ↓ TenantRequestInterceptor 自动透传
Feign Header: tenant-id=1
```

### 目标三：灵活排除

不是所有场景都需要租户隔离。框架需要提供灵活的排除机制：

| 排除粒度 | 方式 | 场景示例 |
|---------|------|---------|
| **全局** | `TenantContextHolder.setIgnore(true)` | 系统管理功能 |
| **URL 级** | 配置 `ignoreUrls` | 支付回调、短信回调 |
| **方法级** | `@TenantIgnore` 注解 | 定时任务、数据迁移 |
| **表级** | 配置 `ignoreTables` | 字典表、配置表 |
| **实体级** | `@TenantIgnore` 注解在 DO 上 | 不需要租户隔离的表 |
| **缓存级** | 配置 `ignoreCaches` | 全局共享的缓存 |

## 与 RBAC 数据权限的区别

初学者容易将"多租户"和"数据权限"混淆。两者都是数据过滤，但解决的问题完全不同：

```
┌─────────────────────────────────────────────────────────────┐
│                        数据过滤体系                           │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              多租户（Tenant）                        │    │
│  │                                                     │    │
│  │  问题：商家 A 不能看到商家 B 的数据                  │    │
│  │  粒度：租户级（一个商家 = 一个租户）                 │    │
│  │  实现：MyBatis 拦截器自动追加 WHERE tenant_id = ?   │    │
│  │  特点：全局生效，业务代码无感知                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              数据权限（Data Scope）                  │    │
│  │                                                     │    │
│  │  问题：部门经理只能看到本部门员工的数据              │    │
│  │  粒度：部门/用户级（一个公司内部分权限）             │    │
│  │  实现：MyBatis 拦截器追加 WHERE dept_id IN (...)    │    │
│  │  特点：基于角色配置，不同角色看到不同范围            │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 核心区别

| 维度 | 多租户 | 数据权限 |
|------|--------|---------|
| **解决的问题** | 不同商家之间的数据隔离 | 同一商家内部不同角色的数据范围 |
| **隔离粒度** | 租户级 | 部门/用户级 |
| **过滤字段** | `tenant_id` | `dept_id` / `user_id` |
| **生效方式** | 全局自动生效 | 按角色配置生效 |
| **业务感知** | 完全无感知 | 需要配置数据权限规则 |
| **典型场景** | 商家 A 看不到商家 B 的数据 | 部门经理只能看本部门的订单 |

### 两者的关系

它们是**叠加**的关系，不是互斥的。一条 SQL 可能同时被两个拦截器改写：

```sql
-- 原始 SQL
SELECT * FROM order WHERE status = 1;

-- 多租户拦截器改写（商家 A）
SELECT * FROM order WHERE status = 1 AND tenant_id = 1;

-- 数据权限拦截器继续改写（只能看本部门）
SELECT * FROM order WHERE status = 1 AND tenant_id = 1 AND dept_id IN (100, 101);
```

## 设计约束

在明确了目标之后，还需要考虑一些约束条件：

### 约束一：不能修改 MyBatis-Plus 的 API

业务开发人员已经习惯了 MyBatis-Plus 的 `LambdaQueryWrapper`、`BaseMapper` 等 API。多租户方案不能要求他们使用不同的 API。

### 约束二：性能影响要可控

SQL 改写发生在每次数据库操作时，必须保证性能开销可控。JSQLParser 解析 SQL 是有成本的，需要通过缓存等手段优化。

### 约束三：支持按需关闭

不是所有场景都需要多租户。例如：
- 单元测试时可能不想启用
- 某些内部管理功能不需要租户隔离
- 开发环境调试时可能需要关闭

因此需要提供 `yudao.tenant.enable=false` 的开关。

## 小结

多租户的核心挑战可以归结为四个字：**自动隔离**。在共享 Schema 行级隔离方案下，需要在数据层、缓存层、消息层、RPC 层全面实现自动的租户隔离和上下文传播，同时提供灵活的排除机制和安全防线。
