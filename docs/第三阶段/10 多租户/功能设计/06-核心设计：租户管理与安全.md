# 06-核心设计：租户管理与安全

## 场景引入

你的电商平台已经支持多租户了。某天运营同事问你：

1. "有个新商家要入驻，我加完租户记录后，多久能生效？"——你不想每次都重启应用。
2. "有个定时任务需要遍历所有商家的数据，怎么写？"——你需要一个工具方法来切换租户上下文。
3. "有个商家欠费了，我要禁用他，怎么保证他的请求全部被拦截？"——你需要租户有效性校验。

本章解决的就是这三个问题。

## 设计目标

| 目标 | 说明 |
|------|------|
| **缓存友好** | 租户列表和校验结果缓存，避免每次都查数据库 |
| **编程式切换** | 提供 `TenantUtils.execute()` 工具方法，支持在代码中显式切换租户 |
| **有效性校验** | 检查租户是否被禁用、是否过期，不合法的租户请求直接拒绝 |

## 租户服务接口：TenantFrameworkService

框架定义了一个轻量级的租户服务接口，只包含两个方法：

```java
public interface TenantFrameworkService {

    /**
     * 获得所有租户
     *
     * @return 租户编号数组
     */
    List<Long> getTenantIds();

    /**
     * 校验租户是否合法
     *
     * @param id 租户编号
     */
    void validTenant(Long id);
}
```

**为什么接口这么精简？**

框架层不需要知道租户的名称、套餐、到期时间等业务细节。它只需要：
1. 有哪些租户编号（用于定时任务遍历）
2. 某个租户是否合法（用于请求校验）

具体的业务逻辑由 `TenantCommonApi`（RPC 接口）实现，框架层通过这个接口间接调用。

## 缓存策略：Guava LoadingCache

### 为什么需要缓存？

每次 HTTP 请求都调用 `validTenant()` 校验租户合法性，如果每次都查数据库，在高并发场景下会成为瓶颈。

```
没有缓存：
  请求 1 ──→ SELECT * FROM system_tenant WHERE id = 100
  请求 2 ──→ SELECT * FROM system_tenant WHERE id = 100
  请求 3 ──→ SELECT * FROM system_tenant WHERE id = 100
  ...
  每秒 1000 个请求 = 1000 次数据库查询
```

```
有缓存：
  请求 1 ──→ 缓存未命中 ──→ SELECT * FROM system_tenant WHERE id = 100 ──→ 写入缓存
  请求 2 ──→ 缓存命中 ──→ 直接返回 ✓
  请求 3 ──→ 缓存命中 ──→ 直接返回 ✓
  ...
  每秒 1000 个请求 = 1 次数据库查询
```

### 源码解析

```java
@RequiredArgsConstructor
public class TenantFrameworkServiceImpl implements TenantFrameworkService {

    private final TenantCommonApi tenantApi;

    /**
     * 针对 getTenantIds() 的缓存
     */
    private final LoadingCache<Object, List<Long>> getTenantIdsCache = buildAsyncReloadingCache(
            Duration.ofMinutes(1L), // 过期时间 1 分钟
            new CacheLoader<Object, List<Long>>() {
                @Override
                public List<Long> load(Object key) {
                    return tenantApi.getTenantIdList().getCheckedData();
                }
            });

    /**
     * 针对 validTenant() 的缓存
     */
    private final LoadingCache<Long, CommonResult<Boolean>> validTenantCache = buildAsyncReloadingCache(
            Duration.ofMinutes(1L), // 过期时间 1 分钟
            new CacheLoader<Long, CommonResult<Boolean>>() {
                @Override
                public CommonResult<Boolean> load(Long id) {
                    return tenantApi.validTenant(id);
                }
            });

    @Override
    @SneakyThrows
    public List<Long> getTenantIds() {
        return getTenantIdsCache.get(Boolean.TRUE);
    }

    @Override
    @SneakyThrows
    public void validTenant(Long id) {
        validTenantCache.get(id).checkError();
    }
}
```

### 缓存设计要点

**1. 异步刷新（buildAsyncReloadingCache）**

Guava 的 `buildAsyncReloadingCache` 在缓存过期后，**不会阻塞**当前请求，而是返回旧值，同时异步加载新值。对比同步加载：

```
同步加载（build）：
  缓存过期 ──→ 请求阻塞 ──→ 等待 DB 返回 ──→ 返回新值
  延迟：DB 查询时间（可能 10-100ms）

异步刷新（buildAsyncReloadingCache）：
  缓存过期 ──→ 立即返回旧值（不阻塞） ──→ 后台异步加载新值
  延迟：≈ 0ms
```

**2. 过期时间 1 分钟**

1 分钟是一个平衡点：
- 太短（如 1 秒）：缓存命中率低，等于没缓存
- 太长（如 1 小时）：租户被禁用后，最多 1 小时才能生效
- 1 分钟：绝大多数场景够用，紧急禁用租户最多等 1 分钟

**3. validTenant 的特殊处理**

`validTenant()` 返回的是 `CommonResult<Boolean>`，而不是直接抛异常。这是为了支持缓存：

```java
@Override
public void validTenant(Long id) {
    validTenantCache.get(id).checkError();  // 如果是错误结果，这里会抛异常
}
```

如果缓存的是异常对象，Guava 会认为加载失败并重试。所以先缓存 `CommonResult`，在使用时再判断是否需要抛异常。

### 缓存失效场景

| 场景 | 缓存行为 | 影响 |
|------|---------|------|
| 新增租户 | 1 分钟后 `getTenantIds()` 才能看到 | 新租户最多延迟 1 分钟生效 |
| 禁用租户 | 1 分钟后 `validTenant()` 才会拒绝 | 被禁用的租户最多还能访问 1 分钟 |
| 租户过期 | 同上 | 过期租户最多还能访问 1 分钟 |

如果需要立即生效，可以重启应用或手动清理缓存（框架未暴露此接口，需要扩展）。

## 编程式租户切换：TenantUtils

### 核心方法

`TenantUtils` 提供了三个核心方法，用于在业务代码中显式操作租户上下文：

```java
public class TenantUtils {

    /**
     * 使用指定租户，执行对应的逻辑
     */
    public static void execute(Long tenantId, Runnable runnable) {
        Long oldTenantId = TenantContextHolder.getTenantId();
        Boolean oldIgnore = TenantContextHolder.isIgnore();
        try {
            TenantContextHolder.setTenantId(tenantId);
            TenantContextHolder.setIgnore(false);  // 强制取消忽略
            runnable.run();
        } finally {
            TenantContextHolder.setTenantId(oldTenantId);
            TenantContextHolder.setIgnore(oldIgnore);
        }
    }

    /**
     * 使用指定租户，执行对应的逻辑（带返回值）
     */
    public static <V> V execute(Long tenantId, Callable<V> callable) {
        Long oldTenantId = TenantContextHolder.getTenantId();
        Boolean oldIgnore = TenantContextHolder.isIgnore();
        try {
            TenantContextHolder.setTenantId(tenantId);
            TenantContextHolder.setIgnore(false);
            return callable.call();
        } catch (Exception e) {
            throw new RuntimeException(e);
        } finally {
            TenantContextHolder.setTenantId(oldTenantId);
            TenantContextHolder.setIgnore(oldIgnore);
        }
    }

    /**
     * 忽略租户，执行对应的逻辑
     */
    public static void executeIgnore(Runnable runnable) {
        Boolean oldIgnore = TenantContextHolder.isIgnore();
        try {
            TenantContextHolder.setIgnore(true);
            runnable.run();
        } finally {
            TenantContextHolder.setIgnore(oldIgnore);
        }
    }

    /**
     * 忽略租户，执行对应的逻辑（带返回值）
     */
    public static <V> V executeIgnore(Callable<V> callable) {
        Boolean oldIgnore = TenantContextHolder.isIgnore();
        try {
            TenantContextHolder.setIgnore(true);
            return callable.call();
        } catch (Exception e) {
            throw new RuntimeException(e);
        } finally {
            TenantContextHolder.setIgnore(oldIgnore);
        }
    }
}
```

### 设计要点

**1. 保存-恢复模式**

所有方法都遵循"保存旧值 → 设置新值 → 执行逻辑 → 恢复旧值"的模式：

```java
Long oldTenantId = TenantContextHolder.getTenantId();  // 保存
try {
    TenantContextHolder.setTenantId(tenantId);          // 设置
    runnable.run();                                      // 执行
} finally {
    TenantContextHolder.setTenantId(oldTenantId);       // 恢复
}
```

这保证了即使执行过程中发生异常，上下文也能正确恢复，不会"污染"后续代码。

**2. execute() 强制取消忽略**

`execute()` 方法会设置 `TenantContextHolder.setIgnore(false)`。这是一个重要的设计决策：

```java
TenantContextHolder.setIgnore(false);  // 强制取消忽略
```

**为什么？** 如果当前处于"忽略租户"状态（比如在一个 `@TenantIgnore` 方法中调用了 `TenantUtils.execute(100, ...)`），你显然是想在租户 100 的上下文中执行，而不是继续忽略。

**3. Callable 重载**

除了 `Runnable`，还提供了 `Callable<V>` 重载，支持有返回值的场景：

```java
List<ProductDO> products = TenantUtils.execute(100L, () -> {
    return productMapper.selectList();
});
```

### 典型使用场景

**场景一：定时任务遍历所有租户**

```java
@Component
public class ProductSyncJob {

    @Resource
    private TenantFrameworkService tenantFrameworkService;
    @Resource
    private ProductService productService;

    @XxlJob("productSyncJob")
    public void execute() {
        List<Long> tenantIds = tenantFrameworkService.getTenantIds();
        for (Long tenantId : tenantIds) {
            TenantUtils.execute(tenantId, () -> {
                productService.syncProducts();  // 在每个租户的上下文中执行
            });
        }
    }
}
```

**场景二：跨租户数据查询**

```java
// 管理后台：查询所有租户的商品总数
public Map<Long, Integer> countProductsByTenant() {
    Map<Long, Integer> result = new HashMap<>();
    List<Long> tenantIds = tenantFrameworkService.getTenantIds();
    for (Long tenantId : tenantIds) {
        Integer count = TenantUtils.execute(tenantId, () -> {
            return productMapper.selectCount();
        });
        result.put(tenantId, count);
    }
    return result;
}
```

**场景三：忽略租户查询全局数据**

```java
// 查询所有租户的字典数据（字典表是全局共享的）
public List<DictDataDO> getAllDictData() {
    return TenantUtils.executeIgnore(() -> {
        return dictDataMapper.selectList();
    });
}
```

## 定时任务的多租户支持：@TenantJob

### 问题

定时任务（如 XXL-Job）执行时，没有 HTTP 请求，也就没有 `tenant-id` 请求头。如果直接执行，`TenantContextHolder.getTenantId()` 返回 null，会导致 `getRequiredTenantId()` 抛异常。

### 解决方案

框架提供了 `@TenantJob` 注解和 `TenantJobAspect` 切面，自动将定时任务"分裂"为多个租户的任务：

```java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface TenantJob {
}
```

### 工作流程

```
@TenantJob 标记的 JobHandler 执行
  │
  ▼
TenantJobAspect 拦截
  │
  ├── 1. 调用 tenantFrameworkService.getTenantIds() 获取所有租户
  │
  ├── 2. 使用 parallelStream 并行遍历每个租户
  │     │
  │     └── 对每个租户：
  │           TenantUtils.execute(tenantId, () -> {
  │               joinPoint.proceed();  // 执行原始 Job 逻辑
  │           });
  │
  ├── 3. 收集每个租户的执行结果
  │
  └── 4. 汇总结果，报告成功/失败
```

### 源码解析

```java
@Aspect
@RequiredArgsConstructor
@Slf4j
public class TenantJobAspect {

    private final TenantFrameworkService tenantFrameworkService;

    @Around("@annotation(tenantJob)")
    public void around(ProceedingJoinPoint joinPoint, TenantJob tenantJob) {
        // 获得租户列表
        List<Long> tenantIds = tenantFrameworkService.getTenantIds();
        if (CollUtil.isEmpty(tenantIds)) {
            return;
        }

        // 逐个租户，执行 Job
        Map<Long, String> results = new ConcurrentHashMap<>();
        AtomicBoolean success = new AtomicBoolean(true);
        XxlJobContext xxlJobContext = XxlJobContext.getXxlJobContext();

        tenantIds.parallelStream().forEach(tenantId -> {
            TenantUtils.execute(tenantId, () -> {
                try {
                    XxlJobContext.setXxlJobContext(xxlJobContext);
                    Object result = joinPoint.proceed();
                    results.put(tenantId, StrUtil.toStringOrEmpty(result));
                } catch (Throwable e) {
                    results.put(tenantId, ExceptionUtil.getRootCauseMessage(e));
                    success.set(false);
                    XxlJobHelper.log(StrUtil.format(
                        "[多租户({}) 执行任务({})，发生异常：{}]",
                        tenantId, joinPoint.getSignature(), ExceptionUtils.getStackTrace(e)));
                }
            });
        });

        // 记录执行结果
        if (success.get()) {
            XxlJobHelper.handleSuccess(JsonUtils.toJsonString(results));
        } else {
            XxlJobHelper.handleFail(JsonUtils.toJsonString(results));
        }
    }
}
```

### 设计要点

**1. 并行执行**

使用 `parallelStream()` 并行处理所有租户，提高执行效率。假设平台有 100 个商家，每个商家的任务执行 1 秒：

```
串行：100 秒
并行：≈ 1 秒（取决于 CPU 核数）
```

**2. 幂等性要求**

源码注释中明确指出：

> 注意，需要保证 JobHandler 的幂等性。因为 Job 因为某个租户执行失败重试时，之前执行成功的租户也会再次执行。

这意味着如果商家 A 的任务执行成功，商家 B 的任务执行失败导致整个 Job 重试，商家 A 的任务会再次执行。业务代码必须能容忍重复执行。

**3. 结果收集**

每个租户的执行结果收集到 `results` Map 中，最后汇总到 XXL-Job 的执行日志。如果有任何一个租户执行失败，整个 Job 标记为失败。

### 使用示例

```java
@Component
public class OrderTimeoutJob {

    @XxlJob("orderTimeoutJob")
    @TenantJob  // 自动为每个租户执行一次
    public void execute() {
        // 这段代码会在每个租户的上下文中执行一次
        orderService.cancelTimeoutOrders();
    }
}
```

等价于手动编写：

```java
@XxlJob("orderTimeoutJob")
public void execute() {
    List<Long> tenantIds = tenantFrameworkService.getTenantIds();
    for (Long tenantId : tenantIds) {
        TenantUtils.execute(tenantId, () -> {
            orderService.cancelTimeoutOrders();
        });
    }
}
```

## 租户有效性校验

### 校验时机

租户有效性校验在 `TenantSecurityWebFilter` 中触发（上一章已详述）。校验调用链：

```
TenantSecurityWebFilter.doFilterInternal()
  │
  └── tenantFrameworkService.validTenant(tenantId)
        │
        └── validTenantCache.get(id).checkError()
              │
              └── tenantApi.validTenant(id)  // RPC 调用
                    │
                    └── 检查：租户是否存在？是否被禁用？是否过期？
```

### 校验失败的响应

校验失败时，`validTenant()` 会抛出 `ServiceException`，被 `TenantSecurityWebFilter` 捕获并转换为 JSON 响应：

```java
try {
    tenantFrameworkService.validTenant(tenantId);
} catch (Throwable ex) {
    CommonResult<?> result = globalExceptionHandler.allExceptionHandler(request, ex);
    ServletUtils.writeJSON(response, result);
    return;
}
```

典型的错误响应：

```json
{
    "code": 1002000001,
    "msg": "租户已被禁用",
    "data": null
}
```

## 设计取舍与思考

### 为什么用 Guava LoadingCache 而不是 Redis？

| 维度 | Guava LoadingCache | Redis |
|------|-------------------|-------|
| 延迟 | 本地内存，纳秒级 | 网络调用，毫秒级 |
| 依赖 | 无外部依赖 | 需要 Redis 服务 |
| 一致性 | 每个 JVM 实例独立缓存 | 多实例共享 |
| 适用场景 | 租户数量有限（百级） | 租户数量极大（万级） |

对于电商平台，商家数量通常在百到千级，Guava 本地缓存完全够用。即使新增/禁用租户有 1 分钟延迟，在业务上也是可接受的。

### 为什么 TenantJob 使用并行而不是串行？

并行执行可以大幅缩短定时任务的总执行时间。但有代价：

1. **资源竞争**：多个租户同时执行可能争抢数据库连接池
2. **调试困难**：并行执行的日志交织在一起
3. **幂等性要求更高**：部分失败重试时，所有租户都要重来

框架选择并行是基于"定时任务通常在低峰期执行"的假设。如果资源紧张，可以在业务层自行控制并发度。

### 为什么 TenantUtils.execute() 要取消 ignore 标志？

考虑这个场景：

```java
@TenantIgnore  // 忽略租户
public void syncAllDictData() {
    // 这里是全局操作，忽略租户
    
    // 但我现在想在租户 100 的上下文中做点事
    TenantUtils.execute(100L, () -> {
        // 如果不取消 ignore，这里的 SQL 还是会忽略租户
        // 这不是我想要的
        productMapper.selectList();  // 应该只返回租户 100 的商品
    });
}
```

`execute()` 的语义是"我要在指定租户的上下文中执行"，所以必须取消 ignore 标志。而 `executeIgnore()` 的语义是"我要忽略租户执行"，两者互不干扰。

## 小结

租户管理与安全的核心设计可以概括为：

1. **一个接口**：`TenantFrameworkService` 定义了 `getTenantIds()` 和 `validTenant()` 两个最小化方法
2. **两级缓存**：Guava LoadingCache 缓存租户列表和校验结果，1 分钟过期，异步刷新
3. **一个工具类**：`TenantUtils` 提供 `execute()` 和 `executeIgnore()` 两种编程式切换方式
4. **一个注解**：`@TenantJob` 配合 `TenantJobAspect` 自动为定时任务实现多租户遍历

设计原则是：**缓存优先减少数据库压力，工具方法简化业务代码，注解驱动降低使用门槛**。
