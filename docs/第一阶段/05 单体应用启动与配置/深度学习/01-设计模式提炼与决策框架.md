# 设计模式提炼与决策框架

---

## 1. 空壳容器模式

### 1.1 设计

`yudao-server` 不包含任何业务逻辑，它是一个纯粹的组装层：

```java
@SpringBootApplication(scanBasePackages = "${yudao.info.base-package}")
public class YudaoServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(YudaoServerApplication.class, args);
    }
}
```

所有功能通过 Maven 依赖引入。注释/取消注释 pom.xml 中的依赖，就能组装出不同的应用。

### 1.2 核心收益

- **同一代码库，多种部署形态**：只含 system → 系统管理服务；含 system + mall → 电商服务
- **新项目可复用**：fork 后改 `yudao.info.base-package`，删除不需要的模块依赖
- **模块可插拔**：注释掉一个依赖，该模块的所有 Controller、Service、Mapper 都不会被扫描到

### 1.3 动态包扫描

启动类的 `scanBasePackages` 使用属性占位符：

```java
scanBasePackages = {"${yudao.info.base-package}.server", "${yudao.info.base-package}.module"}
```

这意味着改包名只需要改 `yudao.info.base-package` 一个配置项，不需要改启动类。两个扫描路径（`.server` 和 `.module`）避免扫描到 `yudao-common` 和 `yudao-framework` 的内部类。

## 2. DefaultController：通配回退

### 2.1 问题

当模块被禁用后，对应的 API 端点返回 404。用户看到 404 不知道是路径写错了还是模块没安装。

### 2.2 设计

```java
@RestController
public class DefaultController {
    @RequestMapping("/**")
    public CommonResult<?> defaultApi() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(), "模块未安装...");
    }
}
```

当模块启用后，Spring MVC 的路由优先级规则（精确匹配 > 通配匹配）会自动让 `DefaultController` 退位。

### 2.3 为什么不用 @ConditionalOnClass

另一种方案是用 `@ConditionalOnClass` 判断模块是否存在，不存在则注册 501 处理器。但这需要为每个模块写一个条件判断，而通配方案用一个 Controller 覆盖所有情况。

## 3. 哨兵值自动配置

### 3.1 设计

`IdTypeEnvironmentPostProcessor` 在 Spring 容器刷新之前运行，读取 JDBC URL 判断数据库类型，将配置中的 `id-type: NONE`（哨兵值）替换为 `AUTO`（MySQL）或 `INPUT`（PostgreSQL/Oracle）。

```yaml
mybatis-plus:
  global-config:
    db-config:
      id-type: NONE    # 启动时自动替换
```

### 3.2 为什么用 EnvironmentPostProcessor

| 机制 | 执行时机 | 能修改 Environment | 能创建 Bean |
|------|---------|------------------|-----------|
| BeanFactoryPostProcessor | Bean 创建前 | 能 | 不能 |
| EnvironmentPostProcessor | 所有 Bean 创建前 | 能 | 不能 |
| ApplicationListener | 容器刷新后 | 能（但太晚） | 能 |

`EnvironmentPostProcessor` 是最早能修改配置的扩展点，确保所有 Bean 创建时看到的都是替换后的值。

## 4. 配置分层

### 4.1 三层配置

```
application.yaml          ← 公共配置（所有环境共享）
application-local.yaml    ← 本地开发（关闭验证码、Mock 登录、本地数据库）
application-dev.yaml      ← 开发服务器（连接远程基础设施）
```

### 4.2 设计选择：文件分离 vs 条件逻辑

| 方案 | 优点 | 缺点 |
|------|------|------|
| 文件分离（当前选择） | 清晰，每个环境一个文件 | 文件数量多 |
| 单文件 + `@Profile` 条件 | 文件少 | 复杂，容易出错 |

yudao-cloud 选择了文件分离。当环境超过 3 个时，文件分离的优势更明显——每个文件独立可读，不需要在脑中展开条件逻辑。

## 5. VO 命名规范

### 5.1 五种 VO

| 类型 | 命名 | 用途 | 字段数量 |
|------|------|------|---------|
| 创建/修改请求 | `*SaveReqVO` | POST 创建和 PUT 修改共用 | 必填字段 |
| 分页查询请求 | `*PageReqVO` | 分页查询参数 | 可选过滤字段 |
| 列表查询请求 | `*ListReqVO` | 不分页的列表查询 | 可选过滤字段 |
| 完整响应 | `*RespVO` | 返回给前端的完整数据 | 全部字段 |
| 简单响应 | `*SimpleRespVO` | 下拉框等轻量场景 | ID + 名称 |

### 5.2 为什么不用一个 VO 做所有事

"God VO"反模式：一个 `*VO` 既做请求又做响应，既有分页参数又有业务字段。

问题：
- 前端不知道哪些字段是请求用的，哪些是响应用的
- 修改一个场景可能影响其他场景
- 分页查询的参数和响应字段混在一起，无法用 `@Valid` 做精确校验

### 5.3 SaveReqVO 同时用于创建和修改

创建和修改共用一个 VO（而不是 `CreateReqVO` + `UpdateReqVO`），因为两者的字段几乎完全相同，区别仅在于修改时有 `id` 字段。用 `@NotNull(groups = Update.class)` 标记 `id` 字段即可区分。

## 小结

| 模式/决策 | 核心思想 |
|----------|---------|
| 空壳容器 | 启动类不含逻辑，通过依赖组装应用 |
| 通配回退 | 利用路由优先级做优雅降级 |
| 哨兵值 | 配置用特殊值标记，启动时自动替换 |
| 配置文件分离 | 每个环境一个文件，避免条件逻辑 |
| 严格 VO 命名 | 每种场景一个 VO 类型，拒绝 God VO |
