# 第一个完整业务：部门管理

> 本章是"单体应用启动与配置"功能实现的最后一篇。前面四章分别讲了启动类、配置体系、模块组装和业务模块结构。本章将这些知识串联起来，**端到端地走一遍"部门管理"这个完整业务的实现过程**，让读者看到第一阶段所有组件是如何协同工作的。

## 1. 我们要实现什么

部门管理是企业后台系统中最基础的功能之一。一个电商公司的典型部门结构：

```
芋道科技（根部门）
├── 电商运营部
│   ├── 商品管理组
│   ├── 订单处理组
│   └── 客户服务组
├── 技术研发部
│   ├── 后端开发组
│   └── 前端开发组
└── 市场营销部
```

我们需要实现的 API：

| 操作 | HTTP 方法 | 路径 | 权限 |
|------|----------|------|------|
| 创建部门 | POST | `/admin-api/system/dept/create` | `system:dept:create` |
| 更新部门 | PUT | `/admin-api/system/dept/update` | `system:dept:update` |
| 删除部门 | DELETE | `/admin-api/system/dept/delete` | `system:dept:delete` |
| 查询部门列表 | GET | `/admin-api/system/dept/list` | `system:dept:query` |
| 查询部门详情 | GET | `/admin-api/system/dept/get` | `system:dept:query` |
| 查询精简列表 | GET | `/admin-api/system/dept/list-all-simple` | 无需权限 |

## 2. 从零开始的实现步骤

假设我们要从零实现这个功能，按照项目的分层规范，步骤是：

```
Step 1: 定义 DO 实体       → 映射数据库表结构
Step 2: 创建 Mapper        → 数据访问层
Step 3: 定义 VO 对象        → 请求参数和响应数据
Step 4: 实现 Service       → 业务逻辑
Step 5: 编写 Controller    → HTTP 接口
```

下面逐步讲解每一步。由于这些代码已经在项目中存在（上一章已详细展示），这里聚焦于**每一步的设计决策和实现要点**。

### Step 1: 定义 DO 实体

首先，需要有一张 `system_dept` 数据库表。对应的 DO 实体：

```java
// 来源：yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/dal/dataobject/dept/DeptDO.java
@TableName("system_dept")
@KeySequence("system_dept_seq")
@Data
@EqualsAndHashCode(callSuper = true)
public class DeptDO extends TenantBaseDO {

    public static final Long PARENT_ID_ROOT = 0L;  // 根部门的 parentId

    @TableId
    private Long id;
    private String name;
    private Long parentId;
    private Integer sort;
    private Long leaderUserId;
    private String phone;
    private String email;
    private Integer status;

}
```

**设计决策**：

1. **继承 `TenantBaseDO` 而非 `BaseDO`**：因为部门数据需要按租户隔离。`TenantBaseDO` 在 `BaseDO` 基础上增加了 `tenantId` 字段，配合多租户拦截器，自动在 SQL 中追加 `WHERE tenant_id = ?`

2. **定义 `PARENT_ID_ROOT = 0L`**：根部门的 `parentId` 用 0 表示，而不是 null。这样可以用统一的字段查询所有层级的部门

3. **`@KeySequence` 注解**：为 Oracle、PostgreSQL 等数据库提供主键序列支持。MySQL 不需要，但加上也无害——这是多数据库兼容的设计

### Step 2: 创建 Mapper

```java
// 来源：yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/dal/mysql/dept/DeptMapper.java
@Mapper
public interface DeptMapper extends BaseMapperX<DeptDO> {

    default List<DeptDO> selectList(DeptListReqVO reqVO) {
        return selectList(new LambdaQueryWrapperX<DeptDO>()
                .likeIfPresent(DeptDO::getName, reqVO.getName())
                .eqIfPresent(DeptDO::getStatus, reqVO.getStatus()));
    }

    default DeptDO selectByParentIdAndName(Long parentId, String name) {
        return selectOne(DeptDO::getParentId, parentId, DeptDO::getName, name);
    }

    default Long selectCountByParentId(Long parentId) {
        return selectCount(DeptDO::getParentId, parentId);
    }

    default List<DeptDO> selectListByParentId(Collection<Long> parentIds) {
        return selectList(DeptDO::getParentId, parentIds);
    }

    default List<DeptDO> selectListByLeaderUserId(Long id) {
        return selectList(DeptDO::getLeaderUserId, id);
    }

}
```

**设计决策**：

1. **继承 `BaseMapperX`**：获得增强的 CRUD 方法（分页查询、批量操作、便捷查询等），而不需要写 XML

2. **`default` 方法 + `LambdaQueryWrapperX`**：查询逻辑直接在接口中实现。`*IfPresent` 系列方法（如 `likeIfPresent`、`eqIfPresent`）在参数为 null 时自动跳过条件，避免了传统 MyBatis 中大量的 `<if test>` XML

3. **按业务场景命名方法**：`selectByParentIdAndName`、`selectCountByParentId` 等方法名直接表达业务含义，比通用的 `selectList(wrapper)` 更易读

### Step 3: 定义 VO 对象

根据业务需要，定义三种 VO：

**DeptSaveReqVO**（创建/修改请求）：

```java
// 来源：yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/dept/vo/dept/DeptSaveReqVO.java
@Schema(description = "管理后台 - 部门创建/修改 Request VO")
@Data
public class DeptSaveReqVO {

    @Schema(description = "部门编号", example = "1024")
    private Long id;  // 修改时有值，创建时为空

    @Schema(description = "部门名称", requiredMode = Schema.RequiredMode.REQUIRED, example = "芋道")
    @NotBlank(message = "部门名称不能为空")
    @Size(max = 30, message = "部门名称长度不能超过 30 个字符")
    private String name;

    @Schema(description = "父部门 ID", example = "1024")
    private Long parentId;

    @Schema(description = "显示顺序", requiredMode = Schema.RequiredMode.REQUIRED, example = "1024")
    @NotNull(message = "显示顺序不能为空")
    private Integer sort;

    @Schema(description = "负责人的用户编号", example = "2048")
    private Long leaderUserId;

    @Schema(description = "联系电话", example = "15601691000")
    @Size(max = 11, message = "联系电话长度不能超过11个字符")
    private String phone;

    @Schema(description = "邮箱", example = "yudao@iocoder.cn")
    @Email(message = "邮箱格式不正确")
    @Size(max = 50, message = "邮箱长度不能超过 50 个字符")
    private String email;

    @Schema(description = "状态,见 CommonStatusEnum 枚举", requiredMode = Schema.RequiredMode.REQUIRED, example = "1")
    @NotNull(message = "状态不能为空")
    @InEnum(value = CommonStatusEnum.class, message = "修改状态必须是 {value}")
    private Integer status;

}
```

**设计决策**：

1. **创建和修改共用一个 VO**：`id` 字段在创建时为空，修改时有值。这样 Controller 只需要一个 `@RequestBody` 参数

2. **使用 JSR-303 校验注解**：`@NotBlank`、`@NotNull`、`@Size`、`@Email`、`@InEnum` 等注解配合 `@Valid` 实现参数自动校验

3. **`@Schema` 注解**：为 Swagger 文档提供字段描述、示例值、是否必填等信息

### Step 4: 实现 Service

```java
// 来源：yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/service/dept/DeptServiceImpl.java
@Service
@Validated
@Slf4j
public class DeptServiceImpl implements DeptService {

    @Resource
    private DeptMapper deptMapper;

    @Override
    @CacheEvict(cacheNames = RedisKeyConstants.DEPT_CHILDREN_ID_LIST, allEntries = true)
    public Long createDept(DeptSaveReqVO createReqVO) {
        if (createReqVO.getParentId() == null) {
            createReqVO.setParentId(DeptDO.PARENT_ID_ROOT);
        }
        // 1. 校验父部门的有效性
        validateParentDept(null, createReqVO.getParentId());
        // 2. 校验部门名的唯一性
        validateDeptNameUnique(null, createReqVO.getParentId(), createReqVO.getName());
        // 3. VO → DO 转换并插入
        DeptDO dept = BeanUtils.toBean(createReqVO, DeptDO.class);
        deptMapper.insert(dept);
        return dept.getId();
    }

    @Override
    @CacheEvict(cacheNames = RedisKeyConstants.DEPT_CHILDREN_ID_LIST, allEntries = true)
    public void updateDept(DeptSaveReqVO updateReqVO) {
        if (updateReqVO.getParentId() == null) {
            updateReqVO.setParentId(DeptDO.PARENT_ID_ROOT);
        }
        // 1. 校验自己存在
        validateDeptExists(updateReqVO.getId());
        // 2. 校验父部门的有效性
        validateParentDept(updateReqVO.getId(), updateReqVO.getParentId());
        // 3. 校验部门名的唯一性
        validateDeptNameUnique(updateReqVO.getId(), updateReqVO.getParentId(), updateReqVO.getName());
        // 4. 更新
        DeptDO updateObj = BeanUtils.toBean(updateReqVO, DeptDO.class);
        deptMapper.updateById(updateObj);
    }

    @Override
    @CacheEvict(cacheNames = RedisKeyConstants.DEPT_CHILDREN_ID_LIST, allEntries = true)
    public void deleteDept(Long id) {
        // 1. 校验存在
        validateDeptExists(id);
        // 2. 校验无子部门
        if (deptMapper.selectCountByParentId(id) > 0) {
            throw exception(DEPT_EXITS_CHILDREN);
        }
        // 3. 删除
        deptMapper.deleteById(id);
    }

    @Override
    public List<DeptDO> getDeptList(DeptListReqVO reqVO) {
        List<DeptDO> list = deptMapper.selectList(reqVO);
        list.sort(Comparator.comparing(DeptDO::getSort));
        return list;
    }

    // ... 其他方法省略，完整代码见上一章
}
```

**Service 层的核心职责**：

1. **业务校验**：父部门有效性、名称唯一性、是否有子部门等
2. **VO → DO 转换**：`BeanUtils.toBean(createReqVO, DeptDO.class)`
3. **调用 Mapper**：执行数据库操作
4. **缓存管理**：增删改操作清空缓存，查询操作使用缓存
5. **异常抛出**：校验失败时通过 `ServiceExceptionUtil.exception()` 抛出业务异常

### Step 5: 编写 Controller

```java
// 来源：yudao-module-system-server/src/main/java/cn/iocoder/yudao/module/system/controller/admin/dept/DeptController.java
@Tag(name = "管理后台 - 部门")
@RestController
@RequestMapping("/system/dept")
@Validated
public class DeptController {

    @Resource
    private DeptService deptService;

    @PostMapping("create")
    @Operation(summary = "创建部门")
    @PreAuthorize("@ss.hasPermission('system:dept:create')")
    public CommonResult<Long> createDept(@Valid @RequestBody DeptSaveReqVO createReqVO) {
        Long deptId = deptService.createDept(createReqVO);
        return success(deptId);
    }

    @PutMapping("update")
    @Operation(summary = "更新部门")
    @PreAuthorize("@ss.hasPermission('system:dept:update')")
    public CommonResult<Boolean> updateDept(@Valid @RequestBody DeptSaveReqVO updateReqVO) {
        deptService.updateDept(updateReqVO);
        return success(true);
    }

    @DeleteMapping("delete")
    @Operation(summary = "删除部门")
    @PreAuthorize("@ss.hasPermission('system:dept:delete')")
    public CommonResult<Boolean> deleteDept(@RequestParam("id") Long id) {
        deptService.deleteDept(id);
        return success(true);
    }

    @GetMapping("/list")
    @Operation(summary = "获取部门列表")
    @PreAuthorize("@ss.hasPermission('system:dept:query')")
    public CommonResult<List<DeptRespVO>> getDeptList(DeptListReqVO reqVO) {
        List<DeptDO> list = deptService.getDeptList(reqVO);
        return success(BeanUtils.toBean(list, DeptRespVO.class));
    }

    @GetMapping(value = {"/list-all-simple", "/simple-list"})
    @Operation(summary = "获取部门精简信息列表")
    public CommonResult<List<DeptSimpleRespVO>> getSimpleDeptList() {
        List<DeptDO> list = deptService.getDeptList(
                new DeptListReqVO().setStatus(CommonStatusEnum.ENABLE.getStatus()));
        return success(BeanUtils.toBean(list, DeptSimpleRespVO.class));
    }

    @GetMapping("/get")
    @Operation(summary = "获得部门信息")
    @PreAuthorize("@ss.hasPermission('system:dept:query')")
    public CommonResult<DeptRespVO> getDept(@RequestParam("id") Long id) {
        DeptDO dept = deptService.getDept(id);
        return success(BeanUtils.toBean(dept, DeptRespVO.class));
    }

}
```

**Controller 层的核心职责**：

1. **参数接收**：`@RequestBody` 接收 JSON，`@RequestParam` 接收查询参数
2. **参数校验**：`@Valid` 触发 JSR-303 校验
3. **权限校验**：`@PreAuthorize("@ss.hasPermission('...')")` 检查权限
4. **调用 Service**：传递参数，获取结果
5. **DO → VO 转换**：`BeanUtils.toBean(list, DeptRespVO.class)`
6. **包装返回**：`success()` 包装为 `CommonResult`

## 4. 一个请求的完整生命周期

让我们以"创建部门"为例，追踪从前端到数据库的完整链路：

```
1. 前端发送请求
   POST /admin-api/system/dept/create
   Headers: Authorization: Bearer eyJhbG...
   Body: {"name": "电商运营部", "parentId": 100, "sort": 1, "status": 1}

2. URL 前缀注入（YudaoWebAutoConfiguration）
   /admin-api/system/dept/create → 路由到 DeptController
   （因为 DeptController 在 controller.admin 包下，自动加上 /admin-api 前缀）

3. Spring MVC 路由匹配
   匹配到 DeptController.createDept() 方法

4. 参数校验（@Valid）
   Spring MVC 自动校验 DeptSaveReqVO 中的 @NotBlank、@NotNull 等约束
   如果校验失败，直接返回 400 错误

5. 权限校验（@PreAuthorize）
   @ss.hasPermission('system:dept:create') 检查当前用户的权限
   如果没有权限，返回 401 错误

6. Controller 调用 Service
   Long deptId = deptService.createDept(createReqVO);

7. Service 业务逻辑
   a. 设置默认 parentId（如果为空则设为 0）
   b. validateParentDept() — 校验父部门存在且不会形成环路
   c. validateDeptNameUnique() — 校验同级部门名称不重复
   d. BeanUtils.toBean(createReqVO, DeptDO.class) — VO → DO
   e. deptMapper.insert(dept) — 插入数据库

8. Mapper 执行 SQL
   INSERT INTO system_dept (name, parent_id, sort, status, tenant_id, creator, create_time, deleted)
   VALUES ('电商运营部', 100, 1, 1, 1, '1', 1611460870401, 0)

9. 数据库返回
   dept.getId() = 1024（自增主键）

10. Controller 包装返回
    return success(1024)
    → CommonResult<Long>

11. 序列化响应
    {"code": 0, "data": 1024, "msg": "success"}
```

## 5. 第一阶段所有组件的协同

部门管理这个看似简单的功能，实际上用到了第一阶段所有五个组件的知识：

```
第一阶段组件在部门管理中的体现：

01 Maven 多模块结构
   └── yudao-module-system-server 的 pom.xml 定义了对 mybatis、security 等 starter 的依赖

02 共享基础类型与工具
   ├── CommonResult.success(deptId)          ← 统一响应体
   ├── ServiceExceptionUtil.exception()      ← 统一异常抛出
   └── BeanUtils.toBean(vo, do.class)        ← 对象转换工具

03 数据访问增强
   ├── DeptDO extends TenantBaseDO           ← 实体基类（审计字段 + 租户ID）
   ├── DeptMapper extends BaseMapperX        ← 增强 Mapper（便捷查询、批量操作）
   └── LambdaQueryWrapperX.likeIfPresent()   ← 条件构建器（自动跳过空值）

04 Web 层基础设施
   ├── /admin-api 前缀自动注入               ← URL 前缀注入机制
   ├── GlobalExceptionHandler                ← 全局异常处理（ServiceException → CommonResult）
   └── @Operation + @Tag                     ← Swagger 文档自动生成

05 单体应用启动与配置
   ├── YudaoServerApplication                ← 启动入口，扫描 system 模块的 Bean
   ├── application.yaml + application-local.yaml  ← 配置数据库、Redis 等
   └── pom.xml 引入 yudao-module-system-server    ← 模块组装
```

这就是为什么第一阶段要按这个顺序学习——每个组件都在为下一个组件打基础，最终在第五个组件（单体应用启动与配置）中汇聚成一个可运行的应用。

## 6. 扩展：如何新增一个类似的 CRUD 功能

理解了部门管理的实现后，你可以按照相同的模式新增任何 CRUD 功能。以"商品分类"为例：

```
1. 创建 CategoryDO
   ├── 继承 TenantBaseDO
   ├── 定义字段：id, name, parentId, sort, status, icon, ...
   └── @TableName("product_category")

2. 创建 CategoryMapper
   ├── 继承 BaseMapperX<CategoryDO>
   └── 定义查询方法：selectList, selectByParentIdAndName, ...

3. 创建 VO 对象
   ├── CategorySaveReqVO（创建/修改请求）
   ├── CategoryListReqVO（列表查询条件）
   ├── CategoryRespVO（完整响应）
   └── CategorySimpleRespVO（精简响应，用于下拉框）

4. 实现 CategoryService + CategoryServiceImpl
   ├── 校验逻辑：父分类有效性、名称唯一性
   ├── VO → DO 转换
   └── 缓存管理

5. 编写 CategoryController
   ├── @RequestMapping("/product/category")
   ├── @PreAuthorize 权限校验
   └── DO → VO 转换
```

这就是项目中所有 CRUD 功能的统一模式。掌握了这个模式，你就掌握了整个项目的开发范式。

## 本章小结

通过部门管理这个端到端的示例，我们看到了：

1. **分层架构的实践**：DO → Mapper → Service → Controller，每层职责清晰
2. **对象转换的时机**：VO→DO 在 Service 层，DO→VO 在 Controller 层
3. **校验的分层**：JSR-303 校验在 Controller 层，业务校验在 Service 层
4. **前五个组件的协同**：Maven 结构、基础类型、数据访问、Web 基础设施、单体启动，缺一不可
