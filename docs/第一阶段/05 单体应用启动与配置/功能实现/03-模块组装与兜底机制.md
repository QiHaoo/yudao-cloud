# 模块组装与兜底机制

> 本章是"单体应用启动与配置"功能实现的第三篇。在功能设计中，我们分析了模块组装的两个核心挑战：如何通过 Maven 依赖控制模块的启停，以及如何优雅处理未启用模块的请求。
>
> 本章的目标是：**动手实现模块的启用/禁用流程**，理解 DefaultController 的兜底机制和 OpenFeign 的单体排除策略。

## 1. Maven 依赖作为模块开关

### 1.1 模块组装清单

yudao-server 的 `pom.xml` 本质上是一份**模块组装清单**。哪些业务模块被包含，完全取决于 `pom.xml` 中声明了哪些依赖。

默认启用的模块（两个）：

```xml
<!-- 来源：yudao-server/pom.xml -->

<!-- 系统管理模块：用户、角色、菜单、部门、字典等 -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-system-server</artifactId>
    <version>${revision}</version>
</dependency>

<!-- 基础设施模块：代码生成、文件管理、配置管理、操作日志等 -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-infra-server</artifactId>
    <version>${revision}</version>
</dependency>
```

### 1.2 实操：启用商城模块

假设你正在开发电商后台，需要启用商品（product）、交易（trade）和营销（promotion）三个模块。

**第一步**：取消注释对应的依赖

在 `yudao-server/pom.xml` 中，找到被注释的商城模块依赖，取消注释：

```xml
<!-- 取消注释这三个 -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-product-server</artifactId>
    <version>${revision}</version>
</dependency>
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-trade-server</artifactId>
    <version>${revision}</version>
</dependency>
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-promotion-server</artifactId>
    <version>${revision}</version>
</dependency>
```

**第二步**：重新编译

```bash
mvn clean compile -pl yudao-server -am
```

`-pl yudao-server` 只编译 yudao-server 模块，`-am`（also-make）会同时编译它依赖的模块。

**第三步**：重启应用

重新运行 `YudaoServerApplication`，商城模块的 Controller、Service、Mapper 会被自动扫描并注册为 Spring Bean。

### 1.3 为什么用注释而不是 Maven Profile？

Maven 的 Profile 也能实现条件依赖：

```xml
<!-- 另一种方案：用 Maven Profile 控制模块 -->
<profiles>
    <profile>
        <id>mall</id>
        <dependencies>
            <dependency>
                <groupId>cn.iocoder.cloud</groupId>
                <artifactId>yudao-module-product-server</artifactId>
            </dependency>
        </dependencies>
    </profile>
</profiles>
```

但项目选择了更简单的"注释/取消注释"方案：

| 维度 | 注释方案 | Maven Profile 方案 |
|------|---------|-------------------|
| 操作成本 | 改一行注释 | 需要记住 Profile 名称 |
| 可见性 | 打开 pom.xml 一目了然 | 需要查看 activeProfiles |
| 灵活性 | 随意组合 | Profile 通常是预定义的组合 |
| 学习成本 | 零 | 需要理解 Profile 机制 |

对于学习项目来说，**简单直观**比**工程化**更重要。

## 2. DefaultController 兜底机制

### 2.1 问题

当前端请求了一个未启用模块的 API 路径时（比如商城模块未开启，但请求了 `/admin-api/product/**`），Spring MVC 找不到对应的 Controller 方法，默认返回一个 404 错误页面。这对前端开发者非常不友好——他们无法区分"接口写错了"和"模块没开启"。

### 2.2 DefaultController 实现

```java
// 来源：yudao-server/src/main/java/cn/iocoder/yudao/server/controller/DefaultController.java
package cn.iocoder.yudao.server.controller;

import cn.iocoder.yudao.framework.common.pojo.CommonResult;
import cn.iocoder.yudao.framework.common.util.servlet.ServletUtils;
import lombok.extern.slf4j.Slf4j;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

import javax.annotation.security.PermitAll;
import javax.servlet.http.HttpServletRequest;

import static cn.iocoder.yudao.framework.common.exception.enums.GlobalErrorCodeConstants.NOT_IMPLEMENTED;

/**
 * 默认 Controller，解决部分 module 未开启时的 404 提示。
 */
@RestController
@Slf4j
public class DefaultController {

    @RequestMapping("/admin-api/bpm/**")
    public CommonResult<Boolean> bpm404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[工作流模块 yudao-module-bpm - 已禁用][参考 https://doc.iocoder.cn/bpm/ 开启]");
    }

    @RequestMapping("/admin-api/mp/**")
    public CommonResult<Boolean> mp404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[微信公众号 yudao-module-mp - 已禁用][参考 https://doc.iocoder.cn/mp/build/ 开启]");
    }

    @RequestMapping(value = { "/admin-api/product/**", // 商品中心
            "/admin-api/trade/**", // 交易中心
            "/admin-api/promotion/**" }) // 营销中心
    public CommonResult<Boolean> mall404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[商城系统 yudao-module-mall - 已禁用][参考 https://doc.iocoder.cn/mall/build/ 开启]");
    }

    @RequestMapping("/admin-api/erp/**")
    public CommonResult<Boolean> erp404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[ERP 模块 yudao-module-erp - 已禁用][参考 https://doc.iocoder.cn/erp/build/ 开启]");
    }

    @RequestMapping("/admin-api/crm/**")
    public CommonResult<Boolean> crm404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[CRM 模块 yudao-module-crm - 已禁用][参考 https://doc.iocoder.cn/crm/build/ 开启]");
    }

    @RequestMapping("/admin-api/ai/**")
    public CommonResult<Boolean> ai404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[AI 大模型 yudao-module-ai - 已禁用][参考 https://doc.iocoder.cn/ai/build/ 开启]");
    }

    @RequestMapping("/admin-api/iot/**")
    public CommonResult<Boolean> iot404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[IoT 物联网 yudao-module-iot - 已禁用][参考 https://doc.iocoder.cn/iot/build/ 开启]");
    }

    @RequestMapping("/admin-api/pay/**")
    public CommonResult<Boolean> pay404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[支付模块 yudao-module-pay - 已禁用][参考 https://doc.iocoder.cn/pay/build/ 开启]");
    }

    @RequestMapping("/admin-api/report/**")
    public CommonResult<Boolean> report404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[报表模块 yudao-module-report - 已禁用][参考 https://doc.iocoder.cn/report/ 开启]");
    }

    /**
     * 测试接口：打印 query、header、body
     */
    @RequestMapping(value = { "/test" })
    @PermitAll
    public CommonResult<Boolean> test(HttpServletRequest request) {
        log.info("Query: {}", ServletUtils.getParamMap(request));
        log.info("Header: {}", ServletUtils.getHeaderMap(request));
        log.info("Body: {}", ServletUtils.getBody(request));
        return CommonResult.success(true);
    }

}
```

### 2.3 工作原理

DefaultController 利用了 Spring MVC 的路由匹配规则：**精确路径匹配优先于通配符匹配**。

**当模块未启用时**：

```
前端请求: POST /admin-api/product/spu/create
    │
    ▼
Spring MVC 路由匹配
    │
    ├─ 找到 ProductSpuController？  → 否（模块未引入，Bean 不存在）
    │
    ▼
匹配到 DefaultController 的 /admin-api/product/** 通配符
    │
    ▼
返回: {"code": 501, "msg": "[商城系统 yudao-module-mall - 已禁用][参考 ...]"}
```

**当模块已启用时**：

```
前端请求: POST /admin-api/product/spu/create
    │
    ▼
Spring MVC 路由匹配
    │
    ├─ 找到 ProductSpuController？  → 是（模块已引入，Bean 存在）
    │   路径: /product/spu/create（经过 URL 前缀注入后为 /admin-api/product/spu/create）
    │
    ▼
精确匹配优先于通配符匹配，路由到 ProductSpuController
    │
    ▼
正常处理请求
```

### 2.4 为什么放在 yudao-server 而不是 yudao-common？

`DefaultController` 只在单体模式下有意义。微服务模式下，每个服务独立部署，不存在"模块未启用"的情况——未部署的服务根本不会有请求进来。把它放在 yudao-server 中，恰好只在单体启动时生效。

## 3. OpenFeign 单体排除

### 3.1 问题

业务模块的 API 层（如 `yudao-module-system-api`）使用了 `@FeignClient` 注解来定义 RPC 接口。在微服务模式下，这个注解会让 Spring Cloud 创建一个 HTTP 代理，通过 Nacos 发现服务地址。

但在单体模式下，所有模块都在同一个 JVM 中，Bean 之间可以直接注入，不需要 RPC。如果 OpenFeign 仍然生效，它会尝试去 Nacos 查找服务——但 Nacos 根本没有启用，导致启动失败。

### 3.2 解决方案：Maven 排除

```xml
<!-- 来源：yudao-server/pom.xml -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-spring-boot-starter-rpc</artifactId>
    <!-- 目的：yudao-server 单体启动，禁用 openfeign -->
    <exclusions>
        <exclusion>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-starter-openfeign</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

通过排除 `spring-cloud-starter-openfeign`，OpenFeign 的自动配置不会加载，`@FeignClient` 注解变成了一个无害的"标记"。

### 3.3 单体模式下的调用链路

在单体模式下，当业务代码需要调用 `DeptApi` 时：

```
调用方代码（如 trade 模块的 OrderService）
    │
    ▼
注入 DeptApi Bean
    │
    ├─ 微服务模式：注入的是 Feign 代理 → HTTP 远程调用
    │
    └─ 单体模式：注入的是 DeptApi 的直接实现 Bean → 本地方法调用
```

`DeptApi` 是一个接口，单体模式下由业务模块的 `server` 子模块中的 `DeptApiImpl` 类提供实现：

```java
// 典型的 API 实现（在 server 模块中）
@Service
public class DeptApiImpl implements DeptApi {

    @Resource
    private DeptService deptService;

    @Override
    public CommonResult<DeptRespDTO> getDept(Long id) {
        DeptDO dept = deptService.getDept(id);
        return CommonResult.success(BeanUtils.toBean(dept, DeptRespDTO.class));
    }
}
```

这就是为什么同一份代码能同时支持两种部署模式——接口定义在 `api` 模块，实现在 `server` 模块，调用方只依赖 `api` 模块。

## 4. 完整的模块启停操作手册

### 4.1 启用一个新模块

以启用"支付模块"为例：

**第一步**：在 `yudao-server/pom.xml` 中取消注释

```xml
<!-- 找到被注释的支付模块依赖，取消注释 -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-pay-server</artifactId>
    <version>${revision}</version>
</dependency>
```

**第二步**：检查 DefaultController

确认 `DefaultController` 中有 `/admin-api/pay/**` 的兜底映射。当模块启用后，支付模块自己的 Controller 中的精确路径会自动覆盖这个通配符。

**第三步**：检查配置

某些模块可能需要额外的配置项（如支付模块需要配置回调地址）：

```yaml
# 在 application-local.yaml 中添加
yudao:
  pay:
    order-notify-url: http://localhost:48080/admin-api/pay/notify/order
    refund-notify-url: http://localhost:48080/admin-api/pay/notify/refund
```

**第四步**：重新编译并重启

```bash
mvn clean compile -pl yudao-server -am
```

### 4.2 禁用一个模块

**第一步**：在 `yudao-server/pom.xml` 中注释掉依赖

```xml
<!-- 注释掉支付模块 -->
<!--        <dependency>-->
<!--            <groupId>cn.iocoder.cloud</groupId>-->
<!--            <artifactId>yudao-module-pay-server</artifactId>-->
<!--            <version>${revision}</version>-->
<!--        </dependency>-->
```

**第二步**：重新编译并重启

不需要修改其他任何文件。DefaultController 中的 `/admin-api/pay/**` 兜底映射会自动生效。

### 4.3 模块间依赖关系

需要注意的是，某些模块之间存在依赖关系。例如：

```
yudao-module-trade-server
    └── 依赖 yudao-module-product-api（商品 API）
    └── 依赖 yudao-module-promotion-api（营销 API）
```

启用 trade 模块时，必须同时启用 product 和 promotion 模块（或者至少启用它们的 api 模块）。如果只启用 trade 而不启用 product，启动时会报 Bean 找不到的错误。

## 5. 单体模式与微服务模式的对比

| 维度 | 单体模式 (yudao-server) | 微服务模式 (各 *-server) |
|------|------------------------|------------------------|
| 启动类 | `YudaoServerApplication` | 各模块自己的启动类 |
| 组件扫描 | `scanBasePackages` 统一扫描 | 各服务只扫描自己的包 |
| 模块包含 | 通过 pom.xml 依赖控制 | 每个服务独立部署 |
| 模块间调用 | 直接注入 Bean | OpenFeign RPC |
| OpenFeign | 排除，不加载 | 启用，通过 Nacos 发现 |
| DefaultController | 需要，兜底未启用模块 | 不需要 |
| Nacos | 可选（local 模式不启用） | 必须 |

## 本章小结

模块组装的核心机制：

1. **pom.xml 依赖 = 模块开关**：声明依赖就启用，注释依赖就禁用
2. **DefaultController = 优雅降级**：未启用模块的请求返回 501 而非 404
3. **OpenFeign 排除 = 单体兼容**：排除 Feign 依赖，`@FeignClient` 变成无害标记
4. **`*ApiImpl` = 单体模式的 RPC 替代**：API 接口由本地 Bean 提供实现
