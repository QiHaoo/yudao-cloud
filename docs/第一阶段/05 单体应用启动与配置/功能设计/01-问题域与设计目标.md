# 问题域与设计目标

## 从一个真实场景说起

假设你是一个电商后台系统的开发者。经过前几个阶段的努力，你已经搭建好了：
- 一个 Maven 多模块的项目骨架（第一阶段）
- 一套统一的响应体、异常体系和工具库（第二阶段）
- 数据访问层的增强组件：BaseDO、BaseMapperX、LambdaQueryWrapperX（第三阶段）
- Web 层的基础设施：全局异常处理、URL 前缀注入、Swagger 文档（第四阶段）

这些组件散落在 `yudao-common`、`yudao-framework` 等模块中。现在问题来了：**谁来把它们组装在一起，启动成一个可以运行的应用？**

你可能会想："写一个 `main` 方法不就行了？" 事情远没有这么简单。让我们一步步分析，你会发现一个"启动类"背后隐藏着四个关键的设计挑战。

## 挑战一：多模块项目的组件扫描

### 问题

在单体 Spring Boot 项目中，`@SpringBootApplication` 默认扫描启动类所在的包及其子包。比如启动类在 `com.example` 下，就会自动扫描 `com.example.**` 的所有 Bean。

但 yudao-cloud 是一个多模块项目。业务模块的代码分布在不同的 Maven 模块中：

```
yudao-module-system-server/    → cn.iocoder.yudao.module.system.**
yudao-module-infra-server/     → cn.iocoder.yudao.module.infra.**
yudao-module-product-server/   → cn.iocoder.yudao.module.product.**
```

启动类 `YudaoServerApplication` 在 `cn.iocoder.yudao.server` 包下。如果只扫描这个包，所有业务模块的 Controller、Service、Mapper 都不会被发现。

### 设计目标

需要一种机制，能**自动发现并加载所有已引入的业务模块的 Bean**，而且不需要在启动类上逐个列出每个模块的包名。

## 挑战二：模块的按需组装

### 问题

yudao-cloud 包含十几个业务模块：system、infra、member、pay、bpm、product、trade、crm、erp、iot、ai... 但不是每个项目都需要全部模块。

一个电商后台可能只需要 system + infra + product + trade + promotion。一个 CRM 系统可能只需要 system + infra + crm。

更现实的问题是：**模块越多，启动越慢、编译越久**。开发阶段你可能只关心一两个模块，不想每次都把所有模块编译一遍。

### 设计目标

需要一种**轻量级的模块开关机制**，让开发者通过简单的操作（最好是改一行配置）就能决定包含或排除哪些业务模块。

## 挑战三：多环境配置管理

### 问题

同一个应用在不同环境下的配置差异巨大：

| 环境 | 数据库 | Redis | Nacos | 特殊行为 |
|------|--------|-------|-------|---------|
| 本地开发 (local) | 本机 MySQL | 本机 Redis | 不启用 | 关闭验证码、开启 mock 登录 |
| 开发环境 (dev) | 远程 MySQL | 远程 Redis | 启用 | 开启验证码 |
| 生产环境 (prod) | 生产数据库 | 生产 Redis | 启用 | 严格安全策略 |

如果把所有配置写在一个文件里，用注释切换，既容易出错又不安全（生产密码可能泄露到代码仓库）。

### 设计目标

需要一套**分层的配置体系**：公共配置放一处，环境差异配置按 Profile 隔离，敏感信息通过环境变量或配置中心注入。

## 挑战四：单体与微服务的双模式运行

### 问题

yudao-cloud 的设计目标是**同一套代码既能以单体模式运行，也能以微服务模式运行**。

- **单体模式**：所有模块跑在一个 JVM 进程里，模块之间直接调用 Bean 方法，不需要 RPC
- **微服务模式**：每个模块独立部署为一个服务，模块之间通过 OpenFeign + Nacos 进行 RPC 调用

这意味着同样的 `DeptApi` 接口，在单体模式下应该直接注入实现类的 Bean，在微服务模式下应该通过 Feign 客户端代理去远程调用。如果处理不好，单体启动时会因为找不到 Feign 服务注册而报错。

### 设计目标

需要一套机制，让**同一份业务代码在不同部署模式下透明切换调用方式**，开发者不需要写两套代码。

## 整体设计目标汇总

把四个挑战归纳为一句话：我们需要一个 **"空壳容器"** 模块，它的职责是：

```
┌─────────────────────────────────────────────────────────────┐
│                     yudao-server（空壳）                      │
│                                                             │
│  1. 作为启动入口 → 提供 main 方法，配置组件扫描               │
│  2. 作为组装清单 → 通过 Maven 依赖声明要包含哪些业务模块       │
│  3. 作为配置中心 → 管理公共配置和环境差异配置                  │
│  4. 作为适配层  → 处理单体/微服务模式的差异                   │
│                                                             │
│  它本身不包含任何业务逻辑，所有业务都来自被引入的模块。         │
└─────────────────────────────────────────────────────────────┘
```

用一个类比来说：如果各个业务模块是"器官"，那 yudao-server 就是"躯体"——它把器官组装在一起，提供血液循环（配置）和神经系统（组件扫描），让整个身体能运转起来。

## 与其他阶段的关系

本组件是第一阶段的**终点**，也是整个项目的**启动点**。前面四个组件搭建的基础设施，最终都在这里汇聚：

```
第一阶段组件关系图：

  01 Maven 多模块结构 ──→ 定义了项目的骨架和模块划分
          │
  02 共享基础类型与工具 ──→ 提供了 CommonResult、异常体系、BeanUtils
          │
  03 数据访问增强 ──────→ 提供了 BaseDO、BaseMapperX、LambdaQueryWrapperX
          │
  04 Web 层基础设施 ────→ 提供了全局异常处理、URL 前缀注入、Swagger
          │
          ▼
  05 单体应用启动与配置 ← 把以上所有组件组装成一个可运行的应用
```

## 本章思考题

1. 为什么不在每个业务模块（如 `yudao-module-system-server`）里都放一个启动类，而是要单独建一个 `yudao-server` 模块？如果每个模块都有启动类，会带来什么问题？

2. 假设你在做一个电商后台，只需要 system、infra、product、trade 四个模块。如果采用"在启动类上手动列出每个模块包名"的方案，当团队决定把 product 模块的包名从 `cn.iocoder.yudao.module.product` 改成 `cn.iocoder.yudao.module.goods` 时，你需要修改几处代码？这种方案的维护成本如何？

3. Spring Boot 的 `spring.profiles.active` 机制已经提供了环境切换能力，为什么我们还需要额外的 `application-local.yaml` 和 `application-dev.yaml` 文件，而不是把所有配置都写在 `application.yaml` 里用条件判断？
