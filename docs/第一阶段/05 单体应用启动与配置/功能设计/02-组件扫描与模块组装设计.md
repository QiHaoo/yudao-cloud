# 组件扫描与模块组装设计

## 从一个最小的启动类说起

让我们先看看 `YudaoServerApplication` 的完整代码：

```java
// 来源：yudao-server/src/main/java/cn/iocoder/yudao/server/YudaoServerApplication.java
@SuppressWarnings("SpringComponentScan") // 忽略 IDEA 无法识别 ${yudao.info.base-package}
@SpringBootApplication(scanBasePackages = {"${yudao.info.base-package}.server", "${yudao.info.base-package}.module"},
        excludeName = {
            // RPC 相关
//            "org.springframework.cloud.openfeign.FeignAutoConfiguration",
//            "cn.iocoder.yudao.module.system.framework.rpc.config.RpcConfiguration"
        })
public class YudaoServerApplication {

    public static void main(String[] args) {
        SpringApplication.run(YudaoServerApplication.class, args);
    }

}
```

短短几行代码，却包含了三个关键设计决策。让我们逐一拆解。

## 设计一：动态包扫描路径

### scanBasePackages 的魔法

```java
scanBasePackages = {"${yudao.info.base-package}.server", "${yudao.info.base-package}.module"}
```

这里的 `${yudao.info.base-package}` 不是硬编码的包名，而是一个**配置属性占位符**。它对应的配置值在 `application.yaml` 中：

```yaml
yudao:
  info:
    base-package: cn.iocoder.yudao
```

Spring Boot 在启动时会解析这个占位符，最终生效的扫描路径是：

```
cn.iocoder.yudao.server   → 扫描 yudao-server 模块自身的 Bean（如 DefaultController）
cn.iocoder.yudao.module   → 扫描所有业务模块的 Bean
```

### 为什么用配置属性而不是硬编码？

你可能会问：项目包名 `cn.iocoder.yudao` 不会变，为什么不直接写死？

```java
// 看起来更简单，为什么不用这种方式？
scanBasePackages = {"cn.iocoder.yudao.server", "cn.iocoder.yudao.module"}
```

原因在于**一键改包**的需求。当有人需要把项目作为自己的脚手架，将包名从 `cn.iocoder.yudao` 改成 `com.mycompany.myproject` 时，只需要修改 `yudao.info.base-package` 这一处配置，而不需要去改启动类的代码。

这是一个典型的**"一处配置，全局生效"**的设计思路。

### 扫描范围解析

让我们看看 `cn.iocoder.yudao.module` 这个路径能覆盖什么：

```
cn.iocoder.yudao.module
├── system                          ← yudao-module-system-server
│   ├── controller.admin.**        ← Controller 层（管理后台 API）
│   ├── controller.app.**          ← Controller 层（用户 App API）
│   ├── service.**                 ← Service 层
│   ├── dal.dataobject.**         ← DO 实体
│   ├── dal.mysql.**              ← Mapper 层
│   └── convert.**                 ← 对象转换器
├── infra                           ← yudao-module-infra-server
│   ├── controller.admin.**
│   ├── service.**
│   └── ...
├── product                         ← yudao-module-product-server（如果引入）
│   └── ...
└── ...                             ← 其他业务模块
```

关键点：**扫描路径 `cn.iocoder.yudao.module` 是一个公共前缀**。只要业务模块的代码放在这个前缀下，无论有多少个模块，都会被自动发现。这就是为什么不需要在启动类上逐个列出模块包名。

### IDEA 的 @SuppressWarnings

```java
@SuppressWarnings("SpringComponentScan")
```

IDEA 无法识别 `${yudao.info.base-package}` 这种属性占位符（它在编译期无法解析），会报一个"无法解析包名"的警告。这个注解只是告诉 IDEA "我知道我在做什么，别报警告了"，不影响实际运行。

## 设计二：Maven 依赖作为模块开关

### 机制

yudao-server 的 `pom.xml` 就是一份**模块组装清单**。哪些业务模块被包含，完全取决于 `pom.xml` 中声明了哪些依赖：

```xml
<!-- 来源：yudao-server/pom.xml -->

<!-- 默认启用的模块 -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-system-server</artifactId>
    <version>${revision}</version>
</dependency>
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-module-infra-server</artifactId>
    <version>${revision}</version>
</dependency>

<!-- 默认注释的模块（按需开启） -->
<!-- 会员中心。默认注释，保证编译速度 -->
<!--        <dependency>-->
<!--            <groupId>cn.iocoder.cloud</groupId>-->
<!--            <artifactId>yudao-module-member-server</artifactId>-->
<!--            <version>${revision}</version>-->
<!--        </dependency>-->

<!-- 商城相关模块。默认注释，保证编译速度-->
<!--        <dependency>-->
<!--            <groupId>cn.iocoder.cloud</groupId>-->
<!--            <artifactId>yudao-module-product-server</artifactId>-->
<!--            <version>${revision}</version>-->
<!--        </dependency>-->
```

### 为什么用注释而不是 Profile？

Maven 的 Profile 机制也能实现条件依赖：

```xml
<!-- 另一种方案：用 Maven Profile 控制模块 -->
<profiles>
    <profile>
        <id>mall</id>
        <dependencies>
            <dependency>
                <groupId>cn.iocoder.cloud</groupId>
                <artifactId>yudao-module-product-server</artifactId>
            </dependency>
        </dependencies>
    </profile>
</profiles>
```

但项目选择了更简单的"注释/取消注释"方案，原因是：

| 维度 | 注释方案 | Maven Profile 方案 |
|------|---------|-------------------|
| 操作成本 | 改一行注释 | 需要记住 Profile 名称 |
| 可见性 | 打开 pom.xml 一目了然 | 需要查看 activeProfiles |
| 灵活性 | 随意组合 | Profile 通常是预定义的组合 |
| 学习成本 | 零 | 需要理解 Profile 机制 |

对于一个学习项目来说，**简单直观**比**工程化**更重要。

### 模块不存在时的行为

如果前端请求了一个未启用模块的接口（比如商城模块未开启，但请求了 `/admin-api/product/**`），会发生什么？

答案是：**不会返回 404，而是返回一个友好的错误提示**。这就是 `DefaultController` 的作用。

## 设计三：DefaultController 优雅降级

### 问题

当前端请求了一个未启用模块的 API 路径时，Spring MVC 找不到对应的 Controller 方法，会返回一个默认的 404 错误页面。这对前端开发者非常不友好——他们无法区分"接口写错了"和"模块没开启"。

### 解决方案

`DefaultController` 为每个可选模块预定义了一组兜底的请求映射：

```java
// 来源：yudao-server/src/main/java/cn/iocoder/yudao/server/controller/DefaultController.java

@RestController
@Slf4j
public class DefaultController {

    @RequestMapping("/admin-api/bpm/**")
    public CommonResult<Boolean> bpm404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[工作流模块 yudao-module-bpm - 已禁用][参考 https://doc.iocoder.cn/bpm/ 开启]");
    }

    @RequestMapping("/admin-api/mp/**")
    public CommonResult<Boolean> mp404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[微信公众号 yudao-module-mp - 已禁用][参考 https://doc.iocoder.cn/mp/build/ 开启]");
    }

    @RequestMapping(value = { "/admin-api/product/**",  // 商品中心
            "/admin-api/trade/**",                      // 交易中心
            "/admin-api/promotion/**" })                // 营销中心
    public CommonResult<Boolean> mall404() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(),
                "[商城系统 yudao-module-mall - 已禁用][参考 https://doc.iocoder.cn/mall/build/ 开启]");
    }

    // ... 其他模块类似

}
```

### 工作原理

当模块**未启用**时：

```
前端请求: POST /admin-api/product/spu/create
    │
    ▼
Spring MVC 路由匹配
    │
    ├─ 找到 ProductSpuController？  → 否（模块未引入，Bean 不存在）
    │
    ▼
匹配到 DefaultController 的 /admin-api/product/** 通配符
    │
    ▼
返回: {"code": 501, "msg": "[商城系统 yudao-module-mall - 已禁用][参考 ...]"}
```

当模块**已启用**时：

```
前端请求: POST /admin-api/product/spu/create
    │
    ▼
Spring MVC 路由匹配
    │
    ├─ 找到 ProductSpuController？  → 是（模块已引入，Bean 存在）
    │
    ▼
精确匹配优先于通配符匹配，路由到 ProductSpuController
    │
    ▼
正常处理请求
```

这里利用了 Spring MVC 的一个规则：**精确路径匹配优先于通配符匹配**。当模块启用后，其 Controller 中定义的精确路径（如 `/product/spu/create`）会覆盖 DefaultController 中的通配符路径（如 `/admin-api/product/**`）。

### 为什么放在 yudao-server 而不是 yudao-common？

`DefaultController` 只在单体模式下有意义。微服务模式下，每个服务独立部署，不存在"模块未启用"的情况——未部署的服务根本不会有请求进来。把它放在 yudao-server 中，恰好只在单体启动时生效。

## 设计四：OpenFeign 在单体模式下的处理

### 问题

业务模块的 API 层（如 `yudao-module-system-api`）使用了 `@FeignClient` 注解来定义 RPC 接口：

```java
// 来源：yudao-module-system-api/.../DeptApi.java
@FeignClient(name = ApiConstants.NAME) // name = "system-server"
public interface DeptApi {
    // ...
}
```

在微服务模式下，这个注解会让 Spring Cloud 创建一个 HTTP 代理，通过 Nacos 发现服务地址并发起远程调用。

但在单体模式下，所有模块都在同一个 JVM 中，Bean 之间可以直接注入，不需要 RPC。如果 OpenFeign 仍然生效，它会尝试去 Nacos 查找 `system-server` 服务——但 Nacos 根本没有启用，导致启动失败。

### 解决方案：Maven 排除

yudao-server 的 pom.xml 中，对 RPC 模块做了依赖排除：

```xml
<!-- 来源：yudao-server/pom.xml -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-spring-boot-starter-rpc</artifactId>
    <!-- 目的：yudao-server 单体启动，禁用 openfeign -->
    <exclusions>
        <exclusion>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-starter-openfeign</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

通过排除 `spring-cloud-starter-openfeign`，OpenFeign 的自动配置不会加载，`@FeignClient` 注解变成了一个无害的"标记"——它不会创建任何代理。

### 单体模式下的调用链路

在单体模式下，当业务代码需要调用 `DeptApi` 时：

```
调用方代码（如 trade 模块的 OrderService）
    │
    ▼
注入 DeptApi Bean
    │
    ├─ 微服务模式：注入的是 Feign 代理 → HTTP 远程调用
    │
    └─ 单体模式：注入的是 DeptApi 的直接实现 Bean → 本地方法调用
```

等等，`DeptApi` 是一个接口，单体模式下谁来提供实现？

答案是：业务模块的 `server` 子模块中，通常会有一个 `DeptApiImpl` 类实现 `DeptApi` 接口，并标注 `@Service` 注解。在单体模式下，Spring 会直接注入这个实现类。

```java
// 典型的 API 实现（在 server 模块中）
@Service
public class DeptApiImpl implements DeptApi {

    @Resource
    private DeptService deptService;

    @Override
    public CommonResult<DeptRespDTO> getDept(Long id) {
        DeptDO dept = deptService.getDept(id);
        return CommonResult.success(BeanUtils.toBean(dept, DeptRespDTO.class));
    }

    // ...
}
```

这就是为什么同一份代码能同时支持两种部署模式——接口定义在 `api` 模块，实现在 `server` 模块，调用方只依赖 `api` 模块。

## 完整启动流程

把以上设计串起来，一次单体启动的完整流程是这样的：

```
1. JVM 启动
   │
2. 加载 application.yaml（公共配置）
   │
3. 激活 spring.profiles.active=local
   │
4. 加载 application-local.yaml（本地配置，覆盖公共配置）
   │
5. 解析 yudao.info.base-package = cn.iocoder.yudao
   │
6. @SpringBootApplication 开始组件扫描
   │
   ├─ 扫描 cn.iocoder.yudao.server.**
   │   └─ 发现 DefaultController（兜底路由）
   │
   └─ 扫描 cn.iocoder.yudao.module.**
       ├─ 发现 system 模块的所有 Bean
       │   ├─ DeptController, DeptService, DeptMapper ...
       │   └─ DeptApiImpl（实现 DeptApi 接口）
       ├─ 发现 infra 模块的所有 Bean
       │   ├─ ConfigController, FileController ...
       │   └─ ...
       └─ product、trade 等模块未引入 → 对应 Bean 不存在
           └─ DefaultController 的通配符路由兜底
   │
7. Spring 容器初始化完成
   │
8. 应用启动在 48080 端口
```

## 与微服务模式的对比

| 维度 | 单体模式 (yudao-server) | 微服务模式 (各 *-server) |
|------|------------------------|------------------------|
| 启动类 | YudaoServerApplication | 各模块自己的启动类 |
| 组件扫描 | scanBasePackages 统一扫描 | 各服务只扫描自己的包 |
| 模块包含 | 通过 pom.xml 依赖控制 | 每个服务独立部署 |
| 模块间调用 | 直接注入 Bean | OpenFeign RPC |
| OpenFeign | 排除，不加载 | 启用，通过 Nacos 发现 |
| DefaultController | 需要，兜底未启用模块 | 不需要，服务不存在则网关报错 |
| Nacos | 可选（local 模式不启用） | 必须（注册发现 + 配置中心） |

## 本章思考题

1. `scanBasePackages` 配置了两个路径：`${yudao.info.base-package}.server` 和 `${yudao.info.base-package}.module`。为什么不合并成一个 `${yudao.info.base-package}` 来扫描整个项目？这会带来什么问题？

2. 假设你引入了 `yudao-module-product-server` 依赖，但忘记取消 `DefaultController` 中 `/admin-api/product/**` 的兜底映射。此时请求商品接口会发生什么？Spring MVC 的路由匹配规则如何保证精确匹配优先？

3. 在单体模式下排除了 OpenFeign，但如果某个模块的 `@FeignClient` 接口没有对应的本地实现类（`*ApiImpl`），启动时会报什么错？这种设计有没有潜在的风险？
