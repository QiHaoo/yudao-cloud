# 业务模块内部结构设计

## 从一个请求说起：创建商品

假设你正在开发电商后台的商品管理功能。前端发起一个请求：

```
POST /admin-api/product/spu/create
Content-Type: application/json
Authorization: Bearer eyJhbG...

{
  "name": "iPhone 16 Pro",
  "categoryId": 100,
  "price": 799900,
  "description": "全新 iPhone 16 Pro..."
}
```

这个请求从进入网关到最终写入数据库，经过了哪些层？每层的职责是什么？数据形态如何变化？

```
HTTP 请求
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ Controller 层（ProductSpuController）                  │
│   接收 SaveReqVO，校验参数，调用 Service               │
│   数据形态：SaveReqVO → Service 调用                   │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ Service 层（ProductSpuServiceImpl）                    │
│   业务逻辑：校验分类、生成 SPU 编号、保存商品           │
│   数据形态：SaveReqVO → DO → Mapper 调用              │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ Mapper 层（ProductSpuMapper）                          │
│   数据访问：继承 BaseMapperX，拼装 SQL                  │
│   数据形态：DO → SQL → 数据库                          │
└─────────────────────────────────────────────────────┘
    │
    ▼
  MySQL 表 product_spu
```

让我们看看每一层在代码中的实际位置和结构。

### 为什么要拆成 api 和 server？

想象一个场景：交易模块（trade）需要查询商品信息来生成订单。如果交易模块直接依赖商品模块的 server：

```
yudao-module-trade-server
    └── 依赖 yudao-module-product-server
        └── 依赖 yudao-module-product-api
```

这会导致：
1. **传递依赖膨胀**：trade 模块会把 product 的所有依赖（MyBatis-Plus、Service、Controller...）全部拉进来
2. **循环依赖风险**：如果 product 也需要查交易状态，两个 server 模块会互相依赖
3. **微服务化困难**：微服务模式下，trade 调 product 走的是 HTTP，不需要 product 的 Service 实现

拆分后的依赖关系：

```
yudao-module-trade-server
    └── 依赖 yudao-module-product-api    ← 只依赖轻量的 API 模块
        └── 包含：Feign 接口 + DTO + 枚举
            不包含：Controller、Service、Mapper
```

## api 模块：跨模块契约

### 核心组成

api 模块包含三类代码：

**1. Feign 接口定义**

```java
// 来源：yudao-module-system-api/.../api/dept/DeptApi.java
@FeignClient(name = ApiConstants.NAME) // name = "system-server"
@Tag(name = "RPC 服务 - 部门")
public interface DeptApi {

    String PREFIX = ApiConstants.PREFIX + "/dept";

    @GetMapping(PREFIX + "/get")
    @Operation(summary = "获得部门信息")
    CommonResult<DeptRespDTO> getDept(@RequestParam("id") Long id);

    @GetMapping(PREFIX + "/list")
    @Operation(summary = "获得部门信息数组")
    CommonResult<List<DeptRespDTO>> getDeptList(@RequestParam("ids") Collection<Long> ids);

    @GetMapping(PREFIX + "/valid")
    @Operation(summary = "校验部门是否合法")
    CommonResult<Boolean> validateDeptList(@RequestParam("ids") Collection<Long> ids);

    default Map<Long, DeptRespDTO> getDeptMap(Collection<Long> ids) {
        List<DeptRespDTO> list = getDeptList(ids).getCheckedData();
        return CollectionUtils.convertMap(list, DeptRespDTO::getId);
    }
}
```

关键点：
- `@FeignClient(name = ApiConstants.NAME)`：在微服务模式下用于 Nacos 服务发现
- `PREFIX = RpcConstants.RPC_API_PREFIX + "/system/dept"`：使用 `/rpc-api/system/dept` 前缀，与业务 API（`/admin-api/system/dept`）隔离
- 接口中可以有 `default` 方法，提供便捷的封装

**2. DTO（数据传输对象）**

```java
// 来源：yudao-module-system-api/.../api/dept/dto/DeptRespDTO.java
@Data
public class DeptRespDTO {
    private Long id;
    private String name;
    private Long parentId;
    private Integer sort;
    private Integer status;
}
```

DTO 比 DO 更轻量（不包含数据库注解），比 VO 更通用（不包含前端展示专用字段）。它是模块间通信的"通用语言"。

**3. 枚举常量**

```java
// 来源：yudao-module-system-api/.../enums/ApiConstants.java
public class ApiConstants {
    public static final String NAME = "system-server";
    public static final String PREFIX = RpcConstants.RPC_API_PREFIX + "/system";
    public static final String VERSION = "1.0.0";
}
```

```java
// 来源：yudao-module-system-api/.../enums/ErrorCodeConstants.java
public interface ErrorCodeConstants {
    // 用户模块 1-001-000-000
    ErrorCode USER_NOT_EXISTS = new ErrorCode(1001000001, "用户不存在");
    ErrorCode USER_USERNAME_EXISTS = new ErrorCode(1001000002, "用户名已经存在");
    // ...
}
```

## server 模块：完整实现

### Controller 层：接收请求，返回响应

Controller 层是 HTTP 请求的入口，它的职责是：
1. 接收请求参数（VO）
2. 参数校验（`@Valid`）
3. 权限校验（`@PreAuthorize`）
4. 调用 Service
5. 转换结果为响应 VO
6. 包装为 `CommonResult` 返回

```java
// 来源：yudao-module-system-server/.../controller/admin/dept/DeptController.java
@Tag(name = "管理后台 - 部门")
@RestController
@RequestMapping("/system/dept")
@Validated
public class DeptController {

    @Resource
    private DeptService deptService;

    @PostMapping("create")
    @Operation(summary = "创建部门")
    @PreAuthorize("@ss.hasPermission('system:dept:create')")
    public CommonResult<Long> createDept(@Valid @RequestBody DeptSaveReqVO createReqVO) {
        Long deptId = deptService.createDept(createReqVO);
        return success(deptId);
    }

    @GetMapping("/list")
    @Operation(summary = "获取部门列表")
    @PreAuthorize("@ss.hasPermission('system:dept:query')")
    public CommonResult<List<DeptRespVO>> getDeptList(DeptListReqVO reqVO) {
        List<DeptDO> list = deptService.getDeptList(reqVO);
        return success(BeanUtils.toBean(list, DeptRespVO.class));  // DO → VO 转换
    }

    @GetMapping("/get")
    @Operation(summary = "获得部门信息")
    @PreAuthorize("@ss.hasPermission('system:dept:query')")
    public CommonResult<DeptRespVO> getDept(@RequestParam("id") Long id) {
        DeptDO dept = deptService.getDept(id);
        return success(BeanUtils.toBean(dept, DeptRespVO.class));
    }
}
```

### VO 命名规范

项目中 VO 的命名遵循严格的规范，一看名字就知道用途：

| VO 类型 | 命名模式 | 用途 | 示例 |
|---------|---------|------|------|
| 保存请求 | `*SaveReqVO` | 创建和修改共用 | `DeptSaveReqVO` |
| 分页查询 | `*PageReqVO` | 分页查询条件 | `PostPageReqVO` |
| 列表查询 | `*ListReqVO` | 非分页列表查询条件 | `DeptListReqVO` |
| 完整响应 | `*RespVO` | 返回完整字段 | `DeptRespVO` |
| 精简响应 | `*SimpleRespVO` | 下拉选项等轻量场景 | `DeptSimpleRespVO` |

以部门为例，看看不同 VO 的字段差异：

```java
// 保存请求：包含 id（修改时有值，创建时为空）
public class DeptSaveReqVO {
    private Long id;         // 修改时传入
    private String name;
    private Long parentId;
    private Integer sort;
    private Long leaderUserId;
    private String phone;
    private String email;
    private Integer status;
}

// 完整响应：包含 id 和 createTime
public class DeptRespVO {
    private Long id;
    private String name;
    private Long parentId;
    private Integer sort;
    private Long leaderUserId;
    private String phone;
    private String email;
    private Integer status;
    private LocalDateTime createTime;  // 比 SaveReqVO 多了这个
}

// 精简响应：只包含最常用的字段
public class DeptSimpleRespVO {
    private Long id;
    private String name;
    private Long parentId;
}
```

这种设计避免了一个"万能 VO"带来的问题：前端不需要的字段也会传输，增加带宽开销和安全风险。

### Service 层：业务逻辑的家

Service 层是业务逻辑的核心所在。它的设计遵循一个原则：**Controller 不做业务逻辑，只做参数校验和调用编排；Mapper 不做业务逻辑，只做数据访问**。

```java
// 来源：yudao-module-system-server/.../service/dept/DeptService.java（接口）
public interface DeptService {
    Long createDept(DeptSaveReqVO createReqVO);
    void updateDept(DeptSaveReqVO updateReqVO);
    void deleteDept(Long id);
    DeptDO getDept(Long id);
    List<DeptDO> getDeptList(DeptListReqVO reqVO);
    // ...
}
```

```java
// 来源：yudao-module-system-server/.../service/dept/DeptServiceImpl.java（实现）
@Service
@Validated
public class DeptServiceImpl implements DeptService {

    @Resource
    private DeptMapper deptMapper;

    @Override
    @CacheEvict(cacheNames = RedisKeyConstants.DEPT_CHILDREN_ID_LIST, allEntries = true)
    public Long createDept(DeptSaveReqVO createReqVO) {
        if (createReqVO.getParentId() == null) {
            createReqVO.setParentId(DeptDO.PARENT_ID_ROOT);
        }
        // 校验父部门的有效性
        validateParentDept(null, createReqVO.getParentId());
        // 校验部门名的唯一性
        validateDeptNameUnique(null, createReqVO.getParentId(), createReqVO.getName());

        // 插入部门
        DeptDO dept = BeanUtils.toBean(createReqVO, DeptDO.class);  // VO → DO
        deptMapper.insert(dept);
        return dept.getId();
    }

    // ...
}
```

注意几个关键设计：
- **Service 接口的返回值是 DO，不是 VO**。这样其他 Service 调用时不需要做 VO/DO 转换
- **VO → DO 转换在 Service 层完成**，使用 `BeanUtils.toBean()`
- **DO → VO 转换在 Controller 层完成**，同样是 `BeanUtils.toBean()`

### Mapper 层：增强的数据访问

```java
// 来源：yudao-module-system-server/.../dal/mysql/dept/DeptMapper.java
@Mapper
public interface DeptMapper extends BaseMapperX<DeptDO> {

    default List<DeptDO> selectList(DeptListReqVO reqVO) {
        return selectList(new LambdaQueryWrapperX<DeptDO>()
                .likeIfPresent(DeptDO::getName, reqVO.getName())
                .eqIfPresent(DeptDO::getStatus, reqVO.getStatus()));
    }

    default DeptDO selectByParentIdAndName(Long parentId, String name) {
        return selectOne(DeptDO::getParentId, parentId, DeptDO::getName, name);
    }

    default Long selectCountByParentId(Long parentId) {
        return selectCount(DeptDO::getParentId, parentId);
    }
}
```

关键点：
- 继承 `BaseMapperX`（而非原生 `BaseMapper`），获得批量操作、分页等增强方法
- 使用 `LambdaQueryWrapperX` 的 `*IfPresent` 方法，当参数为 null 时自动跳过该条件
- 查询方法使用 `default` 方法直接在接口中实现，不需要 XML

### DO 实体：数据库的镜像

```java
// 来源：yudao-module-system-server/.../dal/dataobject/dept/DeptDO.java
@TableName("system_dept")
@KeySequence("system_dept_seq")  // 用于 Oracle、PostgreSQL 等数据库的主键自增
@Data
@EqualsAndHashCode(callSuper = true)
public class DeptDO extends TenantBaseDO {

    public static final Long PARENT_ID_ROOT = 0L;

    @TableId
    private Long id;
    private String name;
    private Long parentId;
    private Integer sort;
    private Long leaderUserId;
    private String phone;
    private String email;
    private Integer status;
}
```

DO 继承 `TenantBaseDO`（而非直接继承 `BaseDO`），因为它需要支持多租户。`TenantBaseDO` 在 `BaseDO` 的基础上增加了 `tenantId` 字段。

## 对象转换的两种方式

在 Controller → Service → Mapper 的调用链中，数据需要在 VO、DO、DTO 之间转换。项目提供了两种方式：

### 方式一：BeanUtils.toBean()（通用转换）

```java
// VO → DO
DeptDO dept = BeanUtils.toBean(createReqVO, DeptDO.class);

// DO → VO
DeptRespVO vo = BeanUtils.toBean(dept, DeptRespVO.class);

// DO → DTO
DeptRespDTO dto = BeanUtils.toBean(dept, DeptRespDTO.class);
```

适用于**字段名和类型基本一致**的简单转换。底层基于反射实现，使用方便但性能略低。

### 方式二：MapStruct（复杂转换）

当转换逻辑复杂时（需要查关联数据、字段名不一致等），使用 MapStruct：

```java
// 来源：yudao-module-system-server/.../convert/user/UserConvert.java
@Mapper
public interface UserConvert {

    UserConvert INSTANCE = Mappers.getMapper(UserConvert.class);

    default List<UserRespVO> convertList(List<AdminUserDO> list, Map<Long, DeptDO> deptMap) {
        return CollectionUtils.convertList(list, user -> convert(user, deptMap.get(user.getDeptId())));
    }

    default UserRespVO convert(AdminUserDO user, DeptDO dept) {
        UserRespVO userVO = BeanUtils.toBean(user, UserRespVO.class);
        if (dept != null) {
            userVO.setDeptName(dept.getName());  // 从关联的部门对象中取名称
        }
        return userVO;
    }
}
```

这里 `UserRespVO` 比 `AdminUserDO` 多了一个 `deptName` 字段，需要额外查询部门表才能填充。这种场景用 `BeanUtils.toBean()` 无法完成，必须用 MapStruct 或手写转换。

## 包结构设计决策

### 为什么 controller 下分 admin 和 app？

```
controller/
├── admin/      ← 管理后台 API（/admin-api/**）
└── app/        ← 用户端 API（/app-api/**）
```

这两类 API 的区别不仅仅是 URL 前缀不同：

| 维度 | admin API | app API |
|------|----------|---------|
| URL 前缀 | `/admin-api/product/**` | `/app-api/product/**` |
| 使用者 | 商家/管理员 | 普通消费者 |
| 权限模型 | RBAC + @PreAuthorize | 用户身份 + 业务规则 |
| 数据范围 | 管理后台可见所有数据 | 只能看自己的数据 |
| 操作 | CRUD 全功能 | 通常只有读和部分写 |

URL 前缀的自动注入由 `YudaoWebAutoConfiguration` 完成（详见第四章），它根据 Controller 所在的包路径自动添加前缀：

```java
// WebProperties 中的默认配置
private Api adminApi = new Api("/admin-api", "**.controller.admin.**");
private Api appApi = new Api("/app-api", "**.controller.app.**");
```

### 为什么 dal 下分 dataobject 和 mysql？

```
dal/
├── dataobject/     ← DO 实体（纯数据结构）
├── mysql/          ← Mapper 接口（数据访问逻辑）
└── redis/          ← Redis DAO（如果有）
```

将 DO 和 Mapper 分开存放，而不是放在同一个包下，原因是：
- **DO 是"名词"**：代表数据库表结构，变化频率低
- **Mapper 是"动词"**：包含查询逻辑，可能经常增加新方法
- 如果放在同一个包下，当表很多时，目录会非常臃肿

## 完整请求生命周期

让我们以"创建商品"为例，追踪一个请求从头到尾的完整生命周期：

```
1. 前端发送请求
   POST /admin-api/product/spu/create
   Body: {"name": "iPhone 16 Pro", "categoryId": 100, "price": 799900}

2. URL 前缀注入生效
   实际路由: /product/spu/create
   匹配到: ProductSpuController.createSpu()

3. 参数校验
   @Valid 校验 SaveReqVO 中的 @NotNull、@NotBlank 等约束

4. 权限校验
   @PreAuthorize("@ss.hasPermission('product:spu:create')")
   检查当前用户是否有 product:spu:create 权限

5. Controller 调用 Service
   Long spuId = productSpuService.createSpu(createReqVO);

6. Service 业务逻辑
   - 校验分类是否存在
   - 生成 SPU 编号
   - VO → DO 转换: ProductSpuDO spu = BeanUtils.toBean(createReqVO, ProductSpuDO.class)
   - 调用 Mapper 插入: productSpuMapper.insert(spu)

7. Mapper 执行 SQL
   INSERT INTO product_spu (name, category_id, price, ...) VALUES (?, ?, ?, ...)

8. 返回结果
   Controller: return success(spuId)
   响应体: {"code": 0, "data": 1024, "msg": "success"}
```

数据在各层的形态变化：

```
前端 JSON
    │
    ▼ (Spring MVC 反序列化)
SaveReqVO {name, categoryId, price, ...}
    │
    ▼ (BeanUtils.toBean)
ProductSpuDO {id, name, categoryId, price, createTime, ...}
    │
    ▼ (MyBatis-Plus)
SQL: INSERT INTO product_spu (...)
    │
    ▼
MySQL 表 product_spu
```
