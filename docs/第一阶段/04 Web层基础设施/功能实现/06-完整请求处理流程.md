# 完整请求处理流程

> 本章是功能实现的最后一章。前五章分别实现了 URL 前缀注入、全局异常处理、过滤器链、Swagger 文档、访问日志。本章将它们串联起来，展示一个 HTTP 请求从到达服务器到返回响应的完整生命周期。

## 1. 场景设定

以电商后台的"创建商品"接口为例：

```java
package cn.iocoder.yudao.module.product.controller.admin;

import cn.iocoder.yudao.framework.common.pojo.CommonResult;
import io.swagger.v3.oas.annotations.Operation;
import io.swagger.v3.oas.annotations.tags.Tag;

@Tag(name = "商品管理")
@RestController
@RequestMapping("/product/spu")
public class AdminProductSpuController {

    @Resource
    private ProductService productService;

    @Operation(summary = "创建商品")
    @PostMapping("/create")
    public CommonResult<Long> createProduct(@Valid @RequestBody ProductSaveReqVO reqVO) {
        return CommonResult.success(productService.createProduct(reqVO));
    }
}
```

前端发起请求：
```
POST /admin-api/product/spu/create
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Content-Type: application/json
tenant-id: 1

{
    "name": "iPhone 15 Pro",
    "price": 8999,
    "categoryId": 100
}
```

## 3. 异常场景的处理流程

如果 `productService.createProduct()` 抛出了 `ServiceException`：

```
⑧ Controller 执行
  │
  ├─ productService.createProduct(reqVO)
  │   └─ 抛出 ServiceException(PRODUCT_NAME_DUPLICATE)
  │
  ▼
┌─────────────────────────────────────────────────────────────────┐
│ GlobalExceptionHandler.serviceExceptionHandler()                │
│    ├─ IGNORE_ERROR_MESSAGES 不包含此消息 → 打印 warn 日志        │
│    ├─ 跳过 ServiceExceptionUtil 的堆栈帧                        │
│    ├─ 打印实际业务代码的堆栈位置                                  │
│    └─ return CommonResult.error(1001001, "商品名称已存在")       │
└─────────────────────────────────────────────────────────────────┘
  │
  ▼
GlobalResponseBodyHandler 记录 CommonResult 到 Request Attribute
  │
  ▼
ApiAccessLogFilter 记录访问日志
  │
  ├─ resultCode = 1001001
  ├─ resultMsg = "商品名称已存在"
  │
  ▼
浏览器收到响应：
{
    "code": 1001001,
    "msg": "商品名称已存在",
    "data": null
}
```

如果抛出的是未知异常（如 `NullPointerException`）：

```
⑧ Controller 执行
  │
  └─ 抛出 NullPointerException

┌─────────────────────────────────────────────────────────────────┐
│ GlobalExceptionHandler.defaultExceptionHandler()                │
│    ├─ ex.getCause() 不是 ServiceException → 继续                │
│    ├─ handleTableNotExists() → 不包含 "doesn't exist" → 继续   │
│    ├─ log.error("[defaultExceptionHandler]", ex)                │
│    │   └─ 打印完整堆栈到日志文件                                  │
│    ├─ createExceptionLog(req, ex)                              │
│    │   ├─ 记录用户信息、异常类名、堆栈、请求参数等                  │
│    │   └─ apiErrorLogApi.createApiErrorLogAsync(errorLog)      │
│    └─ return CommonResult.error(500, "系统异常")                │
└─────────────────────────────────────────────────────────────────┘
  │
  ▼
浏览器收到响应：
{
    "code": 500,
    "msg": "系统异常",
    "data": null
}
```

前端只看到"系统异常"，不会看到任何堆栈信息。但后端日志和数据库中记录了完整的异常详情。

## 4. Filter 中的异常处理

如果异常发生在 Filter 中（比如 `ApiEncryptFilter` 解密失败），处理方式不同：

```
ApiEncryptFilter.doFilterInternal()
  │
  ├─ 解密请求体失败，抛出异常
  │
  ▼
catch (Exception ex)
  │
  ├─ globalExceptionHandler.allExceptionHandler(request, ex)
  │   └─ 返回 CommonResult.error(...)
  │
  ├─ ServletUtils.writeJSON(response, result)
  │   └─ 直接写入响应，不经过 DispatcherServlet
  │
  └─ return（不调用 filterChain.doFilter，请求到此终止）
```

Filter 中的异常不会经过 `@ExceptionHandler`，所以需要手动调用 `allExceptionHandler` 方法。

## 5. URL 前缀注入的完整流程

```
请求 URL: /admin-api/product/spu/create

DispatcherServlet 查找 Handler
  │
  ▼
RequestMappingHandlerMapping.getHandler()
  │
  ├─ 遍历所有注册的 Controller
  │
  ├─ 对 AdminProductSpuController 执行匹配：
  │   ├─ @RestController? ✓
  │   ├─ 包名 cn.iocoder.yudao.module.product.controller.admin
  │   │   匹配 **.controller.admin.**? ✓
  │   └─ 注入前缀 /admin-api
  │
  ├─ @RequestMapping("/product/spu") + 前缀 "/admin-api"
  │   = "/admin-api/product/spu"
  │
  ├─ @PostMapping("/create")
  │   = "/admin-api/product/spu/create"
  │
  └─ 匹配成功 → HandlerMethod
```

## 6. WebFrameworkUtils 的数据流

`WebFrameworkUtils` 是连接各个组件的"数据桥梁"：

```
Spring Security Filter
  │
  ├─ 解析 Token → userId = 1, userType = ADMIN
  ├─ WebFrameworkUtils.setLoginUserId(request, 1)
  ├─ WebFrameworkUtils.setLoginUserType(request, ADMIN)
  │
  ▼
Controller 执行（可以读取 userId/userType）
  │
  ▼
GlobalResponseBodyHandler
  │
  ├─ WebFrameworkUtils.setCommonResult(request, result)
  │
  ▼
ApiAccessLogFilter
  │
  ├─ WebFrameworkUtils.getLoginUserId(request) → 1
  ├─ WebFrameworkUtils.getLoginUserType(request) → ADMIN
  ├─ WebFrameworkUtils.getCommonResult(request) → {code: 0, ...}
  │
  ▼
GlobalExceptionHandler（如果发生异常）
  │
  ├─ WebFrameworkUtils.getLoginUserId(request) → 1
  ├─ WebFrameworkUtils.getLoginUserType(request) → ADMIN
```

所有数据都通过 Request Attribute 传递，避免了组件之间的直接依赖。

1. 查看所有接口文档
2. 点击"Authorize"按钮，输入 Token
3. 选择接口，填写参数，点击"Execute"
4. 查看请求/响应详情

### 7.2 使用 curl 测试

```bash
# 创建商品
curl -X POST http://localhost:8080/admin-api/product/spu/create \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..." \
  -H "tenant-id: 1" \
  -d '{"name":"iPhone 15 Pro","price":8999,"categoryId":100}'

# 正常响应
{"code":0,"msg":"","data":12345}

# 异常响应（参数校验失败）
curl -X POST http://localhost:8080/admin-api/product/spu/create \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..." \
  -d '{}'

# 响应
{"code":400,"msg":"请求参数不正确:商品名称不能为空","data":null}
```

### 7.3 验证 CORS

```bash
# 预检请求
curl -X OPTIONS http://localhost:8080/admin-api/product/spu/create \
  -H "Origin: http://localhost:3000" \
  -H "Access-Control-Request-Method: POST"

# 响应头中应包含：
# Access-Control-Allow-Origin: http://localhost:3000
# Access-Control-Allow-Credentials: true
# Access-Control-Allow-Methods: POST
```

### 7.4 验证 XSS 过滤

```bash
# 提交包含 XSS 的商品名称
curl -X POST http://localhost:8080/admin-api/product/spu/create \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..." \
  -d '{"name":"<script>alert(1)</script>iPhone","price":8999}'

# Controller 接收到的 name 参数已经被清理：
# "name": "iPhone"
```

## 9. 小结

本章通过一个完整的 HTTP 请求生命周期，串联了前五章实现的所有组件：

1. **CORS Filter**：处理跨域
2. **CacheRequestBodyFilter**：缓存请求体
3. **TenantContextFilter**：设置租户上下文
4. **ApiAccessLogFilter**：记录访问日志
5. **XssFilter**：XSS 防护
6. **Spring Security**：认证鉴权
7. **URL 前缀注入**：自动拼接 `/admin-api`
8. **Controller**：执行业务逻辑
9. **GlobalExceptionHandler**：异常翻译为 CommonResult
10. **GlobalResponseBodyHandler**：记录返回值

这些组件通过 `WebFilterOrderEnum` 控制执行顺序，通过 `WebFrameworkUtils` 传递上下文数据，通过 `CommonResult` 统一响应格式。它们共同构成了 Web 层的基础设施。
