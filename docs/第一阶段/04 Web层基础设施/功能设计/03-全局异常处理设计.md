# 全局异常处理设计

## 为什么需要全局异常处理？

在[第 1 章](01-问题域与设计目标.md)中，我们看到了一个没有异常处理的电商后台面临的窘境：NullPointerException 的堆栈信息直接暴露给前端，数据库表名、SQL 语句、服务器路径一览无余。

更实际的问题是：一个电商系统可能有几百个 Controller 方法，每个方法都可能抛出各种异常。如果让每个方法自己处理异常：

```java
// 反面教材：每个方法都 try-catch
@GetMapping("/list")
public CommonResult<List<ProductRespVO>> list(@Valid ProductPageReqVO reqVO) {
    try {
        return CommonResult.success productService.list(reqVO));
    } catch (ServiceException e) {
        return CommonResult.error(e.getCode(), e.getMessage());
    } catch (AccessDeniedException e) {
        return CommonResult.error(FORBIDDEN);
    } catch (Exception e) {
        log.error("未知异常", e);
        return CommonResult.error(INTERNAL_SERVER_ERROR);
    }
}
```

这段代码在每个方法中重复出现，而且一旦遗漏某个 catch 分支，异常就会泄露给前端。

全局异常处理的设计目标是：**Controller 只管正常业务逻辑，所有异常由一个统一的处理器兜底**。

## 两层异常处理架构

项目的异常处理分为两层：

```
请求进入
    │
    ├── 在 Filter 层发生异常
    │       │
    │       ▼
    │   allExceptionHandler(request, ex)  ← 第一层：手动调用
    │       │
    │       ▼
    │   返回 CommonResult
    │
    └── 在 Controller 层发生异常
            │
            ▼
        @ExceptionHandler 方法          ← 第二层：Spring MVC 自动拦截
            │
            ▼
        返回 CommonResult
```

### 第一层：@ExceptionHandler（Spring MVC 层）

这是标准的 Spring MVC 异常处理机制。`@RestControllerAdvice` 注解让 `GlobalExceptionHandler` 成为全局的异常处理器，`@ExceptionHandler` 注解声明了每种异常对应的处理方法。

Spring MVC 在执行 Controller 方法时，如果抛出异常，会自动查找匹配的 `@ExceptionHandler` 方法。

### 第二层：allExceptionHandler（Filter 层）

Filter 不走 Spring MVC 的流程，所以 `@ExceptionHandler` 对 Filter 中的异常无效。但项目中有多个 Filter（ApiAccessLogFilter、ApiEncryptFilter 等），它们也可能抛出异常。

为此，`GlobalExceptionHandler` 提供了一个公开的 `allExceptionHandler(request, ex)` 方法，Filter 可以在 catch 块中手动调用它：

```java
try {
    chain.doFilter(request, response);
} catch (Exception ex) {
    // Filter 层异常，手动调用全局异常处理
    CommonResult<?> result = globalExceptionHandler.allExceptionHandler(request, ex);
    ServletUtils.writeJSON(response, result);
}
```

`allExceptionHandler` 内部通过 `instanceof` 判断异常类型，复用了所有 `@ExceptionHandler` 方法的逻辑，保证了 Filter 层和 Controller 层的异常处理结果一致。

## 异常处理优先级

框架对异常的处理遵循明确的优先级顺序：

```
异常发生
    │
    ▼
参数校验异常？ ──── 是 ──→ 返回 400 + 具体错误信息
    │                      （如"请求参数缺失:productId"）
    │ 否
    ▼
业务异常 ServiceException？ ──── 是 ──→ 返回对应错误码 + 业务消息
    │                                  （如"商品库存不足"）
    │ 否
    ▼
权限不足 AccessDeniedException？ ──── 是 ──→ 返回 403
    │
    │ 否
    ▼
系统异常（兜底） ──→ 记录完整日志 + 持久化错误日志
                    + 返回 500 + 通用错误消息
```

### 第一优先级：参数校验异常

参数校验异常是最常见的异常类型，通常是因为前端传参错误。框架为每种参数校验异常都提供了精确的错误信息：

| 异常类型 | 场景 | 返回示例 |
|---------|------|---------|
| `MissingServletRequestParameterException` | 缺少必填参数 | `请求参数缺失:productId` |
| `MethodArgumentNotValidException` | @Valid 校验失败 | `请求参数不正确:商品名称不能为空` |
| `BindException` | 参数绑定失败 | `请求参数不正确:价格格式错误` |
| `ConstraintViolationException` | 方法级参数校验失败 | `请求参数不正确:页码必须大于0` |
| `HttpMessageNotReadableException` | 请求体解析失败 | `请求参数类型错误:null` |
| `MethodArgumentTypeMismatchException` | 参数类型不匹配 | `请求参数类型错误:abc` |

这些异常都返回 `BAD_REQUEST`（错误码 400），附带人类可读的错误描述。前端可以直接将这个描述展示给用户。

### 第二优先级：业务异常 ServiceException

`ServiceException` 是项目中自定义的业务异常，用于表达业务规则不满足的情况：

```java
// 业务代码中抛出业务异常
public void updateProductStock(Long productId, Integer count) {
    ProductDO product = productMapper.selectById(productId);
    if (product.getStock() < count) {
        throw new ServiceException(PRODUCT_STOCK_NOT_ENOUGH);  // "商品库存不足"
    }
    // ... 更新库存
}
```

`ServiceException` 携带错误码和错误消息。异常处理器直接将它们转换为 `CommonResult`：

```java
@ExceptionHandler(value = ServiceException.class)
public CommonResult<?> serviceExceptionHandler(ServiceException ex) {
    // 只打印第一层调用栈（跳过 ServiceExceptionUtil），使用 warn 级别
    if (!IGNORE_ERROR_MESSAGES.contains(ex.getMessage())) {
        StackTraceElement[] stackTraces = ex.getStackTrace();
        for (StackTraceElement stackTrace : stackTraces) {
            if (ObjUtil.notEqual(stackTrace.getClassName(), ServiceExceptionUtil.class.getName())) {
                log.warn("[serviceExceptionHandler]\n\t{}", stackTrace);
                break;
            }
        }
    }
    return CommonResult.error(ex.getCode(), ex.getMessage());
}
```

关键设计决策：

1. **不打印完整堆栈**：ServiceException 是业务预期的异常，不是系统 Bug。打印完整堆栈会产生大量无用日志。只打印第一个业务调用栈帧，方便定位是哪行业务代码抛出的异常。
2. **忽略特定消息**：`IGNORE_ERROR_MESSAGES` 集合中的消息不会打印日志。比如"无效的刷新令牌"是高频异常（Token 过期时每次请求都会触发），打印日志会淹没其他重要信息。

### 第三优先级：权限异常 AccessDeniedException

当用户尝试访问没有权限的接口时，Spring Security 会抛出 `AccessDeniedException`（通过 `@PreAuthorize` 注解触发）：

```java
@ExceptionHandler(value = AccessDeniedException.class)
public CommonResult<?> accessDeniedExceptionHandler(HttpServletRequest req, AccessDeniedException ex) {
    log.warn("[accessDeniedExceptionHandler][userId({}) 无法访问 url({})]",
            WebFrameworkUtils.getLoginUserId(req), req.getRequestURL(), ex);
    return CommonResult.error(FORBIDDEN);
}
```

日志中记录了用户 ID 和请求 URL，方便安全审计。

### 第四优先级：系统异常（兜底处理）

所有未被前面处理器捕获的异常，都会落入 `defaultExceptionHandler`：

```java
@ExceptionHandler(value = Exception.class)
public CommonResult<?> defaultExceptionHandler(HttpServletRequest req, Throwable ex) {
    // 特殊处理：ServiceException 被包装在其他异常中的情况
    if (ex.getCause() != null && ex.getCause() instanceof ServiceException) {
        return serviceExceptionHandler((ServiceException) ex.getCause());
    }

    // 情况一：处理表不存在的异常
    CommonResult<?> tableNotExistsResult = handleTableNotExists(ex);
    if (tableNotExistsResult != null) {
        return tableNotExistsResult;
    }

    // 情况二：处理异常
    log.error("[defaultExceptionHandler]", ex);
    createExceptionLog(req, ex);
    return CommonResult.error(INTERNAL_SERVER_ERROR.getCode(), INTERNAL_SERVER_ERROR.getMsg());
}
```

这里有两个特殊处理：

#### 特殊处理一：ServiceException 被包装

有时 ServiceException 会被其他框架（如 Guava 的 `UncheckedExecutionException`）包装一层。`defaultExceptionHandler` 会检查 `ex.getCause()`，如果是 ServiceException 就走业务异常的处理逻辑。

#### 特殊处理二：表不存在异常

这是一个非常实用的"用户体验优化"。项目中有些模块（如商城、工作流、支付等）是可选的，用户可能没有导入对应的数据库表。当访问这些模块的接口时，MySQL 会报 "Table 'xxx' doesn't exist" 错误。

框架通过检查异常消息中的表名前缀（如 `product_`、`bpm_`、`pay_`），返回一个友好的提示：

```
[商城系统 yudao-module-mall - 已禁用][参考 https://cloud.iocoder.cn/mall/build/ 开启]
```

而不是让前端展示一个莫名其妙的 "Internal Server Error"。

目前支持的模块前缀映射：

| 表名前缀 | 对应模块 |
|---------|---------|
| `report_` | 报表模块 |
| `bpm_` | 工作流模块 |
| `mp_` | 微信公众号 |
| `product_`/`promotion_`/`trade_` | 商城系统 |
| `erp_` | ERP 系统 |
| `wms_` | WMS 仓库管理 |
| `crm_` | CRM 系统 |
| `mes_` | MES 系统 |
| `im_` | IM 即时通讯 |
| `pay_` | 支付平台 |
| `ai_` | AI 大模型 |
| `iot_` | IoT 物联网 |

## 错误日志的持久化

系统异常（非参数校验、非业务异常、非权限异常）不仅在控制台打印日志，还会持久化到数据库，方便后续排查。

### buildExceptionLog：构建完整的诊断信息

```java
private void buildExceptionLog(ApiErrorLogCreateReqDTO errorLog, HttpServletRequest request, Throwable e) {
    // 用户信息
    errorLog.setUserId(WebFrameworkUtils.getLoginUserId(request));
    errorLog.setUserType(WebFrameworkUtils.getLoginUserType(request));

    // 异常信息
    errorLog.setExceptionName(e.getClass().getName());
    errorLog.setExceptionMessage(ExceptionUtil.getMessage(e));
    errorLog.setExceptionRootCauseMessage(ExceptionUtil.getRootCauseMessage(e));
    errorLog.setExceptionStackTrace(ExceptionUtil.stacktraceToString(e));

    // 异常发生位置
    StackTraceElement stackTraceElement = e.getStackTrace()[0];
    errorLog.setExceptionClassName(stackTraceElement.getClassName());
    errorLog.setExceptionFileName(stackTraceElement.getFileName());
    errorLog.setExceptionMethodName(stackTraceElement.getMethodName());
    errorLog.setExceptionLineNumber(stackTraceElement.getLineNumber());

    // 请求信息
    errorLog.setTraceId(TracerUtils.getTraceId());
    errorLog.setApplicationName(applicationName);
    errorLog.setRequestUrl(request.getRequestURI());
    errorLog.setRequestMethod(request.getMethod());
    errorLog.setUserAgent(ServletUtils.getUserAgent(request));
    errorLog.setUserIp(ServletUtils.getClientIP(request));

    // 请求参数（合并 query + body）
    Map<String, Object> requestParams = MapUtil.<String, Object>builder()
            .put("query", ServletUtils.getParamMap(request))
            .put("body", ServletUtils.getBody(request)).build();
    errorLog.setRequestParams(JsonUtils.toJsonString(requestParams));

    errorLog.setExceptionTime(LocalDateTime.now());
}
```

这些信息包含了排查问题所需的完整上下文：

```
┌─────────────────────────────────────────────────────────┐
│ 错误日志                                                  │
├─────────────────────────────────────────────────────────┤
│ 谁出的问题？                                              │
│   userId: 123                                           │
│   userType: ADMIN                                       │
│   userIp: 192.168.1.100                                 │
│   userAgent: Chrome/120.0                                │
├─────────────────────────────────────────────────────────┤
│ 出了什么问题？                                            │
│   exceptionName: java.lang.NullPointerException         │
│   exceptionMessage: null                                │
│   exceptionRootCauseMessage: Cannot invoke ...          │
│   exceptionStackTrace: (完整堆栈)                        │
├─────────────────────────────────────────────────────────┤
│ 在哪里出的问题？                                          │
│   requestUrl: /admin-api/product/spu/update             │
│   requestMethod: PUT                                    │
│   exceptionClassName: ProductSpuServiceImpl             │
│   exceptionMethodName: updateSpu                        │
│   exceptionLineNumber: 156                              │
├─────────────────────────────────────────────────────────┤
│ 请求了什么？                                              │
│   requestParams: {"query":{},"body":{"id":1,"name":..}} │
├─────────────────────────────────────────────────────────┤
│ 追踪信息                                                 │
│   traceId: abc123                                       │
│   applicationName: yudao-product-server                 │
│   exceptionTime: 2024-01-15 14:30:25                    │
└─────────────────────────────────────────────────────────┘
```

有了这些信息，排查问题时不再需要"盲人摸象"——你知道谁、在什么时间、调了什么接口、传了什么参数、在哪里崩了。

### 异步持久化

错误日志的持久化通过 `ApiErrorLogCommonApi.createApiErrorLogAsync()` 异步执行。这避免了日志写入操作阻塞异常响应。即使日志写入失败，也不会影响用户收到错误响应：

```java
private void createExceptionLog(HttpServletRequest req, Throwable e) {
    ApiErrorLogCreateReqDTO errorLog = new ApiErrorLogCreateReqDTO();
    try {
        buildExceptionLog(errorLog, req, e);
        apiErrorLogApi.createApiErrorLogAsync(errorLog);
    } catch (Throwable th) {
        log.error("[createExceptionLog][url({}) log({}) 发生异常]",
                req.getRequestURI(), JsonUtils.toJsonString(errorLog), th);
    }
}
```

## GlobalResponseBodyHandler 的辅助作用

`GlobalResponseBodyHandler` 不是异常处理器，但它是异常处理体系的重要辅助。它实现了 `ResponseBodyAdvice` 接口，在 Controller 返回 `CommonResult` 后、响应写入 HTTP 之前，将结果存入 Request 属性：

```java
@ControllerAdvice
public class GlobalResponseBodyHandler implements ResponseBodyAdvice {

    @Override
    public boolean supports(MethodParameter returnType, Class converterType) {
        return returnType.getMethod() != null
                && returnType.getMethod().getReturnType() == CommonResult.class;
    }

    @Override
    public Object beforeBodyWrite(Object body, ...) {
        WebFrameworkUtils.setCommonResult(
            ((ServletServerHttpRequest) request).getServletRequest(),
            (CommonResult<?>) body
        );
        return body;  // 不修改返回值，只是记录
    }
}
```

这个设计的巧妙之处在于：**它不自动包装返回值**（很多网上的做法是自动将返回值包上 `CommonResult`），而是只做记录。Controller 自己负责返回 `CommonResult`。

为什么不自动包装？项目注释中给出了理由：

> GlobalResponseBodyHandler 本质上是 AOP，它不应该改变 Controller 返回的数据结构。

自动包装看似方便，但会带来问题：
- Controller 返回 `String` 时，自动包装成 `CommonResult<String>`，但 `String` 需要用 `StringHttpMessageConverter` 处理，而不是 `MappingJackson2HttpMessageConverter`
- 接口返回类型变得不透明，IDE 无法正确推断
- 无法对不同接口返回不同的包装结构

## 与 Spring Security 的协作

`AccessDeniedException` 的来源是 Spring Security 的 `@PreAuthorize` 注解。当用户访问一个没有权限的接口时，Spring Security 的 AOP 拦截器会抛出这个异常。

异常处理链路：

```
用户请求 /admin-api/system/user/delete
    │
    ▼
Spring Security Filter Chain
    │
    ▼
@PreAuthorize("@ss.hasPermission('system:user:delete')")
    │
    ▼
权限检查失败 → 抛出 AccessDeniedException
    │
    ▼
GlobalExceptionHandler.accessDeniedExceptionHandler()
    │
    ▼
CommonResult.error(FORBIDDEN)  // {"code": 403, "msg": "权限不足"}
```

## 本章小结

全局异常处理的设计遵循三个原则：

1. **分层处理**：参数校验异常返回精确的错误信息，业务异常返回业务错误码，系统异常记录完整日志并返回通用错误消息
2. **两层架构**：`@ExceptionHandler` 处理 Controller 层异常，`allExceptionHandler()` 处理 Filter 层异常，逻辑统一
3. **安全默认**：系统异常只返回通用错误消息（"系统异常，请联系管理员"），不暴露堆栈、SQL、表名等内部信息
