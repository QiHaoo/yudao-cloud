# 问题域与设计目标

## 从一个电商后台的"裸奔"说起

假设你正在从零搭建一个电商后台管理系统。Controller 写好了，商品查询、订单管理、库存更新都能跑了。但上线前你发现了一系列问题：

**问题一：异常信息直接暴露给前端**

```java
// 商品库存更新接口
@PutMapping("/stock")
public CommonResult<Boolean> updateStock(@Valid @RequestBody ProductStockUpdateReqVO reqVO) {
    productService.updateStock(reqVO.getId(), reqVO.getCount());
    return CommonResult.success(true);
}
```

如果 `productService.updateStock` 内部抛了一个 `NullPointerException`，前端会收到什么？如果没有任何异常处理，Spring Boot 的默认行为是返回一个白底黑字的 Whitelabel Error Page，或者一个包含完整堆栈的 JSON。这意味着数据库表名、SQL 语句、甚至服务器路径都可能泄露给用户。

**问题二：URL 前缀的"复制粘贴地狱"**

电商后台有管理端和用户端两套 API。管理端的接口前缀是 `/admin-api`，用户端（C 端顾客）的前缀是 `/app-api`。如果在每个 Controller 上都硬编码前缀：

```java
@RestController
@RequestMapping("/admin-api/product/spu")  // 管理端
public class AdminProductSpuController { ... }

@RestController
@RequestMapping("/app-api/product/spu")   // 用户端
public class AppProductSpuController { ... }
```

当项目需要改前缀（比如从 `/admin-api` 改成 `/manage-api`），你要改多少个文件？如果有人忘了加前缀呢？如果前缀拼写不一致呢？

**问题三：跨域请求被浏览器拦截**

前端跑在 `localhost:3000`，后端跑在 `localhost:8080`。浏览器的同源策略会阻止前端调用后端接口。每个接口都需要配置 CORS 头，但你不可能在每个 Controller 方法上都加 `@CrossOrigin`。

**问题四：XSS 攻击的隐患**

电商系统的商品名称、商品描述、用户评论等内容都是用户可输入的。如果用户提交了 `<script>alert('XSS')</script>` 作为商品名称，而你没有做任何过滤，这段脚本就会在其他用户浏览商品时执行——可能窃取 Cookie、冒充用户下单。

**问题五：API 文档的手动维护**

前后端分离开发中，API 文档是协作的基础。如果靠手动维护 Swagger 注解和文档，一旦接口变更而文档没有同步，就会出现前后端联调时"文档说有这个字段，实际没有"的尴尬。

**问题六：线上问题难以排查**

用户反馈"下单失败了"，但你无法复现。你不知道用户调了哪个接口、传了什么参数、耗时多久、返回了什么。没有访问日志，排查问题就像盲人摸象。

**问题七：敏感数据的传输安全**

电商系统中，用户的收货地址、支付密码、手机号等敏感信息通过 HTTP 传输。如果被抓包，这些数据就完全暴露了。

---

这些问题的共同特征是：**它们不是某个业务接口的逻辑问题，而是所有接口都需要面对的"横切关注点"（Cross-Cutting Concerns）**。如果让每个 Controller 自己处理，代码会严重重复，而且极易遗漏。

这就是 `yudao-spring-boot-starter-web` 要解决的核心问题。

## 设计目标

### 目标一：统一异常处理——"Controller 只管业务，不管善后"

每个 Controller 方法只负责正常流程，返回 `CommonResult.success(...)`。所有的异常——参数校验失败、权限不足、业务异常、系统崩溃——都由一个全局的异常处理器统一捕获，转换为结构化的 `CommonResult` 返回给前端。

```
正常请求  →  Controller  →  CommonResult.success(data)
                            ↓ (发生异常)
                    GlobalExceptionHandler  →  CommonResult.error(code, msg)
```

这意味着前端永远收到统一格式的 JSON 响应，不会看到 Whitelabel Error Page 或裸露的堆栈信息。

### 目标二：URL 前缀自动注入——"Controller 不写前缀，框架帮你加"

Controller 只需关注自己的业务路径（如 `/product/spu`），框架根据 Controller 所在的包路径自动判断并注入前缀：

- `cn.iocoder.yudao.module.product.controller.admin.*` → 自动加 `/admin-api` 前缀
- `cn.iocoder.yudao.module.product.controller.app.*` → 自动加 `/app-api` 前缀

改前缀？改一处配置即可。

### 目标三：全局安全防护——"透明的保护层"

- **CORS**：一个全局 Filter 处理所有跨域请求，Controller 无感知
- **XSS**：两层防护——Filter 层清理 Query 参数，Jackson 反序列化层清理 JSON Body
- **API 加解密**：敏感接口通过 `@ApiEncrypt` 注解声明，框架自动解密请求、加密响应

### 目标四：API 文档自动生成——"代码即文档"

基于 OpenAPI 3 规范 + Knife4j 增强，自动从代码中生成交互式 API 文档。自动注入公共参数（租户 ID、认证 Token），开发者不需要手动维护文档。

### 目标五：访问日志自动采集——"每一个请求都有迹可循"

自动记录每个 API 请求的完整信息：谁调的、调了什么、传了什么参数、返回了什么、耗时多久。敏感字段（password、token）自动脱敏。

## 组件清单

`yudao-spring-boot-starter-web` 内部包含 7 个子模块，各自对应一个独立的自动配置类：

| 子模块 | 自动配置类 | 核心能力 | 配置前缀 |
|--------|-----------|---------|---------|
| web | `YudaoWebAutoConfiguration` | URL 前缀注入、全局异常处理、CORS、RestTemplate | `yudao.web` |
| xss | `YudaoXssAutoConfiguration` | XSS 过滤（Filter + Jackson 反序列化两层） | `yudao.xss` |
| encrypt | `YudaoApiEncryptAutoConfiguration` | API 请求/响应加解密 | `yudao.api-encrypt` |
| swagger | `YudaoSwaggerAutoConfiguration` | OpenAPI 3 文档 + Knife4j 增强 | `yudao.swagger` |
| apilog | `YudaoApiLogAutoConfiguration` | 访问日志自动采集 | `yudao.access-log` |
| jackson | `YudaoJacksonAutoConfiguration` | Jackson 全局配置（Long→String 等） | — |
| banner | `YudaoBannerAutoConfiguration` | 自定义启动 Banner | — |
| desensitize | （自动注册） | 字段脱敏（手机号、邮箱、身份证等） | — |

后续章节将逐一深入讲解核心子模块的设计。

## 本章小结

Web 层基础设施要解决的核心矛盾是：**横切关注点不应侵入业务代码**。异常处理、安全防护、日志采集、文档生成——这些都是每个接口都需要的，但又不属于任何特定接口的业务逻辑。`yudao-spring-boot-starter-web` 的设计哲学是：

1. **声明式**：通过注解（`@ApiAccessLog`、`@ApiEncrypt`）声明意图，框架负责执行
2. **约定优于配置**：URL 前缀通过包路径自动推断，不需要手动配置
3. **可插拔**：每个子模块都可以通过配置开关独立启用或禁用
4. **安全默认**：XSS 过滤默认开启，敏感字段自动脱敏
