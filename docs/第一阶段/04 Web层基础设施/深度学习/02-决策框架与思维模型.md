## 1. Filter 排序：依赖链推导法

### 1.1 问题

10 个 Filter 的执行顺序不能随意排列。错误的顺序会导致数据丢失或逻辑错误。

### 1.2 推导过程

Filter 排序不是凭直觉，而是从依赖关系推导：

```
依赖关系：
  ApiEncryptFilter 需要读取 Body → 必须在 CacheRequestBodyFilter 之后
  ApiAccessLogFilter 需要读取原始参数 → 必须在 XssFilter 之前（XssFilter 会修改参数）
  TenantSecurityFilter 需要知道用户身份 → 必须在 Spring Security 之后
  ApiAccessLogFilter 需要知道用户身份 → 必须在 TenantContextFilter 之后
  XssFilter 需要缓存的 Body → 必须在 CacheRequestBodyFilter 之后

推导出的顺序：
  1. CORS（最先，因为跨域检查失败应立即返回）
  2. Body 缓存（让后续 Filter 可以重复读取 Body）
  3. API 解密（在 Body 缓存之后，需要读取 Body）
  4. 租户上下文（在访问日志之前，日志需要租户信息）
  5. 访问日志（在 XSS 之前，记录原始参数）
  6. XSS 清理（在 Spring Security 之前，先清理再认证）
  7. Spring Security（默认 -100）
  8. 租户安全（在 Spring Security 之后，需要用户身份）
  9. Demo 模式（最后，只拦截写操作）
```

### 1.3 决策框架

给 Filter 排序时，用这个三步法：

**第一步：画依赖图** — 列出每个 Filter 的输入依赖（它需要什么数据？）

**第二步：拓扑排序** — 根据依赖关系确定先后顺序

**第三步：分层归组** — 将无依赖关系的 Filter 归入同一优先级层

```
层 1（无依赖）：CORS
层 2（依赖层 1）：Body 缓存
层 3（依赖层 2）：API 解密、访问日志、XSS（三者可并行，但 Filter 是串行的）
层 4（框架层）：Spring Security
层 5（依赖层 4）：租户安全、Demo 模式
```

### 1.4 WebFilterOrderEnum 的设计

项目用 `Integer.MIN_VALUE` 到 `Integer.MAX_VALUE` 的整数区间来排序，而不是用 `@Order` 注解。原因：

1. **跨模块共享**：`WebFilterOrderEnum` 在 `yudao-common` 中定义，所有 Starter 都能引用
2. **数值间距预留**：相邻 Filter 之间留有间隔（如 `MIN_VALUE + 500`），方便后续插入新 Filter
3. **一目了然**：看到数值就知道相对位置，不需要查找注解

## 2. 异常处理：两层架构

### 2.1 问题

Spring MVC 的 `@ExceptionHandler` 只能捕获 Controller 层的异常。如果 Filter 层抛出异常（如 XSS Filter 中的错误），`@ExceptionHandler` 不会生效。

### 2.2 设计选择

yudao-cloud 用一个 `allExceptionHandler` 方法统一处理两层异常：

```
Controller 层异常 → @ExceptionHandler（Spring MVC 自动调用）
Filter 层异常 → Filter 中手动调用 globalExceptionHandler.allExceptionHandler()
```

`allExceptionHandler` 是一个 public 方法，内部通过 `instanceof` 分派到具体的处理方法。`@ExceptionHandler` 注解的方法也调用同一个分派逻辑。

### 2.3 异常优先级链

```
ValidationException（参数校验）
  → 400 BAD_REQUEST，精确的字段错误信息

ServiceException（业务异常）
  → 自定义 code + message，WARN 级别日志

AccessDeniedException（权限不足）
  → 403 FORBIDDEN

Exception（兜底）
  → 500 INTERNAL_SERVER_ERROR，ERROR 级别日志
```

关键设计：**ServiceException 只打印第一个业务栈帧**（跳过 `ServiceExceptionUtil` 的工具方法帧），并且对高频消息（如 "invalid refresh token"）完全抑制日志。这是"日志噪音控制"的实践。

### 2.4 友好错误处理：表不存在

当未安装的模块对应的数据库表不存在时，MySQL 会报 "Table 'xxx' doesn't exist"。`handleTableNotExists` 方法通过字符串匹配检测表名前缀（`product_`、`bpm_`、`pay_` 等），返回友好的"模块未安装"提示和配置链接，而不是 500 错误。

这是一个"用字符串匹配做降级"的务实做法——虽然脆弱（表名前缀可能变化），但在用户友好的报错和工程严谨性之间取得了平衡。

## 3. 日志脱敏：递归 JSON 树遍历

### 3.1 问题

访问日志需要记录请求参数和响应体，但必须脱敏敏感字段（如 `password`、`token`）。参数可能是嵌套的 JSON 结构，敏感字段可能出现在任意层级。

### 3.2 设计选择

`ApiAccessLogFilter` 的 `sanitizeJson(JsonNode, String[])` 方法递归遍历 JSON 树：

```
JsonNode
  ├── ObjectNode → 遍历字段，移除匹配的 key，递归处理子节点
  ├── ArrayNode → 遍历元素，递归处理每个元素
  └── 其他 → 直接返回
```

脱敏在两个层面进行：
- **注解级**：`@ApiAccessLog(sanitizeKeys = {"idCard", "phone"})` — 业务自定义
- **框架级**：`SANITIZE_KEYS = {"password", "token", "accessToken", "refreshToken"}` — 硬编码兜底

两者合并，确保不会遗漏。

### 3.3 取舍

| 方案 | 优点 | 缺点 |
|------|------|------|
| 递归遍历 JSON 树 | 能处理任意嵌套结构 | 性能开销（序列化 + 遍历 + 重新序列化） |
| 只脱敏顶层字段 | 性能好 | 嵌套结构中的敏感字段会泄露 |
| 正则替换 JSON 字符串 | 不需要解析 JSON | 容易误伤（如字段值中包含 "password"） |

yudao-cloud 选择了最安全的方案（递归遍历），因为日志脱敏是安全关键操作，宁可慢一点也不能泄露。

## 4. 异步日志 + 容错

### 4.1 设计模式

异常日志和访问日志都遵循同一个模式：

```java
try {
    apiAccessLogApi.createApiAccessLogAsync(accessLog);
} catch (Throwable ignored) {
    // 日志记录失败不影响用户请求
}
```

两个关键设计：
1. **异步**：日志持久化不阻塞用户请求
2. **容错**：日志系统的故障不会传播到业务系统

### 4.2 为什么用 try-catch 包裹异步调用

即使 `createApiAccessLogAsync` 是异步的，调用本身可能抛异常（如序列化失败、队列满）。如果不 catch，日志系统的异常会导致用户的 API 请求返回 500。

这是一个"防御性编程"的例子——对非关键路径（日志）的异常，宁可丢失也不能影响关键路径（业务响应）。

## 5. CORS 配置的微妙之处

### 5.1 问题

```java
corsConfiguration.addAllowedOriginPattern("*");  // 为什么不是 addAllowedOrigin("*")？
corsConfiguration.setAllowCredentials(true);
```

当 `allowCredentials = true`（允许携带 Cookie）时，浏览器禁止 `Access-Control-Allow-Origin: *`。必须用具体的 Origin 值或 `allowedOriginPattern("*")`。

### 5.2 背景

这是 HTTP CORS 规范的安全限制：如果允许携带凭证，响应头不能用通配符 `*`，因为这等于允许任意网站携带用户的 Cookie 发起请求。

`addAllowedOriginPattern("*")` 是 Spring 5.3 引入的新方法，它在响应时回显请求的 `Origin` 头（而不是返回 `*`），从而满足浏览器的安全检查。

### 5.3 决策框架

```
需要 CORS 支持吗？
├── 否 → 不配置
└── 是 → 需要携带 Cookie/Authorization？
    ├── 否 → addAllowedOrigin("*") 简单直接
    └── 是 → addAllowedOriginPattern("*") 或指定具体域名
        ├── 开发环境 → pattern("*") 方便调试
        └── 生产环境 → 指定具体域名更安全
```

## 小结

| 决策 | 核心判断维度 |
|------|------------|
| Filter 排序 | 从依赖关系推导，而非凭直觉 |
| 异常两层处理 | Filter 层和 Controller 层需要统一的异常处理入口 |
| 日志脱敏 | 安全关键操作选择最保守的方案 |
| 异步日志容错 | 非关键路径的异常不能影响关键路径 |
| CORS 配置 | allowCredentials 和 allowedOrigin 的组合约束 |
