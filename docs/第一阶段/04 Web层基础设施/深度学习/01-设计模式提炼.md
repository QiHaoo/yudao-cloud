# 设计模式提炼

---

## 1. 双通道处理模式

### 1.1 问题：一个请求，两条数据路径

HTTP 请求的数据通过两个不同的技术栈到达业务代码：

- **Query 参数、Header、Attribute** — 通过 `HttpServletRequest.getParameter()` 等 Servlet API
- **JSON Body** — 通过 Jackson 的 `@RequestBody` 反序列化，完全绕过 Servlet API

这意味着如果安全过滤（如 XSS 清理）只在 Servlet 层做，JSON Body 中的恶意代码会直接到达业务对象。

### 1.2 模式：同一策略，两个入口

yudao-cloud 的 XSS 防护在两个通道上各放一个守卫，共享同一个清洁策略：

```
请求进入
  ├── Query/Header/Attribute → XssFilter → XssRequestWrapper → 调用 XssCleaner.clean()
  └── JSON Body → Jackson → XssStringJsonDeserializer → 调用 XssCleaner.clean()
                                                              ↑
                                                        同一个策略实例
```

`XssCleaner` 是策略接口，`JsoupXssCleaner` 是默认实现。两个通道注入同一个 bean，确保清理逻辑一致。

### 1.3 通用化

双通道模式适用于任何"同一数据通过两个技术栈到达"的场景：

| 场景 | 通道 A（Servlet 层） | 通道 B（反序列化层） |
|------|-------------------|-------------------|
| XSS 清理 | XssRequestWrapper | XssStringJsonDeserializer |
| 参数脱敏 | Filter 中的 `getParameter()` 拦截 | 自定义 `JsonDeserializer` |
| 数据校验 | Filter 中预校验 | `@Valid` + 自定义 Validator |
| 审计日志 | Filter 中记录原始参数 | 反序列化后记录结构化数据 |

### 1.4 判断标准

是否需要双通道？关键问题是：**你的数据是否只通过 JSON Body 到达？**

- 如果所有 API 都用 `@RequestBody`，只需要 Jackson 层的处理
- 如果有文件上传（`multipart/form-data`）或 URL 参数，就需要 Filter 层
- 如果两种都有，就必须双通道

## 2. Filter-Interceptor 协作模式

### 2.1 问题：Filter 能力不足

Servlet Filter 能访问整个请求生命周期，但**无法获取 Spring MVC 的 `HandlerMethod`**（即"这个请求最终由哪个 Controller 的哪个方法处理"）。而 `HandlerMethod` 是读取注解（如 `@ApiAccessLog`）的前提。

反过来，Spring MVC 的 `HandlerInterceptor` 能访问 `HandlerMethod`，但它的执行时机在 Filter 链之后，无法拦截 Filter 层的异常，也无法记录请求开始时间。

### 2.2 模式：Request Attribute 传递元数据

```
Filter 层（外层，能包裹整个请求）
  │
  ▼
  ApiAccessLogFilter: 记录 beginTime
  │
  ▼
  filterChain.doFilter()
  │
  ▼
  Spring MVC 层（内层）
  │
  ├── HandlerInterceptor.preHandle()
  │     └── 将 HandlerMethod 存入 request.setAttribute("HANDLER_METHOD", ...)
  │
  ├── Controller 执行
  │
  └── HandlerInterceptor.afterCompletion()
  │
  ▼
  回到 Filter 层
  │
  └── ApiAccessLogFilter: 
        读取 request.getAttribute("HANDLER_METHOD") → 获取注解
        计算耗时 → 持久化日志
```

Filter 负责"何时记录"（时间戳、异常捕获），Interceptor 负责"记录什么"（通过 `HandlerMethod` 读取注解元数据）。两者通过 Request Attribute 传递信息。

### 2.3 通用化

这个模式适用于任何需要"Filter 的生命周期控制 + Spring MVC 元数据"的场景：

- **访问日志**：Filter 记录时间/IP，Interceptor 提供方法注解
- **权限校验**：Filter 做粗粒度检查（URL 级别），Interceptor 做细粒度检查（方法注解）
- **性能监控**：Filter 计算总耗时，Interceptor 提供方法名称用于指标标签

### 2.4 注意事项

Request Attribute 是一个隐式的"通道"，没有类型安全保证。建议：
- 用常量定义 attribute key（如 `ATTRIBUTE_HANDLER_METHOD`）
- 在 Interceptor 中 `null` 检查后再使用
- 考虑用 `ThreadLocal` 替代（但要注意清理）

## 3. 模板方法 + 路径守卫模式

### 3.1 问题：多个 Filter 共享相同的"是否过滤"逻辑

项目中有多个 Filter 只处理 API 请求（`/admin-api/**` 和 `/app-api/**`），对 Swagger、Actuator 等非 API 路径不生效。如果每个 Filter 都自己实现路径判断，代码会重复。

### 3.2 模式

```
OncePerRequestFilter（Spring 提供，保证每个请求只执行一次）
  └── ApiRequestFilter（抽象类，只做一件事：shouldNotFilter 判断路径）
        ├── XssFilter
        ├── ApiAccessLogFilter
        ├── TenantContextFilter
        └── 其他 API Filter
```

`ApiRequestFilter` 的 `shouldNotFilter()` 方法检查请求路径是否以 `/admin-api` 或 `/app-api` 开头。子类只需要实现 `doFilterInternal()`，不需要关心路径过滤。

### 3.3 通用化

"模板方法 + 路径守卫"适用于任何"多个组件共享相同的前置条件判断"的场景：

```
AbstractBaseClass
  ├── shouldProcess(input)  ← 共享的前置条件
  └── 子类实现具体逻辑
```

其他应用：
- 多个 Interceptor 共享"只处理特定 Controller"的逻辑
- 多个 AOP 切面共享"只拦截特定包"的逻辑
- 多个消息消费者共享"只处理特定队列"的逻辑

## 4. Convention over Configuration：URL 前缀注入

### 4.1 设计选择

URL 前缀注入有四种方案，项目选择了"包路径约定"：

| 方案 | 实现方式 | 代码侵入性 | 冗余信息 |
|------|---------|-----------|---------|
| A: 硬编码前缀 | 每个 Controller 上写 `@RequestMapping("/admin-api/...")` | 高 | 前缀在每个 Controller 重复 |
| B: 自定义注解 | `@AdminApi` 注解标记 Controller | 中 | 注解和包路径都表达了"这是 admin API" |
| C: 配置文件映射 | 配置文件指定哪些包对应哪些前缀 | 低 | 需要维护映射关系 |
| **D: 包路径约定** | `**.controller.admin.**` → `/admin-api` | **零** | **无——信息已在包路径中** |

方案 D 是典型的 Convention over Configuration：包路径本身就包含了"这是 admin API"的信息，不需要再通过注解或配置重复表达。

### 4.2 实现机制

通过 `WebMvcRegistrations` 替换默认的 `RequestMappingHandlerMapping`：

```java
@Bean
public WebMvcRegistrations webMvcRegistrations(WebProperties webProperties) {
    return new WebMvcRegistrations() {
        @Override
        public RequestMappingHandlerMapping getRequestMappingHandlerMapping() {
            RequestMappingHandlerMapping mapping = new RequestMappingHandlerMapping();
            mapping.setPathPrefixes(buildPathPrefixes(webProperties));
            return mapping;
        }
    };
}
```

`setPathPrefixes` 接收一个 `Map<String, Predicate<Class<?>>>`：
- Key 是前缀（如 `/admin-api`）
- Value 是匹配条件（如"在 `**.controller.admin.**` 包下的 `@RestController`"）

### 4.3 安全收益

URL 前缀不仅是代码组织手段，还有安全价值。源码注释解释：

> 通过前缀，Nginx 可以只转发 `/api/*`，防止 Swagger 和 Actuator 端点被意外暴露

这是一个"架构决策同时服务于安全"的例子。

### 4.4 通用化

Convention over Configuration 适用于任何"信息已经存在于代码结构中，不需要重复表达"的场景：

- **包路径 → API 类型**（本项目的做法）
- **方法名 → HTTP 方法**（`getUser` → GET, `createUser` → POST）
- **类名 → 表名**（JPA 的 `@Entity` 默认行为）
- **配置文件名 → Profile**（`application-dev.yaml` → `dev` profile）

判断标准：如果开发者已经通过命名或结构表达了意图，就不需要额外的注解或配置。

## 5. 反模式拒绝：ResponseBodyAdvice 自动包装

### 5.1 常见做法

很多 Spring 教程推荐用 `ResponseBodyAdvice` 自动将 Controller 返回值包装为 `CommonResult`：

```java
@ControllerAdvice
public class AutoWrapAdvice implements ResponseBodyAdvice<Object> {
    @Override
    public Object beforeBodyWrite(Object body, ...) {
        if (body instanceof CommonResult) return body;  // 已包装则跳过
        return CommonResult.success(body);  // 自动包装
    }
}
```

这样 Controller 可以直接返回业务对象，不需要手动包装。

### 5.2 为什么 yudao-cloud 不这样做

`GlobalResponseBodyHandler` 实现了 `ResponseBodyAdvice`，但**只记录返回值用于日志，不做自动包装**。源码注释解释了三个原因：

1. **String 返回值问题**：`ResponseBodyAdvice` 在 `StringHttpMessageConverter` 之前执行。如果返回值是 String，包装后的 `CommonResult` 会被当作 String 写入响应，而不是 JSON
2. **IDE 类型推断不透明**：Controller 方法签名返回 `String`，但实际响应是 `CommonResult<String>`，IDE 和前端开发者看到的类型不一致
3. **无法自定义包装**：某些接口需要返回非标准格式（如文件下载、流式响应），自动包装会干扰

### 5.3 取舍

| 维度 | 自动包装 | 手动包装（yudao-cloud 的选择） |
|------|---------|--------------------------|
| 代码简洁性 | 更简洁（省一行 `CommonResult.success()`） | 每个方法多一行 |
| 类型安全 | 不透明 | Controller 返回类型 = 实际响应类型 |
| 特殊场景 | 需要排除逻辑 | 天然支持 |
| 新人理解 | "魔法"——返回值怎么变了？ | 显式——看到什么就是什么 |

yudao-cloud 选择了显式，牺牲了一点简洁性换取可理解性。这是一个"少即是多"的设计哲学。

## 小结

| 模式 | 核心思想 | 适用场景 |
|------|---------|---------|
| 双通道处理 | 同一策略在 Servlet 层和反序列化层各设一个守卫 | 数据通过多个技术栈到达 |
| Filter-Interceptor 协作 | Filter 控制生命周期，Interceptor 提供元数据 | 需要两者能力的场景 |
| 模板方法 + 路径守卫 | 基类判断"是否处理"，子类实现"怎么处理" | 多个组件共享前置条件 |
| Convention over Configuration | 包路径本身就是配置 | 信息已存在于代码结构中 |
| 反模式拒绝 | 显式包装优于"魔法"包装 | 类型安全优先于代码简洁 |
