# 设计模式提炼与决策框架

---

## 1. 三层模块体系：从混乱到秩序

### 1.1 模式：BOM + Framework + Business 分层

```
yudao-dependencies (BOM)      ← 版本字典：所有第三方依赖的版本号集中在这里
  │
yudao-framework (17个 Starter) ← 能力层：每个 Starter 封装一个横切关注点
  │
yudao-module-* (业务模块)       ← 业务层：每个模块拆为 api + server
```

**依赖方向**：Business → Framework → BOM（单向，不可反向）

这个分层解决了三个问题：
- **版本统一**：BOM 集中管理版本号，避免模块间版本冲突
- **能力复用**：Framework 层的 Starter 被所有业务模块共享
- **关注点分离**：业务代码不混入基础设施逻辑

### 1.2 模式：api/server 契约拆分

每个业务模块拆为两个子模块：

```
yudao-module-system/
  ├── yudao-module-system-api/     ← 契约：Feign 接口 + DTO + 枚举
  └── yudao-module-system-server/  ← 实现：Controller + Service + Mapper + DO
```

其他模块只依赖 `-api`，不依赖 `-server`。这保证了：
- **依赖树轻量**：引入 `system-api` 不会带进 Service、Mapper 等实现类
- **微服务可拆**：`-api` 天然就是 Feign 客户端，拆微服务时不需要改代码
- **循环依赖可解**：框架层需要调用业务层时，依赖 `-api` 而非 `-server`

### 1.3 模式：CommonApi 反转依赖

框架 Starter 需要调用业务模块（如日志框架需要调用系统模块保存日志），但框架不能依赖业务模块（否则反向了）。

解决方案：在 `yudao-common` 中定义接口（`CommonApi`），业务模块实现它：

```
yudao-common 定义 ApiErrorLogCommonApi（接口）
  ↑
yudao-spring-boot-starter-web 依赖 yudao-common，调用接口
  ↑
yudao-module-system-server 实现接口
```

关键约束：每个模块的 CommonApi 不超过 1-3 个——它是框架级的稳定契约，不是业务 API。

## 2. 决策：单体/微服务双模式

### 2.1 核心机制

同一套代码既能当单体运行，也能当微服务运行。切换方式：

```
单体模式：
  pom.xml 注释掉 spring-cloud-starter-openfeign
  → @FeignClient 注解失效
  → Spring 自动注入本地 @RestController 实现
  → 同 JVM 调用，无 HTTP

微服务模式：
  pom.xml 引入 spring-cloud-starter-openfeign
  → @FeignClient 生效
  → 通过 Nacos 发现服务，HTTP 调用
```

### 2.2 为什么用注释而不是 Maven Profile

| 方案 | 操作方式 | 可见性 | 学习成本 |
|------|---------|--------|---------|
| 注释/取消注释 pom.xml | 直接改文件 | 高（打开 pom 就能看到） | 低 |
| Maven Profile | `mvn package -P prod` | 低（需要看 profile 定义） | 中 |
| Spring Profile | `spring.profiles.active=prod` | 中 | 中 |

yudao-cloud 选择了最简单的方案。这是一个"学习项目"的权衡——可理解性优先于工程严谨性。

### 2.3 DefaultController 通配回退

当某个模块被禁用（如注释掉 pom 依赖）后，对应的 API 端点不存在了。默认行为是 404，但 404 不够友好。

`DefaultController` 用 `/**` 通配路由匹配所有路径，返回 HTTP 501（Not Implemented）和模块安装链接。当模块启用后，Spring MVC 的"精确匹配优先于通配"规则会自动让 `DefaultController` 退位。

这个设计利用了一个不太为人知的 Spring MVC 行为：精确路径 > 通配路径。

## 3. ${revision} + flatten：Maven 版本管理

### 3.1 问题

多模块项目中，如果每个模块的版本号都写死（如 `<version>1.0.0</version>`），发版时需要改几十个 pom.xml。

### 3.2 解决方案

根 POM 中定义 `${revision}` 变量，所有模块引用它。`flatten-maven-plugin` 在 `install` 时将变量替换为实际值，确保发布到 Maven 仓库的 pom 不含变量。

两种 flatten 模式：
- `oss`：普通模块，生成扁平化的 pom
- `bom`：BOM 模块，保留 `<dependencyManagement>` 结构

### 3.3 为什么不直接用 Maven 的 `${revision}`

Maven 原生的 `${revision}` 在 `mvn install` 时不会替换——发布到本地仓库的 pom 仍然包含 `${revision}`，其他项目引用时会解析失败。`flatten-maven-plugin` 解决了这个问题。

## 4. 业界方案对比

| 方案 | yudao-cloud | Spring Cloud 官方示例 | DDD 脚手架 |
|------|-------------|---------------------|-----------|
| 模块划分 | 按业务域 | 按技术层 | 按限界上下文 |
| 模块间通信 | api/server 拆分 + Feign | 独立仓库 + Feign | 共享内核 + 防腐层 |
| 依赖管理 | BOM + 根 POM | 独立版本 | 共享内核 |
| 适用规模 | 中大型单体可拆 | 微服务原生 | 复杂业务域 |

yudao-cloud 的方案适合"从单体逐步拆微服务"的演进路径。如果一开始就确定是微服务架构，独立仓库 + CI/CD 流水线更合适。

## 小结

| 模式/决策 | 核心思想 |
|----------|---------|
| 三层模块体系 | BOM 管版本、Framework 管能力、Business 管业务 |
| api/server 拆分 | 契约与实现分离，为微服务化做准备 |
| CommonApi 反转 | 框架通过接口调用业务，避免循环依赖 |
| 注释切换模式 | 可理解性优先于工程严谨性 |
| 通配回退控制器 | 利用 Spring MVC 路由优先级做优雅降级 |
