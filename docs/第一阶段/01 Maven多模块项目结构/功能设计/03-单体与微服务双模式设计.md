# 单体与微服务双模式设计

---

## 1. 一个两难的选择

在项目初期，团队面临一个经典的技术决策：

**选单体还是微服务？**

单体应用的优势在早期非常明显：
- 本地开发简单，一个 JVM 跑起来就能调试所有功能
- 没有网络调用开销，模块间直接方法调用
- 部署简单，一个 JAR 包搞定

但微服务的优势在后期也很诱人：
- 各服务独立部署，按需扩容
- 技术栈可以差异化（某些模块用 JDK 17，某些用 JDK 8）
- 故障隔离，一个模块挂了不影响其他

**大多数团队的困境**：选了单体，后期拆分成本极高；选了微服务，前期开发效率低下。

yudao-cloud 的解法是：**同一套代码，同时支持两种模式**。

## 2. 核心机制：模块化 + 组件扫描

### 单体模式的工作原理

单体模式的关键是 `yudao-server` 模块。它是一个"空壳"——自身几乎没有业务代码，只做两件事：

**第一件事：通过 Maven 依赖组装业务模块**

```xml
<!-- yudao-server/pom.xml -->
<dependencies>
    <!-- 必选模块：默认启用 -->
    <dependency>
        <groupId>cn.iocoder.cloud</groupId>
        <artifactId>yudao-module-system-server</artifactId>
    </dependency>
    <dependency>
        <groupId>cn.iocoder.cloud</groupId>
        <artifactId>yudao-module-infra-server</artifactId>
    </dependency>

    <!-- 可选模块：注释掉就是不启用 -->
    <!--
    <dependency>
        <groupId>cn.iocoder.cloud</groupId>
        <artifactId>yudao-module-mall-product-server</artifactId>
    </dependency>
    <dependency>
        <groupId>cn.iocoder.cloud</groupId>
        <artifactId>yudao-module-pay-server</artifactId>
    </dependency>
    -->
</dependencies>
```

想启用哪个模块，取消注释对应的 `<dependency>` 即可。

**第二件事：通过组件扫描加载所有 Bean**

```java
@SpringBootApplication(
    scanBasePackages = {
        "${yudao.info.base-package}.server",   // cn.iocoder.yudao.server
        "${yudao.info.base-package}.module"    // cn.iocoder.yudao.module
    }
)
public class YudaoServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(YudaoServerApplication.class, args);
    }
}
```

`scanBasePackages` 设置为 `cn.iocoder.yudao.module`，这意味着所有业务模块（system、infra、mall 等）的 `@Controller`、`@Service`、`@Mapper` 都会被自动扫描和注册到 Spring 容器中。

### 微服务模式的工作原理

每个业务模块都有自己的启动类：

```java
// yudao-module-system-server/src/.../SystemServerApplication.java
@SpringBootApplication
public class SystemServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(SystemServerApplication.class, args);
    }
}
```

以及自己的 `application.yaml`：

```yaml
# yudao-module-system-server/src/main/resources/application.yaml
spring:
  application:
    name: system-server
  cloud:
    nacos:
      discovery:
        enabled: true    # 启用服务注册
      config:
        enabled: true    # 启用配置中心
```

在微服务模式下，每个模块作为一个独立的 Spring Boot 应用运行，通过 Nacos 注册和发现，通过 Feign 进行 RPC 调用。

## 3. 模式切换的关键配置

| 配置项 | 单体模式 | 微服务模式 |
|--------|---------|-----------|
| `spring.cloud.nacos.discovery.enabled` | `false` | `true` |
| `spring.cloud.nacos.config.enabled` | `false` | `true` |
| OpenFeign 依赖 | 排除 | 引入 |
| 服务注册 | 不需要 | Nacos |
| Feign 调用方式 | 本地解析（同 JVM 内直接调用） | HTTP 远程调用 |

### 单体模式下的 Feign 调用

一个常见的疑问是：在单体模式下，`@FeignClient` 标注的接口怎么工作？

答案是：**Feign 被排除了，Feign 客户端根本不会被创建**。

```xml
<!-- yudao-server/pom.xml -->
<dependency>
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-spring-boot-starter-rpc</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-starter-openfeign</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

在单体模式下，`DeptApi` 接口的实现是 `DeptApiImpl`（一个 `@RestController`），它直接作为 Spring Bean 被注入和调用，没有 HTTP 调用开销：

```
单体模式:
  mall-server 中的代码 → DeptApi → DeptApiImpl（直接方法调用）→ DeptService → DeptMapper

微服务模式:
  mall-server 中的代码 → DeptApi → OpenFeign → HTTP → system-server → DeptApiImpl → DeptService → DeptMapper
```

## 4. 禁用模块的优雅降级

当某个模块在单体模式下被注释掉（禁用）时，如果前端发了请求到该模块的 URL，会发生什么？

答案是 `DefaultController` 的兜底处理：

```java
// yudao-server/src/.../DefaultController.java
@RestController
public class DefaultController {

    @GetMapping({"admin-api/bpm/**", "app-api/bpm/**"})
    public CommonResult<?> bpmApi() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(), "请启动 bpm 服务，或取消 yudao-module-bpm-server 的注释");
    }

    @GetMapping({"admin-api/pay/**", "app-api/pay/**"})
    public CommonResult<?> payApi() {
        return CommonResult.error(NOT_IMPLEMENTED.getCode(), "请启动 pay 服务，或取消 yudao-module-pay-server 的注释");
    }
    // ... 更多模块
}
```

用户收到的不是 500 错误或一片空白，而是一个明确的提示信息，告诉他如何启用这个功能。这就是"优雅降级"。

## 5. Profile 配置体系

yudao-cloud 使用 Spring Profile 来区分不同环境的配置：

```
yudao-server/src/main/resources/
├── application.yaml              ← 通用配置（所有环境共享）
├── application-local.yaml        ← 本地开发环境
└── application-dev.yaml          ← 开发服务器环境
```

### 通用配置 (application.yaml)

包含所有环境都需要的配置，主要是框架层面的：

```yaml
spring:
  application:
    name: yudao-server

# MyBatis-Plus 配置
mybatis-plus:
  global-config:
    db-config:
      id-type: NONE              # 由 IdTypeEnvironmentPostProcessor 根据数据库自动切换
      logic-delete-field: deleted
      logic-delete-value: 1
      logic-not-delete-value: 0

# API 文档
springdoc:
  api-docs:
    enabled: true

# 多租户
yudao:
  tenant:
    enable: true
```

### 本地环境配置 (application-local.yaml)

包含本地开发时的数据库、Redis 等连接信息：

```yaml
server:
  port: 48080

spring:
  cloud:
    nacos:
      discovery:
        enabled: false           # 单体模式：关闭 Nacos
      config:
        enabled: false

  datasource:
    dynamic:
      datasource:
        master:
          url: jdbc:mysql://127.0.0.1:3306/yudao-cloud
          username: root
          password: 123456

  redis:
    host: 127.0.0.1
    port: 6379
```

### 开发环境配置 (application-dev.yaml)

连接开发服务器的 Nacos 和数据库：

```yaml
spring:
  cloud:
    nacos:
      discovery:
        enabled: true            # 微服务模式：启用 Nacos
        server-addr: 127.0.0.1:8848
      config:
        enabled: true
        server-addr: 127.0.0.1:8848
```

## 6. 微服务模式下的部署拓扑

```
                     浏览器 / App
                         │
                         ▼
                ┌─────────────────┐
                │  Gateway (48080)│  ← API 网关：路由 + Token 校验
                └────────┬────────┘
                         │
           ┌─────────────┼─────────────┐
           │             │             │
    ┌──────▼──────┐ ┌────▼──────┐ ┌────▼──────┐
    │system (48081)│ │infra(48082)│ │pay (48083)│  ← 独立 JVM
    └─────────────┘ └───────────┘ └───────────┘
           │             │             │
           └─────────────┼─────────────┘
                         │
                   ┌─────▼─────┐
                   │   Nacos   │  ← 服务注册 + 配置中心
                   └───────────┘
```

每个 `-server` 模块都是一个独立的 Spring Boot 应用，有自己的端口和 `application.yaml`。它们通过 Nacos 注册自己，通过 Feign + Nacos 发现彼此。
