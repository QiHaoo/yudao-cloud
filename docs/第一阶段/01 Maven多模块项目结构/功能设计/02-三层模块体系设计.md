# 三层模块体系设计

---

## 1. 为什么需要三层？

上一章我们确定了"按业务领域拆分"的大方向。但在实际操作中，直接把项目拆成十几个业务模块会遇到一个关键问题：**依赖管理的复杂度**。

想象一下，每个业务模块都需要用到 MyBatis-Plus、Redis、Spring Security。如果每个模块的 pom.xml 都自己声明这些依赖的版本：

```xml
<!-- system 模块 -->
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-boot-starter</artifactId>
    <version>3.5.16</version>
</dependency>

<!-- infra 模块 -->
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-boot-starter</artifactId>
    <version>3.5.14</version>  <!-- 不小心用了旧版本 -->
</dependency>
```

当 MyBatis-Plus 的 API 在 3.5.14 和 3.5.16 之间有不兼容的变更时，系统就会出现难以排查的 bug。

更糟糕的是，如果 MyBatis-Plus 要升级到 3.5.17，你需要逐个修改所有模块的 pom.xml。漏改一个就是一个隐患。

**yudao-cloud 的解法是三层模块体系**：

```
第一层：BOM（版本清单）
    ↓ 声明所有依赖的版本
第二层：框架层（能力封装）
    ↓ 封装通用技术能力，对外暴露 Starter
第三层：业务模块（业务实现）
    ↓ 引入 Starter，专注业务逻辑
```

每一层有明确的职责，层与层之间是单向依赖关系。

## 2. 第一层：BOM — 版本统一管理

### 什么是 BOM？

BOM（Bill of Materials）是 Maven 的一种标准实践。它本身不包含任何代码，只有一个 `pom.xml`，里面声明了一堆依赖的版本号。其他模块通过 `<scope>import</scope>` 导入 BOM 后，就不再需要写 `<version>` 标签了。

### yudao-dependencies 的设计

```xml
<!-- yudao-dependencies/pom.xml -->
<project>
    <!-- 注意：没有 parent！这是一个独立的 POM -->
    <groupId>cn.iocoder.cloud</groupId>
    <artifactId>yudao-dependencies</artifactId>
    <version>${revision}</version>
    <packaging>pom</packaging>

    <properties>
        <!-- Spring 生态 -->
        <spring.boot.version>2.7.18</spring.boot.version>
        <spring.cloud.version>2021.0.9</spring.cloud.version>
        <spring.cloud.alibaba.version>2021.0.6.2</spring.cloud.alibaba.version>

        <!-- 数据层 -->
        <mybatis-plus.version>3.5.16</mybatis-plus.version>
        <druid.version>1.2.28</druid.version>
        <redisson.version>4.4.0</redisson.version>

        <!-- 工具库 -->
        <lombok.version>1.18.46</lombok.version>
        <mapstruct.version>1.6.3</mapstruct.version>
        <hutool-5.version>5.8.44</hutool-5.version>

        <!-- ... 70+ 个版本属性 -->
    </properties>

    <dependencyManagement>
        <!-- 先导入上游 BOM -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>${spring.boot.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
        <!-- ... 更多上游 BOM -->

        <!-- 再声明自己的框架模块 -->
        <dependency>
            <groupId>cn.iocoder.cloud</groupId>
            <artifactId>yudao-common</artifactId>
            <version>${revision}</version>
        </dependency>
        <!-- ... 所有框架 Starter -->

        <!-- 最后声明第三方库 -->
        <dependency>
            <groupId>com.baomidou</groupId>
            <artifactId>mybatis-plus-boot-starter</artifactId>
            <version>${mybatis-plus.version}</version>
        </dependency>
        <!-- ... 所有第三方库 -->
    </dependencyManagement>
</project>
```

### 关键设计决策

**1. BOM 没有 parent**

这是一个有意的设计。BOM 通常会被其他项目导入（比如你自己的其他项目，或者公司内的其他项目），如果它有 parent（比如 `yudao` 根 POM），那么导入方也会被迫拉入这个 parent 的所有依赖管理，造成意料之外的版本冲突。独立的 BOM 是一个干净的"版本字典"。

**2. BOM 放在根 POM 的第一个 `<module>`**

```xml
<!-- yudao-cloud/pom.xml -->
<modules>
    <module>yudao-dependencies</module>  <!-- 必须第一个 -->
    <module>yudao-gateway</module>
    <module>yudao-framework</module>
    <module>yudao-server</module>
    <!-- ... -->
</modules>
```

Maven 按 `<module>` 顺序构建。BOM 必须先构建，因为根 POM 的 `<dependencyManagement>` 导入了它（`<scope>import</scope>`）。如果 BOM 还没 install 到本地仓库，其他模块就找不到版本声明。

**3. 先导入上游 BOM，再声明自己的依赖**

```xml
<dependencyManagement>
    <!-- 第一步：导入上游 BOM -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-dependencies</artifactId>
        ...
    </dependency>

    <!-- 第二步：声明自己的框架模块 -->
    <dependency>
        <groupId>cn.iocoder.cloud</groupId>
        <artifactId>yudao-common</artifactId>
        ...
    </dependency>

    <!-- 第三步：声明第三方库 -->
    <dependency>
        <groupId>com.baomidou</groupId>
        <artifactId>mybatis-plus-boot-starter</artifactId>
        ...
    </dependency>
</dependencyManagement>
```

这个顺序很重要：如果上游 BOM 已经声明了某个依赖的版本（比如 Spring Boot BOM 声明了 Jackson 的版本），而你在第三步用不同的版本重新声明了同一个依赖，你的声明会覆盖上游的。这就是"精确控制"的手段。

### 版本统一的魔法：${revision} + flatten-maven-plugin

传统 Maven 多模块项目有一个痛点：每个模块的 `<version>` 都要写死，比如 `1.0.0-SNAPSHOT`。如果要发版，需要逐个修改所有模块的版本号。

yudao-cloud 用 `${revision}` 属性解决了这个问题：

```xml
<!-- 根 POM 和 BOM 都声明 -->
<properties>
    <revision>2026.04-jdk8-SNAPSHOT</revision>
</properties>

<!-- 所有模块引用 -->
<version>${revision}</version>
```

但 Maven 在 install/deploy 时，不会自动把 `${revision}` 替换为实际值。安装到仓库的 POM 会保留字面量 `${revision}`，导致下游消费者无法解析。

这就是 **flatten-maven-plugin** 的作用：它在构建时生成一个"扁平化"的 POM，把 `${revision}` 替换为 `2026.04-jdk8-SNAPSHOT`。

```
源码中的 POM:     <version>${revision}</version>
                        ↓ flatten-maven-plugin
安装到仓库的 POM:  <version>2026.04-jdk8-SNAPSHOT</version>
```

两种 flatten 模式：

| 模式 | 使用者 | 行为 |
|------|--------|------|
| `oss` | 根 POM 和所有普通模块 | 解析 `${revision}`，移除 `<dependencyManagement>` 和 `<pluginManagement>`，生成干净的消费级 POM |
| `bom` | `yudao-dependencies` | 解析 `${revision}`，但**保留** `<dependencyManagement>`——因为 BOM 的核心价值就是导出版本管理 |

## 3. 第二层：框架层 — 通用能力封装

框架层 (`yudao-framework/`) 包含 17 个自定义 Spring Boot Starter，每个封装一个通用技术能力。

### 为什么叫 "Spring Boot Starter"？

Spring Boot 的核心设计理念是"约定优于配置"。一个 Starter 就是一个"开箱即用"的模块——引入依赖后自动配置好，不需要手动编写大量样板代码。

比如，你引入 `spring-boot-starter-web`，Spring Boot 自动帮你配好内嵌 Tomcat、Jackson 序列化、DispatcherServlet。你只需要写 Controller。

yudao-cloud 的框架层遵循同样的理念：

```
引入 yudao-spring-boot-starter-web
  → 自动配置全局异常处理
  → 自动配置 URL 前缀注入 (/admin-api, /app-api)
  → 自动配置 CORS 跨域
  → 自动配置 XSS 防护
  → 自动配置 Swagger/Knife4j 文档
  → 自动配置 API 访问日志
```

### 框架层的模块列表

| 类别 | 模块 | 封装的能力 |
|------|------|-----------|
| 基础 | `yudao-common` | 共享类型（CommonResult、PageParam、ServiceException）、工具类 |
| 数据 | `yudao-spring-boot-starter-mybatis` | MyBatis-Plus 增强、多数据源、审计字段自动填充 |
| Web | `yudao-spring-boot-starter-web` | 全局异常处理、URL 前缀注入、CORS、XSS、Swagger、访问日志 |
| 安全 | `yudao-spring-boot-starter-security` | Token 认证、Spring Security 集成、权限校验 |
| 缓存 | `yudao-spring-boot-starter-redis` | Redis + Redisson 集成 |
| RPC | `yudao-spring-boot-starter-rpc` | OpenFeign 声明式调用、Sentinel 熔断 |
| 消息 | `yudao-spring-boot-starter-mq` | Event/Redis/RocketMQ/RabbitMQ/Kafka |
| 任务 | `yudao-spring-boot-starter-job` | XXL-Job 定时任务 |
| 其他 | ... | Excel、WebSocket、监控、限流、测试等 |

### 框架层的依赖规则

框架层有一条铁律：**框架 Starter 不能依赖业务模块**。

```
yudao-spring-boot-starter-web → yudao-common  ✓
yudao-spring-boot-starter-web → yudao-module-system  ✗
```

如果框架层依赖了业务模块，就会形成循环依赖：业务模块依赖框架，框架又依赖业务模块。这会导致编译失败。

那框架层怎么调用业务模块的功能呢？答案是 `yudao-common` 中的 `CommonApi` 接口（详见第四章"跨模块通信"）。

## 4. 第三层：业务模块 — 业务实现

业务模块 (`yudao-module-*`) 是项目的核心，每个模块对应一个业务领域。

### 每个业务模块的内部结构

```
yudao-module-system/
├── pom.xml                         ← 聚合 POM
├── yudao-module-system-api/        ← API 契约（Feign 接口 + DTO）
│   └── src/main/java/
│       └── cn.iocoder.yudao.module.system.api/
│           ├── dept/
│           │   ├── DeptApi.java        ← Feign 客户端接口
│           │   └── DeptRespDTO.java    ← 传输对象
│           ├── user/
│           │   └── AdminUserApi.java
│           └── ApiConstants.java       ← 服务名、URL 前缀
│
└── yudao-module-system-server/     ← 实现（Controller + Service + Mapper）
    └── src/main/java/
        └── cn.iocoder.yudao.module.system/
            ├── controller/admin/    ← 管理端接口
            ├── service/             ← 业务逻辑
            ├── dal/                 ← 数据访问
            ├── convert/             ← 对象转换
            └── enums/               ← 模块枚举
```

### api 和 server 的分工

| 子模块 | 角色 | 依赖 | 使用者 |
|--------|------|------|--------|
| `xxx-api` | API 契约 | 只依赖 `yudao-common`（轻量） | 被其他模块引用（用于 RPC 调用） |
| `xxx-server` | 实现 | 依赖框架 Starter + 其他模块的 API | 自身就是服务提供者 |

这种拆分的核心价值是**依赖轻量化**。当 mall 模块需要调用 system 模块的部门查询接口时：

```
yudao-module-mall-server
  └── 依赖 yudao-module-system-api  ← 只有接口和 DTO，很轻量
      └── DeptApi.getDept()         ← 通过 Feign 调用
```

如果 mall 依赖的是 `yudao-module-system-server`（整个实现），就会把 system 的所有代码（Controller、Service、Mapper、所有依赖）都拉进来，造成依赖膨胀。

### 业务模块的依赖规则

1. **可以依赖**：框架 Starter、其他模块的 `-api`
2. **不能依赖**：其他模块的 `-server`

```
yudao-module-mall-server
  ├── yudao-spring-boot-starter-mybatis  ✓  （框架 Starter）
  ├── yudao-spring-boot-starter-web      ✓  （框架 Starter）
  ├── yudao-module-system-api            ✓  （其他模块的 API）
  └── yudao-module-system-server         ✗  （其他模块的实现！）
```

## 5. 三层之间的依赖关系全景

```
                    yudao-dependencies (BOM)
                         │
                    ┌────┴────┐
                    │ 根 POM  │  ← import BOM
                    └────┬────┘
                         │
          ┌──────────────┼──────────────┐
          │              │              │
   yudao-common    starter-web    starter-mybatis  ← 框架层
          │              │              │
          └──────────────┼──────────────┘
                         │
        ┌────────────────┼────────────────┐
        │                │                │
  system-api        infra-api        mall-api  ← 业务 API
        │                │                │
        │    ┌───────────┘                │
        │    │                            │
  system-server  infra-server  mall-server     ← 业务实现
```

**单向依赖规则**：
- 业务模块 → 框架 Starter → yudao-common → 第三方库
- 业务模块 → 其他业务模块的 API（不是实现）
- 框架层不能反向依赖业务模块
