# Maven 多模块项目结构：问题域与设计目标

---

## 1. 从一个真实场景说起

假设你正在从零搭建一个电商后台管理系统。项目刚启动时，所有代码都放在一个 Java 工程里：

```
shop-admin/
├── src/main/java/
│   ├── controller/        ← 用户、商品、订单的 Controller 全在一起
│   ├── service/           ← 所有 Service 也在一起
│   ├── mapper/            ← 所有 Mapper 也在一起
│   └── entity/            ← 所有实体类也在一起
└── pom.xml                ← 一个 POM，所有依赖都堆在里面
```

这个结构在项目初期完全没问题——三五个人的小团队，快速迭代，一切都很高效。

但随着业务增长，问题开始浮现：

**问题一：依赖爆炸**

pom.xml 从最初的 20 个依赖膨胀到 80 个。MyBatis-Plus、Flowable 工作流引擎、微信 SDK、支付宝 SDK、XXL-Job、RocketMQ……全部堆在一起。你只是想改一下商品模块的代码，却要等整个项目编译 5 分钟。

**问题二：职责不清**

新来的同事想加一个"优惠券"功能，但他找不到优惠券相关的代码应该放在哪里。Controller 目录里已经有 30 多个文件了，Service 也是。他不确定该复用哪个工具类，也不知道已有的"促销"模块和他要做的"优惠券"是什么关系。

**问题三：无法独立部署**

促销活动期间流量暴增，你只想给订单模块多分配几台机器。但因为所有模块打包在一起，你不得不把整个应用部署 10 份——即使用户模块、商品模块根本不需要扩容。

**问题四：团队协作冲突**

10 个开发者在同一个仓库里改代码，每天都有 merge 冲突。小王改了公共工具类，小李的编译就挂了。

## 2. 核心问题：如何组织一个大型 Java 项目？

这些问题的根源在于**单体模块结构无法承载大型项目的复杂度**。我们需要一种项目组织方式，能够：

1. **按职责拆分代码** — 每个模块只关注自己的领域，有清晰的边界
2. **统一管理依赖版本** — 避免不同模块引入同一依赖的不同版本导致冲突
3. **按需组合部署** — 想用哪些模块就组装哪些，像搭积木一样
4. **渐进式扩展** — 新增一个业务模块不需要改动已有模块
5. **支持两种部署模式** — 开发阶段单体运行方便调试，生产环境按需拆分为微服务

---

## 3. 设计目标

| 目标 | 具体要求 |
|------|---------|
| **模块化** | 业务模块之间通过明确的 API 契约通信，不直接访问彼此的内部实现 |
| **版本统一** | 所有模块使用同一版本号，一次改版本号即可全局生效 |
| **依赖集中管理** | 第三方库的版本在 BOM 中统一声明，子模块不写 `<version>` |
| **框架与业务分离** | 通用能力（Web、数据库、缓存等）封装为框架层，业务模块只关心业务逻辑 |
| **单体/微服务双模式** | 同一套代码既能以单体模式运行（一个 JVM），也能拆分为独立的微服务 |
| **按需引入** | 业务模块可以自由组合——电商系统用 mall + pay，CRM 系统只用 crm |

## 4. 替代方案对比

在设计 yudao-cloud 的项目结构时，有几种常见的组织方式可选：

### 方案 A：单模块单体

```
shop-admin/
└── pom.xml    ← 所有代码和依赖都在一个模块里
```

- **优点**：结构最简单，零配置开销
- **缺点**：上述四个问题全部存在
- **适用场景**：原型验证、个人项目

### 方案 B：按层拆分

```
shop-admin/
├── shop-controller/    ← 所有 Controller
├── shop-service/       ← 所有 Service
├── shop-dao/           ← 所有 Mapper 和实体
└── shop-common/        ← 公共工具
```

- **优点**：关注点分层清晰
- **缺点**：无法按业务领域拆分，新增业务模块时仍然要在每一层都加代码。无法独立部署单个业务
- **适用场景**：中型项目，团队不超过 5 人

### 方案 C：按业务领域拆分（yudao-cloud 采用）

```
yudao-cloud/
├── yudao-dependencies/          ← BOM：版本统一管理
├── yudao-framework/             ← 框架层：通用能力封装
│   ├── yudao-common/
│   ├── yudao-spring-boot-starter-web/
│   ├── yudao-spring-boot-starter-mybatis/
│   └── ...
├── yudao-server/                ← 单体启动入口
├── yudao-gateway/               ← API 网关
└── yudao-module-*/              ← 业务模块：每个模块独立自治
    ├── yudao-module-system/     ← 系统管理
    ├── yudao-module-infra/      ← 基础设施
    ├── yudao-module-mall/       ← 电商
    └── ...
```

- **优点**：业务模块自治、可独立部署、可自由组合、框架与业务解耦
- **缺点**：初始搭建成本较高，需要设计好模块间的通信契约
- **适用场景**：大型项目、多人团队、需要长期演进的系统

### 方案 D：微服务拆分（独立仓库）

```
system-service/          ← 独立 Git 仓库
infra-service/           ← 独立 Git 仓库
mall-service/            ← 独立 Git 仓库
```

- **优点**：彻底隔离，各服务独立版本控制
- **缺点**：跨仓库协调成本极高，依赖版本难以统一，本地开发需要同时拉多个仓库
- **适用场景**：超大型团队（50+ 人），各服务由不同团队完全独立维护

yudao-cloud 选择了**方案 C**，并在其基础上设计了一套精巧的三层模块体系，兼顾了模块化、版本管理和双模式部署的需求。

## 思考题

1. 如果你的电商系统只需要商品管理和订单管理两个功能，你会选择方案 C 还是方案 B？为什么？
2. 方案 C 和方案 D 的核心区别是什么？在什么团队规模下应该考虑从 C 迁移到 D？
3. "版本统一管理"和"各模块独立版本号"各有什么优缺点？yudao-cloud 为什么选择前者？
