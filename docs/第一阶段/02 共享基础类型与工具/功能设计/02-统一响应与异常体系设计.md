# 统一响应与异常体系设计

---

## 1. API 响应的三种形态

一个 HTTP 接口的响应，在技术上有三种可能的形态：

**形态一：直接返回业务对象**

```json
{
    "id": 1,
    "name": "技术部",
    "parentId": 0
}
```

前端拿到数据就能用，但遇到错误时（比如部门不存在），返回什么？一个 404 空白页？一个 HTML 错误页面？前端无法统一处理。

**形态二：返回带状态码的对象**

```json
{
    "code": 0,
    "msg": "",
    "data": {
        "id": 1,
        "name": "技术部"
    }
}
```

成功时 `code=0`，失败时 `code=100200001, msg="部门不存在"`。前端只需要判断 `code === 0` 就知道请求是否成功。

**形态三：RESTful 风格，用 HTTP 状态码**

```
成功：200 OK + JSON
失败：400 Bad Request / 404 Not Found / 500 Internal Server Error + JSON
```

前端需要同时处理 HTTP 状态码和响应体中的业务错误码，逻辑更复杂。

yudao-cloud 选择了**形态二**，通过 `CommonResult<T>` 实现。

---

## 2. CommonResult：统一响应封装

```java
@Data
public class CommonResult<T> implements Serializable {
    private Integer code;   // 错误码（0 = 成功）
    private String msg;     // 人类可读的错误信息
    private T data;         // 响应数据
}
```

### 为什么用 0 表示成功？

这是国内 Java 项目的一个约定俗成。HTTP 用 200 表示成功，但业务 API 用 0 更方便——前端判断 `if (code === 0)` 比 `if (code === 200)` 更简洁，也避免了和 HTTP 状态码混淆。

### 关键方法设计

| 方法 | 用途 | 设计考量 |
|------|------|---------|
| `success(T data)` | 创建成功响应 | 只接受 data，msg 固定为空——成功的消息没有意义 |
| `error(ErrorCode errorCode)` | 创建错误响应 | 从 ErrorCode 对象构造，确保错误码和消息的一致性 |
| `error(ErrorCode errorCode, Object... params)` | 带参数的错误 | 支持 `{}` 占位符，如 `"部门 {} 不存在"` → `"部门 123 不存在"` |
| `isSuccess()` / `isError()` | 判断状态 | 标记 `@JsonIgnore`——不序列化到 JSON，仅供 Java 代码使用 |
| `checkError()` | 检查并抛异常 | 如果是错误，直接抛 `ServiceException`——用于 Feign 调用后的错误传播 |
| `getCheckedData()` | 获取数据或抛异常 | 先 `checkError()` 再返回 `data`——一行代码完成"检查+取值" |

### Feign 调用中的典型用法

```java
// 调用方代码
CommonResult<DeptRespDTO> result = deptApi.getDept(id);
DeptRespDTO dept = result.getCheckedData();  // 成功返回数据，失败抛 ServiceException
```

`checkError()` 和 `getCheckedData()` 的存在，让基于 `CommonResult` 的 RPC 调用和直接方法调用一样简洁——调用方不需要手动判断 `isSuccess()`。

## 3. ErrorCode：错误码设计

### 错误码的结构

yudao-cloud 使用 10 位数字错误码，分为 4 段：

```
T-SSS-MMM-EEE
│   │   │   └── 错误序号（3位，模块内自增）
│   │   └────── 子模块编号（3位）
│   └────────── 系统编号（3位）
└────────────── 类型（1位，固定为 1 = 业务异常）
```

| 系统编号 | 模块 |
|---------|------|
| 001 | 基础设施（infra） |
| 002 | 系统管理（system） |
| 004 | 会员（member） |
| 007 | 支付（pay） |
| 008 | 商品（product） |
| 009 | 工作流（bpm） |
| 011 | 交易（trade） |

例如：
- `100200001` = 业务异常-系统管理-用户子模块-第 1 个错误（用户不存在）
- `100200002` = 业务异常-系统管理-用户子模块-第 2 个错误（用户名已存在）

### 全局错误码

| 错误码 | 常量 | 含义 |
|--------|------|------|
| 0 | `SUCCESS` | 成功 |
| 400 | `BAD_REQUEST` | 请求参数不正确 |
| 401 | `UNAUTHORIZED` | 账号未登录 |
| 403 | `FORBIDDEN` | 没有操作权限 |
| 404 | `NOT_FOUND` | 请求未找到 |
| 500 | `INTERNAL_SERVER_ERROR` | 系统异常 |
| 501 | `NOT_IMPLEMENTED` | 功能未实现/未开启 |
| 999 | `UNKNOWN` | 未知错误 |

全局错误码占用 `[0, 999]` 区间，与 HTTP 状态码对齐，方便前端统一处理。

## 4. 异常体系

### 两种异常

```
RuntimeException
  ├── ServiceException  ← 业务异常（用户输入错误、业务规则违反）
  └── ServerException   ← 系统异常（数据库挂了、配置错误）
```

| 异常类型 | 场景 | 处理方式 | 日志级别 |
|---------|------|---------|---------|
| `ServiceException` | 用户名已存在、订单已支付 | 返回明确的错误信息给前端 | INFO（不打堆栈） |
| `ServerException` | 数据库连接失败、NPE | 返回"系统异常"给前端，记录完整堆栈 | ERROR |

### 为什么 ServiceException 是 final 类？

```java
public final class ServiceException extends RuntimeException {
```

`final` 意味着不能被继承。这是一个刻意的设计决策：

1. **避免异常层次过深** — 如果 `OrderNotFoundException extends ServiceException`，`ProductNotFoundException extends ServiceException`，调用方需要 catch 多个异常类型
2. **统一处理** — `GlobalExceptionHandler` 只需要 `catch (ServiceException e)` 就能处理所有业务异常
3. **错误码即分类** — 通过 `ErrorCode` 对象携带错误码，比异常类型本身更能精确描述错误

### ServiceExceptionUtil：异常创建的工厂

```java
// 使用方式
throw exception(DEPT_NOT_FOUND);                           // 简单错误
throw exception(DEPT_NAME_DUPLICATE, name);                // 带参数的错误
throw exception(DEPT_PARENT_NOT_EXITS, reqVO.getParentId()); // 多参数
```

`ServiceExceptionUtil.exception()` 方法做了两件事：
1. 格式化错误消息（替换 `{}` 占位符）
2. 创建 `ServiceException` 实例

在业务代码中，通常通过静态导入简化使用：

```java
import static cn.iocoder.yudao.framework.common.exception.util.ServiceExceptionUtil.exception;

public void validateDeptNameUnique(Long id, String name) {
    DeptDO dept = deptMapper.selectByParentIdAndName(parentId, name);
    if (dept != null && !dept.getId().equals(id)) {
        throw exception(DEPT_NAME_DUPLICATE, name);
    }
}
```

## 5. 全局异常处理链

```
请求到达 Controller
  │
  ├─→ 参数校验失败（@Validated）
  │     → MethodArgumentNotValidException
  │     → GlobalExceptionHandler 捕获
  │     → 返回 CommonResult.error(400, "用户名不能为空")
  │
  ├─→ 业务代码抛 ServiceException
  │     → GlobalExceptionHandler.serviceExceptionHandler()
  │     → 返回 CommonResult.error(serviceException)
  │
  ├─→ 代码 bug（NPE、ClassCastException）
  │     → GlobalExceptionHandler.defaultExceptionHandler()
  │     → 记录完整堆栈日志
  │     → 持久化错误日志（通过 ApiErrorLogCommonApi）
  │     → 返回 CommonResult.error(500, "系统异常")
  │
  └─→ 正常返回
        → Controller 返回 CommonResult.success(data)
        → GlobalResponseBodyHandler 记录响应（用于访问日志）
        → 返回 JSON
```

`GlobalExceptionHandler` 是一个 `@RestControllerAdvice`，它捕获所有异常并转换为 `CommonResult`。关键设计：

- **ServiceException 不打堆栈** — 业务异常是预期内的，打堆栈浪费日志空间
- **ServerException 打完整堆栈** — 系统异常需要排查，必须记录详细信息
- **错误日志持久化** — 通过 `ApiErrorLogCommonApi` 将错误日志写入数据库，方便事后排查
