# 共享基础类型与工具：问题域与设计目标

---

## 1. 从一个重复代码说起

假设你正在开发电商后台的多个业务模块。每个模块都需要：

1. **统一的 API 响应格式** — 用户模块返回 `{code: 0, msg: "", data: {...}}`，商品模块也返回同样的格式
2. **分页查询** — 用户列表需要分页，商品列表也需要分页，分页参数和返回格式一模一样
3. **异常处理** — 用户不存在时抛异常，商品不存在时也抛异常，异常格式和错误码体系要统一
4. **对象转换** — DO 转 VO、VO 转 DTO，每个模块都在做同样的事

如果没有一个共享模块，每个业务模块都会自己实现一套：

```
// system 模块
public class SystemResult<T> { private int code; private String msg; private T data; }

// mall 模块
public class MallResult<T> { private int code; private String msg; private T data; }

// pay 模块
public class PayResult<T> { private int code; private String msg; private T data; }
```

这不仅是代码重复的问题，更严重的是：三个类的字段名可能不一致（`code` vs `errCode`），成功码可能不同（`0` vs `200`），序列化行为可能不同。前端对接三个模块就要写三套解析逻辑。

## 2. 设计目标

`yudao-common` 是整个项目最底层的模块，所有其他模块都依赖它。它的设计目标是：

| 目标 | 具体要求 |
|------|---------|
| **统一 API 契约** | 所有接口使用相同的响应封装（`CommonResult`）、分页参数（`PageParam`）、分页结果（`PageResult`） |
| **统一异常体系** | 业务异常（`ServiceException`）和系统异常（`ServerException`）有统一的错误码分配机制 |
| **统一对象转换** | 提供 `BeanUtils` 作为基础转换工具，支持单对象、列表、分页结果的转换 |
| **常用工具类** | 集合操作、日期处理、JSON 序列化、HTTP 工具等，避免各模块重复造轮子 |
| **跨模块 RPC 契约** | 在 `CommonApi` 接口中定义框架级的跨模块调用契约，打破框架层→业务层的反向依赖 |
| **轻量级** | 大部分依赖使用 `provided` scope，不强制传递给消费者 |

## 3. 模块定位

```
yudao-cloud 的依赖层次：

yudao-module-xxx-server
  ├── yudao-spring-boot-starter-web
  ├── yudao-spring-boot-starter-mybatis
  └── yudao-module-xxx-api
        └── yudao-common  ← 所有模块的最底层依赖
```

`yudao-common` 处于依赖链的最底端，它：
- **不依赖** 任何框架 Starter 或业务模块
- **被所有模块** 直接或间接依赖
- **只包含** POJO、枚举、工具类、接口定义——不包含任何 Spring 配置或自动装配逻辑

这种"纯定义 + 纯工具"的定位确保了它不会引入运行时依赖冲突。
