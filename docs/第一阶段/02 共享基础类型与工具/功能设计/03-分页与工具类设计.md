# 分页与工具类设计

---

## 1. 分页：每个系统都需要的能力

电商后台几乎每个列表页面都需要分页：用户列表、商品列表、订单列表、日志列表。分页看似简单，但要做得好需要考虑几个问题：

- 前端传什么参数？`page` 还是 `pageNo`？从 0 开始还是从 1 开始？
- 后端返回什么？只有数据列表？还是列表 + 总条数？
- 不分页的场景怎么处理？（比如导出 Excel 时需要查全部数据）
- 排序怎么传？`sort=name&order=asc` 还是 `sortingFields[0].field=name&sortingFields[0].order=asc`？

yudao-cloud 用四个类统一了分页的输入输出。

---

## 2. 分页参数：PageParam

```java
@Data
public class PageParam implements Serializable {
    public static final Integer PAGE_SIZE_NONE = -1; // -1 = 不分页

    @NotNull @Min(1)           private Integer pageNo = 1;
    @NotNull @Min(1) @Max(200) private Integer pageSize = 10;
}
```

### 设计决策

**pageNo 从 1 开始**

从 0 开始是程序员的直觉，但从 1 开始更符合普通人的认知。前端传 `pageNo=1` 就是第一页，不需要解释。

**pageSize 上限 200**

这是一个防呆设计。如果前端不小心传了 `pageSize=99999`，数据库会一次性查出近 10 万条数据，可能导致内存溢出或数据库慢查询。200 是一个合理的上限——如果需要更多数据，应该用导出功能而不是翻页。

**PAGE_SIZE_NONE = -1**

当 `pageSize=-1` 时，`BaseMapperX.selectPage()` 会跳过分页，直接查全部数据。典型场景：
- 导出 Excel
- 生成下拉框选项
- 数据迁移

## 3. 分页结果：PageResult

```java
@Data
public final class PageResult<T> implements Serializable {
    private Long total;      // 总条数
    private List<T> list;    // 当前页数据
}
```

`PageResult` 是所有分页查询的统一返回类型。注意它是 `final` 的——不需要被继承。

## 4. 带排序的分页：SortablePageParam

```java
@Data
public class SortablePageParam extends PageParam {
    private List<SortingField> sortingFields;
}

@Data
public class SortingField implements Serializable {
    public static final String ORDER_ASC = "asc";
    public static final String ORDER_DESC = "desc";
    private String field;    // 排序字段名
    private String order;    // asc 或 desc
}
```

支持多字段排序。前端可以传：

```
pageNo=1&pageSize=10&sortingFields[0].field=createTime&sortingFields[0].order=desc
```

## 5. 工具类总览

`yudao-common` 提供了丰富的工具类，覆盖日常开发中最常见的需求。以下是按类别整理的核心工具：

### 对象转换：BeanUtils

基于 Hutool 的 `BeanUtil`，封装了项目中最常用的转换场景：

| 方法 | 用途 |
|------|------|
| `toBean(source, TargetClass)` | 单对象转换（DO → VO） |
| `toBean(list, TargetClass)` | 列表转换（List<DO> → List<VO>） |
| `toBean(pageResult, TargetClass)` | 分页结果转换（PageResult<DO> → PageResult<VO>） |
| `copyProperties(source, target)` | 属性拷贝（忽略 null 值） |

使用示例：

```java
// DO → VO
DeptRespVO vo = BeanUtils.toBean(deptDO, DeptRespVO.class);

// List<DO> → List<VO>
List<DeptRespVO> voList = BeanUtils.toBean(deptDOList, DeptRespVO.class);

// PageResult<DO> → PageResult<VO>
PageResult<DeptRespVO> voPage = BeanUtils.toBean(deptPage, DeptRespVO.class);
```

对于复杂的对象映射（需要额外字段、条件映射等），项目推荐使用 MapStruct + `default` 方法。

### 集合操作：CollectionUtils

这是项目中最大的工具类（374 行），封装了 Stream API 的常见用法：

| 方法 | 等价的 Stream 代码 |
|------|-------------------|
| `convertList(list, mapper)` | `list.stream().map(mapper).filter(Objects::nonNull).collect(toList())` |
| `convertSet(list, mapper)` | 同上，收集为 Set |
| `convertMap(list, keyMapper, valueMapper)` | `list.stream().collect(toMap(keyMapper, valueMapper))` |
| `convertMultiMap(list, keyMapper, valueMapper)` | `list.stream().collect(groupingBy(keyMapper, mapping(valueMapper, toList())))` |
| `filterList(list, predicate)` | `list.stream().filter(predicate).collect(toList())` |
| `diffList(oldList, newList, matcher)` | 比较新旧列表，返回 `[创建列表, 更新列表, 删除列表]` |

`diffList` 方法特别有用——在"批量更新"场景中，比较前端提交的列表和数据库现有列表，自动生成需要新增、修改、删除的记录。

### 日期处理：LocalDateTimeUtils + DateUtils

| 工具类 | 职责 |
|--------|------|
| `DateUtils` | `Date` 和 `LocalDateTime` 之间的互转，以及常用日期操作 |
| `LocalDateTimeUtils` | `LocalDateTime` 的全面操作：解析、范围判断、时间桶生成 |

`LocalDateTimeUtils.getDateRangeList()` 用于生成统计图表的时间轴——比如"按天统计最近 30 天的订单量"：

```java
List<LocalDateTime[]> buckets = LocalDateTimeUtils.getDateRangeList(
    startTime, endTime, DateIntervalEnum.DAY
);
// 返回: [[12-01 00:00, 12-02 00:00], [12-02 00:00, 12-03 00:00], ...]
```

### JSON 处理：JsonUtils

基于 Jackson 的 JSON 工具，做了两项关键定制：

1. **LocalDateTime 序列化为时间戳** — 默认行为是将 `LocalDateTime` 序列化为毫秒时间戳（如 `1704067200000`），而非 ISO 字符串。如果字段标注了 `@JsonFormat(pattern="yyyy-MM-dd")`，则使用指定格式
2. **Long 类型安全序列化** — 超过 JavaScript 安全整数范围（2^53）的 `Long` 值自动序列化为字符串，避免前端精度丢失

### 其他工具

| 工具类 | 核心能力 |
|--------|---------|
| `HttpUtils` | URL 编解码、HTTP 请求封装、Basic Auth 解析 |
| `ServletUtils` | 获取客户端 IP/UserAgent、读取请求体、写 JSON 响应 |
| `MoneyUtils` | 分转元、百分比计算、金额乘法（电商必备） |
| `NumberUtils` | 安全的数字解析、经纬度距离计算 |
| `CacheUtils` | Guava LoadingCache 构建（异步刷新 / 同步刷新） |
| `TracerUtils` | 获取 SkyWalking 的 traceId |
| `SpringUtils` | 获取 Spring Bean、判断是否生产环境 |
| `ValidationUtils` | 手机号/URL 格式校验、编程式 JSR-303 校验 |

## 6. 自定义校验注解

除了工具类，`yudao-common` 还提供了三个自定义校验注解：

| 注解 | 用途 | 示例 |
|------|------|------|
| `@InEnum` | 校验值是否在枚举的合法范围内 | `@InEnum(CommonStatusEnum.class)` |
| `@Mobile` | 校验中国手机号格式 | `@Mobile` |
| `@Telephone` | 校验电话号码格式 | `@Telephone` |

`@InEnum` 是最常用的一个。它通过 `ArrayValuable` 接口与枚举配合：

```java
// 枚举实现 ArrayValuable
public enum CommonStatusEnum implements ArrayValuable<Integer> {
    ENABLE(0), DISABLE(1);

    @Override
    public Integer[] array() {
        return ARRAYS;
    }
    public static final Integer[] ARRAYS = Arrays.stream(values())
            .map(CommonStatusEnum::getCode).toArray(Integer[]::new);
}

// VO 中使用
@InEnum(CommonStatusEnum.class)
private Integer status;  // 只能传 0 或 1
```

## 思考题

1. `PageParam` 的 `pageSize` 上限为什么是 200 而不是 1000？在什么场景下 200 可能不够用？
2. `BeanUtils.toBean()` 是基于反射的运行时转换，而 MapStruct 是编译期代码生成。在什么场景下应该用 `BeanUtils`，什么场景下应该用 MapStruct？
3. `CollectionUtils.diffList()` 的 `matcher` 参数是一个 `BiFunction<T, T, Boolean>`，用来判断两条记录是否是"同一条"。为什么不用 `equals()` 而是用自定义的匹配函数？
