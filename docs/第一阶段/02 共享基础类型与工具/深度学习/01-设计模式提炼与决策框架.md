# 设计模式提炼与决策框架

---

## 1. 异常体系：final + 错误码

### 1.1 常见做法

大多数项目用异常继承体系区分业务异常：

```java
class OrderNotFoundException extends ServiceException { }
class ProductOutOfStockException extends ServiceException { }
class PaymentFailedException extends ServiceException { }
```

GlobalExceptionHandler 中为每个异常类写一个 `@ExceptionHandler`。

### 1.2 yudao-cloud 的做法

`ServiceException` 是 `final` 类，不能被继承。所有业务异常都用同一个类，通过错误码区分：

```java
// 抛出
throw new ServiceException(ErrorCodeConstants.ORDER_NOT_FOUND);
// 或工具方法
throw ServiceExceptionUtil.exception(ErrorCodeConstants.ORDER_NOT_FOUND);

// GlobalExceptionHandler 中只需要一个方法
@ExceptionHandler(ServiceException.class)
public CommonResult<?> serviceExceptionHandler(ServiceException ex) {
    return CommonResult.error(ex.getCode(), ex.getMessage());
}
```

### 1.3 设计推理

| 维度 | 异常继承体系 | final + 错误码 |
|------|------------|---------------|
| 异常类型数量 | 每个业务场景一个类 | 只有一个类 |
| GlobalExceptionHandler 复杂度 | 每个异常一个 handler | 一个 handler 处理所有 |
| 新增业务异常 | 新建类 + 注册 handler | 新增错误码常量 |
| 编译期安全 | 强（catch 特定异常） | 弱（错误码是 int） |
| 可读性 | `catch (OrderNotFoundException)` 直观 | `ex.getCode() == 1001001` 不直观 |

yudao-cloud 选择了右边。核心理由：**减少类的数量**。一个大型电商系统可能有几百种业务异常，如果每种都建一个类，光异常类就有几百个。而错误码只是一个 int 常量，管理成本低得多。

### 1.4 错误码设计：10 位数字的结构化

```
T-SSS-MMM-EEE
│   │   │   │
│   │   │   └── 错误序号（000-999）
│   │   └────── 模块编号（000-999）
│   └────────── 子系统编号（0-9）
└────────────── 类型（1=用户错误，2=系统错误）
```

示例：`1001001` = 用户错误(1) + 系统模块(001) + 第一个错误(001)

这个结构让错误码本身就携带了上下文信息（哪个子系统、哪个模块、什么类型），不需要查文档就能大致定位问题。

### 1.5 决策框架

```
异常体系选择：
├── 项目规模小（<50 个业务异常）→ 异常继承也可以
├── 项目规模大（>100 个业务异常）→ final + 错误码
├── 需要 catch 特定异常做特殊处理？ → 异常继承
└── 异常处理逻辑统一（只返回 code+msg）？ → final + 错误码
```

## 2. API 响应风格选型

### 2.1 三种方案

| 方案 | 成功响应 | 错误响应 | 例子 |
|------|---------|---------|------|
| A: 裸对象 | 直接返回业务对象 | HTTP 状态码 | `200 {"name":"商品A"}` / `400 Bad Request` |
| B: 业务码包装 | `{"code":0, "data":...}` | `{"code":1001001, "msg":"..."}` | yudao-cloud 的选择 |
| C: 纯 RESTful | HTTP 状态码 + 业务对象 | HTTP 状态码 + 错误体 | `201 Created` / `422 Unprocessable Entity` |

### 2.2 yudao-cloud 选择方案 B 的理由

1. **前端处理简单**：前端只需要判断 `code === 0`，不需要同时处理 HTTP 状态码和业务码
2. **语义不混淆**：HTTP 状态码表达传输层状态（请求是否到达、格式是否正确），业务码表达业务层状态（订单是否存在、余额是否足够）
3. **`code=0` 表示成功**：这是中文 Java 社区的惯例，不是 HTTP 的 `200`

### 2.3 取舍

方案 B 的缺点是违反了 RESTful 的语义——所有响应都是 HTTP 200，错误信息在 body 中。这使得 HTTP 层的监控工具（如按状态码统计错误率）失效。

缓解方式：`GlobalExceptionHandler` 对 HTTP 层错误（400、404、405、500）仍然设置正确的 HTTP 状态码，只有业务层错误用 200 + 业务码。

## 3. CommonApi：框架调业务的依赖反转

### 3.1 问题

框架 Starter（如日志 Starter）需要调用业务模块保存日志，但框架不能依赖业务模块。

### 3.2 模式

```
yudao-common（最底层）
  ├── 定义 ApiErrorLogCommonApi（接口）
  └── 定义 /rpc-api/ URL 前缀约定

yudao-spring-boot-starter-web（框架层）
  └── 注入 ApiErrorLogCommonApi，调用接口方法

yudao-module-system-server（业务层）
  └── 实现 ApiErrorLogCommonApi，提供实际逻辑
```

这本质上是依赖反转原则（DIP）在模块层面的应用：框架依赖抽象（接口），业务实现抽象。

### 3.3 两层 API 体系

| 层级 | 命名 | 稳定性 | 数量 | 示例 |
|------|------|--------|------|------|
| CommonApi | `*CommonApi` | 极高（框架级） | 1-3 个/模块 | `ApiErrorLogCommonApi`、`AuthCommonApi` |
| 业务 Api | `*Api` | 中（业务级） | 按需 | `AdminUserApi`、`ProductApi` |

CommonApi 的数量必须严格控制——它是框架和业务之间的"窄接口"，变更成本极高（所有依赖方都要改）。如果一个接口有 10 个方法，考虑拆分。

### 3.4 optional 依赖策略

API 模块中 OpenFeign 和 Validation 用 `optional=true` 而非 `provided`：

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
    <optional>true</optional>
</dependency>
```

区别：
- `provided`：依赖不会传递，运行时由容器提供（如 Servlet API）
- `optional`：依赖不会传递，但运行时可选（有则用，没有不报错）

`optional` 适合"单体模式下不需要 Feign，但微服务模式下需要"的场景。`provided` 适合"运行环境一定提供"的场景。

## 4. 工具类设计哲学

### 4.1 薄封装原则

yudao-cloud 的工具类不是从零实现，而是对 Hutool/Jackson/Stream 的薄封装，增加：
- null 安全检查
- 项目特定类型支持（如 `PageResult`）
- Lambda 字段名提取（如 `BeanUtils.toBean`）

### 4.2 BeanUtils vs MapStruct

| 维度 | BeanUtils（反射） | MapStruct（编译期代码生成） |
|------|-----------------|------------------------|
| 性能 | 慢（反射调用） | 快（生成的代码等同手写） |
| 灵活性 | 运行时动态（可处理 Map → DO） | 编译期静态（需要明确的类型） |
| 复杂映射 | 不支持（同名属性才能映射） | 支持（`@Mapping` 注解） |
| 适用场景 | 简单的同名属性拷贝 | 复杂的跨对象数据组装 |

yudao-cloud 两种都用：简单的用 `BeanUtils.toBean()`，复杂的用 MapStruct `*Convert` 接口。

## 小结

| 模式/决策 | 核心思想 |
|----------|---------|
| final ServiceException | 用错误码替代异常继承，减少类数量 |
| 业务码包装响应 | HTTP 状态码和业务码分离 |
| CommonApi 依赖反转 | 框架通过接口调用业务，控制接口数量 |
| optional vs provided | 运行时可选 vs 运行时必须 |
| 薄封装工具类 | 不造轮子，包装已有库增加 null 安全 |
