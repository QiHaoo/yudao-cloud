# 实体基类与自动填充设计

---

## 1. 从一个"谁改了这条数据"的需求说起

电商后台出了一个 bug：某商品的价格被改成了 0 元，导致以亏本价卖出了 100 单。运营找到你，问"谁改的？什么时候改的？"

你查看数据库，发现 `update_time` 是 NULL，`updater` 也是 NULL。原来负责写商品模块的同事忘了在更新时设置这两个字段。

这不是个例。在实际项目中，审计字段的缺失是常见问题：

```
场景 1：新来的开发者不知道要设置 audit 字段
场景 2：紧急修复 bug 时手忙脚乱，忘了设置
场景 3：通过批量更新脚本修改数据，脚本里没有 audit 逻辑
场景 4：前端直接把整个对象传过来，audit 字段被覆盖成空值
```

我们需要一种机制，让审计字段的填充**自动发生**，开发者不需要关心，也无法遗忘。

## 2. BaseDO：所有实体的基类

yudao-cloud 定义了 `BaseDO` 作为所有数据对象的基类：

```java
// 源码位置：cn.iocoder.yudao.framework.mybatis.core.dataobject.BaseDO
@Data
public abstract class BaseDO implements Serializable, TransPojo {

    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime createTime;

    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updateTime;

    @TableField(fill = FieldFill.INSERT, jdbcType = JdbcType.VARCHAR)
    private String creator;

    @TableField(fill = FieldFill.INSERT_UPDATE, jdbcType = JdbcType.VARCHAR)
    private String updater;

    @TableLogic
    private Boolean deleted;
}
```

### 字段设计决策

**为什么 `creator` 和 `updater` 用 String 而不是 Long？**

虽然目前存的是用户 ID（Long 类型），但注释说明了原因："未来可能会存在非数值的情况，留好拓展性"。比如：
- 系统定时任务自动创建的数据，creator 可能存 "SYSTEM"
- 第三方平台同步的数据，creator 可能存 "WECHAT_SYNC"
- 预留了对 UUID 主键的支持

**为什么 `@TableField(fill = ...)` 有两个级别？**

| 级别 | 含义 | 哪些字段使用 |
|------|------|-------------|
| `FieldFill.INSERT` | 仅在 INSERT 时自动填充 | `createTime`、`creator` |
| `FieldFill.INSERT_UPDATE` | INSERT 和 UPDATE 时都自动填充 | `updateTime`、`updater` |

这符合审计语义：创建时间和创建人只在创建时记录一次，更新时间和更新人在每次修改时都要更新。

**为什么 `deleted` 用 Boolean 而不是 Integer？**

语义更清晰：`deleted = true` 表示已删除，`deleted = false` 表示未删除。MyBatis-Plus 的 `@TableLogic` 支持 Boolean 类型，查询时自动追加 `WHERE deleted = false`。

### TenantBaseDO：多租户扩展

在多租户场景下，需要在每张表上记录数据属于哪个租户。`TenantBaseDO` 在 `BaseDO` 基础上增加了 `tenantId` 字段：

```java
// 源码位置：cn.iocoder.yudao.framework.tenant.core.db.TenantBaseDO
@Data
@EqualsAndHashCode(callSuper = true)
public abstract class TenantBaseDO extends BaseDO {

    /**
     * 多租户编号
     */
    private Long tenantId;
}
```

注意：`TenantBaseDO` 不在 mybatis starter 中，而是在 tenant starter 中。这是有意为之——不是所有项目都需要多租户功能，如果放在 mybatis starter 中，所有 DO 都会多一个不必要的字段。

继承关系：

```
BaseDO（mybatis starter）
  ├── 所有非租户业务模块的 DO 直接继承
  │
  └── TenantBaseDO（tenant starter）
        ├── AdminUserDO
        ├── DeptDO
        ├── RoleDO
        └── ... 所有需要租户隔离的 DO
```

## 3. clean() 方法：防御前端覆盖审计字段

`BaseDO` 中有一个容易被忽略但非常重要的方法：

```java
public void clean() {
    this.creator = null;
    this.createTime = null;
    this.updater = null;
    this.updateTime = null;
}
```

### 为什么需要 clean()

考虑这个场景：前端提交了一个更新请求，JSON 体中包含了所有字段：

```json
{
    "id": 1,
    "username": "zhangsan",
    "nickname": "张三",
    "createTime": "2024-01-01 00:00:00",
    "creator": "1",
    "updateTime": "2024-01-01 00:00:00",
    "updater": "1"
}
```

反序列化后，`createTime` 和 `creator` 被前端传入的值覆盖了。如果你直接把这个对象传给 MyBatis-Plus 的 `updateById`，`DefaultDBFieldHandler` 会发现 `createTime` 不是 null，就不填充了——前端传入的错误值被直接写入数据库。

`clean()` 方法的作用是：在更新操作前，清空审计字段，让自动填充机制重新赋值。

```java
// 典型用法
public void updateUser(AdminUserUpdateReqVO reqVO) {
    AdminUserDO user = BeanUtils.toBean(reqVO, AdminUserDO.class);
    user.clean();  // 清空审计字段，避免前端传入的值覆盖
    userMapper.updateById(user);
}
```

## 4. DefaultDBFieldHandler：自动填充的实现

`DefaultDBFieldHandler` 是 `MetaObjectHandler` 的实现类，负责在插入和更新时自动填充审计字段。

### 插入时的填充逻辑

```java
// 源码位置：cn.iocoder.yudao.framework.mybatis.core.handler.DefaultDBFieldHandler
@Override
public void insertFill(MetaObject metaObject) {
    if (Objects.nonNull(metaObject) && metaObject.getOriginalObject() instanceof BaseDO) {
        BaseDO baseDO = (BaseDO) metaObject.getOriginalObject();

        LocalDateTime current = LocalDateTime.now();
        // 创建时间为空，则以当前时间为插入时间
        if (Objects.isNull(baseDO.getCreateTime())) {
            baseDO.setCreateTime(current);
        }
        // 更新时间为空，则以当前时间为更新时间
        if (Objects.isNull(baseDO.getUpdateTime())) {
            baseDO.setUpdateTime(current);
        }

        Long userId = SecurityFrameworkUtils.getLoginUserId();
        // 当前登录用户不为空，创建人为空，则当前登录用户为创建人
        if (Objects.nonNull(userId) && Objects.isNull(baseDO.getCreator())) {
            baseDO.setCreator(userId.toString());
        }
        // 当前登录用户不为空，更新人为空，则当前登录用户为更新人
        if (Objects.nonNull(userId) && Objects.isNull(baseDO.getUpdater())) {
            baseDO.setUpdater(userId.toString());
        }
    }
}
```

几个关键设计：

1. **"为空才填充"原则**。如果业务代码已经显式设置了值（比如数据迁移脚本），自动填充不会覆盖。这保留了灵活性。

2. **从 SecurityFrameworkUtils 获取当前用户**。这依赖于 Spring Security 的上下文。在定时任务、消息队列消费者等没有用户上下文的场景中，`getLoginUserId()` 返回 null，creator/updater 就不会被填充（保持 null）。

3. **creator 和 updater 都转成 String**。虽然用户 ID 是 Long，但统一转成 String 存储，与字段类型一致。

### 更新时的填充逻辑

```java
@Override
public void updateFill(MetaObject metaObject) {
    // 更新时间为空，则以当前时间为更新时间
    Object modifyTime = getFieldValByName("updateTime", metaObject);
    if (Objects.isNull(modifyTime)) {
        setFieldValByName("updateTime", LocalDateTime.now(), metaObject);
    }

    // 当前登录用户不为空，更新人为空，则当前登录用户为更新人
    Object modifier = getFieldValByName("updater", metaObject);
    Long userId = SecurityFrameworkUtils.getLoginUserId();
    if (Objects.nonNull(userId) && Objects.isNull(modifier)) {
        setFieldValByName("updater", userId.toString(), metaObject);
    }
}
```

更新时只填充 `updateTime` 和 `updater`，不碰 `createTime` 和 `creator`——这符合审计语义。

### 自动填充的时序

```
业务代码调用 mapper.insert(entity) 或 mapper.updateById(entity)
    │
    ▼
MyBatis-Plus 拦截到操作，检查实体字段上的 @TableField(fill=...)
    │
    ▼
发现需要填充的字段，调用 DefaultDBFieldHandler
    │
    ▼
DefaultDBFieldHandler 检查字段是否已有值
    ├── 有值 → 不覆盖（尊重业务代码的显式赋值）
    └── 无值 → 自动填充（当前时间 / 当前用户）
    │
    ▼
MyBatis 执行 SQL，将填充后的值写入数据库
```

## 5. @TableLogic：统一的软删除

`BaseDO` 中的 `deleted` 字段标注了 `@TableLogic`：

```java
@TableLogic
private Boolean deleted;
```

这行注解的效果：

| 操作 | 原始 SQL | 实际 SQL |
|------|---------|---------|
| `deleteById(1)` | `DELETE FROM table WHERE id = 1` | `UPDATE table SET deleted = true WHERE id = 1 AND deleted = false` |
| `selectById(1)` | `SELECT * FROM table WHERE id = 1` | `SELECT * FROM table WHERE id = 1 AND deleted = false` |
| `selectList(wrapper)` | `SELECT * FROM table WHERE ...` | `SELECT * FROM table WHERE ... AND deleted = false` |

所有继承 `BaseDO` 的表自动获得软删除能力，不需要额外配置。

### 软删除与物理删除的取舍

| 维度 | 软删除 | 物理删除 |
|------|--------|---------|
| 数据恢复 | 容易，改 deleted 标记即可 | 困难，需要从备份恢复 |
| 审计追溯 | 可以查到"谁在什么时候删的" | 数据已不存在 |
| 存储空间 | 占用更多空间 | 立即释放 |
| 查询性能 | 所有查询都多一个 `AND deleted = false` 条件 | 无额外开销 |
| 唯一约束 | 同一记录"删了再建"可能冲突 | 不冲突 |

yudao-cloud 选择软删除作为默认策略，因为电商系统中数据的审计追溯非常重要。但通过配置，你可以对特定表使用物理删除（设置 `@TableLogic(value = "false", delval = "false")` 即可禁用）。

## 6. 完整的继承体系

将 BaseDO、TenantBaseDO 和业务 DO 的关系放在一起看：

```
BaseDO
├── createTime      @TableField(fill = INSERT)
├── updateTime      @TableField(fill = INSERT_UPDATE)
├── creator         @TableField(fill = INSERT)
├── updater         @TableField(fill = INSERT_UPDATE)
├── deleted         @TableLogic
└── clean()         清空审计字段
    │
    ├── 直接继承（无租户场景）
    │   ├── PayAppDO
    │   ├── PayChannelDO
    │   └── ...
    │
    └── TenantBaseDO extends BaseDO
        ├── tenantId
        │
        ├── AdminUserDO extends TenantBaseDO
        │   ├── id           @TableId
        │   ├── username
        │   ├── password
        │   └── ...
        │
        ├── DeptDO extends TenantBaseDO
        │   ├── id           @TableId
        │   ├── name
        │   ├── parentId
        │   └── ...
        │
        └── RoleDO extends TenantBaseDO
            ├── id           @TableId
            ├── name
            └── ...
```

3. `clean()` 方法清空了四个审计字段。如果前端传入的 `createTime` 是正确的（比如数据导入场景），`clean()` 会误清吗？你认为 `clean()` 应该在什么时机调用？
