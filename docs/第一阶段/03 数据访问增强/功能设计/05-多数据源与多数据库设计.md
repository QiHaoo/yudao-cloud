# 多数据源与多数据库设计

---

## 1. 从一个"换个数据库"的需求说起

你的电商系统一直用 MySQL 开发，一切顺利。突然，客户说："我们的生产环境只能用 PostgreSQL，因为公司标准。"

你开始检查代码，发现了这些需要改的地方：

1. **ID 生成**：MySQL 用自增（`AUTO_INCREMENT`），PostgreSQL 用序列（`SEQUENCE`）。你的实体类 `@TableId` 策略要改。
2. **SQL 语法**：MySQL 的 `FIND_IN_SET` 函数在 PostgreSQL 中不存在，需要用 `POSITION` 替代。
3. **分页语法**：虽然 MySQL 和 PostgreSQL 都用 `LIMIT`，但 Oracle 用 `ROWNUM`，SQL Server 用 `TOP`。
4. **Quartz 定时任务**：不同数据库的 `driverDelegateClass` 不同。

如果这些差异散落在每个业务模块中，"换数据库"就变成了一场噩梦。我们需要把这些差异集中管理，让业务代码不感知数据库类型。

## 2. 多数据源：dynamic-datasource

在生产环境中，一个应用可能需要连接多个数据库。yudao-cloud 集成了 `dynamic-datasource` 库来实现多数据源管理。

### 2.1 配置方式

在 `application.yaml` 中配置多个数据源：

```yaml
spring:
  datasource:
    dynamic:
      primary: master            # 默认数据源
      datasource:
        master:                  # 主库
          url: jdbc:mysql://localhost:3306/yudao
          username: root
          password: root123
        slave:                   # 从库
          url: jdbc:mysql://slave-host:3306/yudao
          username: readonly
          password: readonly123
```

### 2.2 @DS 注解切换数据源

通过 `@DS` 注解可以在方法级别切换数据源：

```java
@DS("slave")  // 使用从库
public List<ProductDO> listProducts() {
    return productMapper.selectList();
}
```

yudao-cloud 定义了数据源常量：

```java
// 源码位置：cn.iocoder.yudao.framework.datasource.core.enums.DataSourceEnum
public interface DataSourceEnum {
    String MASTER = "master";  // 主库，推荐使用 @Master 注解
    String SLAVE = "slave";    // 从库，推荐使用 @Slave 注解
}
```

实际上，`dynamic-datasource` 库本身就提供了 `@Master` 和 `@Slave` 注解，比 `@DS("master")` 语义更清晰。

### 2.3 数据源路由的原理

```
Service 方法调用
    │
    ▼
dynamic-datasource 的 AOP 拦截器
    │  检查方法/类上的 @DS 注解
    ▼
DynamicRoutingDataSource
    │  根据注解值选择对应的数据源
    ▼
对应的 DataSource（master 或 slave）
    │
    ▼
Connection → SQL 执行
```

`DynamicRoutingDataSource` 实现了 `javax.sql.DataSource` 接口，对上层代码透明——Mapper 不需要知道当前用的是哪个数据源。

## 3. ID 生成策略的自动适配

不同数据库的主键生成策略不同：

| 数据库 | 常用 ID 策略 | 机制 |
|--------|------------|------|
| MySQL | 自增 | `AUTO_INCREMENT` |
| PostgreSQL | 序列 | `SEQUENCE` + `nextval()` |
| Oracle | 序列 | `SEQUENCE` + `nextval()` |
| 达梦 | 自增 | 类似 MySQL |
| SQL Server | 自增 | `IDENTITY` |

如果在实体类上硬编码 `@TableId(type = IdType.AUTO)`，换到 PostgreSQL 就报错。反之硬编码 `IdType.INPUT`，MySQL 又不适用。

### 3.1 IdTypeEnvironmentPostProcessor 的解决方案

yudao-cloud 的方案是：**配置 `id-type: NONE`，启动时自动检测数据库类型并切换**。

```java
// 源码位置：cn.iocoder.yudao.framework.mybatis.config.IdTypeEnvironmentPostProcessor
@Slf4j
public class IdTypeEnvironmentPostProcessor implements EnvironmentPostProcessor {

    // 使用 INPUT（用户输入 ID）的数据库类型
    private static final Set<DbType> INPUT_ID_TYPES = SetUtils.asSet(
        DbType.ORACLE, DbType.ORACLE_12C,
        DbType.POSTGRE_SQL, DbType.KINGBASE_ES, DbType.DB2, DbType.H2);

    @Override
    public void postProcessEnvironment(ConfigurableEnvironment environment, SpringApplication application) {
        DbType dbType = getDbType(environment);
        if (dbType == null) return;

        // 如果用户已经显式配置了 id-type（非 NONE），不进行处理
        IdType idType = getIdType(environment);
        if (idType != IdType.NONE) return;

        // Oracle、PostgreSQL 等 → INPUT（由序列生成 ID）
        if (INPUT_ID_TYPES.contains(dbType)) {
            setIdType(environment, IdType.INPUT);
            return;
        }
        // MySQL、达梦等 → AUTO（数据库自增）
        setIdType(environment, IdType.AUTO);
    }
}
```

工作流程：

```
应用启动
    │
    ▼
IdTypeEnvironmentPostProcessor 执行（比 MyBatis-Plus 自动配置更早）
    │
    ├── 读取 spring.datasource.dynamic.primary
    ├── 读取主库的 JDBC URL
    ├── JdbcUtils.getDbType(url) 识别数据库类型
    │
    ├── Oracle/PostgreSQL/Kingbase/H2 → 设置 id-type = INPUT
    │   └── 配合 @KeySequence 注解使用序列生成 ID
    │
    └── MySQL/达梦 → 设置 id-type = AUTO
        └── 使用数据库自增
```

### 3.2 KeyGenerator 的注册

对于使用 `INPUT` 策略的数据库，还需要注册对应的 `KeyGenerator`：

```java
// 源码位置：YudaoMybatisAutoConfiguration
@Bean
@ConditionalOnProperty(prefix = "mybatis-plus.global-config.db-config", name = "id-type", havingValue = "INPUT")
public IKeyGenerator keyGenerator(ConfigurableEnvironment environment) {
    DbType dbType = IdTypeEnvironmentPostProcessor.getDbType(environment);
    switch (dbType) {
        case POSTGRE_SQL: return new PostgreKeyGenerator();
        case ORACLE:
        case ORACLE_12C:  return new OracleKeyGenerator();
        case H2:          return new H2KeyGenerator();
        case KINGBASE_ES: return new KingbaseKeyGenerator();
        case DM:          return new DmKeyGenerator();
    }
    throw new IllegalArgumentException("找不到合适的 IKeyGenerator");
}
```

`KeyGenerator` 的作用：在 INSERT 之前，先从数据库序列获取下一个 ID 值，设置到实体对象中。这样 MyBatis-Plus 执行 INSERT 时就有了 ID。

`@ConditionalOnProperty` 确保这个 Bean 只在 `id-type = INPUT` 时才注册，MySQL 环境下不会创建多余的 Bean。

### 3.3 @KeySequence 注解

对于使用序列的数据库，实体类需要标注 `@KeySequence`：

```java
@TableName("system_users")
@KeySequence("system_users_seq")  // 序列名
public class AdminUserDO extends TenantBaseDO {
    @TableId
    private Long id;
}
```

`@KeySequence` 对 MySQL 不生效（被忽略），所以写在实体类上不会有副作用。这是一种"兼容性注解"——在任何数据库下都能正常工作。

## 4. 数据库类型的枚举与 SQL 差异

### 4.1 DbTypeEnum

yudao-cloud 定义了自己的数据库类型枚举，补充了 MyBatis-Plus `DbType` 缺少的信息：

```java
// 源码位置：cn.iocoder.yudao.framework.mybatis.core.enums.DbTypeEnum
public enum DbTypeEnum {
    H2(DbType.H2, "H2", ""),
    MY_SQL(DbType.MYSQL, "MySQL", "FIND_IN_SET('#{value}', #{column}) <> 0"),
    ORACLE(DbType.ORACLE, "Oracle", "FIND_IN_SET('#{value}', #{column}) <> 0"),
    POSTGRE_SQL(DbType.POSTGRE_SQL, "PostgreSQL", "POSITION('#{value}' IN #{column}) <> 0"),
    SQL_SERVER(DbType.SQL_SERVER, "Microsoft SQL Server", "CHARINDEX(...)..."),
    DM(DbType.DM, "DM DBMS", "FIND_IN_SET('#{value}', #{column}) <> 0"),
    KINGBASE_ES(DbType.KINGBASE_ES, "KingbaseES", "POSITION('#{value}' IN #{column}) <> 0"),
    OCEAN_BASE(DbType.OCEAN_BASE, "OceanBase", "FIND_IN_SET('#{value}', #{column}) <> 0");
    // ...
}
```

每个枚举值包含三个信息：
1. **mpDbType**：MyBatis-Plus 的数据库类型
2. **productName**：JDBC 元数据中的数据库产品名（用于匹配）
3. **findInSetTemplate**：该数据库的 `FIND_IN_SET` 等价 SQL

### 4.2 FIND_IN_SET 的跨数据库兼容

`FIND_IN_SET` 是 MySQL 特有的函数，用于判断一个值是否在一个逗号分隔的字符串中：

```sql
-- MySQL
SELECT * FROM product WHERE FIND_IN_SET('1', category_ids);

-- PostgreSQL（用 POSITION 替代）
SELECT * FROM product WHERE POSITION('1' IN category_ids) <> 0;

-- SQL Server（用 CHARINDEX 替代）
SELECT * FROM product WHERE CHARINDEX(',' + '1' + ',', ',' + category_ids + ',') <> 0;
```

`MyBatisUtils.findInSet` 方法封装了这个差异：

```java
// 源码位置：cn.iocoder.yudao.framework.mybatis.core.util.MyBatisUtils
public static String findInSet(String column, Object value) {
    DbType dbType = JdbcUtils.getDbType();
    return DbTypeEnum.getFindInSetTemplate(dbType)
            .replace("#{column}", column)
            .replace("#{value}", StrUtil.toString(value));
}
```

业务代码只需要调用 `MyBatisUtils.findInSet("category_ids", categoryId)`，不用关心当前是什么数据库。

### 4.3 JdbcUtils：数据库类型检测

```java
// 源码位置：cn.iocoder.yudao.framework.mybatis.core.util.JdbcUtils
public static DbType getDbType(String url) {
    return com.baomidou.mybatisplus.extension.toolkit.JdbcUtils.getDbType(url);
}

public static DbType getDbType() {
    DataSource dataSource;
    try {
        DynamicRoutingDataSource dynamicRoutingDataSource = SpringUtils.getBean(DynamicRoutingDataSource.class);
        dataSource = dynamicRoutingDataSource.determineDataSource();
    } catch (NoSuchBeanDefinitionException e) {
        dataSource = SpringUtils.getBean(DataSource.class);
    }
    try (Connection conn = dataSource.getConnection()) {
        return DbTypeEnum.find(conn.getMetaData().getDatabaseProductName());
    } catch (SQLException e) {
        throw new IllegalArgumentException(e.getMessage());
    }
}
```

两种重载：
- `getDbType(String url)`：从 JDBC URL 解析数据库类型（用于启动阶段，还没有连接）
- `getDbType()`：从当前数据源连接获取数据库类型（用于运行时，兼容多数据源场景）

运行时的 `getDbType()` 方法会优先尝试获取 `DynamicRoutingDataSource`（多数据源场景），如果不存在则回退到普通的 `DataSource`。

## 5. Quartz 定时任务的数据库适配

`IdTypeEnvironmentPostProcessor` 不仅处理 ID 策略，还顺便处理了 Quartz 的数据库适配：

```java
public void setJobStoreDriverIfPresent(ConfigurableEnvironment environment, DbType dbType) {
    String driverClass = environment.getProperty(QUARTZ_JOB_STORE_DRIVER_KEY);
    if (StrUtil.isNotEmpty(driverClass)) return;  // 用户已配置，不覆盖

    switch (dbType) {
        case POSTGRE_SQL:
            driverClass = "org.quartz.impl.jdbcjobstore.PostgreSQLDelegate";
            break;
        case ORACLE:
        case ORACLE_12C:
            driverClass = "org.quartz.impl.jdbcjobstore.oracle.OracleDelegate";
            break;
        case SQL_SERVER:
        case SQL_SERVER2005:
            driverClass = "org.quartz.impl.jdbcjobstore.MSSQLDelegate";
            break;
        case DM:
        case KINGBASE_ES:
            driverClass = "org.quartz.impl.jdbcjobstore.StdJDBCDelegate";
            break;
    }
}
```

Quartz 的 `JobStore` 需要知道数据库方言才能正确操作数据库表。如果不配置 `driverDelegateClass`，Quartz 在 PostgreSQL 上可能无法正常工作。这个自动配置让用户不需要手动设置。

## 6. 完整的数据库兼容架构

```
应用启动
    │
    ▼
IdTypeEnvironmentPostProcessor
    ├── 从 JDBC URL 检测数据库类型
    ├── 设置 id-type（AUTO 或 INPUT）
    └── 设置 Quartz driverDelegateClass
    │
    ▼
YudaoMybatisAutoConfiguration
    ├── 注册分页插件
    ├── 注册自动填充处理器
    ├── 注册 KeyGenerator（仅 INPUT 模式）
    └── 注册 JacksonTypeHandler
    │
    ▼
运行时
    ├── BaseMapperX.insertBatch → 检测 SQL Server，降级为逐条插入
    ├── MyBatisUtils.findInSet  → 根据数据库类型选择 SQL 模板
    ├── QueryWrapperX.limitN    → 根据数据库类型选择分页语法
    └── JdbcUtils.getDbType     → 从当前数据源连接获取数据库类型
```

业务代码只需要：

1. 继承 `BaseDO` 或 `TenantBaseDO`
2. 继承 `BaseMapperX`
3. 使用 `LambdaQueryWrapperX` 构建查询
4. 不需要关心底层是 MySQL 还是 PostgreSQL

## 7. 设计决策：为什么不在运行时动态切换数据库方言？

一个自然的问题是：既然要支持多数据库，为什么不在运行时根据当前数据源动态切换方言，而是在启动时确定？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **启动时确定**（yudao-cloud 采用） | 简单可靠，无运行时开销 | 不支持同一个应用连接不同类型的数据库 |
| **运行时切换** | 灵活，支持混合数据库 | 复杂度高，每条 SQL 都要判断数据库类型 |

yudao-cloud 的实际场景是：一个部署实例只连接一种数据库（MySQL 或 PostgreSQL，不会混合）。即使有多数据源（主从），它们也是同类型的。所以"启动时确定"是最简单可靠的方案。

## 思考题

1. 如果你的电商系统需要同时连接 MySQL（主业务库）和 PostgreSQL（数据分析库），`IdTypeEnvironmentPostProcessor` 的方案还适用吗？你会怎么改造？
2. `MyBatisUtils.findInSet` 通过字符串模板替换生成 SQL。如果传入的 `value` 包含 SQL 特殊字符（如单引号），会不会有 SQL 注入风险？
3. `JdbcUtils.getDbType()` 在运行时通过获取数据库连接来检测数据库类型。如果连接池满了，这个调用会不会阻塞？你有什么优化思路？
4. 为什么 Quartz 的 `driverDelegateClass` 放在 `IdTypeEnvironmentPostProcessor` 中处理，而不是单独写一个 `EnvironmentPostProcessor`？这种"顺便处理"的设计有什么优缺点？
