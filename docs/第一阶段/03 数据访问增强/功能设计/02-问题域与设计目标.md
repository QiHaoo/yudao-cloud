# 数据访问增强：问题域与设计目标

---

## 1. 从一段"能跑但很累"的代码说起

假设你在开发电商后台的用户管理模块。使用原生 MyBatis-Plus 后，一个典型的 Mapper 方法长这样：

```java
@Mapper
public interface AdminUserMapper extends BaseMapper<AdminUserDO> {

    default PageResult<AdminUserDO> selectPage(UserPageReqVO reqVO) {
        LambdaQueryWrapper<AdminUserDO> wrapper = new LambdaQueryWrapper<>();
        // 问题 1：每个条件都要判断 null
        if (StringUtils.hasText(reqVO.getUsername())) {
            wrapper.like(AdminUserDO::getUsername, reqVO.getUsername());
        }
        if (reqVO.getStatus() != null) {
            wrapper.eq(AdminUserDO::getStatus, reqVO.getStatus());
        }
        if (reqVO.getDeptId() != null) {
            wrapper.eq(AdminUserDO::getDeptId, reqVO.getDeptId());
        }
        if (reqVO.getBeginCreateTime() != null && reqVO.getEndCreateTime() != null) {
            wrapper.between(AdminUserDO::getCreateTime, reqVO.getBeginCreateTime(), reqVO.getEndCreateTime());
        }
        wrapper.orderByDesc(AdminUserDO::getId);

        // 问题 2：分页结果要手动转换
        Page<AdminUserDO> page = new Page<>(reqVO.getPageNo(), reqVO.getPageSize());
        Page<AdminUserDO> result = selectPage(page, wrapper);
        return new PageResult<>(result.getRecords(), result.getTotal());
    }

    // 问题 3：简单的按字段查询也要写好几行
    default AdminUserDO selectByUsername(String username) {
        return selectOne(new LambdaQueryWrapper<AdminUserDO>()
            .eq(AdminUserDO::getUsername, username));
    }
}
```

这段代码能跑，但存在三个痛点：

**痛点一：空值判断的噪音**。分页查询通常有 5-10 个筛选条件，每个条件都要写 `if (xxx != null)`。这些判断和业务逻辑无关，纯粹是防御性编程的噪音。

**痛点二：分页结果类型转换**。MyBatis-Plus 返回的是 `Page<T>`，但业务层需要的是 `PageResult<T>`。每个分页方法都要做这个转换。

**痛点三：简单查询太啰嗦**。"按用户名查用户"是最常见的操作，但每次都得 new 一个 LambdaQueryWrapper，写 `.eq(...)`，再调 `selectOne`。

这些问题在每个 Mapper 中重复出现。当你的电商系统有 50 张表时，这些重复代码加起来可能有上千行。

## 2. 审计字段的另一个烦恼

除了查询的重复，还有一个贯穿所有表的问题：**审计字段**。

每张表都有 `create_time`、`update_time`、`creator`、`updater` 这些字段。业务代码不应该手动设置它们——创建时间应该是"现在"，创建人应该是"当前登录用户"。

但如果依赖开发者手动赋值，总有人会忘记：

```java
AdminUserDO user = new AdminUserDO();
user.setUsername("zhangsan");
user.setPassword("...");
// 忘了设置 createTime、creator → 数据库里这两个字段是 NULL
mapper.insert(user);
```

我们需要一种机制，让这些字段在插入和更新时**自动填充**，业务代码完全不需要关心。

---

## 3. 软删除的一致性问题

另一个需要统一处理的问题是**逻辑删除**。

在电商系统中，"删除"通常不是物理删除（`DELETE FROM`），而是标记删除（`UPDATE SET deleted = true`）。这样可以保留数据用于审计，也方便恢复。

但如果每张表自己实现软删除逻辑，就会出现不一致：
- 用户表用 `deleted` 字段（Boolean）
- 商品表用 `is_deleted` 字段（Integer，0/1）
- 订单表用 `del_flag` 字段（String，"0"/"1"）

查询时也要各自处理：有的表自动追加 `WHERE deleted = false`，有的表不会。这会让开发者困惑——"我明明删了这条数据，为什么还能查到？"

我们需要一个统一的软删除方案，让所有表使用相同的字段名、相同的类型、相同的查询过滤逻辑。

## 4. 多数据库的兼容挑战

yudao-cloud 是一个开源项目，用户可能使用不同的数据库：

| 数据库 | 使用场景 |
|--------|---------|
| MySQL | 最常见，开发和生产都用 |
| PostgreSQL | 部分企业的标准数据库 |
| Oracle | 大型企业、政府项目 |
| 达梦（DM） | 国产化替代 |
| SQL Server | 部分 .NET 转 Java 的企业 |
| 人大金仓 | 国产化替代 |

不同的数据库在 SQL 语法上有差异。例如"分页"：

```sql
-- MySQL / PostgreSQL / 达梦
SELECT * FROM table LIMIT 10 OFFSET 20

-- Oracle
SELECT * FROM (SELECT t.*, ROWNUM rn FROM table t WHERE ROWNUM <= 30) WHERE rn > 20

-- SQL Server
SELECT * FROM table ORDER BY id OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY
```

ID 生成策略也不同：MySQL 用自增，PostgreSQL 和 Oracle 用序列。如果硬编码一种策略，换数据库就要改代码。

我们需要一种机制，让数据访问层能**自动适配**不同的数据库，业务代码不需要感知差异。

## 5. 多数据源与读写分离

在生产环境中，电商系统通常有多个数据源：

```
                    ┌─────────────┐
                    │   应用程序   │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
       ┌──────▼──────┐  ┌─▼──────┐  ┌──▼───────┐
       │  主库(Master) │  │ 从库1  │  │  从库2   │
       │  读写        │  │ 只读   │  │  只读    │
       └─────────────┘  └────────┘  └──────────┘
```

- 写操作（INSERT/UPDATE/DELETE）走主库
- 读操作（SELECT）走从库，分担主库压力
- 不同业务模块可能使用不同的数据库（如系统模块用 MySQL，日志模块用 Elasticsearch）

我们需要一种方式，让开发者通过简单的注解就能切换数据源，而不是手动管理连接。

## 6. 设计目标

综合以上问题，`yudao-spring-boot-starter-mybatis` 的设计目标是：

| 目标 | 具体要求 |
|------|---------|
| **消除查询噪音** | `LambdaQueryWrapperX` 的 `*IfPresent` 方法自动跳过空值条件 |
| **统一分页接口** | `BaseMapperX.selectPage` 直接返回 `PageResult`，支持"不分页"模式 |
| **简化常用查询** | `selectOne(field, value)` 等快捷方法，一行代码完成单字段查询 |
| **审计字段自动填充** | `DefaultDBFieldHandler` 自动填充 createTime/updateTime/creator/updater |
| **统一软删除** | `BaseDO.deleted` + `@TableLogic`，所有表统一行为 |
| **多数据库兼容** | `IdTypeEnvironmentPostProcessor` 自动选择 ID 策略，`DbTypeEnum` 封装 SQL 差异 |
| **多数据源支持** | 集成 dynamic-datasource，通过 `@DS` 注解切换数据源 |
| **批量操作增强** | `insertBatch`/`updateBatch`/`deleteBatch`，兼容 SQL Server 等特殊数据库 |

## 7. 整体架构

以下是 yudao-spring-boot-starter-mybatis 的核心组件及其关系：

```
yudao-spring-boot-starter-mybatis
│
├── 实体层
│   ├── BaseDO              ← 所有 DO 的基类（审计字段 + 软删除）
│   └── TenantBaseDO        ← 多租户基类（在 tenant starter 中，加 tenantId）
│
├── Mapper 层
│   ├── BaseMapperX<T>      ← 所有 Mapper 的基接口
│   │   ├── selectPage      → 返回 PageResult，支持 PAGE_SIZE_NONE
│   │   ├── selectOne       → 单字段/双字段/三字段快捷查询
│   │   ├── selectOneForUpdate → 悲观锁查询
│   │   ├── insertBatch     → 批量插入（兼容 SQL Server）
│   │   ├── updateBatch     → 批量更新
│   │   └── deleteBatch     → 批量删除
│   └── MPJBaseMapper       ← MyBatis-Plus Join 提供的连表能力
│
├── 查询构建层
│   ├── LambdaQueryWrapperX ← 扩展 LambdaQueryWrapper
│   │   └── *IfPresent      → 自动跳过空值的条件方法
│   ├── QueryWrapperX       ← 扩展 QueryWrapper（字符串方式）
│   └── MPJLambdaWrapperX   ← 扩展 MPJLambdaWrapper（连表方式）
│
├── 自动填充
│   └── DefaultDBFieldHandler ← MetaObjectHandler 实现
│       ├── insertFill      → 填充 createTime、updateTime、creator、updater
│       └── updateFill      → 填充 updateTime、updater
│
├── 数据源
│   ├── DataSourceEnum      → MASTER/SLAVE 常量
│   └── dynamic-datasource  → 第三方库，提供 @DS 注解
│
├── 数据库兼容
│   ├── IdTypeEnvironmentPostProcessor → 启动时自动切换 ID 策略
│   ├── DbTypeEnum          → 数据库类型枚举 + SQL 模板
│   ├── JdbcUtils           → 从 JDBC URL/连接获取数据库类型
│   └── MyBatisUtils        → findInSet 等跨数据库工具方法
│
└── 自动配置
    ├── YudaoMybatisAutoConfiguration → 注册插件、填充处理器、KeyGenerator
    └── YudaoDataSourceAutoConfiguration → 事务管理、Druid 配置
```
