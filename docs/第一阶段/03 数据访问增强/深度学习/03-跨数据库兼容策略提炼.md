# 跨数据库兼容策略提炼

---

## 1. 总体策略：在哪里做抽象

跨数据库兼容有两种极端做法：

- **完全不抽象**：代码里写死 MySQL 语法，换数据库时逐个修改
- **完全抽象**：用 Hibernate 的 HQL 或 JOOQ 这样的数据库无关 DSL，彻底屏蔽 SQL 差异

yudao-cloud 选择了一条中间路线：**在差异点上做最小化抽象，其余地方直接写 SQL**。

```
抽象层（自己实现）          不抽象（直接用）
├── ID 策略自动检测          ├── SQL 语法（MyBatis XML/注解）
├── FIND_IN_SET 模板替换     ├── 分页（MyBatis-Plus 自动处理）
├── Quartz delegate 配置     ├── 基本 CRUD（ORM 框架处理）
└── SQL Server 批量插入降级
```

这个策略的前提是：项目使用的数据库差异点是有限的、可枚举的。如果差异点太多（比如同时支持 MySQL、Oracle、DB2 且大量使用各数据库特有函数），这个策略的维护成本会急剧上升。

---

## 2. 差异点一：ID 策略

### 2.1 问题

MySQL 用 `AUTO_INCREMENT`，PostgreSQL 用 `SEQUENCE`，Oracle 用 `SEQUENCE` 但语法不同。MyBatis-Plus 的 `@TableId(type = IdType.AUTO)` 只在 MySQL 上有效。

### 2.2 解决方案

`IdTypeEnvironmentPostProcessor` 在启动时读取 JDBC URL，自动将 `id-type: NONE`（哨兵值）替换为实际值：

| 数据库 | JDBC URL 特征 | 替换为 |
|--------|-------------|--------|
| MySQL | `jdbc:mysql://...` | `AUTO` |
| PostgreSQL | `jdbc:postgresql://...` | `INPUT` |
| Oracle | `jdbc:oracle://...` | `INPUT` |

业务代码中的实体类不需要任何改动：

```java
@TableName("product")
public class ProductDO extends BaseDO {
    @TableId(type = IdType.AUTO)  // 启动时自动适配
    private Long id;
}
```

### 2.3 局限

这个方案假设一个应用只连一种数据库。如果同一个应用同时连 MySQL 和 PostgreSQL（通过 `@DS` 切换数据源），启动期检测只能检测到主数据源的类型，从数据源的 ID 策略可能不正确。

这种情况需要运行时检测 + 动态设置 ID 策略，复杂度会显著上升。

## 3. 差异点二：FIND_IN_SET

### 3.1 问题

MySQL 的 `FIND_IN_SET('1', '1,2,3')` 函数用于判断一个值是否在逗号分隔的字符串中。这个函数在其他数据库中不存在，但语法差异很大：

| 数据库 | 等价写法 |
|--------|---------|
| MySQL | `FIND_IN_SET(column, '1,2,3') <> 0` |
| PostgreSQL | `POSITION(column IN '1,2,3') <> 0` |
| SQL Server | `CHARINDEX(',' + column + ',', ',1,2,3,') <> 0` |
| Oracle | `FIND_IN_SET(column, '1,2,3') <> 0`（Oracle 有同名函数） |
| H2 | 不支持 |

### 3.2 解决方案

`DbTypeEnum` 为每个数据库定义了 `FIND_IN_SET` 的 SQL 模板：

```java
public enum DbTypeEnum {
    MY_SQL(DbType.MYSQL, "MySQL", "FIND_IN_SET({0}, {1}) <> 0"),
    POSTGRE_SQL(DbType.POSTGRE_SQL, "PostgreSQL", "POSITION({0} IN {1}) <> 0"),
    SQL_SERVER(DbType.SQL_SERVER, "Microsoft SQL Server",
               "CHARINDEX(',' + {0} + ',', ',' + {1} + ',') <> 0"),
    // ...
}
```

`MyBatisUtils.findInSet()` 在运行时检测当前数据库类型，选择对应模板，替换占位符：

```java
public static Expression findInSet(String column, Object value) {
    DbType dbType = JdbcUtils.getDbType();
    String template = DbTypeEnum.getFindInSetTemplate(dbType);
    return new RawExpression(MessageFormat.format(template, column, value));
}
```

### 3.3 设计取舍

**为什么用模板字符串而不是 JSQLParser AST？**

JSQLParser 可以构建跨数据库的 SQL AST（抽象语法树），理论上更"正确"。但实际项目中：
- `FIND_IN_SET` 的差异点有限，模板字符串足够
- JSQLParser 学习成本高，团队不一定熟悉
- 模板字符串更直观，调试时直接看字符串比看 AST 树方便

**为什么在运行时检测而不是启动时？**

因为 `findInSet()` 可能在不同数据源上被调用。启动时只能检测主数据源，无法覆盖 `@DS` 切换的从数据源。

## 4. 差异点三：批量插入

### 4.1 问题

SQL Server 在批量插入后无法正确返回自增 ID。MyBatis-Plus 的 `Db.saveBatch()` 依赖 `useGeneratedKeys`，SQL Server 上会抛异常。

### 4.2 解决方案

`BaseMapperX.insertBatch()` 检测当前数据库类型，SQL Server 降级为逐行插入：

```java
default Boolean insertBatch(Collection<T> entities) {
    if (JdbcUtils.isSQLServer()) {
        for (T entity : entities) {
            insert(entity);  // 逐行插入，能正确获取 ID
        }
        return true;
    }
    return Db.saveBatch(entities);  // 其他数据库用批量插入
}
```

### 4.3 代价

逐行插入的性能远低于批量插入。对于 10000 条记录：
- 批量插入：1 次网络往返 + 数据库内部批量处理
- 逐行插入：10000 次网络往返 + 10000 次事务提交

如果 SQL Server 上的批量导入是高频操作，需要考虑替代方案：
1. 使用 SQL Server 的 `BULK INSERT` 语句
2. 先批量插入不带 ID，再用 `MERGE` 或 `OUTPUT` 获取 ID
3. 接受不返回 ID 的方案，批量插入后用其他字段查询回填

## 5. 差异点四：Quartz 委托类

### 5.1 问题

Quartz 的 `driverDelegateClass` 需要根据数据库类型选择：

| 数据库 | 委托类 |
|--------|--------|
| MySQL/PostgreSQL | `org.quartz.impl.jdbcjobstore.StdJDBCDelegate` |
| SQL Server | `org.quartz.impl.jdbcjobstore.MSSQLDelegate` |
| Oracle | `org.quartz.impl.jdbcjobstore.oracle.OracleDelegate` |

如果配置错误，Quartz 无法正常建表和调度。

### 5.2 解决方案

`IdTypeEnvironmentPostProcessor` 在检测 ID 策略的同时，也设置了 Quartz 的 `driverDelegateClass`。这是前面提到的"一个处理器做两件事"的设计争议点。

### 5.3 是否应该拆分

从职责分离的角度，应该拆为两个 `EnvironmentPostProcessor`：
- `IdTypeEnvironmentPostProcessor`：只管 ID 策略
- `QuartzDelegateEnvironmentPostProcessor`：只管 Quartz 配置

从简洁性的角度，两者都需要读 JDBC URL 判断数据库类型，合并可以避免重复代码。

判断标准：如果这两个配置的**变更频率不同**或**失败影响范围不同**，应该拆分。如果它们总是同时变更、同时失败（比如换数据库时都要改），合并是合理的。

## 6. 兼容策略的维护模式

### 6.1 增加新数据库支持的步骤

当需要支持一个新数据库（如达梦 DM）时：

1. **`DbTypeEnum` 中增加枚举值**：定义 `productName`、`findInSetTemplate`
2. **检查 `IdTypeEnvironmentPostProcessor`**：新数据库的 ID 策略是 `AUTO` 还是 `INPUT`
3. **检查 `insertBatch`**：新数据库是否支持批量插入返回 ID
4. **检查 Quartz `driverDelegateClass`**：新数据库是否有专用委托类
5. **检查业务代码中的原生 SQL**：是否有不兼容的函数或语法

步骤 1-4 都在框架层，改一处即可生效。步骤 5 需要逐个排查业务代码——这是最小化抽象策略的代价。

### 6.2 测试策略

跨数据库兼容的测试不能只在 MySQL 上跑。最低限度需要：

1. **启动测试**：用每种数据库的内存版本（H2、嵌入式 PostgreSQL）启动应用
2. **CRUD 测试**：对每个实体执行增删改查，覆盖自动填充、软删除、分页
3. **FIND_IN_SET 测试**：确保跨数据库函数替换正确
4. **批量操作测试**：特别是 SQL Server 的降级路径

H2 模式在测试中广泛使用，但 H2 不支持 `FIND_IN_SET`——这意味着 H2 测试无法覆盖这个差异点。如果项目大量使用 `FIND_IN_SET`，需要额外针对 PostgreSQL 或 SQL Server 做集成测试。

## 7. 业界方案对比

| 方案 | 抽象级别 | 优点 | 缺点 |
|------|---------|------|------|
| yudao-cloud（最小化抽象） | 函数级模板替换 | 简单直观，性能好 | 差异点多时维护成本高 |
| MyBatis-Plus 内置方言 | SQL 重写层 | 框架自动处理分页等 | 只覆盖常见差异 |
| JOOQ | DSL 级 | 类型安全，方言自动切换 | 学习成本高，与 MyBatis 生态不兼容 |
| Hibernate/JPA | HQL 级 | 最完整的抽象 | 与 MyBatis 生态不兼容 |
| Flyway/Liquibase 迁移脚本 | DDL 级 | 只管结构差异 | 不管运行时 SQL 差异 |

yudao-cloud 的选择适合"主要用 MySQL，偶尔需要兼容其他数据库"的场景。如果项目需要在多种数据库上平等运行，JOOQ 或 Hibernate 是更合适的选择。

---

## 小结

跨数据库兼容的核心问题不是"怎么写兼容代码"，而是"在哪些点上做抽象"。yudao-cloud 的策略可以概括为：

1. **识别有限的差异点**：ID 策略、特定函数、批量操作、第三方库配置
2. **在差异点上做最小化抽象**：模板替换、启动期探测、降级处理
3. **其余地方直接写 SQL**：利用 MyBatis-Plus 和 MyBatis-Plus Join 处理大部分兼容

这个策略的适用边界是：差异点可枚举，且不需要在多种数据库上平等运行。超出这个边界，需要更系统化的数据库抽象层。
