# 设计模式提炼

---

本章从数据访问增强组件中提取四个可复用的设计模式。这些模式不是"最佳实践清单"，而是从实际代码中识别出的、在其他项目中遇到类似问题时可以直接套用的解决方案。

---

## 1. 条件链式扩展模式（*IfPresent）

### 1.1 问题：空值判断的"噪音污染"

任何接受查询参数的 API 都面临同一个问题：前端传来的参数可能为空，而业务逻辑要求"空参数意味着不过滤"。最直觉的写法：

```java
LambdaQueryWrapper<AdminUserDO> wrapper = new LambdaQueryWrapper<>();
if (reqVO.getUsername() != null) {
    wrapper.like(AdminUserDO::getUsername, reqVO.getUsername());
}
if (reqVO.getStatus() != null) {
    wrapper.eq(AdminUserDO::getStatus, reqVO.getStatus());
}
if (reqVO.getBeginTime() != null && reqVO.getEndTime() != null) {
    wrapper.between(AdminUserDO::getCreateTime, reqVO.getBeginTime(), reqVO.getEndTime());
} else if (reqVO.getBeginTime() != null) {
    wrapper.ge(AdminUserDO::getCreateTime, reqVO.getBeginTime());
} else if (reqVO.getEndTime() != null) {
    wrapper.le(AdminUserDO::getCreateTime, reqVO.getEndTime());
}
```

6 个条件就占了 12 行，其中一半是 `if` 判断。更糟糕的是，`between` 的"单边有值"场景需要额外写 `else if` 分支——这个边界情况很容易被遗忘。

### 1.2 模式：继承 + 条件守卫 + 返回类型协变

`LambdaQueryWrapperX` 的解决方案是三个设计手法的组合：

**手法一：继承扩展**

```
LambdaQueryWrapper<T>          ← 框架提供的查询构建器
  └── LambdaQueryWrapperX<T>   ← 增加 *IfPresent 方法
```

不修改框架源码，不使用装饰器——直接继承，利用 Java 的方法覆盖机制。

**手法二：方法内部条件守卫**

每个 `*IfPresent` 方法内部先判断值是否有效，无效则直接 `return this`，不追加条件：

```java
public LambdaQueryWrapperX<T> likeIfPresent(SFunction<T, ?> column, String val) {
    if (!StringUtils.hasText(val)) {  // 空字符串、null、纯空白 → 跳过
        return this;
    }
    return (LambdaQueryWrapperX<T>) super.like(column, val);
}
```

关键细节：不同类型的"空"定义不同。字符串用 `StringUtils.hasText()`（拒绝 null、空串、纯空白），对象用 `ObjectUtil.isNotEmpty()`，集合用 `isEmpty()`。这比统一用 `!= null` 更严格，避免了 `"  "` 这样的无效值被当作有效查询条件。

**手法三：返回类型协变（Covariant Return Type）**

父类 `LambdaQueryWrapper` 的方法返回 `LambdaQueryWrapper<T>`。如果 `LambdaQueryWrapperX` 不覆盖这些方法，调用链会"断裂"：

```java
// 如果不覆盖 eq()，这行代码编译失败——eq() 返回的是父类类型
wrapperX.eqIfPresent(...)
         .eq(AdminUserDO::getStatus, 1)  // ← 返回 LambdaQueryWrapper，没有 eqIfPresent
         .likeIfPresent(...)              // ← 编译错误！
```

解决方案是覆盖所有父类方法，返回子类类型：

```java
@Override
public LambdaQueryWrapperX<T> eq(SFunction<T, ?> column, Object val) {
    return (LambdaQueryWrapperX<T>) super.eq(column, val);
}
```

### 1.3 模式的通用化

这个模式适用于任何"链式 API 需要增加条件性方法"的场景。通用结构：

```
原始构建器 Builder
  └── BuilderX extends Builder
        + withXIfValid(value, predicate)   ← 新增的条件方法
        + 原有方法覆盖为返回 BuilderX       ← 保持链式不断裂
```

适用场景举例：
- HTTP 请求构建器：`headerIfPresent(name, value)` 跳过 null header
- 配置构建器：`optionIfNonDefault(key, value)` 只设置非默认值
- 流式过滤器：`filterIf(predicate, element)` 条件性过滤

### 1.4 已知风险

这个模式有一个维护隐患：如果父类（MyBatis-Plus）在新版本中增加了方法，而 `LambdaQueryWrapperX` 没有覆盖它，链式调用会在新方法处返回父类类型，后续的 `*IfPresent` 方法就不可用了。

这是继承扩展的固有代价——子类与父类的 API 表面积耦合。缓解方式：
1. 升级 MyBatis-Plus 版本后，运行已有的查询测试
2. 如果框架提供了 `SFunction` 版本的方法但子类没有覆盖，至少编译期不会报错（只是运行时可能行为不符预期）

## 2. 非破坏性自动填充模式（Fill-if-Empty）

### 2.1 问题：审计字段的两种需求冲突

每个实体都需要 `createTime`、`updateTime`、`creator`、`updater` 这些审计字段。自动填充是标准做法，但有两种需求场景冲突：

- **正常业务操作**：用户通过 API 创建/修改数据，应该自动填充当前用户和时间
- **数据迁移/批量导入**：从旧系统迁移数据时，需要保留原始的创建时间和创建人

如果自动填充是"无条件覆盖"，迁移脚本就必须用原生 SQL 绕过 ORM，或者在迁移后手动修正数据。

### 2.2 模式：null 作为"请填充"信号

`DefaultDBFieldHandler` 的核心设计是：**只在字段为 null 时填充，已有的值不覆盖**。

```java
public void insertFill(MetaObject metaObject) {
    if (metaObject.getOriginalObject() instanceof BaseDO) {
        BaseDO baseDO = (BaseDO) metaObject.getOriginalObject();
        if (baseDO.getCreateTime() == null) {
            strictInsertFill(metaObject, "createTime", LocalDateTime.class, LocalDateTime.now());
        }
        if (baseDO.getCreator() == null) {
            Long userId = SecurityFrameworkUtils.getLoginUserId();
            if (userId != null) {
                strictInsertFill(metaObject, "creator", String.class, userId.toString());
            }
        }
        // ... updateTime, updater 同理
    }
}
```

语义约定：
- 字段为 `null` → "我不知道这个值，请帮我填"
- 字段有值 → "我知道这个值，请尊重我的设置"

这个约定让两种场景自然共存：
- API 请求不传审计字段 → null → 自动填充
- 迁移脚本显式设置 `createTime`、`creator` → 有值 → 保留

### 2.3 防御层：clean() 方法

自动填充解决了"谁来填"的问题，但还有一个"谁来改"的问题：如果前端恶意传入 `creator` 和 `createTime` 字段，Spring 的数据绑定会直接设置到实体上，自动填充就会被绕过。

`BaseDO.clean()` 方法是第二道防线，在 Controller 层调用：

```java
@PostMapping("/create")
public CommonResult<Long> create(@Valid @RequestBody ProductSaveReqVO reqVO) {
    ProductDO product = BeanUtils.toBean(reqVO, ProductDO.class);
    product.clean();  // 清除审计字段，确保自动填充生效
    return success(productService.createProduct(product));
}
```

### 2.4 模式的通用化

"非破坏性自动填充"适用于任何需要"默认值但允许覆盖"的场景：

| 场景 | null 信号 | 自动填充 | 覆盖场景 |
|------|----------|---------|---------|
| 审计字段 | 未传入 | 当前用户/时间 | 数据迁移 |
| 订单号 | 未指定 | 系统生成 | 导入旧订单 |
| 多租户 ID | 未传入 | 当前租户 | 跨租户数据操作 |
| 数据来源 | 未指定 | "API" | 批量导入标记为 "IMPORT" |

实现要点：
1. 填充逻辑在 `MetaObjectHandler`（或类似拦截器）中，不在业务代码里
2. 判断条件是 `== null`，不是 `equals(defaultValue)`
3. 安全防御（clean）和自动填充是两层独立机制

## 3. 哨兵值 + 启动期探测模式（Sentinel Value）

### 3.1 问题：数据库切换时的隐式配置

MyBatis-Plus 的 `@TableId(type = IdType.AUTO)` 对 MySQL 是自增，对 PostgreSQL 无效（需要用 `INPUT`）。如果项目要支持多种数据库，开发者必须在配置文件中手动指定 `id-type`，而这个值取决于运行时的数据库类型——同一个配置文件不能同时适用于两种数据库。

### 3.2 模式：哨兵值 + EnvironmentPostProcessor

`IdTypeEnvironmentPostProcessor` 的设计分两步：

**第一步：配置文件使用哨兵值**

```yaml
mybatis-plus:
  global-config:
    db-config:
      id-type: NONE    # 哨兵值，表示"自动检测"
```

`NONE` 不是一个有效的数据库策略，它是一个信号："请帮我决定"。

**第二步：启动期探测并替换**

`IdTypeEnvironmentPostProcessor` 实现 Spring 的 `EnvironmentPostProcessor` 接口，在 Spring 容器刷新之前运行：

```java
public class IdTypeEnvironmentPostProcessor implements EnvironmentPostProcessor {
    @Override
    public void postProcessEnvironment(ConfigurableEnvironment environment, SpringApplication application) {
        // 读取 JDBC URL
        String jdbcUrl = environment.getProperty("spring.datasource.dynamic.datasource.master.url");
        // 根据 URL 判断数据库类型
        boolean isMySQL = jdbcUrl.contains("mysql");
        // 替换哨兵值
        Properties props = new Properties();
        props.put("mybatis-plus.global-config.db-config.id-type", isMySQL ? "AUTO" : "INPUT");
        environment.getPropertySources().addLast(new PropertiesPropertySource("id-type-fix", props));
    }
}
```

### 3.3 为什么不用运行时检测

另一种方案是在运行时通过 `JdbcUtils.getDbType()` 获取数据库类型，然后在代码中动态选择 ID 策略。这两种方案的对比：

| 维度 | 启动期探测 | 运行时检测 |
|------|----------|----------|
| 性能 | 零开销，一次性 | 每次调用都有开销（开连接读元数据） |
| 复杂度 | 简单，标准 Spring 扩展点 | 需要缓存机制，要考虑连接池 |
| 灵活性 | 不支持运行时切换数据源 | 支持 |
| 适用场景 | 部署时确定数据库类型 | 同一应用连接多种数据库 |

yudao-cloud 选择启动期探测，因为实际部署中一个应用不会在运行时切换数据库类型。

### 3.4 模式的通用化

哨兵值模式适用于任何"配置值取决于运行环境，但在部署时可以确定"的场景：

```
配置文件: feature.mode = AUTO  ← 哨兵值
启动期:   探测环境 → 替换为 FEATURE_A 或 FEATURE_B
运行时:   使用确定的值，无需再判断
```

典型应用：
- 连接池参数：根据可用内存自动设置 `maxPoolSize`
- 缓存策略：根据是否有 Redis 服务决定 `local` 还是 `redis`
- 文件存储：根据部署环境选择 `local`、`oss`、`s3`

### 3.5 一个设计争议

`IdTypeEnvironmentPostProcessor` 同时还配置了 Quartz 的 `driverDelegateClass`——一个处理器做了两件不相关的事。这违反了单一职责原则。

为什么这样做？因为两者都需要读取 JDBC URL 判断数据库类型，提取公共逻辑到一个处理器比创建两个处理器更简单。这是"简洁"对"职责分离"的权衡。如果你的项目规模更大、这两个配置的生命周期不同，应该拆分为两个独立的 Processor。

## 4. 渐进式继承链模式

### 4.1 问题：单一扩展点不够用

MyBatis-Plus 的 `BaseMapper` 提供了标准 CRUD。但项目需要：
1. 连表查询能力（来自 mybatis-plus-join）
2. 分页查询、批量操作、快捷查询等增强方法

如果只有一层继承 `BaseMapperX extends BaseMapper`，那连表能力要么自己实现（重复造轮子），要么放弃。

### 4.2 模式：分层继承，每层一个职责

```
BaseMapper<T>                    ← 第一层：标准 CRUD（框架提供）
  └── MPJBaseMapper<T>           ← 第二层：连表查询（第三方库提供）
        └── BaseMapperX<T>       ← 第三层：项目增强（自己实现）
```

每层只增加一个维度的能力，通过 `default` 方法实现，不需要任何实现类。

这和 `LambdaQueryWrapperX` 的继承模式类似，但有一个关键区别：`LambdaQueryWrapperX` 需要覆盖父类方法来修复返回类型（协变返回），而 `BaseMapperX` 只需要新增方法，不覆盖已有方法。

### 4.3 用 default 方法替代实现类

`BaseMapperX` 中所有方法都是 `default` 方法：

```java
public interface BaseMapperX<T> extends MPJBaseMapper<T> {

    default PageResult<T> selectPage(PageParam pageParam, Wrapper<T> queryWrapper) {
        // 分页逻辑
        IPage<T> page = MyBatisUtils.buildPage(pageParam);
        page = selectPage(page, queryWrapper);
        return new PageResult<>(page.getRecords(), page.getTotal());
    }
}
```

好处：
- 所有 Mapper 自动获得增强能力，不需要写实现代码
- 增强逻辑集中在一处，修改时不需要改 50 个 Mapper
- 如果某个 Mapper 需要特殊行为，直接覆盖 `default` 方法

### 4.4 适用场景与风险

这个模式适用于"基础能力 + 可选增强"的分层场景。关键约束是继承链不能太深——三层已经是合理的上限。更深的继承链会导致：
- 方法来源难以追踪（这个方法是在哪一层定义的？）
- 覆盖行为复杂化（覆盖父类方法可能影响更上层的行为）
- 测试困难（需要理解整个继承链才能 mock）

---

## 小结

| 模式 | 核心思想 | 适用场景 |
|------|---------|---------|
| 条件链式扩展 | 继承 + 方法内守卫 + 协变返回 | 链式 API 需要增加条件性方法 |
| 非破坏性自动填充 | null 作为"请填充"信号 | 默认值可覆盖的场景 |
| 哨兵值 + 启动期探测 | 配置用特殊值标记，启动时替换 | 配置值取决于部署环境 |
| 渐进式继承链 | 每层增加一个职责维度 | 基础能力 + 可选增强的分层 |
