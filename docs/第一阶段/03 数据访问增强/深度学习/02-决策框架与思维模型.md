# 决策框架与思维模型

---

本章提炼数据访问增强组件中的五个关键设计决策。每个决策都给出了"为什么选 A 而不是 B"的推理过程，形成可复用的判断框架。

---

## 1. 软删除 vs 物理删除

### 1.1 决策矩阵

| 维度 | 软删除（@TableLogic） | 物理删除 |
|------|---------------------|---------|
| 数据恢复 | 直接改 `deleted=0` 即可 | 需要从备份恢复，代价高 |
| 审计追踪 | 保留完整记录 | 删除后无痕迹 |
| 存储空间 | 永久占用（随数据增长） | 立即释放 |
| 查询性能 | 所有查询自动带 `WHERE deleted=0`，索引需要包含此列 | 无额外开销 |
| 唯一约束 | 冲突——已删除的记录仍占唯一索引 | 无冲突 |
| 外键关联 | 删除后关联查询返回空，需要业务处理 | 级联删除或数据库报错 |

### 1.2 yudao-cloud 的选择与权衡

默认使用软删除，通过 `BaseDO.deleted` 字段和 `@TableLogic` 注解实现。但这是一个"默认值可覆盖"的设计——如果某个表不需要软删除，可以在 Mapper 中覆盖注解或使用原生 SQL。

**唯一约束冲突的处理**：软删除最大的技术难题是唯一约束。比如"用户名唯一"，删除用户 admin 后，无法再创建同名用户（因为 `deleted=1` 的记录仍在索引中）。

常见解决方案：
1. **组合唯一索引**：`(username, deleted)` —— 但 `deleted` 是 Boolean，只有 0 和 1，第二次删除就会冲突
2. **deleted 改为删除时间戳**：`(username, deleted_at)` —— 每次删除时间戳不同，唯一索引自然不冲突
3. **应用层校验**：创建前先查是否存在已删除的同名记录，存在则先物理删除或改名

yudao-cloud 用的是方案 3（应用层校验），因为实现最简单。如果数据量大、删除频繁，方案 2（时间戳）更健壮。

### 1.3 决策框架

```
需要软删除吗？
├── 数据是否需要恢复？ → 是 → 软删除
├── 是否有审计合规要求？ → 是 → 软删除
├── 是否有唯一约束冲突？ → 是 → 物理删除（或时间戳软删除）
├── 数据量是否巨大？ → 是 → 物理删除（或定期归档）
└── 以上都不是 → 默认软删除（更安全）
```

## 2. selectOne vs selectFirstOne vs selectOneForUpdate

### 2.1 三种场景，三种方法

这三个方法看起来都是"查一条记录"，但解决的是完全不同的并发问题：

| 方法 | 行为 | 适用场景 | 并发安全性 |
|------|------|---------|----------|
| `selectOne` | 查到多条则抛异常 | 唯一约束保证只有一条 | 如果唯一约束存在则安全 |
| `selectFirstOne` | 查到多条返回第一条 | 并发插入可能导致重复 | 不安全——可能返回错误记录 |
| `selectOneForUpdate` | 加行锁，其他事务等待 | 需要读-改-写的事务场景 | 安全——但性能开销大 |

### 2.2 selectFirstOne 的存在理由

一个自然的问题：如果 `selectOne` 已经能处理唯一记录，为什么还需要 `selectFirstOne`？

答案是**并发插入的竞态条件**。考虑这个场景：

```
时刻 T1: 事务 A 插入订单号 ORD001（未提交）
时刻 T2: 事务 B 插入订单号 ORD001（未提交，因为唯一约束要等 A 提交后才检查）
时刻 T3: 事务 A 提交，唯一索引生效
时刻 T4: 事务 B 提交，违反唯一约束 → 但有些数据库配置下不报错而是允许
时刻 T5: 查询订单号 ORD001 → 返回两条记录 → selectOne 抛异常
```

`selectFirstOne` 的设计意图是：在"理论上有唯一约束但实际可能因为并发出现多条"的情况下，选择容忍而不是崩溃。

### 2.3 决策框架

```
查询单条记录时：
├── 唯一约束是否存在且可信？ → selectOne
├── 可能存在并发插入导致的重复？ → selectFirstOne
├── 需要读取后立即修改（先查后改）？ → selectOneForUpdate
└── 不确定？ → 默认 selectOne（fail-fast 更容易发现问题）
```

### 2.4 selectOneForUpdate 的使用边界

`selectOneForUpdate` 加行锁，意味着其他事务对该行的读写都会阻塞。使用它需要满足三个条件：

1. **必须在事务内**：`@Transactional` 注解或编程式事务
2. **锁粒度要小**：查询条件必须命中索引，否则退化为表锁
3. **持有时间要短**：锁内只做必要的读-改-写，不要调用外部服务

典型使用场景：扣减库存、余额转账、订单状态流转。

## 3. betweenIfPresent：便利性 vs 严格性

### 3.1 设计选择

`betweenIfPresent` 在只提供一个边界时，自动降级为 `>=` 或 `<=`：

```java
// 两个边界都有 → BETWEEN
wrapper.betweenIfPresent(ProductDO::getPrice, 10, 100);
// SQL: price BETWEEN 10 AND 100

// 只有下界 → >=
wrapper.betweenIfPresent(ProductDO::getPrice, 10, null);
// SQL: price >= 10

// 只有上界 → <=
wrapper.betweenIfPresent(ProductDO::getPrice, null, 100);
// SQL: price <= 100

// 都没有 → 不加条件
wrapper.betweenIfPresent(ProductDO::getPrice, null, null);
// SQL: (无变化)
```

### 3.2 替代方案

| 方案 | 行为 | 优点 | 缺点 |
|------|------|------|------|
| A: 自动降级（当前选择） | 单边 → `>=` 或 `<=` | 使用简单，不需要写 if-else | 语义不精确——"价格区间"和"最低价格"是不同的业务含义 |
| B: 严格模式 | 必须两个都有或两个都没有 | 语义精确 | 使用麻烦，前端日期选择器经常只选一个 |
| C: 提供两个独立方法 | `betweenStrict` + `geOrLe` | 最灵活 | API 表面积增大 |

### 3.3 决策框架

```
区间查询的空值处理：
├── 业务上"单边有值"是否有意义？
│   ├── 是（如日期范围，用户可能只选开始日期）→ 自动降级
│   └── 否（如经纬度，必须成对）→ 严格模式
├── 前端控件是否容易产生单边空值？ → 是 → 自动降级
└── 安全敏感？（如金额范围，单边可能查出过多数据）→ 严格模式
```

## 4. String vs Long 用于审计字段

### 4.1 反直觉的选择

`BaseDO` 中 `creator` 和 `updater` 是 `String` 类型，而不是 `Long`。当前系统的用户 ID 是 Long 型自增主键，用 String 似乎多此一举。

源码注释解释了原因：

> 当前使用 SysUser 的 id，使用 String 类型，是因为未来可能存在非数字的 id，留有拓展性

### 4.2 决策推理

这是一个"为未来可能性买单"的决策。评估方式：

| 假设 | 用 Long | 用 String |
|------|---------|----------|
| 用户 ID 永远是数字 | 效率更高，类型安全 | 可以工作，但浪费一点存储 |
| 未来引入 UUID 或第三方 ID | 需要改字段类型（数据库迁移） | 不需要改 |
| 需要记录特殊主体（如 "SYSTEM"） | 不行，Long 无法表示 | 可以 |

yudao-cloud 已经有 "SYSTEM" 这样的特殊主体需求（定时任务没有登录用户），所以 String 不仅仅是"为未来准备"，而是当下就需要。

### 4.3 决策框架

```
ID 字段类型选择：
├── 是否只用于人类用户？ → Long
├── 是否需要表示非人类主体（系统、外部服务、机器人）？ → String
├── 是否可能对接外部系统（UUID、字符串格式 ID）？ → String
├── 是否有性能敏感的批量查询（亿级关联查询）？ → Long（索引更紧凑）
└── 不确定？ → String（扩展性更好，代价可控）
```

## 5. 启动期检测 vs 运行时检测

### 5.1 问题

数据库方言检测（如 `FIND_IN_SET` 的替代方案选择）可以在两个时机进行：

- **启动期**：`IdTypeEnvironmentPostProcessor` 在 Spring 启动时读取 JDBC URL，确定数据库类型
- **运行时**：`JdbcUtils.getDbType()` 在每次调用时获取连接元数据

### 5.2 对比

| 维度 | 启动期检测 | 运行时检测 |
|------|----------|----------|
| 实现复杂度 | 低（标准 Spring 扩展点） | 中（需要缓存，处理连接池） |
| 性能 | 零开销 | 需要开连接或维护缓存 |
| 混合数据库支持 | 不支持 | 支持 |
| 错误发现时机 | 启动时报错（fail-fast） | 运行时报错 |
| 配置变更响应 | 需要重启 | 自动适应 |

### 5.3 yudao-cloud 的混合策略

项目中两种方式都在用，但用于不同场景：

- **启动期**：`IdTypeEnvironmentPostProcessor` 用于 ID 策略——这个值在整个应用生命周期中不会变
- **运行时**：`JdbcUtils.getDbType()` 用于 `findInSet()` 等 SQL 构建——因为多数据源场景下，不同数据源可能连不同数据库

这个区分揭示了一个判断原则：**如果配置值在运行期间不会变，用启动期检测；如果可能因调用上下文不同而不同，用运行时检测**。

### 5.4 决策框架

```
检测时机选择：
├── 检测结果在整个运行期间是否恒定？
│   ├── 是 → 启动期检测（fail-fast + 零开销）
│   └── 否 → 运行时检测
├── 是否需要支持多种环境？（如同一应用连 MySQL 和 PostgreSQL）
│   ├── 是 → 运行时检测
│   └── 否 → 启动期检测
└── 检测失败是否应阻止启动？ → 是 → 启动期检测
```

## 小结

五个决策的共同特征：**没有绝对正确的答案，只有在特定约束下的最优选择**。每个决策都可以用一个简单的判断框架来指导：

| 决策 | 核心判断维度 |
|------|------------|
| 软删除 vs 物理删除 | 数据恢复需求 vs 唯一约束冲突 |
| selectOne 系列 | 并发场景的确定性程度 |
| betweenIfPresent | "单边有值"是否有业务意义 |
| String vs Long | 是否需要表示非人类主体 |
| 检测时机 | 运行期间是否恒定 |
