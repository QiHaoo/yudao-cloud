# 02 - 核心设计：流程引擎集成

## 流程定义：从需求到可执行模型

### BPMN 2.0 基础

BPMN（Business Process Model and Notation）是 OMG 组织制定的流程建模标准。它用一套图形符号描述业务流程，让业务人员和技术人员用同一种语言沟通。

一个典型的请假审批流程的 BPMN 模型：

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  开始事件  │───▶│  用户任务  │───▶│ 排他网关   │───▶│  用户任务  │───▶│  结束事件  │
│  (Start)  │    │ 主管审批  │    │ (XOR)    │    │ 人事备案  │    │  (End)   │
└──────────┘    └──────────┘    └─────┬────┘    └──────────┘    └──────────┘
                                      │
                                      │ 驳回
                                      ▼
                                ┌──────────┐
                                │  用户任务  │
                                │ 修改申请   │
                                └──────────┘
```

### BPMN 核心元素

| 元素 | 图形 | 含义 | Flowable API |
|------|------|------|-------------|
| 开始事件 | ○ | 流程起点 | `startEvent()` |
| 结束事件 | ◎ | 流程终点 | `endEvent()` |
| 用户任务 | □ | 需要人工处理 | `userTask()` |
| 服务任务 | □(齿轮) | 自动执行代码 | `serviceTask()` |
| 排他网关 | ◇ | 条件分支（只走一条） | `exclusiveGateway()` |
| 并行网关 | ◇(+) | 同时走所有分支 | `parallelGateway()` |
| 包含网关 | ◇(○) | 满足条件的分支都走 | `inclusiveGateway()` |

### Flowable Model vs ProcessDefinition

这是两个容易混淆的概念：

```
设计时                              运行时
┌──────────────────┐    部署    ┌──────────────────┐
│  Model（模型）     │──────────▶│  ProcessDefinition│
│                  │           │  （流程定义）       │
│  - 可以反复编辑    │           │  - 只读，不可修改   │
│  - 存储 BPMN XML  │           │  - 有版本号        │
│  - 可以有多个版本  │           │  - 每次部署+1      │
└──────────────────┘           └──────────────────┘
```

类比：Model 是"设计图纸"，ProcessDefinition 是"施工蓝图"。你可以反复修改图纸，但一旦交付施工（部署），施工蓝图就定稿了。

yudao-cloud 通过 `BpmProcessDefinitionInfoDO` 扩展了 Flowable 的 ProcessDefinition：

```java
// 来源：yudao-module-bpm-server/.../definition/BpmProcessDefinitionInfoDO.java
@TableName("bpm_process_definition_info")
public class BpmProcessDefinitionInfoDO extends BaseDO {

    private String processDefinitionId; // 关联 Flowable 的 ProcessDefinition ID
    private String modelId;             // 关联 Flowable 的 Model ID
    private Integer modelType;          // 模型类型：BPMN / 简易模式
    private String category;            // 流程分类编码

    // 表单配置
    private Integer formType;           // 表单类型：内置动态表单 / 业务表单
    private Long formId;                // 关联 BpmFormDO
    private String formConf;            // 表单配置（冗余）
    private List<String> formFields;    // 表单项（冗余）
    private String formCustomCreatePath; // 自定义表单的提交路径
    private String formCustomViewPath;   // 自定义表单的查看路径

    // 权限控制
    private List<Long> startUserIds;    // 可发起人（为空则所有人可发起）
    private List<Long> startDeptIds;    // 可发起部门
    private List<Long> managerUserIds;  // 可管理人

    // 流程行为
    private Boolean allowCancelRunningProcess; // 允许撤销审批中的流程
    private Boolean allowWithdrawTask;         // 允许审批人撤回任务
    private Integer autoApprovalType;          // 自动审批类型
}
```

## 流程部署：从模型到可执行定义

### 部署流程

```
1. 设计流程模型（BPMN XML 或简易模式）
2. 配置表单（选择内置表单或自定义表单路径）
3. 配置权限（谁能发起、谁能管理）
4. 点击"发布"
5. 调用 Flowable 的 repositoryService.createDeployment() 部署
6. 生成 ProcessDefinition（带版本号）
7. 保存 BpmProcessDefinitionInfoDO（扩展信息）
```

### 版本管理

每次部署都会生成新版本的 ProcessDefinition。Flowable 通过 `key + version` 唯一标识一个流程定义：

```
key = "leave_approval", version = 1  ← 第一版
key = "leave_approval", version = 2  ← 修改后重新部署
key = "leave_approval", version = 3  ← 最新版
```

已发起的流程实例绑定到特定版本，不受新版本影响。新发起的流程使用最新版本。

## 用户任务：谁来审批

### 候选人与候选组

用户任务节点需要指定"谁来审批"。Flowable 提供了多种指派方式：

| 指派方式 | 说明 | 举例 |
|---------|------|------|
| assignee | 直接指定审批人 | 张三 |
| candidateUsers | 候选人列表（多人竞争） | 张三、李四、王五 |
| candidateGroups | 候选组（组内竞争） | 部门经理组 |

yudao-cloud 通过 `BpmUserGroupDO` 管理用户组：

```java
// 来源：yudao-module-bpm-server/.../definition/BpmUserGroupDO.java
@TableName("bpm_user_group")
public class BpmUserGroupDO extends BaseDO {
    private Long id;
    private String name;         // 组名
    private String description;  // 描述
    private Integer status;      // 状态
    private Set<Long> userIds;   // 成员用户编号
}
```

用户组的设计让审批配置更灵活：不需要在每个流程节点上指定具体的用户，而是指定"部门经理组"这样的角色组，人员变动时只需调整组成员。

### 表单设计

表单是审批人看到和填写的内容。yudao-cloud 支持两种表单模式：

**模式一：内置动态表单（NORMAL）**

通过 `BpmFormDO` 定义表单结构：

```java
// 来源：yudao-module-bpm-server/.../definition/BpmFormDO.java
@TableName("bpm_form")
public class BpmFormDO extends BaseDO {
    private Long id;
    private String name;           // 表单名
    private Integer status;        // 状态
    private String conf;           // 表单配置
    private List<String> fields;   // 表单项数组
    private String remark;         // 备注
}
```

`fields` 字段存储的是 [form-generator](https://github.com/JakHuang/form-generator) 生成的 JSON 配置，前端根据这个 JSON 动态渲染表单。

**模式二：业务表单（CUSTOM）**

直接跳转到业务模块的自定义页面：

```java
private String formCustomCreatePath; // 发起时的页面路径，如 "/bpm/oa/leave/create"
private String formCustomViewPath;   // 查看时的页面路径，如 "/bpm/oa/leave/view"
```

这种模式适合复杂的业务表单，比如采购申请需要动态添加商品列表、计算总价等，用动态表单难以实现。

### 任务监听器

任务生命周期中可以触发自定义逻辑：

```java
// 来源：yudao-module-bpm-server/.../definition/BpmProcessListenerDO.java
@TableName("bpm_process_listener")
public class BpmProcessListenerDO extends BaseDO {
    private String name;      // 监听器名字
    private String type;      // 监听类型：execution / task
    private String event;     // 监听事件：
                              //   execution: start, end
                              //   task: create, assignment, complete, delete, update, timeout
    private String valueType; // 值类型：class / delegateExpression / expression
    private String value;     // 值（类名或表达式）
}
```

监听器的三种值类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| class | Java 类全名 | `com.example.MyTaskListener` |
| delegateExpression | Spring Bean 名称 | `${myTaskListener}` |
| expression | SpEL 表达式 | `${myBean.doSomething(task)}` |

## 流程变量：传递业务数据

### 什么是流程变量

流程变量是绑定到流程实例上的键值对数据，在整个流程生命周期内可访问。它的核心作用：

1. **条件分支判断**：网关根据变量值决定走哪条路径
2. **表单数据存储**：用户填写的表单数据存入流程变量
3. **业务数据传递**：将业务实体 ID 传递给服务任务

### 流程变量的生命周期

```
发起流程时设置变量
  │
  ▼
用户任务节点读取变量（展示表单数据）
  │
  ▼
审批时更新变量（填写审批意见）
  │
  ▼
排他网关根据变量值选择分支
  │
  ▼
服务任务读取变量（执行业务逻辑）
  │
  ▼
流程结束，变量保存在历史表中
```

### 条件表达式

排他网关通过条件表达式决定走向：

```xml
<!-- BPMN XML 示例 -->
<exclusiveGateway id="gateway1" />
<sequenceFlow id="flow1" sourceRef="gateway1" targetRef="task_approve">
  <conditionExpression>${amount <= 50000}</conditionExpression>
</sequenceFlow>
<sequenceFlow id="flow2" sourceRef="gateway1" targetRef="task_manager">
  <conditionExpression>${amount > 50000}</conditionExpression>
</sequenceFlow>
```

yudao-cloud 还支持通过 `BpmProcessExpressionDO` 管理可复用的条件表达式：

```java
// 来源：yudao-module-bpm-server/.../definition/BpmProcessExpressionDO.java
@TableName("bpm_process_expression")
public class BpmProcessExpressionDO extends BaseDO {
    private Long id;
    private String name;        // 表达式名字
    private Integer status;     // 状态
    private String expression;  // 表达式内容
}
```

## 服务任务：自动执行的节点

服务任务（Service Task）不需要人工干预，由引擎自动执行。典型用途：

```
1. 自动审批：满足条件时自动通过（如金额小于 1000 元的报销单）
2. 调用外部系统：发送通知、更新业务状态
3. 数据校验：检查库存、验证权限
4. 定时等待：延迟一段时间后继续
```

### 自动审批

yudao-cloud 通过 `autoApprovalType` 字段支持自动审批：

```java
// BpmProcessDefinitionInfoDO 中
private Integer autoApprovalType; // 自动审批类型
```

自动审批的典型场景：
- 发起人自己就是最终审批人（如部门经理给自己请假）
- 满足特定条件的低风险操作（如金额小于阈值的采购）

## 流程实例的生命周期

```
                    ┌─────────────────────────────────────────────┐
                    │              流程实例生命周期                   │
                    └─────────────────────────────────────────────┘

    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │  未开始   │───▶│  审批中   │───▶│ 审批通过  │    │ 审批不通过│
    │ NOT_START│    │ RUNNING  │    │ APPROVE  │    │ REJECT   │
    │   (-1)   │    │   (1)    │    │   (2)    │    │   (3)    │
    └──────────┘    └────┬─────┘    └──────────┘    └──────────┘
                         │
                         │  发起人取消
                         ▼
                    ┌──────────┐
                    │  已取消   │
                    │ CANCEL   │
                    │   (4)    │
                    └──────────┘
```

状态枚举定义：

```java
// 来源：yudao-module-bpm-api/.../task/BpmProcessInstanceStatusEnum.java
public enum BpmProcessInstanceStatusEnum {
    NOT_START(-1, "未开始"),
    RUNNING(1, "审批中"),
    APPROVE(2, "审批通过"),
    REJECT(3, "审批不通过"),
    CANCEL(4, "已取消");
}
```

### 状态转换的触发

| 转换 | 触发者 | 触发条件 |
|------|--------|---------|
| NOT_START → RUNNING | 发起人 | 提交申请 |
| RUNNING → APPROVE | 最后一个审批人 | 通过 |
| RUNNING → REJECT | 任意审批人 | 拒绝 |
| RUNNING → CANCEL | 发起人/管理员 | 取消流程 |

## 历史记录：流程执行轨迹

Flowable 的 `HistoryService` 记录了流程执行的完整轨迹：

```
流程实例 #12345 的执行轨迹：
──────────────────────────────────────────────────
时间            操作人    节点          动作
──────────────────────────────────────────────────
2024-01-15 09:00  张三    开始事件      发起流程
2024-01-15 09:00  张三    填写申请      提交表单
2024-01-15 10:30  李四    主管审批      通过 → "同意请假"
2024-01-15 14:00  王五    人事备案      通过 → "已备案"
2024-01-15 14:00  系统    结束事件      流程完成
──────────────────────────────────────────────────
```

每次审批操作都会记录：
- 谁操作的（userId）
- 在哪个节点（taskName）
- 做了什么决定（approve/reject）
- 审批意见（comment）
- 操作时间

## 抄送机制

`BpmProcessInstanceCopyDO` 实现了流程抄送功能：

```
审批流程：
  发起人 → 主管审批 → 人事备案 → 结束

抄送：
  发起时抄送给 → HR 助理（知会）
  主管通过时抄送给 → 部门总监（知会）
```

抄送只通知不参与审批，让相关人员了解流程进展。

## OA 请假：一个完整的集成示例

yudao-cloud 内置了一个 OA 请假的示例，展示工作流与业务模块的集成方式：

### 数据模型

```java
// 来源：yudao-module-bpm-server/.../oa/BpmOALeaveDO.java
@TableName("bpm_oa_leave")
public class BpmOALeaveDO extends BaseDO {
    private Long id;                // 请假表单主键
    private Long userId;            // 申请人
    private Integer type;           // 请假类型
    private String reason;          // 请假原因
    private LocalDateTime startTime; // 开始时间
    private LocalDateTime endTime;   // 结束时间
    private Long day;               // 请假天数
    private Integer status;         // 审批结果
    private String processInstanceId; // 关联的流程实例 ID
}
```

### 集成流程

```
1. 员工在前端填写请假表单
2. 前端调用 BpmOALeaveController.createBpmOALeave()
3. 后端：
   a. 保存 BpmOALeaveDO 到数据库
   b. 调用 BpmProcessInstanceApi.createProcessInstanceByKey() 发起流程
   c. 将 BpmOALeaveDO.id 作为流程变量传递
4. 流程引擎：
   a. 创建流程实例
   b. 生成第一个用户任务（主管审批）
   c. 通知主管有待办任务
5. 主管审批：
   a. 查看待办任务，加载请假表单数据
   b. 点击"通过"或"拒绝"
   c. 流程引擎更新流程实例状态
   d. 如果是最后一个节点，更新 BpmOALeaveDO.status
```

### 关键设计：流程与业务的双向关联

```
BpmOALeaveDO                    Flowable
┌──────────────────┐           ┌──────────────────┐
│ id: 100          │           │ ProcessInstance  │
│ userId: 张三      │           │ id: "12345"      │
│ type: 年假        │◀────────▶│ businessKey: 100 │
│ processInstanceId │           │ processVariables │
│   : "12345"      │           │   {leaveId: 100} │
└──────────────────┘           └──────────────────┘
```

通过 `processInstanceId` 和 `businessKey` 实现双向查询：
- 从业务数据查流程状态：`leaveDO.getProcessInstanceId()` → 查询 Flowable
- 从流程查业务数据：`processInstance.getBusinessKey()` → 查询 BpmOALeaveDO

## 流程表达式：动态审批人

`BpmProcessExpressionDO` 支持用表达式动态确定审批人：

```java
// 来源：yudao-module-bpm-server/.../definition/BpmProcessExpressionDO.java
@TableName("bpm_process_expression")
public class BpmProcessExpressionDO extends BaseDO {
    private Long id;
    private String name;       // 表达式名字
    private Integer status;    // 状态
    private String expression; // 表达式
}
```

表达式示例：
- `${bpmUserService.getDeptManager(startUserId)}`：获取发起人的部门经理
- `${bpmUserService.getUserRoleUsers('finance_approver')}`：获取财务审批角色的所有人

## 设计模式提炼

### 1. 策略模式

表单类型（NORMAL/CUSTOM）对应不同的表单渲染策略；监听器值类型（class/delegateExpression/expression）对应不同的执行策略。

### 2. 模板方法模式

Flowable 引擎定义了流程执行的骨架（开始 → 用户任务 → 网关 → 结束），具体节点行为由用户配置的监听器、条件表达式填充。

### 3. 观察者模式

流程监听器（BpmProcessListenerDO）本质上是观察者：在任务创建、完成、删除等事件发生时，通知注册的监听器执行逻辑。

### 4. 桥接模式

`processInstanceId` 桥接了业务数据（BpmOALeaveDO）和流程引擎（Flowable ProcessInstance），让两者可以独立演化。
