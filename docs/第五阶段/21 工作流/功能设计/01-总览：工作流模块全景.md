# 01 - 总览：工作流模块全景

## 场景引入

一家有 200 人的公司，行政部门每天要处理大量的审批请求：

- **请假审批**：员工提交请假申请 → 直属主管审批 → 人事备案
- **采购审批**：部门提交采购申请 → 部门经理审批 → 金额超过 5 万需要总经理审批 → 财务确认
- **报销审批**：员工提交报销单 → 直属主管审批 → 财务审核 → 出纳打款

这些流程有几个共同特点：

1. **有明确的步骤**：每一步由谁审批、做什么操作，都是预先定义好的
2. **有条件分支**：金额超过某个阈值需要更高级别的审批
3. **需要记录轨迹**：谁在什么时间做了什么决定，需要完整留痕
4. **与业务数据关联**：请假单、采购单、报销单的数据需要和审批流程绑定

手动管理这些流程效率低下、容易出错。**工作流引擎**就是为了解决这个问题而生的——它将流程的定义和执行自动化，让开发者专注于业务逻辑，而非流程控制。

## 为什么选择 Flowable

Java 生态中有多个工作流引擎可选：

| 引擎 | 特点 | 适用场景 |
|------|------|---------|
| Activiti | 老牌引擎，社区活跃 | 传统 OA 系统 |
| Flowable | Activiti 的分支，性能更好，API 更现代 | 企业级应用首选 |
| Camunda | 功能最全，支持 DMN/CMMN | 复杂业务流程 |
| jBPM | Red Hat 支持，与 KIE 集成 | 与规则引擎配合 |

yudao-cloud 选择 **Flowable** 的原因：

1. **与 Activiti 兼容**：Flowable 是从 Activiti 6.x 分支出来的，API 高度兼容，迁移成本低
2. **Spring Boot 集成好**：官方提供 `flowable-spring-boot-starter`，开箱即用
3. **轻量级**：相比 Camunda，Flowable 更轻量，适合中小规模项目
4. **BPMN 2.0 标准**：支持标准的流程建模语言，流程定义可移植

## 核心能力

### 1. 流程建模

流程建模是定义"审批流程长什么样"的过程。支持两种方式：

- **BPMN 可视化设计器**：通过拖拽方式设计流程图，生成标准的 BPMN 2.0 XML
- **仿钉钉简易设计器**：用简化的树形结构定义流程，降低使用门槛

建模阶段的核心实体：

```
BpmCategoryDO（流程分类）
├── name: "OA 审批"、"财务审批"、"人事审批"
└── code: 分类标识

BpmFormDO（表单定义）
├── name: "请假申请表"
├── conf: 表单配置（JSON）
└── fields: 表单项数组（基于 form-generator）

BpmProcessDefinitionInfoDO（流程定义扩展信息）
├── processDefinitionId: 关联 Flowable 的 ProcessDefinition
├── modelId: 关联 Flowable 的 Model
├── modelType: 模型类型（BPMN / 简易模式）
├── formType: 表单类型（内置动态表单 / 业务表单）
├── startUserIds: 可发起人（为空则所有人可发起）
└── startDeptIds: 可发起部门
```

### 2. 流程部署

流程模型设计完成后，需要**部署**才能使用。部署的过程是将 BPMN XML 交给 Flowable 引擎解析，生成可执行的流程定义。

```
设计阶段                          运行阶段
┌──────────┐    部署    ┌──────────┐    发起    ┌──────────┐
│ 流程模型  │──────────▶│ 流程定义  │──────────▶│ 流程实例  │
│ (Model)  │           │ (Process │           │ (Process │
│          │           │ Definition)│          │ Instance)│
└──────────┘           └──────────┘           └──────────┘
  可编辑、可版本管理      只读、有版本号          正在运行的流程
```

一个模型可以多次部署，每次部署生成新版本的流程定义。已发起的流程实例不受新版本影响。

### 3. 流程发起

用户选择一个流程定义，填写表单数据，提交后创建流程实例。发起时需要传递：

- **表单数据**：用户的申请内容（如请假天数、采购金额）
- **流程变量**：用于条件分支的判断依据

### 4. 任务审批

流程运行过程中，会在各个**用户任务（User Task）**节点生成待办任务。审批人可以：

- **通过（Approve）**：任务完成，流程流转到下一个节点
- **拒绝（Reject）**：任务驳回，根据配置退回到发起人或上一个节点
- **转办（Delegate）**：将任务转给其他人处理
- **加签（Copy）**：抄送给其他人知会

### 5. 流程监控

管理员可以查看所有流程实例的状态：

- 运行中的流程有哪些、卡在哪个节点
- 已完成的流程审批轨迹
- 已取消的流程取消原因

## 与业务模块的集成

工作流模块最大的价值在于**与业务模块解耦集成**。以请假审批为例：

```
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│   员工提交     │         │   工作流引擎   │         │   OA 请假表   │
│   请假申请     │────────▶│   发起流程     │────────▶│   保存数据    │
│              │         │              │         │              │
│  BpmOALeave  │         │  Flowable    │         │  bpm_oa_leave│
│  Controller  │         │  Engine      │         │  表           │
└──────────────┘         └──────────────┘         └──────────────┘
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
              ┌──────────┐ ┌──────────┐ ┌──────────┐
              │ 主管审批   │ │ 人事审批   │ │ 审批完成   │
              │          │ │          │ │          │
              │ 通知员工  │ │ 更新状态   │ │ 更新状态   │
              └──────────┘ └──────────┘ └──────────┘
```

集成的关键在于 `BpmOALeaveDO`：

```java
// 来源：yudao-module-bpm-server/.../oa/BpmOALeaveDO.java
private Long id;               // 请假表单主键
private Long userId;           // 申请人
private Integer type;          // 请假类型
private String reason;         // 请假原因
private LocalDateTime startTime; // 开始时间
private LocalDateTime endTime;   // 结束时间
private Long day;              // 请假天数
private Integer status;        // 审批结果（复用 BpmProcessInstanceStatusEnum）
private String processInstanceId; // 关联的流程实例 ID
```

`processInstanceId` 是连接业务数据和流程引擎的桥梁：通过它可以查询流程的当前状态、审批历史等。

## Flowable 引擎的核心表

Flowable 使用 `ACT_` 前缀的表存储流程运行时数据，yudao-cloud 在此基础上创建了 `bpm_` 前缀的扩展表：

| 表前缀 | 说明 | 举例 |
|--------|------|------|
| `ACT_RE_*` | Repository（流程定义） | `ACT_RE_PROCDEF`（流程定义）、`ACT_RE_MODEL`（模型） |
| `ACT_RU_*` | Runtime（运行时） | `ACT_RU_TASK`（待办任务）、`ACT_RU_EXECUTION`（执行实例） |
| `ACT_HI_*` | History（历史） | `ACT_HI_PROCINST`（历史流程实例）、`ACT_HI_TASKINST`（历史任务） |
| `ACT_GE_*` | General（通用） | `ACT_GE_PROPERTY`（引擎属性） |
| `bpm_*` | 扩展信息 | `bpm_process_definition_info`（流程定义扩展）、`bpm_form`（表单） |

为什么需要扩展表？因为 Flowable 的 `ProcessDefinition` 不支持自定义字段（如图标、描述、可发起人等），所以 `BpmProcessDefinitionInfoDO` 用相同 ID 关联，存储这些扩展信息。

## 设计决策：为什么不用 Activiti？

| 维度 | Flowable | Activiti |
|------|----------|----------|
| 性能 | 更好，针对高并发优化 | 一般 |
| Spring Boot 集成 | 官方 starter，配置简单 | 需要额外配置 |
| BPMN 2.0 支持 | 完整 | 完整 |
| 社区活跃度 | 活跃，版本迭代快 | 相对放缓 |
| API 设计 | 更现代，更简洁 | 老派 |
| 学习资料 | 丰富（中文社区尤其活跃） | 丰富 |

## 设计决策：为什么用扩展表而非修改 Flowable 源码？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 修改 Flowable 源码 | 数据完全统一 | 升级困难，维护成本高 |
| 扩展表（当前方案） | 与 Flowable 解耦，升级无影响 | 需要额外维护关联关系 |
| 自定义引擎 | 完全可控 | 工作量巨大，不现实 |

扩展表方案是业界通行做法：在不侵入引擎的前提下，通过 ID 关联扩展信息。

## 思考题

1. 如果需要支持"会签"（多个审批人全部通过才算通过）和"或签"（任意一人通过即可），在 BPMN 中如何建模？
2. 流程变量和表单数据是什么关系？为什么需要分开存储？
3. 如果一个审批流程需要调用外部系统（如查询库存、检查预算），应该用什么类型的节点？
4. 如何处理"审批人离职"的情况？候选人都不在了，流程怎么继续？
