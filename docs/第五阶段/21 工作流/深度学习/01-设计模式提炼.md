# 工作流模块设计模式提炼

## 1. 外观模式：对 Flowable API 的封装简化

### 1.1 模式定义

外观模式（Facade Pattern）为子系统中的一组接口提供一个一致的界面，定义了一个高层接口，使得子系统更加容易使用。外观模式通过简化复杂系统的接口，降低了客户端与子系统之间的耦合度。

### 1.2 项目应用

Flowable 是一个功能强大的工作流引擎，其 API 众多且复杂。项目通过 Service 层对 Flowable API 进行封装，提供简洁易用的接口给业务层调用。

**封装层次：**

```
┌─────────────────────────────────────────────────────┐
│                   业务层（Controller）                │
│         调用 BpmModelService、BpmProcessInstanceService │
└───────────────────────────┬─────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────┐
│                   Service 层（外观）                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ BpmModel    │  │ BpmProcess  │  │ BpmTask     │  │
│  │ Service     │  │ Definition  │  │ Service     │  │
│  │             │  │ Service     │  │             │  │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘  │
│         │               │               │          │
│         ▼               ▼               ▼          │
│  ┌─────────────────────────────────────────────┐   │
│  │        Flowable API 封装层                   │   │
│  └───────────────────┬─────────────────────────┘   │
└──────────────────────┼─────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│                  Flowable 引擎                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ Repository  │  │ Runtime     │  │ Task        │  │
│  │ Service     │  │ Service     │  │ Service     │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
└─────────────────────────────────────────────────────┘
```

### 1.5 可迁移经验

**适用场景：**
- 集成复杂的第三方框架或 SDK
- 需要简化子系统的接口
- 需要在子系统和客户端之间建立解耦层

**设计要点：**
1. **接口简化**：将复杂的 API 调用封装为简洁的业务方法
2. **异常转换**：将框架异常转换为业务异常
3. **参数转换**：将业务 DTO 转换为框架所需的参数
4. **结果转换**：将框架返回结果转换为业务 DTO

**反模式警示：**
- 避免外观类过于庞大，应按业务领域拆分
- 避免过度封装，保留必要的灵活性

## 2. 观察者模式：流程事件监听

### 2.1 模式定义

观察者模式（Observer Pattern）定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并自动更新。观察者模式实现了对象之间的松耦合。

### 2.2 项目应用

工作流引擎在流程执行过程中会产生各种事件（如任务创建、任务完成、流程启动、流程结束等）。项目通过监听器机制，在这些事件发生时执行相应的业务逻辑。

**事件监听架构：**

```
┌─────────────────────────────────────────────────────┐
│                  Flowable 引擎                       │
│                                                      │
│  事件产生：                                           │
│  ├── 任务创建事件                                     │
│  ├── 任务完成事件                                     │
│  ├── 流程启动事件                                     │
│  └── 流程结束事件                                     │
└───────────────────────┬─────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│                   事件监听器                          │
│  ┌─────────────────────────────────────────────┐    │
│  │  BpmUserTaskListener                        │    │
│  │  - 监听任务创建/完成事件                      │    │
│  │  - 执行 HTTP 回调通知                        │    │
│  └─────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────┐    │
│  │  BpmProcessInstanceStatusEventListener      │    │
│  │  - 监听流程状态变更事件                      │    │
│  │  - 同步状态到业务系统                        │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│                   业务系统                           │
│  - 更新业务表单状态                                   │
│  - 发送通知消息                                       │
│  - 记录操作日志                                       │
└─────────────────────────────────────────────────────┘
```

### 2.3 代码示例

**任务监听器实现：**

```java
// 源码位置：yudao-module-bpm-server/src/main/java/cn/iocoder/yudao/module/bpm/service/task/listener/BpmUserTaskListener.java
@Component
@Slf4j
@Scope("prototype")
public class BpmUserTaskListener implements TaskListener {

    public static final String DELEGATE_EXPRESSION = "${bpmUserTaskListener}";

    @Resource
    private BpmProcessInstanceService processInstanceService;

    @Setter
    private Expression listenerConfig;

    @Override
    public void notify(DelegateTask delegateTask) {
        // 1. 获取所需基础信息
        ProcessInstance processInstance = processInstanceService
            .getProcessInstance(delegateTask.getProcessInstanceId());
        BpmSimpleModelNodeVO.ListenerHandler listenerHandler = 
            parseListenerConfig(listenerConfig);

        // 2. 发起 HTTP 请求通知业务系统
        listenerHandler.getBody().add(new BpmSimpleModelNodeVO.HttpRequestParam()
            .setKey("processInstanceId")
            .setType(BpmHttpRequestParamTypeEnum.FIXED_VALUE.getType())
            .setValue(delegateTask.getProcessInstanceId()));
        listenerHandler.getBody().add(new BpmSimpleModelNodeVO.HttpRequestParam()
            .setKey("assignee")
            .setType(BpmHttpRequestParamTypeEnum.FIXED_VALUE.getType())
            .setValue(delegateTask.getAssignee()));
        listenerHandler.getBody().add(new BpmSimpleModelNodeVO.HttpRequestParam()
            .setKey("taskDefinitionKey")
            .setType(BpmHttpRequestParamTypeEnum.FIXED_VALUE.getType())
            .setValue(delegateTask.getTaskDefinitionKey()));
        listenerHandler.getBody().add(new BpmSimpleModelNodeVO.HttpRequestParam()
            .setKey("taskId")
            .setType(BpmHttpRequestParamTypeEnum.FIXED_VALUE.getType())
            .setValue(delegateTask.getId()));

        BpmHttpRequestUtils.executeBpmHttpRequest(processInstance,
                listenerHandler.getPath(), listenerHandler.getHeader(), 
                listenerHandler.getBody(), false, null);
    }
}
```

### 2.4 设计优势

**解耦业务逻辑：**
- 工作流引擎不需要知道业务系统如何处理事件
- 业务系统不需要关心工作流引擎的实现细节

**可扩展性：**
- 可以添加多个监听器，每个监听器处理不同的事件
- 监听器可以独立开发和部署

**灵活性：**
- 监听器可以通过配置动态启用或禁用
- 支持同步和异步两种事件处理方式

### 2.5 可迁移经验

**适用场景：**
- 系统需要在某个操作完成后执行额外的逻辑
- 多个模块需要响应同一个事件
- 需要解耦事件产生者和事件消费者

**设计要点：**
1. **事件对象设计**：封装事件相关的信息，便于监听器使用
2. **监听器注册**：通过 Spring 的事件机制或自定义的监听器注册中心
3. **异步支持**：对于耗时的监听器逻辑，应支持异步执行
4. **异常隔离**：一个监听器的异常不应影响其他监听器

**进阶方案：**
- 使用 Spring Event 机制实现进程内事件
- 使用消息队列实现跨进程事件

## 3. 策略模式：不同审批节点的处理策略

### 3.1 问题场景

工作流中的审批节点有多种类型：
- **用户任务（User Task）**：指定用户审批
- **服务任务（Service Task）**：自动执行
- **网关（Gateway）**：条件分支
- **子流程（Sub Process）**：嵌套流程

每种节点类型的处理逻辑不同，需要根据节点类型选择不同的处理策略。

### 3.2 项目应用

项目通过策略模式，将不同审批节点的处理逻辑封装为独立的策略类：

**策略类结构：**

```
BpmTaskAssignStrategy                    ← 策略接口
├── BpmTaskAssignUserStrategy            ← 指定用户策略
├── BpmTaskAssignRoleStrategy            ← 指定角色策略
├── BpmTaskAssignDeptLeaderStrategy      ← 部门负责人策略
├── BpmTaskAssignStartUserStrategy       ← 发起人策略
└── BpmTaskAssignMultiStrategy           ← 多人策略（会签/或签）
```

### 3.3 代码示例

**候选人表达式：**

```java
// 源码位置：yudao-module-bpm-server/src/main/java/cn/iocoder/yudao/module/bpm/framework/flowable/core/candidate/expression/BpmTaskAssignLeaderExpression.java
@Component("bpmTaskAssignLeaderExpression")
public class BpmTaskAssignLeaderExpression implements FlowableContextAware {

    /**
     * 获得任务的部门负责人
     *
     * @param delegateTask 任务对象
     * @return 部门负责人用户编号
     */
    public Collection<String> execute(DelegateTask delegateTask) {
        // 1. 获取发起人
        String startUserId = delegateTask.getProcessInstance()
            .getStartUserId();
        // 2. 查询发起人的部门负责人
        Long deptId = getUserDeptId(Long.parseLong(startUserId));
        // 3. 返回部门负责人
        return getDeptLeaderIds(deptId);
    }
}
```

**发起人表达式：**

```java
// 源码位置：yudao-module-bpm-server/src/main/java/cn/iocoder/yudao/module/bpm/framework/flowable/core/candidate/expression/BpmTaskAssignStartUserExpression.java
@Component("bpmTaskAssignStartUserExpression")
public class BpmTaskAssignStartUserExpression {

    /**
     * 获得流程发起人
     *
     * @param delegateTask 任务对象
     * @return 发起人用户编号
     */
    public Collection<String> execute(DelegateTask delegateTask) {
        return Collections.singletonList(
            delegateTask.getProcessInstance().getStartUserId()
        );
    }
}
```

### 3.4 BPMN 中的配置

```xml

    <extensionElements>

        <flowable:taskListener 
            event="create" 
            expression="${bpmTaskAssignStartUserExpression.execute(task)}"/>
    </extensionElements>

```

### 3.5 可迁移经验

**适用场景：**
- 同一操作有多种实现方式，需要根据条件动态选择
- 需要在运行时根据配置选择处理逻辑
- 需要支持用户自定义扩展

**设计要点：**
1. **策略接口统一**：所有策略实现相同的接口
2. **Spring Bean 集成**：策略类注册为 Spring Bean，通过名称获取
3. **表达式驱动**：通过 BPMN 表达式配置使用哪个策略
4. **上下文传递**：策略类可以获取流程上下文信息

**反模式警示：**
- 避免策略类之间有依赖关系
- 避免在一个策略类中处理多种逻辑

## 4. 可迁移经验：如何在业务系统中集成工作流引擎

### 4.1 集成架构设计

```
┌─────────────────────────────────────────────────────┐
│                    业务系统                          │
│  ┌─────────────────────────────────────────────┐    │
│  │  业务表单（请假、报销、采购等）               │    │
│  └─────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────┐    │
│  │  业务 Service（调用工作流 API）              │    │
│  └─────────────────────────────────────────────┘    │
└───────────────────────┬─────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│                 工作流中间层                          │
│  ┌─────────────────────────────────────────────┐    │
│  │  BpmProcessInstanceService                  │    │
│  │  - 启动流程                                  │    │
│  │  - 查询流程状态                              │    │
│  │  - 审批任务                                  │    │
│  └─────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────┐    │
│  │  事件监听器                                  │    │
│  │  - 流程状态变更 → 更新业务表单状态           │    │
│  │  - 任务创建 → 发送通知                      │    │
│  └─────────────────────────────────────────────┘    │
└───────────────────────┬─────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│                  Flowable 引擎                       │
│  - 流程定义管理                                       │
│  - 流程实例执行                                       │
│  - 任务管理                                           │
└─────────────────────────────────────────────────────┘
```

### 4.2 关键集成点

**1. 流程定义与业务表单绑定**

```java
// 流程定义关联业务表单
public class BpmProcessDefinitionDO {
    private String id;           // 流程定义 ID
    private String name;         // 流程名称
    private Long formId;         // 关联的业务表单 ID
    private String formType;     // 表单类型（内置/外置）
}
```

**2. 流程实例与业务数据关联**

```java
// 启动流程时传递业务数据
public void startProcess(Long userId, String processDefinitionKey, 
                         Map<String, Object> variables) {
    // 1. 创建业务表单数据
    Long formId = createFormData(variables);
    // 2. 启动流程实例
    variables.put("formId", formId);
    ProcessInstance instance = runtimeService
        .startProcessInstanceByKey(processDefinitionKey, variables);
    // 3. 关联业务数据和流程实例
    bindFormAndProcess(formId, instance.getId());
}
```

**3. 审批时更新业务状态**

```java
// 审批通过后更新业务状态
public void approveTask(Long userId, Long taskId, boolean approved) {
    // 1. 完成 Flowable 任务
    taskService.complete(taskId, variables);
    // 2. 如果是最后一个节点，更新业务状态
    if (isProcessEnd(processInstanceId)) {
        updateBusinessStatus(formId, approved ? "APPROVED" : "REJECTED");
    }
}
```

### 4.4 关键设计决策

**为什么使用中间层封装？**
- 隔离 Flowable API 的复杂性
- 统一异常处理和日志记录
- 便于后续更换工作流引擎

**为什么表单和流程分离？**
- 表单是业务数据，流程是控制逻辑
- 分离后可以独立演进
- 支持多种表单类型（内置、外置）

**为什么使用事件监听器？**
- 解耦工作流引擎和业务系统
- 支持异步处理，提高性能
- 便于扩展和维护

### 4.5 反模式与陷阱

**反模式 1：业务逻辑写在流程定义中**
```java
// 错误示例：在 ServiceTask 中写业务逻辑
public class SendEmailServiceTask implements JavaDelegate {
    @Override
    public void execute(DelegateExecution execution) {
        // 复杂的业务逻辑...
    }
}
```
- 应该将业务逻辑放在业务系统中，通过事件监听器触发

**反模式 2：直接操作 Flowable 表**
```java
// 错误示例：直接更新 Flowable 的表
UPDATE ACT_RU_TASK SET ASSIGNEE = 'user1' WHERE ID_ = 'taskId';
```
- 应该通过 Flowable API 操作，避免数据不一致

**反模式 3：忽略流程版本管理**
- 流程定义更新后，正在运行的流程实例应该继续使用旧版本
- 新的流程实例使用新版本

## 总结

工作流模块通过外观模式简化了 Flowable API 的使用，通过观察者模式实现了流程事件的松耦合处理，通过策略模式支持了不同审批节点的灵活配置。这些设计模式使得工作流引擎能够优雅地集成到业务系统中。

**核心设计要点：**
1. 外观模式将复杂的 Flowable API 封装为简洁的业务接口
2. 观察者模式让业务系统能够响应流程事件，实现松耦合集成
3. 策略模式支持多种审批节点类型的灵活配置
4. 分层架构将工作流引擎与业务系统解耦，便于独立演进
