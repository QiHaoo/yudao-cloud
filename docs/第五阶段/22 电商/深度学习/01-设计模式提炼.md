# 电商模块设计模式提炼

## 1. 分库分表策略：4 个子服务的数据划分

### 1.1 模式定义

分库分表是将数据分散到多个数据库或表中的策略，目的是解决单库单表的性能瓶颈。垂直分库按业务领域划分，水平分表按数据量划分。

### 1.2 项目应用

电商模块采用垂直分库策略，将不同业务领域的数据存储在独立的数据库中：

**子服务划分：**

```
┌─────────────────────────────────────────────────────┐
│                   电商系统                            │
├─────────────┬─────────────┬─────────────┬─────────────┤
│  商品服务   │  交易服务   │  营销服务   │  统计服务   │
│  Product    │  Trade      │  Promotion  │  Statistics │
└──────┬──────┴──────┬──────┴──────┬──────┴──────┬──────┘
       │             │             │             │
       ▼             ▼             ▼             ▼
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ product  │  │ trade    │  │promotion │  │statistics│
│ 数据库   │  │ 数据库   │  │ 数据库   │  │ 数据库   │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
```

**目录结构：**

```
yudao-module-mall/
├── yudao-module-product-api/          ← 商品 API
├── yudao-module-product-server/       ← 商品服务实现
├── yudao-module-trade-api/            ← 交易 API
├── yudao-module-trade-server/         ← 交易服务实现
├── yudao-module-promotion-api/        ← 营销 API
├── yudao-module-promotion-server/     ← 营销服务实现
├── yudao-module-statistics-api/       ← 统计 API
└── yudao-module-statistics-server/    ← 统计服务实现
```

### 1.3 数据库设计

**商品服务数据库（product）：**

```sql

    id BIGINT PRIMARY KEY,
```

### 1.5 可迁移经验

**适用场景：**
- 业务领域边界清晰，数据量大
- 不同业务的读写比例差异大
- 需要独立扩展不同业务的数据库

**设计要点：**
1. **领域驱动划分**：按业务领域而非技术层次划分数据库
2. **API 模块独立**：每个子服务有独立的 API 模块，定义接口和 DTO
3. **数据隔离**：不同服务的数据完全隔离，避免跨库查询
4. **RPC 通信**：服务间通过 Feign 调用，传递 DTO 而非直接访问数据库

**反模式警示：**
- 避免跨库 JOIN 查询，应通过 RPC 调用聚合数据
- 避免循环依赖，服务间的依赖关系应该是单向的

## 2. 库存扣减的并发控制

### 2.1 问题场景

库存扣减是典型的高并发场景，多用户同时购买可能导致超卖：

```
时间线     用户A              用户B
  │      查询库存=1
  │                          查询库存=1
  │      扣减1，库存=0
  │                          扣减1，库存=-1（超卖！）
  ▼
```

### 2.2 解决方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 数据库悲观锁 | 数据一致性强 | 性能差，可能死锁 | 库存量小，并发低 |
| 数据库乐观锁 | 性能好 | 冲突时需要重试 | 冲突概率低 |
| Redis 原子操作 | 性能最好 | 需要额外维护 Redis | 高并发场景 |
| 分布式锁 | 简单直观 | 性能一般 | 中等并发 |

### 2.3 项目应用

项目采用数据库乐观锁 + SQL 原子操作的方式：

**方案一：SQL 原子扣减**

```sql
UPDATE product_sku 
SET stock = stock - 1 
WHERE id = 123 AND stock >= 1
```

**方案二：乐观锁（版本号）**

```sql
UPDATE product_sku 
SET stock = stock - 1, version = version + 1 
WHERE id = 123 AND version = 1
```

### 2.4 代码示例

**库存扣减实现：**

```java
// 伪代码示例
public void updateSkuStock(ProductSkuUpdateStockReqDTO reqDTO) {
    for (ProductSkuUpdateStockReqDTO.Item item : reqDTO.getItems()) {
        // 使用 SQL 原子操作扣减库存
        int affected = productSkuMapper.updateStock(item.getId(), item.getIncrCount());
        if (affected == 0) {
            // 库存不足，抛出异常
            throw new ServiceException(STOCK_NOT_ENOUGH);
        }
    }
}
```

**Mapper SQL：**

```xml
<update id="updateStock">
    UPDATE product_sku
    SET stock = stock + #{incrCount}
    WHERE id = #{id}
    <if test="incrCount &lt; 0">
        AND stock >= #{-incrCount}
    </if>
</update>
```

### 2.5 Redis 原子操作方案（扩展）

对于超高并发场景，可以使用 Redis 的原子操作：

```java
// 伪代码示例
public boolean deductStock(Long skuId, int count) {
    String key = "stock:" + skuId;
    // 使用 Lua 脚本保证原子性
    String script = 
        "local stock = tonumber(redis.call('GET', KEYS[1])) " +
        "if stock >= tonumber(ARGV[1]) then " +
        "    redis.call('DECRBY', KEYS[1], ARGV[1]) " +
        "    return 1 " +
        "else " +
        "    return 0 " +
        "end";
    Long result = redisTemplate.execute(
        new DefaultRedisScript<>(script, Long.class),
        Collections.singletonList(key),
        String.valueOf(count)
    );
    return result == 1;
}
```

### 2.6 可迁移经验

**适用场景：**
- 库存、余额、优惠券数量等共享资源的并发扣减
- 需要防止超卖或透支

**设计要点：**
1. **原子操作优先**：使用数据库或 Redis 的原子操作，避免先查后改
2. **边界校验**：在扣减时校验剩余数量是否足够
3. **失败重试**：乐观锁冲突时可以重试，但要设置最大重试次数
4. **异步对账**：定期对账，发现并修复数据不一致

**进阶方案：**
- 库存预热：将库存从数据库加载到 Redis
- 库存分桶：将库存拆分到多个桶，减少竞争
- 异步扣减：先扣减 Redis，异步同步到数据库

## 3. 订单状态机设计

### 3.1 状态定义

```
┌─────────────────────────────────────────────────────┐
│                    订单状态流转                       │
│                                                      │
│   ┌─────────┐      ┌─────────┐      ┌─────────┐    │
│   │ UNPAID  │ ───► │UNDELIVER│ ───► │DELIVERED│    │
│   │ (待支付) │      │ (待发货) │      │ (已发货) │    │
│   └────┬────┘      └─────────┘      └────┬────┘    │
│        │                                  │         │
│        │                                  ▼         │
│        │                            ┌─────────┐    │
│        │                            │COMPLETED│    │
│        │                            │ (已完成) │    │
│        │                            └─────────┘    │
│        │                                            │
│        ▼                                            │
│   ┌─────────┐                                       │
│   │CANCELED │                                       │
│   │ (已取消) │                                       │
│   └─────────┘                                       │
└─────────────────────────────────────────────────────┘
```

### 3.2 状态枚举

```java
// 源码位置：yudao-module-trade-api/src/main/java/cn/iocoder/yudao/module/trade/enums/order/TradeOrderStatusEnum.java
@RequiredArgsConstructor
@Getter
public enum TradeOrderStatusEnum implements ArrayValuable<Integer> {
    UNPAID(0, "待支付"),
    UNDELIVERED(10, "待发货"),
    DELIVERED(20, "已发货"),
    COMPLETED(30, "已完成"),
    CANCELED(40, "已取消");

    private final Integer status;
    private final String name;

    // 状态判断方法
    public static boolean isUnpaid(Integer status) {
        return ObjectUtil.equal(UNPAID.getStatus(), status);
    }

    public static boolean isUndelivered(Integer status) {
        return ObjectUtil.equal(UNDELIVERED.getStatus(), status);
    }

    public static boolean isDelivered(Integer status) {
        return ObjectUtil.equals(status, DELIVERED.getStatus());
    }

    public static boolean isCanceled(Integer status) {
        return ObjectUtil.equals(status, CANCELED.getStatus());
    }

    public static boolean isCompleted(Integer status) {
        return ObjectUtil.equals(status, COMPLETED.getStatus());
    }

    // 判断是否有过已付款状态
    public static boolean havePaid(Integer status) {
        return ObjectUtils.equalsAny(status, 
            UNDELIVERED.getStatus(),
            DELIVERED.getStatus(), 
            COMPLETED.getStatus());
    }

    // 判断是否有过已发货状态
    public static boolean haveDelivered(Integer status) {
        return ObjectUtils.equalsAny(status, 
            DELIVERED.getStatus(), 
            COMPLETED.getStatus());
    }
}
```

### 3.3 状态转换规则

| 当前状态 | 目标状态 | 触发条件 |
|---------|---------|---------|
| 待支付 | 待发货 | 支付成功 |
| 待支付 | 已取消 | 用户取消 / 超时未支付 |
| 待发货 | 已发货 | 商家发货 |
| 已发货 | 已完成 | 用户确认收货 / 自动确认 |
| 已完成 | - | 终态 |
| 已取消 | - | 终态 |

### 3.4 代码示例

**订单状态校验：**

```java
// 伪代码示例
public void payOrder(Long orderId) {
    TradeOrderDO order = tradeOrderMapper.selectById(orderId);
    // 校验订单状态：只有待支付状态才能支付
    if (!TradeOrderStatusEnum.isUnpaid(order.getStatus())) {
        throw new ServiceException(ORDER_STATUS_NOT_UNPAID);
    }
    // 更新状态为待发货
    order.setStatus(TradeOrderStatusEnum.UNDELIVERED.getStatus());
    order.setPayTime(LocalDateTime.now());
    tradeOrderMapper.updateById(order);
}

public void deliverOrder(Long orderId) {
    TradeOrderDO order = tradeOrderMapper.selectById(orderId);
    // 校验订单状态：只有待发货状态才能发货
    if (!TradeOrderStatusEnum.isUndelivered(order.getStatus())) {
        throw new ServiceException(ORDER_STATUS_NOT_UNDELIVERED);
    }
    // 更新状态为已发货
    order.setStatus(TradeOrderStatusEnum.DELIVERED.getStatus());
    order.setDeliveryTime(LocalDateTime.now());
    tradeOrderMapper.updateById(order);
}
```

### 3.5 可迁移经验

**适用场景：**
- 订单、工单、审批单等有明确生命周期的业务对象
- 需要防止非法的状态转换

**设计要点：**
1. **枚举定义状态**：使用枚举类定义所有可能的状态
2. **语义化方法**：提供 `isXxx()` 和 `haveXxx()` 方法，提高代码可读性
3. **状态校验集中**：在 Service 层统一校验状态转换的合法性
4. **乐观锁保护**：使用版本号或状态条件更新，防止并发冲突

**进阶方案：**
- 使用状态机框架（如 Spring Statemachine）
- 将状态转换规则配置化
- 支持状态转换的钩子函数

## 4. 促销规则引擎的设计

### 4.1 问题场景

电商促销规则复杂多样：
- 满减：满 100 减 10
- 折扣：全场 8 折
- 优惠券：指定商品立减 20
- 积分抵扣：100 积分抵 1 元
- 组合优惠：买 3 件打 7 折

这些规则可能叠加使用，需要一个灵活的规则引擎来处理。


### 4.6 可迁移经验

**适用场景：**
- 促销规则复杂且多变
- 需要支持规则的动态配置
- 规则之间可能有优先级或叠加关系

**设计要点：**

**反模式警示：**
- 避免在一个方法中处理所有促销类型
- 避免硬编码促销规则
- 避免忽略优惠的边界情况（如优惠后价格为负数）

## 5. 可迁移经验：电商系统的核心架构模式

### 5.1 领域驱动设计（DDD）


### 5.2 核心设计原则

**1. 单一职责原则**
- 每个微服务只负责一个业务领域
- 商品服务只管商品，订单服务只管订单

**2. 服务自治原则**
- 每个服务有独立的数据库
- 服务间通过 API 通信，不直接访问对方数据库

**3. 最终一致性原则**
- 分布式事务采用最终一致性
- 通过消息队列实现异步处理

### 5.3 关键架构模式

**1. API 网关模式**

```
客户端 → API 网关 → 商品服务
                   → 订单服务
                   → 支付服务
```

**2. 服务发现模式**

```
商品服务注册到 Nacos ← 订单服务从 Nacos 获取商品服务地址
```

**3. 断路器模式**

```
订单服务调用商品服务 → 如果商品服务不可用 → 快速失败，返回默认值
```

**4. 事件驱动模式**

```
订单创建事件 → 库存扣减服务
            → 积分增加服务
            → 消息通知服务
```

### 5.4 数据一致性方案

**场景一：下单扣库存**

```
方案：本地消息表 + 定时任务

1. 订单服务：创建订单 + 写入消息表（事务）
2. 定时任务：扫描消息表，调用库存服务扣减
3. 库存服务：扣减库存，返回成功
4. 订单服务：更新消息状态为已完成
```

**场景二：支付回调更新订单**

```
方案：支付服务发送消息 → 订单服务消费消息

1. 支付服务：收到支付回调，更新支付状态
2. 支付服务：发送 MQ 消息
3. 订单服务：消费消息，更新订单状态
```

### 5.5 性能优化策略

**1. 缓存策略**
- 商品详情页：Redis 缓存
- 库存数据：Redis 预扣减
- 用户 Session：Redis 存储

**2. 异步处理**
- 订单创建后异步扣库存
- 支付成功后异步发通知
- 物流状态异步同步

**3. 数据库优化**
- 读写分离：主库写，从库读
- 分库分表：按用户 ID 分片
- 索引优化：订单号、用户 ID 等常用查询字段

### 5.6 可迁移经验

**适用场景：**
- 中大型电商系统
- 需要支持高并发和高可用
- 业务领域边界清晰

**设计要点：**
1. **领域驱动划分**：按业务领域划分服务，而非技术层次
2. **API 优先设计**：先定义 API 接口，再实现业务逻辑
3. **异步解耦**：通过消息队列实现服务间异步通信
4. **最终一致性**：接受短暂的数据不一致，通过补偿机制保证最终一致

**反模式警示：**
- 避免分布式单体：服务拆分了，但数据库还是共用
- 避免过度拆分：微服务数量过多，导致运维复杂
- 避免同步调用链过长：A → B → C → D，应改为异步

## 总结

电商模块通过分库分表策略实现了数据的水平扩展，通过乐观锁和 SQL 原子操作保证了库存扣减的并发安全，通过状态机模式管理订单的生命周期，通过策略模式实现了灵活的促销规则引擎。这些设计模式为构建高性能、高可用的电商系统提供了坚实的基础。

**核心设计要点：**
1. 分库分表按业务领域划分，实现服务自治
2. 库存扣减使用 SQL 原子操作，简单高效
3. 订单状态机通过枚举和状态判断方法，清晰管理订单生命周期
4. 促销规则引擎通过策略模式和配置化，支持灵活扩展
5. 电商系统的核心是领域驱动设计和最终一致性
