# 02 - 核心设计：商品与交易

## 商品模型：SPU 与 SKU 的深层设计

### 从"一个商品"说起

在日常生活中，我们说"买一部 iPhone 15"，但实际上购买的是一个非常具体的版本——比如"iPhone 15 黑色 128GB"。这个"具体版本"才是真正的交易对象。

电商领域用两个层次来描述商品：

- **SPU（Standard Product Unit）**：标准产品单元，回答"这是什么商品"
- **SKU（Stock Keeping Unit）**：库存量单位，回答"具体买哪个版本"

这种分层不是随意设计的，而是源于现实中商品的**组合爆炸**问题：

```
手机型号（SPU）: iPhone 15
├── 颜色: 黑色、白色、蓝色（3 种）
├── 存储: 128GB、256GB、512GB（3 种）
└── 组合: 3 × 3 = 9 个 SKU
```

如果每个组合都独立管理，数据量会爆炸式增长。SPU/SKU 分层让公共信息（名称、描述、品牌、分类）只存一份在 SPU 上，差异信息（价格、库存、属性组合）存在 SKU 上。

### ProductSpuDO 的设计哲学

源码中 `ProductSpuDO` 的字段可以分为四组：

**第一组：商品描述信息**

```java
// 来源：yudao-module-product-server/.../spu/ProductSpuDO.java
private String name;           // 商品名称
private String keyword;        // 搜索关键字
private String introduction;   // 简介（列表页展示）
private String description;    // 详情（富文本，详情页展示）
```

这些字段是所有 SKU 共享的。无论用户选黑色还是白色，看到的商品描述都是一样的。

**第二组：分类与品牌**

```java
private Long categoryId;  // 关联 ProductCategoryDO
private Long brandId;     // 关联 ProductBrandDO
```

分类和品牌挂在 SPU 上而非 SKU 上，因为"iPhone 15 是苹果品牌的手机"这个事实不会因为颜色不同而改变。

**第三组：聚合字段（从 SKU 派生）**

```java
private Integer price;       // 取所有 SKU 中最低价
private Integer marketPrice; // 取所有 SKU 中最低市场价
private Integer costPrice;   // 取所有 SKU 中最低成本价
private Integer stock;       // 所有 SKU 库存求和
```

这些字段是从 SKU **冗余聚合**而来。为什么不直接查 SKU 表？因为在商品列表页展示时，需要快速显示"价格从 xxx 元起"和"库存充足/紧张"，每次都 JOIN SKU 表会很慢。这是一种典型的**读优化**设计。

**第四组：状态与统计**

```java
private Integer status;          // -1 回收站 / 0 下架 / 1 上架
private Integer salesCount;      // 真实销量
private Integer virtualSalesCount; // 虚拟销量（运营设置，营造热销氛围）
private Integer browseCount;     // 浏览量
```

虚拟销量是一个有趣的运营手段：新上架的商品真实销量为 0，但可以设置虚拟销量为 500，让用户觉得"这个商品很畅销"。

### ProductSkuDO 的属性设计

SKU 的核心是 `properties` 字段，它用 JSON 数组存储属性组合：

```java
// 来源：yudao-module-product-server/.../sku/ProductSkuDO.java
@TableField(typeHandler = JacksonTypeHandler.class)
private List<Property> properties;

@Data
public static class Property {
    private Long propertyId;     // 属性项编号（关联 ProductPropertyDO）
    private String propertyName; // 属性项名称（冗余）
    private Long valueId;        // 属性值编号（关联 ProductPropertyValueDO）
    private String valueName;    // 属性值名称（冗余）
}
```

存储格式示例：

```json
[
  {"propertyId": 1, "propertyName": "颜色", "valueId": 10, "valueName": "黑色"},
  {"propertyId": 2, "propertyName": "存储", "valueId": 20, "valueName": "128GB"}
]
```

注意属性名和属性值名都被**冗余**存储了。这是因为属性名可能会被修改（比如把"颜色"改成"机身颜色"），但已生成的 SKU 属性组合需要保持历史快照。这和订单冗余商品信息是同一个设计思想：**关键时刻拍照，而非持续引用**。

### 单规格 vs 多规格

很多商品其实没有多种规格，比如一瓶矿泉水。系统通过 `specType` 字段区分：

```
specType = false（单规格）
  → 系统自动创建默认属性（ID_DEFAULT = 0, NAME_DEFAULT = "默认"）
  → 只生成一个 SKU
  → 用户无需选择规格，直接购买

specType = true（多规格）
  → 用户自定义属性项和属性值
  → 生成多个 SKU 组合
  → 用户需要选择规格后才能购买
```

这个设计避免了为简单商品强制维护属性的繁琐操作，体现了**渐进式复杂度**的理念。

## 订单流程：从购物车到完成

### 完整流程

```
┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
│ 浏览商品 │───▶│ 加购    │───▶│ 下单    │───▶│ 支付    │───▶│ 发货    │───▶│ 收货    │
│        │    │ 购物车  │    │        │    │        │    │        │    │ 完成    │
└────────┘    └────────┘    └────────┘    └────────┘    └────────┘    └────────┘
                  │              │              │              │
                  │              │              │              │
                  ▼              ▼              ▼              ▼
            CartDO         TradeOrderDO    PayOrderDO    物流系统
            选中商品        生成订单主表     调用支付服务    更新物流单号
            计算价格        生成订单明细     扣减库存        通知用户
            选择优惠        应用优惠券       记录支付流水
```

### 第一步：购物车

购物车的设计非常简洁：

```java
// 来源：yudao-module-trade-server/.../cart/CartDO.java
private Long userId;      // 用户编号
private Long spuId;       // 商品 SPU
private Long skuId;       // 商品 SKU
private Integer count;    // 购买数量
private Boolean selected; // 是否选中
```

`selected` 字段用于"部分结算"场景：用户购物车里有 5 件商品，但只想结算其中 3 件，通过 `selected = true` 标记要结算的商品。

### 第二步：下单

下单时，系统将购物车中的商品信息"拍照"存入订单。订单主表 `TradeOrderDO` 记录整体信息，订单明细 `TradeOrderItemDO` 记录每个商品的快照。

关键的价格计算公式：

```
应付金额 = 商品原价总和
         - 优惠券减免
         - 积分抵扣
         - 促销活动折扣
         - VIP 减免
         + 运费
         + 手动调价
```

这个公式在 `TradeOrderDO.payPrice` 的注释中有完整说明：

```java
// 来源：yudao-module-trade-server/.../order/TradeOrderDO.java
/**
 * 应付金额（总），单位：分
 *
 * = totalPrice        商品原价
 * - couponPrice        优惠券减免
 * - pointPrice         积分抵扣
 * - discountPrice      促销折扣
 * + deliveryPrice      运费
 * + adjustPrice        手动调价
 * - vipPrice           VIP 减免
 */
private Integer payPrice;
```

### 第三步：订单状态机

订单状态是整个交易流程的核心控制字段。状态流转如下：

```
                        用户取消                超时未支付
                    ┌───────────┐          ┌───────────┐
                    │           │          │           │
                    ▼           │          ▼           │
┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
│ 待支付 │───▶│ 已支付 │───▶│ 已发货 │───▶│ 已收货 │───▶│ 已完成 │
│  0    │    │  10   │    │  20   │    │  30   │    │  40   │
└──────┘    └──────┘    └──────┘    └──────┘    └──────┘
                │
                │  申请退款
                ▼
           ┌──────────┐
           │  退款中   │
           │   11     │
           └──────────┘
```

每个状态转换都会记录在 `TradeOrderLogDO` 中：

```java
// 来源：yudao-module-trade-server/.../order/TradeOrderLogDO.java
private Long orderId;       // 订单编号
private Integer beforeStatus; // 操作前状态
private Integer afterStatus;  // 操作后状态
private Integer operateType;  // 操作类型
private String content;       // 日志内容
private Long userId;          // 操作人
private Integer userType;     // 操作人类型（用户/管理员/系统）
```

这个日志表记录了订单的完整生命周期，便于排查问题和审计。

### 库存扣减：下单扣 vs 支付扣

库存扣减时机是电商系统中最经典的争议之一：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 下单扣库存 | 不会超卖 | 恶意下单会锁库存，导致真正想买的人买不到 | 高价值商品、库存紧张 |
| 支付扣库存 | 库存利用率高 | 可能超卖（多人同时支付最后一件） | 一般商品 |
| 预扣 + 超时释放 | 兼顾两者 | 实现复杂 | 秒杀、抢购 |

yudao-cloud 的默认策略是**下单扣库存**：用户下单时立即扣减 SKU 的 `stock`，如果订单超时未支付或用户取消，再回滚库存。

### 售后流程

售后状态独立于订单状态，通过 `TradeOrderItemDO.afterSaleStatus` 跟踪每个订单项的售后情况：

```
AfterSaleDO（售后单）
├── way: Integer        ← 售后方式：仅退款 / 退货退款
├── type: Integer       ← 售后类型
├── status: Integer     ← 售后状态
├── applyReason: String ← 申请原因
├── refundPrice: Integer← 退款金额
├── auditUserId: Long   ← 审批人
└── auditReason: String ← 审批备注
```

售后流程：

```
用户申请 → 商家审核 → [退货退款需要] 用户寄回商品 → 商家确认收货 → 退款
  │              │                                        │
  │              ▼                                        │
  │         拒绝申请                                      │
  │              │                                        │
  └──────────────┴────────────────────────────────────────┘
                    所有路径最终更新 AfterSaleDO.status
```

## 佣金分账：推广员体系

### 分销模型

yudao-cloud 采用**二级分销**模型，这是中国电商法规允许的最深层级：

```
推广员 A
├── 直推用户 B（一级关系）
│   ├── B 下单 → A 获得一级佣金（firstBrokeragePrice）
│   └── 直推用户 C（二级关系）
│       └── C 下单 → A 获得二级佣金，B 获得一级佣金
```

### 佣金生命周期

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 佣金产生  │───▶│ 佣金冻结  │───▶│ 佣金解冻  │───▶│ 佣金提现  │
│          │    │          │    │          │    │          │
│ 用户下单  │    │ frozenDays│    │ 确认收货  │    │ 申请提现  │
│ 记录佣金  │    │ 冻结天数  │    │ 可用余额  │    │ 审核转账  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

`BrokerageUserDO` 中用两个字段管理佣金状态：

```java
private Integer brokeragePrice; // 可用佣金（已解冻，可提现）
private Integer frozenPrice;    // 冻结佣金（待确认收货后解冻）
```

`BrokerageRecordDO` 记录每笔佣金的详细信息：

```java
private Long userId;          // 佣金归属用户
private Integer price;        // 佣金金额
private Integer totalPrice;   // 当时的总佣金（便于对账）
private Integer status;       // 状态
private Integer frozenDays;   // 冻结天数
private LocalDateTime unfreezeTime; // 解冻时间
private Long sourceUserId;    // 来源用户（谁下单产生的佣金）
private Integer sourceUserLevel; // 来源用户等级（一级/二级）
```

### 提现流程

```
推广员申请提现 → 管理员审核 → 通过 → 调用支付服务转账 → 完成
      │                           │
      │                           ▼
      │                      拒绝 → 退回佣金
      ▼
  BrokerageWithdrawDO
  ├── price: 提现金额
  ├── feePrice: 手续费
  ├── type: 提现类型（微信/支付宝/银行卡）
  └── status: 审核状态
```

## 运费计算

运费通过 `DeliveryExpressTemplateDO` 运费模板管理：

```java
// 来源：yudao-module-trade-server/.../delivery/DeliveryExpressTemplateDO.java
private String name;     // 模板名称
private Integer chargeMode; // 计费方式（按件/按重量/按体积）
```

运费模板关联两个子表：
- `DeliveryExpressTemplateChargeDO`：收费规则（如首件 10 元，续件 5 元）
- `DeliveryExpressTemplateFreeDO`：包邮条件（如满 99 元包邮）

## 设计模式提炼

### 1. 快照模式

订单冗余商品信息、SKU 冗余属性名——在**关键时刻**将关联数据"拍照"存入当前表，避免后续数据变更影响历史记录。

**适用场景**：订单、发票、审批记录等需要保持历史准确性的场景。

### 2. 聚合冗余模式

SPU 上冗余了 SKU 的最低价和总库存——这是一种**读优化**策略，用写入时的额外计算换取读取时的高性能。

**适用场景**：列表页展示、统计汇总、搜索结果。

### 3. 模板 + 实例模式

优惠券模板定义规则，用户领取时生成实例——一套规则服务无数实例，降低管理成本。

**适用场景**：优惠券、消息模板、审批流程模板。

### 4. 状态机模式

订单状态通过枚举定义有限状态，每次状态转换都有明确的触发条件和副作用——保证业务流转的可预测性。

**适用场景**：订单、售后、审批等有明确生命周期的业务实体。

## 思考题

1. 如果要支持"同一个 SKU 在不同仓库有不同库存"，你会如何扩展现有模型？
2. 订单的 `payPrice` 计算涉及多个优惠维度叠加，如果两个优惠互斥（如秒杀商品不能用优惠券），你会在哪个层面实现这个校验？
3. 分销体系中，如果推广员 A 的上级 B 被封禁，A 的佣金应该如何处理？
4. 虚拟销量和真实销量分开存储的设计，在统计分析时会带来什么挑战？
