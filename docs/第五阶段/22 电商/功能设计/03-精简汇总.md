## 订单状态机速查

```
状态码    状态名      可执行操作
─────────────────────────────────────────────────
  0      待支付      支付、取消（用户/系统超时）
 10      已支付      发货、申请退款
 11      退款中      退款确认
 20      已发货      确认收货、申请退货退款
 30      已收货      完成、申请退货退款
 40      已完成      评价、申请售后
 -1      已取消      （终态）
```

### 售后状态

```
状态码    状态名      说明
─────────────────────────────────────────────────
  0      未售后      正常状态
  1      售后中      已申请，待审核
  2      售后成功    退款/退货退款完成
  3      售后被拒    商家拒绝
```

## 核心设计决策

### 1. SPU/SKU 分层

| 决策 | 选择 | 原因 |
|------|------|------|
| 属性存储 | JSON 数组嵌入 SKU 表 | 属性组合是 SKU 的内在属性，非独立实体 |
| 价格聚合 | SPU 冗余 SKU 最低价 | 列表页高频读取，避免 JOIN |
| 单规格处理 | 自动创建默认属性 | 统一处理逻辑，减少条件分支 |

### 2. 订单设计

| 决策 | 选择 | 原因 |
|------|------|------|
| 商品信息 | 快照冗余到订单项 | 历史订单不受商品变更影响 |
| 价格字段 | 单位统一为"分" | 避免浮点数精度问题 |
| 库存扣减 | 下单时扣减 | 防止超卖，保证数据一致性 |
| 营销信息 | 冗余优惠券 ID 到订单 | 退款时需要归还优惠券 |

### 3. 促销体系

| 决策 | 选择 | 原因 |
|------|------|------|
| 优惠券 | 模板 + 实例两层 | 规则统一管理，实例独立生命周期 |
| 满减送 | JSON 存储阶梯规则 | 灵活支持多阶梯，避免额外表 |
| 秒杀库存 | 独立于商品库存 | 避免秒杀活动影响正常库存 |
| 拼团 | 虚拟成团选项 | 保证用户体验，避免"凑不齐人" |

### 4. 分销体系

| 决策 | 选择 | 原因 |
|------|------|------|
| 分销层级 | 最多二级 | 中国电商法规要求 |
| 佣金状态 | 冻结 + 可用两态 | 防止确认收货前提现导致资金风险 |
| 提现方式 | 微信/支付宝/银行卡 | 覆盖主流提现渠道 |
