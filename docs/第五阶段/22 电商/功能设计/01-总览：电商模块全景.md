# 01 - 总览：电商模块全景

## 场景引入

假设你正在为一家中型电商公司搭建后台管理系统。运营团队提出了以下需求：

- **商品管理**：上架一款手机，有黑色/白色两种颜色，128GB/256GB 两种存储容量，每种组合价格和库存不同
- **促销活动**：双十一要做限时秒杀，还要发满 300 减 50 的优惠券
- **交易闭环**：用户从浏览商品、加购物车、下单、支付、收货到评价，整个链路要完整
- **数据决策**：老板每天要看销售额、订单量、退款率等经营数据

一个完整的电商后台，需要同时覆盖**商品、促销、交易、统计**四大领域。这四个领域各有边界，但又紧密协作。

## 四个子服务

yudao-module-mall 采用**按领域拆分**的微服务架构，将电商业务拆分为四个独立的子服务：

```
yudao-module-mall/
├── yudao-module-product-api/      ← 商品服务 API（Feign 客户端 + DTO）
├── yudao-module-product-server/   ← 商品服务实现
├── yudao-module-promotion-api/    ← 促销服务 API
├── yudao-module-promotion-server/ ← 促销服务实现
├── yudao-module-trade-api/        ← 交易服务 API
├── yudao-module-trade-server/     ← 交易服务实现
├── yudao-module-statistics-api/   ← 统计服务 API
└── yudao-module-statistics-server/← 统计服务实现
```

这种拆分遵循 DDD（领域驱动设计）的**限界上下文**原则：每个子服务拥有自己的数据表、业务逻辑和 API 边界，通过 Feign 进行跨服务调用。

## 商品子服务：SPU/SKU 模型

商品子服务是整个电商系统的**数据基石**。核心挑战在于：同一款手机有多个颜色、多个存储容量，每种组合的价格和库存都不同。这就引出了电商领域最核心的概念——**SPU 与 SKU**。

### 核心概念

| 概念 | 全称 | 含义 | 举例 |
|------|------|------|------|
| SPU | Standard Product Unit | 标准产品单元，一组商品的统称 | iPhone 15 |
| SKU | Stock Keeping Unit | 库存量单位，最小可售卖单元 | iPhone 15 黑色 128GB |

一个 SPU 下可以有多个 SKU。SPU 描述的是"这是什么商品"，SKU 描述的是"具体买哪个版本"。

### 实体关系

```
ProductSpuDO（商品 SPU）
├── id: Long                    ← SPU 编号
├── name: String                ← 商品名称，如"iPhone 15"
├── categoryId: Long            ← 关联 ProductCategoryDO（商品分类）
├── brandId: Long               ← 关联 ProductBrandDO（品牌）
├── picUrl: String              ← 商品封面图
├── sliderPicUrls: List<String> ← 轮播图
├── specType: Boolean           ← 规格类型：false 单规格 / true 多规格
├── price: Integer              ← 价格（取 SKU 最低价）
├── stock: Integer              ← 库存（所有 SKU 库存求和）
├── salesCount: Integer         ← 销量
└── status: Integer             ← 状态：-1 回收站 / 0 下架 / 1 上架

ProductSkuDO（商品 SKU）
├── id: Long                    ← SKU 编号
├── spuId: Long                 ← 关联 SPU
├── properties: List<Property>  ← 属性组合，如 [{颜色:黑色}, {存储:128GB}]
├── price: Integer              ← 售价（分）
├── marketPrice: Integer        ← 市场价（分）
├── costPrice: Integer          ← 成本价（分）
├── stock: Integer              ← 库存
├── barCode: String             ← 条码
├── picUrl: String              ← SKU 图片
└── salesCount: Integer         ← 销量
```

### 辅助实体

```
ProductCategoryDO（商品分类）
├── parentId: Long              ← 父分类（支持二级树形结构）
├── name: String                ← 分类名称
└── status: Integer             ← 启用状态

ProductBrandDO（商品品牌）
├── name: String                ← 品牌名称
├── picUrl: String              ← 品牌图片
└── status: Integer             ← 启用状态

ProductPropertyDO（商品属性项）
├── name: String                ← 属性名，如"颜色"、"存储容量"

ProductPropertyValueDO（商品属性值）
├── propertyId: Long            ← 关联属性项
└── name: String                ← 属性值，如"黑色"、"128GB"
```

### SPU 与属性的关系

当 SPU 的 `specType = true`（多规格）时，通过属性项和属性值的组合生成多个 SKU：

```
iPhone 15（SPU）
├── 属性项：颜色 → 属性值：黑色、白色
├── 属性项：存储 → 属性值：128GB、256GB
│
└── 生成 SKU 组合：
    ├── SKU-1: 黑色 + 128GB → 价格 5999 元，库存 100
    ├── SKU-2: 黑色 + 256GB → 价格 6999 元，库存 80
    ├── SKU-3: 白色 + 128GB → 价格 5999 元，库存 60
    └── SKU-4: 白色 + 256GB → 价格 6999 元，库存 50
```

当 `specType = false`（单规格）时，系统会自动创建一个默认属性（`ProductPropertyDO.ID_DEFAULT`）和默认属性值（`ProductPropertyValueDO.ID_DEFAULT`），只生成一个 SKU。

## 促销子服务

促销子服务负责各种营销活动的管理。在电商场景中，常见的促销手段包括：

### 优惠券（CouponTemplateDO / CouponDO）

采用**模板 + 实例**的两层设计：

```
CouponTemplateDO（优惠券模板）          CouponDO（优惠券实例）
├── name: "双十一满减券"                ├── templateId: 关联模板
├── totalCount: 1000（发放总量）         ├── userId: 领取用户
├── takeLimitCount: 1（每人限领）        ├── status: 未使用/已使用/已过期
├── usePrice: 30000（满 300 元可用）     ├── validStartTime/validEndTime
├── discountType: PRICE（固定金额）      └── useOrderId: 使用的订单号
├── discountPrice: 5000（减 50 元）
└── productScope: ALL（全场通用）
```

模板定义了优惠券的规则，用户领取时生成实例。这种设计让"一张券"和"一万张券"共享同一套规则，只需管理模板即可。

### 满减送（RewardActivityDO）

满减送活动支持**阶梯式优惠**，通过 `rules` 字段存储多条规则：

```
规则 1：满 100 元 → 减 10 元 + 包邮
规则 2：满 300 元 → 减 50 元 + 包邮 + 赠送积分 500
规则 3：满 500 元 → 减 100 元 + 包邮 + 赠送积分 1000 + 赠送优惠券
```

### 限时折扣（DiscountActivityDO）

在指定时间段内对商品进行打折。一个活动关联多个商品（`DiscountProductDO`），每个商品可以设置不同的折扣比例。

### 秒杀（SeckillActivityDO）

限时限量的超低价促销。核心设计要点：

- **独立库存**：秒杀有自己独立的 `stock` 和 `totalStock`，与商品主库存隔离
- **限购控制**：`totalLimitCount`（总限购）和 `singleLimitCount`（单次限购）
- **时段配置**：通过 `configIds` 关联秒杀时段配置

### 拼团（CombinationActivityDO）

多人组团享受优惠价的社交电商模式：

- `userSize`：几人成团（如 3 人团）
- `limitDuration`：成团时限（如 24 小时内）
- `virtualGroup`：虚拟成团（人数不足时系统自动补人，保证用户体验）

## 交易子服务

交易子服务是整个电商系统的**业务核心**，承载从购物车到订单完成的完整交易链路。

### 购物车（CartDO）

```
CartDO
├── userId: Long       ← 用户编号
├── spuId: Long        ← 商品 SPU
├── skuId: Long        ← 商品 SKU
├── count: Integer     ← 购买数量
└── selected: Boolean  ← 是否选中（结算时只处理选中的商品）
```

### 订单（TradeOrderDO / TradeOrderItemDO）

订单采用**主表 + 明细表**的经典设计：

```
TradeOrderDO（订单主表）
├── no: String              ← 订单流水号（如 1146347329394184195）
├── userId: Long            ← 下单用户
├── status: Integer         ← 订单状态
├── totalPrice: Integer     ← 商品原价总和
├── discountPrice: Integer  ← 优惠金额
├── deliveryPrice: Integer  ← 运费
├── payPrice: Integer       ← 实际支付金额
├── payOrderId: Long        ← 关联支付订单
├── deliveryType: Integer   ← 配送方式（快递/自提）
├── receiverName: String    ← 收件人
├── receiverMobile: String  ← 收件人手机
└── refundStatus: Integer   ← 售后状态

TradeOrderItemDO（订单明细）
├── orderId: Long           ← 关联订单主表
├── spuId/skuId: Long       ← 关联商品
├── spuName: String         ← 冗余商品名（避免关联查询）
├── properties: List<Property> ← 冗余属性快照
├── price: Integer          ← 商品单价
├── count: Integer          ← 购买数量
├── payPrice: Integer       ← 应付金额
└── afterSaleStatus: Integer← 售后状态
```

订单明细中**大量冗余**了商品信息（名称、属性、图片），这是一种典型的**快照设计**：下单时将商品信息"拍照"存入订单，即使后续商品信息变更，订单数据也不受影响。

### 售后（AfterSaleDO）

支持两种售后方式：
- **仅退款**：未发货或无需退货的场景
- **退货退款**：已收货但不满意，需要退回商品

### 分销（BrokerageUserDO / BrokerageRecordDO / BrokerageWithdrawDO）

分销体系采用**二级分佣**模型：

```
推广员 A 发展了用户 B，用户 B 又发展了用户 C
├── 用户 C 下单 → A 获得一级佣金，B 获得二级佣金
├── 佣金先冻结（frozenPrice），确认收货后解冻到可用余额（brokeragePrice）
└── 推广员申请提现 → 审核通过 → 转账到微信/支付宝/银行卡
```

### 物流（DeliveryExpressTemplateDO）

运费模板支持按**计费方式**（按件/按重量/按体积）计算运费，并支持设置包邮条件。

## 统计子服务

统计子服务不直接参与业务流程，而是通过**定时任务**聚合其他子服务的数据，为运营决策提供支撑。

### TradeStatisticsDO（交易统计）

以天为维度聚合：
- `orderCreateCount`：创建订单数
- `orderPayCount`：支付订单数
- `orderPayPrice`：支付总金额
- `afterSaleCount`：退款订单数
- `brokerageSettlementPrice`：已结算佣金

### ProductStatisticsDO（商品统计）

聚合商品维度的数据，如销量排行、库存预警等。

### 统计服务的特殊性

统计服务不拥有自己的业务写入表，它通过 RPC 调用其他服务的 API 获取数据，然后写入统计表。这种设计体现了**CQRS（命令查询职责分离）**的思想：业务服务负责写入，统计服务负责读取和聚合。

## 子服务间的协作关系

```
                    ┌──────────────┐
                    │  商品服务     │
                    │  Product     │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
     ┌────────────┐ ┌────────────┐ ┌────────────┐
     │  促销服务   │ │  交易服务   │ │  统计服务   │
     │  Promotion │ │  Trade     │ │  Statistics│
     └────────────┘ └────────────┘ └────────────┘

数据流：
1. 促销服务 ← 读取商品信息（价格、库存）用于活动校验
2. 交易服务 ← 读取商品信息用于下单；读取促销信息用于计算优惠
3. 统计服务 ← 聚合交易数据、商品数据生成统计报表
4. 交易服务 → 扣减商品库存、核销优惠券
```

## 设计决策：为什么拆成四个子服务？

| 考量维度 | 拆分的理由 | 不拆分的问题 |
|---------|-----------|------------|
| 业务边界 | 商品、促销、交易、统计各自有独立的业务闭环 | 单体代码耦合严重，改促销逻辑可能影响交易 |
| 数据隔离 | 每个子服务只操作自己的表，降低锁竞争 | 所有表在一个库中，高并发时互相影响 |
| 独立部署 | 促销高峰期（双十一）可以单独扩容促销和交易服务 | 必须整体扩容，资源浪费 |
| 团队协作 | 不同团队可以并行开发不同子服务 | 代码冲突频繁，发布互相阻塞 |

但拆分也带来了代价：跨服务调用增加了复杂度和延迟。项目通过**Feign 声明式 RPC** 和**统一的 API 模块**（`*-api`）来管理这种复杂度。

## 思考题

1. SPU 和 SKU 的关系本质上是什么设计模式？如果要支持"同一个 SKU 在不同地区价格不同"，你会如何扩展数据模型？
2. 订单明细为什么要冗余商品信息而不是关联查询？这种"快照模式"在什么场景下会带来问题？
3. 统计服务为什么不直接查询业务服务的数据库，而是通过 RPC 获取数据？
4. 如果要把促销服务从"优惠券 + 满减 + 秒杀 + 拼团"进一步拆分为独立服务，你会怎么划分边界？
