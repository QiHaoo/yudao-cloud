# 02 - 核心设计：客户与商机管理

## 场景引入

上一章我们看到了 CRM 的全景——从线索到回款的销售漏斗。这一章聚焦几个核心设计问题：

1. **客户池机制**：公海和私海如何运作？客户回收的规则怎么设计？
2. **商机管理**：商机的阶段流转怎么建模？如何跟踪销售漏斗？
3. **合同与回款**：合同审批流程怎么设计？回款如何与合同关联？
4. **数据权限**：CRM 的数据权限和系统模块的数据权限有什么不同？

## 客户池：公海与私海

### 设计动机

销售小李手上有 200 个客户，但他只认真跟进了 50 个。剩下 150 个客户"沉睡"着，没有任何销售在关注。如果竞争对手这时候来挖客户，公司完全不知道。

客户池机制要解决的核心问题是：**让客户资源始终处于被有效管理的状态**。

### 公海与私海模型

```
┌─────────────────────────────────────────────────────────────┐
│                       客户池全景                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                    公海池 (Public Pool)                │   │
│  │                                                     │   │
│  │  来源：                                               │   │
│  │  ├── 新导入的客户（批量 Excel 导入）                   │   │
│  │  ├── 线索转化后的客户（未分配负责人）                   │   │
│  │  ├── 被回收的客户（超过 N 天未跟进）                   │   │
│  │  └── 离职销售的客户                                   │   │
│  │                                                     │   │
│  │  特征：ownerUserId = null（无负责人）                 │   │
│  └──────────────────────────┬──────────────────────────┘   │
│                             │                               │
│          捞取（领取）        │        回收（自动/手动）       │
│                             ▼                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ 销售A私海 │  │ 销售B私海 │  │ 销售C私海 │  │ 销售D私海 │   │
│  │  30个客户 │  │  45个客户 │  │  28个客户 │  │  52个客户 │   │
│  │          │  │          │  │          │  │          │   │
│  │ 特征：    │  │ 特征：    │  │ 特征：    │  │ 特征：    │   │
│  │ owner    │  │ owner    │  │ owner    │  │ owner    │   │
│  │ UserId   │  │ UserId   │  │ UserId   │  │ UserId   │   │
│  │ = 销售A  │  │ = 销售B  │  │ = 销售C  │  │ = 销售D  │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 核心实体：CrmCustomerDO

```java
public class CrmCustomerDO extends BaseDO {
    private Long id;                    // 客户编号
    private String name;                // 客户名称

    // ========== 跟进状态 ==========
    private Boolean followUpStatus;     // 跟进状态
    private LocalDateTime contactLastTime;   // 最后跟进时间
    private String contactLastContent;  // 最后跟进内容
    private LocalDateTime contactNextTime;   // 下次联系时间

    // ========== 负责人 ==========
    private Long ownerUserId;           // 负责人（null = 公海客户）
    private LocalDateTime ownerTime;    // 成为负责人的时间

    // ========== 状态标记 ==========
    private Boolean lockStatus;         // 锁定状态（锁定后不回收）
    private Boolean dealStatus;         // 成交状态

    // ========== 联系方式 ==========
    private String mobile;              // 手机
    private String telephone;           // 电话
    private String qq;                  // QQ
    private String wechat;              // 微信
    private String email;               // 邮箱

    // ========== 基础信息 ==========
    private Integer areaId;             // 所在地
    private String detailAddress;       // 详细地址
    private Integer industryId;         // 所属行业
    private Integer level;              // 客户等级
    private Integer source;             // 客户来源
    private String remark;              // 备注
}
```

**关键设计：`ownerUserId` 是公海/私海的判断依据。**

- `ownerUserId = null`：客户在公海池，任何人都可以领取
- `ownerUserId = 某销售ID`：客户在该销售的私海中

这种设计比用一个显式的 `poolType` 字段更自然——负责人本身就是"归属"的语义。

### 客户回收机制

```
定时任务（每天凌晨执行）
  │
  ▼
扫描所有私海客户
  │
  ├── 条件 1：contactLastTime < (今天 - contactExpireDays)
  │     → 超过 N 天未跟进
  │     → 回收：ownerUserId = null
  │
  ├── 条件 2：dealStatus = false 且 ownerTime < (今天 - dealExpireDays)
  │     → 超过 M 天未成交
  │     → 回收：ownerUserId = null
  │
  └── 条件 3：lockStatus = true
        → 已锁定的客户不回收
        → 跳过
```

**配置实体：**

```java
// CrmCustomerPoolConfigDO - 客户池配置
public class CrmCustomerPoolConfigDO {
    private Long id;
    private Integer contactExpireDays;    // 联系过期天数（如 30 天）
    private Integer dealExpireDays;       // 成交过期天数（如 90 天）
    private Boolean enabled;              // 是否启用自动回收
}
```

### 客户数量限制

防止某个销售"囤积"过多客户：

```java
// CrmCustomerLimitConfigDO - 客户数量限制
public class CrmCustomerLimitConfigDO {
    private Long id;
    private Integer type;      // 限制类型：全局/部门/个人
    private Integer maxCount;  // 最大客户数量
}
```

领取客户时的校验逻辑：

```
销售 A 申请领取公海客户
  │
  ▼
校验流程
  │
  ├── 1. 查询销售 A 当前持有的客户数量
  ├── 2. 查询销售 A 适用的限制配置
  │     ├── 先查个人限制
  │     ├── 没有则查部门限制
  │     └── 都没有则查全局限制
  ├── 3. 当前数量 >= maxCount？
  │     ├── 是 → 拒绝："您的客户数量已达上限，请先释放部分客户"
  │     └── 否 → 允许领取
  └── 4. 更新客户 ownerUserId = 销售 A 的 ID
```

## 商机管理

### 商机的业务含义

商机（Business）是"有可能成交的交易机会"。以电商 B2B 场景为例：

- 客户"XX 贸易公司"想采购 100 台空调 → 这是一个商机
- 客户"YY 餐饮集团"要定制 500 套制服 → 这是另一个商机

商机和客户是**一对多**关系：一个客户可以有多个商机（不同时期的不同需求）。

### 商机阶段流转

CRM 的商机通常有一个**销售阶段**（Sales Stage），从初步接触到最终成交：

```
商机阶段流转
  │
  ├── 1. 初步接触    (10%)   → 刚建立联系，了解需求
  ├── 2. 需求确认    (20%)   → 明确客户需要什么
  ├── 3. 方案报价    (40%)   → 提供解决方案和报价
  ├── 4. 商务谈判    (60%)   → 讨论价格、条款
  ├── 5. 合同签订    (80%)   → 签订合同
  └── 6. 成交/丢失   (100%)  → 最终结果
```

括号中的百分比是**赢单概率**，用于销售预测。

### 商机实体

```java
public class CrmBusinessDO extends BaseDO {
    private Long id;                    // 商机编号
    private String name;                // 商机名称
    private Long customerId;            // 客户编号
    private Long contactId;             // 联系人编号
    private Integer statusId;           // 销售阶段（关联 CrmBusinessStatusDO）
    private BigDecimal price;           // 商机金额
    private BigDecimal discountPercent; // 折扣率
    private LocalDateTime dealTime;     // 预计成交时间
    private Long ownerUserId;           // 负责人
    private Boolean dealStatus;         // 成交状态
    private Integer source;             // 商机来源
    private String remark;              // 备注
}
```

### 销售阶段配置

销售阶段不是硬编码的，而是通过配置表管理：

```java
public class CrmBusinessStatusDO extends BaseDO {
    private Long id;            // 阶段编号
    private String name;        // 阶段名称
    private Integer sort;       // 排序
    private Integer probability; // 赢单概率（百分比）
}
```

这样不同行业、不同公司可以自定义销售阶段。比如：
- 标准 B2B：初步接触 → 需求确认 → 方案报价 → 商务谈判 → 合同签订
- 电商招商：意向沟通 → 资质审核 → 样品确认 → 合同签订 → 入驻完成

### 商机与产品

一个商机涉及哪些产品？通过 `CrmBusinessProductDO` 关联：

```
CrmBusinessDO (商机)
  │
  └── 1:N ── CrmBusinessProductDO (商机产品)
                ├── productId    - 产品编号
                ├── productName  - 产品名称（冗余）
                ├── productPrice - 产品单价
                ├── quantity     - 数量
                └── totalPrice   - 小计金额
```

**为什么冗余存储 productName 和 productPrice？**

因为产品信息可能会变（改名、调价），但商机创建时的价格应该固定。冗余存储保证了历史商机的产品信息不会因为产品变更而"失真"。

## 合同管理

### 合同实体

```java
public class CrmContractDO extends BaseDO {
    private Long id;                    // 合同编号
    private String name;                // 合同名称
    private String no;                  // 合同编号（业务编号）
    private Long customerId;            // 客户编号
    private Long businessId;            // 商机编号（可选）

    // ========== 跟进信息 ==========
    private LocalDateTime contactLastTime;   // 最后跟进时间
    private Long ownerUserId;           // 负责人

    // ========== 审批信息 ==========
    private String processInstanceId;   // 工作流实例编号
    private Integer auditStatus;        // 审批状态

    // ========== 合同信息 ==========
    private LocalDateTime orderDate;    // 下单日期
    private LocalDateTime startTime;    // 开始时间
    private LocalDateTime endTime;      // 结束时间
    private BigDecimal totalProductPrice; // 产品总金额
    private BigDecimal discountPercent; // 整单折扣
    private BigDecimal totalPrice;      // 合同总金额
    private Long signContactId;         // 客户签约人
    private Long signUserId;            // 公司签约人
    private String remark;              // 备注
}
```

### 合同审批流程

合同通常需要审批。CRM 模块与工作流模块（BPM）集成：

```
销售创建合同
  │
  ▼
提交审批
  │
  ├── 创建 ProcessInstance（工作流实例）
  ├── 设置 CrmContractDO.processInstanceId
  └── 设置 CrmContractDO.auditStatus = 审核中

审批流程（由 BPM 模块驱动）
  │
  ├── 销售主管审批
  │     ├── 通过 → 继续
  │     └── 驳回 → auditStatus = 驳回，销售可修改后重新提交
  │
  └── 总监审批（金额超过阈值时）
        ├── 通过 → auditStatus = 审核通过
        └── 驳回 → auditStatus = 驳回
```

### 合同产品明细

合同中包含哪些产品，通过 `CrmContractProductDO` 关联：

```
CrmContractDO (合同)
  │
  └── 1:N ── CrmContractProductDO (合同产品)
                ├── productId      - 产品编号
                ├── productName    - 产品名称
                ├── productPrice   - 产品单价
                ├── quantity       - 数量
                ├── totalPrice     - 小计金额
                └── discountPercent - 单品折扣
```

### 合同金额计算

```
合同总金额计算逻辑：
  │
  ├── totalProductPrice = SUM(每个产品的小计金额)
  │
  ├── discountPrice = totalProductPrice * discountPercent / 100
  │
  └── totalPrice = totalProductPrice - discountPrice
```

## 回款管理

### 回款的业务含义

签了合同不代表钱到手。合同约定"分三期付款"，每期付款就是一条回款记录。

### 回款实体

```java
public class CrmReceivableDO extends BaseDO {
    private Long id;                    // 回款编号
    private Long customerId;            // 客户编号
    private Long contractId;            // 合同编号
    private Long ownerUserId;           // 负责人
    private BigDecimal price;           // 回款金额
    private LocalDateTime returnTime;   // 回款时间
    private Integer returnType;         // 回款方式（现金/转账/支票）
    private Integer auditStatus;        // 审批状态
    private String remark;              // 备注
}
```

### 回款与合同的关系

```
CrmContractDO (合同：总金额 100,000 元)
  │
  ├── 1:N ── CrmReceivableDO (回款记录)
  │           ├── 回款 1：30,000 元（定金）
  │           ├── 回款 2：40,000 元（发货后）
  │           └── 回款 3：30,000 元（验收后）
  │
  └── 已回款总额 = SUM(receivable.price)
      剩余待回款 = totalPrice - 已回款总额
```

### 回款审批

回款也需要审批，防止虚假回款：

```
销售录入回款记录
  │
  ▼
提交审批
  │
  ├── 财务确认到账
  │     ├── 确认 → auditStatus = 审核通过
  │     └── 未到账 → auditStatus = 驳回
  │
  └── 更新合同的已回款金额
```

## 跟进记录

### 设计思路

跟进记录是 CRM 的"日志系统"——记录每一次与客户的沟通。它不是客户表的一个字段，而是独立的实体。

**为什么不放在客户表里？**

```
方案 A：客户表加一个 lastContactContent 字段
  ├── 优点：简单
  └── 缺点：只能存最后一次，历史记录丢失

方案 B：独立的跟进记录表（CrmFollowUpRecordDO）
  ├── 优点：完整的历史记录，支持时间线展示
  └── 缺点：多一张表，查询需要 JOIN
```

显然方案 B 更合理。跟进是一个持续的过程，不是一次性记录。

### 跟进记录实体

```java
public class CrmFollowUpRecordDO extends BaseDO {
    private Long id;                    // 记录编号
    private Integer bizType;            // 业务类型（客户/商机/合同）
    private Long bizId;                 // 业务 ID
    private Integer type;               // 跟进方式（电话/拜访/邮件/微信）
    private String content;             // 跟进内容
    private LocalDateTime nextTime;     // 下次跟进时间
    private List<String> fileUrls;      // 附件（录音、照片等）
}
```

**`bizType` + `bizId` 的多态设计：**

跟进记录可以关联客户、商机、合同等多种业务实体。用 `bizType` 区分类型，`bizId` 存储对应实体的 ID。这是一种常见的**多态关联**模式。

### 跟进与客户状态联动

创建跟进记录时，自动更新客户的跟进状态：

```
创建跟进记录
  │
  ▼
CrmFollowUpRecordService.createFollowUpRecord()
  │
  ├── 1. 创建 CrmFollowUpRecordDO
  ├── 2. 更新关联客户的跟进状态
  │     ├── CrmCustomerDO.contactLastTime = 当前时间
  │     ├── CrmCustomerDO.contactLastContent = 跟进内容
  │     └── CrmCustomerDO.contactNextTime = 下次跟进时间
  └── 3. 重置回收计时器
        → 因为 contactLastTime 更新了，回收定时任务不会回收这个客户
```

## 数据权限：CRM 专属设计

### 为什么 CRM 需要自己的权限表？

系统模块的数据权限（`DataPermissionRule`）是基于**组织架构**的：按部门、按岗位、按角色。但 CRM 的权限需求不同：

```
场景 1：销售 A 把客户转移给销售 B
  ├── 系统权限：不变（两人可能在同一部门）
  └── CRM 权限：需要更新（A 失去权限，B 获得权限）

场景 2：大客户需要多人协作
  ├── 系统权限：无法表达"这个客户由 A 和 B 共同负责"
  └── CRM 权限：需要支持负责人 + 协作者 + 只读用户

场景 3：销售总监看所有客户
  ├── 系统权限：可以做到（全部数据权限）
  └── 但如果总监换了，需要重新配置
```

### CrmPermissionDO 设计

```java
public class CrmPermissionDO extends BaseDO {
    private Long id;            // 权限编号
    private Integer bizType;    // 业务类型（客户/商机/合同/回款）
    private Long bizId;         // 业务 ID
    private Long userId;        // 用户编号
    private Integer level;      // 权限级别
}
```

权限级别：

| 级别 | 值 | 说明 |
|------|---|------|
| 负责人 | 1 | 完全控制（编辑、删除、转移） |
| 协作者 | 2 | 可查看、可编辑 |
| 只读 | 3 | 只能查看 |

### 权限查询流程

```
用户请求查询客户列表
  │
  ▼
权限过滤
  │
  ├── 1. 系统数据权限过滤
  │     → 根据用户的部门、角色，过滤出"可见"的客户
  │
  ├── 2. CRM 权限过滤
  │     → 查询 CrmPermissionDO，找出 userId = 当前用户的记录
  │     → 合并到查询条件中
  │
  └── 3. 返回结果
        → 用户看到的是"系统权限 OR CRM 权限"的并集
```

### 客户转移

销售 A 离职，客户转给销售 B：

```
客户转移流程
  │
  ▼
CrmCustomerService.transferCustomer()
  │
  ├── 1. 更新 CrmCustomerDO.ownerUserId = B
  ├── 2. 更新 CrmPermissionDO
  │     ├── 删除 A 的权限记录
  │     └── 新增 B 的权限记录（level = 负责人）
  ├── 3. 关联数据同步转移
  │     ├── 该客户的商机 → ownerUserId = B
  │     ├── 该客户的合同 → ownerUserId = B
  │     └── 该客户的回款 → ownerUserId = B
  └── 4. 记录操作日志
```

## 线索转化

### 线索到客户的转化

线索（CrmClueDO）是未经验证的潜在客户。转化流程：

```
线索（CrmClueDO）
  │
  ├── 确认有合作意向
  │
  ▼
转化操作
  │
  ├── 1. 创建 CrmCustomerDO（从线索信息复制）
  ├── 2. 创建 CrmContactDO（联系人）
  ├── 3. 创建 CrmPermissionDO（分配权限）
  ├── 4. 更新线索状态为"已转化"
  └── 5. 可选：自动创建商机
```

### 线索实体

```java
public class CrmClueDO extends BaseDO {
    private Long id;            // 线索编号
    private String name;        // 线索名称（客户名称）
    private String contactName; // 联系人名称
    private String contactPhone; // 联系电话
    private String contactEmail; // 联系邮箱
    private Integer source;     // 线索来源
    private Long ownerUserId;   // 负责人
    private Integer status;     // 状态（待跟进/已转化/已关闭）
    private String remark;      // 备注
}
```

### 线索也有公海机制

和客户一样，线索也有公海/私海：

```
线索公海
  │
  ├── 新导入的线索 → 进入公海
  ├── 销售领取 → 进入私海
  └── 超时未跟进 → 回收到公海
```

## 思考题

1. 客户的 `ownerUserId` 为 null 表示公海客户。如果需要支持"多个销售共同负责一个客户"，这个设计需要怎么改？
2. 跟进记录用 `bizType + bizId` 实现多态关联。如果后续需要支持"对某个跟进记录评论"，应该怎么扩展？
3. 合同审批走工作流引擎。如果工作流模块不可用（比如没部署 BPM），CRM 的合同功能还能用吗？应该怎么降级？
4. 回款记录的审批和合同的审批用的是同一个审批引擎吗？如果审批规则不同怎么办？
