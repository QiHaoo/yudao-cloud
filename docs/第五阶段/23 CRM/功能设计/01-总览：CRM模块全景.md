# CRM 模块全景

## 场景引入

想象你是一家电商公司的销售总监。你手下有 20 个销售，每人手里握着几十个客户。问题来了：

- 销售小王离职了，他的客户资料怎么办？新来的销售怎么接手？
- 同一个客户被两个销售同时跟进，客户抱怨"你们公司内部都不沟通的吗？"
- 销售说"我这个月谈了 50 个客户"，但成交了几个？卡在哪个环节了？
- 财务问"这个季度签了多少合同？回款多少？"，销售支支吾吾答不上来。

**CRM（Customer Relationship Management，客户关系管理）** 就是为了解决这些问题而生的。它不是一个简单的通讯录，而是一套完整的**销售过程管理工具**。

### 为什么电商也需要 CRM？

很多人以为 CRM 只适合 B2B 企业销售。其实不然：

- **电商 B2B 业务**：批发客户、分销商、供应商关系管理
- **大客户运营**：VIP 客户维护、专属客服对接
- **售后回访**：订单完成后的客户满意度跟踪
- **招商管理**：平台招商、商户入驻跟进

核心价值是：**让客户资源属于公司，而不是属于某个销售个人**。

### 销售漏斗模型

CRM 的核心是一个**销售漏斗**，从线索到成交层层转化：

```
        ┌─────────────────────┐
        │       线索 (Leads)    │  ← 来源：广告、展会、网站注册
        │   "有个客户来咨询了"    │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │     客户 (Customer)   │  ← 确认有合作意向
        │   "这个客户值得跟进"    │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │     商机 (Business)   │  ← 明确需求，准备报价
        │   "客户要买 100 台设备" │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │     合同 (Contract)   │  ← 双方确认，签约
        │   "合同签了，准备发货"  │
        └──────────┬──────────┘
                   ▼
        ┌─────────────────────┐
        │     回款 (Receivable) │  ← 按合同约定收款
        │   "第一笔款已到账"     │
        └─────────────────────┘
```

每经过一层，数量都在减少——这就是"漏斗"。CRM 帮你追踪每个阶段的转化率，找出卡点。

|--------|------|------|

| `CrmContactDO` | `crm_contact` | 客户的联系人，一个客户可有多个联系人 |

| `CrmReceivableDO` | `crm_receivable` | 合同的回款记录 |
| `CrmProductDO` | `crm_product` | 产品目录，商机和合同中引用 |

| `CrmFollowUpRecordDO` | `crm_follow_up_record` | 跟进记录，记录每次沟通内容 |

## 客户池机制：公海与私海

这是 CRM 最核心的设计之一。以电商场景为例：

### 问题场景

销售小李手上有 200 个客户，但他只认真跟进了 50 个。剩下 150 个客户没人管，白白流失。

### 解决方案：客户池

```
┌─────────────────────────────────────────────────────────────────┐
│                         客户池机制                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                     公海池 (Public Pool)                  │  │
│   │                                                         │  │
│   │   未分配的客户  │  被回收的客户  │  新导入的客户           │  │
│   │                                                         │  │
│   │   规则：任何人都可以"捞取"到自己的私海                     │  │
│   └─────────────────────────────────────────────────────────┘  │
│                          │                                      │
│                          │ 捞取/分配                            │
│                          ▼                                      │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│   │ 销售A私海 │  │ 销售B私海 │  │ 销售C私海 │  │ 销售D私海 │      │
│   │  30个客户 │  │  45个客户 │  │  28个客户 │  │  52个客户 │      │
│   └──────────┘  └──────────┘  └──────────┘  └──────────┘      │
│                                                                 │
│   回收规则（示例）：                                             │
│   ├── 超过 30 天未跟进 → 自动回收到公海                         │
│   ├── 超过 90 天未成交 → 自动回收到公海                         │
│   └── 销售离职 → 全部客户回收到公海                             │
│                                                                 │
│   限制规则：                                                     │
│   ├── 每个销售最多持有 N 个客户                                  │
│   └── 有进行中商机的客户不回收                                   │
└─────────────────────────────────────────────────────────────────┘
```

### 配置实体

```java
// CrmCustomerPoolConfigDO - 客户池配置
public class CrmCustomerPoolConfigDO {
    private Long id;
    private Integer contactExpireDays;    // 联系过期天数（多久没联系就回收）
    private Integer dealExpireDays;       // 成交过期天数
    private Boolean enabled;              // 是否启用自动回收
}

// CrmCustomerLimitConfigDO - 客户数量限制
public class CrmCustomerLimitConfigDO {
    private Long id;
    private Integer type;      // 限制类型：全局/部门/个人
    private Integer maxCount;  // 最大客户数量
}
```

### 电商场景举例

- **新注册的批发商**：自动进入公海池，销售可以申请认领
- **VIP 客户**：长期不跟进会被回收，分配给更积极的销售
- **销售离职**：所有客户自动回到公海，由主管重新分配

## 与权限系统集成

CRM 模块的权限控制比普通业务模块更复杂。除了菜单权限（谁能访问 CRM 模块），还需要**数据权限**（谁能看哪些客户）。

### 权限层级

```
┌─────────────────────────────────────────────────────────────────┐
│                       CRM 权限模型                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Level 1: 菜单权限 (Menu Permission)                           │
│   ├── crm:customer:query    查看客户                            │
│   ├── crm:customer:create   新建客户                            │
│   ├── crm:customer:update   编辑客户                            │
│   └── crm:customer:delete   删除客户                            │
│                                                                 │
│   Level 2: 数据权限 (Data Permission)                           │
│   ├── 全部数据权限  ────  销售总监：看所有客户                   │
│   ├── 部门数据权限  ────  销售经理：看本部门的客户               │
│   ├── 个人数据权限  ────  普通销售：只看自己的客户               │
│   └── 自定义权限    ────  特殊情况：指定某些客户                 │
│                                                                 │
│   Level 3: CRM 专属权限 (CrmPermissionDO)                       │
│   ├── 负责人 (Owner)   ──── 客户的主要负责人                    │
│   ├── 协作者 (Collaborator) ── 可以查看和编辑                   │
│   └── 读取者 (Reader)  ──── 只能查看                           │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │ CrmPermissionDO                                         │  │
│   │ ├── bizType: 业务类型（客户/商机/合同等）                │  │
│   │ ├── bizId: 业务ID                                       │  │
│   │ ├── userId: 用户ID                                      │  │
│   │ └── level: 权限级别（负责人/协作者/读取者）              │  │
│   └─────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 权限控制流程

```
用户请求查询客户列表
       │
       ▼
┌──────────────┐
│  菜单权限检查  │ ──── 无权限 → 返回 403
└──────┬───────┘
       │ 有权限
       ▼
┌──────────────┐
│  数据权限过滤  │ ──── 追加 WHERE 条件
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ CRM 权限过滤  │ ──── 根据 CrmPermissionDO 进一步过滤
└──────┬───────┘
       │
       ▼
   返回客户列表
```

## 关键设计决策

### 1. 线索与客户分离

为什么不让销售直接创建客户？

```
线索 (CrmClueDO)                    客户 (CrmCustomerDO)
├── 来源不确定                       ├── 已确认有合作意向
├── 信息可能不完整                   ├── 信息经过验证
├── 可以批量导入                     ├── 关联商机和合同
└── 转化后才成为客户                 └── 有明确的负责人
```

**好处**：
- 避免销售随意创建客户，污染客户库
- 线索可以批量导入，经过筛选后再转化为客户
- 线索转化过程本身就是一次质量筛选

### 2. 跟进记录独立设计

为什么不把跟进内容放在客户表的某个字段里？

```java
// CrmFollowUpRecordDO - 独立的跟进记录
public class CrmFollowUpRecordDO {
    private Long id;
    private Integer bizType;    // 关联的业务类型（客户/商机）
    private Long bizId;         // 关联的业务ID
    private Integer type;       // 跟进方式（电话/拜访/邮件/微信）
    private String content;     // 跟进内容
    private LocalDateTime nextTime;  // 下次跟进时间
}
```

**好处**：
- 跟进是持续的过程，不是一次性记录
- 可以按时间线查看完整的跟进历史
- 支持多种跟进方式的分类统计

### 3. 权限系统的双层设计

为什么 CRM 需要自己的权限表 `CrmPermissionDO`，而不能只用系统模块的权限？

```
系统权限 (system模块)                  CRM权限 (CrmPermissionDO)
├── 控制菜单访问                       ├── 控制数据访问
├── 基于角色分配                       ├── 基于业务关系分配
├── 相对静态                           ├── 动态变化（转移、共享）
└── 通用框架                           └── CRM 专属逻辑
```

**场景**：
- 销售 A 把客户转移给销售 B，系统权限不变，但 CRM 权限需要更新
- 销售总监需要查看所有客户，但不是所有系统功能
- 某个大客户需要多人协作，每个人有不同的权限级别

## 小结

| 维度 | 说明 |
|------|------|
| **定位** | 销售过程管理，从线索到回款的全生命周期 |
| **核心实体** | 线索、客户、联系人、商机、合同、回款 |
| **关键机制** | 客户池（公海/私海）、数据权限、跟进记录 |
| **适用场景** | B2B 电商、大客户运营、供应商管理、招商管理 |
| **集成点** | System（权限）、Pay（收款）、BPM（审批）、ERP（订单） |

CRM 模块的核心价值在于：**把销售过程从"人治"变成"机制治"**。销售凭个人能力的时代过去了，现在靠的是系统化的流程管理。
