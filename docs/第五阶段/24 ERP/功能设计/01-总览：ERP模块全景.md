# 01 - 总览：ERP 模块全景

## 场景引入

你做了一个电商系统，商品卖得不错，但问题来了：

- 采购部门问"我们这个月进了多少货？从哪些供应商进的？"
- 销售部门问"这个月出了多少单？回款多少？"
- 仓库管理员问"库存还够不够？哪些商品快缺货了？"
- 财务问"这个月应收多少？应付多少？利润多少？"

**ERP（Enterprise Resource Planning，企业资源计划）** 就是为了解决这些问题而生的。它不是一个独立的系统，而是一套**企业内部资源流转的管理框架**。

### 为什么电商系统需要 ERP？

很多人以为 ERP 只适合制造业。但电商同样需要管理：

- **采购**：从供应商处进货，需要管理采购订单、入库、退货
- **销售**：向客户出货，需要管理销售订单、出库、退货
- **库存**：实时掌握库存数量，避免超卖或积压
- **财务**：应收应付、收付款管理、账户余额

核心价值是：**让企业内部的"进、销、存、财"四条线打通，数据不再孤岛**。

## 四大子模块

### 1. 采购管理

采购管理覆盖从"找供应商"到"货入库"的全流程：

```
采购流程
  │
  ├── 供应商管理 (ErpSupplierDO)
  │     └── 维护供应商信息：名称、联系方式、税率、银行信息
  │
  ├── 采购订单 (ErpPurchaseOrderDO)
  │     └── 向供应商下单：产品、数量、价格、税率
  │
  ├── 采购入库 (ErpPurchaseInDO)
  │     └── 供应商发货后入库：关联采购订单
  │
  └── 采购退货 (ErpPurchaseReturnDO)
        └── 质量问题退货：关联采购入库单
```

### 2. 销售管理

销售管理覆盖从"接客户订单"到"货出库"的全流程：

```
销售流程
  │
  ├── 客户管理 (ErpCustomerDO)
  │     └── 维护客户信息：名称、联系方式、税率、银行信息
  │
  ├── 销售订单 (ErpSaleOrderDO)
  │     └── 接受客户订单：产品、数量、价格、税率
  │
  ├── 销售出库 (ErpSaleOutDO)
  │     └── 发货出库：关联销售订单
  │
  └── 销售退货 (ErpSaleReturnDO)
        └── 客户退货：关联销售出库单
```

### 3. 库存管理

库存管理是 ERP 的核心枢纽——采购入库增加库存，销售出库减少库存：

```
库存变动
  │
  ├── 采购入库  →  库存增加
  ├── 采购退货  →  库存减少
  ├── 销售出库  →  库存减少
  ├── 销售退货  →  库存增加
  ├── 库存盘点  →  库存调整
  └── 库存调拨  →  仓库间转移

库存实体 (ErpStockDO)
  ├── productId    - 产品编号
  ├── warehouseId  - 仓库编号
  └── count        - 库存数量
```

### 4. 财务管理

财务管理跟踪企业的资金流动：

```
财务流程
  │
  ├── 账户管理 (ErpAccountDO)
  │     └── 管理银行账户、现金账户
  │
  ├── 收款单 (ErpFinanceReceiptDO)
  │     └── 销售回款：客户付钱给我们
  │
  └── 付款单 (ErpFinancePaymentDO)
        └── 采购付款：我们付钱给供应商
```

## 订单状态流转

采购订单和销售订单共享同一套审核状态机：

```
ErpAuditStatus（审核状态）
  │
  ├── 0 - 草稿 (DRAFT)
  │     → 新建订单的初始状态
  │
  ├── 10 - 审核中 (AUDITING)
  │     → 提交审批，等待审核
  │
  ├── 20 - 审核通过 (AUDIT_SUCCESS)
  │     → 审批通过，可以执行入库/出库
  │
  ├── 30 - 审核驳回 (AUDIT_REJECT)
  │     → 审批驳回，可以修改后重新提交
  │
  └── 40 - 已完成 (COMPLETED)
        → 全部入库/出库完成
```

### 采购订单状态流转

```
创建采购订单 (草稿)
  │
  ├── 提交审核 → 审核中
  │     ├── 通过 → 审核通过
  │     │     │
  │     │     ├── 创建采购入库单
  │     │     │     └── 入库完成 → 更新 inCount
  │     │     │
  │     │     ├── 创建采购退货单
  │     │     │     └── 退货完成 → 更新 returnCount
  │     │     │
  │     │     └── inCount + returnCount >= totalCount → 已完成
  │     │
  │     └── 驳回 → 审核驳回
  │           └── 修改后重新提交
  │
  └── 直接删除（草稿状态才能删除）
```

### 销售订单状态流转

```
创建销售订单 (草稿)
  │
  ├── 提交审核 → 审核中
  │     ├── 通过 → 审核通过
  │     │     │
  │     │     ├── 创建销售出库单
  │     │     │     └── 出库完成 → 更新 outCount
  │     │     │
  │     │     ├── 创建销售退货单
  │     │     │     └── 退货完成 → 更新 returnCount
  │     │     │
  │     │     └── outCount + returnCount >= totalCount → 已完成
  │     │
  │     └── 驳回 → 审核驳回
  │
  └── 直接删除
```

## 订单金额计算

### 采购订单金额

```java
// ErpPurchaseOrderDO 金额字段
BigDecimal totalCount;          // 合计数量
BigDecimal totalProductPrice;   // 合计产品价格 = SUM(明细数量 * 单价)
BigDecimal totalTaxPrice;       // 合计税额 = SUM(明细税额)
BigDecimal discountPercent;     // 优惠率
BigDecimal discountPrice;       // 优惠金额 = (totalProductPrice + totalTaxPrice) * discountPercent
BigDecimal totalPrice;          // 最终合计 = totalProductPrice + totalTaxPrice - discountPrice
BigDecimal depositPrice;        // 定金金额
```

### 订单明细金额

```java
// ErpPurchaseOrderItemDO 明细字段
Long productId;         // 产品编号
BigDecimal quantity;    // 数量
BigDecimal price;       // 单价
BigDecimal taxPercent;  // 税率
BigDecimal taxPrice;    // 税额 = price * quantity * taxPercent / 100
BigDecimal totalPrice;  // 小计 = price * quantity + taxPrice
```

## 电商场景举例

以一个电商公司的日常运营为例：

1. **采购**：运营发现"蓝牙耳机"库存不足，创建采购订单向供应商采购 500 个
2. **入库**：供应商发货后，仓库管理员创建采购入库单，库存从 50 增加到 550
3. **销售**：客户在小程序下单购买 2 个蓝牙耳机，系统自动创建销售订单
4. **出库**：仓库管理员根据销售订单创建出库单，库存从 550 减少到 548
5. **收款**：财务确认客户已付款，创建收款单
6. **退货**：客户退了 1 个耳机，创建销售退货单，库存从 548 增加到 549

## 思考题

1. 为什么采购订单和销售订单共享同一套审核状态枚举 `ErpAuditStatus`？如果分开定义会有什么问题？
2. 库存表 `ErpStockDO` 用 `productId + warehouseId` 做联合唯一键。如果同一个产品放在不同仓库，库存怎么管理？
3. 采购入库单和销售出库单各自独立创建。如果想实现"销售订单自动触发出库"，应该怎么设计？
4. 财务收款单和付款单的金额单位是什么？为什么不用"分"而用"元"？
