# 02 - 核心设计：进销存与财务

## 场景引入

上一章我们看到了 ERP 的全景——采购、销售、库存、财务四大子模块。这一章深入几个核心设计问题：

1. **订单与明细的主子表设计**：为什么订单头和订单行要分两张表？
2. **库存变动机制**：入库、出库、退货如何影响库存？如何保证并发安全？
3. **审核状态机**：订单状态如何流转？审核通过后如何触发后续业务？
4. **财务收付款**：收付款单如何与采购/销售订单关联？

## 主子表设计：订单头 + 订单行

### 设计动机

一个采购订单包含多个产品。比如"向供应商 A 采购 100 个鼠标 + 200 个键盘"。怎么存储？

```
方案 A：单表设计
  └── 一行存一个订单，产品信息用 JSON 字段
      优点：查询简单
      缺点：无法对产品明细做聚合查询（"鼠标这个月采购了多少？"）

方案 B：主子表设计（ERP 采用的方案）
  └── 订单头表 + 订单明细表
      优点：明细可以独立查询、聚合、统计
      缺点：查询需要 JOIN
```

### 采购订单的主子表

```java
// ErpPurchaseOrderDO - 采购订单头
public class ErpPurchaseOrderDO extends BaseDO {
    private Long id;                    // 订单编号
    private String no;                  // 订单号
    private Integer status;             // 审核状态
    private Long supplierId;            // 供应商编号
    private Long accountId;             // 结算账户
    private LocalDateTime orderTime;    // 下单时间

    // 汇总金额
    private BigDecimal totalCount;          // 合计数量
    private BigDecimal totalProductPrice;   // 合计产品价格
    private BigDecimal totalTaxPrice;       // 合计税额
    private BigDecimal discountPercent;     // 优惠率
    private BigDecimal discountPrice;       // 优惠金额
    private BigDecimal totalPrice;          // 最终合计
    private BigDecimal depositPrice;        // 定金金额

    // 执行进度
    private BigDecimal inCount;         // 已入库数量
    private BigDecimal returnCount;     // 已退货数量
}

// ErpPurchaseOrderItemDO - 采购订单明细
public class ErpPurchaseOrderItemDO extends BaseDO {
    private Long id;                    // 明细编号
    private Long orderId;               // 订单编号（关联订单头）
    private Long productId;             // 产品编号
    private BigDecimal quantity;        // 数量
    private BigDecimal price;           // 单价
    private BigDecimal taxPercent;      // 税率
    private BigDecimal taxPrice;        // 税额
    private BigDecimal totalPrice;      // 小计
    private String remark;              // 备注
}
```

### 金额计算链路

```
订单明细层：
  taxPrice = price * quantity * taxPercent / 100
  totalPrice = price * quantity + taxPrice

订单头层：
  totalProductPrice = SUM(item.price * item.quantity)
  totalTaxPrice = SUM(item.taxPrice)
  discountPrice = (totalProductPrice + totalTaxPrice) * discountPercent / 100
  totalPrice = totalProductPrice + totalTaxPrice - discountPrice
```

### 销售订单的对称设计

销售订单和采购订单结构几乎对称：

```
采购订单                          销售订单
  │                                │
  ├── 供应商 (supplierId)          ├── 客户 (customerId)
  ├── 采购明细 (OrderItemDO)       ├── 销售明细 (OrderItemDO)
  ├── 入库跟踪 (inCount)           ├── 出库跟踪 (outCount)
  └── 退货跟踪 (returnCount)       └── 退货跟踪 (returnCount)
```

**为什么设计成对称的？**

因为采购和销售是镜像关系：
- 采购是"从外部进来的"，对应入库
- 销售是"向外部出去的"，对应出库

对称设计让两套流程的代码结构一致，降低理解和维护成本。

## 库存变动机制

### 库存表设计

```java
// ErpStockDO - 产品库存
public class ErpStockDO extends BaseDO {
    private Long id;            // 库存编号
    private Long productId;     // 产品编号
    private Long warehouseId;   // 仓库编号
    private BigDecimal count;   // 库存数量
}
```

**关键设计：`productId + warehouseId` 联合唯一。**

同一个产品在不同仓库有独立的库存记录。比如"蓝牙耳机"在 A 仓库有 100 个，在 B 仓库有 200 个。

### 库存变动来源

```
库存变动来源
  │
  ├── 采购入库 (ErpPurchaseInDO)
  │     → 入库完成 → ErpStockDO.count += 入库数量
  │
  ├── 采购退货 (ErpPurchaseReturnDO)
  │     → 退货完成 → ErpStockDO.count -= 退货数量
  │
  ├── 销售出库 (ErpSaleOutDO)
  │     → 出库完成 → ErpStockDO.count -= 出库数量
  │
  ├── 销售退货 (ErpSaleReturnDO)
  │     → 退货完成 → ErpStockDO.count += 退货数量
  │
  ├── 库存盘点 (ErpStockCheckDO)
  │     → 盘点完成 → ErpStockDO.count = 盘点数量（直接覆盖）
  │
  └── 库存调拨 (ErpStockMoveDO)
        → 调拨完成 → 源仓库 count -= 数量，目标仓库 count += 数量
```

### 库存变动记录

每次库存变动都会记录一条明细，用于追溯：

```
ErpStockInDO (入库记录)
  ├── id            - 编号
  ├── productId     - 产品编号
  ├── warehouseId   - 仓库编号
  ├── quantity      - 入库数量
  ├── sourceType    - 来源类型（采购入库/销售退货）
  └── sourceId      - 来源单据编号
```

### 并发安全

多个操作同时修改库存怎么办？

```
场景：采购入库 100 个 + 销售出库 50 个 同时执行

方案：数据库乐观锁 / 悲观锁
  │
  ├── 乐观锁：UPDATE erp_stock SET count = count + 100
  │     WHERE id = ? AND version = ?
  │     → 版本号不匹配则重试
  │
  └── 悲观锁：SELECT * FROM erp_stock WHERE id = ? FOR UPDATE
        → 其他事务等待当前事务完成
```

在 ERP 场景中，通常使用数据库的 `UPDATE ... SET count = count + ?` 语句，利用数据库的行锁保证原子性。

## 审核状态机

### 统一审核状态

```java
// ErpAuditStatus - 审核状态枚举
public enum ErpAuditStatus {
    DRAFT(0, "草稿"),
    AUDITING(10, "审核中"),
    AUDIT_SUCCESS(20, "审核通过"),
    AUDIT_REJECT(30, "审核驳回"),
    COMPLETED(40, "已完成");
}
```

### 状态流转规则

```
                ┌──────────────────────────────────────┐
                │                                      │
                ▼                                      │
  ┌──────────┐  提交  ┌──────────┐  通过  ┌──────────┐ │
  │  草稿(0)  │──────▶│ 审核中(10)│──────▶│审核通过(20)│ │
  └──────────┘       └──────────┘       └──────────┘ │
       │                  │                   │        │
       │ 删除             │ 驳回              │ 完成    │
       ▼                  ▼                   ▼        │
    [已删除]         ┌──────────┐       ┌──────────┐   │
                     │审核驳回(30)│       │已完成(40) │   │
                     └──────────┘       └──────────┘   │
                          │                             │
                          │ 修改后重新提交               │
                          └─────────────────────────────┘
```

### 审核通过后的联动

```
采购订单审核通过
  │
  ├── 1. 状态更新为 AUDIT_SUCCESS
  ├── 2. 可以创建采购入库单（关联此订单）
  └── 3. 可以创建采购退货单（关联此订单）

采购入库单完成
  │
  ├── 1. 更新 ErpStockDO.count（库存增加）
  ├── 2. 更新 ErpPurchaseOrderDO.inCount（已入库数量）
  └── 3. 检查是否全部入库
        ├── inCount + returnCount >= totalCount
        │     → 更新订单状态为 COMPLETED
        └── 否则 → 保持 AUDIT_SUCCESS
```

## 采购入库流程

### 流程图

```
采购入库流程
  │
  ├── 1. 选择已审核通过的采购订单
  │
  ├── 2. 创建采购入库单 (ErpPurchaseInDO)
  │     ├── 关联采购订单 (orderId)
  │     ├── 选择仓库 (warehouseId)
  │     └── 填写入库明细
  │           ├── 产品 1：入库 80 个（订单 100 个，分批入库）
  │           └── 产品 2：入库 200 个（全部入库）
  │
  ├── 3. 审核入库单
  │     ├── 通过 → 执行入库
  │     └── 驳回 → 修改后重新提交
  │
  └── 4. 入库完成
        ├── 更新库存：ErpStockDO.count += 入库数量
        ├── 更新订单：ErpPurchaseOrderDO.inCount += 入库数量
        └── 创建库存变动记录
```

### 入库单实体

```java
// ErpPurchaseInDO - 采购入库单
public class ErpPurchaseInDO extends BaseDO {
    private Long id;                // 入库单编号
    private String no;              // 入库单号
    private Integer status;         // 审核状态
    private Long orderId;           // 关联采购订单
    private Long warehouseId;       // 入库仓库
    private LocalDateTime inTime;   // 入库时间
    private BigDecimal totalCount;  // 合计数量
    private BigDecimal totalPrice;  // 合计金额
    private String remark;          // 备注
}
```

### 分批入库

一个采购订单可以分多次入库。比如订单 100 个鼠标，第一批入库 80 个，第二批入库 20 个：

```
采购订单：100 个鼠标
  │
  ├── 入库单 1：80 个 → inCount = 80
  │
  └── 入库单 2：20 个 → inCount = 100（全部完成）
```

## 销售出库流程

### 流程图

```
销售出库流程
  │
  ├── 1. 选择已审核通过的销售订单
  │
  ├── 2. 创建销售出库单 (ErpSaleOutDO)
  │     ├── 关联销售订单 (orderId)
  │     ├── 选择仓库 (warehouseId)
  │     └── 填写出库明细
  │
  ├── 3. 审核出库单
  │     ├── 通过 → 执行出库
  │     └── 驳回 → 修改后重新提交
  │
  └── 4. 出库完成
        ├── 更新库存：ErpStockDO.count -= 出库数量
        ├── 更新订单：ErpSaleOrderDO.outCount += 出库数量
        └── 创建库存变动记录
```

### 出库前的库存校验

```
创建出库单时
  │
  ▼
校验库存
  │
  ├── 查询 ErpStockDO（productId, warehouseId）
  ├── 当前库存 >= 出库数量？
  │     ├── 是 → 允许出库
  │     └── 否 → 提示"库存不足，当前库存 X，需要 Y"
  └── 注意：这里只是预校验，实际扣减在审核通过后
```
