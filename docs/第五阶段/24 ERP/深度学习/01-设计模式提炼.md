# ERP 模块设计模式提炼

## 一、主从表设计模式（采购单+明细）

### 1.1 什么是主从表设计

ERP 系统中，几乎所有的业务单据都采用**主从表**结构。以采购单为例：主表记录采购单的基本信息（供应商、日期、总金额），从表记录具体的采购明细（商品、数量、单价）。

### 1.2 设计模式：事务性主从操作

项目采用**事务性主从操作**模式，在一个事务内完成主表和从表的创建/更新。

**源码位置**：`yudao-module-erp-server/.../service/purchase/ErpPurchaseOrderServiceImpl.java`

```java
@Service
@Validated
public class ErpPurchaseOrderServiceImpl implements ErpPurchaseOrderService {

    @Resource
    private ErpPurchaseOrderMapper purchaseOrderMapper;
    @Resource
    private ErpPurchaseOrderItemMapper purchaseOrderItemMapper;

    @Resource
    private ErpNoRedisDAO noRedisDAO;

    @Override
    @Transactional(rollbackFor = Exception.class)
    public Long createPurchaseOrder(ErpPurchaseOrderSaveReqVO createReqVO) {
        // 1.1 校验订单项的有效性
        List<ErpPurchaseOrderItemDO> purchaseOrderItems = validatePurchaseOrderItems(createReqVO.getItems());
        // 1.2 校验供应商
        supplierService.validateSupplier(createReqVO.getSupplierId());
        // 1.3 校验结算账户
        if (createReqVO.getAccountId() != null) {
            accountService.validateAccount(createReqVO.getAccountId());
        }
        // 1.4 生成订单号，并校验唯一性
        String no = noRedisDAO.generate(ErpNoRedisDAO.PURCHASE_ORDER_NO_PREFIX);
        if (purchaseOrderMapper.selectByNo(no) != null) {
            throw exception(PURCHASE_ORDER_NO_EXISTS);
        }

        // 2.1 插入订单
        ErpPurchaseOrderDO purchaseOrder = BeanUtils.toBean(createReqVO, ErpPurchaseOrderDO.class, in -> in
                .setNo(no).setStatus(ErpAuditStatus.PROCESS.getStatus()));
        calculateTotalPrice(purchaseOrder, purchaseOrderItems);
        purchaseOrderMapper.insert(purchaseOrder);

        // 2.2 插入订单项
        purchaseOrderItems.forEach(o -> o.setOrderId(purchaseOrder.getId()));
        purchaseOrderItemMapper.insertBatch(purchaseOrderItems);
        return purchaseOrder.getId();
    }
}
```

项目使用 Redis 实现**分布式唯一订单号**生成：

### 1.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 事务性操作 | 主表和从表必须在同一个事务中，保证数据一致性 |
| 批量插入 | `insertBatch` 提升从表插入性能 |
| 订单号唯一校验 | Redis 生成 + 数据库唯一索引双重保障 |
| 金额自动计算 | 总金额由明细汇总，避免人工计算错误 |

**迁移建议**：任何有"主单-明细"关系的业务都可以使用此模式。例如：销售订单、发货单、发票、报销单等。

## 二、库存变动的流水记录模式

### 2.1 什么是库存流水

库存流水是记录每一次库存变动的明细账。想象银行账户：余额是当前状态，流水是每一笔交易记录。库存流水的作用：

- **可追溯**：任何库存变动都能找到原因
- **可审计**：支持盘点、对账
- **可回滚**：出错时可以找到问题根源

### 2.2 设计模式：事件溯源 + 最终状态

项目采用**事件溯源**模式，每次库存变动都创建一条流水记录，同时更新库存的最终状态。

**源码位置**：`yudao-module-erp-server/.../service/stock/ErpStockRecordServiceImpl.java`

```java
@Service
@Validated
public class ErpStockRecordServiceImpl implements ErpStockRecordService {

    @Resource
    private ErpStockRecordMapper stockRecordMapper;

    @Resource
    private ErpStockService stockService;

    @Override
    @Transactional(rollbackFor = Exception.class)
    public void createStockRecord(ErpStockRecordCreateReqBO createReqBO) {
        // 1. 更新库存
        BigDecimal totalCount = stockService.updateStockCountIncrement(
                createReqBO.getProductId(), createReqBO.getWarehouseId(), createReqBO.getCount());

        // 2. 创建库存明细
        ErpStockRecordDO stockRecord = BeanUtils.toBean(createReqBO, ErpStockRecordDO.class)
                .setTotalCount(totalCount);
        stockRecordMapper.insert(stockRecord);
    }
}
```

### 2.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 流水 + 最终状态 | 流水记录历史，最终状态方便查询 |
| 业务类型枚举 | 明确库存变动来源，支持多种业务场景 |
| 事务性操作 | 更新库存和创建流水必须在同一事务 |
| 总量快照 | 每条流水记录当时的总量，方便对账 |

**迁移建议**：任何需要"余额 + 流水"的场景都可以使用此模式。例如：账户余额、积分变动、库存管理、工分系统。

## 三、财务凭证的双分录设计

### 3.1 什么是双分录

双分录是会计的基本原则：每笔交易都要同时记录"借方"和"贷方"，且借贷必须相等。ERP 系统中的财务凭证必须遵循这一原则。

### 3.2 设计模式：借贷平衡校验

项目通过**服务层校验**确保借贷平衡。虽然当前 ERP 模块的财务功能相对简化，但核心的收付款记录已经体现了双分录思想。

**源码位置**：`yudao-module-erp-server/.../service/finance/ErpFinancePaymentServiceImpl.java`

```java
@Service
@Validated
public class ErpFinancePaymentServiceImpl implements ErpFinancePaymentService {

    @Override
    @Transactional(rollbackFor = Exception.class)
    public Long createFinancePayment(ErpFinancePaymentSaveReqVO createReqVO) {
        // 1. 校验供应商
        supplierService.validateSupplier(createReqVO.getSupplierId());
        // 2. 校验结算账户
        accountService.validateAccount(createReqVO.getAccountId());

        // 3. 创建付款单
        ErpFinancePaymentDO payment = BeanUtils.toBean(createReqVO, ErpFinancePaymentDO.class);
        financePaymentMapper.insert(payment);

        // 4. 更新账户余额（借方：减少现金，贷方：增加应付）
        accountService.updateAccountBalance(createReqVO.getAccountId(),
            createReqVO.getTotalPrice().negate());

        return payment.getId();
    }
}
```

### 3.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 借贷平衡 | 会计基本原则，保证账目准确 |
| 审核流程 | 财务单据必须经过审核，防止错误 |
| 账户联动 | 收付款自动更新账户余额 |
| 事务性操作 | 凭证创建和余额更新必须在同一事务 |

**迁移建议**：任何需要"记账"的场景都可以使用双分录思想。例如：积分系统、工分系统、虚拟货币系统。

### 4.2 核心设计模式总结

| 模式 | 应用场景 | 核心类 |
|------|---------|--------|
| **主从表** | 采购单、销售单、入库单 | ErpPurchaseOrderService |
| **事件溯源** | 库存流水、账户流水 | ErpStockRecordService |
| **双分录** | 财务凭证、收付款 | ErpFinancePaymentService |
| **状态机** | 单据审核流程 | ErpAuditStatus |
| **分布式ID** | 订单号生成 | ErpNoRedisDAO |

### 4.3 数据库设计规范

**命名规范**：

| 表类型 | 命名格式 | 示例 |
|--------|---------|------|
| 主表 | `erp_{模块}_{单据}` | `erp_purchase_order` |
| 从表 | `erp_{模块}_{单据}_item` | `erp_purchase_order_item` |
| 流水表 | `erp_{模块}_record` | `erp_stock_record` |
| 基础表 | `erp_{模块}` | `erp_product`、`erp_supplier` |

### 4.4 可迁移的设计思想

1. **主从表模式**：任何有"单据+明细"关系的业务都可以使用
2. **事件溯源模式**：任何需要"流水记录"的场景都可以使用
3. **双分录模式**：任何需要"借贷平衡"的记账场景都可以使用
4. **状态机模式**：任何有"审核流程"的业务都可以使用
