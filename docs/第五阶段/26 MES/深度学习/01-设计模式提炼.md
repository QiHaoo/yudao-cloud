# MES 模块设计模式提炼

## 一、工单状态机设计

### 1.1 什么是工单状态机

MES 系统中，生产工单有严格的生命周期：从创建到确认、执行、完成或取消。状态机模式确保状态转换的合法性和可追溯性。

```
┌─────────────────────────────────────────────────────────────────┐
│                      工单状态机                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                    ┌─────────────┐                              │
│                    │   创建工单   │                              │
│                    └──────┬──────┘                              │
│                           │                                     │
│                           ▼                                     │
│                    ┌─────────────┐                              │
│              ┌─────│   草稿(0)   │─────┐                       │
│              │     └──────┬──────┘     │                       │
│              │            │            │                       │
│              │            ▼            │                       │
│              │     ┌─────────────┐     │                       │
│              │     │  已确认(1)  │     │                       │
│              │     └──────┬──────┘     │                       │
│              │            │            │                       │
│              │            ▼            │                       │
│              │     ┌─────────────┐     │                       │
│              │     │  已完成(2)  │     │                       │
│              │     └─────────────┘     │                       │
│              │                         │                       │
│              │     ┌─────────────┐     │                       │
│              └────>│  已取消(3)  │<────┘                       │
│                    └─────────────┘                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 设计模式：枚举状态机

项目采用**枚举 + 状态转换方法**实现状态机，通过 Service 层方法控制状态转换。

**源码位置**：`yudao-module-mes-api/.../enums/pro/MesProWorkOrderStatusEnum.java`

```java
/**
 * MES 生产工单状态枚举
 */
@Getter
@AllArgsConstructor
public enum MesProWorkOrderStatusEnum implements ArrayValuable<Integer> {

    /**
     * 草稿
     *
     * 对应 MesProWorkOrderService#createWorkOrder 方法
     */
    PREPARE(0, "草稿"),

    /**
     * 已确认
     *
     * 对应 MesProWorkOrderService#confirmWorkOrder 方法
     */
    CONFIRMED(1, "已确认"),

    /**
     * 已完成
     *
     * 对应 MesProWorkOrderService#finishWorkOrder 方法
     */
    FINISHED(2, "已完成"),

    /**
     * 已取消
     *
     * 对应 MesProWorkOrderService#cancelWorkOrder 方法
     */
    CANCELED(3, "已取消");

    public static final Integer[] ARRAYS = Arrays.stream(values())
        .map(MesProWorkOrderStatusEnum::getStatus).toArray(Integer[]::new);

    /**
     * 状态值
     */
    private final Integer status;
    /**
     * 状态名
     */
    private final String name;

    @Override
    public Integer[] array() {
        return ARRAYS;
    }
}
```

### 1.3 状态转换的业务逻辑

状态转换通常在 Service 层实现，每个状态转换对应一个业务方法：

```java
// 确认工单：草稿 -> 已确认
public void confirmWorkOrder(Long id) {
    // 1. 校验当前状态
    MesProWorkOrderDO workOrder = validateWorkOrderExists(id);
    if (!MesProWorkOrderStatusEnum.PREPARE.getStatus().equals(workOrder.getStatus())) {
        throw exception(WORK_ORDER_CONFIRM_FAIL_NOT_PREPARE);
    }
    // 2. 更新状态
    workOrderMapper.updateStatus(id, MesProWorkOrderStatusEnum.CONFIRMED.getStatus());
    // 3. 触发后续业务（如生成生产任务）
    generateProductionTasks(workOrder);
}

// 完成工单：已确认 -> 已完成
public void finishWorkOrder(Long id) {
    // 1. 校验当前状态
    MesProWorkOrderDO workOrder = validateWorkOrderExists(id);
    if (!MesProWorkOrderStatusEnum.CONFIRMED.getStatus().equals(workOrder.getStatus())) {
        throw exception(WORK_ORDER_FINISH_FAIL_NOT_CONFIRMED);
    }
    // 2. 更新状态
    workOrderMapper.updateStatus(id, MesProWorkOrderStatusEnum.FINISHED.getStatus());
}
```

### 1.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 枚举定义状态 | 类型安全，编译期检查 |
| Service 层控制转换 | 业务逻辑集中，便于维护 |
| 状态校验前置 | 防止非法状态转换 |
| 状态与业务联动 | 状态转换触发后续业务 |

**迁移建议**：任何有"状态流转"的业务都可以使用此模式。例如：订单状态、工单状态、审批流程、用户状态。

## 二、BOM（物料清单）的树形结构

### 2.1 什么是 BOM

BOM（Bill of Materials）是制造业的核心数据结构。一个产品由多个组件组成，每个组件又由多个子组件组成，形成树形结构。

```
┌─────────────────────────────────────────────────────────────────┐
│                       BOM 树形结构                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                    ┌─────────────┐                              │
│                    │   成品A     │                              │
│                    │  (产品)     │                              │
│                    └──────┬──────┘                              │
│                           │                                     │
│              ┌────────────┼────────────┐                        │
│              │            │            │                        │
│              ▼            ▼            ▼                        │
│        ┌──────────┐ ┌──────────┐ ┌──────────┐                  │
│        │ 组件B    │ │ 组件C    │ │ 原料D    │                  │
│        │ (数量:2) │ │ (数量:1) │ │ (数量:5) │                  │
│        └────┬─────┘ └──────────┘ └──────────┘                  │
│             │                                                   │
│        ┌────┴────┐                                              │
│        │         │                                              │
│        ▼         ▼                                              │
│   ┌──────────┐ ┌──────────┐                                    │
│   │ 零件E    │ │ 零件F    │                                    │
│   │ (数量:3) │ │ (数量:2) │                                    │
│   └──────────┘ └──────────┘                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 设计模式：递归树形结构

项目采用**父节点引用**模式实现 BOM 的树形结构，通过递归查询构建完整的 BOM 树。

**数据库设计**：

```sql
-- BOM 主表
CREATE TABLE mes_md_bom (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100) COMMENT 'BOM名称',
    item_id BIGINT COMMENT '产品ID',
    status INT COMMENT '状态'
);

-- BOM 明细表（树形结构）
CREATE TABLE mes_md_bom_material (
    id BIGINT PRIMARY KEY,
    bom_id BIGINT COMMENT 'BOM ID',
    parent_id BIGINT COMMENT '父节点ID（NULL表示顶级）',
    item_id BIGINT COMMENT '物料ID',
    quantity DECIMAL COMMENT '用量',
    level INT COMMENT '层级'
);
```

### 2.3 树形结构的查询

```java
// 递归查询 BOM 树
public List<BomTreeNode> getBomTree(Long bomId) {
    // 1. 查询所有明细
    List<MesMdBomMaterialDO> materials = bomMaterialMapper.selectListByBomId(bomId);

    // 2. 构建树形结构
    Map<Long, List<MesMdBomMaterialDO>> parentMap = materials.stream()
        .collect(Collectors.groupingBy(MesMdBomMaterialDO::getParentId));

    // 3. 递归构建树
    return buildTree(null, parentMap);
}

private List<BomTreeNode> buildTree(Long parentId,
        Map<Long, List<MesMdBomMaterialDO>> parentMap) {
    List<MesMdBomMaterialDO> children = parentMap.get(parentId);
    if (CollUtil.isEmpty(children)) {
        return Collections.emptyList();
    }
    return convertList(children, child -> {
        BomTreeNode node = new BomTreeNode();
        node.setId(child.getId());
        node.setItemId(child.getItemId());
        node.setQuantity(child.getQuantity());
        node.setChildren(buildTree(child.getId(), parentMap));
        return node;
    });
}
```

### 2.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 父节点引用 | 简化数据库设计，支持无限层级 |
| 递归查询 | 灵活构建任意深度的树 |
| 层级字段 | 方便查询特定层级，优化性能 |
| 版本管理 | BOM 变更时创建新版本，保留历史 |

**迁移建议**：任何有"树形结构"的业务都可以使用此模式。例如：组织架构、商品分类、评论回复、文件目录。

## 三、报工与质检的数据流

### 3.1 什么是报工与质检

报工是生产人员记录生产数量和工时的过程，质检是检验产品质量的过程。两者结合形成完整的生产数据流。

```
┌─────────────────────────────────────────────────────────────────┐
│                      报工与质检数据流                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────┐                                              │
│   │  生产工单   │                                              │
│   └──────┬──────┘                                              │
│          │                                                     │
│          ▼                                                     │
│   ┌─────────────┐      ┌─────────────┐      ┌─────────────┐  │
│   │  生产任务   │ ───> │   报工记录   │ ───> │  工时统计   │  │
│   └──────┬──────┘      └─────────────┘      └─────────────┘  │
│          │                                                     │
│          ▼                                                     │
│   ┌─────────────┐      ┌─────────────┐      ┌─────────────┐  │
│   │  质检任务   │ ───> │  质检记录   │ ───> │  质量统计   │  │
│   └──────┬──────┘      └─────────────┘      └─────────────┘  │
│          │                                                     │
│          ▼                                                     │
│   ┌─────────────┐                                              │
│   │  入库/返工  │                                              │
│   └─────────────┘                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 设计模式：状态驱动的数据流

项目采用**状态驱动**模式，每个环节的状态变化驱动数据流向下一环节。

**报工状态枚举**：

```java
// yudao-module-mes-api/.../enums/pro/MesProFeedbackStatusEnum.java
@Getter
@AllArgsConstructor
public enum MesProFeedbackStatusEnum implements ArrayValuable<Integer> {

    PENDING(0, "待报工"),
    PROCESSING(1, "报工中"),
    COMPLETED(2, "已完成");

    private final Integer status;
    private final String name;
}
```

**质检状态枚举**：

```java
// yudao-module-mes-api/.../enums/qc/MesQcStatusEnum.java
@Getter
@AllArgsConstructor
public enum MesQcStatusEnum implements ArrayValuable<Integer> {

    PENDING(0, "待质检"),
    INSPECTING(1, "质检中"),
    PASSED(2, "合格"),
    REJECTED(3, "不合格");

    private final Integer status;
    private final String name;
}
```

### 3.3 质检结果处理

```java
// 质检结果枚举
public enum MesQcCheckResultEnum {
    PASS(0, "合格"),
    FAIL(1, "不合格"),
    REWORK(2, "返工");

    private final Integer result;
    private final String name;
}
```

质检结果决定后续处理：
- **合格**：产品入库
- **不合格**：报废或降级处理
- **返工**：返回生产环节重新加工

### 3.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 状态驱动 | 每个环节的状态变化触发下一步操作 |
| 数据关联 | 报工关联生产任务，质检关联报工记录 |
| 结果分支 | 质检结果决定不同的后续流程 |
| 统计汇总 | 报工和质检数据支持多维度统计 |

**迁移建议**：任何有"多环节流程"的业务都可以使用此模式。例如：审批流程、订单处理、客服工单。
