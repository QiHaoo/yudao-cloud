# 01 - 起步：支付实体与渠道抽象

## 本章目标

从零搭建支付模块的数据层和渠道抽象层。完成后，你将拥有：

1. 支付订单、退款订单、支付扩展单等核心实体
2. 统一的 `PayClient` 支付渠道接口
3. `PayClientFactory` 工厂，根据渠道编码动态创建支付客户端
4. 微信、支付宝、钱包、Mock 等多种渠道的实现骨架

---

## 一、核心实体设计

### 1.1 PayAppDO -- 支付应用

支付应用是最顶层的概念。一个商户可能有多个应用（如"商城 App"和"商城小程序"），各自有独立的支付配置和回调地址。

```java
// 来源：yudao-module-pay-server/.../dal/dataobject/app/PayAppDO.java
@TableName("pay_app")
public class PayAppDO extends BaseDO {

    @TableId
    private Long id;
    private String appKey;      // 应用标识，业务系统通过此字段调用支付
    private String name;        // 应用名
    private Integer status;     // 状态（启用/禁用）
    private String remark;      // 备注
    private String orderNotifyUrl;    // 支付结果回调地址
    private String refundNotifyUrl;   // 退款结果回调地址
    private String transferNotifyUrl; // 转账结果回调地址
}
```

设计要点：
- `appKey` 是业务系统调用支付模块的凭证，而非 `id`。这样业务系统不需要关心支付模块内部的主键。
- 三个 `notifyUrl` 分别对应支付、退款、转账三种场景的回调地址，由业务系统在创建应用时配置。

### 1.2 PayChannelDO -- 支付渠道

一个应用下可以配置多个支付渠道（如微信 JSAPI、支付宝 PC 等），每个渠道存储了该支付方式的配置信息。

```java
// 来源：yudao-module-pay-server/.../dal/dataobject/channel/PayChannelDO.java
@TableName(value = "pay_channel", autoResultMap = true)
public class PayChannelDO extends TenantBaseDO {

    private Long id;
    private String code;       // 渠道编码，枚举 PayChannelEnum
    private Integer status;    // 状态（启用/禁用）
    private Double feeRate;    // 渠道费率，单位：百分比
    private String remark;     // 备注
    private Long appId;        // 关联的应用编号
    private PayClientConfig config; // 支付渠道配置（JSON 存储）
}
```

设计要点：
- `config` 字段使用 `PayClientConfig` 接口，不同渠道有不同的配置类（`WxPayClientConfig`、`AlipayPayClientConfig`），通过 Jackson 多态序列化存储到数据库。
- `feeRate` 记录渠道手续费率，在支付成功后计算实际手续费。

### 1.3 PayOrderDO -- 支付订单

支付订单是业务系统发起支付时创建的核心实体。

```java
// 来源：yudao-module-pay-server/.../dal/dataobject/order/PayOrderDO.java
@TableName("pay_order")
public class PayOrderDO extends BaseDO {

    private Long id;              // 订单编号
    private Long appId;           // 应用编号
    private Long channelId;       // 渠道编号（支付成功后回填）
    private String channelCode;   // 渠道编码（支付成功后回填）

    private Long userId;          // 用户编号
    private Integer userType;     // 用户类型

    // 商户相关
    private String merchantOrderId; // 商户订单编号（业务系统的订单号）
    private String subject;         // 商品标题
    private String body;            // 商品描述
    private String notifyUrl;       // 异步通知地址

    // 订单相关
    private Integer price;           // 支付金额，单位：分
    private Double channelFeeRate;   // 渠道手续费率
    private Integer channelFeePrice; // 渠道手续金额
    private Integer status;          // 支付状态，枚举 PayOrderStatusEnum
    private String userIp;           // 用户 IP
    private LocalDateTime expireTime;    // 订单失效时间
    private LocalDateTime successTime;   // 支付成功时间
    private Long extensionId;        // 支付成功的扩展单编号
    private String no;               // 支付成功的外部订单号

    // 退款相关
    private Integer refundPrice;     // 退款总金额

    // 渠道相关
    private String channelUserId;    // 渠道用户编号（如微信 openid）
    private String channelOrderNo;   // 渠道订单号
}
```

设计要点：
- `merchantOrderId` 是业务系统的订单号，通过 `appId + merchantOrderId` 可以唯一确定一笔支付单，避免重复创建。
- `status` 的状态流转：`WAITING(0) -> SUCCESS(10) -> REFUND(20)` 或 `WAITING(0) -> CLOSED(30)`。
- `channelId` 和 `channelCode` 在支付成功后才回填，因为用户可能在提交支付时选择了不同的渠道。

### 1.4 PayOrderExtensionDO -- 支付订单扩展

每次调用支付渠道都会生成一条扩展记录。一个支付订单可以对应多次渠道调用（比如第一次超时了，用户重新发起）。

```java
// 来源：yudao-module-pay-server/.../dal/dataobject/order/PayOrderExtensionDO.java
@TableName(value = "pay_order_extension", autoResultMap = true)
public class PayOrderExtensionDO extends BaseDO {

    private Long id;
    private String no;              // 外部订单号（调用渠道时使用）
    private Long orderId;           // 关联的支付订单编号
    private Long channelId;         // 渠道编号
    private String channelCode;     // 渠道编码
    private String userIp;          // 用户 IP
    private Integer status;         // 支付状态
    private Map<String, String> channelExtras; // 渠道额外参数
    private String channelErrorCode;    // 渠道错误码
    private String channelErrorMsg;     // 渠道错误信息
    private String channelNotifyData;   // 渠道通知内容
}
```

为什么需要扩展表？因为同一个支付订单可能被多次提交到不同的渠道。例如：
1. 用户第一次选择微信支付，但超时未完成
2. 用户第二次选择支付宝支付，成功了

此时 `PayOrderDO` 只记录最终成功的渠道，而 `PayOrderExtensionDO` 记录每一次的渠道调用。

## 二、支付渠道抽象

### 2.1 PayClient 接口

`PayClient` 是支付渠道的核心抽象接口，定义了支付、退款、转账三大类操作。

```java
// 来源：yudao-module-pay-server/.../framework/pay/core/client/PayClient.java
public interface PayClient<Config> {

    Long getId();           // 获得渠道编号
    Config getConfig();     // 获得渠道配置

    // 支付相关
    PayOrderRespDTO unifiedOrder(PayOrderUnifiedReqDTO reqDTO);           // 统一下单
    PayOrderRespDTO parseOrderNotify(Map<String, String> params, String body, Map<String, String> headers); // 解析支付回调
    PayOrderRespDTO getOrder(String outTradeNo);                           // 查询订单状态

    // 退款相关
    PayRefundRespDTO unifiedRefund(PayRefundUnifiedReqDTO reqDTO);        // 统一退款
    PayRefundRespDTO parseRefundNotify(Map<String, String> params, String body, Map<String, String> headers); // 解析退款回调
    PayRefundRespDTO getRefund(String outTradeNo, String outRefundNo);     // 查询退款状态

    // 转账相关
    PayTransferRespDTO unifiedTransfer(PayTransferUnifiedReqDTO reqDTO);  // 统一转账
    PayTransferRespDTO getTransfer(String outTradeNo);                     // 查询转账状态
    PayTransferRespDTO parseTransferNotify(Map<String, String> params, String body, Map<String, String> headers); // 解析转账回调
}
```

设计要点：
- 使用泛型 `Config` 约束配置类型，确保每个实现类都有对应的配置类。
- 统一的入参/出参 DTO，屏蔽不同渠道的参数差异。

### 2.2 AbstractPayClient -- 模板方法抽象类

`AbstractPayClient` 使用模板方法模式，在 `unifiedOrder()` 等方法中统一处理参数校验和异常捕获，子类只需实现 `doUnifiedOrder()` 等具体逻辑。

```java
// 来源：yudao-module-pay-server/.../framework/pay/core/client/impl/AbstractPayClient.java
@Slf4j
public abstract class AbstractPayClient<Config extends PayClientConfig> implements PayClient<Config> {

    private final Long channelId;
    private final String channelCode;
    protected Config config;

    public AbstractPayClient(Long channelId, String channelCode, Config config) {
        this.channelId = channelId;
        this.channelCode = channelCode;
        this.config = config;
    }

    // 初始化（子类实现）
    public final void init() {
        doInit();
        log.debug("[init][客户端({}) 初始化完成]", getId());
    }
    protected abstract void doInit();

    // 配置变更时刷新
    public final void refresh(Config config) {
        if (config.equals(this.config)) {
            return;
        }
        log.info("[refresh][客户端({})发生变化，重新初始化]", getId());
        this.config = config;
        this.init();
    }

    // 模板方法：统一异常处理
    @Override
    public final PayOrderRespDTO unifiedOrder(PayOrderUnifiedReqDTO reqDTO) {
        ValidationUtils.validate(reqDTO);
        PayOrderRespDTO resp;
        try {
            resp = doUnifiedOrder(reqDTO);
        } catch (ServiceException ex) {
            throw ex;
        } catch (Throwable ex) {
            log.error("[unifiedOrder][客户端({}) request({}) 发起支付异常]",
                    getId(), toJsonString(reqDTO), ex);
            throw buildPayException(ex);
        }
        return resp;
    }

    // 子类实现具体的渠道调用
    protected abstract PayOrderRespDTO doUnifiedOrder(PayOrderUnifiedReqDTO reqDTO) throws Throwable;

    // ... 其他方法类似
}
```

模板方法模式的好处：
1. 子类不需要重复写参数校验、异常捕获、日志记录的代码
2. 统一的异常处理保证了错误信息的一致性
3. `init()` 和 `refresh()` 方法支持配置热更新

## 三、PayClientFactory -- 支付客户端工厂

### 3.1 工厂接口

```java
// 来源：yudao-module-pay-server/.../framework/pay/core/client/PayClientFactory.java
public interface PayClientFactory {

    // 获得支付客户端
    PayClient getPayClient(Long channelId);

    // 创建或更新支付客户端
    <Config extends PayClientConfig> PayClient createOrUpdatePayClient(
            Long channelId, String channelCode, Config config);
}
```

### 3.2 工厂实现

```java
// 来源：yudao-module-pay-server/.../framework/pay/core/client/impl/PayClientFactoryImpl.java
@Slf4j
public class PayClientFactoryImpl implements PayClientFactory {

    // 支付客户端缓存，key 是渠道编号
    private final ConcurrentMap<Long, AbstractPayClient<?>> clients = new ConcurrentHashMap<>();

    // 渠道编码 -> 客户端 Class 的映射
    private final Map<PayChannelEnum, Class<? extends PayClient<?>>> clientClass = new ConcurrentHashMap<>();

    public PayClientFactoryImpl() {
        // 微信支付客户端
        clientClass.put(WX_PUB, WxPubPayClient.class);
        clientClass.put(WX_LITE, WxLitePayClient.class);
        clientClass.put(WX_APP, WxAppPayClient.class);
        clientClass.put(WX_BAR, WxBarPayClient.class);
        clientClass.put(WX_NATIVE, WxNativePayClient.class);
        clientClass.put(WX_WAP, WxWapPayClient.class);
        // 支付宝支付客户端
        clientClass.put(ALIPAY_WAP, AlipayWapPayClient.class);
        clientClass.put(ALIPAY_QR, AlipayQrPayClient.class);
        clientClass.put(ALIPAY_APP, AlipayAppPayClient.class);
        clientClass.put(ALIPAY_PC, AlipayPcPayClient.class);
        clientClass.put(ALIPAY_BAR, AlipayBarPayClient.class);
        // 钱包支付客户端
        clientClass.put(WALLET, WalletPayClient.class);
        // Mock 支付客户端
        clientClass.put(MOCK, MockPayClient.class);
    }

    @Override
    public PayClient getPayClient(Long channelId) {
        AbstractPayClient<?> client = clients.get(channelId);
        if (client == null) {
            log.error("[getPayClient][渠道编号({}) 找不到客户端]", channelId);
        }
        return client;
    }

    @Override
    public <Config extends PayClientConfig> PayClient createOrUpdatePayClient(
            Long channelId, String channelCode, Config config) {
        AbstractPayClient<Config> client = (AbstractPayClient<Config>) clients.get(channelId);
        if (client == null) {
            // 不存在则创建
            client = this.createPayClient(channelId, channelCode, config);
            client.init();
            clients.put(client.getId(), client);
        } else {
            // 已存在则刷新配置
            client.refresh(config);
        }
        return client;
    }

    private <Config extends PayClientConfig> AbstractPayClient<Config> createPayClient(
            Long channelId, String channelCode, Config config) {
        PayChannelEnum channelEnum = PayChannelEnum.getByCode(channelCode);
        Assert.notNull(channelEnum, String.format("支付渠道(%s) 为空", channelCode));
        Class<?> payClientClass = clientClass.get(channelEnum);
        Assert.notNull(payClientClass, String.format("支付渠道(%s) Class 为空", channelCode));
        return (AbstractPayClient<Config>) ReflectUtil.newInstance(payClientClass, channelId, config);
    }
}
```

设计要点：
- 使用 `ConcurrentHashMap` 缓存已创建的客户端实例，避免重复创建。
- `createOrUpdatePayClient` 方法支持创建新客户端和更新已有客户端的配置（热更新）。
- 通过反射 `ReflectUtil.newInstance()` 动态创建客户端实例，实现了工厂与具体实现类的解耦。

## 四、统一请求/响应 DTO

### 4.1 统一下单请求

```java
// 来源：yudao-module-pay-server/.../framework/pay/core/client/dto/order/PayOrderUnifiedReqDTO.java
@Data
public class PayOrderUnifiedReqDTO {
    @NotEmpty(message = "用户 IP 不能为空")
    private String userIp;

    @NotEmpty(message = "外部订单编号不能为空")
    private String outTradeNo;      // 外部订单号
    @NotEmpty(message = "商品标题不能为空")
    private String subject;          // 商品标题
    private String body;             // 商品描述
    @NotEmpty(message = "支付结果的回调地址不能为空")
    private String notifyUrl;        // 回调地址
    private String returnUrl;        // 同步跳转地址

    @NotNull(message = "支付金额不能为空")
    @DecimalMin(value = "0", inclusive = false)
    private Integer price;           // 支付金额，单位：分
    @NotNull(message = "支付过期时间不能为空")
    private LocalDateTime expireTime; // 过期时间

    private Map<String, String> channelExtras; // 渠道额外参数
    private String displayMode;      // 展示模式
}
```

设计要点：
- `outTradeNo` 使用 `PayOrderExtensionDO.no` 而非 `PayOrderDO.id`，因为每次渠道调用都需要唯一的外部订单号。
- `channelExtras` 用于传递渠道特有的参数（如微信公众号的 `openid`）。
- `displayMode` 控制支付结果的展示方式（如二维码、跳转链接等）。
