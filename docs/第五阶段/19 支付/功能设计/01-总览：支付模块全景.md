# 01 - 总览：支付模块全景

## 场景引入

假设你在做一个电商系统，用户下单后需要支付。你需要对接支付宝和微信支付，让用户可以选择其中一种方式完成付款。

听起来很简单，但真正做起来，你会发现一堆问题：

- **多渠道对接**：支付宝有 PC 网站支付、手机网站支付、App 支付、扫码支付、条码支付；微信有 JSAPI 支付、小程序支付、Native 支付、App 支付、H5 支付、付款码支付。每种支付方式的 SDK 调用方式、参数格式、回调格式都不一样。
- **支付回调**：支付宝和微信都会在支付成功后异步通知你，你需要验签、处理幂等、更新订单状态，然后通知业务系统。
- **退款**：退款不是简单的"把钱退回去"，你需要调用渠道的退款接口，等待退款回调，处理部分退款、多次退款。
- **转账**：有些场景需要给用户打款，比如提现、分账，这又是一套独立的流程。
- **对账**：你需要定期同步支付渠道的订单状态，处理那些回调丢失的情况。

如果每个业务模块都自己对接支付宝和微信，代码会重复得一塌糊涂。而且，当你需要新增一个支付渠道（比如银联云闪付）时，所有业务模块都要改。

**支付模块就是为了解决这个问题：把支付能力抽象成一个独立的模块，业务系统只需要调用统一的 API，不需要关心底层对接的是支付宝还是微信。**

---

## 支付模块的核心能力

支付模块提供五大核心能力：

| 能力 | 说明 | 场景举例 |
|------|------|---------|
| **支付下单** | 创建支付订单，调用支付渠道，返回支付参数给前端 | 用户在电商系统下单，选择微信支付 |
| **支付回调** | 接收支付渠道的异步通知，更新订单状态 | 微信支付成功后，通知你的服务器 |
| **退款** | 创建退款订单，调用退款接口，处理退款回调 | 用户申请退款，电商系统调用支付模块发起退款 |
| **转账** | 给用户打款 | 用户提现到微信零钱 |
| **钱包** | 用户余额管理，支持充值和消费 | 用户充值到平台钱包，用钱包余额支付 |

### 核心概念关系

在深入代码之前，先理清几个核心概念的关系：

```
PayApp (支付应用)
  │
  ├── PayChannel (支付渠道) × N
  │     一个应用下可以配置多个支付渠道
  │     例如：微信 JSAPI、支付宝 PC、微信小程序
  │
  ├── PayOrder (支付订单) × N
  │     一个应用下的所有支付订单
  │
  ├── PayRefund (退款订单) × N
  │     一个应用下的所有退款订单
  │
  └── PayTransfer (转账订单) × N
        一个应用下的所有转账订单
```

- **PayApp（支付应用）**：最顶层的概念。一个商户可能有多个应用，比如"商城 App"和"商城小程序"是两个不同的应用，各自有独立的支付配置。
- **PayChannel（支付渠道）**：一个应用下的具体支付方式。每个渠道存储了该支付方式的配置信息（如 appId、密钥等）。
- **PayOrder（支付订单）**：业务系统发起支付时创建的订单。一个支付订单关联一个应用和一个渠道。
- **PayOrderExtension（支付订单扩展）**：每次调用支付渠道都会生成一条扩展记录。一个支付订单可以对应多次渠道调用（比如第一次超时了，用户重新发起）。
- **PayRefund（退款订单）**：一个支付订单可以有多次退款（部分退款场景）。
- **PayTransfer（转账订单）**：给用户打款的记录。

## 支付渠道抽象

支付模块最核心的设计是**支付渠道抽象**。不管底层是支付宝还是微信，对外暴露的接口都是一样的。

### 支付渠道枚举

项目定义了所有支持的支付渠道：

```
微信系：
  wx_pub     - 微信 JSAPI 支付（公众号网页）
  wx_lite    - 微信小程序支付
  wx_app     - 微信 App 支付
  wx_native  - 微信 Native 支付（扫码）
  wx_wap     - 微信 Wap 网站支付（H5）
  wx_bar     - 微信付款码支付

支付宝系：
  alipay_pc  - 支付宝 PC 网站支付
  alipay_wap - 支付宝 Wap 网站支付
  alipay_app - 支付宝 App 支付
  alipay_qr  - 支付宝扫码支付
  alipay_bar - 支付宝条码支付

其他：
  mock       - 模拟支付（开发测试用）
  wallet     - 钱包支付（平台余额）
```

### PayClient 接口

所有支付渠道都实现 `PayClient` 接口，它定义了三大类操作：

```
PayClient
  │
  ├── 支付相关
  │   ├── unifiedOrder()       // 统一下单
  │   ├── parseOrderNotify()   // 解析支付回调
  │   └── getOrder()           // 查询订单状态
  │
  ├── 退款相关
  │   ├── unifiedRefund()      // 统一退款
  │   ├── parseRefundNotify()  // 解析退款回调
  │   └── getRefund()          // 查询退款状态
  │
  └── 转账相关
      ├── unifiedTransfer()    // 统一转账
      ├── parseTransferNotify()// 解析转账回调
      └── getTransfer()        // 查询转账状态
```

这种设计的妙处在于：

1. **策略模式**：业务代码不需要知道具体是哪个支付渠道，只需要调用 `PayClient` 的统一方法。
2. **易于扩展**：新增支付渠道（比如银联云闪付），只需要新增一个 `PayClient` 实现类，不需要修改任何业务代码。
3. **配置驱动**：支付渠道的配置存储在数据库中，运行时通过 `PayClientFactory` 动态创建客户端实例。

### 类继承体系

```
PayClient (接口)
  │
  └── AbstractPayClient (抽象类，模板方法模式)
        │  统一处理参数校验、异常捕获、日志记录
        │
        ├── AbstractWxPayClient (微信支付基类)
        │     ├── WxPubPayClient    (JSAPI)
        │     ├── WxLitePayClient   (小程序)
        │     ├── WxAppPayClient    (App)
        │     ├── WxNativePayClient (Native)
        │     ├── WxWapPayClient    (H5)
        │     └── WxBarPayClient    (付款码)
        │
        ├── AbstractAlipayPayClient (支付宝基类)
        │     ├── AlipayPcPayClient  (PC)
        │     ├── AlipayWapPayClient (Wap)
        │     ├── AlipayAppPayClient (App)
        │     ├── AlipayQrPayClient  (扫码)
        │     └── AlipayBarPayClient (条码)
        │
        ├── WalletPayClient (钱包支付)
        └── MockPayClient   (模拟支付)
```

`AbstractPayClient` 使用了**模板方法模式**：它在 `unifiedOrder()` 中统一处理参数校验和异常捕获，然后调用子类的 `doUnifiedOrder()` 完成具体的渠道调用。这样每个子类只需要关注"如何调用渠道 SDK"这一件事。

## 支付订单状态机

支付订单有一个清晰的状态流转：

```
                    ┌──────────────────┐
                    │   WAITING (0)    │
                    │   待支付          │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
    ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
    │ SUCCESS (10)│  │ CLOSED (30) │  │  (超时关闭)   │
    │ 支付成功     │  │ 支付关闭     │  │             │
    └──────┬──────┘  └─────────────┘  └─────────────┘
           │
           ▼
    ┌─────────────┐
    │ REFUND (20) │
    │ 已退款       │
    └─────────────┘
```

- **WAITING → SUCCESS**：收到支付渠道的支付成功回调
- **WAITING → CLOSED**：订单超时未支付，或用户主动关闭
- **SUCCESS → REFUND**：退款成功后，状态变为已退款

注意：全部退款后状态是 REFUND（20），而不是 CLOSED（30）。CLOSED 专门表示"未支付就关闭了"。

## 一个支付请求的完整生命周期

以电商系统中用户选择微信小程序支付为例，看看一个支付请求从头到尾经历了什么：

```
用户 (微信小程序)
  │
  │  1. 点击"立即支付"
  ▼
电商订单服务 (TradeOrderService)
  │
  │  2. 通过 Feign 调用 PayOrderApi.createOrder()
  │     传入：appKey, merchantOrderId, subject, price, expireTime
  ▼
支付模块 - PayOrderService.createOrder()
  │
  │  3. 创建 PayOrderDO，状态为 WAITING
  │
  │  4. 返回支付订单编号
  ▼
电商订单服务
  │
  │  5. 调用 PayOrderApi (或 AppPayOrderController) submit()
  │     传入：支付订单编号, channelCode="wx_lite", channelExtras
  ▼
支付模块 - PayOrderService.submitOrder()
  │
  │  6. 创建 PayOrderExtensionDO（扩展记录）
  │
  │  7. 通过 PayClientFactory 获取 WxLitePayClient
  │
  │  8. 调用 WxLitePayClient.unifiedOrder()
  │     → 调用微信 SDK 的统一下单接口
  │     → 返回 prepay_id 等支付参数
  ▼
微信支付服务器
  │
  │  9. 返回支付参数（prepay_id, 签名等）
  ▼
支付模块
  │
  │  10. 将支付参数封装为 PayOrderSubmitRespVO 返回
  ▼
电商订单服务
  │
  │  11. 将支付参数返回给小程序
  ▼
用户 (微信小程序)
  │
  │  12. 调用 wx.requestPayment() 拉起微信支付
  ▼
微信支付服务器
  │
  │  13. 用户完成支付
  │
  │  14. 异步通知支付模块：POST /pay/notify/order/{channelId}
  ▼
支付模块 - PayNotifyController.notifyOrder()
  │
  │  15. 通过 PayClient.parseOrderNotify() 验签并解析回调数据
  │
  │  16. PayOrderService.notifyOrder() 更新订单状态为 SUCCESS
  │
  │  17. 创建 PayNotifyTaskDO，异步通知业务系统
  ▼
电商订单服务
  │
  │  18. 收到支付成功通知，更新订单状态为"已支付"
  ▼
用户看到"支付成功"
```

## 与业务模块的集成方式

支付模块通过两种方式与业务模块集成：

### 方式一：Feign RPC（推荐）

业务模块通过 `-api` 模块中的 Feign 客户端调用支付模块：

- `PayOrderApi` - 创建支付单、查询支付单
- `PayRefundApi` - 创建退款单、查询退款单
- `PayTransferApi` - 创建转账单、查询转账单

### 方式二：支付回调通知

支付模块在支付/退款/转账成功后，通过 `PayNotifyTask` 异步通知业务系统。通知地址配置在 `PayAppDO` 中：

- `orderNotifyUrl` - 支付成功通知地址
- `refundNotifyUrl` - 退款成功通知地址
- `transferNotifyUrl` - 转账成功通知地址

通知机制采用**指数退避重试**策略：首次通知后，如果业务系统没有返回成功，会按照 15s、15s、30s、3min、30min、30min、30min、1h 的间隔重试，最多 9 次。

## 钱包支付

除了对接外部支付渠道，支付模块还内置了**钱包支付**能力：

```
PayWalletDO (钱包)
  ├── userId        - 用户编号
  ├── userType      - 用户类型（会员/管理员）
  ├── balance       - 余额（单位：分）
  ├── freezePrice   - 冻结金额
  ├── totalExpense  - 累计支出
  └── totalRecharge - 累计充值
```

钱包支付的流程和外部支付渠道一样走 `PayClient` 接口，只是 `WalletPayClient` 直接操作本地数据库，不需要调用外部 API。

钱包还支持**充值套餐**（`PayWalletRechargePackageDO`），运营可以配置"充 100 送 10"之类的活动。
