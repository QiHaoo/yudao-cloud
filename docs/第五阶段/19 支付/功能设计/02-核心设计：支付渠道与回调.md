# 02 - 核心设计：支付渠道与回调

## 场景引入

上一章我们看到了支付模块的全景。这一章深入两个最核心的设计问题：

1. **支付渠道抽象**：如何用一套统一的接口对接支付宝、微信、银联等完全不同的支付 SDK？
2. **支付回调处理**：支付渠道的异步通知格式各不相同，如何统一验签、解析、幂等处理？

---

## 支付渠道抽象设计

### 问题域

每个支付渠道的 SDK 都有自己的调用方式：

```
微信支付：
  WxPayService.createOrder(WxPayUnifiedOrderRequest)  → WxPayUnifiedOrderResult

支付宝：
  AlipayClient.execute(AlipayTradePayRequest)          → AlipayTradePayResponse
```

参数格式不同、返回值不同、异常处理不同。如果业务代码直接调用这些 SDK，会和具体支付渠道强耦合。

### 设计目标

1. **统一接口**：业务代码不关心底层是哪个支付渠道
2. **易于扩展**：新增支付渠道只需要实现接口，不改业务代码
3. **配置驱动**：渠道配置存储在数据库中，运行时动态创建客户端
4. **统一异常处理**：不同渠道的异常都转换为统一的 `PayClientException`

### PayClient 接口设计

```java
public interface PayClient<Config> {

    Long getId();           // 渠道编号
    Config getConfig();     // 渠道配置

    // ========== 支付相关 ==========
    PayOrderRespDTO unifiedOrder(PayOrderUnifiedReqDTO reqDTO);
    PayOrderRespDTO parseOrderNotify(Map<String, String> params, String body, Map<String, String> headers);
    PayOrderRespDTO getOrder(String outTradeNo);

    // ========== 退款相关 ==========
    PayRefundRespDTO unifiedRefund(PayRefundUnifiedReqDTO reqDTO);
    PayRefundRespDTO parseRefundNotify(Map<String, String> params, String body, Map<String, String> headers);
    PayRefundRespDTO getRefund(String outTradeNo, String outRefundNo);

    // ========== 转账相关 ==========
    PayTransferRespDTO unifiedTransfer(PayTransferUnifiedReqDTO reqDTO);
    PayTransferRespDTO parseTransferNotify(Map<String, String> params, String body, Map<String, String> headers);
    PayTransferRespDTO getTransfer(String outTradeNo);
}
```

这个接口的设计有几个值得注意的地方：

**1. 泛型 `Config`**

每个支付渠道需要不同的配置。微信需要 appId、mchId、apiV3Key、证书路径；支付宝需要 appId、privateKey、alipayPublicKey。用泛型可以让每个实现类定义自己的配置类型，而不需要用 `Map<String, String>` 这种弱类型方式。

**2. 三组平行的方法**

支付、退款、转账三组方法结构完全对称：
- `unifiedXxx()` - 发起请求
- `parseXxxNotify()` - 解析回调
- `getXxx()` - 查询状态

这种对称性让代码结构非常清晰，新增一种业务类型（比如"分账"）时，也只需要按同样的模式增加一组方法。

**3. `parseXxxNotify` 的参数**

回调解析方法接收三个参数：`params`（URL 参数）、`body`（请求体）、`headers`（请求头）。这是因为不同支付渠道的回调格式不同：
- 支付宝：URL 参数 + 表单参数
- 微信：XML 请求体 或 JSON 请求体 + 签名在 Header 中

统一接收所有数据，让每个实现类自己决定怎么解析。

### AbstractPayClient 模板方法

`AbstractPayClient` 是一个典型的**模板方法模式**应用：

```java
public abstract class AbstractPayClient<Config extends PayClientConfig> implements PayClient<Config> {

    // 模板方法：统一参数校验 + 异常处理
    @Override
    public final PayOrderRespDTO unifiedOrder(PayOrderUnifiedReqDTO reqDTO) {
        ValidationUtils.validate(reqDTO);        // 1. 参数校验
        PayOrderRespDTO resp;
        try {
            resp = doUnifiedOrder(reqDTO);        // 2. 调用子类实现
        } catch (ServiceException ex) {
            throw ex;                             // 3a. 业务异常直接抛
        } catch (Throwable ex) {
            throw buildPayException(ex);          // 3b. 系统异常包装后抛
        }
        return resp;
    }

    // 抽象方法：子类实现具体的渠道调用
    protected abstract PayOrderRespDTO doUnifiedOrder(PayOrderUnifiedReqDTO reqDTO) throws Throwable;
}
```

这里用了 `final` 关键字修饰模板方法，防止子类绕过统一的异常处理逻辑。子类只需要实现 `doUnifiedOrder()`，专注于"如何调用渠道 SDK"。

### 继承体系

```
PayClient<Config>                              (接口)
  │
  └── AbstractPayClient<Config>                (抽象类，模板方法)
        │
        ├── AbstractWxPayClient               (微信支付基类)
        │     │  封装 WxPayService 的初始化和通用逻辑
        │     │
        │     ├── WxPubPayClient               (JSAPI 支付)
        │     ├── WxLitePayClient              (小程序支付)
        │     ├── WxAppPayClient               (App 支付)
        │     ├── WxNativePayClient            (Native 支付)
        │     ├── WxWapPayClient               (H5 支付)
        │     └── WxBarPayClient               (付款码支付)
        │
        ├── AbstractAlipayPayClient            (支付宝基类)
        │     │  封装 AlipayClient 的初始化和通用逻辑
        │     │
        │     ├── AlipayPcPayClient             (PC 网站支付)
        │     ├── AlipayWapPayClient            (Wap 网站支付)
        │     ├── AlipayAppPayClient            (App 支付)
        │     ├── AlipayQrPayClient             (扫码支付)
        │     └── AlipayBarPayClient            (条码支付)
        │
        ├── WalletPayClient                    (钱包支付，直接操作数据库)
        └── MockPayClient                      (模拟支付，开发测试用)
```

两层继承的设计：
- `AbstractWxPayClient` / `AbstractAlipayPayClient`：封装渠道 SDK 的初始化逻辑（证书加载、客户端创建等），让具体的支付方式实现类只关注"调用哪个 API"。
- 具体实现类：每种支付方式的 `doUnifiedOrder()` 实现不同，比如 JSAPI 支付需要传 openid，Native 支付需要传二维码回调 URL。

### PayClientFactory 工厂

客户端实例通过工厂创建和管理：

```java
public interface PayClientFactory {
    PayClient getPayClient(Long channelId);
    <Config extends PayClientConfig> PayClient createOrUpdatePayClient(
        Long channelId, String channelCode, Config config);
}
```

工厂的职责：
1. **创建客户端**：根据渠道编码（如 `wx_lite`）和配置，创建对应的 `PayClient` 实例
2. **缓存客户端**：同一个渠道的客户端只创建一次，后续从缓存获取
3. **热更新**：当渠道配置变更时，调用 `createOrUpdatePayClient` 更新客户端实例

这种设计让支付渠道的配置可以存储在数据库中，运营人员在后台修改配置后，客户端实例会自动刷新，不需要重启服务。

## 支付下单流程

### 两步式设计

支付下单被拆分为两个独立的步骤：

```
步骤一：创建支付订单 (createOrder)
  │
  │  业务系统调用 PayOrderApi.createOrder()
  │  传入：appKey, merchantOrderId, subject, price, expireTime
  │
  │  创建 PayOrderDO，状态为 WAITING
  │  返回：支付订单编号 (id)
  │
  ▼
步骤二：提交支付 (submitOrder)
  │
  │  前端/业务系统调用 AppPayOrderController.submit()
  │  传入：支付订单编号, channelCode, channelExtras
  │
  │  1. 创建 PayOrderExtensionDO
  │  2. 调用 PayClient.unifiedOrder()
  │  3. 返回支付参数（前端用于拉起支付）
  │
  ▼
```

**为什么要分成两步？**

1. **关注点分离**：创建订单是业务逻辑（需要业务系统参与），提交支付是技术逻辑（只需要前端选择支付方式）。
2. **灵活性**：同一个订单可以尝试不同的支付方式。比如用户先选了微信支付但没成功，可以切换到支付宝重新支付，而不需要重新创建订单。
3. **幂等性**：创建订单是幂等的（相同 merchantOrderId 不会重复创建），提交支付可以重试。

### 创建支付订单

业务系统通过 Feign 调用 `PayOrderApi.createOrder()`：

```java
// 请求参数
PayOrderCreateReqDTO {
    appKey: "商城应用",           // 应用标识
    merchantOrderId: "ORDER_001", // 业务系统的订单号
    subject: "iPhone 15 Pro",    // 商品标题
    body: "256GB 黑色钛金属",      // 商品描述
    price: 999900,               // 支付金额（单位：分）
    expireTime: "2024-01-01 12:00:00", // 过期时间
    userId: 1024,                // 用户编号
    userType: 2                  // 用户类型（会员）
}

// 返回
支付订单编号 (Long)
```

创建订单时会做以下事情：
1. 根据 `appKey` 查找 `PayAppDO`，获取应用编号
2. 校验相同应用下 `merchantOrderId` 是否已存在（幂等）
3. 创建 `PayOrderDO`，状态为 `WAITING(0)`
4. 返回支付订单编号

### 提交支付

前端拿到支付订单编号后，用户选择支付方式，调用提交接口：

```java
// 请求参数
PayOrderSubmitReqVO {
    id: 1024,                    // 支付订单编号
    channelCode: "wx_lite",      // 支付渠道编码
    channelExtras: {             // 渠道额外参数
        "openid": "oXXXX"       // 微信 JSAPI/小程序需要 openid
    }
}

// 返回
PayOrderSubmitRespVO {
    // 不同渠道返回不同的支付参数
    // 微信小程序：prepay_id, 签名等
    // 支付宝：支付表单 HTML 或 支付链接
    // Native：二维码 URL
}
```

提交支付的内部流程：

```
submitOrder()
  │
  ├── 1. 校验订单存在且状态为 WAITING
  │
  ├── 2. 创建 PayOrderExtensionDO（扩展记录）
  │     记录本次渠道调用的渠道、时间、状态
  │
  ├── 3. 通过 PayClientFactory 获取 PayClient
  │
  ├── 4. 调用 PayClient.unifiedOrder()
  │     │
  │     ├── 微信：调用 WxPayService.unifiedOrder()
  │     ├── 支付宝：调用 AlipayClient.execute()
  │     └── 钱包：直接扣减余额
  │
  ├── 5. 更新 PayOrderExtensionDO
  │     记录渠道返回的参数或错误信息
  │
  ├── 6. 更新 PayOrderDO
  │     设置 channelId, channelCode, channelUserId 等
  │
  └── 7. 返回支付参数给前端
```

## 支付回调处理

### 回调入口

所有支付渠道的回调都走同一个入口：

```
POST /pay/notify/order/{channelId}    - 支付回调
POST /pay/notify/refund/{channelId}   - 退款回调
POST /pay/notify/transfer/{channelId} - 转账回调
```

`channelId` 是渠道编号，用于找到对应的 `PayClient` 来验签和解析。

### 回调处理流程

```
支付渠道 (支付宝/微信)
  │
  │  POST /pay/notify/order/123
  │  Body: 支付结果数据
  ▼
PayNotifyController.notifyOrder()
  │
  ├── 1. 通过 channelId 获取 PayClient
  │
  ├── 2. 调用 PayClient.parseOrderNotify()
  │     │
  │     ├── 验签：验证请求确实来自支付渠道
  │     ├── 解析：提取订单号、支付状态、支付金额等
  │     └── 返回 PayOrderRespDTO
  │
  ├── 3. 调用 PayOrderService.notifyOrder()
  │     │
  │     ├── 幂等校验：订单已经是 SUCCESS 就不再处理
  │     ├── 更新 PayOrderDO 状态为 SUCCESS
  │     ├── 更新 PayOrderExtensionDO 的回调数据
  │     └── 创建 PayNotifyTaskDO（异步通知业务系统）
  │
  └── 4. 返回 "success" 给支付渠道
       (支付宝和微信都要求返回特定的成功字符串)
```

### 幂等性设计

支付回调的幂等处理至关重要。支付渠道可能会多次通知同一笔订单（网络超时重试等），如果每次都更新订单状态和通知业务系统，会导致重复处理。

幂等的关键点：

1. **订单状态校验**：在 `notifyOrder()` 中，首先检查订单状态。如果已经是 `SUCCESS`，直接返回，不再处理。

2. **扩展记录去重**：通过 `channelOrderNo`（渠道订单号）查找对应的 `PayOrderExtensionDO`，避免重复处理同一笔渠道回调。

3. **通知任务去重**：创建 `PayNotifyTaskDO` 时，检查是否已存在相同 `dataId` 和 `type` 的未完成任务。

## 退款流程

### 退款与支付的关系

```
PayOrderDO (支付订单)
  │
  ├── PayRefundDO (退款订单 1) - 部分退款 100 元
  ├── PayRefundDO (退款订单 2) - 部分退款 50 元
  └── PayRefundDO (退款订单 3) - 全额退款剩余金额
```

一个支付订单可以有多次退款（部分退款场景），每次退款都会创建一个 `PayRefundDO`。

### 退款流程

```
业务系统
  │
  │  PayRefundApi.createRefund()
  │  传入：支付订单编号, 退款金额, 退款原因, 商户退款单号
  ▼
PayRefundService.createRefund()
  │
  ├── 1. 校验支付订单存在且已支付
  ├── 2. 校验退款金额不超过可退金额
  │     可退金额 = 支付金额 - 已退款金额
  ├── 3. 创建 PayRefundDO，状态为 WAITING(0)
  ├── 4. 调用 PayClient.unifiedRefund()
  ├── 5. 更新退款订单状态
  └── 6. 创建 PayNotifyTaskDO（异步通知业务系统）
```

### 退款回调

退款回调的处理逻辑和支付回调类似：

```
POST /pay/notify/refund/{channelId}
  │
  ├── PayClient.parseRefundNotify() - 验签并解析
  ├── PayRefundService.notifyRefund() - 更新退款状态
  │     ├── 更新 PayRefundDO 状态为 SUCCESS
  │     ├── 更新 PayOrderDO 的 refundPrice
  │     └── 如果全部退款，更新 PayOrderDO 状态为 REFUND
  └── 创建 PayNotifyTaskDO - 通知业务系统
```

## 转账流程

转账是支付的反向操作——把钱打给用户。典型场景是用户提现。

### 转账单实体

```
PayTransferDO
  ├── id                  - 编号
  ├── no                  - 转账单号
  ├── appId               - 应用编号
  ├── channelId           - 渠道编号
  ├── channelCode         - 渠道编码
  ├── merchantTransferId  - 商户转账单号
  ├── subject             - 转账标题
  ├── price               - 转账金额（分）
  ├── userAccount         - 收款人账号
  ├── userName            - 收款人姓名
  ├── status              - 转账状态
  └── channelTransferNo   - 渠道转账单号
```

### 转账流程

```
业务系统
  │
  │  PayTransferApi.createTransfer()
  ▼
PayTransferService.createTransfer()
  │
  ├── 创建 PayTransferDO
  ├── 调用 PayClient.unifiedTransfer()
  └── 创建 PayNotifyTaskDO
```

## 通知机制设计

### 为什么需要独立的通知机制？

支付成功后，需要通知业务系统（比如电商订单服务）更新订单状态。但通知不能是"一次性"的——如果业务系统暂时不可用，通知就会丢失。

所以支付模块设计了独立的**通知任务机制**：把通知请求存储到数据库中，通过定时任务不断重试，直到成功。

### 通知任务实体

```
PayNotifyTaskDO
  ├── id               - 编号
  ├── appId            - 应用编号
  ├── type             - 通知类型（ORDER/REFUND/TRANSFER）
  ├── dataId           - 数据编号（关联具体的支付/退款/转账单）
  ├── merchantOrderId  - 商户订单编号
  ├── status           - 通知状态
  ├── notifyUrl        - 通知地址（从 PayAppDO 获取）
  ├── nextNotifyTime   - 下一次通知时间
  ├── notifyTimes      - 已通知次数
  └── maxNotifyTimes   - 最大通知次数
```

### 重试策略

通知采用**指数退避**策略：

```
通知次数:  1    2    3    4     5      6      7      8      9
间隔(秒):  15   15   30   180   1800   1800   1800   3600   -
累计(分):  0.25 0.5  1    4     34     64     94     154    -
```

算上首次通知，一共 9 次。如果 9 次都失败，通知任务标记为失败，需要人工介入。

### 通知日志

每次通知都会记录日志（`PayNotifyLogDO`），包括：
- 通知时间
- 通知结果（成功/失败）
- 响应内容
- 错误信息

这些日志对于排查问题非常有价值——当商户说"我收到支付成功的回调但订单没更新"时，你可以查看通知日志，确认通知是否发出、业务系统返回了什么。

## 与业务模块的集成

### PayAppService：应用管理

`PayAppDO` 是支付模块的顶层实体，代表一个支付应用：

```
PayAppDO
  ├── id              - 应用编号
  ├── appKey          - 应用标识（唯一）
  ├── name            - 应用名
  ├── status          - 状态
  ├── orderNotifyUrl  - 支付结果回调地址
  ├── refundNotifyUrl - 退款结果回调地址
  └── transferNotifyUrl - 转账结果回调地址
```

业务系统在调用 `PayOrderApi.createOrder()` 时，通过 `appKey` 指定使用哪个支付应用。支付模块根据 `appKey` 找到 `PayAppDO`，然后使用该应用下配置的支付渠道。

### 集成示例：电商系统对接支付

```
1. 后台配置
   └── 支付管理 → 应用管理 → 创建"商城应用"
       └── 渠道管理 → 配置微信小程序支付、支付宝 PC 支付

2. 电商订单服务代码
   @Resource
   private PayOrderApi payOrderApi;

   public void payOrder(Long orderId) {
       // 1. 创建支付单
       PayOrderCreateReqDTO reqDTO = new PayOrderCreateReqDTO();
       reqDTO.setAppKey("商城应用");
       reqDTO.setMerchantOrderId(orderId.toString());
       reqDTO.setSubject("商品标题");
       reqDTO.setPrice(999900);
       Long payOrderId = payOrderApi.createOrder(reqDTO).getCheckedData();

       // 2. 返回支付单号给前端，前端选择支付方式后调用 submit 接口
   }

   // 3. 收到支付成功回调
   public void handlePayNotify(PayOrderNotifyReqDTO notify) {
       // 更新订单状态为"已支付"
   }
```

## 思考题

1. `AbstractPayClient` 为什么用 `final` 修饰模板方法？如果子类需要绕过统一的异常处理，该怎么办？
2. 支付回调的幂等性为什么不能只靠数据库唯一索引？还需要在代码层面做状态校验？
3. 通知机制的重试间隔为什么是递增的，而不是固定间隔？如果用固定 1 分钟的间隔会有什么问题？
4. 如果你需要支持"分账"功能（把一笔支付的金额分给多个商户），你会怎么扩展 `PayClient` 接口？
