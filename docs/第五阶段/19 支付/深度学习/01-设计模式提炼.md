# 支付模块设计模式提炼

## 1. 策略模式：PayClient 的多渠道实现

### 1.1 模式定义

策略模式（Strategy Pattern）定义了一系列算法，将每个算法封装起来，并使它们可以互相替换。策略模式让算法独立于使用它的客户端而变化。

### 1.2 项目应用

支付模块需要对接多个支付渠道（支付宝、微信支付等），每个渠道的 API 接口、签名方式、回调格式都不同。通过策略模式，将不同渠道的实现封装为独立的策略类，客户端代码无需关心具体使用的是哪个渠道。

**核心类结构：**

```
PayClient<Config>                          ← 策略接口
├── AbstractPayClient<Config>              ← 抽象策略基类（模板方法）
│   ├── AbstractAlipayPayClient            ← 支付宝策略抽象
│   │   ├── AlipayAppPayClient             ← 支付宝 APP 支付
│   │   ├── AlipayWapPayClient             ← 支付宝 WAP 支付
│   │   ├── AlipayPcPayClient              ← 支付宝 PC 支付
│   │   └── AlipayQrPayClient              ← 支付宝扫码支付
│   └── AbstractWxPayClient                ← 微信支付策略抽象
│       ├── WxPayAppPayClient              ← 微信 APP 支付
│       ├── WxPayJsPayClient               ← 微信 JSAPI 支付
│       └── WxPayNativePayClient           ← 微信 Native 支付
```

### 1.3 代码示例

**策略接口定义：**

```java
// 源码位置：yudao-module-pay/yudao-module-pay-server/src/main/java/cn/iocoder/yudao/module/pay/framework/pay/core/client/PayClient.java
public interface PayClient<Config> {
    Long getId();
    Config getConfig();

    // 支付相关
    PayOrderRespDTO unifiedOrder(PayOrderUnifiedReqDTO reqDTO);
    PayOrderRespDTO parseOrderNotify(Map<String, String> params, String body, Map<String, String> headers);
    PayOrderRespDTO getOrder(String outTradeNo);

    // 退款相关
    PayRefundRespDTO unifiedRefund(PayRefundUnifiedReqDTO reqDTO);
    PayRefundRespDTO parseRefundNotify(Map<String, String> params, String body, Map<String, String> headers);
    PayRefundRespDTO getRefund(String outTradeNo, String outRefundNo);

    // 转账相关
    PayTransferRespDTO unifiedTransfer(PayTransferUnifiedReqDTO reqDTO);
    PayTransferRespDTO getTransfer(String outTradeNo);
    PayTransferRespDTO parseTransferNotify(Map<String, String> params, String body, Map<String, String> headers);
}
```

**策略接口的使用（客户端代码）：**

客户端通过 `PayClientFactory` 获取对应的 `PayClient` 实例，然后调用统一接口方法，无需关心底层是支付宝还是微信：

```java
// 伪代码示例
PayClient client = payClientFactory.getPayClient(channelId);
PayOrderRespDTO result = client.unifiedOrder(reqDTO);
```

### 1.4 可迁移经验

**适用场景：**
- 需要对接多个第三方服务（如短信服务商、邮件服务商、存储服务商）
- 同一功能有多种实现方式，需要根据配置动态切换
- 系统需要支持多种协议或格式

**设计要点：**
1. **统一抽象接口**：定义所有策略必须实现的方法，确保客户端代码的一致性
2. **泛型配置支持**：使用 `PayClient<Config>` 泛型，让每个策略可以有自己的配置类型
3. **工厂模式配合**：通过工厂类根据渠道编码创建对应的策略实例
4. **配置热更新**：支持运行时更新配置并重新初始化客户端

**迁移步骤：**
1. 定义策略接口，提取公共方法
2. 为每种实现创建独立的策略类
3. 创建工厂类，根据条件选择策略
4. 使用依赖注入管理策略实例的生命周期

## 2. 模板方法模式：支付回调处理流程

### 2.1 模式定义

模板方法模式（Template Method Pattern）在父类中定义一个算法的骨架，将某些步骤延迟到子类中实现。模板方法使得子类可以在不改变算法结构的情况下，重新定义算法的某些步骤。

### 2.2 项目应用

`AbstractPayClient` 是模板方法模式的典型应用。它定义了支付操作的完整流程（参数校验 → 执行操作 → 异常处理 → 日志记录），将具体的业务逻辑延迟到子类实现。

**模板方法的执行流程：**

```
unifiedOrder(reqDTO)                    ← 模板方法（final）
    ├── ValidationUtils.validate(reqDTO) ← 步骤1：参数校验（父类实现）
    ├── doUnifiedOrder(reqDTO)           ← 步骤2：执行支付（子类实现）
    ├── catch ServiceException           ← 步骤3a：业务异常处理（父类实现）
    └── catch Throwable                  ← 步骤3b：系统异常包装（父类实现）
```

### 2.3 代码示例

**模板方法定义：**

```java
// 源码位置：yudao-module-pay/yudao-module-pay-server/src/main/java/cn/iocoder/yudao/module/pay/framework/pay/core/client/impl/AbstractPayClient.java
public abstract class AbstractPayClient<Config extends PayClientConfig> implements PayClient<Config> {

    @Override
    public final PayOrderRespDTO unifiedOrder(PayOrderUnifiedReqDTO reqDTO) {
        // 1. 参数校验（固定步骤）
        ValidationUtils.validate(reqDTO);
        // 2. 执行统一下单（可变步骤）
        PayOrderRespDTO resp;
        try {
            resp = doUnifiedOrder(reqDTO);
        } catch (ServiceException ex) {
            throw ex;
        } catch (Throwable ex) {
            // 3. 异常处理（固定步骤）
            log.error("[unifiedOrder][客户端({}) request({}) 发起支付异常]",
                    getId(), toJsonString(reqDTO), ex);
            throw buildPayException(ex);
        }
        return resp;
    }

    // 抽象方法，由子类实现
    protected abstract PayOrderRespDTO doUnifiedOrder(PayOrderUnifiedReqDTO reqDTO) throws Throwable;
}
```

**子类实现示例（支付宝）：**

```java

        BeanUtil.copyProperties(config, alipayConfig, false);

}
```

### 2.4 可迁移经验

**适用场景：**
- 多个类有相同的算法骨架，但某些步骤的实现不同
- 需要控制子类的扩展点，只允许在特定位置进行定制
- 需要在算法的固定步骤中添加横切关注点（如日志、事务、权限）

**设计要点：**
1. **模板方法声明为 final**：防止子类覆盖整个算法骨架
2. **钩子方法**：提供可选的钩子方法，让子类可以选择性覆盖
3. **异常处理统一**：在父类中统一处理异常，子类只需关注业务逻辑
4. **日志记录集中**：在模板方法中统一记录日志，避免子类重复代码

**反模式警示：**
- 避免模板方法过于复杂，步骤过多时应考虑拆分
- 避免子类之间的依赖，每个子类应该独立

## 3. 状态机模式：支付单状态流转

### 3.1 模式定义

状态机模式（State Machine Pattern）将对象的行为封装在不同的状态类中，对象的状态改变时，其行为也随之改变。状态转换由状态对象自身控制，而不是由外部逻辑控制。

### 3.2 项目应用

支付单有明确的生命周期：未支付 → 已支付 → 已退款/已关闭。虽然项目没有使用显式的状态机框架，但通过枚举类和状态判断方法，实现了隐式的状态机管理。

**支付单状态流转图：**

```
                    ┌──────────────┐
                    │   WAITING    │
                    │   (未支付)    │
                    └──────┬───────┘
                           │
            ┌──────────────┼──────────────┐
            │              │              │
            ▼              ▼              ▼
    ┌───────────┐  ┌───────────┐  ┌───────────┐
    │  SUCCESS  │  │  CLOSED   │  │  REFUND   │
    │ (支付成功) │  │ (支付关闭) │  │  (已退款)  │
    └───────────┘  └───────────┘  └───────────┘
```

### 3.3 代码示例

**状态枚举定义：**

```java
// 源码位置：yudao-module-pay/yudao-module-pay-api/src/main/java/cn/iocoder/yudao/module/pay/enums/order/PayOrderStatusEnum.java
@Getter
@AllArgsConstructor
public enum PayOrderStatusEnum implements ArrayValuable<Integer> {
    WAITING(0, "未支付"),
    SUCCESS(10, "支付成功"),
    REFUND(20, "已退款"),
    CLOSED(30, "支付关闭");

    private final Integer status;
    private final String name;

    // 状态判断方法
    public static boolean isWaiting(Integer status) {
        return Objects.equals(status, WAITING.getStatus());
    }

    public static boolean isSuccess(Integer status) {
        return Objects.equals(status, SUCCESS.getStatus());
    }

    public static boolean isRefund(Integer status) {
        return Objects.equals(status, REFUND.getStatus());
    }

    public static boolean isSuccessOrRefund(Integer status) {
        return ObjectUtils.equalsAny(status, SUCCESS.getStatus(), REFUND.getStatus());
    }

    public static boolean isClosed(Integer status) {
        return Objects.equals(status, CLOSED.getStatus());
    }
}
```

**状态转换逻辑示例：**

```java
// 在 PayOrderServiceImpl 中
public void notifyOrder(Long channelId, PayOrderRespDTO notify) {
    PayOrderDO order = orderMapper.selectById(notify.getOutTradeNo());
    // 状态校验：只有未支付状态才能变为已支付
    if (!PayOrderStatusEnum.isWaiting(order.getStatus())) {
        log.warn("[notifyOrder][订单({})状态不是未支付，忽略通知]", order.getId());
        return;
    }
    // 更新状态为已支付
    order.setStatus(PayOrderStatusEnum.SUCCESS.getStatus());
    order.setPayTime(notify.getSuccessTime());
    orderMapper.updateById(order);
}
```

### 3.4 可迁移经验

**适用场景：**
- 对象有明确的生命周期和状态转换规则
- 不同状态下对象的行为差异较大
- 需要防止非法的状态转换

**设计要点：**
1. **枚举定义状态**：使用枚举类定义所有可能的状态，避免魔法数字
2. **状态判断方法**：提供语义化的状态判断方法，提高代码可读性
3. **转换规则集中**：将状态转换规则集中在 Service 层，避免分散在各处
4. **乐观锁保护**：使用版本号或状态条件更新，防止并发状态冲突

**进阶方案：**
- 对于复杂的状态机，可以考虑使用 Spring Statemachine 框架
- 将状态转换规则抽离为配置，支持动态调整

## 4. 可迁移经验：如何设计可扩展的支付抽象层

### 4.1 设计原则

**单一职责原则：**
- `PayClient` 接口只负责与支付渠道交互
- `PayOrderService` 负责支付单的生命周期管理
- `PayNotifyService` 负责回调通知的处理

**开闭原则：**
- 新增支付渠道只需实现 `PayClient` 接口
- 无需修改现有的业务代码

**依赖倒置原则：**
- 业务层依赖 `PayClient` 抽象接口
- 具体的支付渠道实现通过依赖注入

### 4.2 分层架构

```
┌─────────────────────────────────────────────────────┐
│                   业务层（Trade）                     │
│         调用 PayOrderService 创建支付单              │
└───────────────────────────┬─────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────┐
│                支付服务层（Pay）                      │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ PayOrder    │  │ PayRefund   │  │ PayNotify   │  │
│  │ Service     │  │ Service     │  │ Service     │  │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘  │
│         │               │               │          │
│         ▼               ▼               ▼          │
│  ┌─────────────────────────────────────────────┐   │
│  │           PayClientFactory                   │   │
│  └───────────────────┬─────────────────────────┘   │
└──────────────────────┼─────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│               支付渠道层（Channel）                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │  Alipay     │  │  WxPay      │  │  UnionPay   │  │
│  │  Client     │  │  Client     │  │  Client     │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
└─────────────────────────────────────────────────────┘
```

### 4.3 扩展新渠道的步骤

1. **创建配置类**：继承 `PayClientConfig`，定义渠道特有的配置项
2. **实现客户端类**：继承 `AbstractPayClient`，实现 `doUnifiedOrder` 等抽象方法
3. **注册到工厂**：在 `PayClientFactory` 中注册新渠道的创建逻辑
4. **配置渠道信息**：在数据库中配置渠道的参数（appId、密钥等）

### 4.4 关键设计决策

**为什么使用泛型配置？**
```java
public interface PayClient<Config> { ... }
public abstract class AbstractPayClient<Config extends PayClientConfig> { ... }
```
- 每个渠道的配置项不同（支付宝有证书模式，微信有 API 密钥）
- 泛型确保类型安全，避免运行时类型转换

**为什么模板方法用 final？**
```java
@Override
public final PayOrderRespDTO unifiedOrder(PayOrderUnifiedReqDTO reqDTO) { ... }
```
- 防止子类覆盖统一的异常处理和日志记录逻辑
- 确保所有渠道的行为一致性

**为什么提供多个子类（如 AlipayAppPayClient、AlipayWapPayClient）？**
- 同一渠道的不同支付方式，API 调用方式不同
- 将差异封装在独立的类中，避免一个类过于庞大

### 4.5 反模式与陷阱

**反模式 1：大而全的 if-else**
```java
// 错误示例
if ("alipay".equals(channel)) {
    // 支付宝逻辑
} else if ("wxpay".equals(channel)) {
    // 微信逻辑
}
```
- 违反开闭原则，新增渠道需要修改此处代码
- 应使用策略模式 + 工厂模式替代

**反模式 2：配置硬编码**
```java
// 错误示例
private static final String ALIPAY_APP_ID = "xxx";
```
- 配置应从数据库或配置中心读取，支持动态更新

**反模式 3：忽略异常差异**
- 不同渠道的错误码和错误信息格式不同
- 应在渠道层将异常翻译为统一的业务异常

## 总结

支付模块通过策略模式实现了多渠道支付的灵活扩展，通过模板方法模式统一了支付操作的流程和异常处理，通过状态机模式管理支付单的生命周期。这些设计模式的组合使用，使得支付模块具有良好的可扩展性和可维护性。

**核心设计要点：**
1. 策略模式让新增支付渠道只需实现接口，无需修改业务代码
2. 模板方法模式确保所有渠道的行为一致性，减少重复代码
3. 状态机模式通过枚举和状态判断方法，清晰地管理支付单生命周期
4. 分层架构将业务逻辑与渠道实现解耦，便于独立演进
