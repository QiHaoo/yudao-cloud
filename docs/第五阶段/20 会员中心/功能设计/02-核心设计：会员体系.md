# 02 - 核心设计：会员体系

上一章我们看到了会员中心的全景。这一章深入几个核心设计问题：

1. **会员实体设计**：MemberUserDO 需要存储哪些信息？为什么这么设计？

## MemberUserDO：会员实体设计

### 字段分析

```java
public class MemberUserDO extends TenantBaseDO {

    // ========== 账号信息 ==========
    private Long id;              // 用户ID
    private String mobile;        // 手机号（主要标识）
    private String password;      // 加密后的密码（BCrypt）
    private Integer status;       // 帐号状态（启用/禁用）
    private String registerIp;    // 注册 IP
    private Integer registerTerminal; // 注册终端（小程序/H5/App）
    private String loginIp;       // 最后登录 IP
    private LocalDateTime loginDate;  // 最后登录时间

    // ========== 基础信息 ==========
    private String nickname;      // 用户昵称
    private String avatar;        // 用户头像
    private String name;          // 真实名字
    private Integer sex;          // 性别
    private LocalDateTime birthday;   // 出生日期
    private Integer areaId;       // 所在地（关联 Area 表）
    private String mark;          // 用户备注

    // ========== 其它信息 ==========
    private Integer point;        // 积分
    private List<Long> tagIds;    // 会员标签列表
    private Long levelId;         // 会员级别编号
    private Integer experience;   // 会员经验
    private Long groupId;         // 用户分组编号
}
```

### 设计要点

**1. 为什么用 mobile 而不是 username 作为主要标识？**

Admin 用户用 username（用户名）登录，因为后台管理员的用户名是管理员分配的，具有唯一性。

Member 用户用 mobile（手机号）登录，因为：
- C 端用户没有"用户名"的概念，手机号是最自然的标识
- 短信验证码登录是主流方式，手机号是必需的
- 手机号具有天然的唯一性
- 手机号可以用于找回密码、接收通知

**2. 为什么 password 是可选的？**

不是所有会员都会设置密码。微信小程序登录、手机号验证码登录的用户可能根本没有密码。password 字段只有在用户选择"手机号 + 密码"登录时才会有值。

**3. 为什么 areaId 是 Integer 而不是字符串？**

areaId 关联的是系统模块的 `Area` 表（行政区划表），存储的是地区的数字编码。这样设计的好处：
- 标准化：使用国家行政区划编码，不会出现"北京市"和"北京"这种不一致
- 节省存储：数字比字符串更省空间
- 便于查询：可以按省、市、区做聚合查询

**4. tagIds 为什么用逗号分隔的 List 而不是关联表？**

`tagIds` 用 `LongListTypeHandler` 存储为逗号分隔的字符串。这是一种**反范式设计**：
- 优点：查询简单，不需要 JOIN 关联表
- 缺点：无法通过标签反查用户（需要全表扫描）
- 适用场景：标签数量不多，主要是"给用户打标签"而不是"查某个标签下有哪些用户"

如果需要"按标签查用户"的能力，应该改用关联表。

**5. 为什么继承 TenantBaseDO 而不是 BaseDO？**

MemberUserDO 继承 `TenantBaseDO`，意味着会员数据是**租户隔离**的。在 SaaS 模式下，不同租户（商户）的会员数据是完全隔离的。

## 会员登录方式

### 登录方式概览

会员模块支持四种登录方式，每种方式的适用场景不同：

```
登录方式                    适用场景              是否需要密码
─────────────────────────────────────────────────────────────
手机号 + 密码              H5/App 老用户          是
手机号 + 验证码            H5/App 通用            否
微信小程序一键登录          小程序                 否
社交登录（code 授权码）     公众号/支付宝生活号     否
```

### 手机号 + 密码登录

最传统的登录方式：

```
用户输入手机号 + 密码
  │
  ▼
AppAuthController.login()
  │
  ├── 1. 根据手机号查找 MemberUserDO
  ├── 2. 校验密码（BCrypt 比对）
  ├── 3. 校验账号状态
  └── 4. 生成 OAuth2 令牌返回
```

### 手机号 + 验证码登录

更安全、更便捷的登录方式：

```
用户请求发送验证码
  │
  ▼
AppAuthController.sendSmsCode()
  │
  ├── 1. 生成 6 位随机验证码
  ├── 2. 存入 Redis（5 分钟过期）
  └── 3. 调用 system 模块发送短信

用户输入验证码
  │
  ▼
AppAuthController.smsLogin()
  │
  ├── 1. 从 Redis 获取验证码比对
  ├── 2. 根据手机号查找或自动创建用户
  └── 3. 生成 OAuth2 令牌返回
```

**自动创建用户**：如果手机号不存在，会自动创建一个新的 MemberUserDO。这是 C 端产品的常见设计——用户第一次用验证码登录时就自动注册了。

### 微信小程序一键登录

微信小程序特有的登录方式：

```
小程序端调用 wx.login() 获取 code
  │
  ▼
AppAuthController.weixinMiniAppLogin()
  │
  ├── 1. 用 code 调用微信接口获取 openid
  ├── 2. 根据 openid 查找已绑定的 MemberUserDO
  │     ├── 找到 → 直接登录
  │     └── 没找到 → 自动创建用户并绑定
  └── 3. 生成 OAuth2 令牌返回
```

### 社交登录

支持微信公众号、支付宝等社交平台的 OAuth 登录：

```
用户点击"微信登录"
  │
  ▼
AppAuthController.socialAuthRedirect()
  │
  └── 返回微信授权页面 URL

用户授权后回调
  │
  ▼
AppAuthController.socialLogin()
  │
  ├── 1. 用 code 换取社交平台用户信息
  ├── 2. 查找已绑定的 MemberUserDO
  │     ├── 找到 → 直接登录
  │     └── 没找到 → 自动创建用户并绑定
  └── 3. 生成 OAuth2 令牌返回
```

## OAuth2 令牌体系复用

### 设计决策

会员登录和 Admin 登录**共用同一套 OAuth2 令牌体系**。这意味着：

- 令牌格式相同（JWT）
- 令牌存储方式相同（Redis）
- 令牌刷新逻辑相同
- 令牌校验逻辑相同

### 如何区分 Admin 和 Member 的令牌？

通过 `UserTypeEnum` 区分：

```
UserTypeEnum
  ├── ADMIN (1)   - 管理员
  └── MEMBER (2)  - 会员
```

令牌中包含 `userType` 字段，Spring Security 的过滤器根据 `userType` 决定走哪套用户加载逻辑。

### 复用的好处

1. **减少重复代码**：令牌的生成、存储、刷新、校验逻辑不需要写两遍
2. **统一管理**：可以在同一个地方管理所有令牌（查看在线用户、强制下线等）
3. **网关统一校验**：API 网关只需要一套令牌校验逻辑

### 令牌中的用户信息

```
OAuth2AccessTokenDO
  ├── userId      - 用户编号
  ├── userType    - 用户类型（ADMIN/MEMBER）
  ├── accessToken - 访问令牌
  ├── refreshToken - 刷新令牌
  └── expiresTime - 过期时间
```

## 收货地址管理

### 地址实体

```java
public class MemberAddressDO extends BaseDO {
    private Long id;              // 编号
    private Long userId;          // 用户编号
    private String name;          // 收件人名称
    private String mobile;        // 手机号
    private Long areaId;          // 地区编号
    private String detailAddress; // 收件详细地址
    private Boolean defaultStatus; // 是否默认
}
```

### 设计要点

**1. 为什么 areaId 和 detailAddress 分开？**

```
areaId: 110105  →  北京市 朝阳区
detailAddress: 望京 SOHO T1 20 层
```

分开的好处：
- areaId 关联行政区划表，可以做地区统计（"北京有多少订单"）
- detailAddress 只存具体门牌号，节省空间
- 前端可以用级联选择器选择省市区，体验更好

**2. defaultStatus 的处理逻辑**

一个用户可以有多个收货地址，但只能有一个默认地址。设置默认地址时需要：
1. 将该用户所有地址的 `defaultStatus` 设为 `false`
2. 将目标地址的 `defaultStatus` 设为 `true`

这是一个典型的**唯一约束**场景，但在代码层面处理（而不是数据库唯一索引），因为"每个用户一个默认地址"这个约束需要跨行操作。

**3. 为什么地址属于 BaseDO 而不是 TenantBaseDO？**

地址本身不直接关联租户，它是通过 `userId` 间接关联的。会员用户的租户隔离已经保证了数据安全。

## 会员等级与积分

### 等级体系

```
MemberLevelDO (等级定义)
  ├── id                - 编号
  ├── name              - 等级名称（如"普通会员"、"银卡会员"、"金卡会员"）
  ├── level             - 等级数值（1、2、3...）
  ├── experience        - 升级所需经验
  ├── discountPercent   - 享受折扣（如 95 表示 95 折）
  ├── icon              - 等级图标
  ├── backgroundUrl     - 等级背景图
  └── status            - 状态
```

等级配置示例：

| 等级 | 名称 | 升级经验 | 折扣 |
|------|------|---------|------|
| 1 | 普通会员 | 0 | 100（无折扣） |
| 2 | 银卡会员 | 1000 | 98 |
| 3 | 金卡会员 | 5000 | 95 |
| 4 | 钻石会员 | 20000 | 90 |

### 积分 vs 经验

积分和经验是两个不同的概念：

| 维度 | 积分 (point) | 经验 (experience) |
|------|-------------|-------------------|
| 用途 | 消费抵扣、兑换商品 | 升级等级 |
| 可消费 | 是 | 否 |
| 获取方式 | 签到、消费、活动 | 签到、消费 |
| 减少 | 可以（消费扣减） | 不会减少 |
| 存储 | MemberUserDO.point | MemberUserDO.experience |

**为什么不直接用积分作为升级依据？**

因为积分是可以消费的。如果用户攒了 5000 积分升到金卡，然后用积分兑换了一个商品，积分降到 3000，那他的等级要不要降？降级会带来很差的用户体验。

所以用经验作为升级依据——经验只增不减（或者有明确的减少规则），等级就不会因为消费而降级。

### 积分记录

```java
public class MemberPointRecordDO extends BaseDO {
    private Long id;          // 编号
    private Long userId;      // 用户编号
    private Integer point;    // 积分值（正数增加，负数减少）

    private String bizId;     // 业务编号
    private String description; // 描述
}
```

积分变动时会创建一条记录，用于积分明细查询和对账。

## 签到体系

### 签到规则

```java
public class MemberSignInConfigDO extends BaseDO {
    private Long id;          // 规则编号
    private Integer day;      // 签到第 x 天
    private Integer point;    // 奖励积分
    private Integer experience; // 奖励经验
    private Integer status;   // 状态
}
```

运营可以在后台配置签到规则，比如：
- 第 1 天：10 积分 + 5 经验
- 第 2 天：10 积分 + 5 经验
- 第 3 天：15 积分 + 8 经验
- 连续 7 天：30 积分 + 15 经验

### 签到记录

```java
public class MemberSignInRecordDO extends BaseDO {
    private Long id;          // 编号
    private Long userId;      // 用户编号
    private Integer day;      // 签到天数
    private Integer point;    - 获得积分
    private Integer experience; // 获得经验
    private LocalDateTime signInTime; // 签到时间
}
```

### 签到流程

```
用户点击"签到"
  │
  ▼
MemberSignInService.signIn()
  │
  ├── 1. 检查今天是否已签到（防重复签到）
  ├── 2. 计算连续签到天数
  │     ├── 查询昨天是否有签到记录
  │     │     ├── 有 → 连续天数 = 上次连续天数 + 1
  │     │     └── 没有 → 连续天数 = 1（重新开始）
  ├── 3. 根据连续天数查找签到规则
  ├── 4. 创建签到记录
  ├── 5. 增加积分和经验
  └── 6. 返回签到结果
```

## 标签与分组

### 标签 (MemberTagDO)

```java
public class MemberTagDO extends BaseDO {
    private Long id;          // 标签编号
    private String name;      // 标签名称
    private Integer status;   // 状态
}
```

标签是**多对多**关系：一个用户可以有多个标签，一个标签可以属于多个用户。

存储方式：MemberUserDO.tagIds 存储为逗号分隔的标签 ID 列表。

使用场景：
- 运营给用户打标签："高价值用户"、"活跃用户"
- 营销活动按标签筛选目标用户

### 分组 (MemberGroupDO)

```java
public class MemberGroupDO extends BaseDO {
    private Long id;          // 分组编号
    private String name;      // 分组名称
    private Integer status;   // 状态
}
```

分组是**一对多**关系：一个用户只能属于一个分组。

存储方式：MemberUserDO.groupId 存储分组 ID。

使用场景：
- 将用户分组管理："VIP 客户群"、"潜在流失用户群"
- 不同分组的用户享受不同的运营策略

### 标签 vs 分组

| 维度 | 标签 | 分组 |
|------|------|------|
| 关系 | 多对多 | 一对多 |
| 存储 | tagIds (逗号分隔) | groupId (单个值) |
| 场景 | 用户画像、精细化运营 | 用户管理、分群运营 |
| 灵活性 | 高（一个用户可以有多个标签） | 低（一个用户只能在一个分组） |

## RPC API 设计

### MemberUserApi

```java
@FeignClient(name = "yunai-member")
public interface MemberUserApi {

    CommonResult<MemberUserRespDTO> getUser(Long id);
```
