# 01 - 总览：会员中心全景

## 场景引入

你做了一个电商系统，后台管理已经有了（系统模块），但 C 端用户（买家）的体系还没有。你需要：

- 用户能在小程序/公众号/H5 上注册、登录
- 用户能管理自己的收货地址
- 用户能查看自己的订单、收藏商品
- 用户有积分、等级，可以享受不同的优惠
- 用户每天签到可以领积分

这些功能都属于**会员中心**的范畴。

## 会员模块的定位

在 yudao-cloud 的架构中，用户体系分为两套：

```
┌─────────────────────────────────────────────────────────────┐
│                        用户体系                               │
│                                                              │
│  ┌──────────────────────┐    ┌──────────────────────┐       │
│  │   Admin 用户体系      │    │   Member 用户体系     │       │
│  │   (后台管理人员)       │    │   (C 端消费者)        │       │
│  │                      │    │                      │       │
│  │  AdminUserDO         │    │  MemberUserDO        │       │
│  │  ┌────────────────┐  │    │  ┌────────────────┐  │       │
│  │  │ username       │  │    │  │ mobile         │  │       │
│  │  │ password       │  │    │  │ password       │  │       │
│  │  │ nickname       │  │    │  │ nickname       │  │       │
│  │  │ dept_id        │  │    │  │ avatar         │  │       │
│  │  │ post_id        │  │    │  │ sex            │  │       │
│  │  │ ...            │  │    │  │ birthday       │  │       │
│  │  └────────────────┘  │    │  │ point          │  │       │
│  │                      │    │  │ level_id       │  │       │
│  │  所在模块：            │    │  │ experience     │  │       │
│  │  yudao-module-system │    │  │ ...            │  │       │

│                              │  所在模块：            │       │
│                              │  yudao-module-member  │       │
│                              └──────────────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### Admin 用户 vs Member 用户

| 维度 | Admin 用户 | Member 用户 |
|------|-----------|------------|
| 面向对象 | 后台管理人员（运营、客服、管理员） | C 端消费者（买家、用户） |
| 登录方式 | 账号 + 密码 | 手机号 + 验证码、微信小程序、公众号 |
| 用户标识 | username（用户名） | mobile（手机号） |
| 组织关系 | 部门、岗位 | 无（松散结构） |
| 权限模型 | RBAC（角色 + 菜单权限） | 无精细权限，按等级区分 |
| 所在模块 | yudao-module-system | yudao-module-member |
| 数据隔离 | 租户级隔离 | 租户级隔离 |
| 扩展信息 | 部门、岗位、角色 | 积分、等级、经验、标签、分组 |

为什么要分两套？因为两者的业务模型完全不同：

- Admin 用户关心的是**组织架构**（部门、岗位、角色权限）
- Member 用户关心的是**用户画像**（积分、等级、标签、消费行为）

如果强行合并到一张表，要么字段大量冗余，要么逻辑复杂到无法维护。

## 核心能力概览

### 1. 会员注册与登录

会员模块支持多种登录方式：

```
登录方式
  │
  ├── 手机号 + 密码        (AppAuthController.login)
  ├── 手机号 + 验证码      (AppAuthController.smsLogin)
  ├── 微信小程序一键登录    (AppAuthController.weixinMiniAppLogin)
  ├── 社交登录（code 授权） (AppAuthController.socialLogin)
  └── 刷新令牌             (AppAuthController.refreshToken)
```

### 2. 收货地址管理

```
收货地址
  │
  ├── 新增地址      (AppAddressController.createAddress)
  ├── 修改地址      (AppAddressController.updateAddress)
  ├── 删除地址      (AppAddressController.deleteAddress)
  ├── 查询地址列表   (AppAddressController.getAddressList)
  └── 设置默认地址   (默认地址标记在 AddressDO.defaultStatus 字段)
```

### 3. 积分体系

```
积分变动
  │
  ├── 签到送积分      (MemberSignInService)
  ├── 消费送积分      (MemberPointService.addPoint)
  ├── 积分扣减        (MemberPointService.subPoint)
  └── 积分明细查询    (MemberPointRecordDO)
```

### 4. 等级体系

```
等级体系
  │
  ├── MemberLevelDO        - 等级定义（名称、升级经验、折扣）
  ├── MemberExperienceRecordDO - 经验变动记录
  └── MemberLevelRecordDO  - 等级变动记录
```

等级与经验的关系：
- 用户通过消费、签到等行为获得经验值
- 经验值达到等级阈值时自动升级
- 不同等级享受不同的折扣

### 5. 签到体系

```
签到体系
  │
  ├── MemberSignInConfigDO  - 签到规则（第 N 天奖励多少积分/经验）
  ├── MemberSignInRecordDO  - 签到记录
  └── 连续签到奖励机制
```

签到规则示例：
| 签到天数 | 奖励积分 | 奖励经验 |
|---------|---------|---------|
| 第 1 天 | 10 | 5 |
| 第 2 天 | 10 | 5 |
| 第 3 天 | 15 | 8 |
| 第 7 天 | 30 | 15 |

### 6. 标签与分组

```
标签 (MemberTagDO)
  │
  └── 给会员打标签，用于精细化运营
      例如："高价值用户"、"活跃用户"、"新用户"

分组 (MemberGroupDO)
  │
  └── 将会员分组，用于定向营销
      例如："VIP 客户群"、"潜在流失用户群"
```

## 核心实体关系

```
MemberUserDO (会员用户)
  │
  ├── 1:N ── MemberAddressDO (收货地址)
  │
  ├── 1:N ── MemberPointRecordDO (积分记录)
  │
  ├── 1:N ── MemberExperienceRecordDO (经验记录)
  │
  ├── 1:N ── MemberLevelRecordDO (等级变动记录)
  │
  ├── 1:N ── MemberSignInRecordDO (签到记录)
  │
  ├── N:1 ── MemberLevelDO (会员等级)
  │
  ├── N:1 ── MemberGroupDO (会员分组)
  │
  └── N:N ── MemberTagDO (会员标签，通过 tagIds 字段存储)
```

## 与电商模块的关系

会员模块是电商模块的基础依赖：

```
电商模块 (yudao-module-trade)
  │
  ├── 下单时查询会员信息
  │   MemberUserApi.getUser(id) → 获取会员昵称、手机号
  │
  ├── 下单时查询收货地址
  │   MemberAddressApi.getAddress(id) → 获取收货人、手机号、地址
  │
  ├── 计算会员折扣
  │   MemberLevelApi.getLevel(levelId) → 获取折扣百分比
  │
  ├── 消费后增加积分和经验
  │   MemberPointApi.addPoint(userId, point, bizType)
  │
  └── 展示会员等级和积分
      MemberUserApi.getUser(id) → 返回 point, levelId, experience
```

## App 端 vs Admin 端

会员模块的 Controller 分为两套：

### App 端（C 端用户）

路径前缀：`/app-api/member/`（由 URL 前缀注入机制自动添加）

```
/member/auth/*          - 认证相关（登录/登出/刷新令牌）
/member/address/*       - 收货地址管理
/member/point/*         - 积分明细查询
/member/level/*         - 等级信息查询
/member/sign-in/*       - 签到
```

### Admin 端（后台管理）

路径前缀：`/admin-api/member/`（由 URL 前缀注入机制自动添加）

```
/member/user/*          - 会员用户管理（查询、编辑、导出）
/member/address/*       - 收货地址管理
/member/level/*         - 等级管理（增删改查）
/member/point/*         - 积分记录查询
/member/tag/*           - 标签管理
/member/group/*         - 分组管理
/member/sign-in/*       - 签到配置和记录
/member/config/*        - 会员配置
```

## 思考题

1. 为什么 Admin 用户和 Member 用户要分成两张表，而不是共用一张用户表？
2. 会员登录复用了 system 模块的 OAuth2 令牌体系，这样设计有什么好处？如果会员模块自己实现一套令牌体系会怎样？
3. 积分和经验有什么区别？为什么不直接用积分作为升级依据？
4. 标签和分组的区别是什么？什么场景用标签，什么场景用分组？
