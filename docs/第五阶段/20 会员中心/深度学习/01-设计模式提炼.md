# 会员中心设计模式提炼

## 1. 双 Controller 模式：admin-api vs app-api 的职责分离

### 1.1 模式定义

双 Controller 模式是指同一个业务模块同时提供两套 API 接口：一套面向后台管理员（admin-api），一套面向终端用户（app-api）。两套接口共享同一个 Service 层，但有不同的权限控制、数据范围和业务逻辑。

### 1.2 项目应用

会员中心模块同时服务于两个角色：
- **管理员**：管理会员信息、配置会员等级、查看积分记录等
- **会员用户**：查看自己的信息、签到、查看积分、管理地址等

**目录结构对比：**

```
yudao-module-member-server/src/main/java/cn/iocoder/yudao/module/member/controller/
├── admin/                              ← 管理后台 API
│   ├── address/AddressController.java
│   ├── config/MemberConfigController.java
│   ├── level/MemberLevelController.java
│   ├── point/MemberPointRecordController.java
│   ├── signin/MemberSignInConfigController.java
│   ├── tag/MemberTagController.java
│   └── user/MemberUserController.java
└── app/                                ← 用户端 API
    ├── address/AppAddressController.java
    ├── auth/AppAuthController.java
    ├── level/AppMemberLevelController.java
    ├── point/AppMemberPointRecordController.java
    └── signin/AppMemberSignInRecordController.java
```

### 1.3 代码示例

**Admin Controller 示例：**

```java
// 源码位置：yudao-module-member-server/src/main/java/cn/iocoder/yudao/module/member/controller/admin/point/MemberPointRecordController.java
@RestController
@RequestMapping("/member/point-record")
public class MemberPointRecordController {

    @Resource
    private MemberPointRecordService pointRecordService;

    @GetMapping("/page")
    @PreAuthorize("@ss.hasPermission('member:point-record:query')")
    public CommonResult<PageResult<MemberPointRecordDO>> getPointRecordPage(
            @Valid MemberPointRecordPageReqVO pageReqVO) {
        return success(pointRecordService.getPointRecordPage(pageReqVO));
    }
}
```

**App Controller 示例：**

```java
// 源码位置：yudao-module-member-server/src/main/java/cn/iocoder/yudao/module/member/controller/app/point/AppMemberPointRecordController.java
@RestController
@RequestMapping("/app-api/member/point-record")
public class AppMemberPointRecordController {

    @Resource
    private MemberPointRecordService pointRecordService;

    @GetMapping("/page")
    public CommonResult<PageResult<MemberPointRecordDO>> getPointRecordPage(
            @Valid AppMemberPointRecordPageReqVO pageReqVO) {
        // 从 SecurityContextHolder 获取当前登录用户
        Long userId = SecurityFrameworkUtils.getLoginUserId();
        return success(pointRecordService.getPointRecordPage(userId, pageReqVO));
    }
}
```

**Service 接口同时支持两种调用：**

```java
// 源码位置：yudao-module-member-server/src/main/java/cn/iocoder/yudao/module/member/service/point/MemberPointRecordService.java
public interface MemberPointRecordService {
    // 管理员查询：查询所有用户的积分记录
    PageResult<MemberPointRecordDO> getPointRecordPage(MemberPointRecordPageReqVO pageReqVO);

    // 会员查询：只查询自己的积分记录
    PageResult<MemberPointRecordDO> getPointRecordPage(Long userId, AppMemberPointRecordPageReqVO pageReqVO);
}
```

### 1.4 设计优势

**URL 前缀隔离：**
- Admin API：`/admin-api/member/...`
- App API：`/app-api/member/...`

**权限控制差异：**
- Admin API：使用 `@PreAuthorize` 注解，基于角色和权限控制
- App API：基于用户身份，只能访问自己的数据

**数据范围差异：**
- Admin API：可以查看所有会员的数据
- App API：只能查看自己的数据，通过 `userId` 过滤

### 1.5 可迁移经验

**适用场景：**
- C 端应用需要同时支持管理后台和用户端
- 管理后台和用户端的权限模型不同
- 需要复用同一套业务逻辑

**设计要点：**
1. **Service 层复用**：两套 Controller 共用同一个 Service，避免业务逻辑重复
2. **VO 分离**：Admin 和 App 使用不同的 VO 类，因为参数和响应可能不同
3. **URL 前缀规范**：统一使用 `/admin-api/` 和 `/app-api/` 前缀
4. **权限注解区分**：Admin 使用权限注解，App 使用用户身份校验

**反模式警示：**
- 避免在 Controller 中编写业务逻辑，应下沉到 Service 层
- 避免 Admin 和 App 的 Service 实现重复，应通过参数区分

## 2. 积分/余额的并发安全设计

### 2.1 问题场景

积分和余额是典型的共享资源，多线程并发修改可能导致数据不一致：

```
时间线     线程A              线程B
  │      读取余额=100
  │                          读取余额=100
  │      扣减50，余额=50
  │                          扣减30，余额=70（错误！应该是20）
  ▼
```

### 2.2 解决方案

**方案一：数据库乐观锁**

```java
// 伪代码示例
UPDATE member_user 
SET balance = balance - 50, version = version + 1 
WHERE id = 123 AND version = 1
```

**方案二：数据库悲观锁**

```java
// 伪代码示例
SELECT * FROM member_user WHERE id = 123 FOR UPDATE
UPDATE member_user SET balance = balance - 50 WHERE id = 123
```

**方案三：SQL 原子操作（推荐）**

```java
// 伪代码示例
UPDATE member_user SET balance = balance - 50 WHERE id = 123 AND balance >= 50
```

### 2.3 项目应用

项目采用 SQL 原子操作 + 业务校验的方式：

```java
// 积分记录创建
public void createPointRecord(Long userId, Integer point, MemberPointBizTypeEnum bizType, String bizId) {
    // 1. 校验用户是否存在
    MemberUserDO user = memberUserMapper.selectById(userId);
    if (user == null) {
        throw exception(USER_NOT_EXISTS);
    }
    // 2. 创建积分记录
    MemberPointRecordDO record = new MemberPointRecordDO();
    record.setUserId(userId);
    record.setPoint(point);
    record.setBizType(bizType.getType());
    record.setBizId(bizId);
    pointRecordMapper.insert(record);
    // 3. 更新用户积分（SQL 原子操作）
    memberUserMapper.updatePoint(userId, point);
}
```

### 2.4 代码示例

**积分记录 Service：**

```java

public interface MemberPointRecordService {
    /**
     * 创建用户积分记录
     *
     * @param userId  用户ID
     * @param point   变动积分（正数增加，负数扣减）
     * @param bizType 业务类型
     * @param bizId   业务编号
     */
    void createPointRecord(Long userId, Integer point, MemberPointBizTypeEnum bizType, String bizId);
}
```

### 2.5 可迁移经验

**适用场景：**
- 账户余额、积分、库存等共享资源的并发修改
- 需要记录变更历史的场景

**设计要点：**
1. **原子操作优先**：使用 SQL 的 `SET balance = balance - #{amount}` 而非先查后改
2. **边界校验**：在 SQL 中加入 `WHERE balance >= #{amount}` 防止负数
3. **记录变更日志**：每次变更都创建记录，便于审计和问题排查
4. **幂等设计**：通过 `bizType + bizId` 唯一索引防止重复操作

**并发控制方案对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 乐观锁 | 性能好，无锁等待 | 冲突时需要重试 | 读多写少，冲突概率低 |
| 悲观锁 | 数据一致性强 | 性能差，可能死锁 | 冲突概率高，金额敏感 |
| SQL 原子操作 | 简单高效 | 不适合复杂业务逻辑 | 简单的增减操作 |

## 3. 会员等级自动升降级的规则引擎

### 3.1 问题场景

会员等级通常基于经验值自动升降级，规则可能包括：
- 消费 1 元 = 1 经验值
- 签到一次 = 10 经验值
- 经验值达到 1000 升级为银卡
- 经验值达到 5000 升级为金卡
- 连续 90 天无消费降级

### 3.2 项目应用

项目通过经验值机制实现等级自动升级：

```
用户行为 → 增加经验值 → 检查是否满足升级条件 → 自动升级
```

**核心类：**

```java
// 源码位置：yudao-module-member-server/src/main/java/cn/iocoder/yudao/module/member/service/level/MemberLevelService.java
public interface MemberLevelService {
    /**
     * 增加会员经验
     *
     * @param userId     会员ID
     * @param experience 经验
     * @param bizType    业务类型
     * @param bizId      业务编号
     */
    void addExperience(Long userId, Integer experience, MemberExperienceBizTypeEnum bizType, String bizId);
}
```

### 3.3 代码示例

**等级配置数据结构：**

```java
// 会员等级 DO
public class MemberLevelDO {
    private Long id;                    // 等级编号
    private String name;                // 等级名称（如：普通会员、银卡、金卡）
    private Integer level;              // 等级值（用于排序）
    private Integer experience;         // 升级所需经验值
    private Integer status;             // 状态（启用/禁用）
    private BigDecimal discount;        // 折扣率
}
```

**经验值变更记录：**

```java
// 会员经验记录 DO
public class MemberExperienceRecordDO {
    private Long id;
    private Long userId;                // 用户编号
    private Integer experience;         // 变动经验值
    private Integer bizType;            // 业务类型（消费、签到等）
    private String bizId;               // 业务编号
    private LocalDateTime createTime;   // 创建时间
}
```

### 3.4 设计思路

**经验值增加流程：**

```
1. 记录经验值变更（MemberExperienceRecordDO）
2. 更新用户总经验值（MemberUserDO.experience）
3. 查询当前等级配置（MemberLevelDO）
4. 判断是否满足升级条件
5. 如果满足，更新用户等级（MemberUserDO.level）
```

**等级升降级判断：**

```java
// 伪代码
public void addExperience(Long userId, Integer experience, ...) {
    // 1. 记录变更
    createExperienceRecord(userId, experience, bizType, bizId);
    // 2. 更新总经验值
    memberUserMapper.updateExperience(userId, experience);
    // 3. 查询用户当前信息
    MemberUserDO user = memberUserMapper.selectById(userId);
    // 4. 查询所有启用的等级配置
    List<MemberLevelDO> levels = memberLevelMapper.selectListByStatus(ENABLE);
    // 5. 找到用户应该处于的等级
    MemberLevelDO targetLevel = levels.stream()
        .filter(level -> user.getExperience() >= level.getExperience())
        .max(Comparator.comparing(MemberLevelDO::getLevel))
        .orElse(null);
    // 6. 如果等级发生变化，更新用户等级
    if (targetLevel != null && !targetLevel.getLevel().equals(user.getLevel())) {
        memberUserMapper.updateLevel(userId, targetLevel.getLevel());
    }
}
```

### 3.5 可迁移经验

**适用场景：**
- 会员等级体系
- 用户成长值系统
- 积分兑换规则

**设计要点：**
1. **规则配置化**：将等级规则存储在数据库中，支持动态调整
2. **经验值与等级解耦**：经验值是连续的，等级是离散的，通过配置映射
3. **变更记录完整**：每次经验值变更都记录，便于审计
4. **异步处理**：等级变更可以异步处理，不影响主流程性能

**进阶方案：**
- 使用规则引擎（如 Drools）处理复杂的升降级规则
- 支持多维度评估（消费金额 + 活跃度 + 信用分）

### 4.2 核心模块设计

**会员用户模块（MemberUser）：**
- 存储用户基础信息
- 关联等级、积分、余额等聚合数据
- 支持扩展字段（如自定义属性）

**会员等级模块（MemberLevel）：**
- 等级配置管理
- 等级权益定义
- 自动升降级规则

**会员积分模块（MemberPoint）：**
- 积分获取规则
- 积分消费规则
- 积分过期处理

**会员钱包模块（MemberWallet）：**
- 余额充值
- 余额消费
- 冻结/解冻

### 4.3 数据库设计要点

**用户表设计：**

```sql
CREATE TABLE member_user (
    id BIGINT PRIMARY KEY,
    nickname VARCHAR(50),
    avatar VARCHAR(255),
    mobile VARCHAR(20),
    level INT DEFAULT 0,           -- 当前等级
    experience INT DEFAULT 0,      -- 总经验值
    point INT DEFAULT 0,           -- 当前积分
    balance INT DEFAULT 0,         -- 余额（单位：分）
    frozen_balance INT DEFAULT 0,  -- 冻结余额
    status INT DEFAULT 0,          -- 状态
    create_time DATETIME,
    update_time DATETIME
);
```

**积分记录表设计：**

```sql
CREATE TABLE member_point_record (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    point INT NOT NULL,            -- 变动积分（正数增加，负数扣减）
    biz_type INT NOT NULL,         -- 业务类型
    biz_id VARCHAR(64),            -- 业务编号（用于幂等）
    create_time DATETIME,
    UNIQUE KEY uk_biz (biz_type, biz_id)  -- 幂等索引
);
```

### 4.4 关键设计决策

**为什么余额单位用分而不是元？**
- 避免浮点数精度问题
- 整数运算性能更好
- 与支付系统保持一致

**为什么积分和余额分开设计？**
- 积分有获取和消费的双向流动
- 余额主要是消费和充值
- 两者的业务规则和过期策略不同

**为什么经验值和等级分开存储？**
- 经验值是连续的数值，用于计算等级
- 等级是离散的状态，用于业务逻辑判断
- 分离后可以灵活调整等级规则

### 4.5 反模式与陷阱

**反模式 1：用户信息过度聚合**
```java
// 错误示例：将所有信息都存在用户表
public class MemberUserDO {
    private String address;  // 地址应该单独建表
    private String remark;   // 备注应该单独建表
}
```

**反模式 2：积分扣减不校验**
```java
// 错误示例：直接扣减，不检查余额
public void deductPoint(Long userId, Integer point) {
    memberUserMapper.updatePoint(userId, -point);  // 可能变成负数
}
```

**反模式 3：等级规则硬编码**
```java
// 错误示例：等级规则写在代码里
if (experience >= 5000) {
    return "金卡";
} else if (experience >= 1000) {
    return "银卡";
}
```

## 总结

会员中心模块通过双 Controller 模式实现了管理后台和用户端的职责分离，通过 SQL 原子操作保证了积分/余额的并发安全，通过经验值机制实现了等级自动升降级。这些设计模式为 C 端用户体系提供了坚实的基础。

**核心设计要点：**
1. 双 Controller 模式让同一套 Service 同时服务两个角色，减少代码重复
2. 并发安全通过 SQL 原子操作实现，简单高效
3. 等级体系通过经验值和等级配置解耦，支持灵活调整
4. C 端用户体系需要分层设计，关注数据一致性
