# 设备与数据流实现

## 1. 设备管理实现

### 1.1 设备数据模型

IoT 设备的核心数据对象 `IotDeviceDO` 定义了设备的完整信息：

```java
// 文件位置：yudao-module-iot/yudao-module-iot-server/src/main/java/cn/iocoder/yudao/module/iot/dal/dataobject/device/IotDeviceDO.java

@TableName(value = "iot_device", autoResultMap = true)
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class IotDeviceDO extends TenantBaseDO {

    @TableId
    private Long id;
    
    /**
     * 设备名称，在产品内唯一，用于标识设备
     */
    private String deviceName;
    
    /**
     * 设备备注名称
     */
    private String nickname;
    
    /**
     * 产品编号，关联 IotProductDO
     */
    private Long productId;
    
    /**
     * 产品标识，冗余自 IotProductDO
     */
    private String productKey;
    
    /**
     * 设备类型，冗余自 IotProductDO
     */
    private Integer deviceType;
    
    /**
     * 网关设备编号，子设备需要关联的网关设备 ID
     */
    private Long gatewayId;
    
    /**
     * 设备状态，枚举 IotDeviceStateEnum
     */
    private Integer state;
    
    /**
     * 最后上线时间
     */
    private LocalDateTime onlineTime;
    
    /**
     * 最后离线时间
     */
    private LocalDateTime offlineTime;
    
    /**
     * 设备激活时间
     */
    private LocalDateTime activeTime;
    
    /**
     * 固件编号，关联 IotOtaFirmwareDO
     */
    private Long firmwareId;
    
    /**
     * 设备密钥，用于设备认证
     */
    private String deviceSecret;
    
    /**
     * 设备位置的纬度
     */
    private BigDecimal latitude;
    
    /**
     * 设备位置的经度
     */
    private BigDecimal longitude;
    
    /**
     * 设备配置，JSON 格式，可下发给 device 进行自定义配置
     */
    private String config;
}
```

**关键设计说明**：
- 继承 `TenantBaseDO` 支持多租户隔离
- 冗余 `productKey` 和 `deviceType` 字段，避免频繁关联查询产品表
- `gatewayId` 支持网关-子设备拓扑关系
- `deviceSecret` 用于设备认证，安全性设计

`IotDeviceService` 接口提供了完整的设备管理能力：


**核心功能说明**：

1. **缓存操作**：使用 `Guava LoadingCache` 缓存设备信息，`getDeviceFromCache` 方法支持按 ID 或 `productKey + deviceName` 查询，适用于高频设备消息处理场景。

2. **网关拓扑管理**：
   - 后台操作：管理员手动绑定/解绑子设备到网关
   - 设备上报：网关设备主动上报拓扑关系（添加/删除/查询）

3. **设备动态注册**：
   - 直连设备：一型一密模式，设备通过产品密钥动态获取设备密钥
   - 子设备：通过网关代理转发注册请求

### 1.3 设备控制器层实现

```java
// 文件位置：yudao-module-iot/yudao-module-iot-server/src/main/java/cn/iocoder/yudao/module/iot/controller/admin/device/IotDeviceController.java

@Tag(name = "管理后台 - IoT 设备")
@RestController
@RequestMapping("/iot/device")
@Validated
public class IotDeviceController {

    @Resource
    private IotDeviceService deviceService;
    @Resource
    private IotProductService productService;

    @PostMapping("/create")
    @Operation(summary = "创建设备")
    @PreAuthorize("@ss.hasPermission('iot:device:create')")
    public CommonResult<Long> createDevice(@Valid @RequestBody IotDeviceSaveReqVO createReqVO) {
        return success(deviceService.createDevice(createReqVO));
    }

    @PutMapping("/bind-gateway")
    @Operation(summary = "绑定子设备到网关")
    @PreAuthorize("@ss.hasPermission('iot:device:update')")
    public CommonResult<Boolean> bindDeviceGateway(@Valid @RequestBody IotDeviceBindGatewayReqVO reqVO) {
        deviceService.bindDeviceGateway(reqVO.getSubIds(), reqVO.getGatewayId());
        return success(true);
    }

    @GetMapping("/sub-device-list")
    @Operation(summary = "获取网关的子设备列表")
    @PreAuthorize("@ss.hasPermission('iot:device:query')")
    public CommonResult<List<IotDeviceRespVO>> getSubDeviceList(@RequestParam("gatewayId") Long gatewayId) {
        List<IotDeviceDO> list = deviceService.getDeviceListByGatewayId(gatewayId);
        // 补充产品名称
        Map<Long, IotProductDO> productMap = convertMap(productService.getProductList(), IotProductDO::getId);
        return success(convertList(list, device -> {
            IotDeviceRespVO respVO = BeanUtils.toBean(device, IotDeviceRespVO.class);
            MapUtils.findAndThen(productMap, device.getProductId(),
                    product -> respVO.setProductName(product.getName()));
            return respVO;
        }));
    }

    @GetMapping("/get-auth-info")
    @Operation(summary = "获得设备连接信息")
    @PreAuthorize("@ss.hasPermission('iot:device:auth-info')")
    public CommonResult<IotDeviceAuthInfoRespVO> getDeviceAuthInfo(@RequestParam("id") Long id) {
        return success(deviceService.getDeviceAuthInfo(id));
    }

    @GetMapping("/location-list")
    @Operation(summary = "获取设备位置列表", description = "获取有经纬度信息的设备列表，用于地图展示")
    @PreAuthorize("@ss.hasPermission('iot:device:query')")
    public CommonResult<List<IotDeviceRespVO>> getDeviceLocationList() {
        List<IotDeviceDO> devices = deviceService.getDeviceListByHasLocation();
        // 转换并返回
        Map<Long, IotProductDO> productMap = convertMap(productService.getProductList(), IotProductDO::getId);
        return success(convertList(devices, device -> {
            IotDeviceRespVO respVO = BeanUtils.toBean(device, IotDeviceRespVO.class);
            MapUtils.findAndThen(productMap, device.getProductId(),
                    product -> respVO.setProductName(product.getName()));
            return respVO;
        }));
    }
}
```

**API 设计要点**：
- 统一使用 `CommonResult` 封装响应
- 权限控制：`@PreAuthorize` 注解配合 `@ss` SpEL Bean
- 数据拼接：批量获取关联数据，使用 `MapUtils.findAndThen` 高效填充 VO

### 3.2 消息流转架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         设备消息流转架构                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌──────────────┐      ┌──────────────┐      ┌──────────────┐         │
│   │   设备端      │ ───> │   网关层     │ ───> │   服务端     │         │
│   │  (MQTT/CoAP) │      │  (协议适配)  │      │  (业务处理)  │         │
│   └──────────────┘      └──────────────┘      └──────────────┘         │
│                                  │                    │                 │
│                                  ▼                    ▼                 │
│                          ┌──────────────┐      ┌──────────────┐         │
│                          │   消息总线   │      │   时序数据库 │         │
│                          │ (Redis/RMQ)  │      │  (TDengine)  │         │
│                          └──────────────┘      └──────────────┘         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**消息处理流程**：

1. **上行消息**（设备 → 服务端）：
   - 设备通过 MQTT/CoAP/HTTP 协议发送消息到网关
   - 网关解码消息，调用 `handleUpstreamDeviceMessage`
   - 服务端处理业务逻辑（属性更新、事件上报、OTA 等）
   - 消息持久化到 TDengine 时序数据库

2. **下行消息**（服务端 → 设备）：
   - 服务端调用 `sendDeviceMessage` 发送指令
   - 消息通过消息总线投递到网关
   - 网关编码消息，通过对应协议发送到设备

**上行消息处理流程**：


## 总结

### 核心实现要点

1. **设备管理**：完整的 CRUD + 缓存优化 + 多租户支持
2. **设备认证**：安全的密钥认证机制 + 动态注册
3. **消息处理**：支持多协议（MQTT/CoAP/HTTP）+ 高性能消息总线
4. **时序存储**：TDengine 存储设备消息，支持海量数据
5. **OTA 升级**：完整的固件管理和升级流程
6. **告警系统**：灵活的告警配置 + 实时告警触发

### 关键技术选型

| 场景 | 技术选型 | 说明 |
|------|---------|------|
| 消息协议 | MQTT (EMQX) | 物联网标准协议，低功耗、高可靠 |
| 时序数据库 | TDengine | 国产高性能时序数据库 |
| 消息总线 | Redis/RocketMQ | 支持本地/分布式两种模式 |
| 设备缓存 | Guava LoadingCache | 高频访问场景优化 |

### 扩展建议

1. **协议扩展**：实现 `IotProtocol` 接口即可添加新协议
2. **消息总线扩展**：实现 `IotMessageBus` 接口即可接入新消息中间件
3. **物模型扩展**：支持自定义物模型属性和事件
