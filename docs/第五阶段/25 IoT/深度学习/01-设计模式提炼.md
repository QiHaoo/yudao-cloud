# IoT 模块设计模式提炼

## 一、多协议适配器模式（MQTT/CoAP/HTTP）

### 1.1 什么是多协议适配器

IoT 平台需要支持多种设备通信协议：MQTT 适合低带宽场景、CoAP 适合受限设备、HTTP 适合传统设备。多协议适配器模式的核心是**统一接口 + 协议实现分离**。

```
│   │                    协议管理器                            │  │
│   │              IotProtocolManager                         │  │

│                             │                                   │
│          ┌──────────────────┼──────────────────┐               │

│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐        │
│   │   HTTP      │   │   MQTT      │   │   CoAP      │        │
│   │   协议      │   │   协议      │   │   协议      │        │
```

### 1.2 设计模式：策略模式 + 工厂模式

项目采用**策略模式**定义协议接口，**工厂模式**根据配置创建协议实例。

**源码位置**：`yudao-module-iot-gateway/.../protocol/IotProtocol.java`

```java
/**
 * IoT 协议接口
 *
 * 定义传输层协议的生命周期管理
 */
public interface IotProtocol {

    /**
     * 获取协议实例 ID
     */
    String getId();

    /**
     * 获取服务器 ID（用于消息追踪，全局唯一）
     */
    String getServerId();

    /**
     * 获取协议类型
     */
    IotProtocolTypeEnum getType();

    /**
     * 启动协议服务
     */
    void start();

    /**
     * 停止协议服务
     */
    void stop();

    /**
     * 检查协议服务是否正在运行
     */
    boolean isRunning();
}
```

### 1.3 协议管理器实现

**源码位置**：`yudao-module-iot-gateway/.../protocol/IotProtocolManager.java`

```java
@Slf4j
public class IotProtocolManager implements SmartLifecycle {

    private final IotGatewayProperties gatewayProperties;

    /**
     * 协议实例列表
     */
    private final List<IotProtocol> protocols = new ArrayList<>();

    @Getter
    private volatile boolean running = false;

    @Override
    public void start() {
        if (running) {
            return;
        }
        List<IotGatewayProperties.ProtocolProperties> protocolConfigs = gatewayProperties.getProtocols();
        if (CollUtil.isEmpty(protocolConfigs)) {
            log.info("[start][没有配置协议实例，跳过启动]");
            return;
        }

        for (IotGatewayProperties.ProtocolProperties config : protocolConfigs) {
            if (BooleanUtil.isFalse(config.getEnabled())) {
                log.info("[start][协议实例 {} 未启用，跳过]", config.getId());
                continue;
            }
            IotProtocol protocol = createProtocol(config);
            if (protocol == null) {
                continue;
            }
            protocol.start();
            protocols.add(protocol);
        }
        running = true;
        log.info("[start][协议管理器启动完成，共启动 {} 个协议实例]", protocols.size());
    }

    @Override
    public void stop() {
        if (!running) {
            return;
        }
        for (IotProtocol protocol : protocols) {
            try {
                protocol.stop();
            } catch (Exception e) {
                log.error("[stop][协议实例 {} 停止失败]", protocol.getId(), e);
            }
        }
        protocols.clear();
        running = false;
        log.info("[stop][协议管理器已停止]");
    }

    /**
     * 创建协议实例
     */
    private IotProtocol createProtocol(IotGatewayProperties.ProtocolProperties config) {
        IotProtocolTypeEnum protocolType = IotProtocolTypeEnum.of(config.getProtocol());
        if (protocolType == null) {
            log.error("[createProtocol][协议实例 {} 的协议类型 {} 不存在]",
                config.getId(), config.getProtocol());
            return null;
        }
        switch (protocolType) {
            case HTTP:
                return createHttpProtocol(config);
            case TCP:
                return createTcpProtocol(config);
            case UDP:
                return createUdpProtocol(config);
            case COAP:
                return createCoapProtocol(config);
            case WEBSOCKET:
                return createWebSocketProtocol(config);
            case MQTT:
                return createMqttProtocol(config);
            case EMQX:
                return createEmqxProtocol(config);
            case MODBUS_TCP_CLIENT:
                return createModbusTcpClientProtocol(config);
            case MODBUS_TCP_SERVER:
                return createModbusTcpServerProtocol(config);
            default:
                throw new IllegalArgumentException(String.format(
                    "[createProtocol][协议实例 %s 的协议类型 %s 暂不支持]",
                    config.getId(), protocolType));
        }
    }
}
```

### 1.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 统一接口 | 所有协议实现相同的生命周期接口 |
| 配置驱动 | 通过配置文件决定启用哪些协议 |
| SmartLifecycle | 集成 Spring 生命周期，自动启停 |
| 失败隔离 | 单个协议启动失败不影响其他协议 |

**迁移建议**：任何需要支持多种协议/格式的场景都可以使用此模式。例如：消息队列适配（Kafka/RabbitMQ/RocketMQ）、数据库适配（MySQL/PostgreSQL/Oracle）。

## 二、设备影子（Device Shadow）设计

### 2.1 什么是设备影子

设备影子是 IoT 平台的核心概念。当设备离线时，云端仍然保存设备的"影子"状态，应用可以通过影子下发指令，设备上线后自动同步。

```

│                       设备影子机制                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────┐                    ┌─────────────┐           │
│   │   设备      │                    │   云端      │           │
│   │  (离线)     │                    │  (影子)     │           │
│   └──────┬──────┘                    └──────┬──────┘           │
│          │                                  │                   │
│          │   1. 设备离线                     │                   │
│          │ ───────────────────────────────> │                   │
│          │                                  │                   │
│          │                    2. 应用下发指令 │                   │
│          │                    (更新影子)     │                   │
│          │                    <──────────── │                   │
│          │                                  │                   │
│          │   3. 设备上线                     │                   │

│          │                                  │                   │
│          │   4. 同步影子状态                 │                   │
```


### 2.3 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 影子状态持久化 | 设备离线时仍可查询和修改 |
| 版本号控制 | 避免配置冲突，支持乐观锁 |
| 差分同步 | 只同步变化的属性，减少数据传输 |
| 时间戳标记 | 支持数据排序和过期判断 |

**迁移建议**：任何需要"离线状态同步"的场景都可以使用此模式。例如：移动端离线缓存、分布式配置中心、多人协作编辑。

## 三、时序数据存储策略

### 3.1 什么是时序数据

IoT 设备产生的数据具有明显的时间序列特征：温度、湿度、位置等数据按时间顺序产生，查询时也通常按时间范围查询。

### 3.2 设计模式：时间分片 + 消息总线

项目采用**消息总线**模式处理设备上报数据，支持多种后端存储。

**源码位置**：`yudao-module-iot-core/.../messagebus/core/IotMessageBus.java`

```java
/**
 * IoT 消息总线接口
 */
public interface IotMessageBus {

    /**
     * 发送消息
     *
     * @param topic 消息主题
     * @param message 消息内容
     */
    void send(String topic, IotLocalMessage message);

    /**
     * 订阅消息
     *
     * @param topic 消息主题
     * @param subscriber 消息订阅者
     */
    void subscribe(String topic, IotMessageSubscriber subscriber);
}
```

**本地消息总线实现**：

```java
// yudao-module-iot-core/.../messagebus/core/local/IotLocalMessageBus.java
public class IotLocalMessageBus implements IotMessageBus {

    private final Map<String, List<IotMessageSubscriber>> subscribers = new ConcurrentHashMap<>();

    @Override
    public void send(String topic, IotLocalMessage message) {
        List<IotMessageSubscriber> topicSubscribers = subscribers.get(topic);
        if (CollUtil.isEmpty(topicSubscribers)) {
            return;
        }
        for (IotMessageSubscriber subscriber : topicSubscribers) {
            try {
                subscriber.onMessage(message);
            } catch (Exception e) {
                log.error("[onMessage][主题 {} 订阅者处理异常]", topic, e);
            }
        }
    }

    @Override
    public void subscribe(String topic, IotMessageSubscriber subscriber) {
        subscribers.computeIfAbsent(topic, k -> new CopyOnWriteArrayList<>())
            .add(subscriber);
    }
}
```

### 3.3 消息主题设计

项目定义了多种设备消息主题：


### 3.4 可迁移经验

**设计要点**：

| 设计决策 | 理由 |
|---------|------|
| 消息总线解耦 | 设备数据采集与存储分离，灵活扩展 |
| 主题路由 | 按设备类型、消息类型路由，支持订阅过滤 |
| 本地/分布式可切换 | 开发环境用本地，生产环境用 Redis/RocketMQ |
| 异步处理 | 消息异步消费，不阻塞设备上报 |

**迁移建议**：任何需要处理"时序数据"或"事件流"的场景都可以使用此模式。例如：日志收集、监控指标、用户行为追踪。
