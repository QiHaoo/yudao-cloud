# 01 - 总览：IoT 模块全景

## 场景引入

你做了一个电商系统，现在想扩展智能硬件业务——卖智能插座、智能灯泡、温湿度传感器。用户买了设备后，需要：

- 在 App 上查看设备是否在线
- 远程控制设备（开关灯、调节亮度）
- 查看设备上报的数据（温度、湿度、用电量）
- 设备异常时收到告警通知

这些功能都需要一个 **IoT（物联网）平台**来支撑。

### 为什么电商系统需要 IoT？

电商卖硬件设备越来越常见：

- **智能家居**：智能插座、智能灯泡、扫地机器人
- **工业设备**：传感器、PLC、工业网关
- **农业设备**：土壤传感器、自动灌溉控制器
- **共享设备**：共享充电宝、共享按摩椅

核心价值是：**让设备成为"可管理的商品"，而不只是一次性卖出的硬件**。

## 核心能力分层

### 1. 产品与物模型

产品是设备的"模板"。比如"智能温湿度传感器"是一个产品，卖出的 1000 个传感器是 1000 个设备。

```
产品 (IotProductDO)
  │
  ├── 产品标识 (productKey)   ← 唯一标识，用于设备认证
  ├── 产品密钥 (productSecret) ← 用于一型一密动态注册
  ├── 设备类型 (deviceType)    ← 直连设备 / 网关设备 / 网关子设备
  ├── 联网方式 (netType)       ← WiFi / 蜂窝 / 以太网 / LoRa
  ├── 协议类型 (protocolType)  ← MQTT / Modbus / TCP / WebSocket
  │
  └── 物模型 (ThingModel)
        ├── 属性 (Property)    ← 设备的状态，如"温度: 25.5°C"
        ├── 服务 (Service)     ← 设备的能力，如"设置亮度: 80%"
        └── 事件 (Event)       ← 设备的通知，如"温度过高告警"
```

**物模型是 IoT 的核心概念**——它定义了"这个设备能做什么、会报告什么"。以智能灯泡为例：

```
智能灯泡物模型
  │
  ├── 属性
  │     ├── power (bool)      - 开关状态
  │     ├── brightness (int)  - 亮度 (0-100)
  │     └── color (string)    - 颜色 (#FFFFFF)
  │
  ├── 服务
  │     ├── setPower (bool)   - 设置开关
  │     ├── setBrightness (int) - 设置亮度
  │     └── setColor (string) - 设置颜色
  │
  └── 事件
        ├── offline ()        - 设备离线
        └── error (code)      - 设备故障
```

### 2. 设备管理

```
设备 (IotDeviceDO)
  │
  ├── 设备标识
  │     ├── deviceName        - 设备名称（产品内唯一）
  │     ├── productKey        - 产品标识（冗余自 ProductDO）
  │     ├── deviceSecret      - 设备密钥（用于认证）
  │     └── serialNumber      - 设备序列号
  │
  ├── 设备状态
  │     ├── state             - 在线/离线/未激活
  │     ├── onlineTime        - 最后上线时间
  │     ├── offlineTime       - 最后离线时间
  │     └── activeTime        - 激活时间
  │
  ├── 设备关系
  │     ├── productId         - 产品编号
  │     ├── gatewayId         - 网关设备编号（子设备用）
  │     └── groupIds          - 设备分组集合
  │
  └── 设备位置
        ├── latitude          - 纬度
        └── longitude         - 经度
```

### 3. 规则引擎

规则引擎定义"设备数据来了之后怎么处理"：

```
规则引擎
  │
  ├── 数据规则 (IotDataRuleDO)
  │     └── 定义数据过滤条件
  │         例：当温度 > 35°C 时触发
  │
  ├── 场景规则 (IotSceneRuleDO)
  │     └── 定义联动动作
  │         例：当温度 > 35°C 时，自动打开空调
  │
  └── 数据流转 (IotDataSinkDO)
        └── 定义数据去向
            例：将设备数据写入 MySQL / 推送到 Kafka / 调用 HTTP 接口
```

数据流转支持多种目标：

| 数据目标 | 配置类 | 说明 |
|---------|--------|------|
| MySQL | `IotDataSinkDatabaseConfig` | 写入数据库表 |
| HTTP | `IotDataSinkHttpConfig` | 调用 HTTP 接口 |
| Kafka | `IotDataSinkKafkaConfig` | 推送到 Kafka 主题 |
| MQTT | `IotDataSinkMqttConfig` | 转发到另一个 MQTT 主题 |
| RabbitMQ | `IotDataSinkRabbitMQConfig` | 推送到 RabbitMQ 队列 |
| Redis | `IotDataSinkRedisConfig` | 写入 Redis |
| RocketMQ | `IotDataSinkRocketMQConfig` | 推送到 RocketMQ |
| TCP | `IotDataSinkTcpConfig` | 转发到 TCP 服务 |
| WebSocket | `IotDataSinkWebSocketConfig` | 推送到 WebSocket |

### 4. 告警与运维

```
告警体系
  │
  ├── 告警配置 (IotAlertConfigDO)
  │     ├── 关联产品/设备
  │     ├── 触发条件（如"温度 > 40°C"）
  │     └── 通知方式（站内信/短信/邮件）
  │
  ├── 告警记录 (IotAlertRecordDO)
  │     ├── 告警时间
  │     ├── 告警内容
  │     └── 处理状态
  │
  └── OTA 升级
        ├── 固件管理 (IotOtaFirmwareDO)
        │     └── 固件版本、文件地址
        ├── 升级任务 (IotOtaTaskDO)
        │     └── 目标设备、升级策略
        └── 升级记录 (IotOtaTaskRecordDO)
              └── 每个设备的升级状态
```

## 设备认证机制

### 一机一密

每个设备出厂时预烧录唯一的 `deviceSecret`。设备连接时用 `productKey + deviceName + deviceSecret` 认证。

```
设备连接流程
  │
  ├── 1. 设备发送认证请求
  │     ├── productKey: "a1B2c3D4"
  │     ├── deviceName: "sensor_001"
  │     └── 签名: HMAC-SHA256(deviceSecret, ...)
  │
  ├── 2. IoT 平台验证
  │     ├── 根据 productKey + deviceName 查找 IotDeviceDO
  │     ├── 用 deviceSecret 验证签名
  │     └── 校验设备状态（是否已激活、是否被禁用）
  │
  └── 3. 认证通过 → 建立连接
```

### 一型一密动态注册

产品级别的密钥，设备首次连接时动态获取设备密钥：

```
动态注册流程
  │
  ├── 1. 产品开启 registerEnabled
  ├── 2. 设备用 productKey + productSecret 请求注册
  ├── 3. IoT 平台分配 deviceSecret
  └── 4. 设备保存 deviceSecret，后续用一机一密认证
```

## 设备类型

```
设备类型 (deviceType)
  │
  ├── 直连设备 (DEVICE)
  │     → 直接连接 IoT 平台
  │     → 例：WiFi 温湿度传感器
  │
  ├── 网关设备 (GATEWAY)
  │     → 作为子设备的代理连接 IoT 平台
  │     → 例：Zigbee 网关
  │
  └── 网关子设备 (GATEWAY_SUB_DEVICE)
        → 通过网关设备间接连接
        → 例：Zigbee 门磁传感器（通过 Zigbee 网关上报数据）
```

## 电商场景举例

1. **商品关联设备**：商城里的"智能温湿度传感器"商品，关联 IoT 产品 `productKey`
2. **用户购买后激活**：用户收到设备，扫码绑定到自己的账号，设备自动激活
3. **数据上报**：设备每 30 秒上报一次温度和湿度数据
4. **规则触发**：温度超过 35°C 时，自动发送告警通知给用户
5. **远程控制**：用户在 App 上点击"打开空调"，平台下发控制指令给空调设备
6. **OTA 升级**：设备固件有新版本时，平台推送升级任务给设备

## 思考题

1. 为什么要把"产品"和"设备"分开管理，而不是直接管理设备？
2. 物模型（ThingModel）的作用是什么？如果没有物模型，设备数据怎么解析？
3. 数据流转（DataSink）支持写入数据库、推送到消息队列等多种目标。为什么不在设备上报时直接写入数据库？
4. 网关设备和直连设备有什么区别？什么场景需要用网关？
