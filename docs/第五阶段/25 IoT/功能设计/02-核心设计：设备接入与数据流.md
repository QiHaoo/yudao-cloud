# 02 - 核心设计：设备接入与数据流

## 场景引入

上一章我们看到了 IoT 的全景——产品、设备、规则引擎、告警。这一章深入几个核心设计问题：

1. **设备认证**：设备怎么证明自己是合法的？密钥怎么管理？
2. **物模型**：设备上报的 JSON 数据怎么解析？怎么定义"温度"和"湿度"？
3. **设备消息流转**：设备上报的数据经过哪些环节？最终存到哪里？
4. **设备影子**：设备离线时，怎么缓存最后一次状态？

## 设备认证：三元组与密钥管理

### 设备三元组

每个设备有三个核心标识，称为"三元组"：

```
设备三元组
  │
  ├── productKey    - 产品标识（同一个产品的所有设备共享）
  ├── deviceName    - 设备名称（产品内唯一）
  └── deviceSecret  - 设备密钥（每个设备唯一）
```

认证时，设备用 `productKey + deviceName + deviceSecret` 生成签名，IoT 平台验证签名的合法性。

### 认证流程

```
设备                          IoT 平台
 │                              │
 │  CONNECT(productKey,         │
 │    deviceName, signature)    │
 │─────────────────────────────>│
 │                              │
 │                              │  1. 根据 productKey + deviceName
 │                              │     查找 IotDeviceDO
 │                              │
 │                              │  2. 用 deviceSecret 验证签名
 │                              │     ├── 通过 → 继续
 │                              │     └── 失败 → 拒绝连接
 │                              │
 │                              │  3. 检查设备状态
 │                              │     ├── 已激活 → 允许连接
 │                              │     └── 未激活 → 拒绝连接
 │                              │
 │  CONNACK(成功)               │
 │<─────────────────────────────│
 │                              │
 │  设备上线 → 更新 state = ONLINE
 │  记录 onlineTime
```

### 密钥存储

```java
// IotDeviceDO 中的认证字段
private String deviceSecret;    // 设备密钥
private Long firmwareId;        // 固件编号
```

**为什么 deviceSecret 需要加密存储？**

设备密钥是设备的"身份证"，泄露后可以伪造设备。在数据库中应该加密存储，查询时解密使用。

## 物模型：设备能力的标准化描述

### 物模型的三个维度

```
物模型 (ThingModel)
  │
  ├── 属性 (Property)
  │     → 设备的"状态"
  │     → 例：当前温度 = 25.5°C
  │     → 例：开关状态 = 开
  │
  ├── 服务 (Service)
  │     → 设备的"能力"
  │     → 例：设置亮度(参数: brightness)
  │     → 例：重启设备()
  │
  └── 事件 (Event)
        → 设备的"通知"
        → 例：温度过高告警(参数: currentTemp, threshold)
        → 例：设备离线()
```

### 属性定义示例

以智能温湿度传感器为例：

```json
    },
    {

      "dataType": "double",

    },
    {
      "identifier": "power",
```


### 物模型与设备消息的关联

设备上报的数据必须符合物模型的定义：

```
设备上报消息：
  {
    "method": "property.report",
    "params": {
      "temperature": 25.5,
      "humidity": 65.2
    }
  }

解析过程：
  │
  ├── 1. 根据 deviceId 查找 IotProductDO → 获取 productId
  ├── 2. 根据 productId 查找物模型
  ├── 3. 按物模型定义解析 params
  │     ├── temperature: double → 25.5 ✓
  │     ├── humidity: double → 65.2 ✓
  │     └── unknown_field → 忽略（物模型中没有定义）
  └── 4. 存储到 IotDevicePropertyDO
```

## 设备消息流转

### 消息类型

```
设备消息类型
  │
  ├── 属性上报 (property.report)
  │     → 设备主动上报属性值
  │     → 例：{"temperature": 25.5}
  │
  ├── 属性设置 (property.set)
  │     → 平台下发属性设置指令
  │     → 例：{"power": true}
  │
  ├── 服务调用 (service.invoke)
  │     → 平台调用设备服务
  │     → 例：setBrightness(80)
  │
  └── 事件上报 (event.report)
        → 设备上报事件
        → 例：{"errorCode": "TEMP_HIGH"}
```

### 消息流转链路

```
设备上报消息
  │
  ▼
协议层接收 (MQTT/TCP/Modbus/WebSocket)
  │
  ▼
消息解析
  │
  ├── 1. 识别设备（productKey + deviceName）
  ├── 2. 认证校验
  ├── 3. 解析消息体
  │
  ▼
消息存储
  │
  ├── IotDeviceMessageDO (消息记录)
  │     └── 存储原始消息，用于调试和追溯
  │
  ├── IotDevicePropertyDO (属性快照)
  │     └── 更新设备最新属性值
  │
  └── 设备状态更新
        ├── state = ONLINE
        └── onlineTime = 当前时间
  │
  ▼
规则引擎
  │
  ├── 数据规则 (IotDataRuleDO)
  │     └── 过滤条件匹配？
  │           ├── 匹配 → 执行数据流转
  │           └── 不匹配 → 结束
  │
  ├── 场景规则 (IotSceneRuleDO)
  │     └── 联动条件匹配？
  │           ├── 匹配 → 执行联动动作
  │           └── 不匹配 → 结束
  │
  └── 数据流转 (IotDataSinkDO)
        ├── 写入数据库
        ├── 推送到消息队列
        ├── 调用 HTTP 接口
        └── 转发到其他 MQTT 主题
```

### 消息实体

```java
// IotDeviceMessageDO - 设备消息
public class IotDeviceMessageDO extends BaseDO {
    private Long id;            // 消息编号
    private Long deviceId;      // 设备编号
    private Long productId;     // 产品编号
    private String productKey;  // 产品标识
    private String deviceName;  // 设备名称
    private Integer direction;  // 消息方向（上行/下行）
    private Integer type;       // 消息类型（属性/服务/事件）
    private String content;     // 消息内容（JSON）
}
```

### 属性快照

```java
// IotDevicePropertyDO - 设备属性（最新值）
public class IotDevicePropertyDO extends BaseDO {
    private Long id;            // 编号
    private Long deviceId;      // 设备编号
    private String identifier;  // 属性标识（如 "temperature"）
    private String value;       // 属性值（字符串形式）
    private Long reportTime;    // 上报时间
}
```

**为什么属性快照只存最新值？**

因为属性查询的高频场景是"这个设备现在温度多少？"，而不是"这个设备 10 分钟前温度多少？"。历史数据通过 `IotDeviceMessageDO` 或数据流转到外部存储查询。

## 设备影子

### 设计动机

设备可能离线（网络断了、设备关机）。但用户想"设置灯泡为红色"，即使设备当前离线。

设备影子（Device Shadow）是一个 JSON 文档，存储设备的"期望状态"和"报告状态"：

```
设备影子 (Device Shadow)
  │
  ├── desired (期望状态)
  │     → 用户/平台希望设备达到的状态
  │     → 例：{"power": true, "brightness": 80}
  │
  └── reported (报告状态)
        → 设备实际上报的状态
        → 例：{"power": true, "brightness": 50}
```

### 影子同步流程

```
场景：用户设置灯泡亮度为 80，但设备离线

1. 用户调用 API → 平台更新 desired = {"brightness": 80}
2. 设备当前离线 → 不下发指令

...

3. 设备重新上线
4. 设备请求同步影子
5. 平台返回 desired = {"brightness": 80}
6. 设备执行 → 设置亮度为 80
7. 设备上报 → reported = {"brightness": 80}
8. 平台更新 reported
```

## 规则引擎设计

### 数据规则

数据规则定义"什么条件的数据触发什么动作"：

```
数据规则示例
  │
  ├── 条件：产品 = 温湿度传感器 AND 温度 > 35
  │
  └── 动作：
        ├── 数据流转 → 写入告警数据库
        ├── 数据流转 → 推送到 Kafka 主题
        └── 数据流转 → 调用 HTTP 接口（通知业务系统）
```

### 场景规则

场景规则定义"设备之间的联动"：

```
场景规则示例
  │
  ├── 触发条件：
  │     设备 A 的温度 > 35°C
  │
  └── 联动动作：
        ├── 设置设备 B 的 power = true（打开空调）
        ├── 设置设备 C 的 power = true（打开风扇）
        └── 发送通知给用户
```

### 数据流转配置

数据流转支持多种目标，通过配置类实现：

```java
// 基类
public abstract class IotAbstractDataSinkConfig {
    // 公共配置

// 数据库目标
public class IotDataSinkDatabaseConfig extends IotAbstractDataSinkConfig {

    private String username;    // 数据库用户名
    private String password;    // 数据库密码
    private String tableName;   // 目标表名

// HTTP 目标
public class IotDataSinkHttpConfig extends IotAbstractDataSinkConfig {
    private String url;         // HTTP 接口地址
    private String method;      // 请求方法（POST/GET）
    private String headers;     // 请求头
}

// Kafka 目标
public class IotDataSinkKafkaConfig extends IotAbstractDataSinkConfig {
    private String bootstrapServers; // Kafka 地址
    private String topic;            // 主题名称
}
```

## Modbus 协议支持

### Modbus 简介

Modbus 是工业设备常用的通信协议。IoT 模块支持通过 Modbus 网关采集工业设备数据。

```
Modbus 架构
  │
  ├── Modbus 网关（IoT 设备，类型 = GATEWAY）
  │     └── 连接 IoT 平台
  │
  └── Modbus 从设备（IoT 设备，类型 = GATEWAY_SUB_DEVICE）
        ├── 温度传感器 (寄存器地址: 0x0001)
        ├── 压力传感器 (寄存器地址: 0x0002)
        └── 流量计 (寄存器地址: 0x0003)
```

### Modbus 配置

```java
// IotDeviceModbusConfigDO - Modbus 配置
public class IotDeviceModbusConfigDO extends BaseDO {
    private Long deviceId;      // 设备编号
    private String serialPort;  // 串口
    private Integer baudRate;   // 波特率
    private Integer dataBits;   // 数据位
    private Integer stopBits;   // 停止位
    private String parity;      // 校验位
}

// IotDeviceModbusPointDO - Modbus 点位
public class IotDeviceModbusPointDO extends BaseDO {
    private Long deviceId;      // 设备编号
    private String identifier;  // 点位标识（对应物模型属性）
    private Integer functionCode; // 功能码（01/02/03/04）
    private Integer address;    // 寄存器地址
    private Integer dataType;   // 数据类型
}
```

## 设备分组

```java
// IotDeviceGroupDO - 设备分组
public class IotDeviceGroupDO extends TenantBaseDO {
    private Long id;            // 分组编号
    private String name;        // 分组名称
    private String description; // 分组描述
}
```

设备与分组是多对多关系，通过 `IotDeviceDO.groupIds`（逗号分隔）存储。

分组的使用场景：
- "所有温湿度传感器" → 批量下发固件升级
- "A 区域的设备" → 按区域查看设备状态
- "VIP 客户的设备" → 优先技术支持

## 思考题

1. 设备认证用 `productKey + deviceName + deviceSecret` 三元组。如果设备密钥泄露了，怎么处理？
2. 物模型的属性定义了 `accessMode: "r"`（只读）和 `"rw"`（读写）。如果设备上报了一个只读属性的值，平台怎么处理？
3. 设备影子的 `desired` 和 `reported` 不一致时，说明什么？应该怎么处理？
4. Modbus 协议是轮询模式（平台主动读取），MQTT 是推送模式（设备主动上报）。两种模式在数据实时性上有什么区别？
