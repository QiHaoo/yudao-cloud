# 01 - 设计模式提炼：声明式 RPC 调用

本文从声明式 RPC 调用组件中提炼可复用的设计模式，帮助读者理解 OpenFeign 背后的设计思想，以及 *-api 模块设计的工程价值。

## 代理模式：Feign 客户端的 JDK 动态代理

### 模式识别

OpenFeign 的核心是**代理模式（Proxy Pattern）**。当你注入一个 Feign 客户端接口时，Spring 容器中实际存放的是一个 JDK 动态代理对象：

```java
@Resource
private ProductSpuApi productSpuApi;  // 实际是 $Proxy123 代理对象

// 调用时，代理对象拦截方法调用，转换为 HTTP 请求
CommonResult<ProductSpuRespVO> result = productSpuApi.getSpu(1L);
```

代理模式的结构：

```
经典代理模式：
  Client ──调用──> Proxy ──转发──> RealSubject
                (实现同一接口)

Feign 代理：
  OrderService ──调用──> FeignProxy ──HTTP──> ProductController
                   (实现 ProductSpuApi 接口)
```

### 代理模式的三种角色

| 角色 | Feign 中的体现 | 职责 |
|------|---------------|------|
| 抽象主题（Subject） | `ProductSpuApi` 接口 | 定义契约 |
| 代理主题（Proxy） | Feign 生成的 `$Proxy` 对象 | 拦截方法调用，转换为 HTTP |
| 真实主题（RealSubject） | `ProductSpuController` | 实际的业务逻辑 |

### 为什么用 JDK 动态代理而不是 CGLIB

| 维度 | JDK 动态代理 | CGLIB |
|------|-------------|-------|
| 代理方式 | 基于接口 | 基于继承 |
| 要求 | 必须有接口 | 不能代理 final 类 |
| 性能 | 较好 | 略慢（需要生成子类） |
| Spring 中的角色 | 接口代理的默认选择 | 类代理的默认选择 |

Feign 客户端是接口（`interface ProductSpuApi`），天然适合 JDK 动态代理。

### 通用化提炼

代理模式适合以下场景：

| 特征 | Feign 中的体现 |
|------|---------------|
| 调用方不关心实现细节 | 只需调用接口方法 |
| 需要在调用前后做额外处理 | 添加租户 ID、Token、日志等 |
| 实现方在远程（或不同上下文） | 远程 HTTP 服务 |
| 需要统一的错误处理 | ErrorDecoder 统一翻译异常 |

### 电商场景举例：缓存代理

假设商品服务需要频繁查询商品信息，可以使用代理模式实现透明缓存：

```java
// 抽象主题
public interface ProductService {
    ProductDTO getProduct(Long id);
}

// 真实主题
public class ProductServiceImpl implements ProductService {
    public ProductDTO getProduct(Long id) {
        return productMapper.selectById(id);
    }
}

// 缓存代理
public class CachingProductServiceProxy implements ProductService {
    private final ProductService delegate;
    private final Cache<Long, ProductDTO> cache;

    public ProductDTO getProduct(Long id) {
        return cache.get(id, key -> delegate.getProduct(key));
    }
}
```

调用方代码无需修改，缓存对调用方完全透明。

## 工厂模式：Feign 代理的创建

### 模式识别

Feign 代理对象的创建过程是一个典型的**工厂模式**应用。Spring Cloud OpenFeign 使用 `FeignClientFactoryBean` 来创建 Feign 客户端代理：

```
FeignClientFactoryBean（工厂）
    │
    ├── 读取 @FeignClient 注解信息
    ├── 创建 Feign.Builder
    ├── 配置 Encoder / Decoder / ErrorDecoder
    ├── 配置 RequestInterceptor 链
    ├── 配置 LoadBalancer
    └── 调用 Feign.build() 创建代理对象
```

### 工厂模式的结构

```
经典工厂模式：
  Client ──请求──> Factory ──创建──> Product

Feign 工厂：
  Spring Container ──启动──> FeignClientFactoryBean ──创建──> Feign Proxy
```

工厂模式将对象的创建过程封装起来，调用方不需要知道代理对象是如何创建的。

### 通用化提炼

工厂模式适合以下场景：

| 特征 | Feign 中的体现 |
|------|---------------|
| 对象创建过程复杂 | 需要配置多个组件（Encoder、Decoder、拦截器等） |
| 同一类对象有多个实例 | 每个 @FeignClient 接口一个代理 |
| 创建过程需要解耦 | Spring 管理生命周期，业务代码不关心创建 |

## 拦截器链模式：Feign 请求拦截器

### 模式识别

Feign 的 `RequestInterceptor` 是**拦截器链模式（Interceptor Chain Pattern）**的应用。每个拦截器在请求发送前执行，可以修改请求内容：

```
Feign 请求发送流程：

  构建 HTTP 请求
      │
      ▼
  TenantRequestInterceptor
      │  添加 tenant-id header
      ▼
  DataPermissionRequestInterceptor
      │  添加 data-permission header
      ▼
  LoginUserRequestInterceptor
      │  添加 Authorization header
      ▼
  其他自定义拦截器
      │
      ▼
  发送 HTTP 请求
```

### 拦截器链 vs 责任链

| 维度 | 拦截器链 | 责任链 |
|------|---------|--------|
| 处理方式 | 所有拦截器都执行 | 通常只有一个处理器处理 |
| 目的 | 增强请求（添加 header 等） | 决定谁来处理请求 |
| 终止条件 | 遍历完所有拦截器 | 某个处理器处理后终止 |
| 典型应用 | Servlet Filter、Feign Interceptor | 权限校验、审批流程 |

### 通用化提炼

拦截器链模式适合以下场景：

| 特征 | Feign 中的体现 |
|------|---------------|
| 需要在请求前后做通用处理 | 添加租户 ID、Token、日志 |
| 处理逻辑可插拔 | 注册不同的 RequestInterceptor |
| 处理逻辑之间独立 | 每个拦截器只关注自己的职责 |
| 处理顺序可控 | 通过 @Order 控制顺序 |

### 电商场景举例：订单创建的前置校验

```java
public interface OrderCreationInterceptor {
    void beforeCreate(OrderDO order);
    int getOrder();  // 执行顺序
}

// 库存校验拦截器
@Component
public class StockValidationInterceptor implements OrderCreationInterceptor {
    public void beforeCreate(OrderDO order) {
        // 校验库存是否充足
    }
    public int getOrder() { return 1; }
}

// 优惠校验拦截器
@Component
public class PromotionValidationInterceptor implements OrderCreationInterceptor {
    public void beforeCreate(OrderDO order) {
        // 校验优惠是否有效
    }
    public int getOrder() { return 2; }
}

// 风控校验拦截器
@Component
public class RiskControlInterceptor implements OrderCreationInterceptor {
    public void beforeCreate(OrderDO order) {
        // 风控检查
    }
    public int getOrder() { return 3; }
}
```

## API 模块设计：契约优先

### 模式识别

*-api 模块的设计体现了**契约优先（Contract First）**的设计思想：先定义接口契约（Feign 接口 + DTO），再分别实现调用方和被调用方。

```
契约优先的开发流程：

  1. 定义 API 契约（*-api 模块）
     ├── Feign 接口定义
     ├── 请求/响应 DTO 定义
     └── 错误码定义
         │
         ├── 2. 调用方依赖 API 模块
         │   └── 注入 Feign 客户端，直接调用
         │
         └── 3. 实现方依赖 API 模块
             └── 实现 Feign 接口，提供业务逻辑
```

### 与契约优先的对比

| 维度 | 代码优先 | 契约优先（*-api 模块） |
|------|---------|---------------------|
| 开发顺序 | 先写实现，再生成接口 | 先定义接口，再分别实现 |
| 接口一致性 | 可能不一致 | 强制一致（编译时检查） |
| 并行开发 | 不支持 | 支持（定义契约后可并行） |
| 文档 | 后补 | 自动生成（Swagger） |

### *-api 模块的分层设计

```
*-api 模块的内部结构：

  yudao-module-product-api/
  ├── api/                    ← Feign 接口层
  │   ├── spu/
  │   │   ├── ProductSpuApi.java
  │   │   └── dto/
  │   │       └── ProductSpuRespDTO.java
  │   └── sku/
  │       ├── ProductSkuApi.java
  │       └── dto/
  │           └── ProductSkuRespDTO.java
  └── enums/                  ← 常量层
      ├── ApiConstants.java
      └── ErrorCodeConstants.java
```

这种分层设计的好处：
- **接口与 DTO 分离**：同一个 DTO 可以被多个接口复用
- **按业务域组织**：spu、sku 等子目录对应不同的业务域
- **常量统一管理**：服务名、前缀、错误码集中定义

### 通用化提炼

契约优先 + API 模块适合以下场景：

| 特征 | Feign 中的体现 |
|------|---------------|
| 多个服务需要通信 | trade-server 调用 product-server |
| 调用方和实现方独立开发 | 定义契约后可并行开发 |
| 需要编译时类型检查 | 共享 DTO，编译时发现类型错误 |
| 需要统一的接口文档 | Swagger 注解自动生成文档 |

### 电商场景举例：支付模块 API

```java
// yudao-module-pay-api
@FeignClient(name = "pay-server")
public interface PayApi {

    String PREFIX = "/rpc-api/pay";

    @PostMapping(PREFIX + "/create")
    CommonResult<PayCreateRespDTO> createOrder(@RequestBody PayCreateReqDTO reqDTO);

    @GetMapping(PREFIX + "/query")
    CommonResult<PayOrderRespDTO> queryOrder(@RequestParam("id") Long id);
}
```

trade-server 和 pay-server 共享这个接口契约，编译时就能确保调用正确。

## 声明式编程模式

### 模式识别

OpenFeign 是 Java 生态中**声明式编程（Declarative Programming）**的经典案例。声明式编程的核心思想是：**告诉程序"做什么"，而不是"怎么做"**。

```
命令式编程（Imperative）：
  1. 拼接 URL：http://product-server/rpc-api/product/spu/get?id=1
  2. 创建 HttpHeaders
  3. 发起 GET 请求
  4. 解析响应 JSON
  5. 转换为 CommonResult<ProductSpuRespVO>

声明式编程（Declarative）：
  @FeignClient(name = "product-server")
  public interface ProductSpuApi {
      @GetMapping("/rpc-api/product/spu/get")
      CommonResult<ProductSpuRespVO> getSpu(@RequestParam("id") Long id);
  }
  // 使用：productSpuApi.getSpu(1L)
```

### Java 生态中的声明式编程

| 框架/技术 | 声明方式 | 本质 |
|----------|---------|------|
| OpenFeign | `@FeignClient` + `@GetMapping` | HTTP 调用 |
| Spring Data JPA | `findByUsername(String username)` | 数据库查询 |
| MyBatis-Plus | `LambdaQueryWrapper` | SQL 构建 |
| Spring Security | `@PreAuthorize` | 权限校验 |
| Spring Cache | `@Cacheable` | 缓存操作 |
| Spring Transaction | `@Transactional` | 事务管理 |

声明式的共同特点：**用注解或接口定义代替代码实现，框架负责将声明转换为实际行为**。

### 通用化提炼

声明式编程适合以下场景：

| 特征 | Feign 中的体现 |
|------|---------------|
| 操作有固定的模式 | HTTP 请求的构建、发送、响应解析 |
| 实现细节可以标准化 | 序列化、负载均衡、错误处理 |
| 开发者关注业务而非技术 | 只需定义"调用什么方法" |
| 框架可以做优化 | 连接池复用、重试、熔断等 |

## URL 前缀分层设计

### 模式识别

yudao-cloud 的 URL 前缀设计体现了**命名空间（Namespace）**模式：

```
URL 前缀分层：

  /admin-api/{module}/...     ← 管理后台 API
  /app-api/{module}/...       ← 用户端 API
  /rpc-api/{module}/...       ← 服务间 RPC API
```

这种设计避免了不同类型的 API 路径冲突：

```
商品 SPU 的三种 API：

  管理后台：GET /admin-api/product/spu/get?id=1
  用户端：  GET /app-api/product/spu/get?id=1
  RPC 调用：GET /rpc-api/product/spu/get?id=1

三个路径可以映射到同一个 Controller 的不同方法，也可以映射到不同的 Controller。
```

### 通用化提炼

URL 前缀分层适合以下场景：

| 特征 | URL 前缀中的体现 |
|------|-----------------|
| 同一资源有多种访问方式 | 管理后台、用户端、RPC |
| 需要不同的权限控制 | admin-api 需要管理员权限 |
| 需要不同的安全策略 | rpc-api 需要服务间认证 |
| 需要路由到不同服务 | Gateway 根据前缀路由 |

## 反模式警示

### 反模式一：在调用方重新定义 DTO

```java
// 反模式：调用方自己定义 ProductDTO，而不是使用 *-api 模块的 DTO
public class MyProductDTO {
    private Long productId;  // 与 ProductSpuRespDTO 的 id 字段名不同
    private String productName;
}
```

**问题**：字段映射容易出错，编译时无法发现类型错误。

**正确做法**：依赖 *-api 模块，使用共享的 `ProductSpuRespDTO`。

### 反模式二：Feign 接口过于粗粒度

```java
// 反模式：一个接口方法返回所有数据
@FeignClient(name = "product-server")
public interface ProductApi {
    @GetMapping("/rpc-api/product/get-all")
    CommonResult<ProductAllDataDTO> getAllProductData(@RequestParam("id") Long id);
}
```

**问题**：每次调用都传输大量不需要的数据，浪费带宽。

**正确做法**：按业务需求定义细粒度的接口方法。

### 反模式三：忽略错误处理

```java
// 反模式：不检查 Feign 调用结果
public void createOrder(Long spuId) {
    ProductSpuRespDTO spu = productSpuApi.getSpu(spuId).getCheckedData();
    // 如果远程服务返回错误，getCheckedData() 会抛异常
    // 但调用方没有捕获，可能导致订单创建失败
}
```

**正确做法**：使用 Sentinel Fallback 做降级处理，或在调用方捕获异常。

## 总结

声明式 RPC 调用组件的设计模式提炼：

| 模式 | 解决的问题 | 核心思想 |
|------|-----------|---------|
| 代理模式 | 如何透明地调用远程服务 | 接口代理，拦截方法调用 |
| 工厂模式 | 如何创建复杂的代理对象 | 封装创建过程 |
| 拦截器链模式 | 如何在请求前后做通用处理 | 链式拦截，逐个增强 |
| 契约优先模式 | 如何保证调用方和实现方一致 | 先定义契约，再分别实现 |
| 声明式编程 | 如何简化远程调用的编码 | 声明代替编码，框架负责实现 |
| 命名空间模式 | 如何避免 API 路径冲突 | 统一前缀，分层管理 |

这些模式不仅适用于 OpenFeign，也适用于任何 RPC 框架的设计。当你在搭建自己的微服务通信框架时，可以参考这些模式来设计接口定义、代理机制、拦截器链等核心组件。
