# 02 - 问题域与设计目标

## RestTemplate 的痛点

### URL 硬编码问题

使用 RestTemplate 调用远程服务时，URL 是硬编码在代码中的：

```java
// 痛点一：服务地址硬编码
String url = "http://192.168.1.10:48081/admin-api/system/user/get?id=1";
AdminUserRespDTO user = restTemplate.getForObject(url, AdminUserRespDTO.class);

// 痛点二：服务名硬编码（好了一点，但仍有问题）
String url = "http://system-server/admin-api/system/user/get?id=1";
```

问题分析：

```
硬编码 IP 地址：
  ├── 服务迁移时需要修改代码
  ├── 无法利用服务发现机制
  └── 无法自动负载均衡

硬编码服务名：
  ├── 服务名变更时需要修改所有调用方
  ├── URL 路径拼接容易出错
  └── 编译器无法检查 URL 是否正确
```

### 序列化/反序列化手动处理

```java
// 手动拼接请求参数
String url = String.format("http://system-server/admin-api/system/user/get?id=%d", userId);

// 手动处理响应
String response = restTemplate.getForObject(url, String.class);
CommonResult<AdminUserRespDTO> result = JsonUtils.parseObject(response,
    new TypeReference<CommonResult<AdminUserRespDTO>>() {});

if (result.isError()) {
    throw new RuntimeException(result.getMsg());
}
AdminUserRespDTO user = result.getData();
```

这段代码存在以下问题：
- 类型转换繁琐，容易出错
- 错误处理重复，每个调用点都要写
- 编译器无法检查类型正确性

### 横切关注点无法统一处理

```
每次远程调用都需要：
  ├── 添加租户 ID 到请求头
  ├── 添加 Token 到请求头
  ├── 记录调用日志
  ├── 处理超时重试
  └── 处理异常降级

使用 RestTemplate 时，这些逻辑需要在每个调用点重复编写！
```

## 为什么选 OpenFeign 而非 Dubbo

### 技术选型对比

```
场景一：电商系统的服务间调用
  ├── trade-server 调用 product-server 查询商品
  ├── trade-server 调用 system-server 查询用户
  └── trade-server 调用 pay-server 创建支付单

特点：
  ├── 调用频率：中等（每秒几百到几千次）
  ├── 数据量：小（单个对象或列表）
  ├── 延迟要求：毫秒级
  └── 开发效率：优先
```

| 选型考量 | OpenFeign | Dubbo |
|----------|-----------|-------|
| 开发效率 | 高（注解驱动） | 中（需要定义接口+实现） |
| 学习成本 | 低（基于 HTTP） | 中（需要理解协议） |
| 调试便利 | 好（curl/Postman） | 差（需要专用工具） |
| 性能 | 够用 | 更好 |
| 生态集成 | Spring Cloud 原生 | 需要额外适配 |
| 运维成本 | 低 | 中（需要注册中心适配） |

### yudao-cloud 的选择理由

```
选择 OpenFeign 的核心理由：

1. 与 Spring Cloud 生态无缝集成
   ├── 与 Nacos 服务发现集成
   ├── 与 LoadBalancer 负载均衡集成
   └── 与 Sentinel 熔断降级集成

2. HTTP 协议的通用性
   ├── 前端可以直接调用
   ├── 移动端可以直接调用
   ├── 第三方系统可以对接
   └── 网关可以直接转发

3. 开发效率优先
   ├── 注解驱动，代码简洁
   ├── 编译时类型检查
   └── IDE 支持好（代码提示、跳转）

4. 项目定位
   ├── yudao-cloud 是学习项目
   ├── OpenFeign 更容易理解和学习
   └── 适合中小规模微服务
```

## *-api 模块的设计意图

### 接口契约共享

*-api 模块的核心价值是**定义服务间的接口契约**：

```
yudao-module-product-api（接口契约）
├── ProductSpuApi.java              ← "我能提供什么能力"
├── ProductSkuApi.java
└── dto/
    ├── ProductSpuRespVO.java       ← "我返回什么数据"
    └── ProductSkuRespVO.java

trade-server（调用方）
  依赖 yudao-module-product-api
  通过 ProductSpuApi 接口调用 product-server

product-server（实现方）
  依赖 yudao-module-product-api
  实现 ProductSpuApi 接口
```

### DTO 复用

```
没有 *-api 模块：

trade-server                          product-server
    │                                      │
    │  // 自己定义 DTO                     │  // 自己定义 VO
    │  class ProductSpuDTO {               │  class ProductSpuRespVO {
    │      Long id;                        │      Long spuId;
    │      String name;                    │      String spuName;
    │      Integer price;                  │      Integer price;
    │  }                                   │  }
    │                                      │
    │  // 手动映射                          │
    │  ProductSpuDTO dto = new ProductSpuDTO();
    │  dto.setId(vo.getSpuId());           │
    │  dto.setName(vo.getSpuName());       │
    │  dto.setPrice(vo.getPrice());        │


有 *-api 模块：

yudao-module-product-api
    │
    │  // 共享 DTO
    │  class ProductSpuRespVO {
    │      Long id;
    │      String name;
    │      Integer price;
    │  }
    │
    ├── trade-server：直接使用 ProductSpuRespVO
    └── product-server：直接返回 ProductSpuRespVO
```

### 编译时类型检查

```
使用 *-api 模块的优势：

// 编译时就能发现错误
productSpuApi.getSpu("abc");  // 编译错误！参数类型应该是 Long

// 编译时就能发现字段名错误
ProductSpuRespVO spu = productSpuApi.getSpu(1L).getCheckedData();
spu.getSpuId();   // 编译错误！字段名应该是 getId()
spu.getName();    // 正确！
```

## 设计目标

### 目标一：声明式 API

```
设计目标：
  ✓ 使用接口 + 注解定义远程调用
  ✓ 调用方像调用本地方法一样调用远程服务
  ✓ 编译时类型检查
  ✓ IDE 支持（代码提示、跳转、重构）
```

### 目标二：负载均衡

```
设计目标：
  ✓ 自动从 Nacos 获取服务实例列表
  ✓ 自动负载均衡（轮询/随机/权重）
  ✓ 实例故障时自动剔除
  ✓ 支持灰度发布（基于 metadata.version）
```

### 目标三：错误解码

```
设计目标：
  ✓ 统一处理 HTTP 错误响应
  ✓ 将远程异常翻译为本地异常
  ✓ 支持自定义错误解码器
```

### 目标四：Sentinel 集成

```
设计目标：
  ✓ 支持声明式熔断降级
  ✓ 支持 fallback 降级方法
  ✓ 与 Sentinel Dashboard 集成
```

### 目标五：请求拦截

```
设计目标：
  ✓ 统一添加租户 ID 到请求头
  ✓ 统一添加 Token 到请求头
  ✓ 统一记录调用日志
  ✓ 支持自定义拦截器
```
