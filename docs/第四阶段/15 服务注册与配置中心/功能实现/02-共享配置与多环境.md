# 02 - 共享配置与多环境

> 本章介绍如何设计共享配置和多环境策略，让 20 个微服务的配置管理不再混乱。

## 为什么需要共享配置

假设你的电商系统有 5 个微服务，它们都需要以下配置：

```yaml
# 数据库连接池配置
spring:
  datasource:
    druid:
      initial-size: 1
      min-idle: 1
      max-active: 20
      max-wait: 60000

# Redis 配置
spring:
  redis:
    host: 127.0.0.1
    port: 6379
    database: 0

# 日志配置
logging:
  level:
    root: INFO
```

**没有共享配置的问题：**

```
system-server 的 application-local.yaml     ─┐
product-server 的 application-local.yaml    ─┤── 重复 5 份
trade-server 的 application-local.yaml      ─┤── 维护成本高
promotion-server 的 application-local.yaml  ─┤── 修改时容易遗漏
member-server 的 application-local.yaml     ─┘
```

**有共享配置后：**

```
Nacos 上的 application-common.yaml（一份）
    │
    ├── system-server 引用
    ├── product-server 引用
    ├── trade-server 引用
    ├── promotion-server 引用
    └── member-server 引用

修改一处，所有服务生效！
```

## 共享配置的设计方案

### 配置分层设计

yudao-cloud 采用三层配置结构：

```
┌─────────────────────────────────────────────────────────────────┐
│  第一层：本地配置（application.yaml + application-{profile}.yaml）│
│  ├── 服务名、端口、profile 等基础配置                            │
│  └── Nacos 连接信息                                             │
├─────────────────────────────────────────────────────────────────┤
│  第二层：应用专属配置（Nacos 上的 ${service}-{profile}.yaml）     │
│  ├── 数据库连接（每个服务不同）                                  │
│  ├── 业务开关                                                   │
│  └── 服务特有参数                                               │
├─────────────────────────────────────────────────────────────────┤
│  第三层：共享配置（Nacos 上的 application-common.yaml）           │
│  ├── 数据库连接池配置                                           │
│  ├── Redis 配置                                                 │
│  ├── Jackson 序列化配置                                         │
│  └── 日志级别配置                                               │
└─────────────────────────────────────────────────────────────────┘
```

### Nacos 中的配置文件组织

在 Nacos 控制台中，配置文件按以下方式组织：

```
Nacos Server
├── namespace: dev (开发环境)
│   ├── group: DEFAULT_GROUP
│   │   ├── system-server-local.yaml      ← system 专属配置
│   │   ├── product-server-local.yaml     ← product 专属配置
│   │   ├── trade-server-local.yaml       ← trade 专属配置
│   │   ├── promotion-server-local.yaml   ← promotion 专属配置
│   │   └── member-server-local.yaml      ← member 专属配置
│   └── group: SHARED_GROUP
│       └── application-common.yaml       ← 共享配置
│
├── namespace: test (测试环境)
│   ├── group: DEFAULT_GROUP
│   │   ├── system-server-test.yaml
│   │   ├── product-server-test.yaml
│   │   └── ...
│   └── group: SHARED_GROUP
│       └── application-common.yaml
│
└── namespace: prod (生产环境)
    ├── group: DEFAULT_GROUP
    │   ├── system-server-prod.yaml
    │   ├── product-server-prod.yaml
    │   └── ...
    └── group: SHARED_GROUP
        └── application-common.yaml
```

## 实现共享配置

### 第一步：在 Nacos 创建共享配置

在 Nacos 控制台创建共享配置：

- **Data ID**: `application-common.yaml`
- **Group**: `SHARED_GROUP`
- **格式**: YAML
- **内容**:

```yaml
# application-common.yaml - 所有服务共享的配置

# 数据库连接池配置
spring:
  datasource:
    druid:
      initial-size: 1
      min-idle: 1
      max-active: 20
      max-wait: 60000
      time-between-eviction-runs-millis: 60000
      min-evictable-idle-time-millis: 600000
      max-evictable-idle-time-millis: 1800000
      validation-query: SELECT 1 FROM DUAL
      test-while-idle: true
      test-on-borrow: false
      test-on-return: false

# Jackson 序列化配置
spring:
  jackson:
    serialization:
      write-dates-as-timestamps: true
      write-date-timestamps-as-nanoseconds: false
      write-durations-as-timestamps: true
      fail-on-empty-beans: false

# 日志配置
logging:
  level:
    root: INFO
    org.springframework: WARN
```

### 第二步：在微服务中引用共享配置

修改 `application-local.yaml`，添加 `shared-configs` 配置：

```yaml
# yudao-module-system/yudao-module-system-server/src/main/resources/application-local.yaml

spring:
  cloud:
    nacos:
      server-addr: 127.0.0.1:8848
      username: nacos
      password: nacos-admin
      discovery:
        namespace: dev
        group: DEFAULT_GROUP
        metadata:
          version: 1.0.0
      config:
        namespace: dev
        group: DEFAULT_GROUP
        # 共享配置
        shared-configs:
          - data-id: application-common.yaml
            group: SHARED_GROUP
            refresh: true  # 支持热更新
```

**`shared-configs` 配置项解析：**

| 属性 | 作用 | 示例 |
|------|------|------|
| `data-id` | 共享配置的 Data ID | `application-common.yaml` |
| `group` | 共享配置的分组 | `SHARED_GROUP` |
| `refresh` | 是否支持热更新 | `true` |

### 第三步：创建应用专属配置

在 Nacos 控制台为每个服务创建专属配置：

```
Data ID: system-server-local.yaml
Group: DEFAULT_GROUP
格式: YAML
内容:
```

```yaml
# system-server-local.yaml - system 服务专属配置

# 数据源配置（覆盖共享配置中的默认值）
spring:
  datasource:
    dynamic:
      datasource:
        master:
          url: jdbc:mysql://127.0.0.1:3306/ruoyi-vue-pro?useSSL=false&serverTimezone=Asia/Shanghai
          username: root
          password: 123456

# Redis 配置
spring:
  redis:
    host: 127.0.0.1
    port: 6379
    database: 0

# 业务配置
yudao:
  tenant:
    enable: true
  security:
    mock-enable: true
```

## 多环境策略

### 环境切换机制

yudao-cloud 通过 `spring.profiles.active` 切换环境：

```bash
# 本地开发
java -jar system-server.jar --spring.profiles.active=local

# 开发环境部署
java -jar system-server.jar --spring.profiles.active=dev

# 生产环境部署
java -jar system-server.jar --spring.profiles.active=prod
```

### 各环境的配置文件结构

```
resources/
├── application.yaml            ← 主配置（所有环境共享）
├── application-local.yaml      ← 本地开发配置（连接本地 Nacos）
├── application-dev.yaml        ← 开发环境配置（连接开发环境 Nacos）
├── application-test.yaml       ← 测试环境配置（连接测试环境 Nacos）
└── application-prod.yaml       ← 生产环境配置（连接生产环境 Nacos）
```

### 不同环境的 Nacos 配置对比

| 环境 | Nacos 地址 | Namespace | 配置内容 |
|------|-----------|-----------|----------|
| local | `127.0.0.1:8848` | `dev` | 本地数据库、Redis |
| dev | `10.0.0.10:8848` | `dev` | 开发环境数据库、Redis |
| test | `10.0.0.20:8848` | `test` | 测试环境数据库、Redis |
| prod | `172.16.0.10:8848` | `prod` | 生产环境数据库、Redis |

### application-dev.yaml 示例

```yaml
# yudao-module-system/yudao-module-system-server/src/main/resources/application-dev.yaml

spring:
  cloud:
    nacos:
      server-addr: 10.0.0.10:8848  # 开发环境 Nacos 地址
      username: nacos
      password: ${NACOS_PASSWORD:nacos-admin}  # 使用环境变量
      discovery:
        namespace: dev
        group: DEFAULT_GROUP
        metadata:
          version: 1.0.0
      config:
        namespace: dev
        group: DEFAULT_GROUP
        shared-configs:
          - data-id: application-common.yaml
            group: SHARED_GROUP
            refresh: true
```

**注意**：生产环境的敏感信息（密码等）建议使用环境变量或密钥管理服务，不要硬编码在配置文件中。

## 配置优先级详解

理解配置优先级是排查"配置不生效"问题的关键：

```
优先级从高到低：

1. Nacos 上的 ${service-name}-${profile}.yaml    ← 应用专属配置（最高）
2. Nacos 上的 shared-configs                      ← 共享配置
3. 本地 application-${profile}.yaml              ← 本地环境配置
4. 本地 application.yaml                          ← 本地主配置（最低）
```

**覆盖规则示例：**

```
共享配置（application-common.yaml）：
  spring.datasource.druid.max-active: 20

应用专属配置（system-server-local.yaml）：
  spring.datasource.druid.max-active: 50  ← 覆盖共享配置

最终生效：max-active = 50
```

**设计原则：**

- **共享配置做默认值**：存放所有服务通用的配置
- **应用配置做覆盖**：存放服务特有的配置，可覆盖共享配置
- **本地配置做兜底**：确保没有 Nacos 时也能启动

## 配置热更新

### @RefreshScope 注解

当 Nacos 上的配置发生变化时，使用 `@RefreshScope` 注解的 Bean 会自动刷新：

```java
@RestController
@RefreshScope  # 配置变更时自动刷新
@RequestMapping("/test")
public class TestController {

    @Value("${test.config:default}")
    private String testConfig;

    @GetMapping("/config")
    public String testConfig() {
        return testConfig;  # 配置变更后，下次调用会返回新值
    }
}
```

### 配置热更新的工作流程

```
1. 在 Nacos 控制台修改配置
        │
        ▼
2. Nacos Server 推送配置变更事件
        │
        ▼
3. 微服务接收到变更事件
        │
        ▼
4. @RefreshScope 注解的 Bean 被销毁并重建
        │
        ▼
5. @Value 注入新的配置值
        │
        ▼
6. 下次请求使用新配置
```

### 注意事项

- 不是所有配置都支持热更新，只有 `@RefreshScope` 注解的 Bean 才会刷新
- 数据库连接池等复杂组件的配置变更可能需要重启
- 建议在 Nacos 控制台修改配置前，先在测试环境验证

## 配置命名规范

### Data ID 命名规范

| 类型 | 命名规范 | 示例 |
|------|----------|------|
| 应用专属配置 | `${spring.application.name}-${profile}.yaml` | `system-server-local.yaml` |
| 共享配置 | `application-common.yaml` | `application-common.yaml` |
| 业务配置 | `${业务模块}-${profile}.yaml` | `payment-config-local.yaml` |

### Group 命名规范

| 类型 | 命名 | 说明 |
|------|------|------|
| 应用配置 | `DEFAULT_GROUP` | 默认分组，存放应用专属配置 |
| 共享配置 | `SHARED_GROUP` | 共享分组，存放公共配置 |
| 业务配置 | `BUSINESS_GROUP` | 业务分组，存放业务相关配置 |

### Namespace 命名规范

| 环境 | Namespace | 说明 |
|------|-----------|------|
| 开发环境 | `dev` | 开发人员使用 |
| 测试环境 | `test` | 测试人员使用 |
| 生产环境 | `prod` | 生产环境使用 |

## 最佳实践

### 1. 敏感信息管理

```yaml
# 不推荐：硬编码密码
spring:
  datasource:
    password: 123456

# 推荐：使用环境变量
spring:
  datasource:
    password: ${DB_PASSWORD:123456}

# 推荐：使用 Nacos 的加密配置
spring:
  datasource:
    password: ${cipher}AESxxxxxx  # Nacos 支持加密配置
```

### 2. 配置版本管理

建议对 Nacos 上的配置进行版本管理：
- 修改配置前，先在 Nacos 控制台备份当前版本
- 使用"历史版本"功能查看配置变更记录
- 重要配置变更后，及时同步到代码仓库的配置模板中

### 3. 配置变更审批

生产环境的配置变更建议：
- 建立配置变更审批流程
- 先在测试环境验证
- 变更时做好回滚准备
- 记录变更日志

## 本章小结

本章介绍了共享配置和多环境策略的实现：

1. **共享配置设计**：通过 `shared-configs` 实现配置复用，减少重复
2. **配置分层**：本地配置做兜底、共享配置做默认、应用配置做覆盖
3. **多环境隔离**：通过 `namespace` + `profile` 实现环境隔离
4. **配置热更新**：通过 `@RefreshScope` 实现配置动态刷新
