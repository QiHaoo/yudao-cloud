# 02 - 问题域与设计目标

## 微服务通信的前提：服务发现

### 硬编码地址的困境

在单体应用中，所有模块共享同一个进程，模块间调用就是普通的 Java 方法调用。拆分为微服务后，调用变成了网络请求，最原始的方式是硬编码地址：

```java
// 硬编码地址的调用方式
String url = "http://192.168.1.10:48081/admin-api/system/user/get?id=1";
AdminUserRespDTO user = restTemplate.getForObject(url, AdminUserRespDTO.class);
```

这种方式存在严重问题：

```
问题一：地址变更 = 代码变更
  服务 B 迁移到新服务器 → 修改所有调用方的代码 → 重新编译部署

问题二：无法负载均衡
  服务 B 部署了 3 个实例 → 只能写死一个地址 → 其他实例浪费

问题三：无法故障转移
  写死的实例宕机了 → 调用直接失败 → 无法自动切换到其他实例

问题四：环境差异
  开发环境：192.168.1.10:48081
  测试环境：10.0.0.10:48081
  生产环境：172.16.0.10:48081
  每个环境都要改配置？
```

### 服务发现的核心思想

服务发现的本质是一个**动态的地址簿**：

1. **服务启动时**：自动向地址簿注册自己的地址
2. **服务运行时**：定期向地址簿发送心跳，证明自己还活着
3. **服务调用时**：从地址簿查询目标服务的可用地址
4. **服务停止时**：从地址簿注销自己（或被检测为不健康后自动剔除）

```
调用方                              服务注册中心                        被调用方
  │                                    │                                │
  │  1. 查询 product-server 的地址     │                                │
  │ ─────────────────────────────────> │                                │
  │                                    │                                │
  │  2. 返回可用实例列表               │                                │
  │    [192.168.1.20:48082,            │                                │
  │     192.168.1.21:48082]            │                                │
  │ <───────────────────────────────── │                                │
  │                                    │                                │
  │  3. 选择一个实例发起调用           │                                │
  │ ─────────────────────────────────────────────────────────────────> │
  │                                    │                                │
  │                                    │  4. 实例定期发送心跳           │
  │                                    │ <──────────────────────────── │
  │                                    │                                │
```

### Nacos 的服务发现机制

Nacos 提供了两种服务发现模式：

**客户端主动发现**（yudao-cloud 采用的方式）：

```
调用方                     Nacos Server
  │                            │
  │  1. 订阅 product-server    │
  │ ─────────────────────────> │
  │                            │
  │  2. 推送实例列表变更       │
  │ <───────────────────────── │
  │                            │
  │  3. 本地缓存实例列表       │
  │  4. 从本地列表选择实例     │
  │  5. 直接调用目标实例       │
```

**服务端代理发现**：

```
调用方              Nacos Server            被调用方
  │                    │                      │
  │  1. 转发请求       │                      │
  │ ─────────────────> │                      │
  │                    │  2. 选择实例转发      │
  │                    │ ────────────────────> │
  │                    │                      │
  │                    │  3. 返回响应          │
  │ <───────────────── │ <──────────────────── │
```

yudao-cloud 使用 Spring Cloud 的 `DiscoveryClient` + LoadBalancer 实现客户端主动发现，性能更好。

## 配置管理的挑战

### 多环境配置的混乱

一个电商系统至少有三个环境：

```
开发环境（dev）：
  数据库：localhost:3306/dev_db
  Redis：localhost:6379
  支付：沙箱环境
  日志级别：DEBUG

测试环境（test）：
  数据库：10.0.0.10:3306/test_db
  Redis：10.0.0.10:6379
  支付：沙箱环境
  日志级别：INFO

生产环境（prod）：
  数据库：172.16.0.10:3306/prod_db
  Redis：172.16.0.10:6379
  支付：正式环境
  日志级别：WARN
```

传统方式的痛点：
- 每个环境一份配置文件，维护成本高
- 配置散落在各个服务的本地文件中，无法统一查看
- 配置差异可能导致"在我机器上能跑"的问题

### 配置热更新的需求

假设电商系统有一个促销开关，用于控制是否开启秒杀活动：

```yaml
# 传统方式：配置写在本地文件
promotion:
  seckill:
    enabled: true  # 是否开启秒杀
    max-quantity: 10  # 每人限购数量
```

如果需要关闭秒杀活动，传统方式需要：
1. 修改配置文件
2. 重新打包
3. 重启服务

在大促期间，这可能意味着几分钟的服务中断。理想的方式是：修改配置后，服务自动感知并生效，无需重启。

### 多实例配置一致性

当一个微服务部署了 10 个实例时，如何保证所有实例的配置一致？

```
实例 1：promotion.seckill.enabled = true
实例 2：promotion.seckill.enabled = true
实例 3：promotion.seckill.enabled = false  ← 配置不一致！
...
实例 10：promotion.seckill.enabled = true
```

配置不一致会导致：
- 用户请求打到不同实例，得到不同的行为
- 难以排查的 Bug
- 数据不一致

## 设计目标

基于以上问题分析，yudao-cloud 的服务注册与配置中心设计需要达成以下目标：

### 目标一：服务自动注册与发现

```
设计目标：
  ✓ 服务启动时自动注册，无需手动配置
  ✓ 服务停止时自动注销，无需手动清理
  ✓ 调用方通过服务名访问，无需硬编码地址
  ✓ 自动负载均衡，请求均匀分配到多个实例
  ✓ 实例故障时自动剔除，不影响业务
```

### 目标二：配置集中管理

```
设计目标：
  ✓ 所有配置存储在 Nacos，统一查看和管理
  ✓ 支持多环境隔离（dev/test/prod）
  ✓ 支持配置分组（DEFAULT_GROUP / BUSINESS_GROUP）
  ✓ 支持共享配置，减少重复
```

### 目标三：配置热更新

```
设计目标：
  ✓ 配置修改后自动推送到所有实例
  ✓ 无需重启服务即可生效
  ✓ 支持 @RefreshScope 注解实现配置动态刷新
```

### 目标四：环境隔离

```
设计目标：
  ✓ 不同环境的配置完全隔离
  ✓ 开发环境的配置不会影响生产环境
  ✓ 支持命名空间（namespace）实现物理隔离
```

## Nacos 的 AP/CP 模式选择

Nacos 支持两种一致性模型，适用于不同场景：

### AP 模式（高可用）

```
特点：
  - 优先保证服务可用性
  - 允许短暂的数据不一致
  - 适合服务发现场景

为什么服务发现适合 AP？
  - 服务实例列表允许短暂不一致
  - 某个实例短暂不可见，不影响整体业务
  - 高可用比强一致更重要
```

### CP 模式（强一致）

```
特点：
  - 优先保证数据一致性
  - 可能牺牲部分可用性
  - 适合配置管理场景

为什么配置管理适合 CP？
  - 配置数据必须一致
  - 所有实例必须看到相同的配置
  - 配置不一致会导致业务逻辑错误
```

### yudao-cloud 的选择

```yaml
# 服务发现：使用 AP 模式（默认）
spring:
  cloud:
    nacos:
      discovery:
        # Nacos 2.x 默认使用 AP 模式，无需额外配置

# 配置中心：使用 CP 模式（Nacos 内部实现）
# Nacos 配置中心内部使用 Raft 协议保证一致性，无需用户配置
```

```
┌─────────────────────────────────────────────────────────┐
│                      Nacos Server                        │
│                                                         │
│  ┌─────────────────────┐  ┌─────────────────────┐      │
│  │   服务注册中心      │  │     配置中心        │      │
│  │   (AP 模式)         │  │     (CP 模式)       │      │
│  │                     │  │                     │      │
│  │   Distro 协议       │  │     Raft 协议       │      │
│  │   最终一致性        │  │     强一致性        │      │
│  │   高可用            │  │     数据准确        │      │
│  └─────────────────────┘  └─────────────────────┘      │
└─────────────────────────────────────────────────────────┘
```

## 思考题

1. **如果让你设计一个简单的服务注册中心，你会如何实现心跳检测机制？心跳间隔设为多少合适？太短和太长分别会有什么问题？**

2. **配置热更新虽然方便，但也带来了风险。如果有人误操作修改了数据库连接配置，所有实例会立即切换到错误的数据库。如何防范这种风险？**

3. **AP 模式和 CP 模式的选择，本质上是 CAP 理论的权衡。除了服务发现和配置中心，你能举出其他场景中 AP/CP 选择的例子吗？**

4. **在 yudao-cloud 的单体模式中，Nacos 被完全禁用。如果此时你需要配置热更新的能力，有什么替代方案？**
