# 01 - 总览：服务注册与配置中心全景

## 场景引入

假设你正在搭建一个电商后台管理系统，随着业务增长，单体应用已经无法满足需求，你决定将其拆分为微服务架构：

- **system-server**：负责用户、角色、权限等系统管理
- **product-server**：负责商品、分类、SKU 管理
- **trade-server**：负责订单、支付、物流管理
- **promotion-server**：负责优惠券、秒杀、满减等促销活动

拆分后，你立刻面临两个核心问题：

**问题一：服务如何发现彼此？**

当 trade-server 需要调用 product-server 查询商品信息时，它怎么知道 product-server 的地址？如果 product-server 部署了多个实例（比如 3 个），每个实例的 IP 和端口都不同，trade-server 又该如何选择？

在单体时代，所有服务都在同一个进程里，方法调用就是直接的 Java 方法调用。拆分后，服务间调用变成了网络请求，你需要知道对方的 IP:Port。

**问题二：配置如何统一管理？**

每个微服务都有自己的配置文件，比如数据库连接、Redis 地址、业务开关等。当你的微服务数量从 3 个增长到 20 个时：

- 修改数据库密码，需要逐个修改每个服务的配置文件
- 不同环境（开发、测试、生产）的配置不同，管理混乱
- 修改配置后需要重启服务才能生效，影响业务连续性

```
传统方式的痛点：

服务 A ──硬编码──> 192.168.1.10:8081  ──→ 服务 B 实例 1
                  192.168.1.11:8081  ──→ 服务 B 实例 2
                  192.168.1.12:8081  ──→ 服务 B 实例 3

配置管理：
  服务 A 的配置 ──→ application.yaml (本地文件)
  服务 B 的配置 ──→ application.yaml (本地文件)
  服务 C 的配置 ──→ application.yaml (本地文件)
  修改配置？──→ 逐个修改 + 重启所有服务
```

## Nacos 的双重角色

Nacos（Dynamic Naming and Configuration Service）同时扮演两个角色：

### 角色一：服务注册中心

服务注册中心的核心职责是维护一张"服务地址表"。每个微服务启动时，自动将自己的地址注册到 Nacos；其他服务需要调用时，从 Nacos 查询目标服务的地址列表。

```
                    ┌─────────────────────────────┐
                    │        Nacos Server          │
                    │                             │
                    │  服务地址表：                 │
                    │  ┌───────────────────────┐  │
                    │  │ system-server         │  │
                    │  │  ├─ 192.168.1.10:48081│  │
                    │  │  └─ 192.168.1.11:48081│  │
                    │  │ product-server        │  │
                    │  │  ├─ 192.168.1.20:48082│  │
                    │  │  └─ 192.168.1.21:48082│  │
                    │  │ trade-server          │  │
                    │  │  └─ 192.168.1.30:48083│  │
                    │  └───────────────────────┘  │
                    └──────────────┬──────────────┘
                                   │
              ┌────────────────────┼────────────────────┐
              │                    │                    │
              ▼                    ▼                    ▼
    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
    │ system-server│    │product-server│    │ trade-server │
    │              │    │              │    │              │
    │ 启动时注册   │    │ 启动时注册   │    │ 启动时注册   │
    │ 心跳保活     │    │ 心跳保活     │    │ 心跳保活     │
    └──────────────┘    └──────────────┘    └──────────────┘
```

核心机制：
1. **服务注册**：微服务启动时，向 Nacos 发送注册请求，携带服务名、IP、端口、元数据等信息
2. **心跳保活**：注册后，微服务定期向 Nacos 发送心跳，证明自己还活着
3. **服务发现**：调用方通过服务名从 Nacos 查询可用实例列表
4. **健康检查**：Nacos 检测到某实例心跳超时，自动将其从可用列表中剔除

### 角色二：配置中心

配置中心的核心职责是集中管理所有微服务的配置，并支持动态更新。

```
                    ┌─────────────────────────────┐
                    │        Nacos Server          │
                    │                             │
                    │  配置列表：                   │
                    │  ┌───────────────────────┐  │
                    │  │ system-server-dev.yaml │  │
                    │  │ product-server-dev.yaml│  │
                    │  │ trade-server-dev.yaml  │  │
                    │  │ application-common.yaml│  │  ← 共享配置
                    │  └───────────────────────┘  │
                    └──────────────┬──────────────┘
                                   │ 配置变更推送
              ┌────────────────────┼────────────────────┐
              │                    │                    │
              ▼                    ▼                    ▼
    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
    │ system-server│    │product-server│    │ trade-server │
    │              │    │              │    │              │
    │ 启动时拉取   │    │ 启动时拉取   │    │ 启动时拉取   │
    │ 监听变更     │    │ 监听变更     │    │ 监听变更     │
    └──────────────┘    └──────────────┘    └──────────────┘
```

核心机制：
1. **配置拉取**：微服务启动时，从 Nacos 拉取自己的配置
2. **配置监听**：微服务持续监听 Nacos 的配置变更
3. **动态更新**：配置变更时，Nacos 主动推送给微服务，无需重启即可生效
4. **配置共享**：多个微服务可以共享同一份公共配置

## 为什么选 Nacos 而非其他方案

| 维度 | Nacos | Eureka | Consul | Zookeeper |
|------|-------|--------|--------|-----------|
| **开发公司** | 阿里巴巴 | Netflix | HashiCorp | Apache |
| **服务注册** | 支持 | 支持 | 支持 | 支持 |
| **配置中心** | 内置支持 | 不支持 | 支持 KV | 不支持 |
| **健康检查** | TCP/HTTP/MySQL/自定义 | 客户端心跳 | TCP/HTTP/脚本 | Keep Alive |
| **CAP 模型** | AP + CP 可切换 | AP | CP | CP |
| **雪崩保护** | 支持 | 支持 | 不支持 | 不支持 |
| **管理界面** | 自带控制台 | 简单界面 | 自带界面 | 无（需第三方） |
| **Spring Cloud 集成** | Spring Cloud Alibaba 原生 | Spring Cloud Netflix | Spring Cloud Consul | Apache Curator |
| **社区活跃度** | 非常活跃 | 维护模式 | 活跃 | 活跃 |

选择 Nacos 的核心理由：

1. **双重角色**：同时提供服务注册和配置中心，减少基础设施组件数量
2. **Spring Cloud Alibaba 生态**：与项目技术栈完美契合，开箱即用
3. **AP/CP 可切换**：根据场景选择一致性模型（服务发现用 AP，配置管理用 CP）
4. **中文社区**：文档完善，遇到问题容易找到解决方案
5. **阿里生产验证**：在阿里内部大规模使用，经过双十一考验

## 与 Spring Cloud Alibaba 的集成

yudao-cloud 项目使用 Spring Cloud Alibaba 作为微服务框架，Nacos 是其核心组件之一。集成方式非常简洁：

### 依赖引入

在 `yudao-dependencies/pom.xml` 中统一管理版本：

```xml
<spring.cloud.alibaba.version>2021.0.6.2</spring.cloud.alibaba.version>

<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-alibaba-dependencies</artifactId>
    <version>${spring.cloud.alibaba.version}</version>
    <type>pom</type>
    <scope>import</scope>
</dependency>
```

在各微服务的 `pom.xml` 中引入具体依赖：

```xml
<!-- Nacos 服务发现 -->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>

<!-- Nacos 配置中心 -->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-config</artifactId>
</dependency>

<!-- Spring Cloud Bootstrap（Nacos 配置中心需要） -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

### 配置方式

yudao-cloud 采用 `application.yaml` + `spring.config.import` 的方式加载 Nacos 配置（Spring Boot 2.4+ 推荐方式）：

```yaml
# application.yaml
spring:
  config:
    import:
      - optional:classpath:application-${spring.profiles.active}.yaml  # 本地配置
      - optional:nacos:${spring.application.name}-${spring.profiles.active}.yaml  # Nacos 配置
```

### 单体模式 vs 微服务模式

yudao-cloud 支持两种部署模式，通过配置切换：

```yaml
# 单体模式（yudao-server）：禁用 Nacos
spring:
  cloud:
    nacos:
      discovery:
        enabled: false
      config:
        enabled: false

# 微服务模式（system-server 等）：启用 Nacos
spring:
  cloud:
    nacos:
      server-addr: 127.0.0.1:8848
      discovery:
        namespace: dev
        group: DEFAULT_GROUP
      config:
        namespace: dev
        group: DEFAULT_GROUP
```

这种设计的精妙之处在于：同一套业务代码，既可以作为单体应用运行（开发调试方便），也可以作为微服务运行（生产部署灵活）。
