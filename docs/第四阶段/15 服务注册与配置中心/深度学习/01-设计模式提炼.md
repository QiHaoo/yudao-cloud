# 01 - 设计模式提炼：服务注册与配置中心

本文从服务注册与配置中心组件中提炼可复用的设计模式，帮助读者在搭建微服务基础设施时，理解"注册中心"和"配置中心"背后的设计思想。

## 注册中心模式：服务发现的通用架构

### 模式识别

Nacos 作为服务注册中心，其核心是一个**目录服务（Directory Service）**模式：维护一张"服务名 → 实例列表"的映射表，让服务消费者能够动态发现服务提供者。

```
┌─────────────────────────────────────────────────────────────┐
│                    注册中心模式                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  服务提供者（Provider）                                       │
│      │                                                      │
│      │ 1. 注册（Register）                                   │
│      ▼                                                      │
│  ┌─────────────────────────────────┐                        │
│  │        注册中心（Registry）       │                        │
│  │                                 │                        │
│  │  服务名 → 实例列表               │                        │
│  │  ┌─────────────────────────┐   │                        │
│  │  │ system-server           │   │                        │
│  │  │  ├─ 192.168.1.10:48081  │   │                        │
│  │  │  └─ 192.168.1.11:48081  │   │                        │
│  │  │ product-server          │   │                        │
│  │  │  └─ 192.168.1.20:48082  │   │                        │
│  │  └─────────────────────────┘   │                        │
│  └──────────────┬──────────────────┘                        │
│                 │                                            │
│                 │ 2. 发现（Discover）                         │
│                 ▼                                            │
│  服务消费者（Consumer）                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 核心机制：注册、心跳、发现、剔除

注册中心的四个核心机制形成了一个完整的生命周期管理：

```
服务实例生命周期：

  启动 ──注册──▶ 在线 ──心跳续约──▶ 在线 ──...─▶ 心跳超时 ──剔除──▶ 下线
                                    │
                                    │ 主动注销
                                    ▼
                                  下线
```

| 机制 | 触发方 | 方向 | 作用 |
|------|--------|------|------|
| 注册 | 服务实例 | 实例 → 注册中心 | 服务启动时，告诉注册中心"我在这里" |
| 心跳 | 服务实例 | 实例 → 注册中心 | 定期证明"我还活着" |
| 发现 | 服务消费者 | 消费者 ← 注册中心 | 查询目标服务的可用实例列表 |
| 剔除 | 注册中心 | 注册中心内部 | 心跳超时，自动移除不可用实例 |

### 一致性模型的选择：AP vs CP

注册中心面临一个经典的分布式系统问题：**一致性（Consistency）**和**可用性（Availability）**之间的权衡。

```
CAP 定理：
  分布式系统最多同时满足以下三个中的两个：
  - C（一致性）：所有节点看到相同的数据
  - A（可用性）：每个请求都能得到响应
  - P（分区容错）：网络分区时系统仍能工作

注册中心的选择：
  - AP 模型（Nacos 默认）：优先可用性，允许短暂的数据不一致
    适用场景：服务发现（实例列表短暂不一致影响不大）
  - CP 模型（Nacos 可选）：优先一致性，可能牺牲可用性
    适用场景：配置管理（配置必须一致）
```

Nacos 的精妙之处在于：**同一个组件，根据使用场景切换一致性模型**。服务发现用 AP（高可用），配置管理用 CP（强一致）。

### 通用化提炼

注册中心模式适合以下场景：

| 特征 | 服务注册中的体现 |
|------|-----------------|
| 服务实例动态变化 | 实例随时可能上线、下线、扩缩容 |
| 消费者不直接知道提供者地址 | 通过注册中心间接发现 |
| 需要健康检查 | 心跳机制检测实例存活 |
| 需要负载均衡 | 从多个实例中选择一个 |

### 电商场景举例：商品搜索服务

假设电商系统的商品搜索服务部署了 5 个实例，订单服务需要调用它查询商品信息：

```
没有注册中心：
  订单服务硬编码 5 个实例地址
  → 某个实例宕机，订单服务不知道，继续调用失败
  → 新增实例，需要修改订单服务配置并重启

有注册中心：
  订单服务通过服务名 "product-search-server" 查询
  → 注册中心返回当前可用的实例列表
  → 宕机实例自动被剔除，新实例自动注册
  → 订单服务无需关心实例变化
```

## 配置中心模式：集中配置管理

### 模式识别

Nacos 作为配置中心，其核心是一个**集中式配置仓库（Centralized Configuration Repository）**模式：所有配置集中存储，按需拉取，变更推送。

```
┌─────────────────────────────────────────────────────────────┐
│                    配置中心模式                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────┐                        │
│  │        配置中心（Config Center）  │                        │
│  │                                 │                        │
│  │  配置存储 + 变更推送              │                        │
│  └──────────────┬──────────────────┘                        │
│                 │                                            │
│      ┌──────────┼──────────┐                                │
│      │          │          │                                │
│      ▼          ▼          ▼                                │
│  服务 A      服务 B      服务 C                              │
│  (拉取配置)  (拉取配置)  (拉取配置)                           │
│  (监听变更)  (监听变更)  (监听变更)                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 配置加载的时序

配置中心的配置加载发生在 Spring 应用上下文初始化之前，这是一个关键的时序设计：

```
Spring Boot 启动流程：

  1. JVM 启动
       │
  2. BootstrapApplicationListener 监听 ApplicationEnvironmentPreparedEvent
       │
  3. 创建 Bootstrap 上下文
       │
  4. NacosConfigBootstrapConfiguration 初始化
       │  ├─ 读取 spring.cloud.nacos.config 配置
       │  ├─ 从 Nacos Server 拉取配置
       │  └─ 将配置注入到 Spring Environment
       │
  5. 主 Application Context 初始化
       │  ├─ 加载 application.yaml（本地配置）
       │  ├─ 加载 Nacos 配置（通过 spring.config.import）
       │  └─ Bean 创建和注入
       │
  6. 应用就绪
```

这个时序保证了：**配置在 Bean 创建之前就已经就绪**，`@Value` 注入的值来自 Nacos 而不是本地文件。

### 配置推送 vs 配置拉取

配置中心有两种获取配置的方式：

| 方式 | 时机 | 实现 | 优点 | 缺点 |
|------|------|------|------|------|
| 拉取（Pull） | 启动时 | 主动请求配置中心 | 简单可靠 | 配置变更需要重启 |
| 推送（Push） | 运行时 | 配置中心主动通知 | 无需重启 | 需要长连接，实现复杂 |

Nacos 采用**拉取 + 推送**的混合模式：
- 启动时：主动拉取配置
- 运行时：通过长轮询（Long Polling）监听配置变更，变更时推送

### 通用化提炼

配置中心模式适合以下场景：

| 特征 | 配置中心中的体现 |
|------|-----------------|
| 多个服务共享配置 | 数据库连接池、Redis 等公共配置 |
| 配置需要动态更新 | 业务开关、限流阈值等 |
| 不同环境配置不同 | dev / test / prod 环境隔离 |
| 敏感信息需要集中管理 | 数据库密码、API 密钥等 |

## 共享配置分层设计

### 模式识别

yudao-cloud 的配置分层设计体现了**层次化配置（Hierarchical Configuration）**模式：

```
优先级从高到低：

  应用专属配置（Nacos: ${service}-${profile}.yaml）
      │ 可覆盖
      ▼
  共享配置（Nacos: application-common.yaml）
      │ 可覆盖
      ▼
  本地环境配置（application-${profile}.yaml）
      │ 可覆盖
      ▼
  本地主配置（application.yaml）
      │ 可覆盖
      ▼
  代码默认值（@Value 默认值）
```

### 分层设计的三个原则

```
原则一：共享配置做默认值
  ┌─────────────────────────────────────────────┐
  │ application-common.yaml                      │
  │   spring.datasource.druid.max-active: 20     │  ← 默认值
  └─────────────────────────────────────────────┘

原则二：应用配置做覆盖
  ┌─────────────────────────────────────────────┐
  │ system-server-local.yaml                     │
  │   spring.datasource.druid.max-active: 50     │  ← 覆盖为 50
  └─────────────────────────────────────────────┘

原则三：本地配置做兜底
  ┌─────────────────────────────────────────────┐
  │ application-local.yaml                       │
  │   spring.datasource.url: jdbc:mysql://local  │  ← Nacos 不可用时使用
  └─────────────────────────────────────────────┘
```

### 与 Java 类继承的类比

配置分层设计和 Java 的类继承有异曲同工之妙：

| Java 类继承 | 配置分层 |
|------------|---------|
| 父类定义默认行为 | 共享配置定义默认值 |
| 子类覆盖父类方法 | 应用配置覆盖共享配置 |
| 接口定义契约 | 配置 key 定义配置项 |
| 多态：同一接口不同实现 | 多环境：同一 key 不同值 |

### 通用化提炼

层次化配置适合以下场景：

| 特征 | 配置分层中的体现 |
|------|-----------------|
| 大量配置需要复用 | 共享配置存储公共配置 |
| 不同实体有不同配置 | 不同服务有不同配置 |
| 需要支持默认值 | 共享配置做默认，应用配置做覆盖 |
| 需要支持环境隔离 | namespace + profile |

### 电商场景举例：多店铺配置

假设电商 SaaS 平台需要为不同店铺提供不同的配置：

```
全局默认配置：
  商品分页大小: 20
  库存预警阈值: 100
  订单超时时间: 30 分钟

华东区域配置（覆盖）：
  库存预警阈值: 200  ← 华东销量大，阈值调高

某旗舰店配置（覆盖）：
  商品分页大小: 50   ← 旗舰店商品多，分页调大
```

这种分层设计让配置管理既灵活又不混乱。

## 命名空间隔离模式

### 模式识别

Nacos 的命名空间（Namespace）实现了**物理隔离**的配置管理。不同命名空间的数据完全独立，从根本上避免了环境串扰。

```
Nacos Server
├── namespace: dev (开发环境)
│   └── 所有开发环境的服务和配置
│
├── namespace: test (测试环境)
│   └── 所有测试环境的服务和配置
│
└── namespace: prod (生产环境)
    └── 所有生产环境的服务和配置
```

### 两级隔离设计

Nacos 采用两级隔离：Namespace（命名空间） + Group（分组）。

```
Namespace（一级隔离）：环境级别
  ├── dev（开发环境）
  ├── test（测试环境）
  └── prod（生产环境）

Group（二级隔离）：业务级别
  ├── DEFAULT_GROUP（应用配置）
  ├── SHARED_GROUP（共享配置）
  └── BUSINESS_GROUP（业务配置）
```

为什么需要两级隔离？

| 维度 | 只用 Namespace | 只用 Group | 两级都用 |
|------|---------------|-----------|---------|
| 环境隔离 | 可以 | 不够强（逻辑隔离） | 物理隔离 |
| 业务隔离 | 不够灵活 | 可以 | 灵活分组 |
| 权限控制 | 按环境授权 | 按业务授权 | 精细化授权 |

### 通用化提炼

命名空间隔离模式适合以下场景：

| 特征 | Nacos 中的体现 |
|------|---------------|
| 需要环境隔离 | dev / test / prod |
| 防止配置串扰 | 不同 namespace 数据完全独立 |
| 需要权限控制 | 不同 namespace 可设置不同权限 |
| 需要独立管理 | 每个 namespace 独立的配置列表 |

## 单体/微服务双模式设计

### 模式识别

yudao-cloud 的一个精妙设计是：**同一套业务代码，既可以作为单体应用运行，也可以作为微服务运行**。切换方式只需修改配置。

```
单体模式：
  spring.cloud.nacos.discovery.enabled: false
  spring.cloud.nacos.config.enabled: false
  → 所有模块在同一进程内，方法调用直接走 Java 调用

微服务模式：
  spring.cloud.nacos.server-addr: 127.0.0.1:8848
  → 各模块独立进程，通过 Feign + Nacos 通信
```

### 条件装配的实现

这种双模式通过 Spring Boot 的条件装配（Conditional Assembly）实现：

```java
// 只有在微服务模式下（引入了 Feign），才注册 RPC 相关的 Bean
@AutoConfiguration
@ConditionalOnClass(name = "feign.RequestInterceptor")
public class YudaoDataPermissionRpcAutoConfiguration {
    // ...
}

// 只有在有 Nacos 依赖时，才启用服务发现
// spring.cloud.nacos.discovery.enabled: false 时，NacosDiscoveryAutoConfiguration 不生效
```

### 设计价值

| 维度 | 单体模式 | 微服务模式 |
|------|---------|-----------|
| 开发效率 | 高（无需启动 Nacos） | 中（需要 Nacos） |
| 调试难度 | 低（同一进程） | 中（跨进程） |
| 部署复杂度 | 低（一个 JAR） | 高（多个 JAR + Nacos） |
| 扩展性 | 差（垂直扩展） | 好（水平扩展） |
| 适用阶段 | 开发初期、小团队 | 生产环境、大团队 |

这种设计让项目可以在不同阶段选择不同的部署模式，而无需重写代码。

## 设计模式的组合使用

服务注册与配置中心组件的设计模式是相互配合的：

```
┌─────────────────────────────────────────────────────────────┐
│                    设计模式组合全景                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  注册中心模式                                                │
│  ├── 目录服务（服务名 → 实例列表）                            │
│  ├── 心跳保活（定期续约）                                     │
│  ├── 健康检查（超时剔除）                                     │
│  └── 一致性模型选择（AP/CP）                                  │
│                                                             │
│  配置中心模式                                                │
│  ├── 集中式配置仓库                                          │
│  ├── 拉取 + 推送混合模式                                     │
│  └── 配置热更新                                              │
│                                                             │
│  层次化配置模式                                              │
│  ├── 共享配置做默认值                                        │
│  ├── 应用配置做覆盖                                          │
│  └── 本地配置做兜底                                          │
│                                                             │
│  命名空间隔离模式                                            │
│  ├── 一级隔离：Namespace（环境）                              │
│  └── 二级隔离：Group（业务）                                  │
│                                                             │
│  条件装配模式                                                │
│  ├── @ConditionalOnClass（有依赖才启用）                      │
│  └── enabled 配置开关（运行时禁用）                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 横向对比：相同设计手法在其他组件中的应用

### 集中式管理的其他应用

| 组件 | 管理对象 | 存储位置 |
|------|---------|---------|
| 配置中心 | 配置项 | Nacos Config |
| 注册中心 | 服务实例 | Nacos Discovery |
| 字典管理 | 字典数据 | 数据库 |
| 菜单管理 | 菜单权限 | 数据库 |

### 层次化设计的其他应用

| 组件 | 分层方式 | 覆盖规则 |
|------|---------|---------|
| 配置中心 | 共享配置 → 应用配置 → 本地配置 | 高优先级覆盖低优先级 |
| Spring Security | 全局配置 → 模块配置 → 方法注解 | 就近原则 |
| 数据权限 | 类级注解 → 方法级注解 | 方法级覆盖类级 |
| 异常处理 | 全局处理器 → 模块处理器 | 细粒度覆盖粗粒度 |

### 条件装配的其他应用

| 组件 | 条件 | 作用 |
|------|------|------|
| Nacos Discovery | `nacos.discovery.enabled` | 启用/禁用服务发现 |
| Nacos Config | `nacos.config.enabled` | 启用/禁用配置中心 |
| 数据权限 RPC | `@ConditionalOnClass(Feign)` | 有 Feign 才启用 RPC 传播 |
| 多租户 | `yudao.tenant.enable` | 启用/禁用多租户 |

## 总结

服务注册与配置中心组件的设计模式提炼：

| 模式 | 解决的问题 | 核心思想 |
|------|-----------|---------|
| 注册中心模式 | 服务如何发现彼此 | 目录服务 + 心跳保活 + 健康检查 |
| 配置中心模式 | 配置如何统一管理 | 集中存储 + 拉取推送 + 热更新 |
| 层次化配置模式 | 配置如何复用和覆盖 | 分层设计 + 优先级覆盖 |
| 命名空间隔离模式 | 环境如何隔离 | 物理隔离 + 两级分组 |
| 条件装配模式 | 单体/微服务如何切换 | 配置开关 + 条件注解 |

这些模式不仅适用于 Nacos，也适用于任何微服务基础设施的设计。当你在搭建自己的微服务平台时，可以参考这些模式来设计服务注册、配置管理、环境隔离等基础设施。
