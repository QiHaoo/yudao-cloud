# 01 - 设计模式提炼：API 网关

本文从 API 网关组件中提炼可复用的设计模式，帮助读者理解 Spring Cloud Gateway 背后的设计思想，以及网关在微服务架构中的核心价值。

## 过滤器链模式：Gateway 的核心架构

### 模式识别

Spring Cloud Gateway 的核心是**过滤器链模式（Filter Chain Pattern）**。每个请求经过一系列过滤器，每个过滤器负责一个特定的关注点：

```
请求到达 Gateway
    │
    ▼
AccessLogFilter          ← 记录访问日志
    │
    ▼
CorsFilter               ← 处理跨域
    │
    ▼
TokenAuthenticationFilter ← Token 校验
    │
    ▼
GrayLoadBalancer          ← 灰度负载均衡
    │
    ▼
转发到下游服务
```

### 过滤器的两种类型

Gateway 中有两种过滤器类型：

| 类型 | 接口 | 作用范围 | 典型应用 |
|------|------|---------|---------|
| GatewayFilter | `GatewayFilter` | 特定路由 | `RewritePath`、`AddHeader` |
| GlobalFilter | `GlobalFilter` | 所有路由 | `TokenAuthenticationFilter`、`CorsFilter` |

```
GatewayFilter（路由级）：
  spring.cloud.gateway.routes[0].filters:
    - RewritePath=/admin-api/system/v3/api-docs, /v3/api-docs

GlobalFilter（全局级）：
  @Component
  public class TokenAuthenticationFilter implements GlobalFilter { ... }
```

### 过滤器的执行顺序

过滤器通过 `getOrder()` 方法控制执行顺序：

```java
@Component
public class TokenAuthenticationFilter implements GlobalFilter, Ordered {
    @Override
    public int getOrder() {
        return -100;  // 高优先级，先执行
    }
}

@Component
public class AccessLogFilter implements GlobalFilter, Ordered {
    @Override
    public int getOrder() {
        return 0;  // 默认优先级
    }
}
```

执行顺序的规律：
- 值越小，优先级越高，越先执行
- 负数优先于正数
- 相同值的过滤器，注册顺序决定执行顺序

### 与 Servlet Filter 的对比

| 维度 | Gateway GlobalFilter | Servlet Filter |
|------|---------------------|----------------|
| 编程模型 | 响应式（Mono/Flux） | 阻塞式（doFilter） |
| 作用范围 | Gateway 路由 | Servlet 请求 |
| 链式调用 | `chain.filter(exchange)` | `chain.doFilter(request, response)` |
| 上下文 | `ServerWebExchange` | `HttpServletRequest/Response` |

### 通用化提炼

过滤器链模式适合以下场景：

| 特征 | Gateway 中的体现 |
|------|-----------------|
| 请求需要经过多个处理步骤 | 日志、跨域、认证、限流 |
| 处理步骤可插拔 | 可以添加/移除过滤器 |
| 处理步骤有顺序要求 | 认证在限流之前，限流在转发之前 |
| 处理步骤之间共享上下文 | `ServerWebExchange` 传递数据 |

### 电商场景举例：订单创建的前置处理

```java
// 订单创建的过滤器链
public interface OrderCreationFilter {
    Mono<Void> filter(OrderContext context, OrderCreationChain chain);
}

// 库存校验过滤器
@Component
public class StockValidationFilter implements OrderCreationFilter {
    public Mono<Void> filter(OrderContext context, OrderCreationChain chain) {
        return checkStock(context.getOrder())
            .then(chain.filter(context));  // 校验通过，继续下一个过滤器
    }
}

// 风控校验过滤器
@Component
public class RiskControlFilter implements OrderCreationFilter {
    public Mono<Void> filter(OrderContext context, OrderCreationChain chain) {
        return checkRisk(context.getOrder())
            .then(chain.filter(context));
    }
}
```

## 网关模式：微服务的统一入口

### 模式识别

API 网关是**网关模式（Gateway Pattern）**的核心实现。它是微服务架构的统一入口，承担了路由、认证、限流、日志等横切关注点。

```
没有网关：
  前端 → system-server（各自做认证、限流）
  前端 → product-server（各自做认证、限流）
  前端 → trade-server（各自做认证、限流）

有网关：
  前端 → Gateway（统一做认证、限流）→ system-server
                                     → product-server
                                     → trade-server
```

### 网关的核心职责

| 职责 | 实现方式 | 价值 |
|------|---------|------|
| 路由转发 | `spring.cloud.gateway.routes` 配置 | 前端无需知道服务地址 |
| Token 认证 | `TokenAuthenticationFilter` | 统一认证，服务端无需重复实现 |
| 限流熔断 | Sentinel 集成 | 保护后端服务不被压垮 |
| 跨域处理 | `CorsFilter` | 统一跨域策略 |
| 日志记录 | `AccessLogFilter` | 统一日志格式 |
| 灰度发布 | `GrayLoadBalancer` | 支持按版本/标签路由 |

### 网关 vs Nginx

| 维度 | API Gateway | Nginx |
|------|-------------|-------|
| 定位 | 应用层网关 | 反向代理/负载均衡 |
| 动态路由 | 支持（Nacos 集成） | 需要 reload 配置 |
| 业务逻辑 | 可以编写 Java 代码 | 只能用 Lua/配置 |
| 服务发现 | 原生支持 | 需要额外模块 |
| 限流熔断 | Sentinel 集成 | 需要 Lua 脚本 |
| 性能 | 高（响应式） | 极高（C 语言） |

最佳实践：**Nginx 做最外层反向代理 + Gateway 做应用层网关**。

### 通用化提炼

网关模式适合以下场景：

| 特征 | Gateway 中的体现 |
|------|-----------------|
| 微服务数量多 | 10+ 个微服务 |
| 需要统一的横切关注点 | 认证、限流、日志、跨域 |
| 前端需要统一入口 | 所有请求都发到 Gateway |
| 需要动态路由 | 基于 Nacos 的服务发现 |

## 响应式编程模式

### 模式识别

Spring Cloud Gateway 基于 WebFlux 的响应式编程模型。这是与传统 Servlet 模型的根本区别。

```
阻塞式（Servlet）：
  请求 → 线程池分配线程 → 业务处理（阻塞等待 I/O）→ 响应
  问题：线程数 = 并发数，线程是稀缺资源

响应式（WebFlux）：
  请求 → 事件循环 → 回调处理（非阻塞）→ 响应
  优势：少量线程处理大量请求
```

### 响应式的核心概念

| 概念 | 说明 | Gateway 中的体现 |
|------|------|-----------------|
| `Mono<T>` | 0 或 1 个元素的异步序列 | `Mono<Void>` 表示过滤器的异步处理 |
| `Flux<T>` | 0 到 N 个元素的异步序列 | `Flux<ServerSentEvent>` 表示 SSE 流 |
| `Publisher` | 发布者，Mono/Flux 的父接口 | 所有异步操作的起点 |
| `Subscriber` | 订阅者，消费异步数据 | `chain.filter(exchange)` 触发订阅 |

### 响应式的链式调用

```java
@Override
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    return getLoginUser(exchange, token)           // 异步获取用户
        .defaultIfEmpty(LOGIN_USER_EMPTY)          // 默认值
        .flatMap(user -> {                         // 异步处理
            if (user == LOGIN_USER_EMPTY) {
                return chain.filter(exchange);     // 继续过滤器链
            }
            SecurityFrameworkUtils.setLoginUser(exchange, user);
            return chain.filter(exchange.mutate()...build());
        });
}
```

链式调用的特点：
- 每个操作返回新的 `Mono`，形成链式调用
- 没有阻塞，所有操作都是异步的
- `flatMap` 将异步结果展平

### 通用化提炼

响应式编程适合以下场景：

| 特征 | Gateway 中的体现 |
|------|-----------------|
| 高并发 | 网关是所有请求的入口 |
| I/O 密集 | 转发请求到下游服务 |
| 长连接 | WebSocket 支持 |
| 低延迟要求 | 网关的延迟直接影响用户体验 |

## 灰度发布模式

### 模式识别

yudao-cloud 的 `GrayLoadBalancer` 实现了**灰度发布（Gray Release）**模式：根据请求的元数据（version、tag），将请求路由到特定版本的服务实例。

```
灰度路由流程：

  请求 Header: version=2.0.0, tag=gray
      │
      ▼
  GrayLoadBalancer
      │
      ├── 获取所有实例
      ├── 按 version 过滤：只保留 version=2.0.0 的实例
      ├── 按 tag 过滤：只保留 tag=gray 的实例
      └── 随机+权重选择一个实例
```

### 灰度发布的应用场景

| 场景 | 灰度策略 | 说明 |
|------|---------|------|
| 新版本测试 | version=2.0.0 | 只有指定版本的请求才路由到新实例 |
| A/B 测试 | tag=experiment | 不同用户看到不同版本 |
| 金丝雀发布 | tag=canary | 小流量验证新版本 |
| 环境隔离 | tag=dev/test | 不同环境的请求路由到不同实例 |

### 通用化提炼

灰度发布模式适合以下场景：

| 特征 | Gateway 中的体现 |
|------|-----------------|
| 需要渐进式发布 | 新版本逐步替换旧版本 |
| 需要 A/B 测试 | 不同用户看到不同版本 |
| 需要环境隔离 | 开发/测试/生产环境共存 |
| 需要快速回滚 | 切换流量到旧版本 |

## 白名单模式

### 模式识别

`TokenAuthenticationFilter` 中的白名单配置体现了**白名单模式（Whitelist Pattern）**：某些 URL 不需要认证，直接放行。

```
白名单决策流程：

  请求路径: /admin-api/system/auth/login
      │
      ▼
  在白名单中？
      │
      ├── 是 → 直接放行，不做 Token 校验
      │
      └── 否 → 执行 Token 校验流程
```

### 白名单 vs 黑名单

| 维度 | 白名单 | 黑名单 |
|------|--------|--------|
| 策略 | 只允许列表中的项目 | 拒绝列表中的项目 |
| 安全性 | 更安全（默认拒绝） | 可能遗漏 |
| 维护成本 | 新增需要显式添加 | 新增默认允许 |
| 适用场景 | 认证、权限 | 过滤、封禁 |

Gateway 选择白名单的原因：**安全性优先**。只有明确声明的 URL 才能跳过认证，新接口默认需要认证。

### 通用化提炼

白名单模式适合以下场景：

| 特征 | Gateway 中的体现 |
|------|-----------------|
| 大部分请求需要处理 | 绝大部分 URL 需要认证 |
| 少数请求可以跳过 | 登录、注册、健康检查 |
| 安全性要求高 | 默认拒绝，显式允许 |
| 配置可维护 | 白名单集中管理 |

## 反模式警示

### 反模式一：在网关中编写业务逻辑

```java
// 反模式：在网关的过滤器中查询数据库
@Component
public class BusinessFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 在网关中直接查询数据库！
        UserDO user = userMapper.selectById(userId);
        // ...
    }
}
```

**问题**：网关应该是轻量级的转发层，不应该包含业务逻辑。业务逻辑应该在后端服务中。

**正确做法**：网关只做认证、限流、路由等横切关注点，业务逻辑通过 RPC 调用后端服务。

### 反模式二：忽略过滤器顺序

```java
// 反模式：不限流就认证
@Component
public class RateLimitFilter implements GlobalFilter, Ordered {
    public int getOrder() { return -100; }  // 先限流
}

@Component
public class AuthFilter implements GlobalFilter, Ordered {
    public int getOrder() { return -90; }   // 后认证
}
```

**问题**：如果先限流后认证，未认证的请求也会占用限流配额。

**正确做法**：认证在限流之前，只有认证通过的请求才参与限流。

### 反模式三：白名单过于宽泛

```yaml
# 反模式：白名单包含了所有接口
yudao:
  gateway:
    token:
      white-urls:
        - /admin-api/**
        - /app-api/**
```

**问题**：白名单过于宽泛，等于没有认证。

**正确做法**：白名单只包含确实不需要认证的接口（登录、注册、健康检查）。

## 总结

API 网关组件的设计模式提炼：

| 模式 | 解决的问题 | 核心思想 |
|------|-----------|---------|
| 过滤器链模式 | 请求需要经过多个处理步骤 | 链式过滤，逐步增强 |
| 网关模式 | 微服务需要统一入口 | 集中处理横切关注点 |
| 响应式编程 | 高并发下的性能问题 | 非阻塞 I/O，少量线程处理大量请求 |
| 灰度发布模式 | 渐进式发布和 A/B 测试 | 按元数据路由到特定实例 |
| 白名单模式 | 部分请求需要跳过处理 | 默认拒绝，显式允许 |

这些模式不仅适用于 Spring Cloud Gateway，也适用于任何 API 网关的设计。当你在搭建自己的微服务网关时，可以参考这些模式来设计过滤器链、路由机制、认证策略等核心组件。
