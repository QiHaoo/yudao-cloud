# 03 - 跨域与限流配置

> 本章实现网关的跨域处理（CORS）和限流配置。完成本章后，网关将具备统一处理跨域请求的能力，并能通过 Sentinel 进行流量控制。

## 一、跨域处理（CORS）

### 1.1 什么是跨域

浏览器的同源策略限制了跨域请求。当协议、域名、端口任一不同时，就是跨域：

```
同源（允许）：
  http://localhost:8080 → http://localhost:8080  ✓

跨域（被限制）：
  http://localhost:8080 → http://localhost:48080  ✗（端口不同）
  http://localhost:8080 → https://api.example.com  ✗（协议不同）
  http://frontend.com   → http://api.com          ✗（域名不同）
```

在微服务架构中，前端通常部署在 `localhost:8080`，而网关在 `localhost:48080`，跨域问题不可避免。

### 1.2 跨域请求的两种类型

```
简单请求（GET/POST/HEAD）：
  └── 浏览器直接发送，检查响应头中的 CORS 头

复杂请求（PUT/DELETE/自定义头）：
  ├── 1. 浏览器先发送 OPTIONS 预检请求
  │      Origin: http://localhost:8080
  │      Access-Control-Request-Method: PUT
  │
  ├── 2. 服务器返回 CORS 响应头
  │      Access-Control-Allow-Origin: *
  │      Access-Control-Allow-Methods: *
  │
  ├── 3. 浏览器检查通过，发送实际请求
  │
  └── 4. 服务器返回响应
```

### 1.3 CorsFilter 实现

在 Gateway 中通过 `WebFilter` 处理跨域：

```java
// yudao-gateway/src/main/java/cn/iocoder/yudao/gateway/filter/cors/CorsFilter.java
package cn.iocoder.yudao.gateway.filter.cors;

import org.springframework.http.HttpHeaders;
import org.springframework.http.HttpMethod;
import org.springframework.http.HttpStatus;
import org.springframework.http.server.reactive.ServerHttpRequest;
import org.springframework.http.server.reactive.ServerHttpResponse;
import org.springframework.stereotype.Component;
import org.springframework.web.cors.reactive.CorsUtils;
import org.springframework.web.server.ServerWebExchange;
import org.springframework.web.server.WebFilter;
import org.springframework.web.server.WebFilterChain;
import reactor.core.publisher.Mono;

/**
 * 跨域 Filter
 */
@Component
public class CorsFilter implements WebFilter {

    private static final String ALL = "*";
    private static final String MAX_AGE = "3600L";

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        // 非跨域请求，直接放行
        ServerHttpRequest request = exchange.getRequest();
        if (!CorsUtils.isCorsRequest(request)) {
            return chain.filter(exchange);
        }

        // 设置跨域响应头
        ServerHttpResponse response = exchange.getResponse();
        HttpHeaders headers = response.getHeaders();
        headers.add("Access-Control-Allow-Origin", ALL);
        headers.add("Access-Control-Allow-Methods", ALL);
        headers.add("Access-Control-Allow-Headers", ALL);
        headers.add("Access-Control-Max-Age", MAX_AGE);

        // OPTIONS 预检请求，直接返回 200
        if (request.getMethod() == HttpMethod.OPTIONS) {
            response.setStatusCode(HttpStatus.OK);
            return Mono.empty();
        }

        return chain.filter(exchange);
    }

}
```

**关键设计决策：**

| 决策 | 选择 | 理由 |
|------|------|------|
| 实现方式 | WebFilter | 比 GlobalFilter 更早执行，在请求进入 Gateway 核心逻辑之前处理 |
| Allow-Origin | `*`（所有来源） | 开发环境方便，生产环境应限制为具体域名 |
| Allow-Methods | `*`（所有方法） | 简化配置，生产环境可限制为 GET/POST/PUT/DELETE |
| Allow-Headers | `*`（所有头） | 避免自定义头被拒绝，生产环境可限制 |
| Max-Age | 3600 秒 | 预检请求缓存 1 小时，减少 OPTIONS 请求 |

### 1.4 WebFilter vs GlobalFilter

```
WebFilter：
  ├── Spring WebFlux 原生接口
  ├── 在所有 GlobalFilter 之前执行
  ├── 适合：跨域处理、请求日志等通用逻辑
  └── 示例：CorsFilter

GlobalFilter：
  ├── Spring Cloud Gateway 接口
  ├── 通过 order 控制执行顺序
  ├── 适合：路由相关逻辑
  └── 示例：TokenAuthenticationFilter、AccessLogFilter

执行顺序：
  请求 → WebFilter（CorsFilter）→ GlobalFilter（AccessLogFilter → TokenAuthenticationFilter → ...）→ 路由转发
```

### 1.5 生产环境的安全建议

当前配置允许所有来源（`Access-Control-Allow-Origin: *`），在生产环境中应该限制：

```java
// 生产环境的安全配置示例
private static final Set<String> ALLOWED_ORIGINS = Set.of(
    "https://admin.your-domain.com",
    "https://www.your-domain.com"
);

@Override
public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
    String origin = exchange.getRequest().getHeaders().getOrigin();
    if (origin != null && ALLOWED_ORIGINS.contains(origin)) {
        response.getHeaders().add("Access-Control-Allow-Origin", origin);
    }
    // ...
}
```

### 1.6 全局过滤器去重 CORS 头

在 `application.yaml` 中配置了全局过滤器，去重 CORS 响应头：

```yaml
spring:
  cloud:
    gateway:
      default-filters:
        - DedupeResponseHeader=Access-Control-Allow-Credentials Access-Control-Allow-Origin
```

**为什么需要去重？**

当多个地方设置 CORS 头时（如 CorsFilter + Spring Security），响应头中可能出现重复的 `Access-Control-Allow-Origin`。浏览器遇到重复的 CORS 头会拒绝请求。`DedupeResponseHeader` 过滤器自动去重，保留一个。

### 1.7 验证跨域配置

```bash
# 模拟 OPTIONS 预检请求
curl -X OPTIONS http://127.0.0.1:48080/admin-api/system/user/get \
  -H "Origin: http://localhost:8080" \
  -H "Access-Control-Request-Method: GET" \
  -H "Access-Control-Request-Headers: Authorization" \
  -v

# 预期响应头：
# Access-Control-Allow-Origin: *
# Access-Control-Allow-Methods: *
# Access-Control-Allow-Headers: *
# Access-Control-Max-Age: 3600L
```

## 二、访问日志（AccessLogFilter）

### 2.1 AccessLog 数据模型

```java
// yudao-gateway/src/main/java/cn/iocoder/yudao/gateway/filter/logging/AccessLog.java
@Data
public class AccessLog {

    private String traceId;           // 链路追踪编号
    private Long userId;              // 用户编号
    private Integer userType;         // 用户类型
    private Route route;              // 匹配的路由

    private String schema;            // 协议（http/https）
    private String requestMethod;     // 请求方法（GET/POST）
    private String requestUrl;        // 请求路径
    private MultiValueMap<String, String> queryParams;  // 查询参数
    private String requestBody;       // 请求体
    private MultiValueMap<String, String> requestHeaders;  // 请求头
    private String userIp;            // 用户 IP

    private String responseBody;      // 响应体
    private MultiValueMap<String, String> responseHeaders;  // 响应头
    private HttpStatus httpStatus;    // 响应状态码

    private LocalDateTime startTime;  // 开始时间
    private LocalDateTime endTime;    // 结束时间
    private Integer duration;         // 执行时长（毫秒）

}
```

### 2.2 AccessLogFilter 实现

```java
// yudao-gateway/src/main/java/cn/iocoder/yudao/gateway/filter/logging/AccessLogFilter.java
@Slf4j
@Component
public class AccessLogFilter implements GlobalFilter, Ordered {

    @Resource
    private CodecConfigurer codecConfigurer;

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE; // 最高优先级
    }

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 收集请求信息
        ServerHttpRequest request = exchange.getRequest();
        AccessLog gatewayLog = new AccessLog();
        gatewayLog.setRoute(WebFrameworkUtils.getGatewayRoute(exchange));
        gatewayLog.setSchema(request.getURI().getScheme());
        gatewayLog.setRequestMethod(request.getMethodValue());
        gatewayLog.setRequestUrl(request.getURI().getRawPath());
        gatewayLog.setQueryParams(request.getQueryParams());
        gatewayLog.setRequestHeaders(request.getHeaders());
        gatewayLog.setStartTime(LocalDateTime.now());
        gatewayLog.setUserIp(WebFrameworkUtils.getClientIP(exchange));

        // 根据 Content-Type 决定是否读取请求体
        MediaType mediaType = request.getHeaders().getContentType();
        if (MediaType.APPLICATION_FORM_URLENCODED.isCompatibleWith(mediaType)
                || MediaType.APPLICATION_JSON.isCompatibleWith(mediaType)) {
            return filterWithRequestBody(exchange, chain, gatewayLog);
        }
        return filterWithoutRequestBody(exchange, chain, gatewayLog);
    }

    // ... 内部实现省略，详见源码
}
```

**为什么 AccessLogFilter 的 order 是最高优先级？**

AccessLogFilter 需要记录完整的请求信息，包括其他过滤器的处理结果。因此它必须：
- 在其他 GlobalFilter 之前开始（记录请求信息）
- 在其他 GlobalFilter 之后结束（记录响应信息）

它通过装饰器模式包装 Response，在响应写入时才记录日志。

## 三、Sentinel 限流集成

### 3.1 网关限流 vs 服务限流

```
网关限流（Gateway + Sentinel）：
  ├── 在请求进入微服务之前就限流
  ├── 保护所有后端服务
  ├── 基于路由 ID、请求参数限流
  └── 适合：全局限流、API 级别限流

服务限流（Sentinel Starter）：
  ├── 在服务内部限流
  ├── 保护单个服务
  ├── 基于方法、资源限流
  └── 适合：服务级别限流、热点参数限流

两层限流的配合：
  Gateway 限流（第一层）：保护整个系统
      │
      └── 服务限流（第二层）：保护单个服务
```

### 3.2 添加 Sentinel Gateway 依赖

在 `yudao-gateway/pom.xml` 中添加 Sentinel Gateway 依赖：

```xml
<!-- yudao-gateway/pom.xml -->
<dependencies>
    <!-- Sentinel 网关限流 -->
    <dependency>
        <groupId>com.alibaba.cloud</groupId>
        <artifactId>spring-cloud-alibaba-sentinel-gateway</artifactId>
    </dependency>

    <!-- Sentinel 动态规则（Nacos 数据源） -->
    <dependency>
        <groupId>com.alibaba.cloud</groupId>
        <artifactId>spring-cloud-alibaba-sentinel-datasource-nacos</artifactId>
    </dependency>
</dependencies>
```

**依赖说明：**

| 依赖 | 职责 |
|------|------|
| `sentinel-gateway` | 提供 Gateway 专用的限流能力（基于路由 ID 限流） |
| `sentinel-datasource-nacos` | 从 Nacos 动态拉取限流规则 |

### 3.3 Sentinel 配置

在 `application.yaml` 中添加 Sentinel 配置：

```yaml
spring:
  cloud:
    sentinel:
      # Sentinel Dashboard 地址
      transport:
        dashboard: localhost:8080
        port: 8719  # 与 Dashboard 通信的端口

      # 从 Nacos 动态拉取限流规则
      datasource:
        # 网关限流规则
        gateway-flow:
          nacos:
            server-addr: ${spring.cloud.nacos.server-addr}
            data-id: ${spring.application.name}-gateway-flow-rules
            group-id: SENTINEL_GROUP
            data-type: json
            rule-type: gw-flow
        # 网关 API 分组限流规则
        gateway-api-flow:
          nacos:
            server-addr: ${spring.cloud.nacos.server-addr}
            data-id: ${spring.application.name}-gateway-api-rules
            group-id: SENTINEL_GROUP
            data-type: json
            rule-type: gw-api-group
```

**配置项说明：**

| 配置项 | 说明 |
|--------|------|
| `transport.dashboard` | Sentinel Dashboard 地址，用于实时监控和规则管理 |
| `transport.port` | 客户端与 Dashboard 通信的端口 |
| `datasource.gateway-flow` | 网关限流规则的数据源 |
| `datasource.gateway-api-flow` | 网关 API 分组规则的数据源 |
| `rule-type: gw-flow` | 网关限流规则类型 |
| `rule-type: gw-api-group` | 网关 API 分组规则类型 |

### 3.4 限流规则配置

在 Nacos 中创建限流规则配置：

**Data ID**: `gateway-server-gateway-flow-rules`
**Group**: `SENTINEL_GROUP`
**格式**: JSON

```json
[
  {
    "resource": "system-admin-api",
    "grade": 1,
    "count": 1000,
    "strategy": 0,
    "controlBehavior": 0
  },
  {
    "resource": "product-admin-api",
    "grade": 1,
    "count": 500,
    "strategy": 0,
    "controlBehavior": 0
  },
  {
    "resource": "trade-admin-api",
    "grade": 1,
    "count": 200,
    "strategy": 0,
    "controlBehavior": 0
  }
]
```

**规则参数说明：**

| 参数 | 说明 | 取值 |
|------|------|------|
| `resource` | 资源名，对应路由 ID | `system-admin-api` |
| `grade` | 限流模式 | 0=线程数，1=QPS |
| `count` | 限流阈值 | `1000`（每秒 1000 个请求） |
| `strategy` | 流控策略 | 0=直接拒绝，1=关联，2=链路 |
| `controlBehavior` | 流控效果 | 0=快速失败，1=Warm Up，2=排队等待 |

## 四、灰度负载均衡（GrayLoadBalancer）

### 4.1 灰度路由的原理

灰度路由允许根据请求特征将流量导向特定的服务实例：

```
请求头：version=2.0.0, tag=gray

实例列表：
  ├── 实例1: metadata={version: 1.0.0}           ← 不匹配
  ├── 实例2: metadata={version: 2.0.0}           ← 匹配
  ├── 实例3: metadata={version: 2.0.0, tag: gray} ← 匹配
  └── 实例4: metadata={version: 2.0.0, tag: prod} ← tag 不匹配

选择范围：[实例2, 实例3]
最终选择：实例2 或 实例3（随机+权重）
```

### 4.2 GrayLoadBalancer 实现

```java
// yudao-gateway/src/main/java/cn/iocoder/yudao/gateway/filter/grey/GrayLoadBalancer.java
@RequiredArgsConstructor
@Slf4j
public class GrayLoadBalancer implements ReactorServiceInstanceLoadBalancer {

    private static final String VERSION = "version";

    private final ObjectProvider<ServiceInstanceListSupplier> serviceInstanceListSupplierProvider;
    private final String serviceId;

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        HttpHeaders headers = ((RequestDataContext) request.getContext()).getClientRequest().getHeaders();
        ServiceInstanceListSupplier supplier = serviceInstanceListSupplierProvider
                .getIfAvailable(NoopServiceInstanceListSupplier::new);
        return supplier.get(request).next().map(list -> getInstanceResponse(list, headers));
    }

    private Response<ServiceInstance> getInstanceResponse(List<ServiceInstance> instances, HttpHeaders headers) {
        if (CollUtil.isEmpty(instances)) {
            return new EmptyResponse();
        }

        // 1. 按 version 过滤
        String version = headers.getFirst(VERSION);
        List<ServiceInstance> chooseInstances;
        if (StrUtil.isEmpty(version)) {
            chooseInstances = instances;
        } else {
            chooseInstances = CollectionUtils.filterList(instances,
                    instance -> version.equals(instance.getMetadata().get("version")));
            if (CollUtil.isEmpty(chooseInstances)) {
                chooseInstances = instances; // 没有匹配的版本，使用所有实例
            }
        }

        // 2. 按 tag 过滤
        chooseInstances = filterTagServiceInstances(chooseInstances, headers);

        // 3. 随机+权重选择
        return new DefaultResponse(NacosBalancer.getHostByRandomWeight3(chooseInstances));
    }

}
```

### 4.4 灰度发布场景

```
场景：新版本上线，逐步放量

1. 部署新版本实例，metadata.version = 2.0.0
2. 前端请求头携带 version=2.0.0
3. 只有 version=2.0.0 的请求会路由到新版本实例
4. 其他请求仍然路由到旧版本实例
5. 验证无问题后，逐步增加新版本实例，减少旧版本实例
```

## 五、全局异常处理

### 5.1 GlobalExceptionHandler

网关的异常处理与业务服务不同，需要实现 `ErrorWebExceptionHandler`：

```java
// yudao-gateway/src/main/java/cn/iocoder/yudao/gateway/handler/GlobalExceptionHandler.java
@Component
@Order(-1) // 优先级高于默认的异常处理器
@Slf4j
public class GlobalExceptionHandler implements ErrorWebExceptionHandler {

    @Override
    public Mono<Void> handle(ServerWebExchange exchange, Throwable ex) {
        ServerHttpResponse response = exchange.getResponse();
        if (response.isCommitted()) {
            return Mono.error(ex);
        }

        // 转换成 CommonResult
        CommonResult<?> result;
        if (ex instanceof ResponseStatusException) {
            result = responseStatusExceptionHandler(exchange, (ResponseStatusException) ex);
        } else {
            result = defaultExceptionHandler(exchange, ex);
        }

        return WebFrameworkUtils.writeJSON(exchange, result);
    }

    private CommonResult<?> responseStatusExceptionHandler(ServerWebExchange exchange,
                                                           ResponseStatusException ex) {
        log.error("[responseStatusExceptionHandler][uri({}/{}) 发生异常]",
                exchange.getRequest().getURI(), exchange.getRequest().getMethod(), ex);
        return CommonResult.error(ex.getRawStatusCode(), ex.getReason());
    }

    private CommonResult<?> defaultExceptionHandler(ServerWebExchange exchange, Throwable ex) {
        log.error("[defaultExceptionHandler][uri({}/{}) 发生异常]",
                exchange.getRequest().getURI(), exchange.getRequest().getMethod(), ex);
        return CommonResult.error(INTERNAL_SERVER_ERROR.getCode(), INTERNAL_SERVER_ERROR.getMsg());
    }

}
```

**为什么需要单独的异常处理器？**

Gateway 基于 WebFlux，不能使用 `@RestControllerAdvice` + `@ExceptionHandler`（那是 WebMvc 的方式）。必须实现 `ErrorWebExceptionHandler` 接口。

### 5.2 Jackson 配置

Gateway 的 Jackson 配置需要与业务服务保持一致：

```java
// yudao-gateway/src/main/java/cn/iocoder/yudao/gateway/jackson/GatewayJacksonAutoConfiguration.java
@Configuration
public class GatewayJacksonAutoConfiguration {

    @Bean
    public Jackson2ObjectMapperBuilderCustomizer ldtEpochMillisCustomizer() {
        return builder -> builder
                .serializerByType(Long.class, NumberSerializer.INSTANCE)
                .serializerByType(Long.TYPE, NumberSerializer.INSTANCE)
                .serializerByType(LocalDateTime.class, TimestampLocalDateTimeSerializer.INSTANCE)
                .deserializerByType(LocalDateTime.class, TimestampLocalDateTimeDeserializer.INSTANCE);
                // ... 其他序列化配置
    }

    @Bean
    public CodecCustomizer unifyJackson(ObjectMapper om) {
        return configurer -> {
            Jackson2JsonDecoder decoder = new Jackson2JsonDecoder(om);
            Jackson2JsonEncoder encoder = new Jackson2JsonEncoder(om);
            configurer.defaultCodecs().jackson2JsonDecoder(decoder);
            configurer.defaultCodecs().jackson2JsonEncoder(encoder);
        };
    }

}
```

**为什么需要 `CodecCustomizer`？**

Gateway 使用 WebFlux 的 Codec 机制处理 JSON，而不是直接使用 ObjectMapper。通过 `CodecCustomizer` 将自定义的 ObjectMapper 注入到 Codec 中，保证序列化行为一致。
