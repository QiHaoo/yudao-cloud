# 01 - 起步：网关项目结构

> 本章是 API 网关功能实现的起点。完成本章后，你将拥有一个可启动的 Gateway 模块，能够接收请求并根据路由规则转发到下游微服务。

## 前置准备

在开始之前，请确保：

1. 已完成第一阶段的项目骨架搭建（Maven 多模块结构）
2. 已完成第四阶段组件 15（Nacos 集成）和组件 16（OpenFeign）的功能实现
3. 已阅读组件 17 的功能设计文档
4. Nacos Server 已启动，至少有一个微服务（如 system-server）已注册

**为什么 Gateway 是一个独立模块？**

Gateway 是微服务架构的入口，它的职责与业务模块完全不同：

| 维度 | Gateway | 业务模块（如 system-server） |
|------|---------|---------------------------|
| 编程模型 | 响应式（WebFlux） | 阻塞式（Servlet） |
| 依赖 | spring-cloud-starter-gateway | spring-boot-starter-web |
| 主要职责 | 路由转发、认证、限流 | 业务逻辑处理 |
| 是否包含业务代码 | 否 | 是 |

由于 Gateway 基于 WebFlux，而业务模块基于 Servlet，两者不能混在同一个模块中，否则会冲突。

## 第二步：pom.xml 依赖配置

```xml
<!-- yudao-gateway/pom.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
```

**依赖说明：**

| 依赖 | 职责 | 为什么需要 |
|------|------|-----------|
| `spring-cloud-starter-gateway` | Gateway 核心 | 提供路由、过滤器、WebFlux 运行时 |
| `yudao-module-system-api` | 系统模块 API | 网关需要调用 system-server 的 Token 校验接口 |
| `spring-cloud-starter-loadbalancer` | 负载均衡 | 将 `grayLb://service-name` 解析为实际的实例地址 |
| `nacos-discovery` | 服务发现 | 从 Nacos 获取服务实例列表 |
| `nacos-config` | 配置中心 | 从 Nacos 拉取路由等配置 |
| `knife4j-gateway` | API 文档聚合 | 将各服务的 Swagger 文档聚合到网关统一展示 |
| `guava` | 本地缓存 | 缓存 Token 校验结果，减少远程调用 |

**注意排除 `springdoc-openapi-webmvc-core`：**

`yudao-module-system-api` 间接依赖了 WebMvc 版本的 OpenAPI，而 Gateway 使用 WebFlux，两者冲突。需要排除 WebMvc 版本。

## 第四步：启动类

```java
// yudao-gateway/src/main/java/cn/iocoder/yudao/gateway/GatewayServerApplication.java
package cn.iocoder.yudao.gateway;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class GatewayServerApplication {

    public static void main(String[] args) {
        SpringApplication.run(GatewayServerApplication.class, args);
    }

}
```

**为什么启动类如此简洁？**

Gateway 的启动类不需要额外的注解，因为：
- `@SpringBootApplication` 已包含 `@EnableAutoConfiguration`，会自动配置 Gateway
- 路由配置在 `application.yaml` 中，不需要 Java 配置类
- 过滤器通过 `@Component` 自动注册，不需要手动声明

## 第五步：主配置文件 application.yaml

```yaml
# yudao-gateway/src/main/resources/application.yaml
spring:
  application:
    name: gateway-server

  profiles:
    active: local

  # 缓冲区大小，Gateway 需要缓存请求体/响应体用于日志记录
  codec:
    max-in-memory-size: 10MB

  # Jackson 配置：时间戳格式，与前端保持一致
  jackson:
    serialization:
      write-dates-as-timestamps: true
      write-date-timestamps-as-nanoseconds: false
      write-durations-as-timestamps: true
      fail-on-empty-beans: false

  main:
    allow-circular-references: true  # 允许循环依赖（三层架构无法避免）

  config:
    import:
      - optional:classpath:application-${spring.profiles.active}.yaml
      - optional:nacos:${spring.application.name}-${spring.profiles.active}.yaml

  cloud:
    gateway:
      # 路由配置
      routes:
        ## system-server 服务
        - id: system-admin-api
          uri: grayLb://system-server
          predicates:
            - Path=/admin-api/system/**
          filters:
            - RewritePath=/admin-api/system/v3/api-docs, /v3/api-docs
        - id: system-app-api
          uri: grayLb://system-server
          predicates:
            - Path=/app-api/system/**
          filters:
            - RewritePath=/app-api/system/v3/api-docs, /v3/api-docs

        ## infra-server 服务
        - id: infra-admin-api
          uri: grayLb://infra-server
          predicates:
            - Path=/admin-api/infra/**
          filters:
            - RewritePath=/admin-api/infra/v3/api-docs, /v3/api-docs
        - id: infra-app-api
          uri: grayLb://infra-server
          predicates:
            - Path=/app-api/infra/**

        ## product-server 服务
        - id: product-admin-api
          uri: grayLb://product-server
          predicates:
            - Path=/admin-api/product/**
          filters:
            - RewritePath=/admin-api/product/v3/api-docs, /v3/api-docs
        - id: product-app-api
          uri: grayLb://product-server
          predicates:
            - Path=/app-api/product/**

        ## trade-server 服务
        - id: trade-admin-api
          uri: grayLb://trade-server
          predicates:
            - Path=/admin-api/trade/**
          filters:
            - RewritePath=/admin-api/trade/v3/api-docs, /v3/api-docs
        - id: trade-app-api
          uri: grayLb://trade-server
          predicates:
            - Path=/app-api/trade/**

        # ... 更多服务路由，按需添加

      # 禁用 X-Forwarded-Prefix，避免 Swagger 重复带上前缀
      x-forwarded:
        prefix-enabled: false

      # 全局过滤器：去重 CORS 响应头
      default-filters:
        - DedupeResponseHeader=Access-Control-Allow-Credentials Access-Control-Allow-Origin

server:
  port: 48080  # 网关端口

logging:
  file:
    name: ${user.home}/logs/${spring.application.name}.log

# Knife4j 聚合文档配置
knife4j:
  gateway:
    enabled: true
    routes:
      - name: system-server
        service-name: system-server
        url: /admin-api/system/v3/api-docs
      - name: infra-server
        service-name: infra-server
        url: /admin-api/infra/v3/api-docs
      - name: product-server
        service-name: product-server
        url: /admin-api/product/v3/api-docs
      - name: trade-server
        service-name: trade-server
        url: /admin-api/trade/v3/api-docs
      # ... 更多服务的文档路由

--- #################### 芋道相关配置 ####################
yudao:
  info:
    version: 1.0.0
```

**路由配置详解：**

每条路由包含三个核心属性：

```yaml
- id: system-admin-api          # 路由唯一标识，命名规范：{服务名}-{api类型}
  uri: grayLb://system-server    # 目标服务，grayLb 表示灰度负载均衡
  predicates:                    # 断言：请求满足什么条件时匹配此路由
    - Path=/admin-api/system/**  # 路径匹配：以 /admin-api/system/ 开头的请求
  filters:                       # 过滤器：对请求/响应进行处理
    - RewritePath=/admin-api/system/v3/api-docs, /v3/api-docs  # 重写路径
```

**`grayLb://` 协议：**

这是 yudao-cloud 自定义的协议，代替标准的 `lb://` 协议。它在负载均衡的基础上增加了灰度路由能力：

```
grayLb://system-server
  │
  ├── 解析服务名：system-server
  ├── 从 Nacos 获取实例列表
  ├── 根据请求头 version 过滤实例（灰度）
  ├── 根据请求头 tag 过滤实例（环境隔离）
  └── 随机+权重选择一个实例
```

**`RewritePath` 过滤器的作用：**

各服务的 Swagger 文档路径是 `/v3/api-docs`，但通过网关访问时路径变成了 `/admin-api/system/v3/api-docs`。`RewritePath` 负责将路径还原，保证文档能正确返回。

## 路由匹配流程

理解请求如何被路由到正确的服务：

```
请求：GET /admin-api/system/user/get?id=1

1. Gateway 遍历所有路由规则
   │
   ├── 检查 system-admin-api: Path=/admin-api/system/**
   │   └── 匹配！选择此路由
   │
2. 获取目标服务：system-server
   │
3. GrayLoadBalancer 选择实例
   │
   ├── 从 Nacos 获取实例列表
   ├── 按 version/tag 过滤
   └── 随机+权重选择：192.168.1.10:48081
   │
4. 转发请求
   │
   └── GET http://192.168.1.10:48081/admin-api/system/user/get?id=1
```

## 添加新服务的路由

当你新增一个业务模块（如 `crm-server`）时，只需在 `application.yaml` 中添加路由配置：

```yaml
spring:
  cloud:
    gateway:
      routes:
        # ... 已有路由 ...

        ## crm-server 服务
        - id: crm-admin-api
          uri: grayLb://crm-server
          predicates:
            - Path=/admin-api/crm/**
          filters:
            - RewritePath=/admin-api/crm/v3/api-docs, /v3/api-docs
        - id: crm-app-api
          uri: grayLb://crm-server
          predicates:
            - Path=/app-api/crm/**
```

同时在 `knife4j.gateway.routes` 中添加文档路由：

```yaml
knife4j:
  gateway:
    routes:
      # ... 已有路由 ...
      - name: crm-server
        service-name: crm-server
        url: /admin-api/crm/v3/api-docs
```

**路由命名规范：**

| 路由 ID 格式 | 说明 | 示例 |
|-------------|------|------|
| `{module}-admin-api` | 管理后台 API | `system-admin-api` |
| `{module}-app-api` | 用户端 API | `product-app-api` |

## 常见问题排查

### 问题一：启动报错 WebFlux 和 WebMvc 冲突

**现象**：`The bean 'xxx' could not be injected as a 'xxx' because it is a JDK dynamic proxy`

**原因**：Gateway 使用 WebFlux，如果依赖了使用 WebMvc 的组件会冲突。

**解决**：检查 `yudao-module-system-api` 是否排除了 `springdoc-openapi-webmvc-core`。

### 问题二：路由 404

**现象**：访问 `/admin-api/system/xxx` 返回 404

**排查步骤**：
1. 检查 `predicates` 中的 `Path` 配置是否正确
2. 检查目标服务是否已注册到 Nacos
3. 检查目标服务是否正常运行

### 问题三：Knife4j 文档空白

**现象**：访问 `/doc.html` 但没有文档

**排查步骤**：
1. 检查目标服务的 Swagger 是否正常（直接访问目标服务的 `/v3/api-docs`）
2. 检查 `knife4j.gateway.routes` 配置是否正确
3. 检查 `RewritePath` 过滤器是否配置
