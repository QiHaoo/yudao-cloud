# 01 - 总览：API 网关全景

## 场景引入

假设你的电商系统已经拆分为多个微服务，前端需要调用这些服务来完成功能。但你很快会遇到以下问题：

**问题一：前端如何知道调用哪个服务？**

```
前端需要调用的接口：
  ├── 用户管理：http://system-server:48081/admin-api/system/user/get
  ├── 商品管理：http://product-server:48082/admin-api/product/spu/get
  ├── 订单管理：http://trade-server:48083/admin-api/trade/order/get
  └── 促销管理：http://promotion-server:48084/admin-api/promotion/coupon/get

问题：
  ├── 前端需要知道每个服务的地址和端口
  ├── 服务扩缩容时，前端配置需要同步修改
  ├── 跨域问题：不同服务的域名/端口不同
  └── 安全问题：直接暴露所有服务的地址
```

**问题二：认证鉴权在哪里做？**

```
每个服务都需要：
  ├── 解析 Token
  ├── 验证 Token 有效性
  ├── 获取用户信息
  └── 检查权限

问题：
  ├── 认证逻辑在每个服务中重复
  ├── 如果认证逻辑变更，需要修改所有服务
  └── Token 校验的性能开销分散在每个服务
```

**问题三：限流在哪里做？**

```
大促场景：
  ├── 秒杀活动开始，瞬间涌入大量请求
  ├── 如果不限流，服务会被压垮
  └── 限流应该在哪里做？

问题：
  ├── 每个服务单独限流，无法全局控制
  ├── 限流规则分散在各个服务中
  └── 无法实现全局的流量调度
```

## 解决方案：API 网关

API 网关是微服务架构的**统一入口**，所有外部请求都先经过网关，再转发到对应的微服务。

```
                    ┌─────────────────────────────┐
                    │          前端 / 客户端        │
                    └──────────────┬──────────────┘
                                   │
                                   │ 所有请求都发到网关
                                   ▼
                    ┌─────────────────────────────┐
                    │         API Gateway          │
                    │                             │
                    │  ┌───────────────────────┐  │
                    │  │ 路由转发              │  │  ← 根据路径分发到不同服务
                    │  │ Token 校验            │  │  ← 统一认证
                    │  │ 限流熔断              │  │  ← 统一限流
                    │  │ 日志记录              │  │  ← 统一日志
                    │  │ 跨域处理              │  │  ← 统一跨域
                    │  └───────────────────────┘  │
                    └──────────────┬──────────────┘
                                   │
          ┌────────────────────────┼────────────────────────┐
          │                        │                        │
          ▼                        ▼                        ▼
┌──────────────┐        ┌──────────────┐        ┌──────────────┐
│system-server │        │product-server│        │ trade-server │
└──────────────┘        └──────────────┘        └──────────────┘
```

## Spring Cloud Gateway 的响应式架构

### 为什么选择 Spring Cloud Gateway

| 维度 | Spring Cloud Gateway | Zuul 1.x |
|------|---------------------|----------|
| 编程模型 | 响应式（WebFlux） | 阻塞式（Servlet） |
| 性能 | 高 | 中 |
| 长连接支持 | 好（WebSocket） | 差 |
| 过滤器模型 | 链式过滤器 | 拦截器 |
| Spring Cloud 集成 | 原生支持 | 需要额外适配 |
| 维护状态 | 活跃 | 维护模式 |

### 响应式 vs 阻塞式

```
阻塞式（Zuul 1.x）：
  请求 → 线程1 → 业务处理（阻塞等待）→ 响应
  请求 → 线程2 → 业务处理（阻塞等待）→ 响应
  请求 → 线程3 → 业务处理（阻塞等待）→ 响应
  ...
  线程池耗尽 → 新请求等待

响应式（Spring Cloud Gateway）：
  请求 → 事件循环 → 回调处理（非阻塞）→ 响应
  请求 → 事件循环 → 回调处理（非阻塞）→ 响应
  请求 → 事件循环 → 回调处理（非阻塞）→ 响应
  ...
  少量线程处理大量请求
```

响应式的优势：
- 更少的线程，更低的内存开销
- 更高的并发处理能力
- 更好的长连接支持

## 核心能力

### 1. 路由转发

根据请求路径，将请求转发到对应的微服务：

```
请求：GET /admin-api/system/user/get?id=1
  │
  ▼
路由匹配：Path=/admin-api/system/**
  │
  ▼
转发到：system-server
  │
  ▼
实际请求：GET http://system-server:48081/admin-api/system/user/get?id=1
```

### 2. Token 认证

统一处理 Token 校验，将用户信息注入到请求头：

```
请求：GET /admin-api/system/user/get?id=1
Header: Authorization: Bearer xxx
  │
  ▼
TokenAuthenticationFilter
  │
  ├── 从请求头提取 Token
  ├── 调用 system-server 校验 Token
  ├── 获取用户信息（userId、userType、tenantId）
  └── 将用户信息注入到请求头 login-user
  │
  ▼
转发到 system-server（携带 login-user 请求头）
```

### 3. 限流熔断

防止服务被过载请求压垮：

```
请求 → 网关 → Sentinel 限流检查
                  │
                  ├── 未超限 → 正常转发
                  └── 超限 → 返回 429 Too Many Requests
```

### 4. 日志记录

统一记录所有请求的日志：

```
AccessLogFilter 记录：
  ├── 请求方法、URL、参数
  ├── 请求头、请求体
  ├── 响应状态码、响应体
  ├── 用户 IP
  ├── 执行时间
  └── 用户 ID（如果已登录）
```

## 思考题

1. **API 网关和 Nginx 都能做反向代理和负载均衡，它们有什么区别？在什么场景下需要同时使用两者？**

2. **Spring Cloud Gateway 基于 WebFlux 的响应式架构，这意味着它不能使用传统的 Servlet API。如果某个过滤器需要使用 Servlet API（比如获取 HttpServletRequest），该如何处理？**

3. **网关是所有请求的入口，如果网关本身宕机了怎么办？如何保证网关的高可用？**

4. **Token 认证放在网关做，还是放在每个微服务做？两种方式各有什么优缺点？**
