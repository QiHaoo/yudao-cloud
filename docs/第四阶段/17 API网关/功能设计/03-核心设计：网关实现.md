# 03 - 核心设计：网关实现

## 路由配置：基于 Nacos 服务发现的动态路由

### 静态路由配置

yudao-cloud 的路由配置写在 `application.yaml` 中：

```yaml
spring:
  cloud:
    gateway:
      routes:
        ## system-server 服务
        - id: system-admin-api          # 路由编号
          uri: grayLb://system-server    # 目标服务（支持灰度的负载均衡）
          predicates:                    # 断言（匹配条件）
            - Path=/admin-api/system/**  # 路径匹配
          filters:                       # 过滤器
            - RewritePath=/admin-api/system/v3/api-docs, /v3/api-docs

        - id: system-app-api
          uri: grayLb://system-server
          predicates:
            - Path=/app-api/system/**

        ## product-server 服务
        - id: product-admin-api
          uri: grayLb://product-server
          predicates:
            - Path=/admin-api/product/**

        - id: product-app-api
          uri: grayLb://product-server
          predicates:
            - Path=/app-api/product/**
```

### 路由匹配流程

```
请求：GET /admin-api/system/user/get?id=1

1. 遍历所有路由规则
   │
   ├── 检查 route1: Path=/admin-api/system/**
   │   └── 匹配！选择此路由
   │
2. 获取目标服务：system-server
   │
3. 通过 LoadBalancer 选择实例
   │
   ├── Nacos 返回实例列表：
   │   ├── 192.168.1.10:48081
   │   └── 192.168.1.11:48081
   │
   └── 选择：192.168.1.10:48081（随机+权重）
   │
4. 转发请求
   │
   └── GET http://192.168.1.10:48081/admin-api/system/user/get?id=1
```

### grayLb 协议

yudao-cloud 使用自定义的 `grayLb` 协议代替标准的 `lb` 协议，支持灰度路由：

```
grayLb://system-server
  │
  ├── 解析服务名：system-server
  ├── GrayLoadBalancer 选择实例
  │   ├── 获取所有实例
  │   ├── 根据请求头 version 过滤
  │   ├── 根据请求头 tag 过滤
  │   └── 随机+权重选择
  └── 返回选中的实例地址
```

灰度路由的匹配逻辑：

```
请求头：version=2.0.0, tag=gray

实例列表：
  ├── 实例1: metadata={version: 1.0.0}           ← 不匹配
  ├── 实例2: metadata={version: 2.0.0}           ← 匹配
  ├── 实例3: metadata={version: 2.0.0, tag: gray} ← 匹配
  └── 实例4: metadata={version: 2.0.0, tag: prod} ← tag 不匹配

选择范围：[实例2, 实例3]
最终选择：实例2 或 实例3（随机+权重）
```

## TokenAuthenticationFilter：Token 校验 + 用户信息注入

### 过滤器职责

TokenAuthenticationFilter 是网关最核心的过滤器，负责：

1. 提取请求中的 Token
2. 校验 Token 有效性
3. 获取用户信息
4. 将用户信息注入到请求头
5. 转发到下游服务

### 实现细节

```java
@Component
public class TokenAuthenticationFilter implements GlobalFilter, Ordered {

    private final WebClient webClient;
    private final LoadingCache<KeyValue<Long, String>, LoginUser> loginUserCache;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 移除可能伪造的 login-user 请求头
        exchange = SecurityFrameworkUtils.removeLoginUser(exchange);

        // 2. 提取 Token
        String token = SecurityFrameworkUtils.obtainAuthorization(exchange);
        if (StrUtil.isEmpty(token)) {
            return chain.filter(exchange);  // 无 Token，直接放行
        }

        // 3. 校验 Token 并获取用户信息
        return getLoginUser(exchange, token)
            .defaultIfEmpty(LOGIN_USER_EMPTY)
            .flatMap(user -> {
                if (user == LOGIN_USER_EMPTY || user.isExpired()) {
                    return chain.filter(exchange);  // Token 无效或过期，放行（交给服务端判断）
                }

                // 4. 设置登录用户到 exchange 属性
                SecurityFrameworkUtils.setLoginUser(exchange, user);

                // 5. 将用户信息注入到请求头
                ServerWebExchange newExchange = exchange.mutate()
                    .request(builder -> SecurityFrameworkUtils.setLoginUserHeader(builder, user))
                    .build();
                return chain.filter(newExchange);
            });
    }

    @Override
    public int getOrder() {
        return -100;  // 高优先级
    }
}
```

### Token 校验流程

```
请求到达 Gateway
    │
    ▼
提取 Authorization: Bearer xxx
    │
    ├── Token 为空 → 直接放行（交给服务端判断是否需要登录）
    │
    ▼
从缓存获取用户信息
    │
    ├── 缓存命中 → 直接使用
    │
    ├── 缓存未命中 → 调用 system-server 校验 Token
    │   │
    │   ├── GET http://system-server/rpc-api/system/oauth2/token/check?accessToken=xxx
    │   │
    │   ├── 返回成功 → 构建 LoginUser，放入缓存
    │   └── 返回失败 → 返回空
    │
    ▼
注入用户信息到请求头
    │
    ├── login-user: {"id":1,"userType":1,"tenantId":1,...}
    │
    ▼
转发到下游服务
```

### 缓存机制

Token 校验结果会缓存 1 分钟，减少远程调用：

```java
private final LoadingCache<KeyValue<Long, String>, LoginUser> loginUserCache =
    buildAsyncReloadingCache(Duration.ofMinutes(1), new CacheLoader<>() {
        @Override
        public LoginUser load(KeyValue<Long, String> token) {
            // 调用 system-server 校验 Token
            String body = checkAccessToken(token.getKey(), token.getValue()).block();
            return buildUser(body);
        }
    });
```

缓存 Key：`tenantId + token`（不同租户的 Token 独立缓存）

## 租户 ID 透传

### 透传机制

网关从请求头提取租户 ID，转发到下游服务：

```
前端请求
    │
    ▼
Header: tenant-id: 1
    │
    ▼
Gateway 的 TokenAuthenticationFilter
    │
    ├── 从请求头提取 tenant-id
    ├── 从 Token 校验结果中获取 tenantId
    └── 两者取优先级高的（Token 中的 tenantId）
    │
    ▼
注入到请求头 login-user 中
    │
    ▼
下游服务从 login-user 中解析 tenantId
    │
    └── TenantContextHolder.setTenantId(tenantId)
```

### WebFrameworkUtils 工具类

```java
public class WebFrameworkUtils {

    private static final String HEADER_TENANT_ID = "tenant-id";

    public static Long getTenantId(ServerWebExchange exchange) {
        String tenantId = exchange.getRequest().getHeaders().getFirst(HEADER_TENANT_ID);
        return NumberUtil.isNumber(tenantId) ? Long.valueOf(tenantId) : null;
    }

    public static void setTenantIdHeader(Long tenantId, HttpHeaders httpHeaders) {
        if (tenantId == null) {
            return;
        }
        httpHeaders.set(HEADER_TENANT_ID, String.valueOf(tenantId));
    }
}
```

## 白名单配置：无需认证的 URL

### 白名单场景

某些 URL 不需要认证，比如：
- 登录接口
- 注册接口
- 公开的 API
- 健康检查接口

### 实现方式

在 TokenAuthenticationFilter 中，白名单 URL 直接放行：

```java
// 白名单路径（示例）
private static final String[] WHITE_LIST = {
    "/admin-api/system/auth/login",
    "/admin-api/system/auth/refresh-token",
    "/app-api/member/auth/login",
    "/actuator/**"
};

@Override
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    String path = exchange.getRequest().getURI().getPath();

    // 检查是否在白名单中
    if (isWhiteListed(path)) {
        return chain.filter(exchange);  // 直接放行
    }

    // 正常的 Token 校验流程
    // ...
}
```

### 白名单配置化

更优雅的方式是将白名单配置在 `application.yaml` 中：

```yaml
yudao:
  gateway:
    token:
      white-urls:
        - /admin-api/system/auth/login
        - /admin-api/system/auth/refresh-token
        - /app-api/member/auth/login
        - /actuator/**
```

## 跨域配置（CORS）

### 什么是跨域

浏览器的同源策略限制了跨域请求：

```
同源：协议 + 域名 + 端口 都相同
  http://localhost:8080 → http://localhost:8080  ✓

跨源：协议、域名、端口 任一不同
  http://localhost:8080 → http://localhost:48080  ✗（端口不同）
  http://localhost:8080 → https://api.example.com  ✗（协议不同）
```

### yudao-cloud 的跨域处理

在 Gateway 中通过 `WebFilter` 处理跨域：

```java
@Component
public class CorsFilter implements WebFilter {

    private static final String ALL = "*";
    private static final String MAX_AGE = "3600L";

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();

        // 非跨域请求，直接放行
        if (!CorsUtils.isCorsRequest(request)) {
            return chain.filter(exchange);
        }

        // 设置跨域响应头
        ServerHttpResponse response = exchange.getResponse();
        HttpHeaders headers = response.getHeaders();
        headers.add("Access-Control-Allow-Origin", ALL);
        headers.add("Access-Control-Allow-Methods", ALL);
        headers.add("Access-Control-Allow-Headers", ALL);
        headers.add("Access-Control-Max-Age", MAX_AGE);

        // OPTIONS 预检请求，直接返回
        if (request.getMethod() == HttpMethod.OPTIONS) {
            response.setStatusCode(HttpStatus.OK);
            return Mono.empty();
        }

        return chain.filter(exchange);
    }
}
```

### 跨域请求流程

```
浏览器发起跨域请求：
  │
  ├── 简单请求（GET/POST）：
  │   └── 直接发送，浏览器检查响应头
  │
  └── 复杂请求（PUT/DELETE/自定义头）：
      │
      ├── 1. 浏览器先发送 OPTIONS 预检请求
      │      Origin: http://localhost:8080
      │      Access-Control-Request-Method: PUT
      │
      ├── 2. 服务器返回 CORS 响应头
      │      Access-Control-Allow-Origin: *
      │      Access-Control-Allow-Methods: *
      │
      ├── 3. 浏览器检查通过，发送实际请求
      │
      └── 4. 服务器返回响应，携带 CORS 头
```

## 限流配置（Sentinel）

### 网关限流 vs 服务限流

```
网关限流（Gateway）：
  ├── 在请求进入微服务之前就限流
  ├── 保护所有后端服务
  ├── 基于路由、请求参数限流
  └── 适合：全局限流、API 级别限流

服务限流（Sentinel）：
  ├── 在服务内部限流
  ├── 保护单个服务
  ├── 基于方法、资源限流
  └── 适合：服务级别限流、热点参数限流
```

### Sentinel 与 Gateway 集成

```yaml
# pom.xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-alibaba-sentinel-gateway</artifactId>
</dependency>

# application.yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080  # Sentinel Dashboard 地址
```

### 限流规则示例

```yaml
# Sentinel Dashboard 中配置的规则
flows:
  - resource: system-admin-api          # 路由 ID
    grade: 1                            # QPS 模式
    count: 1000                         # 限流阈值
    strategy: 0                         # 直接拒绝
    controlBehavior: 0                  # 快速失败

  - resource: product-admin-api
    grade: 1
    count: 500
    strategy: 0
    controlBehavior: 0
```

## 网关的 application.yaml 结构

### 完整配置结构

```yaml
spring:
  application:
    name: gateway-server              # 服务名

  profiles:
    active: local                     # 激活的环境

  config:
    import:
      - optional:classpath:application-${spring.profiles.active}.yaml
      - optional:nacos:${spring.application.name}-${spring.profiles.active}.yaml

  cloud:
    gateway:
      routes:                         # 路由配置
        - id: system-admin-api
          uri: grayLb://system-server
          predicates:
            - Path=/admin-api/system/**
          filters:
            - RewritePath=/admin-api/system/v3/api-docs, /v3/api-docs
        # ... 更多路由

      x-forwarded:
        prefix-enabled: false         # 禁用前缀转发

      default-filters:                # 全局过滤器
        - DedupeResponseHeader=Access-Control-Allow-Credentials Access-Control-Allow-Origin

server:
  port: 48080                         # 端口

# Knife4j 聚合文档
knife4j:
  gateway:
    enabled: true
    routes:
      - name: system-server
        service-name: system-server
        url: /admin-api/system/v3/api-docs
      # ... 更多文档路由
```

### 配置分层

```
application.yaml                    ← 主配置（路由、端口）
├── application-local.yaml          ← 本地环境（Nacos 地址）
├── application-dev.yaml            ← 开发环境（Nacos 地址）
└── nacos:gateway-server-dev.yaml   ← Nacos 配置（可选，覆盖本地）
```

## 思考题

1. **TokenAuthenticationFilter 的 order 是 -100，为什么选择这个值？如果其他过滤器也需要在 Token 校验之前执行，应该如何设置 order？**

2. **Token 校验结果缓存 1 分钟，如果用户在 1 分钟内被禁用，缓存中的旧数据仍然有效。如何解决这个问题？**

3. **网关的跨域配置允许所有来源（`Access-Control-Allow-Origin: *`），在生产环境中这是否安全？应该如何改进？**

4. **Gateway 的路由配置写在 `application.yaml` 中，如果需要动态修改路由（不停机），有哪些方案？**
