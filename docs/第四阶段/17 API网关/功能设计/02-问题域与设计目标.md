# 02 - 问题域与设计目标

## 为什么需要网关

### 统一入口

没有网关时，前端需要知道每个微服务的地址：

```
前端配置（无网关）：
  用户管理 API：http://system-server:48081
  商品管理 API：http://product-server:48082
  订单管理 API：http://trade-server:48083
  支付管理 API：http://pay-server:48084

问题：
  ├── 服务扩缩容时，前端配置需要同步修改
  ├── 跨域问题：不同服务的域名/端口不同
  ├── 安全问题：直接暴露所有服务的地址
  └── 部署问题：每个服务都需要公网地址
```

有网关后：

```
前端配置（有网关）：
  所有 API：http://gateway:48080

网关路由配置：
  /admin-api/system/**  → system-server
  /admin-api/product/** → product-server
  /admin-api/trade/**   → trade-server
  /admin-api/pay/**     → pay-server
```

### 统一认证

没有网关时，每个微服务都需要独立处理认证：

```
无网关的认证流程：

请求 → system-server → 验证 Token → 业务处理
请求 → product-server → 验证 Token → 业务处理
请求 → trade-server → 验证 Token → 业务处理

问题：
  ├── 认证逻辑在每个服务中重复
  ├── 如果认证逻辑变更，需要修改所有服务
  ├── Token 校验的性能开销分散在每个服务
  └── 每个服务都需要依赖认证相关的代码
```

有网关后：

```
有网关的认证流程：

请求 → Gateway → 验证 Token → 注入用户信息 → 转发到服务 → 业务处理
                    │
                    └── 认证逻辑集中在一个地方
```

### 统一限流

没有网关时，限流分散在各个服务中：

```
无网关的限流：

system-server：限流 1000 QPS
product-server：限流 500 QPS
trade-server：限流 200 QPS

问题：
  ├── 无法全局控制总流量
  ├── 限流规则分散在各个服务中
  └── 无法实现跨服务的流量调度
```

有网关后：

```
有网关的限流：

Gateway：全局限流
  ├── /admin-api/system/**：1000 QPS
  ├── /admin-api/product/**：500 QPS
  ├── /admin-api/trade/**：200 QPS
  └── 全局总限流：5000 QPS
```

## 为什么选 Gateway 而非 Zuul

### 技术对比

| 维度 | Spring Cloud Gateway | Zuul 1.x | Zuul 2.x |
|------|---------------------|----------|----------|
| 编程模型 | 响应式（WebFlux） | 阻塞式（Servlet） | 响应式 |
| 性能 | 高 | 中 | 高 |
| 长连接 | 支持 WebSocket | 不支持 | 支持 |
| 过滤器 | 链式过滤器 | 拦截器 | 过滤器 |
| Spring Cloud | 原生支持 | Netflix 维护 | Netflix 维护 |
| 维护状态 | Spring 社区活跃 | 维护模式 | Netflix 内部使用 |

### 选择理由

```
选择 Spring Cloud Gateway 的理由：

1. Spring Cloud 原生支持
   ├── 与 Nacos 服务发现无缝集成
   ├── 与 LoadBalancer 负载均衡集成
   ├── 与 Sentinel 限流集成
   └── 与 Spring Security 集成

2. 响应式架构
   ├── 更高的并发处理能力
   ├── 更低的资源消耗
   └── 更好的长连接支持

3. 社区活跃
   ├── Spring 社区持续维护
   ├── 文档完善
   └── 问题容易找到解决方案

4. 项目技术栈统一
   ├── yudao-cloud 使用 Spring Cloud Alibaba
   ├── Gateway 是 Spring Cloud 的一等公民
   └── 避免引入额外的框架
```

## 网关 vs Nginx

### 功能对比

| 维度 | API Gateway | Nginx |
|------|-------------|-------|
| 层级 | 应用层（第七层） | 反向代理（第七层） |
| 编程能力 | 强（Java 代码） | 弱（配置 + Lua） |
| 动态路由 | 支持（基于服务发现） | 需要额外模块 |
| 业务逻辑 | 可以编写复杂逻辑 | 不适合 |
| 性能 | 中等 | 高 |
| 适用场景 | 微服务网关 | 反向代理、负载均衡、静态资源 |

### 协作方式

```
生产环境的典型架构：

客户端
  │
  ▼
Nginx（反向代理 + 负载均衡 + 静态资源）
  │
  ├── 静态资源 → 直接返回
  ├── HTTPS 终止
  └── API 请求 → 转发到 Gateway 集群
                      │
                      ▼
               Gateway（业务网关）
                      │
          ┌───────────┼───────────┐
          │           │           │
          ▼           ▼           ▼
    system-server product-server trade-server
```

Nginx 负责：
- HTTPS 终止
- 静态资源服务
- 负载均衡到 Gateway 集群
- 限流（IP 级别）

Gateway 负责：
- 业务路由
- Token 认证
- 业务限流
- 灰度发布

## 设计目标

### 目标一：Token 校验转发

```
设计目标：
  ✓ 统一校验 Token 有效性
  ✓ 将用户信息注入到请求头
  ✓ 支持白名单（无需认证的 URL）
  ✓ Token 校验结果缓存（减少远程调用）
```

### 目标二：路由自动发现

```
设计目标：
  ✓ 基于 Nacos 服务发现的动态路由
  ✓ 无需硬编码服务地址
  ✓ 支持服务实例的自动发现和剔除
  ✓ 支持灰度路由（基于 metadata.version）
```

### 目标三：租户信息透传

```
设计目标：
  ✓ 从请求头提取租户 ID
  ✓ 将租户 ID 转发到下游服务
  ✓ 下游服务通过 TenantContextHolder 获取租户 ID
```

### 目标四：跨域处理

```
设计目标：
  ✓ 统一处理 CORS 跨域
  ✓ 支持 OPTIONS 预检请求
  ✓ 配置允许的域名、方法、头
```

### 目标五：访问日志

```
设计目标：
  ✓ 记录所有请求的详细信息
  ✓ 支持 ELK 日志收集
  ✓ 包含请求/响应体、执行时间、用户信息
```

## 思考题

1. **网关的 Token 校验是每次请求都校验，还是有缓存机制？如果使用缓存，Token 被吊销后多久才能生效？**

2. **网关的路由配置写在 `application.yaml` 中，如果需要动态修改路由规则（不停机），有什么方案？**

3. **如果网关后面有 100 个微服务，每个服务有 10 个实例，网关需要维护多少个连接？如何优化连接管理？**

4. **网关的限流和 Nginx 的限流有什么区别？在什么场景下需要两者配合使用？**
