# 01 - 总览：限流熔断与分布式事务全景

## 场景引入

### 场景一：大促时的系统保护

双十一零点，你的电商系统涌入大量流量：

```
正常流量：1000 QPS
大促流量：100000 QPS  ← 100 倍！

系统容量：
  product-server：5000 QPS
  trade-server：3000 QPS
  pay-server：2000 QPS
```

如果不做任何保护：

```
100000 QPS → product-server（容量 5000）
  │
  ├── 服务响应变慢（从 100ms → 10s）
  ├── 线程池耗尽
  ├── 服务崩溃
  │
  └── 调用方（trade-server）等待超时
      │
      ├── trade-server 线程池耗尽
      ├── trade-server 崩溃
      │
      └── 调用方（gateway）等待超时
          │
          └── 整个系统雪崩！
```

### 场景二：跨服务事务的一致性

用户下单时，需要跨多个服务操作：

```
用户下单流程：
  1. trade-server：创建订单
  2. product-server：扣减库存
  3. member-server：扣减积分
  4. pay-server：创建支付单

问题：
  如果步骤 3 失败了，步骤 1 和 2 的操作如何回滚？
```

```
传统单体事务：
  BEGIN TRANSACTION
    INSERT INTO orders ...
    UPDATE stock SET quantity = quantity - 1 ...
    UPDATE points SET points = points - 100 ...
  COMMIT

  如果 UPDATE points 失败：
    ROLLBACK → 所有操作都撤销

分布式事务：
  trade-server:  INSERT INTO orders ...     ← 已提交
  product-server: UPDATE stock ...          ← 已提交
  member-server: UPDATE points ...          ← 失败！

  如何回滚 trade-server 和 product-server 的操作？
```

## Sentinel：流量控制、熔断降级

### Sentinel 是什么

Sentinel 是阿里巴巴开源的流量控制组件，主要解决以下问题：

```
流量控制（Flow Control）：
  ├── 限制系统的入口流量
  ├── 防止系统被过载请求压垮
  └── 保证系统在安全水位运行

熔断降级（Circuit Breaking）：
  ├── 当依赖的服务不稳定时，自动熔断
  ├── 避免一个服务的故障扩散到整个系统
  └── 提供降级方案，保证核心功能可用

系统负载保护（System Protection）：
  ├── 根据系统的 CPU、内存等指标自动限流
  └── 防止系统在高负载下崩溃
```

### 核心概念

```
资源（Resource）：
  ├── 被保护的对象
  ├── 可以是方法、接口、代码块
  └── 示例：商品查询接口、订单创建方法

规则（Rule）：
  ├── 定义资源的保护策略
  ├── 包括：限流规则、熔断规则、系统规则
  └── 示例：QPS 不超过 1000、慢调用比例超过 50% 时熔断

Sentinel 的工作流程：
  1. 定义资源（@SentinelResource 注解或自动适配）
  2. 配置规则（Dashboard 或代码）
  3. 运行时检查规则，执行保护策略
```

## Seata：分布式事务

### Seata 是什么

Seata 是阿里巴巴开源的分布式事务解决方案，支持多种事务模式：

```
AT 模式（Automatic Transaction）：
  ├── 自动补偿，无需编写回滚代码
  ├── 基于 undo_log 实现
  └── 适合大多数场景

TCC 模式（Try-Confirm-Cancel）：
  ├── 需要手动编写 Try/Confirm/Cancel 方法
  ├── 更灵活，性能更好
  └── 适合对性能要求高的场景

Saga 模式：
  ├── 长事务，每个步骤都有补偿操作
  ├── 适合业务流程长的场景
  └── 适合跨系统事务
```

### 核心概念

```
全局事务（Global Transaction）：
  ├── 由发起方开启的整个分布式事务
  ├── 包含多个分支事务
  └── 由 TC 协调提交或回滚

分支事务（Branch Transaction）：
  ├── 每个参与服务的本地事务
  ├── 向 TC 注册
  └── 根据 TC 指令提交或回滚

事务协调器（Transaction Coordinator, TC）：
  ├── 维护全局事务和分支事务的状态
  ├── 协调提交或回滚
  └── Seata Server 提供

事务管理器（Transaction Manager, TM）：
  ├── 开启、提交、回滚全局事务
  └── 通常是发起方

资源管理器（Resource Manager, RM）：
  ├── 管理分支事务的资源
  ├── 向 TC 注册分支事务
  └── 执行提交或回滚
```

## 两者的关系：保护系统 vs 保证数据一致性

### Sentinel：保护系统可用性

```
Sentinel 的目标：
  ├── 保护系统不被过载流量压垮
  ├── 防止故障扩散（雪崩效应）
  └── 保证核心功能可用

Sentinel 的手段：
  ├── 限流：限制进入系统的流量
  ├── 熔断：当下游服务不稳定时，快速失败
  └── 降级：提供兜底方案

示例：
  商品查询接口限流 1000 QPS
  │
  ├── 正常：直接查询数据库
  └── 超限：返回缓存数据或友好提示
```

### Seata：保证数据一致性

```
Seata 的目标：
  ├── 保证分布式事务的原子性
  ├── 要么全部成功，要么全部回滚
  └── 避免数据不一致

Seata 的手段：
  ├── 全局事务管理
  ├── 分支事务协调
  └── 自动补偿（AT 模式）

示例：
  下单流程（全局事务）
  │
  ├── 创建订单（分支事务 1）
  ├── 扣减库存（分支事务 2）
  ├── 扣减积分（分支事务 3）
  │
  └── 全部成功 → 提交
      任一失败 → 回滚所有
```

### 两者的协作

```
场景：大促下单

1. 请求到达 Gateway
   │
   └── Sentinel 网关限流：检查 QPS 是否超限
       ├── 超限 → 直接拒绝，返回 429
       └── 未超限 → 继续
   │
2. 请求到达 trade-server
   │
   ├── Sentinel 服务限流：检查服务负载
   │   ├── 超限 → 返回降级响应
   │   └── 未超限 → 继续
   │
   ├── Seata 开启全局事务
   │
   ├── 调用 product-server（Sentinel 熔断检查）
   │   ├── 服务不可用 → 熔断，返回降级响应
   │   └── 服务可用 → 扣减库存
   │
   ├── 调用 member-server（Sentinel 熔断检查）
   │   ├── 服务不可用 → 熔断，回滚库存
   │   └── 服务可用 → 扣减积分
   │
   └── 全部成功 → Seata 提交全局事务
       任一失败 → Seata 回滚全局事务
```

## 思考题

1. **Sentinel 的限流和 Seata 的事务回滚，在大促场景下如何配合？如果限流导致部分请求被拒绝，这些请求中已开启的全局事务如何处理？**

2. **Sentinel 的熔断机制是如何判断下游服务"不稳定"的？它和健康检查（Health Check）有什么区别？**

3. **Seata 的 AT 模式需要在每个数据库中创建 undo_log 表。如果某个服务使用的是 NoSQL 数据库（如 MongoDB），AT 模式还能用吗？**

4. **在微服务架构中，如果 Sentinel 和 Seata 同时使用，它们的执行顺序是什么？限流检查在事务开启之前还是之后？**
