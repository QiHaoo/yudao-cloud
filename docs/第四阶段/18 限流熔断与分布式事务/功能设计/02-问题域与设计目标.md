# 02 - 问题域与设计目标

## 微服务的雪崩效应

### 什么是雪崩效应

雪崩效应是指一个服务的故障导致整条调用链路崩溃的现象：

```
正常情况：

Gateway → trade-server → product-server → 数据库
   │           │               │
  10ms       50ms            20ms          总耗时：80ms


异常情况（product-server 响应变慢）：

Gateway → trade-server → product-server → 数据库
   │           │               │
  10ms      5000ms          5000ms         总耗时：10010ms
   │
   └── Gateway 线程被阻塞，无法处理新请求
       │
       └── Gateway 线程池耗尽，所有请求超时
           │
           └── 整个系统不可用！
```

### 雪崩的传播过程

```
阶段一：单个服务变慢
  product-server 响应时间：20ms → 5000ms
  原因：数据库慢查询、资源竞争

阶段二：调用方线程阻塞
  trade-server 调用 product-server，线程阻塞等待
  trade-server 线程池：10/200 → 200/200（耗尽）

阶段三：调用方不可用
  trade-server 无法处理新请求
  所有依赖 trade-server 的服务受影响

阶段四：故障扩散
  Gateway 调用 trade-server，线程阻塞
  Gateway 线程池耗尽
  整个系统不可用

时间线：
  0s: product-server 变慢
  5s: trade-server 线程池开始紧张
  10s: trade-server 线程池耗尽
  15s: Gateway 线程池开始紧张
  20s: 整个系统不可用
```

### 雪崩的根因

```
根本原因：调用链路中的某个环节出现故障，导致上游服务被阻塞

关键因素：
  1. 没有超时控制
     └── 调用方无限等待响应

  2. 没有熔断机制
     └── 持续调用已经故障的服务

  3. 没有限流保护
     └── 大量请求涌入，压垮服务

  4. 没有降级方案
     └── 无法提供兜底响应
```

## 分布式事务的挑战

### CAP 理论

分布式系统无法同时满足以下三个特性：

```
C（Consistency）一致性：
  所有节点在同一时间看到相同的数据

A（Availability）可用性：
  每个请求都能得到响应（不保证是最新数据）

P（Partition Tolerance）分区容错：
  网络分区时，系统仍能继续运行

CAP 定理：
  分布式系统最多只能同时满足其中两个
```

```
CP 系统（一致性 + 分区容错）：
  ├── 放弃可用性
  ├── 网络分区时，拒绝写入
  └── 示例：ZooKeeper、Etcd

AP 系统（可用性 + 分区容错）：
  ├── 放弃一致性
  ├── 网络分区时，允许不一致
  └── 示例：Eureka、Cassandra

CA 系统（一致性 + 可用性）：
  ├── 放弃分区容错
  ├── 不是真正的分布式系统
  └── 示例：单机数据库
```

### 分布式事务的难题

```
单体事务（ACID）：
  A - Atomicity（原子性）：要么全部成功，要么全部失败
  C - Consistency（一致性）：事务前后数据一致
  I - Isolation（隔离性）：并发事务互不影响
  D - Durability（持久性）：提交后数据永久保存

分布式事务的挑战：
  ├── 原子性：跨多个服务，如何保证原子？
  ├── 一致性：网络延迟、服务故障，如何保证一致？
  ├── 隔离性：不同服务的事务如何隔离？
  └── 持久性：部分提交、部分回滚，如何处理？
```

### 最终一致性 vs 强一致性

```
强一致性：
  ├── 事务完成后，所有节点数据一致
  ├── 实现复杂，性能差
  └── 适合：金融转账、库存扣减

最终一致性：
  ├── 允许短暂的数据不一致
  ├── 最终所有节点数据一致
  ├── 实现简单，性能好
  └── 适合：消息推送、数据同步

示例：
  强一致性：下单后，库存立即扣减，用户立即看到最新库存
  最终一致性：下单后，库存可能延迟几秒才扣减，用户可能看到旧库存
```

## 为什么选 Sentinel 而非 Hystrix

### 技术对比

| 维度 | Sentinel | Hystrix |
|------|----------|---------|
| 开发公司 | 阿里巴巴 | Netflix |
| 维护状态 | 活跃 | 维护模式 |
| 熔断策略 | 慢调用比例、异常比例、异常数 | 异常比例 |
| 限流能力 | QPS、线程数、热点参数 | 信号量、线程池 |
| 流控效果 | 快速失败、Warm Up、排队等待 | 快速失败 |
| 规则配置 | 动态推送（Dashboard） | 配置文件 |
| 可视化 | 自带 Dashboard | 需要额外集成 |
| 扩展性 | 丰富扩展点 | 有限 |
| Spring Cloud | 原生支持 | Netflix 维护 |

### 选择理由

```
选择 Sentinel 的理由：

1. 功能更丰富
   ├── 支持更多熔断策略（慢调用、异常比例、异常数）
   ├── 支持热点参数限流
   ├── 支持集群限流
   └── 支持系统自适应保护

2. 动态规则
   ├── 通过 Dashboard 动态修改规则
   ├── 无需重启服务
   └── 支持多种数据源（Nacos、ZooKeeper）

3. 可视化
   ├── 自带 Dashboard
   ├── 实时监控
   └── 规则管理

4. 社区活跃
   ├── 阿里巴巴持续维护
   ├── 文档完善
   └── 问题容易找到解决方案
```

## 为什么选 Seata 而非 TCC/Saga

### 分布式事务方案对比

| 方案 | 实现方式 | 侵入性 | 性能 | 适用场景 |
|------|----------|--------|------|----------|
| 2PC（两阶段提交） | 数据库协议 | 低 | 低 | 传统数据库 |
| TCC（Try-Confirm-Cancel） | 业务代码 | 高 | 高 | 高性能场景 |
| Saga | 补偿操作 | 中 | 中 | 长事务 |
| Seata AT | 自动生成补偿 | 低 | 中 | 大多数场景 |
| 消息最终一致性 | 消息队列 | 中 | 高 | 异步场景 |

### Seata AT 模式的优势

```
AT 模式的核心思想：
  ├── 拦截 SQL，自动生成 undo_log
  ├── 提交时删除 undo_log
  └── 回滚时根据 undo_log 反向补偿

优势：
  ├── 业务代码零侵入（只需加 @GlobalTransactional）
  ├── 自动生成补偿逻辑
  └── 开发效率高

劣势：
  ├── 性能比 TCC 差（需要写 undo_log）
  ├── 需要在每个数据库创建 undo_log 表
  └── 不支持 NoSQL
```

### 选择理由

```
选择 Seata AT 模式的理由：

1. 低侵入性
   ├── 只需加 @GlobalTransactional 注解
   ├── 无需编写补偿代码
   └── 对业务代码影响小

2. 开发效率
   ├── 自动处理回滚逻辑
   ├── 减少开发工作量
   └── 降低出错概率

3. 生态集成
   ├── 与 Spring Cloud 无缝集成
   ├── 支持多种数据库
   └── 社区活跃

4. 项目适用性
   ├── yudao-cloud 是中小型项目
   ├── 对性能要求不是极致
   └── 开发效率优先
```

## 设计目标

### 目标一：自动熔断

```
设计目标：
  ✓ 自动检测下游服务的健康状态
  ✓ 当服务不稳定时，自动熔断
  ✓ 熔断后，提供降级响应
  ✓ 支持半开状态，自动恢复
```

### 目标二：声明式事务

```
设计目标：
  ✓ 使用 @GlobalTransactional 声明全局事务
  ✓ 自动处理分支事务的提交和回滚
  ✓ 支持多种事务模式（AT、TCC、Saga）
  ✓ 与 Spring Boot 自动集成
```

### 目标三：限流保护

```
设计目标：
  ✓ 支持 QPS 和线程数限流
  ✓ 支持热点参数限流
  ✓ 支持集群限流
  ✓ 支持动态规则配置
```

### 目标四：降级方案

```
设计目标：
  ✓ 支持 fallback 降级方法
  ✓ 支持 fallbackFactory 降级工厂
  ✓ 支持返回默认值、缓存数据
  ✓ 保证核心功能可用
```
