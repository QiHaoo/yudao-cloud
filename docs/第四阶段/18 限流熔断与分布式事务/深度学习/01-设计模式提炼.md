# 01 - 设计模式提炼：限流熔断与分布式事务

本文从限流熔断与分布式事务组件中提炼可复用的设计模式，帮助读者理解 Sentinel 和 Seata 背后的设计思想，以及它们在微服务架构中的核心价值。

## 断路器模式：故障隔离的核心机制

### 模式识别

Sentinel 的熔断降级是**断路器模式（Circuit Breaker Pattern）**的经典实现。断路器的核心思想是：当依赖的服务不稳定时，自动"断开"连接，避免故障扩散。

```
断路器的三种状态：

  CLOSED（关闭状态）：
    ├── 正常放行所有请求
    ├── 统计失败率/慢调用率
    └── 当失败率超过阈值 → 转为 OPEN

  OPEN（打开状态）：
    ├── 拒绝所有请求
    ├── 直接返回降级响应
    └── 经过一段时间后 → 转为 HALF-OPEN

  HALF-OPEN（半开状态）：
    ├── 放行部分探测请求
    ├── 如果探测成功 → 转为 CLOSED
    └── 如果探测失败 → 转为 OPEN
```

### 断路器状态机

```
         ┌─────────────────────────────────────────┐
         │                                         │
         ▼                                         │
┌───────────────┐    达到熔断条件    ┌───────────────┐
│   CLOSED      │ ───────────────→ │    OPEN       │
│  （关闭状态）  │                  │  （打开状态）  │
│  正常放行      │                  │  拒绝所有请求  │
└───────────────┘                  └───────┬───────┘
         ▲                                 │
         │                                 │ 熔断时长结束
         │                                 ▼
         │                         ┌───────────────┐
         │  探测成功                │  HALF-OPEN    │
         └──────────────────────── │ （半开状态）   │
                                   │  放行部分请求  │
                                   └───────────────┘
                                         │
                                         │ 探测失败
                                         ▼
                                   回到 OPEN 状态
```

### Sentinel 的三种熔断策略

| 策略 | 触发条件 | 适用场景 |
|------|---------|---------|
| 慢调用比例 | 响应时间超过阈值的请求比例 | 服务响应变慢，但未完全不可用 |
| 异常比例 | 异常请求占总请求的比例 | 服务间歇性报错 |
| 异常数 | 异常请求的绝对数量 | 服务偶尔报错，需要容忍少量异常 |

```java
// 慢调用比例熔断规则
DegradeRule rule = new DegradeRule("getProduct")
    .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
    .setCount(0.5)           // 慢调用比例阈值：50%
    .setSlowRatioThreshold(1000)  // 慢调用阈值：1000ms
    .setTimeWindow(10)       // 熔断时长：10 秒
    .setMinRequestAmount(5)  // 最小请求数：5
    .setStatIntervalMs(10000);  // 统计时长：10 秒
```

### 与健康检查的区别

| 维度 | 断路器（Sentinel） | 健康检查（Health Check） |
|------|-------------------|------------------------|
| 检测粒度 | 单个资源/方法 | 整个服务实例 |
| 触发条件 | 慢调用比例、异常比例 | 服务存活、端口可达 |
| 响应速度 | 实时（每次请求统计） | 定期轮询（秒级延迟） |
| 降级策略 | 返回 Fallback 响应 | 从注册中心摘除实例 |
| 恢复机制 | 半开状态探测 | 重新注册实例 |

两者配合使用：健康检查负责服务实例级别的故障检测，断路器负责资源级别的故障隔离。

### 通用化提炼

断路器模式适合以下场景：

| 特征 | Sentinel 中的体现 |
|------|-----------------|
| 调用远程服务 | trade-server 调用 product-server |
| 远程服务可能不稳定 | 网络抖动、服务过载 |
| 故障可能扩散 | 一个服务慢导致调用方线程池耗尽 |
| 需要快速失败 | 避免无效等待，快速返回降级响应 |

### 电商场景举例：商品服务熔断

```
场景：大促期间商品服务过载

  正常状态（CLOSED）：
    trade-server → product-server → 返回商品信息（100ms）

  商品服务过载：
    trade-server → product-server → 响应变慢（5000ms）
    │
    ├── 慢调用比例：80% > 50%（阈值）
    ├── 触发熔断 → OPEN 状态
    │
    └── 后续请求直接返回降级响应
        "商品服务暂时不可用，请稍后重试"

  10 秒后（HALF-OPEN）：
    trade-server → product-server → 探测请求
    │
    ├── 探测成功 → CLOSED（恢复正常）
    └── 探测失败 → OPEN（继续熔断）
```

## 令牌桶与漏桶模式：流量整形

### 模式识别

Sentinel 的限流机制体现了**令牌桶模式（Token Bucket）**和**漏桶模式（Leaky Bucket）**的设计思想。

```
令牌桶模式：
  ├── 以固定速率向桶中添加令牌
  ├── 请求到来时，从桶中取一个令牌
  ├── 桶满时，新令牌被丢弃
  └── 桶空时，请求被拒绝

  特点：允许突发流量（桶中有积累的令牌）

漏桶模式：
  ├── 请求进入桶中排队
  ├── 以固定速率从桶中漏出请求
  ├── 桶满时，新请求被丢弃
  └── 输出速率恒定

  特点：严格限制输出速率，平滑流量
```

### Sentinel 的三种流控效果

| 流控效果 | 对应模式 | 特点 | 适用场景 |
|---------|---------|------|---------|
| 快速失败 | 令牌桶（无积累） | 超过阈值直接拒绝 | 大多数场景 |
| Warm Up | 令牌桶（预热） | 从低阈值逐步增加到阈值 | 秒杀活动开始时 |
| 排队等待 | 漏桶 | 超过阈值的请求排队，匀速通过 | 对响应时间不敏感的场景 |

### Warm Up 的预热过程

```
Warm Up 预热过程：

  阈值：1000 QPS
  预热时长：10 秒

  时间轴：
    0s   ──── 100 QPS（冷启动，阈值较低）
    2s   ──── 200 QPS
    5s   ──── 500 QPS
    10s  ──── 1000 QPS（达到正常阈值）

  适用场景：
    ├── 秒杀活动开始时，避免冷启动的流量冲击
    ├── 服务重启后，避免大量请求同时涌入
    └── 数据库连接池预热
```

### 排队等待的匀速效果

```
排队等待效果：

  阈值：100 QPS
  超时时间：500ms

  请求到达：
    请求 1 ──→ 立即处理
    请求 2 ──→ 排队等待 10ms → 处理
    请求 3 ──→ 排队等待 20ms → 处理
    ...
    请求 N ──→ 排队等待超时 → 拒绝

  效果：输出速率恒定 100 QPS，平滑流量
```

### 通用化提炼

令牌桶/漏桶模式适合以下场景：

| 特征 | Sentinel 中的体现 |
|------|-----------------|
| 需要限制请求速率 | QPS 限流 |
| 需要处理突发流量 | Warm Up 预热 |
| 需要平滑流量 | 排队等待 |
| 需要保护后端服务 | 防止服务过载 |

### 电商场景举例：秒杀活动限流

```
场景：双十一零点秒杀

  秒杀开始前：
    系统处于冷启动状态，连接池未预热

  使用 Warm Up：
    0s   ──── 允许 100 QPS（预热开始）
    5s   ──── 允许 500 QPS
    10s  ──── 允许 1000 QPS（达到正常容量）

  效果：
    ├── 避免冷启动时的流量冲击
    ├── 让系统逐步预热（连接池、缓存等）
    └── 10 秒后系统完全就绪，正常处理请求
```

## Saga 模式：长事务的最终一致性

### 模式识别

Seata 的 Saga 模式体现了**Saga 模式（Saga Pattern）**的设计思想：将长事务拆分为多个本地事务，每个本地事务都有对应的补偿操作。

```
Saga 模式的核心思想：

  长事务拆分：
    创建订单 → 扣减库存 → 扣减积分 → 创建支付单

  每个步骤的补偿操作：
    创建订单 ←→ 取消订单
    扣减库存 ←→ 恢复库存
    扣减积分 ←→ 恢复积分
    创建支付单 ←→ 取消支付单

  失败时的补偿流程：
    步骤 3 失败 → 补偿步骤 2（恢复库存） → 补偿步骤 1（取消订单）
```

### Saga 模式 vs 两阶段提交

| 维度 | Saga 模式 | 两阶段提交（2PC） |
|------|----------|------------------|
| 一致性 | 最终一致性 | 强一致性 |
| 性能 | 高（无全局锁） | 低（需要全局锁） |
| 适用场景 | 长事务、跨系统 | 短事务、同构数据库 |
| 补偿方式 | 每个步骤有补偿操作 | 协调器统一回滚 |
| 复杂度 | 高（需要实现补偿逻辑） | 低（框架自动处理） |

### Seata Saga 模式的执行流程

```
正向流程：

  步骤 1：创建订单（trade-server）
      │
      ▼
  步骤 2：扣减库存（product-server）
      │
      ▼
  步骤 3：扣减积分（member-server）
      │
      ▼
  步骤 4：创建支付单（pay-server）
      │
      ▼
  全部成功 → 事务完成

补偿流程（步骤 3 失败）：

  步骤 3 失败
      │
      ▼
  补偿步骤 2：恢复库存（product-server）
      │
      ▼
  补偿步骤 1：取消订单（trade-server）
      │
      ▼
  补偿完成（最终一致性）
```

### 通用化提炼

Saga 模式适合以下场景：

| 特征 | Seata 中的体现 |
|------|---------------|
| 事务涉及多个服务 | trade-server、product-server、member-server |
| 事务执行时间长 | 包含远程调用、人工审批等 |
| 不要求强一致性 | 最终一致性即可 |
| 每个步骤可补偿 | 库存可恢复、积分可退回 |

### 电商场景举例：订单创建的 Saga 流程

```java
// Saga 模式的状态机定义
public class OrderSagaDefinition {

    public SagaDefinition<OrderContext> build() {
        return SagaDefinitionBuilder.<OrderContext>builder()
            // 步骤 1：创建订单
            .step("createOrder")
                .compensation(this::cancelOrder)
                .build()
            // 步骤 2：扣减库存
            .step("deductStock")
                .compensation(this::restoreStock)
                .build()
            // 步骤 3：扣减积分
            .step("deductPoints")
                .compensation(this::restorePoints)
                .build()
            .build();
    }

    // 补偿操作
    private void cancelOrder(OrderContext context) {
        tradeOrderMapper.deleteById(context.getOrderId());
    }

    private void restoreStock(OrderContext context) {
        productSpuApi.restoreStock(context.getSpuId(), context.getQuantity());
    }

    private void restorePoints(OrderContext context) {
        memberPointApi.restorePoints(context.getUserId(), context.getPoints());
    }
}
```

## 补偿事务模式：自动回滚的实现

### 模式识别

Seata AT 模式的 undo_log 机制体现了**补偿事务模式（Compensating Transaction Pattern）**的设计思想：通过记录操作前的状态，在需要回滚时执行反向操作。

```
补偿事务的核心思想：

  正向操作：
    UPDATE stock SET quantity = 9 WHERE product_id = 1

  记录补偿信息（undo_log）：
    before image：quantity = 10
    after image：quantity = 9
    rollback SQL：UPDATE stock SET quantity = 10 WHERE product_id = 1

  回滚时：
    执行 rollback SQL → quantity 恢复为 10
    删除 undo_log
```

### AT 模式的 undo_log 机制

```
AT 模式的工作流程：

  1. 拦截业务 SQL
     ├── 解析 SQL，识别表和字段
     ├── 查询 before image（修改前的数据）
     └── 生成 undo_log 记录

  2. 执行业务 SQL
     └── 正常执行 SQL

  3. 生成 after image
     └── 查询修改后的数据

  4. 提交本地事务
     ├── 业务 SQL + undo_log 一起提交
     └── 向 TC 注册分支事务

  5. 全局提交
     └── 删除 undo_log

  6. 全局回滚
     ├── 读取 undo_log
     ├── 执行反向 SQL（INSERT→DELETE, DELETE→INSERT, UPDATE→还原）
     └── 删除 undo_log
```

### TCC 模式：手动补偿的三种操作

TCC 模式需要手动编写三个方法，更灵活但侵入性更高：

```java
// TCC 模式的三个阶段
public interface StockService {

    // Try：预留资源（冻结库存）
    @TwoPhaseBusinessAction(name = "deductStock",
        commitMethod = "confirm", rollbackMethod = "cancel")
    boolean tryDeductStock(
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "quantity") Integer quantity);

    // Confirm：确认提交（真正扣减库存）
    boolean confirm(BusinessActionContext context);

    // Cancel：取消回滚（释放冻结库存）
    boolean cancel(BusinessActionContext context);
}
```

### AT 模式 vs TCC 模式

| 维度 | AT 模式 | TCC 模式 |
|------|---------|---------|
| 实现方式 | 自动补偿（undo_log） | 手动编写 Try/Confirm/Cancel |
| 侵入性 | 低（只需加注解） | 高（需要改造业务代码） |
| 性能 | 中（需要记录 undo_log） | 高（无额外存储开销） |
| 适用场景 | 大多数场景 | 高性能场景、非 SQL 场景 |
| 数据隔离 | 默认读未提交 | 业务层面保证 |

### 通用化提炼

补偿事务模式适合以下场景：

| 特征 | Seata 中的体现 |
|------|---------------|
| 需要保证原子性 | 要么全部成功，要么全部回滚 |
| 操作可逆 | INSERT 可 DELETE，UPDATE 可还原 |
| 分布式环境 | 跨多个数据库/服务 |
| 自动化要求高 | AT 模式自动记录补偿信息 |

### 电商场景举例：库存扣减的补偿

```
场景：下单扣减库存，支付失败需要回滚

  正向流程：
    1. 查询库存：quantity = 10
    2. 执行 SQL：UPDATE stock SET quantity = 9 WHERE product_id = 1
    3. 记录 undo_log：
       - before image：quantity = 10
       - after image：quantity = 9
       - rollback SQL：UPDATE stock SET quantity = 10 WHERE product_id = 1
    4. 提交本地事务（业务 SQL + undo_log）

  支付失败，全局回滚：
    1. 读取 undo_log
    2. 执行 rollback SQL：UPDATE stock SET quantity = 10 WHERE product_id = 1
    3. 删除 undo_log
    4. 库存恢复为 10
```

## 注解驱动模式：声明式事务管理

### 模式识别

Seata 的 `@GlobalTransactional` 注解体现了**注解驱动模式（Annotation-Driven Pattern）**的设计思想：通过注解声明事务边界，框架自动处理事务的开启、提交、回滚。

```java
// 声明式事务：一个注解搞定
@GlobalTransactional
public Long createOrder(TradeOrderCreateReqVO reqVO) {
    // 业务逻辑
    // 框架自动处理：开启事务 → 执行业务 → 提交/回滚
    return orderId;
}
```

### 与 Spring @Transactional 的对比

| 维度 | @Transactional | @GlobalTransactional |
|------|---------------|---------------------|
| 作用范围 | 单个数据库 | 跨多个数据库/服务 |
| 事务协调 | 数据库自身 | Seata TC（事务协调器） |
| 回滚机制 | 数据库 ROLLBACK | undo_log 反向补偿 |
| XID 传播 | 不需要 | 需要通过请求头传播 |
| 性能开销 | 低 | 中（需要与 TC 通信） |

### 注解驱动的设计优势

```
命令式编程：
  // 手动管理事务
  TransactionStatus tx = transactionManager.getTransaction(def);
  try {
      // 业务逻辑
      tradeOrderMapper.insert(order);
      productSpuApi.deductStock(spuId, 1);
      transactionManager.commit(tx);
  } catch (Exception e) {
      transactionManager.rollback(tx);
      throw e;
  }

声明式编程：
  // 注解驱动，框架自动处理
  @GlobalTransactional
  public Long createOrder(TradeOrderCreateReqVO reqVO) {
      tradeOrderMapper.insert(order);
      productSpuApi.deductStock(spuId, 1);
      return order.getId();
  }
```

### 通用化提炼

注解驱动模式适合以下场景：

| 特征 | Seata 中的体现 |
|------|---------------|
| 操作有固定的模式 | 开启事务 → 执行业务 → 提交/回滚 |
| 实现细节可以标准化 | XID 传播、分支注册、undo_log 管理 |
| 开发者关注业务而非技术 | 只需声明"这个方法需要全局事务" |
| 框架可以做优化 | 连接池复用、异步提交等 |

## 降级模式：服务不可用时的兜底方案

### 模式识别

Sentinel 的 FallbackFactory 体现了**降级模式（Fallback/Degradation Pattern）**的设计思想：当服务不可用时，返回一个兜底响应，而不是让整个请求失败。

```
降级模式的三种策略：

  1. 返回默认值
     product-server 不可用 → 返回默认商品信息

  2. 返回缓存数据
     product-server 不可用 → 返回缓存中的商品信息

  3. 返回友好提示
     product-server 不可用 → 返回"商品服务暂时不可用"
```

### FallbackFactory 的典型实现

```java
@Slf4j
@Component
public class ProductSpuApiFallbackFactory implements FallbackFactory<ProductSpuApi> {

    @Override
    public ProductSpuApi create(Throwable cause) {
        return new ProductSpuApi() {
            @Override
            public CommonResult<ProductSpuRespVO> getSpu(Long id) {
                // 1. 记录日志（包含异常信息）
                log.error("[getSpu][商品服务调用失败，商品ID：{}]", id, cause);

                // 2. 返回降级响应
                return CommonResult.error(SERVICE_UNAVAILABLE.getCode(),
                        "商品服务暂时不可用，请稍后重试");

                // 或者返回默认值
                // return CommonResult.success(getDefaultProduct(id));

                // 或者返回缓存数据
                // return CommonResult.success(getFromCache(id));
            }
        };
    }
}
```

### fallback vs fallbackFactory

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| `fallback` | 简单 | 无法获取异常信息 | 简单降级 |
| `fallbackFactory` | 可以获取异常信息 | 稍微复杂 | 生产环境（需要日志记录） |

生产环境推荐使用 `fallbackFactory`，因为它可以记录异常信息，便于排查问题。

### 通用化提炼

降级模式适合以下场景：

| 特征 | Sentinel 中的体现 |
|------|-----------------|
| 调用远程服务 | trade-server 调用 product-server |
| 远程服务可能不可用 | 网络故障、服务崩溃 |
| 需要保证核心功能 | 即使商品服务不可用，订单服务也要可用 |
| 需要友好的用户体验 | 返回友好提示而非系统错误 |

### 电商场景举例：商品服务降级

```
场景：商品服务不可用，但订单服务需要继续工作

  未做降级：
    trade-server → product-server（不可用）
        │
        └── 抛出异常，订单创建失败
            └── 用户看到"系统错误"

  做了降级：
    trade-server → Sentinel 检测到异常
        │
        ├── 触发熔断
        ├── 调用 FallbackFactory
        │   └── 返回默认商品信息
        └── 订单创建成功（使用默认商品信息）
            └── 用户看到"商品信息可能不是最新的"
```

## 设计模式的组合使用

限流熔断与分布式事务组件的设计模式是相互配合的：

```
┌─────────────────────────────────────────────────────────────┐
│                    设计模式组合全景                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  断路器模式（Sentinel）                                      │
│  ├── CLOSED → OPEN → HALF-OPEN 状态机                       │
│  ├── 慢调用比例 / 异常比例 / 异常数                           │
│  └── 故障隔离，防止雪崩                                      │
│                                                             │
│  令牌桶/漏桶模式（Sentinel）                                 │
│  ├── 快速失败：超过阈值直接拒绝                              │
│  ├── Warm Up：预热，逐步增加阈值                             │
│  └── 排队等待：匀速通过                                      │
│                                                             │
│  降级模式（Sentinel + OpenFeign）                            │
│  ├── FallbackFactory：返回兜底响应                           │
│  ├── 返回默认值 / 缓存数据 / 友好提示                        │
│  └── 保证核心功能可用                                        │
│                                                             │
│  Saga 模式（Seata）                                         │
│  ├── 长事务拆分为多个本地事务                                │
│  ├── 每个步骤有对应的补偿操作                                │
│  └── 最终一致性                                              │
│                                                             │
│  补偿事务模式（Seata AT）                                    │
│  ├── undo_log 记录操作前的状态                               │
│  ├── 回滚时执行反向 SQL                                     │
│  └── 自动补偿，无需编写回滚代码                              │
│                                                             │
│  注解驱动模式（Seata）                                       │
│  ├── @GlobalTransactional 声明事务边界                       │
│  ├── 框架自动处理开启/提交/回滚                              │
│  └── 声明式编程，开发者关注业务                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 反模式警示

### 反模式一：在网关中开启全局事务

```java
// 反模式：在 Gateway 的过滤器中使用 @GlobalTransactional
@Component
public class OrderFilter implements GlobalFilter {
    @Override
    @GlobalTransactional  // 网关不应该管理事务！
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // ...
    }
}
```

**问题**：网关是轻量级的转发层，不应该包含事务管理逻辑。事务管理应该在业务服务中。

**正确做法**：在 trade-server 的 Service 层使用 `@GlobalTransactional`。

### 反模式二：熔断后不做任何处理

```java
// 反模式：熔断后直接抛异常，不做降级
@FeignClient(name = "product-server")  // 没有 fallbackFactory
public interface ProductSpuApi {
    // ...
}
```

**问题**：熔断后直接抛异常，调用方需要处理异常，用户体验差。

**正确做法**：定义 FallbackFactory，返回友好的降级响应。

### 反模式三：白名单过于宽泛

```java
// 反模式：在 @GlobalTransactional 中包含耗时操作
@GlobalTransactional
public void createOrder(TradeOrderCreateReqVO reqVO) {
    // 1. 创建订单
    createOrderRecord(reqVO);

    // 2. 发送短信通知（耗时操作，不应该在事务中）
    sendSmsNotification(reqVO.getUserId());

    // 3. 写入日志（耗时操作，不应该在事务中）
    writeAuditLog(reqVO);
}
```

**问题**：事务持有时间过长，占用数据库连接，影响并发性能。

**正确做法**：将非关键操作移到事务外，只在事务中包含必须保证原子性的操作。

### 反模式四：忽略幂等性设计

```java
// 反模式：分支事务不支持幂等
@Override
public void deductStock(Long spuId, Integer quantity) {
    // 直接扣减，不检查是否已经扣减过
    stockMapper.update("UPDATE stock SET quantity = quantity - #{quantity} WHERE spu_id = #{spuId}",
        spuId, quantity);
}
```

**问题**：分支事务可能被重试，如果不支持幂等，会导致重复扣减。

**正确做法**：使用乐观锁或唯一键保证幂等性。

## 总结

限流熔断与分布式事务组件的设计模式提炼：

| 模式 | 解决的问题 | 核心思想 |
|------|-----------|---------|
| 断路器模式 | 故障可能扩散到整个系统 | 自动检测故障，快速失败，防止雪崩 |
| 令牌桶/漏桶模式 | 系统被过载流量压垮 | 限制请求速率，保护后端服务 |
| 降级模式 | 服务不可用时的用户体验 | 返回兜底响应，保证核心功能可用 |
| Saga 模式 | 跨服务的长事务一致性 | 拆分事务，每个步骤有补偿操作 |
| 补偿事务模式 | 分布式环境下的事务回滚 | 记录操作前的状态，回滚时执行反向操作 |
| 注解驱动模式 | 事务管理的复杂性 | 声明式编程，框架自动处理 |

这些模式不仅适用于 Sentinel 和 Seata，也适用于任何微服务架构中的流量控制和事务管理。当你在搭建自己的微服务系统时，可以参考这些模式来设计限流策略、熔断机制、降级方案、分布式事务等核心组件。
