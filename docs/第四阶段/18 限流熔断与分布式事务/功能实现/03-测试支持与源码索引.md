### 1.2 单元测试：Sentinel 资源

```java
// 测试 @SentinelResource 注解的方法
@SpringBootTest
class SentinelResourceTest {

    @Resource
    private TradeOrderService tradeOrderService;

    @Test
    @Disabled("需要 Sentinel Dashboard 配合")
    void testSentinelFlowControl() {
        // Given：配置流控规则（QPS = 10）
        configureFlowRule("testResource", 10);

        // When：发送超过阈值的请求
        int blockedCount = 0;
        for (int i = 0; i < 20; i++) {
            try {
                tradeOrderService.createOrder(buildReqVO());
            } catch (BlockException e) {
                blockedCount++;
            }
        }

        // Then：应该有请求被限流
        assertTrue(blockedCount > 0);
    }

    @Test
    @Disabled("需要 Sentinel Dashboard 配合")
    void testSentinelDegrade() {
        // Given：配置降级规则（慢调用比例 > 50%）
        configureDegradeRule("testResource", 0.5, 1000);

        // When：模拟慢调用
        for (int i = 0; i < 10; i++) {
            try {
                tradeOrderService.slowOperation();  // 耗时 > 1000ms
            } catch (Exception e) {
                // 忽略
            }
        }

        // Then：后续请求应该被熔断
        assertThrows(DegradeException.class, () -> {
            tradeOrderService.slowOperation();
        });
    }

}
```

### 2.2 单元测试：全局事务

```java
@SpringBootTest
class GlobalTransactionalTest {

    @Resource
    private TradeOrderService tradeOrderService;

    @Resource
    private TradeOrderMapper tradeOrderMapper;

    @Test
    @Disabled("需要 Seata Server 配合")
    void testGlobalTransactional_Success() {
        // Given：正常下单
        TradeOrderCreateReqVO reqVO = buildReqVO();

        // When：创建订单
        Long orderId = tradeOrderService.createOrder(reqVO);

        // Then：订单应该创建成功
        assertNotNull(orderId);
        TradeOrderDO order = tradeOrderMapper.selectById(orderId);
        assertNotNull(order);
    }

    @Test
    @Disabled("需要 Seata Server 配合")
    void testGlobalTransactional_Rollback() {
        // Given：模拟扣减积分失败（积分不足）
        TradeOrderCreateReqVO reqVO = buildReqVO();
        reqVO.setUserId(USER_WITH_NO_POINTS);

        // When：创建订单应该失败
        assertThrows(ServiceException.class, () -> {
            tradeOrderService.createOrder(reqVO);
        });

        // Then：订单应该被回滚
        TradeOrderDO order = tradeOrderMapper.selectByUserIdAndSpuId(
                USER_WITH_NO_POINTS, reqVO.getSpuId());
        assertNull(order);
    }

}
```

### 2.3 集成测试：跨服务事务

```java
@SpringBootTest
class CrossServiceTransactionTest {

    @Resource
    private TradeOrderService tradeOrderService;

    @Resource
    private TradeOrderMapper tradeOrderMapper;

    @Resource
    private ProductStockMapper productStockMapper;

    @Test
    @Disabled("需要完整微服务环境")
    void testCrossServiceTransaction_AllSuccess() {
        // Given：正常场景
        Long spuId = 1L;
        Integer originalStock = getStock(spuId);
        TradeOrderCreateReqVO reqVO = buildReqVO(spuId, 1);

        // When：创建订单
        Long orderId = tradeOrderService.createOrder(reqVO);

        // Then：订单创建成功，库存扣减
        assertNotNull(orderId);
        assertEquals(originalStock - 1, getStock(spuId));
    }

    @Test
    @Disabled("需要完整微服务环境")
    void testCrossServiceTransaction_Rollback() {
        // Given：模拟某个分支事务失败
        Long spuId = 1L;
        Integer originalStock = getStock(spuId);
        TradeOrderCreateReqVO reqVO = buildReqVO(spuId, 9999);  // 库存不足

        // When：创建订单应该失败
        assertThrows(ServiceException.class, () -> {
            tradeOrderService.createOrder(reqVO);
        });

        // Then：库存应该回滚到原始值
        assertEquals(originalStock, getStock(spuId));
    }

    private Integer getStock(Long spuId) {
        return productStockMapper.selectBySpuId(spuId).getQuantity();
    }

}
```

### 2.4 验证 undo_log

```sql
-- 在事务执行过程中查看 undo_log
SELECT * FROM undo_log ORDER BY log_created DESC LIMIT 10;

-- 事务提交后，undo_log 应该被删除
-- 如果 undo_log 没有被删除，说明全局提交失败

-- 手动清理过期的 undo_log（开发环境）
DELETE FROM undo_log WHERE log_created < DATE_SUB(NOW(), INTERVAL 1 DAY);
```

### 2.5 测试检查清单

| 测试场景 | 测试方法 | 预期结果 |
|---------|---------|---------|
| 全局事务提交 | 所有分支成功 | undo_log 被删除 |
| 全局事务回滚 | 某个分支失败 | 所有分支回滚，undo_log 被删除 |
| XID 传播 | 检查请求头 | 包含 TX_XID 头 |
| 事务超时 | 执行时间超过 timeoutMills | 自动回滚 |
| 幂等性 | 分支事务重试 | 不会产生重复数据 |

## 四、组件设计精要回顾

### 4.1 Sentinel 核心设计

```
限流熔断的两层架构：

第一层：Gateway 限流
  ├── 在请求进入微服务之前限流
  ├── 基于路由 ID 限流
  └── 保护所有后端服务

第二层：服务限流
  ├── 在服务内部限流
  ├── 基于方法/资源限流
  └── 保护单个服务

两层配合：
  Gateway 限流（全局限流）
      │
      └── 服务限流（局部限流）
          │
          └── 熔断降级（故障隔离）
```

### 4.2 Seata 核心设计

```
AT 模式的工作原理：

1. 拦截 SQL
   ├── 解析 SQL，识别表和字段
   ├── 查询 before image
   └── 生成 undo_log

2. 执行业务 SQL
   └── 正常执行

3. 提交本地事务
   ├── 业务 SQL + undo_log 一起提交
   └── 向 TC 注册分支事务

4. 全局提交
   └── 删除 undo_log

5. 全局回滚
   ├── 根据 undo_log 反向补偿
   └── 删除 undo_log
```

### 4.3 Sentinel 与 Seata 的协作

```
场景：大促下单

1. 请求到达 Gateway
   │
   └── Sentinel 网关限流：检查 QPS
       ├── 超限 → 返回 429
       └── 未超限 → 继续

2. 请求到达 trade-server
   │
   ├── Sentinel 服务限流：检查服务负载
   │   ├── 超限 → 返回降级响应
   │   └── 未超限 → 继续
   │
   ├── @GlobalTransactional 开启全局事务
   │
   ├── 调用 product-server
   │   ├── Sentinel 熔断检查
   │   │   ├── 熔断 → 调用 FallbackFactory
   │   │   └── 正常 → 扣减库存
   │   └── 执行分支事务
   │
   ├── 调用 member-server
   │   ├── Sentinel 熔断检查
   │   └── 执行分支事务
   │
   └── 全部成功 → Seata 提交全局事务
       任一失败 → Seata 回滚全局事务
```

## 五、常见问题排查

| 问题 | 可能原因 | 排查方法 |
|------|----------|----------|
| 限流不生效 | 规则未配置、资源名不匹配 | 检查 Dashboard 规则、检查资源名 |
| 熔断不恢复 | 熔断时长过长、探测请求一直失败 | 检查熔断规则、检查服务健康状态 |
| FallbackFactory 不调用 | 未启用 Sentinel、未指定 fallbackFactory | 检查配置、检查注解 |
| 事务未回滚 | @GlobalTransactional 未生效、XID 未传播 | 检查注解位置、检查请求头 |
| undo_log 冲突 | 并发事务冲突 | 增加重试机制、优化事务粒度 |
| Seata Server 不可达 | 网络问题、配置错误 | 检查网络连通性、检查配置 |
| 全局事务超时 | 业务逻辑耗时过长 | 优化业务逻辑、增加超时时间 |
