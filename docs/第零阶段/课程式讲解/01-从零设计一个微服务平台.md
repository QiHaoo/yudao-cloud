# 从零设计一个微服务平台

## 开篇：我们要做什么

假设你现在接到一个任务：**从零搭建一个电商后台管理系统的技术平台**。

这个平台需要支撑：
- **商品管理** — 商品分类、SPU/SKU、库存
- **订单管理** — 下单、支付、发货、退款
- **用户管理** — 注册、登录、会员等级
- **营销管理** — 优惠券、满减、秒杀
- **内容管理** — CMS 页面、广告位
- **系统管理** — 权限、角色、菜单、字典、配置

而且，老板说了：
1. 这套系统以后要卖给不同的商家，每个商家的数据要**互相隔离**
2. 用户量可能从几百增长到几百万，架构要**能扩展**
3. 团队有 10 个后端开发，要能**并行开发**不同模块
4. 先快速上线 MVP，但不能为了快写一堆烂代码

这四个需求，分别对应了四个核心技术挑战：**多租户**、**可扩展**、**模块化**、**可维护**。

我们一步步来拆解。

## 第一个决策：单体还是微服务？

### 问题

架构选型的第一步，也是最容易让人纠结的一步：到底用单体还是微服务？

### 常见误区

很多人的直觉是"微服务更先进，当然选微服务"。但现实是：

- 微服务带来**网络调用开销**——原来一个方法调用变成一次 HTTP 请求
- 微服务带来**运维复杂度**——需要服务注册、配置中心、链路追踪、网关
- 微服务带来**分布式事务问题**——一个操作跨多个服务，数据一致性怎么办？
- 微服务需要**团队规模支撑**——3 个人的团队拆 10 个服务，光运维就累死了

反过来，纯单体也有问题：
- 所有代码在一个 JVM 里，改一个模块可能影响另一个
- 无法针对热点模块单独扩容
- 大团队协作时代码冲突频繁

### yudao-cloud 的答案：先单体，后微服务

yudao-cloud 选择了一个**两全其美**的方案：同一套代码，既能以单体模式运行，也能以微服务模式部署。

```
                    ┌─────────────────────────────┐
                    │       单体模式（开发阶段）      │
                    │                               │
                    │   ┌───────────────────────┐   │
                    │   │     YudaoServer       │   │
                    │   │  ┌─────┐ ┌─────┐     │   │
                    │   │  │system│ │infra│ ... │   │
                    │   │  └─────┘ └─────┘     │   │
                    │   │     一个 JVM          │   │
                    │   └───────────────────────┘   │
                    └───────────────────────────────┘

                    ┌───────────────────────────────┐
                    │      微服务模式（生产环境）       │
                    │                                 │
                    │   ┌───────┐  ┌───────┐         │
                    │   │system │  │ infra │  ...    │
                    │   │server │  │server │         │
                    │   └───┬───┘  └───┬───┘         │
                    │       │  Nacos   │              │
                    │       └────┬─────┘              │
                    │            │                    │
                    │   ┌────────┴────────┐           │
                    │   │     Gateway     │           │
                    │   └─────────────────┘           │
                    └─────────────────────────────────┘
```

**怎么做到的？** 关键设计：

1. **每个业务模块是一个独立的 Maven 子模块**，有自己的启动类。单体模式下，`yudao-server` 把它们打包进同一个 JVM；微服务模式下，每个模块独立部署。

2. **模块间通过 Feign 接口通信**，而不是直接 Java 方法调用。在单体模式下，Spring Cloud 会自动把 Feign 调用路由到本地的 Controller（零网络开销）；在微服务模式下，Feign 通过 Nacos 发现远程服务并发起 HTTP 调用。

3. **通过配置开关切换**：`spring.cloud.nacos.discovery.enabled=false` 就是单体模式，`=true` 就是微服务模式。

这个设计的精妙之处在于：**开发阶段用单体模式，简单高效；上线后需要扩容时，只需把热点模块拆出来独立部署，其他模块继续跑在单体里**。不需要一次性全部拆分。

## 第二个决策：项目结构怎么组织？

### 问题

一个有十几个业务模块的项目，Maven 工程结构怎么设计？

### 错误示范：一个巨大的单模块

最简单的做法是把所有代码放在一个 `src/main/java` 里。但当项目有几百个类时，问题就来了：
- 编译一次要 5 分钟
- 不知道哪个类属于哪个业务领域
- 改了用户模块的代码，支付模块的测试也挂了

### yudao-cloud 的做法：三层模块设计

```
yudao-cloud/
├── yudao-dependencies/          ← 第一层：依赖版本管理（BOM）
├── yudao-framework/             ← 第二层：框架层（17 个 Starter）
│   ├── yudao-common/
│   ├── yudao-spring-boot-starter-mybatis/
│   ├── yudao-spring-boot-starter-web/
│   ├── yudao-spring-boot-starter-security/
│   └── ...（共 17 个）
├── yudao-module-system/         ← 第三层：业务模块
│   ├── yudao-module-system-api/     ← 对外暴露的 Feign 接口 + DTO
│   └── yudao-module-system-server/  ← 实现：Controller + Service + Mapper
├── yudao-module-infra/
├── yudao-module-pay/
├── yudao-module-mall/
└── ...（共 14 个业务模块）
```

**为什么分三层？**

**第一层 `yudao-dependencies`** — 解决"版本地狱"问题。

想象一下，项目有 70 多个第三方依赖（Spring Boot、MyBatis-Plus、Redis、Flowable...）。如果每个子模块各自声明版本，很快就会出现"A 模块用 MyBatis-Plus 3.5.16，B 模块用 3.5.10"的版本漂移。BOM 把所有版本集中在一个文件里，子模块只声明"我要用 MyBatis-Plus"，不写版本号，版本由 BOM 统一管控。

```
yudao-dependencies/pom.xml（BOM）
  │
  ├── mybatis-plus.version = 3.5.16
  ├── redisson.version = 4.4.0
  ├── flowable.version = 6.8.1
  └── ...（70+ 个版本声明）

yudao-module-system/pom.xml（子模块）
  │
  └── <dependency>
        <groupId>com.baomidou</groupId>
        <artifactId>mybatis-plus-boot-starter</artifactId>
        <!-- 不写版本号！从 BOM 继承 -->
      </dependency>
```

**第二层 `yudao-framework`** — 解决"重复造轮子"问题。

每个业务模块都需要：数据库访问、Redis 缓存、接口文档、异常处理、认证鉴权、日志记录...如果每个模块自己实现一遍，代码重复量巨大，而且不一致。

`yudao-framework` 把这些横切关注点封装成 **Spring Boot Starter**。业务模块只需加一个 Maven 依赖，就自动获得这些能力。这就是"框架层"的价值——**业务开发者专注于业务逻辑，基础设施由框架提供**。

```
yudao-module-system/pom.xml
  │
  ├── <dependency>yudao-spring-boot-starter-mybatis</dependency>
  ├── <dependency>yudao-spring-boot-starter-web</dependency>
  ├── <dependency>yudao-spring-boot-starter-security</dependency>
  └── ... 加上依赖就自动获得能力，零配置
```

**第三层 `yudao-module-*`** — 解决"业务领域划分"问题。

每个业务模块（system、infra、pay、mall...）是一个独立的领域。它们之间通过 `-api` 子模块暴露的 Feign 接口通信，而不是直接引用对方的内部实现。

```
yudao-module-pay-server 依赖 yudao-module-system-api（Feign 接口）
  → 可以调用 system 模块的 DeptApi.getDeptList()
  → 不能直接访问 system 模块的 DeptMapper（内部实现）
```

这就像微服务的"接口契约"——即使在单体模式下，模块间的边界也是清晰的。

### 每个业务模块的内部结构

所有业务模块遵循完全相同的包结构约定：

```
cn.iocoder.yudao.module.system/
├── controller/
│   ├── admin/                    ← 管理端接口
│   │   ├── auth/
│   │   │   ├── AuthController.java
│   │   │   └── vo/               ← 请求/响应 VO
│   │   │       ├── AuthLoginReqVO.java
│   │   │       └── AuthLoginRespVO.java
│   │   └── dept/
│   │       ├── DeptController.java
│   │       └── vo/
│   └── app/                      ← 用户端接口
├── service/
│   ├── auth/
│   │   ├── AdminAuthService.java         ← 接口
│   │   └── AdminAuthServiceImpl.java     ← 实现
│   └── dept/
│       ├── DeptService.java
│       └── DeptServiceImpl.java
├── dal/
│   ├── dataobject/               ← 数据库实体（DO）
│   │   └── dept/
│   │       └── DeptDO.java
│   ├── mysql/                    ← MyBatis Mapper
│   │   └── dept/
│   │       └── DeptMapper.java
│   └── redis/                    ← Redis DAO（需要时）
├── convert/                      ← 对象转换器（MapStruct）
│   └── DeptConvert.java
└── enums/                        ← 模块专属枚举
```

**为什么所有模块都用同一套结构？** 因为当团队有 10 个人并行开发时，如果每个模块的代码组织方式不同，互相看对方的代码就要先花时间理解"这个模块的 Controller 放在哪"。统一结构让任何人打开任何模块都能立即找到自己需要的代码。

## 第三个决策：技术栈怎么选？

### 问题

Java 生态有无数选择。Spring Boot vs Quarkus？MyBatis-Plus vs JPA？Nacos vs Consul？每个选择都有道理，但组合在一起就是一套完整的技术栈。

### yudao-cloud 的技术选型及理由

| 关注点 | 选择 | 为什么选它 | 放弃了什么 |
|--------|------|-----------|-----------|
| 基础框架 | Spring Boot 2.7.18 | 生态最成熟，资料最多，国内社区最大 | Quarkus（更轻量但生态不够成熟） |
| 微服务 | Spring Cloud Alibaba | Nacos 一站式解决注册+配置，国内主流 | Spring Cloud Netflix（已停止维护） |
| ORM | MyBatis-Plus | SQL 可控，学习成本低，适合复杂查询 | JPA/Hibernate（抽象层太厚，复杂查询不灵活） |
| 数据库 | MySQL（默认） | 开源免费，国内使用率最高 | PostgreSQL（功能更强但国内生态略逊） |
| 缓存 | Redis + Redisson | 分布式锁、限流、缓存一站搞定 | Memcached（功能太简单） |
| 网关 | Spring Cloud Gateway | 响应式，性能好，与 Spring 生态无缝集成 | Nginx（不够灵活）、Zuul 1.x（阻塞式） |
| 工作流 | Flowable | 轻量级，BPMN 2.0 标准，社区活跃 | Activiti（维护不积极） |
| API 文档 | SpringDoc + Knife4j | 自动生成 OpenAPI 文档，Knife4j 增强 UI | 手写文档（维护成本高） |

**一个值得注意的决策：JDK 8**

yudao-cloud 的 master 分支选择 JDK 8，而不是 JDK 17 或 21。这不是技术保守，而是**现实考量**：
- 国内大量企业生产环境还在 JDK 8
- 很多中间件（如某些国产数据库驱动）对 JDK 17 支持不完善
- 项目同时维护 `master-jdk17` 分支，给能升级的团队提供选择

但为了安全，Spring Framework 和 Spring Security 被显式升级到了最新补丁版本（5.3.39 和 5.8.16），覆盖了 Spring Boot 2.7.18 默认版本中的已知 CVE。这体现了"兼容性"和"安全性"之间的平衡。

## 从一个请求说起：整体架构长什么样

与其列一张架构图，不如**跟着一个真实的 HTTP 请求走一遍**，看看它经过了哪些层、做了什么。

### 场景：管理员查询部门列表

```
前端发起请求：
GET /admin-api/system/dept/list?name=技术部&status=0
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

### 第一站：API 网关（Gateway）

请求首先到达 Spring Cloud Gateway（端口 48080）。

网关做了两件事：

**1. 路由匹配**

根据 URL 路径找到目标服务。路由规则在 `application.yaml` 中配置：

```yaml
spring.cloud.gateway.routes:
  - id: system-admin-api
    uri: grayLb://system-server    # 负载均衡到 system-server
    predicates:
      - Path=/admin-api/system/**  # 匹配 /admin-api/system/ 开头的请求
    filters:
      - RewritePath= /admin-api/(?<segment>.*), /${segment}
      # 重写路径：/admin-api/system/dept/list → /system/dept/list
```

所以 `GET /admin-api/system/dept/list` 会被转发到 `system-server` 服务的 `/system/dept/list`。

**2. 令牌校验**

网关的 `TokenAuthenticationFilter`（一个 GlobalFilter）拦截请求：

```
① 清除请求中可能伪造的 login-user 头
② 从 Authorization 头提取 Bearer Token
③ 查本地 Guava 缓存（1 分钟 TTL），命中则直接使用
④ 缓存未命中，调用 system-server 的 /oauth2/token/check 接口校验
⑤ 校验通过，把 LoginUser（id, tenantId, scopes）序列化为 JSON，
   URL 编码后放入 login-user 请求头
⑥ 转发请求到下游服务
```

**为什么在网关统一校验令牌？** 如果每个服务都自己校验，就意味着每次请求都要调用一次认证接口。在网关统一校验后，下游服务只需读取 `login-user` 头就能获得用户信息，避免了重复校验的开销。网关还用了 Guava 缓存，同一个令牌 1 分钟内只校验一次。

**为什么用 WebClient 而不是 Feign 调用认证接口？** 因为 Spring Cloud Gateway 是响应式的（基于 WebFlux），而 OpenFeign 是阻塞式的（基于 Servlet），两者不能混用。所以网关用响应式的 `WebClient` 来调用远程接口。

### 第二站：服务端安全过滤器

请求到达 `system-server` 后，经过 Spring Security 的过滤器链。

`TokenAuthenticationFilter`（一个 `OncePerRequestFilter`）做以下事情：

```
① 检查 login-user 头是否存在
② 存在 → 反序列化为 LoginUser 对象（来自网关的校验结果）
③ 不存在 → 尝试从 Authorization 头提取 Token，调用 Feign 接口校验
④ 将 LoginUser 存入 Spring Security 上下文
```

这个设计支持两种部署模式：
- **有网关**：信任网关传来的 `login-user` 头，零额外开销
- **无网关**（如直接通过 Nginx）：服务自己校验 Token

然后 Spring Security 检查 URL 权限：
- 静态资源（*.html, *.css, *.js）→ 直接放行
- 标注了 `@PermitAll` 的接口 → 直接放行
- 其他 → 要求已认证

### 第三站：Controller 层

请求到达 `DeptController`：

```java
@RestController
@RequestMapping("/system/dept")  // 实际 URL: /admin-api/system/dept/...
public class DeptController {

    @GetMapping("/list")
    @PreAuthorize("@ss.hasPermission('system:dept:query')")  // 权限校验
    @Operation(summary = "获取部门列表")
    public CommonResult<List<DeptRespVO>> getDeptList(@Valid DeptListReqVO reqVO) {
        List<DeptDO> list = deptService.getDeptList(reqVO);
        return success(BeanUtils.toBean(list, DeptRespVO.class));
    }
}
```

这里有几个设计要点：

**URL 前缀不是写死的**。Controller 只写 `@RequestMapping("/system/dept")`，`/admin-api` 前缀由框架自动拼接。框架通过 `YudaoWebAutoConfiguration` 的 `WebMvcRegistrations` 实现：扫描 Controller 的包路径，如果在 `*.controller.admin.*` 包下，自动加 `/admin-api` 前缀；如果在 `*.controller.app.*` 包下，自动加 `/app-api` 前缀。

**为什么这样设计？** 因为同一个业务逻辑（比如部门管理）可能同时需要管理端和用户端两个版本的接口。通过包路径区分前缀，Controller 的代码不需要关心自己被部署在哪个 URL 下。

**权限校验是声明式的**。`@PreAuthorize("@ss.hasPermission('system:dept:query')")` 通过 Spring Security 的 SpEL 表达式调用 `SecurityFrameworkServiceImpl.hasPermission()` 方法。这个方法会检查当前用户的权限列表（存在 Redis 中）是否包含指定权限码。如果权限不匹配，直接返回 403，请求不会到达 Service 层。

**统一返回格式**。所有接口都返回 `CommonResult<T>`：

```json
{
  "code": 0,
  "msg": "",
  "data": [
    {"id": 1, "name": "技术部", "status": 0},
    {"id": 2, "name": "产品部", "status": 0}
  ]
}
```

`code=0` 表示成功，非零表示错误。前端只需要判断 `code` 是否为 `0` 就知道请求是否成功。

### 第四站：Service 层

```java
@Service
public class DeptServiceImpl implements DeptService {

    @Override
    public List<DeptDO> getDeptList(DeptListReqVO reqVO) {
        return deptMapper.selectList(reqVO);
    }

    @Override
    @Transactional(rollbackFor = Exception.class)
    @CacheEvict(cacheNames = "dept:children:id:list", allEntries = true)
    public Long createDept(DeptCreateReqVO reqVO) {
        // 1. 校验父部门是否存在
        validateParentDept(reqVO.getParentId());
        // 2. 校验部门名称是否重复
        validateDeptNameUnique(reqVO.getParentId(), null, reqVO.getName());
        // 3. 插入数据库
        DeptDO dept = BeanUtils.toBean(reqVO, DeptDO.class);
        deptMapper.insert(dept);
        return dept.getId();
    }
}
```

Service 层是业务逻辑的核心。几个关键设计：

**接口 + 实现分离**。每个 Service 都有接口（`DeptService`）和实现（`DeptServiceImpl`）。这不仅是为了编程规范，更是为了 Spring AOP 能正常工作——Spring 的代理机制需要接口才能创建 JDK 动态代理。

**缓存通过注解实现**。`@Cacheable` 读缓存，`@CacheEvict` 清缓存。底层是 Redis，但 Service 层完全不关心缓存的具体实现。

**异常通过 `ServiceExceptionUtil.exception()` 抛出**。每个模块在 `ErrorCodeConstants` 中定义自己的错误码范围（如 system 模块是 100100001~100199999），抛出时传入错误码，`GlobalExceptionHandler` 统一捕获并转换为 `CommonResult` 返回。

### 第五站：Mapper 层（数据访问）

```java
@Mapper
public interface DeptMapper extends BaseMapperX<DeptDO> {

    default List<DeptDO> selectList(DeptListReqVO reqVO) {
        return selectList(new LambdaQueryWrapperX<DeptDO>()
                .likeIfPresent(DeptDO::getName, reqVO.getName())
                .eqIfPresent(DeptDO::getStatus, reqVO.getStatus()));
    }
}
```

这里展示了 yudao-cloud 对 MyBatis-Plus 的两层封装：

**`BaseMapperX`** — 扩展了 MyBatis-Plus 的 `BaseMapper`，提供：
- `selectPage()` — 统一分页，返回 `PageResult<T>`（包含总数和数据列表）
- `selectOneForUpdate()` — 悲观锁查询
- `insertBatch()` — 批量插入（自动处理 SQL Server 的兼容性问题）

**`LambdaQueryWrapperX`** — 扩展了 MyBatis-Plus 的 `LambdaQueryWrapper`，核心是 `*IfPresent` 系列方法：

```java
// 传统写法：每个字段都要判空
LambdaQueryWrapper<DeptDO> wrapper = new LambdaQueryWrapper<>();
if (reqVO.getName() != null) {
    wrapper.like(DeptDO::getName, reqVO.getName());
}
if (reqVO.getStatus() != null) {
    wrapper.eq(DeptDO::getStatus, reqVO.getStatus());
}

// LambdaQueryWrapperX 写法：一行搞定
new LambdaQueryWrapperX<DeptDO>()
    .likeIfPresent(DeptDO::getName, reqVO.getName())    // name 为 null 时自动跳过
    .eqIfPresent(DeptDO::getStatus, reqVO.getStatus()); // status 为 null 时自动跳过
```

在管理后台系统中，80% 的查询都是"按可选条件过滤"。`IfPresent` 方法把每个字段的空判断从 3 行缩减到 1 行，几百个查询方法累计下来省掉了几千行样板代码。

### 第六站：响应返回

```
Mapper 返回 List<DeptDO>
  ↓
Service 返回 List<DeptDO>
  ↓
Controller: BeanUtils.toBean(list, DeptRespVO.class) → DO 转 VO
  ↓
Controller: CommonResult.success(voList) → 包装为统一格式
  ↓
GlobalResponseBodyHandler: 记录响应到请求属性（用于访问日志），不修改响应
  ↓
Spring MVC: Jackson 序列化为 JSON
  ↓
Gateway: 转发响应给前端
  ↓
前端收到: {"code": 0, "msg": "", "data": [...]}
```

**DO 到 VO 的转换**：`DeptDO` 是数据库实体，包含所有字段（包括 `creator`、`createTime` 等审计字段）。`DeptRespVO` 是返回给前端的视图对象，只包含前端需要的字段。`BeanUtils.toBean()` 使用 Hutool 的反射拷贝实现转换。对于复杂转换，项目使用 MapStruct 在编译期生成类型安全的映射代码。

## 框架层：为什么需要一个"框架层"？

### 问题

你可能会问：Spring Boot 已经够强大了，为什么还要在它上面再封装一层 `yudao-framework`？

### 答案：Spring Boot 提供的是"工具"，框架层提供的是"规范"

Spring Boot 给你 `@RestController`、`@Service`、`@Mapper` 这些工具，但不告诉你：
- Controller 的返回值应该长什么样？
- 异常应该怎么处理？
- 分页应该怎么实现？
- 审计字段应该怎么填？
- 接口文档应该怎么写？

如果不统一这些规范，10 个人写出来的代码就是 10 种风格。框架层的价值是**把最佳实践固化为代码**，让开发者不需要思考"应该怎么做"，只需要遵循框架提供的模式。

### 框架层的 17 个 Starter

yudao-framework 包含 17 个 Spring Boot Starter，按职责分为两类：

**技术类 Starter（通用基础设施）**

| Starter | 它解决什么问题 |
|---------|--------------|
| `yudao-common` | 所有模块共享的基础类型：`CommonResult`、`PageParam`、`ServiceException`、`BeanUtils` |
| `yudao-spring-boot-starter-mybatis` | 数据库访问：`BaseDO`（审计字段）、`BaseMapperX`（分页/批量）、`LambdaQueryWrapperX`（动态查询） |
| `yudao-spring-boot-starter-web` | Web 层：全局异常处理、CORS、XSS 防护、API 加密、Swagger 文档 |
| `yudao-spring-boot-starter-security` | 认证鉴权：Token 过滤器、Spring Security 配置、`@PreAuthorize` 支持 |
| `yudao-spring-boot-starter-redis` | Redis 集成：缓存配置、Redisson 客户端 |
| `yudao-spring-boot-starter-rpc` | RPC 集成：OpenFeign 配置、Sentinel 限流 |
| `yudao-spring-boot-starter-mq` | 消息队列：支持 Redis Stream、RocketMQ、RabbitMQ、Kafka |
| `yudao-spring-boot-starter-job` | 定时任务：XXL-Job 集成 |
| `yudao-spring-boot-starter-excel` | Excel 导入导出 |
| `yudao-spring-boot-starter-websocket` | WebSocket 支持 |
| `yudao-spring-boot-starter-monitor` | 监控：SkyWalking 链路追踪、Spring Boot Admin |
| `yudao-spring-boot-starter-protection` | 服务保护：分布式锁（Lock4j）、幂等性、限流 |
| `yudao-spring-boot-starter-test` | 测试：`BaseDbUnitTest`（H2 内存数据库）、`BaseMockitoUnitTest` |
| `yudao-spring-boot-starter-env` | 环境配置 |

**业务类 Starter（带业务逻辑）**

| Starter | 它解决什么问题 |
|---------|--------------|
| `yudao-spring-boot-starter-biz-tenant` | 多租户：自动在 SQL 中追加 `WHERE tenant_id = ?` |
| `yudao-spring-boot-starter-biz-data-permission` | 数据权限：按部门/角色过滤数据行 |
| `yudao-spring-boot-starter-biz-ip` | IP 地理位置解析 |

**技术类 vs 业务类的区分标准**：技术类 Starter 不包含任何业务逻辑，任何 Java 项目都能用；业务类 Starter 包含了特定的业务规则（如多租户、数据权限），只适用于有这些需求的项目。框架层用 `biz` 前缀明确标识业务类 Starter，防止技术模块意外依赖业务逻辑。

### Starter 的自动配置原理

每个 Starter 都是一个标准的 Spring Boot 自动配置模块。以 `yudao-spring-boot-starter-web` 为例：

```
yudao-spring-boot-starter-web/
├── src/main/java/
│   └── cn/iocoder/yudao/framework/web/
│       ├── config/
│       │   └── YudaoWebAutoConfiguration.java  ← @AutoConfiguration
│       ├── core/
│       │   ├── handler/
│       │   │   ├── GlobalExceptionHandler.java  ← @RestControllerAdvice
│       │   │   └── GlobalResponseBodyHandler.java
│       │   ├── filter/
│       │   │   ├── CorsFilter.java
│       │   │   └── CacheRequestBodyFilter.java
│       │   └── util/
│       └── ...
└── src/main/resources/
    └── META-INF/
        └── spring.factories  ← 注册自动配置类
```

当业务模块在 `pom.xml` 中添加 `yudao-spring-boot-starter-web` 依赖后，Spring Boot 启动时会自动发现并加载 `YudaoWebAutoConfiguration`，注册所有 Web 层的基础设施 Bean。业务代码**零配置**就能获得全局异常处理、CORS、Swagger 文档等能力。

## 业务模块：怎么拆分业务领域？

### 问题

一个电商系统有那么多功能，怎么决定拆成哪些模块？模块之间怎么通信？

### yudao-cloud 的模块拆分

项目把业务功能拆分为 **14 个业务模块**：

| 模块 | 职责 | 必选/可选 |
|------|------|----------|
| `yudao-module-system` | 系统管理（用户、角色、菜单、字典、认证） | 必选 |
| `yudao-module-infra` | 基础设施（代码生成、文件管理、配置、操作日志） | 必选 |
| `yudao-module-member` | 会员中心 | 可选 |
| `yudao-module-bpm` | 工作流（Flowable） | 可选 |
| `yudao-module-pay` | 支付 | 可选 |
| `yudao-module-mp` | 微信公众号 | 可选 |
| `yudao-module-mall` | 电商（商品/促销/交易/统计） | 可选 |
| `yudao-module-crm` | 客户关系管理 | 可选 |
| `yudao-module-erp` | 企业资源计划 | 可选 |
| `yudao-module-iot` | 物联网 | 可选 |
| `yudao-module-mes` | 生产执行 | 可选 |
| `yudao-module-wms` | 仓储管理 | 可选 |
| `yudao-module-report` | 报表 | 可选 |
| `yudao-module-ai` | AI（需 JDK 17） | 可选 |

**必选 vs 可选的判断标准**：`system` 和 `infra` 是所有业务模块的基础——没有用户认证和文件管理，任何业务都跑不起来。其他模块则根据业务需求按需引入。

### 模块间通信：-api 子模块

每个业务模块拆分为两个子模块：

```
yudao-module-pay/
├── yudao-module-pay-api/       ← 对外接口（Feign 客户端 + DTO）
│   └── cn.iocoder.yudao.module.pay.api.order/
│       ├── PayOrderApi.java         ← Feign 接口
│       └── PayOrderRespDTO.java     ← 返回 DTO
└── yudao-module-pay-server/    ← 内部实现
    └── cn.iocoder.yudao.module.pay.controller/
        └── ...（Controller、Service、Mapper）
```

**`-api` 子模块只包含接口和 DTO**，不包含任何实现代码。其他模块依赖 `-api` 模块来调用这个模块的能力，但不能访问其内部实现。

```
yudao-module-mall-server
  │
  ├── 依赖 yudao-module-pay-api（可以用 PayOrderApi）
  ├── 依赖 yudao-module-system-api（可以用 AdminUserApi）
  │
  ✗ 不能依赖 yudao-module-pay-server（看不到内部实现）
  ✗ 不能依赖 yudao-module-system-server（看不到内部实现）
```

**为什么这样设计？** 想象一下如果没有 `-api` 模块，mall 模块要调用 pay 模块的支付接口，就必须依赖 pay 模块的整个实现包。这意味着 mall 模块能直接访问 pay 模块的 Mapper、Service 等内部类，模块边界形同虚设。`-api` 模块就像一个"接口门面"，暴露最小化的接口，隐藏所有实现细节。

### RPC 接口的 URL 约定

所有跨模块 RPC 调用使用统一的 URL 前缀 `/rpc-api/{module}/...`：

```
管理端接口：/admin-api/system/dept/list      ← 前端调用
用户端接口：/app-api/member/user/get         ← 前端调用
模块间调用：/rpc-api/system/dept/list-all-simple  ← 后端调用
```

三种前缀在同一套 Spring MVC 中运行，通过包路径自动区分。这样做的好处是：
- 网关可以针对 `/admin-api` 和 `/app-api` 设置不同的限流策略
- `/rpc-api` 接口不暴露给前端，安全性更好
- 安全过滤器可以对三种前缀应用不同的认证规则

## 认证鉴权：怎么统一管理身份和权限？

### 问题

一个后台管理系统，最基本的安全需求是：
1. 用户必须**登录**才能访问系统
2. 不同用户有不同的**权限**（比如只有管理员能删除用户）
3. 权限校验要**高效**（不能每次都查数据库）
4. 在微服务架构下，认证信息要在**多个服务间传递**

### yudao-cloud 的认证鉴权架构

```
                    ┌────────────────────────────────────────────────────┐
                    │                  认证鉴权架构                        │
                    │                                                    │
  用户登录 ─────────→│  AuthController                                    │
                    │      ↓ 账密校验                                     │
                    │  AdminAuthService                                  │
                    │      ↓ 生成 OAuth2 Token                           │
                    │  OAuth2TokenService                                │
                    │      ↓ Token 存入 Redis                             │
                    │                                                    │
  后续请求 ─────────→│  Gateway TokenFilter                               │
                    │      ↓ 从 Redis 校验 Token                         │
                    │      ↓ 缓存 LoginUser（1分钟）                      │
                    │      ↓ 设置 login-user 头                           │
                    │  Service TokenFilter                                │
                    │      ↓ 读取 login-user 头                           │
                    │      ↓ 存入 SecurityContext                         │
                    │  @PreAuthorize("@ss.hasPermission('xxx')")         │
                    │      ↓ 检查 Redis 中的权限列表                       │
                    │      ↓ 权限不足 → 403                               │
                    │      ↓ 权限通过 → 执行 Controller                   │
                    └────────────────────────────────────────────────────┘
```

### 令牌机制

yudao-cloud 使用自定义的 OAuth2 令牌机制（不是 Spring Security OAuth2）：

1. **登录**：用户提供用户名+密码（或手机号+验证码），`AdminAuthService` 校验后生成一个 Access Token 和一个 Refresh Token
2. **Token 存储**：Token 信息存入 Redis，key 是 `oauth2_access_token:{token}`，包含用户 ID、租户 ID、权限范围、过期时间
3. **Token 校验**：网关或服务端从 Redis 查询 Token 是否存在且未过期
4. **Token 刷新**：Access Token 过期后，用 Refresh Token 换取新的 Access Token

**为什么用 Redis 存 Token 而不是 JWT？** JWT 的优势是无状态——服务端不需要存储 Token，直接验签即可。但 JWT 有一个致命缺点：**无法主动失效**。如果用户被禁用或修改了权限，已签发的 JWT 仍然有效直到过期。Redis 存储的 Token 可以随时删除，实现即时失效。在管理后台这种对安全性要求较高的场景，Redis 方案更合适。

### 权限模型

```
用户（AdminUser）
  │
  ├── 分配角色（UserRole）
  │     │
  │     └── 角色（Role）
  │           │
  │           ├── 关联菜单（RoleMenu）
  │           │     │
  │           │     └── 菜单（Menu）
  │           │           │
  │           │           ├── 菜单类型=按钮 → 权限标识（如 system:dept:create）
  │           │           └── 菜单类型=菜单 → 页面路由
  │           │
  │           └── 数据范围（DataScope）
  │                 ├── 全部数据
  │                 ├── 本部门数据
  │                 ├── 本部门及以下数据
  │                 └── 自定义部门数据
  │
  └── 部门（Dept）← 数据权限的过滤维度
```

**权限校验的流程**：

```
① 用户登录后，系统查询其所有角色关联的菜单按钮，得到权限标识列表
   （如 ["system:dept:create", "system:dept:update", "system:user:query"]）
② 权限列表存入 Redis（key: user_permission:{userId}）
③ 请求到达 Controller 时，@PreAuthorize 调用 SecurityFrameworkServiceImpl.hasPermission()
④ 该方法从 Redis 加载用户的权限列表（带 Guava 本地缓存，1 分钟 TTL）
⑤ 检查权限列表中是否包含指定的权限标识
⑥ 包含 → 放行；不包含 → 403
```

**数据权限**是另一层过滤——它控制的是"用户能看到哪些数据行"，而不是"用户能不能访问这个接口"。比如两个销售经理都能访问订单列表接口，但 A 只能看到自己部门的订单，B 能看到所有订单。这通过 `@DataPermission` 注解和 `DataPermissionRule` 接口实现，在 SQL 层面自动追加 `WHERE dept_id IN (...)` 条件。

## 数据访问：怎么让数据库操作更高效？

### 问题

一个后台管理系统有几十甚至上百张表，每张表都需要 CRUD。如果每个 Mapper 都从零写 SQL，工作量巨大且容易出错。

### yudao-cloud 的 MyBatis-Plus 封装策略

yudao-cloud 在 MyBatis-Plus 之上做了三层封装，让数据访问层的代码量降到最低：

**第一层：BaseDO — 统一实体基类**

```java
public abstract class BaseDO implements Serializable {

    @TableField(fill = FieldFill.INSERT)       // 插入时自动填充
    private LocalDateTime createTime;
    @TableField(fill = FieldFill.INSERT)
    private String creator;
    @TableField(fill = FieldFill.INSERT_UPDATE) // 插入和更新时自动填充
    private LocalDateTime updateTime;
    @TableField(fill = FieldFill.INSERT_UPDATE)
    private String updater;
    @TableLogic                                  // 逻辑删除
    private Integer deleted;
}
```

所有数据库实体都继承 `BaseDO`。这意味着：
- **审计字段自动填充**：开发者不需要在每个 `insert`/`update` 操作前手动设置 `createTime`、`updateTime`，框架通过 `MetaObjectHandler` 自动处理
- **软删除默认启用**：所有 `delete` 操作自动变成 `UPDATE deleted=1`，所有 `select` 自动追加 `WHERE deleted=0`。数据不会真正从数据库中消失，方便审计和恢复
- **`creator` 用 String 而不是 Long**：虽然当前存的是用户 ID（数字），但用 String 为将来可能的非数字标识符预留了扩展空间

**第二层：BaseMapperX — 统一 Mapper 基类**

```java
public interface BaseMapperX<T> extends MPJBaseMapper<T> {

    // 统一分页：返回 PageResult（包含 total 和 list）
    default PageResult<T> selectPage(PageParam pageParam, LambdaQueryWrapper<T> wrapper) {
        IPage<T> page = this.selectPage(pageParam.buildPage(), wrapper);
        return new PageResult<>(page.getRecords(), page.getTotal());
    }

    // 悲观锁查询
    default T selectOneForUpdate(LambdaQueryWrapper<T> wrapper) {
        wrapper.last("FOR UPDATE");
        return selectOne(wrapper);
    }

    // 批量插入（自动处理 SQL Server 兼容性）
    void insertBatch(@Param("list") List<T> list);
}
```

所有 Mapper 继承 `BaseMapperX`，自动获得分页、悲观锁、批量操作等能力。`selectPage()` 的一个巧妙设计：当 `pageSize == -1` 时，表示"不分页，返回全部数据"（用于导出场景），框架自动降级为 `selectList()`。

**第三层：LambdaQueryWrapperX — 动态查询构建器**

这是对日常开发效率提升最大的一层：

```java
// 传统方式：每个查询条件都要写 if 判断
LambdaQueryWrapper<DeptDO> wrapper = new LambdaQueryWrapper<>();
if (StringUtils.isNotBlank(reqVO.getName())) {
    wrapper.like(DeptDO::getName, reqVO.getName());
}
if (reqVO.getStatus() != null) {
    wrapper.eq(DeptDO::getStatus, reqVO.getStatus());
}
if (reqVO.getBeginTime() != null) {
    wrapper.ge(DeptDO::getCreateTime, reqVO.getBeginTime());
}
if (reqVO.getEndTime() != null) {
    wrapper.le(DeptDO::getCreateTime, reqVO.getEndTime());
}
// 每个字段 3 行代码，10 个字段就是 30 行

// LambdaQueryWrapperX 方式：
new LambdaQueryWrapperX<DeptDO>()
    .likeIfPresent(DeptDO::getName, reqVO.getName())
    .eqIfPresent(DeptDO::getStatus, reqVO.getStatus())
    .geIfPresent(DeptDO::getCreateTime, reqVO.getBeginTime())
    .leIfPresent(DeptDO::getCreateTime, reqVO.getEndTime());
// 每个字段 1 行代码，10 个字段就是 10 行
```

`likeIfPresent` 的逻辑是：如果 `name` 为 null 或空字符串，就跳过这个条件；否则追加 `LIKE` 条件。对于管理后台这种"搜索条件都是可选的"场景，这个模式消除了大量的 `if` 判断。

`betweenIfPresent` 还有一个聪明的设计：如果只提供了开始时间没有结束时间，它会自动降级为 `>=`；如果只提供了结束时间没有开始时间，它会降级为 `<=`。这完美匹配了前端日期范围选择器的常见交互——用户可能只填一个边界。

## 多租户：怎么让一套系统服务多个客户？

### 问题

SaaS 系统的核心挑战：一套代码、一个数据库，怎么保证 A 商家看不到 B 商家的数据？

### yudao-cloud 的多租户方案

yudao-cloud 采用**共享数据库、共享 Schema、行级隔离**的方案：

```
数据库：ruoyi-vue-pro
┌─────────────────────────────────────────────┐
│  system_dept 表                              │
│  ┌────┬──────────┬───────────┐              │
│  │ id │ name     │ tenant_id │              │
│  ├────┼──────────┼───────────┤              │
│  │ 1  │ 技术部   │ 1         │ ← 商家 A    │
│  │ 2  │ 产品部   │ 1         │ ← 商家 A    │
│  │ 3  │ 运营部   │ 2         │ ← 商家 B    │
│  │ 4  │ 财务部   │ 2         │ ← 商家 B    │
│  └────┴──────────┴───────────┘              │
└─────────────────────────────────────────────┘
```

每张需要租户隔离的表都有一个 `tenant_id` 字段。查询时自动追加 `WHERE tenant_id = ?`，确保每个租户只能看到自己的数据。

**怎么实现"自动追加"？** 不是在每个 SQL 里手动写 `WHERE tenant_id = ?`，而是通过 MyBatis-Plus 的 `TenantLineHandler` 拦截器，在 SQL 执行前自动修改 SQL 的 AST（抽象语法树）。

```java
// 开发者写的 SQL（通过 MyBatis-Plus 自动生成）：
SELECT id, name FROM system_dept WHERE status = 0

// 拦截器自动修改后的 SQL：
SELECT id, name FROM system_dept WHERE status = 0 AND tenant_id = 1
```

**业务代码完全不感知租户的存在**。开发者写 Mapper 时不需要关心 `tenant_id`，查询、插入、更新都会被自动处理。插入时，`tenant_id` 也会被自动填充。

### 哪些表需要租户隔离？

不是所有表都需要租户隔离。通过**继承关系**自动判断：

- 继承 `TenantBaseDO` 的实体 → 需要租户隔离（自动追加 `WHERE tenant_id`）
- 继承 `BaseDO` 的实体 → 不需要租户隔离（全局共享数据）
- 标注 `@TenantIgnore` 的实体 → 显式排除

```
BaseDO（无租户字段）
  ├── DictDataDO         ← 字典数据，全局共享
  ├── OAuth2ClientDO     ← OAuth2 客户端，全局共享
  └── TenantBaseDO（有 tenant_id 字段）
        ├── DeptDO       ← 部门，按租户隔离
        ├── UserDO       ← 用户，按租户隔离
        └── OrderDO      ← 订单，按租户隔离
```

**为什么用继承而不是配置文件？** 因为继承关系在编译期就确定了，IDE 能直接看到一个实体是否需要租户隔离。如果用配置文件，开发者可能忘记在配置中添加新表，导致数据泄露。

### 租户上下文传播

租户 ID 需要在整个请求链路中传播：

```
HTTP 请求 → TenantContextWebFilter 从 Header 提取 tenantId
  ↓
Service 调用 → TenantContextHolder（ThreadLocal）持有 tenantId
  ↓
数据库操作 → TenantDatabaseInterceptor 从 TenantContextHolder 读取 tenantId
  ↓
跨服务 RPC → Feign RequestInterceptor 自动传递 tenantId Header
  ↓
消息队列 → MQ Interceptor 自动传递 tenantId
  ↓
定时任务 → @TenantJob AOP 自动设置 tenantId
```

## 跨模块通信：模块之间怎么协作？

### 问题

`mall` 模块需要调用 `pay` 模块创建支付订单，`pay` 模块需要调用 `system` 模块查询用户信息。模块之间怎么通信？

### 单体模式 vs 微服务模式

```
单体模式：
mall-server ──── Feign 调用 ────→ pay-server
  │                                    │
  └──── 同一个 JVM，实际是本地方法调用 ───┘
  （Spring Cloud 自动优化：Feign → 本地 Controller）

微服务模式：
mall-server ──── HTTP 请求 ────→ pay-server
  │                                    │
  └──── 通过 Nacos 服务发现，实际是网络调用 ──┘
```

两种模式下，代码完全一样。开发者只写 Feign 接口，运行时由框架决定是本地调用还是远程调用。

### 跨模块调用的代码示例

`mall` 模块要调用 `pay` 模块的支付能力：

```java
// 1. pay 模块在 -api 子模块中定义接口
@FeignClient(name = "pay-server")
public interface PayOrderApi {

    @GetMapping("/rpc-api/pay/order/create")
    CommonResult<Long> createOrder(@RequestBody PayOrderCreateReqDTO reqDTO);
}

// 2. mall 模块依赖 pay-api，注入 Feign 接口
@Service
public class MallOrderServiceImpl {

    @Resource
    private PayOrderApi payOrderApi;  // 注入 Feign 客户端

    public void createOrder(MallOrderDO order) {
        // 调用 pay 模块创建支付订单
        CommonResult<Long> result = payOrderApi.createOrder(
            new PayOrderCreateReqDTO().setOrderId(order.getId())
        );
        // ...
    }
}
```

### 为什么用 Feign 接口而不是直接 Java 调用？

如果模块间直接 Java 调用（`payOrderService.createOrder()`），就必须依赖 `pay-server` 的实现包。这会导致：
1. **编译期耦合**：mall 模块编译时需要 pay 模块的所有类
2. **无法独立部署**：微服务模式下，两个模块在不同的 JVM 里，无法直接 Java 调用
3. **循环依赖风险**：如果 pay 模块也需要调用 mall 模块，就会产生循环依赖

Feign 接口通过 HTTP 协议解耦了两个模块，`-api` 子模块只包含接口和 DTO（非常轻量），不会引入实现层的依赖。

### 两级 API 体系

项目中有两种级别的跨模块 API：

1. **业务 API**（`-api` 子模块中）：如 `DeptApi`、`PayOrderApi`，面向具体业务场景
2. **通用 API**（`yudao-common` 中）：如 `OAuth2TokenCommonApi`、`PermissionCommonApi`，框架级的跨模块调用

通用 API 放在 `yudao-common` 而不是某个 `-api` 模块中，是因为**框架层（如 security starter）也需要调用系统模块的能力**（如校验 Token、检查权限），但如果把接口放在 `system-api` 中，框架层就会依赖业务模块，产生反向依赖。

## 统一响应与异常处理：怎么让前端开发更顺畅？

### 问题

如果每个接口返回的格式不一样，前端就需要为每个接口写不同的解析逻辑。如果异常处理不统一，前端就不知道该怎么展示错误信息。

### 统一响应格式：CommonResult

```java
public final class CommonResult<T> implements Serializable {
    private int code;    // 错误码，0 表示成功
    private String msg;  // 错误消息
    private T data;      // 业务数据
}
```

所有接口都返回 `CommonResult<T>`。前端只需要：

```javascript
const response = await fetch('/admin-api/system/dept/list');
const result = await response.json();

if (result.code === 0) {
    // 成功，使用 result.data
} else {
    // 失败，展示 result.msg
}
```

### 全局异常处理：GlobalExceptionHandler

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 业务异常
    @ExceptionHandler(ServiceException.class)
    public CommonResult<?> handleServiceException(ServiceException ex) {
        return CommonResult.error(ex.getCode(), ex.getMessage());
    }

    // 参数校验异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public CommonResult<?> handleValidException(MethodArgumentNotValidException ex) {
        String message = ex.getBindingResult().getFieldError().getDefaultMessage();
        return CommonResult.error(ErrorCodeConstants.VALIDATION_ERROR.getCode(), message);
    }

    // 权限不足
    @ExceptionHandler(AccessDeniedException.class)
    public CommonResult<?> handleAccessDeniedException(AccessDeniedException ex) {
        return CommonResult.error(403, "没有权限访问");
    }

    // 兜底：未知异常
    @ExceptionHandler(Exception.class)
    public CommonResult<?> handleException(Exception ex) {
        log.error("未知异常", ex);
        // 记录到错误日志系统
        return CommonResult.error(500, "系统内部错误");
    }
}
```

**设计要点**：

1. **业务异常用 `ServiceException`**：每个模块在 `ErrorCodeConstants` 中定义错误码范围，通过 `ServiceExceptionUtil.exception(ERROR_CODE)` 抛出。前端根据错误码可以做精细化处理（如 `100100001` 表示"部门名称已存在"，前端可以在对应输入框下方显示错误提示）。

2. **未知异常记录到日志系统**：`GlobalExceptionHandler` 在捕获未知异常时，会通过 `ApiErrorLogCommonApi` 把错误信息（包括堆栈、请求参数、用户信息）记录到数据库，方便运维排查。

3. **前端只需处理一种格式**：不管是业务错误、参数错误、权限错误还是系统错误，前端都按 `code !== 0` 的逻辑处理。

## 网关：为什么需要一个"门卫"？

### 问题

如果每个微服务都直接暴露给前端，会有什么问题？

1. **前端需要知道每个服务的地址**——系统服务在 10.0.0.1:8080，支付服务在 10.0.0.2:8081...
2. **跨域问题**——前端同时调用多个服务，每个服务都要配置 CORS
3. **安全问题**——每个服务都要独立做 Token 校验
4. **无法统一限流**——某个接口被恶意刷请求，只能在每个服务里分别限流

### 网关的职责

```
                    ┌──────────────────────────────────┐
                    │           Gateway (48080)          │
                    │                                    │
  浏览器 ──────────→│  ① 路由：根据 URL 找到目标服务      │
                    │  ② 认证：统一校验 Token             │
                    │  ③ 限流：Sentinel 保护后端服务       │
                    │  ④ 跨域：统一配置 CORS              │
                    │  ⑤ 聚合：Knife4j 文档聚合           │
                    │                                    │
                    │  路由规则：                          │
                    │  /admin-api/system/** → system     │
                    │  /admin-api/pay/** → pay           │
                    │  /admin-api/mall/** → product      │
                    │                                    │
                    └──────────────────────────────────┘
```

**路由规则的设计**：网关按 URL 路径前缀把请求分发到不同的微服务。规则是 `/admin-api/{module}/**` → `{module}-server`。这意味着前端只需要知道网关的地址，不需要知道后端有多少个服务。

**Token 校验的缓存策略**：网关对 Token 校验结果做了 Guava 本地缓存（1 分钟 TTL），key 是 `(tenantId, token)`。在 1 分钟内，同一个 Token 的多次请求只校验一次，大幅减少了对 system-server 的调用压力。

**为什么是 WebFlux 而不是 Servlet？** Spring Cloud Gateway 基于 Netty + WebFlux，是完全异步非阻塞的。这意味着一个线程可以处理成千上万个并发连接，而传统的 Servlet 模型每个连接占用一个线程。在网关这种高并发场景下，WebFlux 的性能优势很明显。但这也带来了一个限制：网关内部不能使用 Feign（Feign 是阻塞式的），所以 Token 校验用的是响应式的 WebClient。

## 代码生成：怎么减少重复劳动？

### 问题

每个新的业务功能都需要创建 DO、Mapper、Service、Controller、VO、API...这些代码结构高度相似，手动编写既枯燥又容易出错。

### yudao-cloud 的代码生成器

`yudao-module-infra` 模块内置了一个代码生成器，可以从数据库表自动生成完整的 CRUD 代码：

```
数据库表：mall_product
  ↓ 代码生成器
  ├── MallProductDO.java              ← 数据对象
  ├── MallProductMapper.java          ← Mapper 接口
  ├── MallProductService.java         ← Service 接口
  ├── MallProductServiceImpl.java     ← Service 实现
  ├── MallProductController.java      ← Controller
  ├── MallProductSaveReqVO.java       ← 保存请求 VO
  ├── MallProductPageReqVO.java       ← 分页查询 VO
  ├── MallProductRespVO.java          ← 响应 VO
  ├── MallProductConvert.java         ← MapStruct 转换器
  └── create_tables.sql               ← 建表 SQL
```

生成的代码遵循框架的所有约定（BaseDO 继承、CommonResult 返回、@PreAuthorize 权限注解、LambdaQueryWrapperX 查询），可以直接运行。

**代码生成器的价值不仅是"省时间"，更是"保一致"**。即使团队有 10 个人，生成的代码风格也是完全统一的。

### 核心设计决策回顾

| 决策 | 选择 | 理由 |
|------|------|------|
| 部署模式 | 单体 + 微服务双模式 | 开发简单，按需拆分 |
| 模块间通信 | Feign 接口 + `-api` 子模块 | 编译期隔离，运行时透明切换 |
| 数据访问 | MyBatis-Plus + 三层封装 | SQL 可控，减少样板代码 |
| 认证方式 | Redis 存 Token（非 JWT） | 支持主动失效 |
| 多租户 | 共享 Schema + SQL 自动改写 | 零业务代码改动 |
| 权限模型 | RBAC + 数据权限 | 接口级 + 数据行级双重控制 |
| 异常处理 | 单一异常类型 + 错误码枚举 | 前后端统一格式 |
| URL 设计 | 三前缀（admin-api / app-api / rpc-api） | 区分调用方，便于安全管控 |

### 读者收获

读完这篇讲解，你应该建立了以下认知：

1. **一个微服务平台不是"用了 Spring Cloud 就行"**，而是需要在项目结构、模块划分、框架封装、认证鉴权、数据访问、多租户等十几个维度做出系统性的设计决策
2. **"先单体后微服务"不是妥协，而是务实**——同一套代码支持两种部署模式，让架构随业务需求演进
3. **框架层的价值是"固化最佳实践"**——通过 Spring Boot Starter 机制，把认证、分页、异常处理等横切关注点封装为零配置的能力
4. **模块间通过 Feign 接口通信**，而不是直接 Java 调用——这是同时支持单体和微服务的关键设计
